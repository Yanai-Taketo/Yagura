## 実効値の読み戻し（OS の丸め挙動）

実装（`UdpSyslogListener.ApplyReceiveBufferSize`）は設定値を `Socket.ReceiveBufferSize` に設定した直後に
読み戻し、要求値と異なれば警告ログを出す防御的な作りになっている。本キャンペーンの起動ログ（各
`*.hostlog.txt` の `UDP 受信バッファサイズを N バイトに設定しました。`）はすべて `情報` レベルで
記録されており、**丸めは一度も発生しなかった**（64KB・1MB・4MB・16MB のいずれも要求値どおりに
適用された）。独立した最小プログラムでも同じ 4 値を `Socket.ReceiveBufferSize` に設定→読み戻しで
再確認し、`matches=True` を確認済み:

```
requested=     65536 effective(readback)=     65536 matches=True
requested=   1048576 effective(readback)=   1048576 matches=True
requested=   4194304 effective(readback)=   4194304 matches=True
requested=  16777216 effective(readback)=  16777216 matches=True
```

legacy-lessons.md A-1 が記録する「Windows では約 1MB 超の指定は setsockopt が失敗する」という制約は、
本開発機（Windows 11 ARM64 10.0.26200・.NET SDK 10.0.301）では**再現しなかった**。これは本環境限定の
確認であり、他の Windows バージョン・アーキテクチャでの再現性を保証するものではない——実装側は
丸め・失敗のいずれが将来観測されても起動を止めず警告ログに記録する設計を維持する。

### 読み取れること

1. **OS バッファ破棄（導出値）は 1MB 以上でこのキャンペーンの全 12 セルにおいて厳密にゼロになった**。
   64KB（OS 既定）では総損失のうち相当割合（20000 msg/sec で 24%、60000 msg/sec で 38%）が
   OS レベルの無計測ロスであり、これは ADR-0002 決定 2・architecture.md §3.1 が避けようとしている
   「計測困難なロスへの転化」そのものである。**1MB 以上への拡大でこの種のロスは本開発機では観測されなくなった。**
2. **1MB を超えて 4MB・16MB へ拡大しても、総損失・Q1 破棄件数に一貫した追加の改善は見られない**
   （例: 30000 msg/sec の Q1 破棄は 1MB=7370, 4MB=19074, 16MB=18557 で 1MB が最小、単調な改善ではない）。
   OS バッファ破棄が 1MB 以上でゼロに落ちた時点で律速要因は **Q1（解析段前の内部キュー。容量固定
   1024。[M-1](../../../../docs/design/architecture.md#9-実測待ち一覧)）** に完全に移っており、
   受信バッファをさらに拡大しても Q1 の容量自体は変わらないため、この移行後の変動は
   バッファサイズではなく**実行間のスケジューリング・GC 等の揺らぎ**に支配されていると解釈するのが
   最も整合的である（同一セルの再試行で結果が変わることを別途確認済み——次項参照）。
3. **破棄ゼロを維持できる上限（33 秒間・完全ゼロ）はどのバッファサイズでも 20000 msg/sec 付近が
   境目**であり、1MB のみ 20000 msg/sec で完全ゼロ（Q1・OS 破棄とも 0）を記録した。ただし
   下記の再試行確認のとおりこの境目は揺らぎの範囲内にあり、「1MB だけが特別に達成できる性能」と
   解釈すべきではない。

### 実行間の揺らぎ（重要な留保）

64KB・20000 msg/sec のセルを本キャンペーンとは別に 1 回追加実行したところ、Q1 破棄 0・OS バッファ
破棄（導出値）510 件という、キャンペーン本番の同一セル（Q1 破棄 1655・OS 破棄 532）とは異なる内訳
（ただし総損失規模は近い）を記録した。**この開発機（他プロセスと共有する対話的な Windows 環境）では
20000〜40000 msg/sec 帯の破棄件数に実行間の揺らぎがあり、本表の値は「その回の 1 回の実測」であって
「この開発機の決定論的な性能限界」ではない**——過去の M7-2 実測（`tools/Yagura.Bench/results/
2026-07-05-dev-machine/`）でも同一レート・同一条件で複数回の再試行の間に 0 件〜4 万件超まで
振れる例が記録されている。この限界は architecture.md M-6（基準環境での性能公称値の確定）の対象で
あり、本キャンペーンは「開発機参考値」の位置づけを超えない。

## BurstQ1Drop: バースト 20000 の 64KB vs 4MB 比較

| バッファ | 送出数 | 保存数 | Q1破棄(内部) | OSバッファ破棄(導出値) |
|---|---|---|---|---|
| 64KB (65536) | 20000 | 2250 | 2355 | **15395** |
| 4MB (4194304) | 20000 | 3353 | 16647 | **0** |

（原本: `BurstQ1Drop-20260705-150421.*`（64KB）・`BurstQ1Drop-20260705-150431.*`（4MB））

### 読み取れること

バースト（4 送信ソケットで 20000 通を約 0.14〜0.15 秒に一斉送出）では、64KB では総損失
17,750 件のうち **86.7%（15,395 件）が OS レベルの無計測ロス**として発生した。4MB へ拡大すると
このロスは完全にゼロになり、代わりに**同じ規模の損失が Q1 破棄という「計測される」形**で発生する
（16,647 件）。総損失件数はほぼ同水準（17,750 件 vs 16,647 件）だが、**損失の性質が「アプリからは
見えない OS 側の喪失」から「発生箇所別カウンタに必ず計上される喪失」に変わる**——これは
architecture.md §3.1「受信段では、OS 側の計測不能なロスに転化するブロックを選ばない」という
設計原則が UDP バーストにおいても実際に機能することを示す一次データである。

## 既定値の決定

**`UdpSyslogListenerOptions.DefaultReceiveBufferBytes` を 1 MiB（1,048,576 バイト）とする。**

判断根拠（数値は上表参照）:

1. **主要な判断基準（OS レベル無計測ロスの解消）は 1MB で達成される**——本キャンペーンの全 12 セルで
   OS バッファ破棄（導出値）は 1MB 以上のすべてで厳密にゼロ。4MB・16MB へさらに拡大しても
   この指標に追加の改善はない（すでにゼロ）。
2. **総損失・Q1 破棄の観点でも 1MB を超える拡大に一貫した追加の恩恵は確認できなかった**
   （§「読み取れること」2 参照）。律速要因が OS バッファから Q1（容量固定）へ移った後は、
   バッファをさらに大きくしても Q1 の容量自体は変わらないため理論的にも追加の改善余地は乏しい。
3. **メモリ消費**: 1MB は単一 UDP リスナが保持する妥当な量。4MB・16MB は追加のメモリ確保
   （それぞれ既定の 4 倍・16 倍）に見合う実測上の恩恵が本キャンペーンでは確認できなかった
   ため、「効果が実証された最小の値」を採用する。
4. 上限（`MaxReceiveBufferBytes` = 256MiB）・下限（`MinReceiveBufferBytes` = 64KiB = OS 既定値）は
   実装時の暫定値をそのまま維持する（256MiB は「異常値を弾く安全弁」という位置づけであり、
   本キャンペーンの実測対象ではない）。

**この決定は開発機 1 台・1 回ずつの実測に基づく参考値である**（§「実行間の揺らぎ」参照）。
基準環境・複数回試行での再確認は architecture.md M-6（基準環境の確定）に委ねる。1MB という
既定値そのものが将来の実測で不十分と判明した場合は、本キャンペーンと同じ形式（バッファ×レート
比較表）で再測定し、本 README を更新の上で既定値を見直す。
