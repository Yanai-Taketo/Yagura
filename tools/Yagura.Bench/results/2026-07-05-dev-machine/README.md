# M7-2 実測結果(開発機。2026-07-05)

- 環境: Windows 11 ARM64 25H2(開発機)、.NET 10、SQLite provider。詳細は各 JSON の Environment 節
- 実行: PowerShell(実コンソール相当)からベンチハーネス(M7-1 + 本ブランチの検証器改訂)で実施。全実行でグレースフル停止成立(フォールバック警告なし)
- **注意**: `Throughput-2026*-122x/123x` と `BurstQ1Drop-2026*-1236/1237` の初期分は検証器改訂前(OS 統計を突合式に含む旧式)の実行。旧式の「差分」非ゼロ・NG 判定は OS バッファ破棄の未帰属によるもので、改訂後の解釈では全損失帰属で成立する

## 主要な結果

### UDP 持続流量の破棄ゼロ上限(10 秒 × 各レート)

| レート | OS バッファ破棄(導出) | 判定 |
|---|---|---|
| 〜20,000/s | 0 | 破棄ゼロ維持 |
| 30,000/s | 147 | 限界近傍 |
| 40,000〜60,000/s | 10,035〜24,204 | OS バッファが律速 |

**破棄ゼロ上限 ≈ 20,000〜30,000 msg/sec**(既定のソケット受信バッファ)。破棄はすべて OS ソケットバッファ(Q1 破棄・アプリ内破棄はゼロ)→ 受信バッファ拡大(M-2)が第一の改善レバー。

### TCP 持続流量

50,000/s(実効 49,937)まで破棄ゼロ・完全一致。TCP はフロー制御により受信側律速で自然に抑制される。パイプライン実効上限は約 60,000/s(送出目標 70k/90k とも実効 60k 前後で頭打ち)。

### バースト時の Q1 破棄(M-1 の §3.1 前提検証)

| バースト規模 | 保存 | Q1 破棄 | OS バッファ破棄(導出) |
|---|---|---|---|
| 1,000 | 1,000 | 0 | 0 |
| 2,000 | 1,025 | 975 | 0 |
| 5,000 | 1,025 | 1,377 | 2,598 |
| 10,000 | 2,306 | 1,295 | 6,399 |
| 20,000 | 2,250 | 10,916 | 6,834 |

**Q1 破棄はバースト時に限られ、持続流量(〜30k/s)では発生しない = §3.1 の前提は成立**(恒常発生ではない → 生データ退避の再評価は不要)。Q1 容量 1024 の現仮値はバースト 1,000 まで吸収。

### SQLite 書き込み上限(M-7。TCP 経由で DB を律速にして測定)

| レート(実効) | 保存件数 | スプール退避 |
|---|---|---|
| UDP 1,000〜20,000/s × 15s | 全量 | 0 |
| TCP 29,941/s × 15s | 450,058 | 223(発火開始) |
| TCP 49,950/s × 15s | 749,887 | 16,843(明確な飽和) |
| TCP 61,664/s × 15s | 927,572 | 20,599 |

**SQLite 飽和の開始 ≈ 30,000 msg/sec(開発機参考値)**。全実行で突合成立 = 飽和中も**損失ゼロ**(スプール退避 → drain → 全量保存)。§5.3 の速度不足分類の前提(退避と成功書き込みの混在)が実測で確認できた。
**退避契機の形態(Q2 溢れ / タイムアウト)は現行実装では観測不能**(hostlog に判別情報なし・カウンタも区別なし)——形態の判別可視化は残課題として M-7/M-13 に記録。

### スプール発動 → drain(M-3 入力)

容量 1 MiB + バースト 20,000 で発動誘発 × 3 回: drain 所要 0.204〜0.212 秒(退避 1〜81 件の小規模 backlog)。振動(発動⇔停止の繰り返し)は観測されず。大規模 backlog の drain 所要は SQL Server 実測(オーナー環境)と合わせて追加測定が望ましい。

### ディスク消費(DB-1/SEC-2 相談材料。約 30,000 件保存後の SQLite ファイルサイズ)

| メッセージ(パディング) | DB サイズ | 1 レコード実効 |
|---|---|---|
| 基本(約 100 B)+0 | 5,832,704 B | 約 194 B |
| +100 B | 8,974,336 B | 約 299 B |
| +300 B | 14,950,400 B | 約 498 B |

**レコード単価 ≈ メッセージ長 + 約 95 B(インデックス込み)。本文 1 バイト増 ≈ 単価 1.01〜1.05 B 増**(ほぼ線形)。
