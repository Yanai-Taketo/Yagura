# 通知・途絶検知の lab 検証手順 — ADR-0017 / ADR-0018

> **実施済み(2026-07-24。lab: `WIN-DV7G9OMOF4U` = Windows Server 2025 `10.0.26100`・`yagura.test` DC・Yagura v0.5.0〈`main` 自ビルド〉・保存先 SQL Server 2022 Express・SMTP は smtp4dev 3.15.0)**。2026-07-20 作成。Issue #393。本手順は PR #366(メール通知)の受け入れ基準のうち実 SMTP サーバ・実機負荷を要して未実施のまま残った項目と、ADR-0018 受け入れ基準の「ホットパス追加コストの計測」を lab で実施するためのもの。
> 実測結果は [ADR-0017 改訂履歴 4](../../docs/adr/0017-email-notification.md)(A〜C)と [ADR-0018 改訂履歴 2](../../docs/adr/0018-source-silence-detection.md)(D)へ記録し、Issue #393 にも要約をコメントした。**実施上の差異(ヘッドレス lab)**: A〜C の設定は設定画面ではなく `yagura.json` 手編集で行った(設定画面 UI 経路・テスト送信ボタンは Blazor circuit のため別途 UI 確認とする)。C の平文不発生は `pktmon` が loopback を捕捉しないため、平文 SMTP サーバ自身のセッション記録(受信 SMTP コマンド列)を線上の証跡に用いた。

**実施環境**: Yagura サービスがインストール済みの lab。A〜C には lab 内で自由に止められる SMTP サーバが必要(推奨: 開発用 SMTP サーバ——smtp4dev / Papercut SMTP 等——を lab 内の別ホストに立てる。STARTTLS を有効化できるものを選ぶこと)。D には負荷生成用に `tools/Yagura.Bench`(Windows 上でビルド済み)を使う。

## なぜ CI ではなく lab なのか

| 件 | CI で実施できない理由 |
|---|---|
| A(E2E 送達) | 実 SMTP サーバと受信メールボックスの確認が要る。CI にはどちらもない |
| B(SMTP 停止耐性) | 「SMTP 側の停止・遅延・ブラックホールが受信スループット・UI 応答に影響しない」ことの実測。CI のユニットテストは設計(投入側はキューへ積むだけで戻る)を検証済みだが、実プロセス + 実負荷での確認が未実施 |
| C(平文送信の不発生) | パケットレベルの確認(キャプチャ)が要る。ユニットテストは `SecureSocketOptions.StartTls` 固定を検証済みだが、線上に平文が流れないことの実証は別物 |
| D(ホットパス実測) | ADR-0018 受け入れ基準が「空リストの素通し計測では検証にならない」と明記する実負荷計測。CI の回帰ベンチにはウォッチリスト有効の条件がない |

## 実施の原則

- A〜C は**設定画面から操作する**(手編集ではなく、利用者が実際に通る経路で検証する)
- 結果は本ファイルではなく **ADR-0017 / ADR-0018 の帰結・受け入れ基準の欄と Issue #393 へ記録**する(本ファイルは手順のみを持つ——実施記録の一次置き場にしない)

---

## A. メール通知の E2E 送達(PR #366 受け入れ基準)

**検証する仮説**: 設定画面での設定 → テスト送信成功 → 実際の能動通知(allowlist 内の実事象)がメールとして届く、が一連で成立する。

1. lab の SMTP サーバを起動し、Yagura の管理画面 `/admin/email-notification` で Host・Port・Security(`auto`)・From・To を設定して保存する
2. **テスト送信**を実行し、成功表示と SMTP サーバ側の受信を確認する(監査 2022 が記録されることも確認)
3. 実事象で発火させる: 例として TLS 受信証明書の期限接近(1017)か、DB を停止してスプール退避の開始(1004 系)など、**allowlist に載る Warning 以上の実事象**を 1 つ起こす
4. **期待結果**: 手順 3 の事象がメールで届く。件名・本文に EventId と発生内容が含まれ、イベントログにも同じ事象が記録されている
5. 同一 EventId をもう一度短時間で発火させ、**再送抑制(60 分)で 2 通目が来ない**ことを確認する

## B. SMTP 停止・遅延・ブラックホールでの無影響(PR #366 受け入れ基準)

**検証する仮説**: SMTP 側がどう壊れても、受信スループットと UI 応答は影響を受けない(投入側はキューに積むだけで戻る設計の実証)。

1. A の構成のまま、`tools/Yagura.Bench` の UDP 持続シナリオで受信負荷をかけ、正常時のスループットを記録する
2. 次の 3 状態をそれぞれ作り、負荷をかけ直して比較する:
   - **停止**: SMTP サービスを停止(接続拒否)
   - **ブラックホール**: SMTP ホストへの経路をファイアウォールで DROP(SYN が捨てられ接続タイムアウトまで待つ状態——拒否より悪い)
   - **応答遅延**: 可能なら SMTP サーバ側で応答を遅延させる(smtp4dev 等で困難なら省略し、その旨を記録)
3. 各状態で通知対象の事象を発火させ、メールが送れない状態を実際に作る
4. **期待結果**: いずれの状態でも (a) 受信スループットが正常時と有意差なし (b) 管理画面・閲覧画面の応答が体感で変わらない (c) 送信失敗 1025 が抑制窓付きで記録される (d) SMTP 復旧後、再試行(1 回・5 分後)の生き残りが届く
5. サービス停止(`Restart-Service Yagura`)が SMTP ブラックホール中でも**引き延ばされない**ことを確認する(停止時は応答待ちをしない設計の実証)

## C. `required` で STARTTLS 不成立時に平文送信が発生しない(PR #366 受け入れ基準)

**検証する仮説**: `Security=required` で相手が STARTTLS を広告しない・確立できない場合、送信は失敗し、**平文の MAIL FROM / RCPT TO / DATA が線上に流れない**。

1. SMTP サーバの STARTTLS を無効化する(または STARTTLS 非対応の簡易 SMTP を使う)
2. Yagura 側を `Security=required` にしてテスト送信する
3. Yagura ホストでパケットキャプチャ(`pktmon` / Wireshark)を取りながら実行する
4. **期待結果**: 送信は失敗として表示され(平易化された理由付き)、キャプチャ上は EHLO と応答までで、**AUTH・MAIL FROM 以降の平文コマンドが存在しない**。`auto` に切り替えると同じサーバへ平文で送信される(日和見の対比確認)

## D. 途絶検知のホットパス追加コスト実測(ADR-0018 受け入れ基準)

**検証する仮説**: ウォッチリスト上限(1000 件)登録 + 登録外送信元が混在する受信負荷で、受信スループットへの追加コストが実用上無視できる(ADR-0018 は「空リストの素通し計測では検証にならない」と明記)。

1. ベースライン: ウォッチリスト空(機能無効)で `tools/Yagura.Bench` の UDP 持続シナリオを実行し、スループットを記録する
2. ウォッチリストに 1000 件(上限)を登録する(手編集で機械生成した `Notification:SourceSilence:Watchlist` を投入してよい——負荷条件の準備であり利用者経路の検証ではないため)。**Bench の送信元と一致するアドレスを一部含め、大半は登録外**という混在条件にする
3. 同一シナリオを再実行し、1 と比較する
4. **期待結果**: スループット低下が計測誤差の範囲(目安: CI 回帰ベンチの許容帯と同等)に収まる。有意な低下が出た場合は ADR-0018 受け入れ基準の当該項目に実測値を記録し、対処を Issue 化する
