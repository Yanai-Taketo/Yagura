# ADR-0012: 管理リモート HTTPS の証明書選択の UI 設定化

- 状態: proposed
- 日付: 2026-07-11
- 決定者: YANAI Taketo（予定）
- 関連: [ADR-0010](0010-admin-ui-authentication.md)（管理 UI 認証・リモート管理の解禁。本 ADR は決定 1・決定 4 が確定した「認証必須・HTTPS 必須で解禁されるリモートバインド」の**設定手段**を、`yagura.json` 手編集から管理 UI へ拡張する——ただし決定 1・決定 4 のどの記述も偽にはならないため supersession ではなく独立 ADR。委任事項 15 の CF-5 依存を本 ADR が引き継ぐ）/ [ADR-0003](0003-ui-policy.md)（Blazor Interactive Server 単一モード。新規画面も同方式）/ [ADR-0002](0002-architecture-principles.md) 決定 5（設定ファイル手編集は常に可能——本 ADR は手編集を禁止せず、UI を「既定の設定手段」にする。復旧経路としての手編集は残す）/ [ADR-0004](0004-security-model.md) 決定 2・3（リモート管理は認証 + HTTPS 有効化時に開放。信頼ネットワーク前提 + opt-in 強化）/ [docs/design/architecture.md](../design/architecture.md) §1.1（UI は Host アセンブリを参照できない・UI 用の抽象は `Yagura.Abstractions` に置く参照構造）/ [docs/design/configuration.md](../design/configuration.md) §6・§8（証明書の取り扱い・確定済み設定キー一覧）/ [docs/design/security.md](../design/security.md) §4.3（監査イベント ID の採番）/ Issue #241（起案元。5 ペルソナレビューとオーナー裁定 2026-07-11 を PR/Issue 上に保存）

## 文脈と課題

v0.3 で追加した opt-in セキュリティ機能のうち、**管理リスナのリモート公開 + HTTPS（ADR-0010 Phase 2）だけが管理 UI から設定できず、`yagura.json` の手編集が前提**になっている。証明書は Windows 証明書ストア（`LocalMachine\My`）から**拇印（サムプリント）**で参照する方式（`AdminCertificateProvider`）のため、利用者は `certlm.msc` 等で拇印を手で調べ、JSON に貼り付ける必要がある。

これは CLAUDE.md 確定方針 3「**Web UI ウィザード化（CLI と JSON 手編集を前提にしない）**」および「Windows 管理者が既に持つスキルだけで導入・運用できる」という思想に照らし、最も詰まりやすい箇所（拇印探し + JSON 手貼り）が UI 化から取り残された状態である。認証（`/admin/auth-setup`）・SQL Server 昇格（`/admin/promotion`）は既に「手編集を前提にしない」UI 化が済んでいるのに、管理リモート HTTPS の証明書設定だけが例外になっている。

本 ADR が対象とする設定キー（すべて現状 UI なし・`RestartRequired`。`ConfigurationKeyMetadata.cs` で定義済み）:

- `Admin:RemoteBinding:Enabled`
- `Admin:Https:Enabled`
- `Admin:Https:CertificateThumbprint`
- `Admin:Https:Port`

**この 4 キーはいずれも [configuration.md](../design/configuration.md) §8 の確定済み表に既出であり、本 ADR は新しい設定キーを 1 つも追加しない**（後述の決定 6）。純粋に「既存キーを UI から編集できるようにする」変更であることが、本 ADR を低リスクと判断する最大の論拠である。

本 ADR は「設定手段の付け替え」という表向きの主題の裏で、実体として **(1) 証明書インベントリの読み出し面（列挙 API）の新設** と **(2) 秘密鍵 ACL 付与という特権操作を起動時自動から UI 保存時の対話操作へ移すこと** の 2 つのセキュリティ実体の変更を伴う（田中の指摘）。決定・受け入れ基準はこの 2 点に権限最小化・到達制御・監査の縛りを最初から書き込む。

### スコープ（非目標）

- **syslog over TLS 受信（`Ingestion:Tls:*`）の証明書 UI は本 ADR に含めない**（オーナー裁定 論点 4）。TLS 受信は認証と無関係の受信面であり、脅威モデル・変更頻度・証明書事故時の挙動（管理 HTTPS は停止して loopback へ復旧を逃がす／TLS 受信は Yagura 側を止めず送信元がサイレントに脱落する——鈴木の指摘）がすべて管理面と異なる。関心分離のため**別 ADR・別画面で扱い、別 Issue で追跡する**。ただし証明書選択機構そのもの（証明書ストア列挙・拇印解決・秘密鍵検証）は**ライブラリ層（Abstractions 契約 + Host 実体）で共有**し、TLS 受信の証明書 UI が将来同じ部品に乗れるよう設計する。
- 証明書の取得・生成（ACME・自己署名生成支援等）は範囲外（後述の代替案 (D)）。証明書の発行・更新は利用者／組織の PKI の管轄である（configuration.md §6 の既存方針を継承）。

## 検討した選択肢

### 証明書選択のアプローチ（本 ADR の最重要論点。クリス C-1／C-2）

- **(A) 現状維持（手編集 + 拇印手貼り）**: 実装コストゼロだが CLAUDE.md 確定方針 3 と正面から矛盾する。opt-in セキュリティを「使ってほしい」のに、導入障壁が最も高い箇所を放置する。却下。
- **(B) 拇印手入力欄だけ UI に足す（ストア列挙なし）**: 実装は軽く、保存前検証（決定 2）の価値の大半は拇印欄でも得られる（`AdminCertificateProvider.Load` を保存時に叩けば「見つかった／秘密鍵あり／期限」を即表示できる）。しかし「拇印を `certlm.msc` で調べる」という核心の手間が残り、佐藤ペルソナが最も評価した「certlm.msc とメモ帳の往復を消す」効果を得られない。列挙 UI 固有のコスト（列挙 API 新設・テスト・一覧描画）を払わない代わりに、UI 化の主目的を半分しか達成しない。
- **(C) 証明書ストア列挙 + 選択（採用。オーナー裁定 論点 1）**: 「Windows 管理者スキルで完結」に最も一致し、拇印手貼りを撤廃できる。Host 層に read-only 列挙 API を足す設計コストはあるが、既存の `AdminCertificateProvider` と同層で自然に実装できる。クリス C-3 が指摘した「サブジェクトが似た証明書が並ぶと用途違いをワンクリックで誤選択する」リスクは、**列挙範囲を serverAuth EKU + 秘密鍵ありに最小化し、期限切れ・秘密鍵なし・EKU 不適合を一覧から除外／警告する**ことで抑制する（この表示・除外ルールを受け入れ基準へ格上げする）。
- 決め手: 証明書差し替えは 1〜2 年周期の低頻度だが、その低頻度の操作こそ手順を忘れやすく、拇印探し + JSON 手貼りが最大の詰まりどころになる。誤選択リスクは EKU/秘密鍵フィルタと空状態・期限表示で構造的に下げられるため、(B) の軽さより (C) の完結性を優先する。

### 証明書ストア列挙契約の層配置（リサ 懸念 1・最重要）

- (a) Host 層に列挙契約を置く（ドラフト初稿）: 列挙を呼ぶのは Web 層の画面であり、architecture.md §1.1 の「UI は Host アセンブリを参照できない／UI が Host 管轄実装を型参照するための抽象は `Yagura.Abstractions` に置く」に**違反する**。却下。
- **(b) 契約は `Yagura.Abstractions.Administration`、実体（Windows 実装）は `Yagura.Host`、結線は `Program.cs`（採用）**: 既存の `IAdminAuthenticationAdminService` と同一パターン。参照構造に整合し、単体テストを偽実装で決定的に回せる（決定 5）。

### 秘密鍵 ACL 付与のタイミング（D-3。オーナー裁定 論点 3）

- (a) 起動時自動付与のまま + UI は要否表示のみ: 田中の縛り下ではより保守的だが、UI 保存時点で**サービスアカウント視点の秘密鍵読取可否を検証できない**（`Load` の `HasPrivateKey` は現在のアカウント視点でしか見ない）。「UI では秘密鍵ありと見えたが、サービスアカウントからは読めず再起動後に 1013 縮小継続」という乖離（鈴木 懸念 4）を防げない。
- **(b) UI 保存時に付与 + 読取検証（採用。オーナー裁定 論点 3）**: 事前検証の実効性を優先する。ただし田中の 4 条件（構成対象証明書のみ・付与先は自 identity 固定・冪等・付与失敗は fail-closed 表示）+ 操作者記録を必須前提とし、**Web プロセス権限で ACL 書込・読取検証が実際に可能かは lab 検証を受け入れ条件に含める。不可なら起動時付与へ縮退する**（決定 3・委任事項）。

### ADR の位置づけ（D-5。クリス C-7）

- (a) ADR-0010 の部分 supersession: ADR-0010 決定 1／4 は「設定手段」を UI に限定しておらず（ADR-0002 決定 5 で手編集は常に可能）、UI を**追加**するだけでは決定 1／4 のどの記述も偽にならない。名指しできる偽になる記述が無い以上、部分 supersession は選ぶべきでない。却下。
- **(b) 独立 ADR（採用。オーナー裁定 先行採用 D-5）**: ADR-0010 側に変更を加えず、本 ADR は決定 1／4 が解禁したリモートバインドの設定手段を拡張する独立の決定として起こす。

### 証明書の取得・生成の扱い

- **(D) 証明書の自動生成／取得（ACME 等）**: スコープ過大。閉域前提の製品思想（configuration.md §6 の「持ち込み基本」）と乖離し、新しい外部通信経路を生む。証明書の取得・更新は利用者／組織 PKI の管轄。却下。田中の指摘どおり、列挙は BCL の `X509Store`／`X509Certificate2` で完結し**新規 NuGet 依存はゼロ**で達成できる（後任が安易にライブラリを足さないための歯止めとして本文に明記する）。

## 決定

### 1. 管理リモート HTTPS の証明書設定を管理 UI から行えるようにする

- **認証（`/admin/auth-setup`）とは分離した新規画面**を設け、`Admin:RemoteBinding:Enabled`・`Admin:Https:Enabled`・`Admin:Https:CertificateThumbprint`・`Admin:Https:Port` の編集を画面で完結させる（先行採用 D-1）。通常運用で `yagura.json` 手編集を要さない（手編集は復旧経路として引き続き可能。ADR-0002 決定 5）。
- **認証画面と新規画面は相互リンクする**（佐藤の修正提案 1）。リモートバインド有効化には認証必須という依存があり、行き来が必ず発生するため、双方に導線を置く。
- **設定操作は loopback 管理面から行う前提を維持する**（ADR-0010 決定 3「物理到達 loopback で対話設定」）。リモート公開の設定自体を、認証なしで到達できる現状の loopback 管理リスナ上で対話的に行う。

### 2. 証明書ストア列挙 + 選択 UI（拇印手貼りの撤廃）

- `LocalMachine\My` の証明書を一覧表示し、選択で拇印を自動設定する。**拇印の手入力を不要にする**（オーナー裁定 論点 1 = (C) 採用、先行採用 D-2）。
- **列挙範囲は serverAuth EKU（`1.3.6.1.5.5.7.3.1`）+ 秘密鍵ありに最小化する**（田中 修正提案 1・Q1、オーナー裁定 論点 1）。サーバ認証と無関係な証明書（コード署名・S/MIME・他アプリのクライアント証明書等）を無差別に列挙しない——機内 PKI 構成・他システムの存在が観測される情報開示（田中 懸念 1）と、用途違いのワンクリック誤選択（クリス C-3）を同時に抑える。
- **返却フィールドは表示に必要な最小限**（サブジェクト CN・発行者・有効期限・拇印・秘密鍵の有無）とし、**秘密鍵・エクスポート可能な秘密情報は返さない**。列挙 API は Host 実体を read-only（`X509Store` を `OpenFlags.ReadOnly` で開く。既存 `AdminCertificateProvider.Load` と同一方針）で実装する。
- **期限切れ・秘密鍵なし・EKU 不適合の証明書は一覧から除外または警告表示する**（クリス 修正提案 3、オーナー裁定 論点 1）。この表示・除外ルールは列挙 UI の本体であり、受け入れ基準に格上げする。
- **一覧が空のときの案内を出す**（佐藤 懸念 1・修正提案 3）: 「`LocalMachine\My` に条件を満たす証明書がない旨」と「どのストアに秘密鍵付きで取り込めば一覧に出るか」（`certlm.msc` のローカルコンピューター証明書ストアへの言及を含む）を 1〜2 文で示す。証明書の作り方はスコープ外だが、道しるべは範囲内とする。

### 3. 秘密鍵 ACL 付与を UI 保存時に行う（縛り付き・lab 検証ゲート）

- **UI で証明書を選択・保存した時点で、サービスアカウント（自 identity）に対する秘密鍵読取権限の付与と読取検証を行う**（オーナー裁定 論点 3 = 鈴木案採用）。これにより「サービスアカウントが実際に秘密鍵を読めるか」を保存前検証の一部にでき、再起動後の乖離（鈴木 懸念 4）を防ぐ。
- **田中の 4 条件を必須前提とする**（田中 懸念 2〜4・修正提案 3）:
  - **付与対象は「まさにリスナへ構成する証明書」に限る**（利用者が選んだ任意の証明書の秘密鍵へ権限を広げない）。
  - **付与先 `accountName` は自プロセスの identity（`NT SERVICE\Yagura`）にハードコードし、UI／リモートからの入力を一切受けない**。
  - **冪等にする**（付与前に既存 ACE を確認し、あればスキップ。UI 保存のたびに同一 Read ACE が累積しない）。
  - **付与失敗は fail-closed で表示する**（`AdminCertificatePrivateKeyAccessGranter` が返す失敗理由——ACL 付与失敗・CNG 非ファイル鍵〔HSM/TPM/スマートカード〕で自動付与不可等——を保存前検証に表出する）。
- **read-only 列挙（決定 2）と秘密鍵 ACL 付与（本決定）は別契約に分離する**（リサ 懸念 3・修正提案 3）。列挙は副作用なし（非 `IYaguraWriteService`）、付与は監査対象の書き込み操作として別契約にする。影響範囲の「read-only 列挙」という自己記述と整合させる。
- **lab 検証ゲート**（田中 Q3、オーナー裁定 論点 3）: `SetAccessControl` は対象キーファイルへの `WRITE_DAC` を要する。起動時付与が成立している権限と、**実行時（Web 面）にサービスアカウント自身が ACL を書き換え・読取検証できるか**は別問題である。**Web プロセス権限で ACL 書込・読取検証が可能かを lab 検証することを受け入れ条件に含め、不可と判明した場合は起動時付与へ縮退する**（UI は要否表示のみに留める）。

### 4. 保存前 fail-closed 検証と即時フィードバック（列挙 UI から独立した決定）

- **保存前に fail-closed 検証を行い、理由を即時表示する**。現状は手編集 → 再起動して初めて起動時 fail-closed（イベント ID 1012／1013）で気づくが、それを保存時へ前倒しする。検知対象:
  - 「リモートバインド有効なのに認証・HTTPS が未構成」（ADR-0010 決定 1 の起動時 fail-closed = イベント ID 1012 と対称）。
  - 「証明書が見つからない／秘密鍵にアクセスできない／期限切れ／serverAuth EKU 不適合／秘密鍵 ACL 付与失敗」（決定 3・イベント ID 1013 経路と対称）。
- **この fail-closed 事前検証は列挙 UI とは独立した価値であり、独立した決定として書く**（クリス 修正提案 4）。仮に将来 (B) 拇印欄へ縮退しても、この検証機能は残せる構造にする。既存 `AdminAuthenticationAdminService.ConfigureAsync` の「UI 層で親切に拒否 → 起動時検証が最終防衛線」の二段構え（`WizardValidationException`）と同じ型に流す（リサ 良い点）。
- **fail-closed 不変条件は永続化された現在値に対して検証する**（リサ 懸念 4・修正提案 4）。認証／RemoteBinding／HTTPS の組み合わせ検証は、画面上の値ではなく設定ファイルの現在値に対して行い、**関与する全画面（認証画面・本 ADR の新規画面）が同一の不変条件を表示・拒否する**。画面分割によって「片方の画面から認証を切ると、もう片方で有効な RemoteBinding と組んで fail-closed をすり抜ける」実装ミスを防ぐ。
- **UI 事前検証と起動時検証の一致（乖離ゼロ。D-6。鈴木 懸念 3・修正提案 3）**: 事前検証は `AdminCertificateProvider`（起動時 1012／1013／1016 経路と同一コード）へ寄せ、**「UI 事前検証が緑なら、再起動後に想定外の縮小継続（1012／1013／1016）が起きない」ことを回帰テストで固定する**。`AdminCertificateProvider.Load` は `validOnly:false` で期限切れでも `Succeeded=true` を返し `IsExpired` で区別する設計のため、UI が期限切れ証明書の保存を「通す（警告付き）／止める」いずれにするかを起動時挙動と一致させることを受け入れ条件とする。

### 5. テスト戦略（プラットフォーム依存コードの決定的検証）

- 証明書ストア列挙・拇印解決は実マシン依存でユニットテストしづらい（`AdminCertificateProvider` は `static` + `[SupportedOSPlatform("windows")]`、`X509Store(LocalMachine\My)`）。**契約は `Yagura.Abstractions.Administration` のインターフェースに置き、単体テストは偽実装で決定的に検証し、実ストア接触は統合／E2E テストに限定する**（リサ 懸念 2・修正提案 2。前例: `IAdminHttpsCertificateStatusProbe` の偽実装差し替え、`IngestionTlsDegradedStartupE2ETests`）。#235（WindowsIdentity）と同種のテスト不能コードを制度的に防ぐ。
- CI で実ストア経路を回す方法（テスト用一時証明書を CI 内で生成しストア投入する統合テスト）は委任事項とする（リサ Q4）。

### 6. 反映方式の明示と受信断の最小化（新設キーはゼロ）

- **本 ADR は設定キーを 1 つも追加しない**（リサ Q1・修正提案 5）。対象 4 キーは configuration.md §8 の確定済み表に既出であり、§8 への追記も不要。純粋に既存キーの UI 化のみである旨を本文に明記する（低リスクの主論拠）。
- 対象 4 キーはすべて `RestartRequired`。UI に「保存だけでは反映されない。反映は次回サービス再起動時であり、その間 syslog 受信と管理画面が停止する（= 受信断）」を明示する（佐藤 懸念 2・修正提案 2、鈴木 懸念 1）。文言は「何が・どれくらい・どう戻るか」を含む具体的なものとし、ADR には文面イメージを 1 つ例示する（実装時のブレ防止）。
- **受信断の最小化を UX 要件とする**（鈴木 懸念 1・修正提案 1）: (a) 複数の証明書系キー変更を溜めて **1 回の再起動でまとめて反映するバッチ適用**、(b) 再起動導線に推定断時間・想定ロス量（直近受信レートから概算）の表示、(c)「1 変更 = 1 再起動」を誘発しない UX。再起動導線を設ける場合は確認ダイアログを経由する（ワンクリックにしない）。

### 7. 監査

- **リモートバインド有効化・証明書拇印変更は、認証設定変更（監査 2006）に畳み込まず、additive で独立イベント ID を採番する**（田中 懸念 5・修正提案 4）。「機の公開」という最重要のセキュリティ状態遷移が他のフラグ変更に埋もれることを防ぐ（ADR-0010 決定 6・起動失敗を 1000 番台へ分離した思想と同じ論理）。採番は security.md §4.3 の SEC-5 に送る。
- **秘密鍵付与の監査（現状 2009／2010: 拇印 + 付与先アカウントのみ・起動時自動なので操作者なし）に「誰が付与したか」（ADR-0010 決定 6 の命名空間付き利用者名）を additive で追加する**（田中 懸念 5）。決定 3 で付与を対話操作化した以上、操作者特定を伴わなければ対話化の意味の半分が失われる。

### 8. 既定無認証 loopback で列挙 API が到達可能な残存リスクの明記（D-7）

- 列挙 API・設定 API の到達制御は、オーナー裁定 論点 2（今進める）の結論に従う。**既定（loopback 認証 opt-in 無効）では、ADR-0010 決定 1 が脅威モデルとして明記した「管理リスナへネットワーク到達できる任意のローカル実行主体（低権限サービス・侵害プロセス含む）」が、無認証でこの列挙 API を叩ける**（田中 懸念 1・Q2）。すなわち本 ADR は「無認証 loopback 到達者が機内の serverAuth 証明書インベントリ（サブジェクト CN・発行者・有効期限・拇印）を取得できる」新しい読み出し面を増設する。
- この残存リスクを**本文に明記する**（オーナー裁定 先行採用 D-7）。緩和は決定 2 の列挙範囲最小化（serverAuth EKU + 秘密鍵あり・最小フィールド）で部分的に達成される（無関係証明書のサブジェクトは返らない）。loopback 認証 opt-in（ADR-0010 決定 1）を有効化した環境では、この列挙 API も認証で守られる。

## 受け入れ基準

以下をすべて満たすことを実装 PR（通常コードレビュー）の受け入れ条件とする。

- **列挙フィルタ**: 列挙は serverAuth EKU（`1.3.6.1.5.5.7.3.1`）+ 秘密鍵ありに限定する。無関係証明書を無差別に返さない。返すフィールドは表示に必要な最小限（サブジェクト CN・発行者・有効期限・拇印・秘密鍵の有無）で、秘密鍵・エクスポート可能な秘密情報を返さない。
- **一覧の除外／警告**: 期限切れ・秘密鍵なし・EKU 不適合の証明書は一覧から除外または明確な警告表示とし、無警告で選べる状態を作らない。
- **空状態の案内**: 条件を満たす証明書が 0 件のとき、取り込み先（`LocalMachine\My` に秘密鍵付き）と `certlm.msc` への言及を含む案内文を表示する。
- **拇印手貼りの撤廃**: 通常運用で拇印の手入力・JSON 手編集なしに証明書を選択・保存できる。
- **fail-closed 検証（保存前）**: 「リモートバインド有効なのに認証・HTTPS 未構成」「証明書が見つからない／秘密鍵アクセス不可／期限切れ／EKU 不適合／秘密鍵 ACL 付与失敗」を保存前に検知し理由を表示する。
- **fail-closed は永続値で検証**: 認証／RemoteBinding／HTTPS の組み合わせ検証は永続化された現在値に対して行い、関与する全画面が同一の不変条件を共有する。回帰テストで「片方の画面から不変条件をすり抜けられない」ことを固定する。
- **乖離ゼロ（D-6）**: UI 事前検証が緑なら、再起動後に想定外の縮小継続（イベント ID 1012／1013／1016）が起きない。事前検証と起動時検証を同一コード（`AdminCertificateProvider`）へ寄せ、回帰テストで固定する。
- **秘密鍵 ACL 付与の縛り（決定 3）**: 付与対象は構成対象証明書のみ・付与先は自 identity 固定（UI／リモート入力不可）・冪等・付与失敗は fail-closed 表示・付与操作に操作者記録。
- **lab 検証ゲート（決定 3）**: Web プロセス権限で秘密鍵の ACL 書込・読取検証が可能かを lab で検証する。不可なら起動時付与へ縮退する（本受け入れ基準の付与 UI 部分は縮退版に読み替える）。
- **監査の独立 ID（決定 7）**: リモートバインド有効化・証明書拇印変更を 2006 と別 ID（additive・SEC-5）で採番し、2009／2010 に操作者を additive 追加する。
- **テスト戦略（決定 5）**: 列挙・解決契約は Abstractions に置き、単体は偽実装で決定的に検証、実ストア接触は統合／E2E に限定する。
- **受信断最小化 UX（決定 6）**: 複数キー変更のバッチ適用、再起動導線での推定断時間・想定ロス量表示、「1 変更 = 1 再起動」を誘発しない反映文言。
- **残存リスクの明記（決定 8）**: 既定無認証 loopback で列挙 API が到達可能である点を利用者向けドキュメント／本文に残す。
- **新設キーゼロ（決定 6）**: configuration.md §8 に新しいキーを追加しない。

## 影響範囲

### 層配置（architecture.md §1.1 準拠。`IAdminAuthenticationAdminService` と同一パターン）

- **契約**: 証明書ストア列挙契約（`IAdminCertificateStoreReader` 等・read-only・非 `IYaguraWriteService`）と、証明書系設定の保存・検証サービス契約を **`Yagura.Abstractions.Administration`** に置く。秘密鍵 ACL 付与は監査対象の書き込み操作として**別契約**に分離する（決定 3）。
- **実体**: Windows 実装を **`Yagura.Host`（`AdminCertificateProvider` と同層）** が持つ。列挙は BCL の `X509Store`／`X509Certificate2` で完結し**新規 NuGet 依存を追加しない**。
- **結線**: **`Program.cs`** が DI 結線する。
- **Web 層**: 認証（`/admin/auth-setup`）とは分離した新規画面（Blazor Interactive Server。ADR-0003）を追加し、契約経由で列挙・保存・検証を呼ぶ。既存 `AdminCertificateProvider.Load`（拇印 → 解決 + 秘密鍵／期限検証）を UI の保存前検証に再利用する。

### 対象設定キー

`Admin:RemoteBinding:Enabled`／`Admin:Https:Enabled`／`Admin:Https:CertificateThumbprint`／`Admin:Https:Port`（すべて既出・`RestartRequired`）。**新設キーはゼロ**。

### テスト戦略

契約は Abstractions・単体は偽実装で決定的に検証・実ストア接触は統合／E2E に限定（前例: `IAdminHttpsCertificateStatusProbe` の偽実装差し替え、`IngestionTlsDegradedStartupE2ETests`）。CI での実ストア経路の回し方（テスト用一時証明書の生成・ストア投入）は委任事項。

### 監査

リモートバインド有効化・証明書拇印変更は 2006 と別 ID（additive・SEC-5）。秘密鍵付与（2009／2010）に操作者を additive 追加。いずれも security.md §4.3 の採番方針に従い実装 PR で確定。

## 委任・未解決

- **委任事項（実装 PR + security.md／configuration.md へ送る）**: 新規イベント ID の確定（リモートバインド有効化・証明書拇印変更）／2009・2010 の操作者フィールド追加／再起動導線の具体（バッチ適用・推定断時間表示）／空状態・反映文言の確定文面／CI で実ストア経路を回す統合テストの整備。ADR-0010 が踏んだ手順に倣い、本 PR（ADR のみ）は `docs/design/` の同時更新を含めない——このズレが生じることをここで明記し、実装 PR での追従を約束する（PR #167 リサの教訓）。
- **CF-5 依存の引き継ぎ（オーナー裁定 論点 2）**: 対象キーは `RestartRequired` であり、CF-5（UI 外の受信断なし設定再読み込み。configuration.md §9・ADR-0010 委任事項 15）が未確定な限り、反映 = サービス再起動 = 受信断である。**UI 化はまさに設定変更の頻度を上げる提案であり、「UI 化で設定変更頻度が上がる → CF-5 の優先度を上げる」という因果を本 ADR の委任事項として明示的に引き継ぎ、ADR-0010 委任事項 15 とリンクする**（鈴木 修正提案 5、クリス C-4）。ただし証明書選択自体はロックアウトリスクを大きく上げないため、本 ADR を CF-5 確定にはゲートせず今進める（オーナー裁定 論点 2）。
- **D-3 の lab 検証による縮退可能性（決定 3）**: Web プロセス権限での ACL 書込・読取検証が lab で不可と判明した場合、秘密鍵付与は UI 保存時から起動時付与へ縮退し、UI は要否表示のみに留める。
- **TLS 受信（`Ingestion:Tls:*`）の証明書 UI は別 ADR・別 Issue（オーナー裁定 論点 4）**: 本 ADR には含めない。証明書選択機構はライブラリ層（Abstractions 契約 + Host 実体）で共有し、TLS 受信面の証明書健全性の可視化（期限接近・稼働中使用不能の能動通知、送信元別ハンドシェイク失敗メトリクス `RecordTlsHandshakeFailure` への導線——鈴木 懸念 2）は別 ADR で扱う。別 Issue を起票して追跡する。

## 先送りにする場合の再評価トリガ

- **lab 検証で Web プロセス権限による秘密鍵 ACL 付与・読取検証が不可**と判明した時点で、決定 3 を起動時付与へ縮退する（決定自体の変更は伴わないが、UI 挙動を要否表示のみへ改める）。
- **CF-5 が確定**した時点で、対象キーの反映方式（受信断を伴う再起動）を再読み込みへ切り替える（本 ADR の受信断最小化 UX の前提が変わる）。
- **TLS 受信の証明書 UI を扱う別 ADR が起こされる**時点で、本 ADR が確立した証明書選択機構（Abstractions 契約 + Host 実体）の共有可否を検証する。
