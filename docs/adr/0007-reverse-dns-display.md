# ADR-0007: 閲覧画面の送信元逆引き（PTR）ホスト名表示

- 状態: proposed
- 日付: 2026-07-06
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md)（目的——Windows 管理者の既存スキルで運用できること）、[ADR-0002](0002-architecture-principles.md)（決定 2 受信パイプラインの品質原則・決定 4 外部送信の既定無効）、[ADR-0003](0003-ui-policy.md)（決定 5 文言原則）、[ADR-0004](0004-security-model.md)（決定 1 信頼ネットワーク前提）、[ui.md](../design/ui.md) §4・§7、[configuration.md](../design/configuration.md) §8、[security.md](../design/security.md) §1（閲覧リスナの外向き通信の台帳）

## 文脈と課題

閲覧画面（ログ検索の結果一覧・詳細表示・ダッシュボードの送信元別受信状況）の送信元は IP アドレスで表示している。運用者は「この IP はどの機器か」を機器台帳や `nslookup` で突合しており、この往復が機器同定のボトルネックになる。本製品のターゲット環境（Windows-shop。ADR-0001）では AD 統合 DNS が存在することが多く、**逆引き参照ゾーンが整備されていれば**（AD DS + DNS の導入で前方参照ゾーンは自動作成されるが、**逆引き参照ゾーンは既定では作成されず管理者が手動で作る**——整備状況は環境に依る。整備の案内は configuration.md §10 CF-D6）、送信元 IP の逆引き（PTR）ホスト名の併記により、既存の DNS 資産だけで機器同定が大幅に楽になる。

一方で、この機能には次の緊張があり、方針の確定が必要である:

1. **既存の「ホスト名」との混同**: 詳細表示が現在出している「ホスト名」は **syslog HOSTNAME フィールド（送信者の自己申告）**である。逆引きホスト名は**送信元 IP から DNS で引いた別由来の名前**であり、両者は食い違い得るし、どちらか一方だけが存在することもある。混同させない表示設計が要る
2. **性能**: 描画ごとに DNS を問い合わせることは許されない。特に PTR 未登録の IP の解決失敗は、実測環境では**約 4.5 秒**を要した（環境依存。下記検証記録——DNS 応答自体ではなく OS 名前解決のフォールバック連鎖が主因と推定され、権威 DNS が即答する環境でも同程度の遅延が生じ得る）。検索結果は最大 10,000 件・送信元一覧は最大 1,000 件であり、素朴な実装は画面を破壊する
3. **信頼性**: PTR 未整備の LAN は珍しくない。名前が引けないことは異常ではなく正常系であり、IP のみ表示へのフォールバックが既定の姿である
4. **セキュリティ**: 逆引き名は**送信元（を管理する者）が自分の PTR レコードで任意に操作できる untrusted 値**である（また平文 UDP の DNS 応答は経路上の改ざんもあり得る——untrusted として扱う理由は 1 つではない）。表示の無害化と、認証・同定の根拠に使わない線引きが要る
5. **挙動の透明性と情報流**: 本機能はサーバが自発的に DNS クエリを発する。ADR-0002 決定 4（あらゆるデータの外部送信は既定で無効）との関係の整理に加え、**逆引きクエリは「どの IP からログを受けているか」を DNS 経路上へ載せる情報流**であり、クエリ対象・問い合わせ経路の輪郭を設計が正面から記述する必要がある

### 検証記録（conventions.md「技術的主張の検証」——API の挙動を推測で設計根拠にしない）

実施 2026-07-06。開発機 Windows 11（10.0.26200）・.NET SDK 10.0.301 で `System.Net.Dns` を実機検証した:

- **公開オーバーロード**: `GetHostEntryAsync` は 4 種——`(string)` / `(string, CancellationToken)` / `(string, AddressFamily, CancellationToken)` / `(IPAddress)`。**`IPAddress` を取る版に CancellationToken オーバーロードはない**（IP 文字列を渡せば逆引きになるため string 版で代用できる）
- **PTR 登録あり**（8.8.8.8）: 82 ms で `dns.google` を取得。返却された `AddressList` に前方解決の結果（8.8.8.8, 8.8.4.4）が含まれており、**本 API は逆引きに加えて前方解決も行う**ことを確認した
- **PTR 未登録**（TEST-NET-1 の 192.0.2.1・プライベート帯の 10.255.255.250）: いずれも `SocketException`（ホスト不明）まで**約 4.5 秒**
- **CancellationToken は進行中の解決を中断しない**: PTR 未登録 IP に 300 ms のトークンを渡しても解決は約 4.6 秒完走し、`OperationCanceledException` ではなく元の `SocketException` が返った（トークンは発火済みであることを確認）

**検証範囲の明示と約 4.5 秒の解釈**（ペルソナレビュー（PR #98 クリス）の指摘を受けた追加実測。同日）: 本実測は開発機 1 台・その DNS 構成における値である。追加実測で切り分けた——実測時の OS の DNS 参照先は公開リゾルバ（1.1.1.1）であり、**DNS プロトコル単体の照会（`Resolve-DnsName -Type PTR`・DnsOnly）は 192.0.2.1 に対して NXDOMAIN を 97 ms で即答した**。それでも `GetHostEntryAsync` は約 4.5 秒を要したことから、**遅延の主因は DNS 応答待ちではなく、DNS の否定応答後に OS の名前解決（GetAddrInfoEx 系）が続けるフォールバック連鎖（LLMNR・NetBIOS 等）と推定される**。したがってこの遅延は「権威を持つ社内 DNS が NXDOMAIN を即答する環境」でも同程度に生じ得る（フォールバックを無効化していない限り）。一方、DNS サーバが完全無応答の環境は未検証であり、より長くなり得る。**設計はこの数値に依存しない**——最悪値が変わっても「同期経路に置かない・同時解決数を絞る・失敗（未登録）を必ずキャッシュする」という前提条件は不変である。

この実測から、タイムアウトは「呼び出し側が待つのをやめる」制御としてしか設計できず、裏で OS の解決は完走する。したがって上記 3 点が、設計の選択肢ではなく前提条件になる。

## 検討した選択肢

### 解決のタイミング

- **(a) 表示時解決 + IP 単位キャッシュ（採用）**: 閲覧層が表示のために解決し、結果をメモリキャッシュする。受信経路・DB スキーマ・provider 契約に変更がない。名前は保存されないため逆引き名での検索はできない
- (b) 受信時解決・DB 保存: 逆引き名での検索が可能になる利点はある。却下理由: ①受信パイプラインに DNS という外部依存の遅延・障害モードを持ち込む（「ログを失わない」の主経路を表示都合で複雑化させない。ADR-0002 決定 2）、②保存時点の名前が固定化し、後から引き直した名前と食い違う第 3 の名前を作る、③スキーマ変更と provider 実装への波及、④サーバが能動的に取得して付与する untrusted 値を証跡 DB へ新たに混入させる（syslog HOSTNAME という自己申告値は既に保存しているが、それは**受信したログ原文の一部**であり、サーバ自身が外部照会で取得・付与する値を証跡に混ぜることとは区別する）。要望が反復した場合の再評価対象として残す（再評価トリガ参照）
- (c) 両方（受信時保存 + 表示時再解決）: (b) の費用に加えて「保存された名前と今の名前」の 2 重表示問題を新設する。却下

### 解決対象の IP 範囲

- **(a) プライベート/サイトローカル系の帯域のみ解決（採用）**: 逆引きの解決対象を、ループバック・プライベート帯（RFC 1918）・共有アドレス空間（RFC 6598 の 100.64.0.0/10——CGN・オーバーレイ VPN で内部利用され、プライベート帯と同じく外部の攻撃者が権威を握れない帯域のため対象に含める。再レビュー（PR #98）の指摘で追加）・リンクローカル（169.254/16・fe80::/10）・ULA（fc00::/7）に限定し、**対象帯域外（グローバル等）の IP アドレスは解決しない**（クエリ自体を発しない）。IPv4-mapped IPv6（`::ffff:10.0.0.1` 等）は IPv4 へ正規化してから判定する（帯域判定の単体テストの境界ケースに含める）。根拠: ①本製品の中核ユースケース（信頼ネットワーク内の機器同定。ADR-0004 決定 1）はこの帯域でほぼ完結する、②グローバル帯の逆引きクエリは OS 構成の DNS のフォワーダ/ルートヒント経由で**サイト外の権威サーバまで再帰的に到達し**、「どの IP からログを受けているか」が DNS 経路上の第三者（詐称送信元の場合、その IP の権威サーバ運用者 = 攻撃者自身であり得る）に観測される情報流を作る、③検索結果には詐称された任意の送信元 IP が含まれ得るため、この情報流は「攻撃者が自分の権威サーバへクエリを誘導できる」経路にもなる——対象帯域の限定はこの経路を構造的に閉じる
- (b) 全 IP を解決する: グローバル IP の送信元（インターネット越しの syslog・グローバルアドレスを内部で使う一部の大規模環境）にも名前が出る利点はあるが、上記②③の情報流を常設する。却下——グローバル IP の逆引き表示の要望が反復した場合に、opt-in 設定キーの追加として再評価する（additive-only のためキーは後から足せる。再評価トリガ参照）

### 既定値（オン / オフ）

- **(a) 既定オン・opt-out（採用）**: 根拠は**利用者に負わせる下振れの小ささ**である——①機能の失敗モードは「IP のみ表示に戻る」に閉じ、ログの受信・保存の完全性と交差しない（帰結参照）、②問い合わせは OS の名前解決機構を経由する——実宛先は構成済み DNS サーバに加え、OS 既定のフォールバック（LLMNR・NetBIOS 等。検証記録参照）を含むが、**いずれも OS の既存構成であり本製品が新しい宛先を設定させることはない**、③解決対象はプライベート/サイトローカル系の帯域に限る（前項——サイト外の権威サーバへ再帰到達する情報流を閉じる）、④オフ設定を初日から提供する。機器同定の支援は本製品の中核価値であり、上振れは大きい。**逆引き参照ゾーンの整備率は未検証**（整備なし環境では価値が出ない——帰結）であり、既定の向きは再評価トリガで監視する。なお「新たな信頼境界を跨がない」という整理は **OS の DNS 参照先がサイト管理下にあることを前提とする**——公開リゾルバ（1.1.1.1 等）を直接参照する構成では、プライベート帯の PTR クエリがそのリゾルバへ送信される（実測に使った開発機自身がこの構成だった——検証記録）。この構成での情報の重みは小さい（プライベート IP アドレスの逆引き名の照会であり、公開リゾルバはプライベート帯の権威を持たないため通常は否定応答する）が、事実として利用者向けドキュメントに明記する（CF-D6）
- (b) 既定オフ・opt-in: ADR-0002 決定 4 の字義に最も忠実だが、同決定が禁じるのは「**観測データの製品外への自動送信（テレメトリ）**」であり、利用者インフラ（DNS）の通常利用とは質が異なる——この読み替えが素直に成立するよう、解決対象の帯域限定（サイト外へ再帰到達しない）を採用条件に含めた。却下。ただしこの整理を暗黙にしない: **DNS クエリが発生する事実と停止手段は利用者向けドキュメントに明記し、オフ設定を初日から提供する**（下記決定 4）

### 表示の位置づけ

- **(a) 併記・補完（採用）**: IP アドレスを主、逆引き名を補助として併記する。syslog HOSTNAME（自己申告）は従来どおり独立の項目として残し、置き換えない
- (b) 置換（名前が引けたら IP の代わりに表示）: PTR の操作・陳腐化で誤った名前だけが見える状態を作る。コピー・機器台帳との突合・検索条件（IP）との対応も揺れる。却下

### 逆引き名の検証（FCrDNS）

- **前方確認（FCrDNS——逆引き名を前方解決して元の IP と一致するか確認する手法）は行わない**: 本表示は参考情報であり認証ではない。検証を実装すれば「検証済み」という誤ったお墨付き（PTR と前方の両方を管理する攻撃者は素通しになる）を与える方が害が大きい。untrusted であることを表示設計（無害化 + IP 主・名前補助 + 由来明示）で扱う

### IDN（国際化ドメイン名）の表示

- **復号せず、LDH 検査で扱う（採用）**: Windows DNS サーバの名前検査の既定（Multibyte (UTF8)）では、非 ASCII のコンピュータ名が PTR に UTF-8 生値で載り得る——この値は決定 5 の LDH 検査により**表示されない**（fail-safe）。Punycode（`xn--` 形式）は LDH に適合するため**原文のまま表示し、Unicode への復号は行わない**——復号表示は LDH 検査を通過した名前を similar-looking 文字（ホモグラフ）へ化けさせる偽装経路を再導入するためである。日本語ホスト名が判読不能（`xn--...`）または非表示になる対価は認識のうえで受け入れる（v0.1 の提供言語が日本語のみであることと緊張するが、安全側を優先する。要望が反復した場合の再評価トリガ参照）

## 決定

1. **併記（補完）**: 逆引きホスト名は送信元 IP の**補助表示**とする。syslog HOSTNAME の表示を置き換えず、両者の由来の違い（DNS に登録された名前 / ログに記載された名前）を画面で明示する。表示規約の詳細は [ui.md](../design/ui.md) §4 で定める
2. **表示時解決 + キャッシュ充填型**: 閲覧層（Yagura.Web）のシングルトンサービスが IP → 名前を解決・キャッシュする（成功・未登録の両方に TTL を持つ）。**解決対象はプライベート/サイトローカル系の帯域の IP に限る**（選択肢「解決対象の IP 範囲」——グローバル IP はクエリを発しない）。**解決要求の発行対象は画面に現に描画されている行（ページング後の 1 ページ分・仮想化ウィンドウ内）の IP に限り**、検索結果・一覧の上限件数に対する一括解決要求は行わない。**描画は解決を待たない**——IP は常に即時表示し、キャッシュ命中時のみ名前を同時表示、未解決分は非同期解決の完了時に表示へ反映する。**表示への反映は解決完了 1 件ごとではなく束ねて行う**（Blazor Server の circuit 越し再描画を連発させない。束ねの周期・方式は ui.md §12 UI-10 で数値とともに確定する）。受信パイプライン・DB スキーマ・provider 契約は変更しない。**逆引き名は検索条件に使えない**（v0.1。検索は従来どおり IP）。なお、解決サービスの契約の配置（`Yagura.Abstractions` へ置くか `Yagura.Web` 内で完結させるか）・具体のインターフェース名・キャッシュのデータ構造は**本 ADR の確定事項ではない**——実装 PR で決定し、結果を設計書（ui.md §4 の実装参照）に記録する
3. **資源保護を仕様に含める**: 同時解決数の上限・キャッシュ件数の上限（超過時は新規解決を行わず IP のみ表示——検索結果は送信元詐称を含み得るため、キャッシュを外部入力で無制限に成長させない。**見送りの発生自体はカウンタで観測する**——決定 6）・解決待ちの打ち切り（上記検証記録の制約に基づき「待つのをやめる + 完走結果は事後にキャッシュへ反映」方式）。**解決待ちタスクは受信パイプラインと同一プロセス内で動作するため**（architecture.md §1.1 の単一プロセス構成）、DNS 無応答時に上限到達状態が継続した場合の閲覧層の応答性への間接影響も本決定の保護対象であり、UI-10 の値の確定には「DNS 停止時の閲覧層専有」シナリオの検証を含める。**同時解決数の上限到達時の扱い（後続要求を待たせるか見送るか・見送る場合の計上先）は実装 PR で決定し、UI-10 の行に記録する**。仮値と確定手続きは ui.md §12（UI-10）で追跡する
4. **設定**: `Viewer:ReverseDns:Enabled`（既定 **オン**）。**オフ時は DNS クエリを一切発せず、キャッシュ済みの名前の表示も行わない**（「オフ = 逆引き名は出ない」を単純に保つ）。不正値は**縮小側（無効）で継続**する——本機能は外向きのクエリを発生させるため、不正値で「発生する側」へ倒さない（configuration.md §1 の縮小側原則の適用。詳細は同 §8）。**「オフ時にクエリを発しない」「対象帯域外の IP にクエリを発しない」は自動テストで固定する**（解決 API の呼び出しを解決サービス 1 点に集約し、オフ時・対象帯域外で下位の解決 API へ到達しないことを単体テストが検証する——実装 PR の受け入れ条件）
5. **無害化表示**: 逆引き名は untrusted 値として扱う。①プレーンテキストとして描画する（マークアップ・リンクとして解釈させない）、②ホスト名として妥当な文字集合（英数字・ハイフン・ドット。RFC 1123 の LDH 形式）以外を含む名前、および DNS 名の上限（253 文字）を超える名前は**表示しない**（切り詰め・整形をして見せない——解釈を偽装しない流儀）。**`xn--` で始まるラベル（Punycode）は Unicode へ復号せず原文のまま表示する**（IDN の見た目偽装を復号表示で助けない——選択肢「IDN の表示」）、③逆引き名を検索条件・リンク URL・認可判定に自動使用せず、**監査記録・アプリケーションログ（診断ログ）・Windows イベントログのいずれにも出力しない**（観測は決定 6 のカウンタのみで行い、untrusted 値を証跡系へ持ち込まない。**解決経路の例外・結果オブジェクトを logger へ渡さないこと**を実装 PR のレビュー観点に含める——catch して log する反射的実装が本規約を破る抜け道になるため）
6. **観測**: 解決の**成功 / 未登録（PTR なし）/ 失敗（打ち切り・DNS 障害）/ 見送り（キャッシュ上限）**を発生箇所別カウンタで観測する。計器名は architecture.md §4.1.1 の規則（意味論の異なる事象ごとに独立した計器名）に従い、`yagura.web.reverse_dns.resolved` / `yagura.web.reverse_dns.not_found` / `yagura.web.reverse_dns.failed` / `yagura.web.reverse_dns.skipped` の 4 種を予定する（確定した計器名は実装 PR で同節の一覧に追記する）。DNS インフラの異常が「名前が出ない」という見た目の変化だけで黙って進行しない。なお失敗カウンタは architecture.md §4.6 の**能動通知（イベントログ警告）の対象には加えない**——同節の対象一覧は受信・保存・閲覧提供の継続性に関わる事象（スプール系・縮退・自己検証失敗・証明書期限接近・circuit 上限到達）であり、本機能の失敗はそのいずれにも当たらない参考表示の劣化であるため（DNS インフラ自体の監視はサイトの既存監視の管轄）。この線引きの判断は ui.md §9 にも記録する

## 帰結

- **良くなること**: 機器同定が既存の DNS 資産だけで楽になる（追加の台帳・設定を要しない）。受信経路・スキーマは無変更で、機能の失敗モードは「IP のみ表示に戻る」に閉じる——「ログを失わない」原則と交差しない
- **悪くなること**: 逆引き名で検索はできない（名前が見えるのに検索できないことは誤解を生み得る——検索欄のラベル・ヘルプで手当てする）。キャッシュ TTL の間は付け替え後の古い名前が表示され得る（BCL は DNS レコードの TTL を取得できないため、保持時間は DNS 側の意図と無関係に製品が決める——上限は UI-10 で確定し文書化する）。PTR 未整備環境では価値がなく、「なぜ名前が出ないのか」の問いを生む（operations.md への申し送りで案内する。configuration.md §10 CF-D6）。非 ASCII の日本語ホスト名は表示されない・Punycode は判読不能な原文表示になる（選択肢「IDN の表示」の対価）
- **受け入れるリスク**: ①PTR は送信元側で操作可能であり、誤同定を誘う名前（LDH 内の similar-looking 文字を含む）を表示させられる余地は残る。緩和は決定 5（無害化）・決定 1（IP が常に隣にある併記原則）・由来の明示であり、**表示は参考情報であって同定の証明ではない**ことを利用者向けドキュメントにも明記する。②OS の DNS 参照先が公開リゾルバである構成では、プライベート帯の PTR クエリがサイト外のリゾルバへ送信される（帯域限定後の残余の情報流。CF-D6 で案内し、気にする環境ではオフ設定で停止できる）
- **構造上の位置づけ**: 本機能は**閲覧層が自発的な外向きネットワーク通信を持つ初の機能**である。security.md §1 の「閲覧リスナの外向き通信」の台帳に登録し、以後の外向き通信の追加が「前例がある」で素通りしない構造（台帳更新 = 明示の PR 差分）にする

## 先送りにする場合の再評価トリガ

- **逆引き名での検索・フィルタ**の要望が反復した場合（目安: 独立した利用者から 3 件以上）、却下案 (b)（受信時解決・保存）を個別 ADR で再評価する
- **対象帯域外（グローバル等）の IP の逆引き表示**の要望が反復した場合（同上）、opt-in 設定キーの追加として再評価する（既定の帯域限定は変えない）
- **日本語（非 ASCII）ホスト名の可読表示**の要望が反復した場合（同上）、Punycode の安全な復号表示（ホモグラフ対策付き）を再評価する
- BCL API の制約（キャンセル不可・前方解決の随伴・TTL 情報が取れないこと）が実運用で問題化した場合、専用 DNS クライアント実装の導入を、供給網統治の基準（ADR-0003 決定 3 と同等の検証記録）込みで再評価する
- 既定オンに起因する事故・異議（DNS 負荷・意図しないクエリの報告）が利用者から挙がった場合、既定値の向きを再評価する
- PTR 未整備環境で「なぜ名前が出ないのか」の問い合わせが反復した場合、**状態画面での解決状況の案内**（例: 表示分の大半が未登録である旨と CF-D6 への導線）を再評価する（v0.1 は状態画面のカウンタ + 利用者向けドキュメントで対応する）
