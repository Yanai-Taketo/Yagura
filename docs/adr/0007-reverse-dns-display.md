# ADR-0007: 閲覧画面の送信元逆引き（PTR）ホスト名表示

- 状態: proposed
- 日付: 2026-07-06
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md)（目的——Windows 管理者の既存スキルで運用できること）、[ADR-0002](0002-architecture-principles.md)（決定 2 受信パイプラインの品質原則・決定 4 外部送信の既定無効）、[ADR-0003](0003-ui-policy.md)（決定 5 文言原則）、[ADR-0004](0004-security-model.md)（決定 1 信頼ネットワーク前提）、[ui.md](../design/ui.md) §4・§7、[configuration.md](../design/configuration.md) §8

## 文脈と課題

閲覧画面（ログ検索の結果一覧・詳細表示・ダッシュボードの送信元別受信状況）の送信元は IP アドレスで表示している。運用者は「この IP はどの機器か」を機器台帳や `nslookup` で突合しており、この往復が機器同定のボトルネックになる。本製品のターゲット環境（Windows-shop。ADR-0001）では AD 統合 DNS の逆引き参照ゾーンが整備されていることが多く、送信元 IP の逆引き（PTR）ホスト名を併記すれば、既存の DNS 資産だけで機器同定が大幅に楽になる。

一方で、この機能には次の緊張があり、方針の確定が必要である:

1. **既存の「ホスト名」との混同**: 詳細表示が現在出している「ホスト名」は **syslog HOSTNAME フィールド（送信者の自己申告）**である。逆引きホスト名は**送信元 IP から DNS で引いた別由来の名前**であり、両者は食い違い得るし、どちらか一方だけが存在することもある。混同させない表示設計が要る
2. **性能**: 描画ごとに DNS を問い合わせることは許されない。特に **PTR 未登録の IP の解決失敗は実測で約 4.5 秒かかる**（下記検証記録）。検索結果は最大 10,000 件・送信元一覧は最大 1,000 件であり、素朴な実装は画面を破壊する
3. **信頼性**: PTR 未整備の LAN は珍しくない。名前が引けないことは異常ではなく正常系であり、IP のみ表示へのフォールバックが既定の姿である
4. **セキュリティ**: 逆引き名は**送信元（を管理する者）が自分の PTR レコードで任意に操作できる untrusted 値**である。表示の無害化と、認証・同定の根拠に使わない線引きが要る
5. **挙動の透明性**: 本機能はサーバが自発的に DNS クエリを発する。ADR-0002 決定 4（あらゆるデータの外部送信は既定で無効）との関係を正面から整理する必要がある

### 検証記録（conventions.md「技術的主張の検証」——API の挙動を推測で設計根拠にしない）

実施 2026-07-06。開発機 Windows 11（10.0.26200）・.NET SDK 10.0.301 で `System.Net.Dns` を実機検証した:

- **公開オーバーロード**: `GetHostEntryAsync` は 4 種——`(string)` / `(string, CancellationToken)` / `(string, AddressFamily, CancellationToken)` / `(IPAddress)`。**`IPAddress` を取る版に CancellationToken オーバーロードはない**（IP 文字列を渡せば逆引きになるため string 版で代用できる）
- **PTR 登録あり**（8.8.8.8）: 82 ms で `dns.google` を取得。返却された `AddressList` に前方解決の結果（8.8.8.8, 8.8.4.4）が含まれており、**本 API は逆引きに加えて前方解決も行う**ことを確認した
- **PTR 未登録**（TEST-NET-1 の 192.0.2.1・プライベート帯の 10.255.255.250）: いずれも `SocketException`（ホスト不明）まで**約 4.5 秒**
- **CancellationToken は進行中の解決を中断しない**: PTR 未登録 IP に 300 ms のトークンを渡しても解決は約 4.6 秒完走し、`OperationCanceledException` ではなく元の `SocketException` が返った（トークンは発火済みであることを確認）

この実測から、タイムアウトは「呼び出し側が待つのをやめる」制御としてしか設計できず、裏で OS の解決は完走する。したがって**解決を UI 描画の同期経路に置かないこと・同時解決数を絞ること・失敗（未登録）を必ずキャッシュすること**が、設計の選択肢ではなく前提条件になる。

## 検討した選択肢

### 解決のタイミング

- **(a) 表示時解決 + IP 単位キャッシュ（採用）**: 閲覧層が表示のために解決し、結果をメモリキャッシュする。受信経路・DB スキーマ・provider 契約に変更がない。名前は保存されないため逆引き名での検索はできない
- (b) 受信時解決・DB 保存: 逆引き名での検索が可能になる利点はある。却下理由: ①受信パイプラインに DNS という外部依存の遅延・障害モードを持ち込む（PTR 未登録 1 件が約 4.5 秒——「ログを失わない」の主経路を表示都合で複雑化させない。ADR-0002 決定 2）、②保存時点の名前が固定化し、後から引き直した名前と食い違う第 3 の名前を作る、③スキーマ変更と provider 実装への波及、④保存すれば untrusted 値が証跡 DB に混入する。要望が反復した場合の再評価対象として残す（再評価トリガ参照）
- (c) 両方（受信時保存 + 表示時再解決）: (b) の費用に加えて「保存された名前と今の名前」の 2 重表示問題を新設する。却下

### 既定値（オン / オフ）

- **(a) 既定オン・opt-out（採用）**: 問い合わせ先は OS に構成済みの DNS サーバのみ（新しい宛先を設定させない）であり、信頼ネットワーク前提（ADR-0004 決定 1——送信元・本製品・閲覧者が同一管理主体）では、サイト自身の DNS インフラへの名前解決は新たな信頼境界を跨がない。機器同定は本製品の中核価値であり、既定オフは価値を大半の利用者に届けない
- (b) 既定オフ・opt-in: ADR-0002 決定 4 の字義に最も忠実だが、同決定が禁じるのは「**観測データの製品外への自動送信（テレメトリ）**」であり、利用者インフラ（DNS）の通常利用とは質が異なる——`nslookup` を打つのと同じ向きの通信である。却下。ただしこの整理を暗黙にしない: **DNS クエリが発生する事実と停止手段は利用者向けドキュメントに明記し、オフ設定を初日から提供する**（下記決定 4）

### 表示の位置づけ

- **(a) 併記・補完（採用）**: IP アドレスを主、逆引き名を補助として併記する。syslog HOSTNAME（自己申告）は従来どおり独立の項目として残し、置き換えない
- (b) 置換（名前が引けたら IP の代わりに表示）: PTR の操作・陳腐化で誤った名前だけが見える状態を作る。コピー・機器台帳との突合・検索条件（IP）との対応も揺れる。却下

### 逆引き名の検証（FCrDNS）

- **前方確認（FCrDNS——逆引き名を前方解決して元の IP と一致するか確認する手法）は行わない**: 本表示は参考情報であり認証ではない。検証を実装すれば「検証済み」という誤ったお墨付き（PTR と前方の両方を管理する攻撃者は素通しになる）を与える方が害が大きい。untrusted であることを表示設計（無害化 + IP 主・名前補助 + 由来明示）で扱う

## 決定

1. **併記（補完）**: 逆引きホスト名は送信元 IP の**補助表示**とする。syslog HOSTNAME の表示を置き換えず、両者の由来の違い（DNS に登録された名前 / ログに記載された名前）を画面で明示する。表示規約の詳細は [ui.md](../design/ui.md) §4 で定める
2. **表示時解決 + キャッシュ充填型**: 閲覧層（Yagura.Web）のシングルトンサービスが IP → 名前を解決・キャッシュする（成功・未登録の両方に TTL を持つ）。**描画は解決を待たない**——IP は常に即時表示し、キャッシュ命中時のみ名前を同時表示、未解決分は非同期解決の完了時に表示へ反映する。受信パイプライン・DB スキーマ・provider 契約は変更しない。**逆引き名は検索条件に使えない**（v0.1。検索は従来どおり IP）
3. **資源保護を仕様に含める**: 同時解決数の上限・キャッシュ件数の上限（超過時は新規解決を行わず IP のみ表示——検索結果は送信元詐称を含み得るため、キャッシュを外部入力で無制限に成長させない）・解決待ちの打ち切り（上記検証記録の制約に基づき「待つのをやめる + 完走結果は事後にキャッシュへ反映」方式）。仮値と確定手続きは ui.md §12（UI-10）で追跡する
4. **設定**: `Viewer:ReverseDns:Enabled`（既定 **オン**）。オフ時は DNS クエリを一切発しない。不正値は**縮小側（無効）で継続**する——本機能は外向きのクエリを発生させるため、不正値で「発生する側」へ倒さない（configuration.md §1 の縮小側原則の適用。詳細は同 §8）
5. **無害化表示**: 逆引き名は untrusted 値として扱う。①プレーンテキストとして描画する（マークアップ・リンクとして解釈させない）、②ホスト名として妥当な文字集合（英数字・ハイフン・ドット。RFC 1123 の LDH 形式）以外を含む名前、および DNS 名の上限（253 文字）を超える名前は**表示しない**（切り詰め・整形をして見せない——解釈を偽装しない流儀）、③逆引き名を検索条件・リンク URL・監査記録・認可判定に自動使用しない
6. **観測**: 解決の成功 / 未登録 / 失敗（タイムアウト・DNS 障害）を発生箇所別カウンタで観測する（計器名は architecture.md §4.1.1 の規則に従い `yagura.web.reverse_dns.*`。実装 PR で同節の一覧に追記する）。DNS インフラの異常が「名前が出ない」という見た目の変化だけで黙って進行しない

## 帰結

- **良くなること**: 機器同定が既存の DNS 資産だけで楽になる（追加の台帳・設定を要しない）。受信経路・スキーマは無変更で、機能の失敗モードは「IP のみ表示に戻る」に閉じる——「ログを失わない」原則と交差しない
- **悪くなること**: 逆引き名で検索はできない（名前が見えるのに検索できないことは誤解を生み得る——検索欄のラベル・ヘルプで手当てする）。キャッシュ TTL の間は付け替え後の古い名前が表示され得る。PTR 未整備環境では価値がなく、「なぜ名前が出ないのか」の問いを生む（operations.md への申し送りで案内する。configuration.md §10）
- **受け入れるリスク**: PTR は送信元側で操作可能であり、誤同定を誘う名前を表示させられる余地は残る。緩和は決定 5（無害化）・決定 1（IP が常に隣にある併記原則）・由来の明示であり、**表示は参考情報であって同定の証明ではない**ことを利用者向けドキュメントにも明記する

## 先送りにする場合の再評価トリガ

- **逆引き名での検索・フィルタ**の要望が反復した場合（目安: 独立した利用者から 3 件以上）、却下案 (b)（受信時解決・保存）を個別 ADR で再評価する
- BCL API の制約（キャンセル不可・前方解決の随伴・TTL 情報が取れないこと）が実運用で問題化した場合、専用 DNS クライアント実装の導入を、供給網統治の基準（ADR-0003 決定 3 と同等の検証記録）込みで再評価する
- 既定オンに起因する事故・異議（DNS 負荷・意図しないクエリの報告）が利用者から挙がった場合、既定値の向きを再評価する
