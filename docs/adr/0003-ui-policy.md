# ADR-0003: UI 方針 — Blazor・デザインシステム先行・テーマ

- 状態: proposed
- 日付: 2026-07-04
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md)（導入体験の原則・言語ポリシー）、[ADR-0002](0002-architecture-principles.md)（単一プロセス構成・セットアップウィザード・帰結の UI 資源競合）

## 文脈と課題

Web UI は本製品の顔であり、セットアップウィザード（ADR-0002 決定 5）・ログ検索・ダッシュボードという中核体験を担う。UI の品質は「画面ごとに場当たりで作らない」ことで決まる——デザインの一貫性は、後から整えるものではなく、最初に規約として固定するものである。本 ADR は UI の実装技術・デザインシステムの進め方・テーマ方針を定める。個々の画面設計・具体的な配色値は全体設計書（ui.md）に委ねる。

なお、UI の重い操作（大量ログの検索等）から受信パイプラインを守る防御（結果上限・タイムアウト等。ADR-0002 帰結の宿題）は本 ADR の対象外であり、全体設計書で扱う。

## 決定

### 1. 実装技術: Blazor（Interactive Server 単一モード）

- UI は **Blazor** で実装する（.NET の Razor Components。ADR-0002 決定 1 のランタイム選定に従う）
- レンダリング方式は**全画面 Interactive Server の単一モード**とする。理由は 2 つ:
  - **採用するコンポーネントライブラリ（決定 3）の実証された動作領域に全画面を置くため**。MudBlazor は静的サーバレンダリング下ではスタイル描画のみで対話機能が動作しないことを確認済み（公式 Issue #10946・Discussion #8818、確認日 2026-07-04）
  - **2 方式の混在規約を持たないため**。本製品の同時利用者は LAN 内の少数管理者であり、接続維持（circuit）のサーバ側コストは小さい——公式ドキュメントの目安は最小アプリで circuit あたり約 250 KB、利用者あたり約 273 KB（Microsoft Learn「Host and deploy server-side Blazor apps」、確認日 2026-07-04）であり、数十接続でも数 MB 台にとどまる。一方、方式混在は「この画面はどちらか」の判断・レビュー・実装流儀の二重化を毎 PR で払うことになり、この規模の製品では費用が便益を上回る

  なお、UI 起因のサーバ負荷の主因は接続維持ではなく重いクエリであり、その防御は全体設計書の管轄（文脈と課題を参照）
- WebAssembly レンダリングは採用しない（閉域環境での初回ペイロード・運用の単純さの観点。「検討した選択肢」参照）
- **JavaScript は原則手書きしない**。Blazor で表現できない場合のみ、分離 JS モジュールとして理由と共に配置する。npm・CDN への依存は持たない（閉域完結・供給網統治。依存はすべて NuGet で管理する）

### 2. デザインシステム先行の原則

- **画面を作る前に、デザインシステムを全体設計書（ui.md）に定義する**。定義する内容: デザイントークン（配色・タイポグラフィ・余白・状態色）、共通コンポーネント規約（ボタン・フォーム・テーブル・通知・空状態の標準形）、画面レイアウトの骨格
- **ページ固有の CSS・JavaScript の追加を原則禁止**とする（画面ごとの継ぎ足しがデザイン崩壊の主因になる、という一般則への構造的対策）。**インラインスタイル（`Style=` 属性・style 属性）も禁止対象に含む**
- **例外の手続き**: ページ固有であるべき理由を PR に明記し、共通化候補として ui.md の宿題一覧に登録した上で認める。例外は暗黙に増やさず、記録を残して通す
- 頻出 UI パターンは共通コンポーネントに集約し、ページからコンポーネントライブラリの低レベル API を直接散乱させない。**共通コンポーネントへの部品追加は、外部コントリビュータの貢献の入口として位置づける**（追加の提案手順は ui.md で定義する。ADR-0002 の provider 契約と同じ考え方）

### 3. コンポーネントライブラリ: MudBlazor

- **MudBlazor を採用する**。採用判断の検証記録（いずれも 2026-07-04 確認）:
  - 最新安定版 9.6.0（NuGet flatcontainer API で確認。導入時に再検証する）
  - 採用実績: NuGet 累計ダウンロード約 3,193 万（NuGet Search API）、GitHub Star 10,470
  - 保守状況: 最終コミット 2026-07-03（活発に開発中）
  - ライセンス: MIT（本体の Apache-2.0 との組み合わせに問題なし。NOTICE への記載は導入 PR で行う）
- 採用理由: NuGet 単体で完結し npm を持ち込まない / ライト・ダーク切替を含むテーマシステムを内蔵 / C# のみで完結 / 上記の検証済み採用実績と保守状況
- **MudBlazor が同梱する静的アセット（CSS/JS）も供給網の一部として脆弱性監視・更新追従の対象とする**（具体的な監視手段は詳細設計で定める）
- ライブラリ依存のリスク（メジャー更新追従・保守停滞）への緩和は決定 2 の集約規約が兼ねる: ページがライブラリ API を直接使う面を狭く保つことで、将来の更新・差し替えの影響範囲を限定する

### 4. テーマ: ライト基調 + ダーク切替

- **既定はライトテーマ**（クリーンな業務系）。**ダークテーマへの切替**を UI 上で提供する。OS の設定への追従は選択できる項目とし、既定では利用者の手動選択を尊重する（勝手に切り替わらない）
- **ダークテーマは決定 6 の受け入れ条件（コントラスト等）の対象**であり、ライトの副産物ではない。特にダッシュボードは照明を落とした部屋の大型モニタに長時間掲示される運用を想定ユースケースに含める
- 全色はデザイントークン経由で参照し、テーマ切替で破綻しない構造を最初から守る（直接色指定の禁止）

### 5. UI 文言の原則

- 画面の文言は**平易な日本語**とし、開発用語（provider・opt-in・スプール等）を利用者向けの画面にそのまま出さない（例:「スプール」→「一時保管」）。**エラーメッセージ・通知（メール等）の文言も本原則の対象に含む**
- 用語対応表は全体設計書で管理する。**初版はデザインシステム定義（決定 2）と同時に作成し、新しい用語はそれを導入する PR で追記する**。利用者向けヘルプ・マニュアルも同じ対応表を参照する
- 文言はリソースとして分離し、将来の英語化（ADR-0001 決定 7 の英語 README と同じ文脈）に備える。v0.1 の提供言語は日本語のみとする

### 6. 利用しやすさの下限

- キーボードのみでの主要操作の完遂と、テキストコントラストの確保（WCAG AA 相当を目安）を、**ライト・ダーク両テーマについて**デザインシステムの受け入れ条件に含める。個別基準は全体設計書で定める

## 検討した選択肢

### レンダリング方式

- **(a) 全画面 Interactive Server 単一モード（採用）**: 決定 1 の理由のとおり。コンポーネントライブラリの実証済み動作領域に全画面を置き、方式混在の規約コストを持たない
- (b) 静的 SSR 既定 + 対話画面のみ Interactive Server（本 ADR の初稿案）: 接続維持コストの最小化が狙いだったが、①MudBlazor は静的 SSR 下で対話機能が動作しない（検証済み、2026-07-04）ため SSR 画面ではライブラリの大半が使えず、②LAN 内少数利用者では節約できる接続コスト自体が小さく、③方式混在の判断・レビュー・実装流儀の二重化コストが常設になる。費用対効果が逆転しているため撤回
- (c) WebAssembly（クライアント実行）: サーバ資源は軽くなるが、初回ペイロードが大きく、API 層の追加設計が必要になり、少人数の管理者が LAN 内で使う本製品では利点が費用に見合わない。却下

### コンポーネントライブラリ

- **(a) MudBlazor（採用）**: 決定 3 の検証記録を参照
- (b) Microsoft Fluent UI Blazor（4.14.3 が安定版、5.0 は RC 段階。ライブ検証済み 2026-07-03）: ①目指すデザイン方向（決定 4 の「クリーンな業務系」。オーナーが選定時に Windows Fluent 風を選択しなかった経緯を含む）との適合判断、②大規模改版（v5）の移行期にあり、採用直後にメジャー移行を踏む見通しの悪さ、の 2 点で却下
- (c) ライブラリなし（自前 CSS + 素の Blazor）: 統制は最大だが、フォーム・テーブル・ダイアログ等の基礎部品を品質込みで自作する工数が大きく、デザイン刷新の目的（プロの見た目の底上げ）に対して遠回り。却下
- (d) CSS フレームワーク（Bootstrap / Tailwind 等）+ 素の Blazor: 見た目の底上げはできるが、対話部品（ダイアログ・トースト等）は結局 JS か自作に頼ることになり、決定 1 の「JS を原則書かない」と緊張する。Tailwind はビルドに npm 系ツールチェーンを要する。却下

## 帰結

- 良くなること: 画面の一貫性が規約と部品の両面で担保される。閉域環境で完結する UI スタック。ウィザードやダッシュボードを C# だけで実装でき、レンダリング方式の判断が不要になる
- 悪くなること: MudBlazor という外部依存に UI の広い面を預ける（緩和は決定 3 に記載）。デザインシステム先行は最初の画面が出るまでの工程を 1 つ増やす。全画面が接続維持型になるため、同時閲覧者数に比例したサーバ側状態を持つ（LAN 内少数利用の前提では小さいが、上限の設計は必要——ADR-0004 への申し送り参照）
- 受け入れるリスク: ネットワーク瞬断時に再接続動作が利用者に見える。**再接続中の表示・文言も決定 5 の平易化原則の対象とする**（「サーバが壊れた」と誤解させない）。より重要なリスクとして、**再接続できないまま古いデータが「最新」の顔で表示され続けること**（ダッシュボードの長時間掲示運用で特に危険）があり、最終更新時刻の常時表示・古さの明示（ステール警告）を全体設計書の宿題とする

## ADR-0004（セキュリティモデル）への申し送り

- Interactive Server の常時接続（SignalR circuit）は攻撃面である: 未認証で接続を確立できる画面（特に初期設定前のセットアップウィザード）の扱い、circuit 数の上限、origin 検証を ADR-0004 のスコープに含めること（ADR-0002 §6 の挿入点列挙に「UI 接続層」が明示されていない穴を塞ぐ）
- **circuit 存続中の認証失効の反映**も同スコープに含めること: circuit は接続確立時の認証状態を保持し続けるため、権限剥奪・セッション無効化が既存接続（特に長時間掲示中のダッシュボード）に即時反映されない問題への方針が要る

## 先送りにする場合の再評価トリガ

- MudBlazor の保守が停滞した場合（目安: 安定版リリースが 12 か月途絶）、**重大脆弱性の修正が 90 日以内に提供されない場合**、**メジャー更新の移行コストが本製品の規模に見合わないと判明した場合**（却下案 (b) Fluent UI に適用した基準と対称にする）、もしくはライセンス条件が変更された場合、コンポーネントライブラリ選定を再評価する（決定 2 の集約規約が乗り換えの影響範囲を限定している前提を維持する）。**ライブラリを再評価する場合は、決定 1 のレンダリング方式の根拠 1 つ目（ライブラリの動作領域）も併せて再確認する**（根拠 2 つ目は自立しているが、系譜を明示しておく）
- 英語 UI の提供時期: 利用者からの要望が反復した場合（目安: 独立した利用者から 3 件以上）に個別判断する（文言のリソース分離により追加コストは翻訳のみ）
