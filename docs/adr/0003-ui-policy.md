# ADR-0003: UI 方針 — Blazor・デザインシステム先行・テーマ

- 状態: proposed
- 日付: 2026-07-03
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md)（導入体験の原則・言語ポリシー）、[ADR-0002](0002-architecture-principles.md)（単一プロセス構成・セットアップウィザード・帰結の UI 資源競合）

## 文脈と課題

Web UI は本製品の顔であり、セットアップウィザード（ADR-0002 決定 5）・ログ検索・ダッシュボードという中核体験を担う。UI の品質は「画面ごとに場当たりで作らない」ことで決まる——デザインの一貫性は、後から整えるものではなく、最初に規約として固定するものである。本 ADR は UI の実装技術・デザインシステムの進め方・テーマ方針を定める。個々の画面設計・具体的な配色値は全体設計書（ui.md）に委ねる。

## 決定

### 1. 実装技術: Blazor（Razor Components 統一モデル）

- UI は **Blazor** で実装する（.NET の Razor Components。ADR-0002 決定 1 のランタイム選定に従う）
- レンダリング方式は画面の性質で使い分ける: **通常画面（検索・詳細・一覧）は静的サーバレンダリング（SSR）を既定**とし、**対話性が本質の画面（セットアップウィザード・ダッシュボード等）のみ Interactive Server** とする。既定を SSR に置くのは、接続維持を必要とする画面を最小限に保つため
- WebAssembly レンダリングは採用しない（閉域環境での初回ペイロード・運用の単純さの観点。「検討した選択肢」参照）
- **JavaScript は原則手書きしない**。Blazor で表現できない場合のみ、分離 JS モジュールとして理由と共に配置する。npm・CDN への依存は持たない（閉域完結・供給網統治。依存はすべて NuGet で管理する）

### 2. デザインシステム先行の原則

- **画面を作る前に、デザインシステムを全体設計書（ui.md）に定義する**。定義する内容: デザイントークン（配色・タイポグラフィ・余白・状態色）、共通コンポーネント規約（ボタン・フォーム・テーブル・通知・空状態の標準形）、画面レイアウトの骨格
- **ページ固有の CSS・JavaScript の追加を原則禁止**とする。新しい見た目が必要になったら、まずデザインシステム側に共通部品として追加し、ページはそれを使う。この規約は PR レビューで担保する（画面ごとの継ぎ足しがデザイン崩壊の主因になる、という一般則への構造的対策）
- 頻出 UI パターンは共通コンポーネントに集約し、ページからコンポーネントライブラリの低レベル API を直接散乱させない

### 3. コンポーネントライブラリ: MudBlazor

- **MudBlazor を採用する**（採用時最新安定版 9.6.0 を NuGet API でライブ検証済み、2026-07-03。導入時に再検証する）
- 採用理由: NuGet 単体で完結し npm を持ち込まない / ライト・ダーク切替を含むテーマシステムを内蔵 / C# のみで完結 / Blazor 向けとして最大級の実績とコミュニティ
- ライブラリ依存のリスク（メジャー更新追従・保守停滞）への緩和は決定 2 の集約規約が兼ねる: ページがライブラリ API を直接使う面を狭く保つことで、将来の更新・差し替えの影響範囲を限定する

### 4. テーマ: ライト基調 + ダーク切替

- **既定はライトテーマ**（クリーンな業務系）。**ダークテーマへの切替**を UI 上で提供し、OS の設定への追従も選択できるようにする
- 全色はデザイントークン経由で参照し、テーマ切替で破綻しない構造を最初から守る（直接色指定の禁止）

### 5. UI 文言の原則

- 画面の文言は**平易な日本語**とし、開発用語（provider・opt-in・スプール等）を利用者向けの画面にそのまま出さない（例:「スプール」→「一時保管」）。用語対応は全体設計書で管理する
- 文言はリソースとして分離し、将来の英語化（ADR-0001 決定 7 の英語 README と同じ文脈）に備える。v0.1 の提供言語は日本語のみとする

### 6. 利用しやすさの下限

- キーボードのみでの主要操作の完遂と、テキストコントラストの確保（WCAG AA 相当を目安）をデザインシステムの受け入れ条件に含める。個別基準は全体設計書で定める

## 検討した選択肢

### レンダリング方式

- **(a) 静的 SSR 既定 + 対話画面のみ Interactive Server（採用）**: 接続維持（メモリ・再接続の複雑さ）を対話が本質の画面に限定できる。単一プロセス同居（ADR-0002）の資源競合リスクの抑制にも働く
- (b) 全画面 Interactive Server: 実装は一様になるが、閲覧だけの画面にまで常時接続とサーバ側状態のコストを払う。却下
- (c) WebAssembly（クライアント実行）: サーバ資源は軽くなるが、初回ペイロードが大きく、API 層の追加設計が必要になり、少人数の管理者が LAN 内で使う本製品では利点が費用に見合わない。却下

### コンポーネントライブラリ

- **(a) MudBlazor（採用）**: 上記決定 3 の理由。9.6.0（ライブ検証済み 2026-07-03）
- (b) Microsoft Fluent UI Blazor: Microsoft 純正で Windows 系ツールとの見た目の親和は魅力（4.14.3 が安定版、5.0 は RC 段階。ライブ検証済み 2026-07-03）。しかし①オーナーがデザイン方向として Fluent 風を選択しなかった、②大規模改版（v5）の移行期にあり追従コストの見通しが悪い、③テーマのカスタマイズ自由度は MudBlazor が上、の 3 点で却下
- (c) ライブラリなし（自前 CSS + 素の Blazor）: 統制は最大だが、フォーム・テーブル・ダイアログ等の基礎部品を品質込みで自作する工数が大きく、デザイン刷新の目的（プロの見た目の底上げ）に対して遠回り。却下
- (d) CSS フレームワーク（Bootstrap / Tailwind 等）+ 素の Blazor: 見た目の底上げはできるが、対話部品（ダイアログ・トースト等）は結局 JS か自作に頼ることになり、決定 1 の「JS を原則書かない」と緊張する。Tailwind はビルドに npm 系ツールチェーンを要する。却下

## 帰結

- 良くなること: 画面の一貫性が規約と部品の両面で担保される。閉域環境で完結する UI スタック。ウィザードやダッシュボードを C# だけで実装できる
- 悪くなること: MudBlazor という外部依存に UI の広い面を預ける（緩和は決定 3 に記載）。デザインシステム先行は最初の画面が出るまでの工程を 1 つ増やす
- 受け入れるリスク: Interactive Server 画面はネットワーク瞬断時に再接続動作が利用者に見える。対象画面を限定することで影響範囲を抑える

## 先送りにする場合の再評価トリガ

- MudBlazor の保守が停滞した場合（目安: 安定版リリースが 12 か月途絶、または重大脆弱性への対応遅延）、もしくはライセンス条件が変更された場合、コンポーネントライブラリ選定を再評価する（決定 2 の集約規約が乗り換えの影響範囲を限定している前提を維持する）
- 英語 UI の提供時期: 利用者からの要望が反復した場合（目安: 独立した利用者から 3 件以上）に個別判断する（文言のリソース分離により追加コストは翻訳のみ）
