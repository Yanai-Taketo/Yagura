# ADR-0020: フォワーダ配布キット MSI の管理画面アップロード(管理 UI 認証有効時限定)

- 状態: proposed
- 日付: 2026-07-24
- 決定者: YANAI Taketo
- 関連: [ADR-0008](0008-forwarder-kit-generation.md)(設計条件 9 の「手順は提供しない」無条件文 = 本 ADR の部分 supersession 対象)/ [ADR-0010](0010-admin-ui-authentication.md)(決定 1 の loopback 認証 opt-in・決定 9 の再起案指針)/ [ADR-0011](0011-app-auth-failure-backoff.md)・[ADR-0013](0013-admin-winauth-session.md)(認証基盤の実体)/ [ADR-0004](0004-security-model.md)(決定 5 ACL・決定 7 監査)/ [ADR-0015](0015-gmsa-service-account.md)(ACE 付与先の実行アカウント追随)/ [Issue #283](https://github.com/Yanai-Taketo/Yagura/issues/283)(再起案元。lab 実機棚卸しの記録を含む)/ Issue #130(初回提案。2026-07-09 クローズ)/ Issue #171(配置フォルダ ACL 乖離の教訓)/ PR #167(却下 draft と 5 ペルソナ指摘の保存先)/ [security.md](../design/security.md) §2.4・§2.5・§4.3・§5.1/ [利用者ガイド: Windows イベントログを Yagura へ転送する](../guides/forward-windows-eventlog.md)(手動配置手順)

## 文脈と課題

フォワーダ配布キットの MSI オプトイン同梱(ADR-0008 設計条件 9)は、MSI の配置手段として
**サーバのファイルシステムへの手動配置**(RDP・コンソール・リモート PowerShell)のみを認めている。
管理 UI からのアップロード(Issue #130)は 2026-07-09 の PR #167 ペルソナレビューを経て
**見送り・クローズ**となった。却下理由は次の 3 点に要約される(PR #167 オーナー判断):

1. **判定の問題**: 設計条件 9 の存在理由そのもの(「防御上最も価値の高い書き込み対象」への
   身元保証)を撤回する変更であり、amendment ではなく supersession 相当(田中・クリス)
2. **身元保証の構造的不成立**: 管理リスナが無認証 loopback である当時の構成では、補完統制を
   積んでも「誰が MSI を差し替えたか」を特定できない(監査の操作者欄が常に `127.0.0.1`・
   二段階確認が自己承認に留まる)(田中)
3. **常在する書き込み ACE**: サービスアカウントへの書き込み権限は利用有無に関わらず 24/365
   存在し続け、Web プロセス侵害時の被害範囲を「全端末に配布される MSI の差し替え」まで拡大する。
   低頻度操作の利便性と釣り合わない(クリス)

同時に再評価トリガ「**管理 UI に身元保証を伴う認証(AD 認証 opt-in 等)が導入された時点で、
全ペルソナ指摘を出発点として supersession として再起案する**」が明記された。

### トリガの成立と再起案の前提(Issue #283)

- ADR-0010(管理 UI 認証 opt-in + リモート管理解禁)・ADR-0011(失敗試行の三層防御)・
  ADR-0013(共存セッションモデル)がいずれも accepted・実装済み
- ADR-0010 決定 1 の **loopback 認証 opt-in**(`Admin:Authentication:RequireForLoopback`)により、
  却下理由 2 の「無認証 loopback 経路」を構成で排除できるようになった
- ADR-0010 決定 9 は #130 再起案時の設計指針として「**loopback 認証 opt-in の有効化を
  アップロード機能の利用の前提条件とすることを強く推奨する**」と明記済み
- オーナー裁定(2026-07-16): **認証有効時限定で再起案する**(Issue #283)

### 実機棚卸しの結果(2026-07-18 lab。ADR-0010 先送り節・再評価トリガ ① の消化)

Issue #283 に記録した実測(Windows Server 2025 / yagura.test DC)の要点:

- 管理リスナは `127.0.0.1`/`::1`:8515 のみに束縛されているが、**TCP loopback には ACL が無く**、
  到達主体の集合は「このホストでコードを実行できる者」と一致する。実測で **93 のサービス
  プロセス**が到達可能だった
- DC ではログオン可能な主体が実質管理者相当に限られるが、**本製品の主たる想定配置である
  メンバーサーバでは既定で `BUILTIN\Users`(= Domain Users 全員)が対話ログオン権を持つ**。
  DC での実測を一般化してはならない
- 結論: **loopback 束縛は認可境界として数えられない。MSI アップロード受理の可否は
  認証・認可(管理役割の保持)で決めるべき**であり、Issue #283 の前提条件(認証有効時限定)の
  必要性を実測が裏付けた

### #167 当時から変わったもう一つの前提(Issue #171)

PR #167 の設計はサービスアカウントへの書き込み ACE の**常時付与**を前提としていたが、
その後 Issue #171 で「配置フォルダがデータルートの ACL を継承し、サービスアカウントが
既に書き込み可能になっている」という設計乖離が発見・修正された。現在の実装
(security.md §5.1)は**継承を断ち、サービスアカウントは読み取りのみ**である。
本 ADR はこの確定済みの既定 ACL を出発点とし、これを無条件に緩める設計は採らない。

### 本 ADR の位置づけ

PR #167 の draft(ステージング→アトミックリネーム・入力検証・単一化・SHA256 照合・
二段階確認・監査の統制設計)は**再利用する**(Issue #283 の設計論点「PR #167 の再利用可否」への
回答)。ただし却下理由 1〜3 のそれぞれに構造的な対処を加える:

| 却下理由 | 本 ADR の対処 |
|---|---|
| 1. supersession 相当の変更を amendment に埋めた | **新規 ADR として起案**し、ADR-0008 設計条件 9 の該当文言の部分 supersession を明示する(決定 7) |
| 2. 操作者の身元保証が構造的に不成立 | **管理リスナに無認証の到達経路が存在しない構成でのみ機能を有効化**する(決定 1)。全経路が認証済みになるため、配置・削除の監査に認証済み利用者名が必ず入る |
| 3. 書き込み ACE が常在する | **既定 ACL(読み取り専用)は変えない**。書き込み経路は OS 管理者の明示的な ACE 付与操作によってのみ開き、撤去も自由(決定 2)。ACE が存在するのは管理者がそれを選んだ環境・期間だけになる |

## 検討した選択肢

### 選択肢 A: 現状維持(手動配置のみ)

- 利点: 追加実装ゼロ。攻撃対象領域が増えない。#171 で確定した ACL がそのまま守られる
- 欠点: MSI 同梱運用の版更新(検証済み版の更新は各リリース準備時 + CVE 公表時——
  conventions.md の版運用)のたびに、サーバへの RDP・ファイルシステムアクセスが必要になる。
  管理 UI 認証(ADR-0010)によりその他の管理操作がリモートで完結するようになった今、
  MSI 配置だけが UI の外に残る
- **透明性の注記**(ADR-0008 改訂履歴 1 と同じ作法。ペルソナレビュー クリスの指摘): 本再起案は
  オーナー起点(Issue #283)であり、**独立利用者からの要望・手動配置の負担に関する報告は
  起案時点で 0 件**である。「実害の実測」ではなく再評価トリガの成立とオーナー裁定に基づく
  判断であることを明記し、その裏返しとして決定 6 の撤回基準を証拠要求が対称になる形で定める
- 却下(ただしトリガ成立後の再起案というオーナー裁定自体が A からの移行判断である。
  経路 (a) = 手動配置は本 ADR 採用後も廃止しない)

### 選択肢 B: 認証有効時限定の管理 UI アップロード(採用)

配置経路 (b) を、管理リスナに無認証の到達経路が存在しない構成に限って提供する。
書き込み ACE の扱いで 3 案を比較した:

- **B-1: サービスアカウントへ ACE を常時付与(PR #167 案)**: 却下理由 3 がそのまま残る。
  アップロード機能を使わない環境でも Web プロセス侵害時の被害範囲が拡大する。却下
- **B-2: 既定 ACL は不変とし、OS 管理者の明示操作で ACE を付与した環境・期間のみ書き込み可能
  (採用)**: 「MSI の書き込み経路を開く」という判断そのものを OS レベルの管理者権限
  (フォルダの ACL を変更できる者 = Administrators)に残す。#167 が失った「OS レベルの
  身元保証」を、**経路の開放の時点**に保存する構造になる(日々のアップロード操作の身元保証は
  決定 1 の認証が担う)。ACE を付与しない環境は従来どおり読み取り専用のままであり、
  機能を使わない利用者のリスクプロファイルは一切変わらない(PR #167 佐藤指摘への回答)
- **B-3: 昇格ヘルパープロセス経由の間接書き込み**(サービスアカウントには権限を与えず、
  Administrators 権限の別プロセスへ書き込みを委譲): PR #167 のクリス指摘(時間的露出窓の
  比較が未実施)に答えて再評価した。露出窓の軸では B-3(操作実行中のみ)が B-1(24/365)より
  明確に優れるが、**B-2 の露出窓は「管理者が ACE を付与している期間」であり、運用者が
  自分で制御できる**——使うときだけ開き、使い終えたら閉じる運用が可能で、B-3 の利点の
  大部分を新しい攻撃面(IPC・プロセス起動・昇格境界の実装とテスト)なしに得られる。
  **ただし両者は等価ではない**(ペルソナレビュー クリスの指摘による限定): B-3 の露出窓は
  構造(操作終了で自動閉鎖)が保証するのに対し、B-2 の露出窓は運用者の規律(撤去操作)が
  保証する——閉じ忘れれば開いたままになる。この差は開放継続の周期通知・乖離警告・
  開放状態の常時表示(決定 2)で緩和するが、構造保証には届かない。
  また B-3 は Windows サービスからの対話的昇格が成立しない(サービスは UAC ダイアログを
  出せない)ため、実体は「常駐する第二の昇格サービス」または「スケジュールタスクの
  オンデマンド起動」になり、それ自体が新しい常在の攻撃対象になる。却下

### 選択肢 C: 認証を前提条件にせず補完統制のみで提供(#167 の構図の再掲)

- 却下: 実機棚卸し(文脈と課題)が「loopback 束縛は認可境界として数えられない」ことを
  実測で確定させた。補完統制(入力検証・二段階確認・監査)は操作の**内容**を統制するが、
  操作の**主体**を特定できない構成では監査の中核(誰が)が成立しない。却下理由 2 の再来であり、
  検討の俎上に載せない

## 決定

選択肢 B(B-2)を採用する。ADR-0008 設計条件 9 の配置経路に
「**(b) 管理 UI 経由のアップロード**」を追加する。ただし PR #167 案と異なり、
機能の有効化条件(決定 1)と書き込み経路の開放条件(決定 2)を分離して二重に統制する。

### 1. 有効化条件と UI 表出

アップロード機能(配置経路 (b))は、次の**全条件が成立する構成でのみ**有効化する:

- **(i) 管理 UI 認証が有効**: Windows 統合認証・アプリ独自認証の少なくとも一方が有効
  (ADR-0010 決定 1 の「認証が有効」と同じ単純 OR 判定)
- **(ii) loopback 認証 opt-in が有効**: `Admin:Authentication:RequireForLoopback = true`。
  リモート経由は §2.5 により常に認証必須のため、(i) + (ii) の成立は
  「**管理リスナに無認証の到達経路が存在しない**」ことと等価になる(Issue #283 の前提条件)
- **(iii) アップロード機能自体の明示 opt-in**: 設定キー(名称は実装 PR で確定。
  仮 `Admin:ForwarderKit:MsiUpload:Enabled`。既定 `false`)。認証を有効化しただけで
  書き込み系の新機能が黙って現れることを避ける(「認証が導入されたから MSI アップロードも
  自動的に安全になった」という誤読の否定——ADR-0010 決定 9——を設定構造でも表現する)

条件と表出・拒否の設計:

- **条件不成立時は機能を表出させない。ただし沈黙にしない**(ADR-0008 設計条件 9 の
  「チェックボックスが出ないだけの沈黙にしない」流儀): 生成画面(`/admin/forwarder-kit`)に
  「アップロードが無効であること + どの条件が欠けているか + 手動配置ガイドへの導線」を表示する。
  条件を満たせない・満たさない環境の利用者は、従来の手動配置手順で同じ結果を得られる
- **条件 (ii) の案内には代償の提示を必須で含める**(ペルソナレビュー 佐藤・クリスの指摘):
  (ii) が未設定である旨の案内は、単なる「有効化してください」にしない。loopback 認証 opt-in の
  有効化は ADR-0010 決定 1 が明記した代償——認証手段の全滅時の復旧が設定ファイル手編集 +
  サービス再起動 = **syslog 受信断**(CF-5 確定まで)——を伴うため、**本機能のために (ii) を
  新規有効化する動線では、この代償と、ブレークグラス(アプリ独自の管理者アカウントを最低 1 つ
  保持する——ADR-0010 決定 3)の強い推奨を必ず提示する**。周辺機能(MSI 配置)の利便のために
  コア機能(受信)の可用性リスクを知らずに輸入させない
- **ADR-0011 決定 4(loopback は IP レート制限・グローバルトークンバケットの対象外)は
  本前提構成でも変更しない**(ペルソナレビュー 田中の質問への整合明示): (ii) 有効環境では
  loopback にも認証が課されるが、失敗試行対策の loopback 除外(①②層)の変更は復旧経路
  (ADR-0010 決定 1)の再設計を要し、本 ADR のスコープ外である。残る露出——ローカル実行主体に
  よるアプリ独自認証への低速試行が③バックオフ(完全拒否なし)のみに律速されること——は、
  監査(3004/3006)・1019 昇格(15 分継続)で観測されることをもって受容する
- **fail-closed(起動時)**: (iii) が有効なのに (i) または (ii) が不成立の設定の組み合わせは、
  起動時検証が拒否する(1000 番台エラー。1011/1012 と同型。「なぜ起動しないか・何を直せば
  よいか」の誘導を含める)。設定ファイル手編集でこの組み合わせを作っても起動しない
- **fail-closed(UI)**: 管理画面上の設定変更でも同じ判定を書き込み前に行う(§2.4/§2.5 の
  二段構えと同型)。逆方向も対称に守る——(iii) が有効な稼働構成で (i)(ii) を無効化する
  設定変更は、書き込み前に UI レベルで拒否する(「先にアップロード機能を無効化すること」を案内)
- **エンドポイントの構造的非存在**: アップロード用エンドポイントは条件成立時のみ登録する。
  条件不成立の構成では拒否応答すら返す口が存在しない(「拒否する」より強い「そもそも無い」)
- **認可**: アップロード・削除の実行は認証済みの「管理」役割(既定 `BUILTIN\Administrators`
  SID 判定・AD 管理グループ・アプリ独自アカウント——既存の管理認可判定そのまま)に限る。
  loopback 経由であっても (ii) により必ず認証を経る

### 2. 書き込み経路の開放(ACL)——既定は変えず、開放は OS 管理者の明示操作による

- **既定 ACL は変えない**: #171 で確定した現行構成(継承を断ち、`SYSTEM`/`Administrators` =
  フルコントロール、サービスアカウント = 読み取りのみ。security.md §5.1)を既定として維持する。
  インストーラは引き続き無条件の書き込み ACE を付与しない
- **Yagura 自身は ACL を変更しない(できない)**: サービスアカウントは配置フォルダの
  WRITE_DAC を持たない(§5/§5.1 の ACL 方針)。これは制約ではなく設計である——
  **書き込み経路を開けるのは、フォルダの ACL を変更できる OS 管理者のみ**という構造により、
  「全端末に配る MSI の差し替え口を開く」判断が OS レベルの身元保証(Administrators)の
  管轄に残る。(iii) の設定 opt-in だけでは書き込みは物理的に成立しない
- **開放の手段はドキュメント化された `icacls` 手順のみとする**(生成画面にも表示する。
  コピー可能なコマンドと確認コマンドを併記する)。**インストーラによる ACE 付与は、
  無条件・opt-in プロパティ経由を問わず提供しない**——起案時の draft はインストーラの
  opt-in プロパティを第 2 案として残していたが、ペルソナレビュー(田中・リサ・クリス)が
  指摘した決定 7 の維持範囲(「緩めるインストーラを提供しない」)との矛盾を解消するため、
  起案側判断で落とした。昇格 `msiexec` 経由の付与は `icacls` 1 行に対して固有の利点がなく、
  供給網上最も注視される成果物(インストーラ)に条件分岐を増やすだけである(田中の評価を
  採用)。**UI からの自動付与ボタンも設けない**(Yagura が自分に権限を与える操作を作らない)。
  `icacls` はリモート PowerShell(管理者セッション)からも実行できる見込みであり
  (実機検証は委任 3——再レビュー クリス指摘 2 により断定を避ける)、成立すれば開放操作に
  RDP は必須でなくなる——ただしその到達手段を持つ者は手動配置(経路 (a))も行えることを
  手順書で正直に示す(決定 6 の便益の射程と整合させる)
- **UI は書き込み可否を実挙動で検出し、開放状態を常時表示する**: (i)〜(iii) 成立 +
  ACE 未付与の状態では、アップロード UI は「機能は有効だが書き込み経路が未開放であること +
  付与手順」を案内する(ここも沈黙にしない)。**未開放の案内は削除操作にも同様に適用する**
  (削除もサービスアカウントの「変更」権限を要するため。撤去済み状態で削除を試みて I/O
  エラーになる失敗様式を作らない——ペルソナレビュー 鈴木・リサの指摘)。逆に ACE が付与
  されている間は、**生成画面に「書き込み経路が開いている」ことを常時表示する**——画面を
  開くたびに開放状態が目に入ることが、閉じ忘れの検出可能性を実質的に決める(田中の指摘)
- **撤去は自由**: ACE の撤去手順もガイドに併記する(付与手順の直後に撤去手順を置く構成と
  する)。「使うときだけ開く」運用(版更新の際に付与 → 作業後に撤去)を明示的に支援する
  ——露出窓を運用者が制御できることが B-2 採用の核心である。あわせて、緊急の CVE 対応時は
  まず付与操作が必要になる(撤去済み運用の代償)ことも手順書に併記する(鈴木の指摘)
- **ACL の開放状態の検出は稼働中に機能させる**(ペルソナレビュー 鈴木・クリス・佐藤の指摘に
  より起動時検証依存から強化。検出条件は再レビュー クリス指摘 1 により二系統に確定):
  検出タイミングは**起動時に加えて周期監視(`ActiveNotificationMonitor`。§2.5 の証明書監視と
  同じ枠組み)を併用する**——本製品は再起動 = 受信断のため長期間再起動されず、起動時のみの
  検証では閉じ忘れが数ヶ月単位で沈黙するため。検出は次の二系統とする:
  - **乖離警告((iii) 無効 + 書き込み ACE 残存)**: opt-in を無効化したのに ACE が残っている
    状態は乖離(閉じ忘れ)として**警告**する——#171 の教訓(意図した ACL と実 ACL の乖離は
    検出されるまで気づかれない)を、本機能の運用にも適用する
  - **開放継続の通知((iii) 有効 + 書き込み ACE 存在)**: この状態は設定上「期待内」だが、
    推奨運用(使うときだけ開く)の典型は (iii) を有効のまま ACE だけを開閉するため、
    **閉じ忘れの主たる発生様式はこちら側**である(再レビュー クリス指摘 1)。開放が継続して
    いることを周期監視で観測し、一定期間の継続で通知する(レベル・間隔・抑制窓は委任 7 で
    確定。常置運用を選んだ環境で騒音にならない設計とする)——閉じ忘れの検出を生成画面の
    常時表示(画面を開かなければ見えない)だけに依存させない
  いずれも 1000 番台の能動通知として採番する(委任 7)。なお §5 の起動時 ACL 検証
  (期待 ACE 全体の突合・能動通知)は**本 ADR 起案時点で未実装**(security.md §5.2)であり、
  本機能の検出はそれに相乗りするのではなく、配置フォルダに限定した独立の検出として実装して
  よい(実装順序の整理は委任 7)
- **gMSA 構成**: ACE の付与先は実効実行アカウントから導出する(§5.2 の原則。
  gMSA 構成では `DOMAIN\name$` へ付与する)。手順書は両構成を扱う。**アカウント切替時、
  旧アカウントへ手動付与した本フォルダの書き込み ACE は §5.2 の切替時除去レール
  (インストーラの `icacls /remove`)に載らない**——この残骸の検出を委任 7 に明示的に含める
  (田中の指摘)

### 3. アップロード・削除の操作仕様(PR #167 統制の継承 + レビュー指摘の反映)

**射程**: アップロードの対象は MSI のみとする。キットの他の構成物(設定テンプレート・
`install.ps1`・Lua フィルタ等)はビルド時埋め込みの単一ソース(ADR-0008 設計条件 4)であり、
**実行時の差し替え口は設けない**(ペルソナレビュー リサの指摘による明文化。「他のファイルも
アップロードできるように」という将来の提案は本射程の変更 = ADR の再検討を要する)。

PR #167 draft の技術統制を継承する:

- **ステージング → アトミックリネーム**: アップロード本文は `%TEMP%` 等の広い ACL の領域を
  経由せず、配置フォルダ内のステージング名(検出パターンに一致しない名前。例
  `.uploading-<GUID>.msi`)へ直接ストリーム書き込みする。ファイルとして開けること・
  ProductVersion 読み取り・SHA256 計算のすべてを通過した場合のみ正式ファイル名へ
  アトミックにリネームする。失敗時はステージングファイルを削除し、正式ファイルは変更しない
  (中間状態を `Lookup()` に晒さない)
- **入力検証**: クライアント指定ファイル名は信用しない(保存先パスの構成に使わない)。
  拡張子は `.msi` のみ。サイズ上限(仮値 100 MiB。確定は委任 4)。最終的な格納ファイル名は
  検証(形式チェック)を通過した MSI から読み取った ProductVersion をもとに Yagura が生成し、
  既存の検出パターン(`fluent-bit-*-win64.msi` / `fluent-bit-*-winarm64.msi`)に一致させる
- **単一化**: アップロードは常に「単一の正式ファイルへの新規配置または置換」として扱い、
  複数 MSI 併存状態を構造的に作らない。単一化の単位は**アーキテクチャごと**
  (win64 / winarm64 の各 1 本。ADR-0008 改訂履歴 2 の独立判定と整合)。置換時は
  旧 SHA256・新 SHA256 を並べて表示したうえで明示確認を必須とする
- **SHA256 照合・二段階確認**: 配置確定の時点で SHA256 を計算し、検証済み版の公式ハッシュと
  照合する。一致すれば単一確認、不一致(未知の版を含む)なら既存の版不一致確認と同じ
  二段階確認(UI コンポーネント共通化)を要する。確認文言は専門用語を避け「何が起きているか・
  次に何をすべきか」を平易に示す方向性を委任 4 に含める(PR #167 佐藤指摘)
- **削除**: 配置済み MSI の削除も同じ画面から行える。削除は配置と同じ重みの操作として扱い、
  **常に二段階確認を要し**(PR #167 リサ質問への明確化——配置側の「不一致時のみ」とは
  異なる無条件適用)、削除前の SHA256 を監査に残す

PR #167 レビュー(鈴木)の指摘を設計条件として追加する:

- **circuit を経由しない転送**: アップロードは SignalR circuit の RPC を経由せず、専用の
  HTTP エンドポイント(ストリーミング受信)で受ける。circuit のメッセージサイズ上限を
  緩和する実装は禁止する(circuit 経由の大型メッセージという新しい DoS 面を作らない)。
  エンドポイントは認証セッション Cookie + CSRF 対策(既存の `IAntiforgery` 検証と同型)を要する
- **サイズ超過の事前拒否**: `Content-Length` を申告する要求は本文を読む前に拒否する。
  申告のない(または偽る)要求もストリーミング中の累積カウントで上限到達時点で
  打ち切る(両方を実装する)。ブラウザ側でもファイル選択時にサイズを事前検査し、
  送信前に警告する(PR #167 佐藤質問 3)
- **孤児ステージングファイルの掃除**: 中断(タブクローズ・ネットワーク断・プロセス停止)で
  残ったステージングファイルは、起動時 + 新規アップロード開始時に自動削除する
  (検出パターン非一致のため生成・Lookup には影響しないが、蓄積させない)
- **ディスク空き容量の事前チェック**: アップロード受付前に配置先ボリュームの空き容量を確認し、
  不足時は受け付けない。判定は 1006(空き容量警告)の閾値との単純比較ではなく、
  **「書き込み完了後も 1006 の警告閾値を下回らないこと」**(= 閾値 + 申告サイズ以上の空きが
  あること)とする(ペルソナレビュー 鈴木の指摘による確定。確定値は委任 4)——アップロードを
  受け付けた直後に空き容量警告圏へ突入する設計にしない。事前チェック通過後にスプール退避等が
  並行して空きを消費し書き込みが失敗した場合は、通常の失敗経路(ステージング削除 +
  3000 番台の監査記録)に乗る
- **受信パイプラインとの分離**: アップロード処理(ディスク書き込み・SHA256 計算)は非同期
  I/O で実装し、受信パイプライン(独立した `UdpClient`/`Socket` 系)と直接の排他関係を
  持たない。§2.5 の認証負荷分離と同じ整理をアップロード I/O にも適用し、負荷検証は
  既存ベンチ基盤のシナリオ追加で確認する(委任 5)
- **排他制御**: 同時アップロードは**プロセス全体で単一飛行**とする(進行中に後続の開始要求は
  アーキテクチャの別を問わず明確なエラーで拒否。「最後の書き込みが勝つ」を作らない)。
  単位をアーキテクチャごとにしない理由は、同時実行の実需(win64 と winarm64 を同時に上げる
  場面)が薄いのに対し、排他の実装とテストが単純になる利益が大きいため(ペルソナレビュー
  リサの指摘による確定)
- **生成との整合**: アトミックリネームによりファイル破損は構造的に起きない。リネームと
  同時刻に走った生成がどちらの版を封入したかは、`GENERATED.txt` と監査(2005)の SHA256 で
  事後に一意に判別できる——「配置操作の監査(決定 4)と生成監査の SHA256 を突合する」を
  障害調査手順としてガイドに明記する

### 4. 監査

- **配置(アップロード)・削除を 2000 番台の新規イベントとして記録する**(§4.3 の
  additive-only 採番。本 ADR では意味と記録フィールドを定義し、最終採番は実装 PR で
  次番から確定する——起案時点の次番は 2026・2027)。記録内容:
  **操作者(認証方式 + 認証済み利用者名の命名空間つき表記——決定 1 の前提条件により
  必ず値が入る)**・接続元・**loopback/リモートの別(構造化フィールド。ADR-0011 が
  アカウントキーに同じ次元を持たせた先例に倣う——ペルソナレビュー 田中の指摘)**・
  UTC 時刻・アーキテクチャ・格納ファイル名・SHA256・サイズ・
  ProductVersion・officialHashMatch・versionMismatchAcknowledged(配置)・置換時の旧 SHA256・
  削除前の SHA256(削除)
- **失敗・拒否も記録する**(ADR-0004 決定 7「拒否された試行も監査対象」。PR #167 鈴木指摘):
  検証失敗(拡張子・サイズ超過・ProductVersion 読み取り失敗)・二段階確認の拒否(利用者が
  中止した事実)・書き込み失敗・排他拒否を 3000 番台の新規イベント(起案時点の次番は 3014)に
  記録する。§4.4 の流量制御(集約)の対象に含める
- **ACE 付与・撤去の証跡**: 開放操作(決定 2)は OS 側の操作であり Yagura は実行主体に
  ならないが、ACL 乖離検出(決定 2。起動時 + 周期監視)により「opt-in 設定と実 ACL の乖離」が
  警告として証跡に残る。ACL 変更・フォルダ書き込み自体の OS レベル監査が必要な環境には
  SACL の設定手順を、**ACE 付与手順と同一の手順書内の推奨ステップとして併記する**
  (別ガイドへの参照に留めない——「開くなら見張る」をセットにする。ペルソナレビュー
  田中の指摘)。Yagura 側で代替の監査は作らない(実行主体でない操作の証跡をアプリ層で
  偽装しない)
- **公式ハッシュの来歴**(PR #167 田中指摘): 静的保持する公式 SHA256 の取得手順
  (一次情報源・確認日)・更新頻度(検証済み版の更新トリガ——各リリース準備時 + CVE 公表時
  ——と同期)・射程の限界(照合対象は検証済み単一版のみ。それ以外は常に「不一致」側の
  フローに乗る)を security.md へ明記する(委任 6)

### 5. 受け入れ基準・テスト要件(PR #167 クリス指摘 4 への対応)

他の設計条件と同水準のテスト固定を最初から要求に含める:

- **CI 回帰テスト**: ①fail-closed 起動拒否((iii) 有効 + (i)(ii) 不成立)——1011/1012 と
  同型の実プロセス E2E、②ステージングファイルが `Lookup()` から不可視であることの回帰、
  ③並行アップロード時の排他(単一飛行)のレース試験、④条件不成立時にエンドポイントが
  存在しない(404)ことの検証、⑤サイズ上限・拡張子検証・パス構成にクライアントファイル名を
  使わないことのテスト固定
- **lab 実機検証**(conventions.md「実環境依存の機能は lab 検証を受け入れ条件に含める」):
  ①ACE 付与・撤去手順の実機確認(仮想サービスアカウント・gMSA の両構成。付与 ACE が
  本フォルダ限定で継承等により他へ波及しないことの `icacls` 確認を含む)、
  ②認証有効構成でのアップロード E2E(配置 → 生成 → 監査記録の突合)、
  ③ACL 検出の二系統(乖離警告 = opt-in 無効 + ACE 残存/開放継続通知 = opt-in 有効 +
  ACE 存在の継続)がいずれも**稼働中に**(再起動を要さず)通知に至ることの確認、
  ④アカウント切替時に手動付与 ACE が §5.2 の切替レール(`icacls /remove`)で実際に
  どう扱われるかの実測(再レビュー 田中の指摘——除去されるなら残骸検出は空振りで安全側だが、
  検出仕様の期待値を実測に合わせる)
- **乖離検出(閉じ忘れの稼働中警告)が機能しない状態で、本機能を有効化可能な形で
  リリースしない**(ペルソナレビュー 佐藤・鈴木・クリスの指摘による明文化。露出窓の統制を
  運用者の記憶力だけに委ねない)
- 受け入れ基準を満たすまで機能は既定無効のままリリースに含めない(fail-closed 側に倒す)
- **外部コントリビュータの実装 PR に対する lab 項目の消化は、オーナー側(受領側)の責務と
  する**(persona-review.md が外部設計 PR のペルソナレビューをオーナー側責務とするのと同じ
  整理。ペルソナレビュー リサの指摘——lab を持たないことを実装貢献の参入障壁にしない)

### 6. 効果測定と撤回基準(PR #167 クリス指摘 6 への対応)

- 本機能は opt-in であり、利用実態(手動配置との利用比率・版更新の所要手数)を
  試用フィードバックで測定する
- **撤回基準を期限つき・証拠要求が対称になる形で定める**(ペルソナレビュー クリスの指摘に
  より強化): v1.0 公開後の最初の定期棚卸し(公開後 6 ヶ月以内を目安。実装 Issue のクローズ時に
  追跡 Issue を起票して期限を記す——委任 8)で利用実態をレビューし、①経路 (b) を通じた
  不正・誤配置の実例が観測された場合、②独立利用者からの肯定的フィードバックが 1 件も
  得られない場合、または**③棚卸し時点で利用実態を確認する手段・情報が無い場合**は、
  機能の縮小・撤回(経路 (a) への一本化)を含めて再評価する——「情報なし」を「継続」の
  既定にしない(決定 5 が fail-closed 側に倒したのと同じ流儀を測定にも適用する)。
  「進める」は測定でフィードバックされうる暫定判断である(ADR-0008 改訂履歴 1 と同じ流儀)
- **便益の射程を誇張しない(自己矛盾を含めて明記する。クリス・佐藤の指摘)**:
  - 経路の開放(ACE 付与)に OS 管理者のサーバ側操作(RDP またはリモート PowerShell での
    `icacls`)が残るため、「初回から完全に UI だけで完結」にはならない
  - さらに一段踏み込んで記す: **推奨運用(使うときだけ開く = 毎回開閉)では、版更新
    1 回あたりの総手数は「到達して付与 → UI でアップロード → 到達して撤去」となり、
    手動配置(到達してコピー 1 回)を上回る**。ACE を付与できる到達手段を持つ者は、
    その同じ到達手段で手動配置もできるからである。**本機能の手数上の便益が実際に成立するのは
    ACE を常置する運用のみ**であり、その場合は帰結に記す残余リスク(付与期間中のプロセス
    侵害の波及)が本機能の実質的な代価になる——この構造を隠さない
  - それでも提供する理由は、①常置運用を選ぶ環境(認証・監査・乖離検出・SACL を整えた上で
    UI 完結を優先する環境)に選択肢を与えること、②操作の実行と証跡(認証済み利用者名での
    監査・二段階確認・公式ハッシュ照合)が UI 経路では構造的に揃うこと(手動配置は来歴が
    生成時まで記録されない)、の 2 点である
  - ガイドには**運用モードの比較表(毎回開閉 vs 常置)と、版更新 1 回あたりの手順比較
    (経路 (a) vs (b))を載せ、小規模環境向けの選択目安を示す**(佐藤の指摘。委任 9)。
    この期待値は生成画面の記述にも反映する

### 7. ADR-0008 との関係(部分 supersession の範囲)と設計書追従

- **supersede する範囲**: ADR-0008 設計条件 9「配置フォルダと権限」の
  「既定 ACL は Administrators のみ書き込み可とし、**緩めるインストーラ・手順は提供しない**」の
  無条件文のうち、**「手順は提供しない」の部分のみ**。本 ADR は決定 1〜2 の条件下でのみ
  有効な ACE 付与手順(ドキュメント化された `icacls` 手順)を提供する
- **維持される範囲**: 既定 ACL(Administrators のみ書き込み可・サービスアカウント読み取りのみ)
  そのもの、および「緩めるインストーラ」を提供しないこと——**インストーラによる ACE 付与は
  無条件・opt-in プロパティ経由を問わず提供しない**(決定 2。起案時 draft に残っていた
  インストーラ opt-in プロパティ案はペルソナレビューの指摘——supersede 範囲との矛盾——を
  受けて落とした。この維持範囲の宣言は文字どおり成立する)。MSI 同梱の既定(非同梱)・
  供給網統治の核(リリース物・配布経路に第三者バイナリを含めない)・免責・
  検出/版判定/版整合/識別性の各条件も不変
- docs/adr/README.md の部分 supersession 作法(ADR-0011 先例)に従い、ADR-0008 の status は
  accepted のまま、状態行と設計条件 9 に本 ADR への参照注記を加える(本 ADR の起案 PR で実施)
- **設計書の追従は実装 PR で行う**(ADR-0010 が確立した先例に倣い、ズレの発生と追従対象を
  ここで名指しする): ①security.md §5.1(配置経路 (b) の追加・ACE 付与時の ACL・起動時検証の
  期待値追随)、②security.md §4.3(監査イベント ID の採番)、③configuration.md(設定キー)、
  ④利用者ガイド(ACE 付与・撤去手順、期待値の記述)。本 PR(ADR のみ)はこれらを含めない

## 帰結

- 良くなること:
  - MSI 同梱運用の版更新(検証済み版の更新は各リリース準備時 + CVE 公表時に発生)が、
    認証済みの管理 UI 操作で完結する。RDP・ファイルシステムアクセスの反復が不要になる
  - 配置・削除の監査に**認証済み利用者名が必ず記録される**——PR #167 で田中が指摘した
    「操作者特定の構造的不成立」が、前提条件によって構造的に解消される(記録されない構成では
    機能自体が存在しない)
  - PR #167 の 5 ペルソナ指摘(受け入れ基準・操作者特定・拒否監査・SignalR 転送制約・
    ステージング後始末・ACL 教訓)がすべて設計条件に昇格し、実装 PR の要求として固定される
- 悪くなること(受け入れるトレードオフ):
  - **ACE 付与期間中の Web プロセス侵害は MSI 差し替えに波及しうる**: 決定 2 は露出を
    「管理者が選んだ環境・期間」に限定するが、ゼロにはしない。付与中に Yagura プロセスが
    RCE 水準で侵害された場合、認証・監査はプロセス自身が握っているため統制として機能しない
    (ADR-0010 決定 1 が設定ファイル手編集について明示したのと同じ限界)。緩和は
    ①既定不付与・撤去自由(露出窓の最小化——ただし窓の制御は運用者の規律に依存し、
    B-3 のような構造保証ではない。決定 2 の常時表示・乖離検出はその補助である)、
    ②公式ハッシュ照合と `GENERATED.txt` の来歴(差し替えの事後検知の手がかり——**ただし
    オンボックスの記録・照合値はプロセス侵害シナリオでは偽装されうる**。この脅威の下で
    実効的に独立な証跡は SACL(OS 監査)と配布先に散在する ZIP・`GENERATED.txt` の外部
    コピーに限られる——田中の指摘による正確化。SACL 手順を付与手順とセットで示すのは
    このため)、③閉じ忘れ・開放継続の稼働中検出(決定 2 の二系統の周期監視 + 常時表示)。
    この残余リスクを明示的に受容する
  - **本機能のために loopback 認証 opt-in を新規有効化する環境は、ADR-0010 決定 1 の
    復旧経路の代償(認証手段全滅時の復旧 = 設定ファイル手編集 + サービス再起動 =
    syslog 受信断)を輸入する**(クリスの指摘)。決定 1 の条件案内でこの代償と
    ブレークグラス強推奨を必ず提示することで、知らずに交換させない
  - 有効化条件(認証 + RequireForLoopback + opt-in + ACE)の 4 点が揃って初めて使える機能で
    あり、**導入の手数は PR #167 案より明確に増える**。これは意図的な判断である——
    利便性の絶対量より「使わない環境に一切のコストとリスクを課さない」ことを優先する
  - 実装・テスト・ガイドの保守対象が増える(アップロードエンドポイント・掃除ロジック・
    ACL 期待値追随・lab 検証項目)
  - loopback 認証 opt-in を有効化できない環境(無認証 loopback を復旧経路として維持したい
    環境)は本機能を使えない。手動配置(経路 (a))が引き続き受け皿になる
- 受け入れるリスク:
  - ACE 付与手順の誤操作(対象フォルダ・対象アカウントの誤り)は Yagura からは防げない。
    手順書の具体性(コピー可能なコマンド・確認コマンドの併記)と起動時 ACL 検証の警告で緩和する
  - 検証済み版の更新が止まると、公式ハッシュ照合が「常に不一致」の運用になり二段階確認が
    形骸化しうる。版運用(conventions.md)のリリース時更新が前提であることを委任 6 で明記する

## 先送りにする場合の再評価トリガ

本 ADR が却下(選択肢 A 維持)となった場合、以下のいずれかで再評価する:

- MSI 同梱運用の利用者から、版更新時の手動配置の負担に関する報告が独立利用者から反復した場合
  (目安: 3 件以上)
- Fluent Bit の重大 CVE 公表で検証済み版の緊急更新が必要になり、手動配置の所要時間が
  対応の律速になった実例が生じた場合
- 閲覧 UI 認証を含む認証運用が既定化する方向の再評価(ADR-0004 の既存トリガ)が成立し、
  「認証済みが通常状態」という前提が変わった場合
