# ADR-0004: セキュリティモデル — 信頼ネットワーク前提の定義と防御の既定

- 状態: proposed
- 日付: 2026-07-04
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md) 決定 5（セキュリティ姿勢の委任元）、[ADR-0002](0002-architecture-principles.md) §6（挿入点）、[ADR-0003](0003-ui-policy.md)（UI 接続層の申し送り）

## 文脈と課題

ADR-0001 決定 5 は「信頼ネットワーク前提 + opt-in 強化」を姿勢として定め、前提の成立条件と限界の定義を本 ADR に委任した。また ADR-0002 §6 と ADR-0003 のレビューから、具体的な論点が申し送られている: サービスの実行権限、Web UI の認証の既定と管理操作の防御、保存データ（一時保管領域・組み込み DB・接続資格情報）の保護、UI 接続層（SignalR circuit）の攻撃面、管理操作の監査記録。

本 ADR はこれらの**方針と既定値の向き**を確定する。各決定には v0.1 での到達水準を付す。具体的な実装方式・数値は全体設計書に委ね、委任事項は末尾に一覧化する。

## 決定

### 1. 信頼ネットワーク前提の定義

本製品の既定構成が想定する環境を次のとおり定義する:

- **成立条件**: ①syslog の送信元・本製品・閲覧者が**同一の管理主体**の管理下にある、②設置セグメントがインターネットから直接到達できない（LAN / 管理 VLAN 等で分離されている）、③そのセグメントへ接続できる者を管理主体が把握・統制している
- **成立しない環境の例**: インターネットに露出するホスト、複数組織が共用するネットワーク、ゼロトラスト方針の環境。**中小規模で実際に起きやすい例も明記する**: VPN 経由の在宅勤務端末が同一セグメントに入る構成、保守業者の持ち込み端末の一時接続、ゲスト無線と同居するフラットな LAN——これらは条件③を満たさない可能性があり、該当するなら opt-in 強化（決定 3）の有効化を検討する。利用者が自己判断できる**チェックリストを利用者向けドキュメントに置く**
- **「信頼」はセキュリティ上の前提であって、配送保証ではない**。信頼ネットワーク内でも輻輳・障害によるログの取りこぼしは起こり得る（その対策は ADR-0002 の品質の原則の管轄であり、本 ADR とは独立）

### 2. 既定のセキュリティ姿勢（v0.1: 完動。localhost 判別の成立確認を v0.1 の受け入れ条件に含む）

- **受信**: 平文の UDP/TCP を既定とする（多くのネットワーク機器の既定と合わせ、導入体験を守る）
- **Web UI の閲覧**: 既定で認証なし・LAN に公開する（読み取りのみ）。**公開範囲は設定で縮小できる**（LAN 公開 / localhost のみ、の選択を提供する。認証がなくても「必要になるまで閉じておく」運用を可能にする）
- **管理操作は既定で防御する**: 設定変更・DB 接続先の切替・本番昇格・保持期間変更などの**書き込み系の管理操作は、既定ではサーバ自身（localhost）からのみ実行できる**。初回セットアップウィザードも localhost 限定とする（RDP 越しのサーバ上ブラウザも当然含む）。リモートからの管理操作は、認証 + HTTPS（決定 3）を有効化した場合に開放される
- **localhost 判別の実現方式**: 管理系エンドポイントを **loopback 専用リスナに束縛する方式を第一候補**とする（接続元 IP の事後判定は採らない——判定方式の穴を作らないため）。**制約を明記する**: 同一ホストに TLS 終端のリバースプロキシを前置すると、転送された外部要求が loopback 発として届き、この防御は成立しない。**リバースプロキシ前置構成は既定構成ではサポートせず、認証 opt-in の有効化を必須とする**（利用者向けドキュメントに明記）。loopback 束縛の成立（`127.0.0.1` / `::1` の両系、WebSocket 昇格後の維持を含む）は**一回きりの確認ではなく CI の回帰テストとして固定し、これを v0.1 の受け入れ条件とする**（将来のリスナ構成変更での退行も捕まえる）
- **サーバ上でブラウザを使えない構成**（Server Core、サーバ上のブラウジングを統制で禁止する環境）では、既定構成の管理手段は設定ファイルの手編集（ADR-0002 決定 5 のとおり手編集は引き続き可能）となる。GUI での管理は認証 opt-in 後のリモート管理で行う（この制約は「帰結」にも記載する）

### 3. opt-in 強化の範囲と提供時期

opt-in 強化として以下を提供する。**提供時期はいずれも v1.0 公開まで**（v1.0 公開基準 ADR-0006 に本項を接続する）:

- **TLS の syslog 受信**（RFC 5425、平文と共存可）
- **Web UI の HTTPS**（証明書は利用者持ち込みを基本とし、自己署名の生成支援を検討する。詳細は全体設計書）
- **Web UI の AD 連携認証**（Windows 統合認証）。**認証によるリモート管理の開放は、HTTPS の有効化を前提条件とする**（資格情報・操作内容を平文 HTTP で流さない）
- 認証を有効化した場合、管理操作は「認証済み + 許可された役割」に開放される（役割の粒度は全体設計書で定義。最小は閲覧/管理の 2 段）

### 4. 実行権限の最小化（v0.1: 完動）

- Windows サービスは**専用の低権限アカウント（仮想サービスアカウント `NT SERVICE\` 系）を既定**とし、**LocalSystem では動かさない**。AD 環境向けの gMSA は AD 認証 opt-in とあわせて対応する
- DB への実行時権限は**書き込み・読み取りの必要最小限**とし、スキーマ作成・削除などの管理権限を常時保持しない。管理操作時に必要な管理者資格情報は**都度入力・使用後破棄とし、保持しない**（長時間処理での再入力方針は全体設計書で定める）

### 5. 保存データの保護（v0.1: ACL・DPAPI 完動。削除保証の方式は全体設計書）

- **一時保管領域（スプール）・組み込み DB ファイル・設定ファイルは、サービスアカウントと Administrators のみがアクセスできる ACL で保護する**（ログ本体と同等の保護対象。ADR-0002 §2 の原則の具体化）
- **一時保管領域の消化・破棄後のデータ残置の扱い（削除保証）は全体設計書で定める**（ADR-0002 §2 の委任を明示的に受ける）
- **接続資格情報**: DB 接続は Windows 統合認証を第一推奨とし、パスワード認証を使う場合は設定ファイルに平文で置かず、**DPAPI の machine スコープ**で暗号化して保存する。**帰結として、暗号化データは当該サーバに束縛される——構成のバックアップを別サーバへ復元する際・サーバ交換の際は資格情報の再入力を要する**（この制約は利用者向けドキュメントとバックアップ手順に明記する）
- 本番昇格後の旧・組み込み DB ファイルは残置しない（削除または明示の退避を昇格手順に含める。管理外の複製を作らない）
- テレメトリ・外部送信は既定で行わない（ADR-0002 決定 4 で確定済み。参照）

### 6. UI 接続層（SignalR circuit）の防御

- **origin 検証を常時有効**とする（同一サイト以外からの接続確立を拒否）（v0.1: 完動）
- **circuit 数に上限を設ける**（v0.1: 完動。既定値は**障害時の閲覧集中——NOC 等で同時多数が一斉に画面を開く利用特性——を考慮して**全体設計書で確定。上限到達と現在値は観測可能にする。上限到達時に既存・新規のどちらを守るかも同所で定める）
- **認証状態の失効は既存 circuit に反映する**（**認証 opt-in 有効時に適用**——認証なしの既定構成には失効対象が存在しない）: 権限剥奪・セッション無効化時、既存接続を再認証に誘導する。ただし**読み取り専用の掲示用途（常時表示のダッシュボード）を定期失効で不意に切らない配慮**を全体設計書の論点に含める（無人の監視画面が深夜に認証画面へ化ける事故の防止）
- 未認証で接続できる画面は最小化する（決定 2 の localhost 限定により、初期ウィザードの未認証 circuit はローカルのみとなる）

### 7. 管理操作の監査記録（v0.1: 完動）

- **設定変更に限らず、管理操作全般**（設定変更・DB 切替・本番昇格・保持期間変更・一時保管領域の手動操作等）と、**拒否された試行**（非 localhost からの管理要求、認証失敗）を監査記録の対象とする
- 記録内容は「誰が（認証有効時は利用者名、無効時は接続元）・いつ・何を・変更前後の要約」。**認証なし構成では接続元はほぼ常に localhost であり、個人の特定は OS のログオン監査（RDP・コンソールのログオンイベント）との時刻突合が前提となる**——この限界を利用者向けドキュメントに明記する（「接続元を記録しているから個人を特定できる」という誤解を与えない）
- **記録先はアプリの記録に加えて Windows イベントログへ併記する**。Administrators は記録ファイルに触れられるが、イベントログは消去自体が痕跡（イベント ID 1102）を残し、既存のログ転送・監視のレールにも乗る。耐タンパの到達水準は「UI からの改変不能 + イベントログ併記」までとし、ローカル管理者による完全な証跡消去への耐性は本製品の守備範囲外とする（限界の明示）
- **監査記録の書き込み不能は管理操作自体を妨げない**（DB 障害中の復旧操作が記録系の巻き添えで止まる事態を排除する）。記録の失敗はそれ自体を観測可能にする

## 検討した選択肢

### Web UI の既定認証

- (a) 既定で認証必須（初期パスワード発行等）: 安全側だが、初回体験に必ず資格情報管理が入り、ADR-0001 決定 3（手編集・追加手順なしで最初のログ閲覧）と衝突する。パスワード初期化・失念対応という運用負担も既定で背負わせる。却下
- **(b) 閲覧は認証なし（公開範囲は設定可能）+ 管理操作は localhost 限定（採用）**: 読み取りは信頼ネットワーク前提に委ね、書き込みだけを物理的な操作場所で防御する。認証基盤なしで「設定を書き換えられる穴」を塞げる
- (c) 閲覧も管理も認証なし: 管理操作の防御がゼロになり、LAN 上の誰でも DB 接続先を差し替えられる。監査観点の指摘（PR #7 のレビュー記録参照）のとおり却下

### 管理操作の防御方式

- **(a) localhost 限定（採用）**: サーバにログオンできる者 = OS レベルで統制された管理者、という Windows 運用の既存統制（ログオン監査・特権管理）に相乗りする。追加の資格情報管理が不要。**代償**: リモート管理は認証 opt-in まで不可（RDP 前提）、サーバ上でブラウザを使えない構成では GUI 管理が使えない——いずれも「帰結」に明記
- (b) 管理用パスワード（アプリ独自）: リモート管理が最初から可能になるが、独自資格情報の保管・失念・共有という新しい管理対象を生む。AD 認証（決定 3）と重複投資になる。却下
- (c) 防御なし（ドキュメントで注意喚起のみ）: 「読んだ人しか守られない」であり、opt-out 設計の思想（ADR-0002）と矛盾。却下

### サービスの実行アカウント

- **(a) 仮想サービスアカウント既定（採用）**: 追加のアカウント管理なしで LocalSystem より大幅に権限を絞れる
- (b) LocalSystem: インストールは最も簡単だが、UI・受信・解析という外部入力を扱うプロセスに OS 全権を与えることになる。単一プロセス構成（ADR-0002）では UI の脆弱性がそのまま OS 全権に届くため却下
- (c) 利用者作成のカスタムアカウント必須: 権限設計の自由度は最大だが、初回セットアップの手数が増える。opt-in（gMSA 等）として残し、既定にはしない

## 帰結

- 良くなること: 「認証なしでも管理操作は守られる」既定になり、導入の簡単さと最低限の防御が両立する。監査・審査で問われる項目（権限・保存データ・証跡）に文書で答えられる
- 悪くなること:
  - **v0.x 期間中、管理操作は常にサーバ上での操作（RDP 等）が前提になる**（認証 + HTTPS の opt-in 提供で解消。管理操作の想定頻度の目安は全体設計書で示す）
  - Server Core 等、サーバ上でブラウザを使えない構成では GUI 管理が使えない（設定ファイル手編集または認証 opt-in 後のリモート管理で代替）
  - リバースプロキシ前置構成は既定のままではサポートされない（認証 opt-in が必須）
  - ACL・DPAPI・監査記録の実装コストが v0.1 から掛かる。DPAPI の machine スコープ採用により、サーバ移行時は資格情報の再入力が必要
- 受け入れるリスク: 既定構成では閲覧が認証なしであり、信頼ネットワーク内の全員がログを読める（公開範囲の localhost 縮小で緩和可能）。ログに機微情報が含まれる環境では AD 認証の有効化が必要である旨を利用者向けドキュメントで明示する

## 先送りにする場合の再評価トリガ

- 決定 2 の「閲覧は認証なし」既定: 利用者から「閲覧にも認証が欲しい」という要望が反復した場合（目安: 独立した利用者から 3 件以上）、または実環境でのインシデント報告があった場合、既定の向きを再評価する
- 決定 2 の loopback 束縛が実機確認で成立しない場合（受け入れ条件のとおり）、管理操作の防御方式自体を本 ADR の amendment または supersession で再決定する
- 決定 3 の提供時期: v1.0 公開基準（ADR-0006）の起案時に本 ADR の決定 3 と整合を取り、乖離があれば amendment する
