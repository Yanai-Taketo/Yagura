# ADR-0008: フォワーダ配布キットの動的生成(管理 UI)

- 状態: accepted
- 日付: 2026-07-06
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md)(スコープ)/ [ADR-0004](0004-security-model.md)(管理面の分離・監査・ACL)/ PR #99・#101(静的キットと実機検証の証跡)/ [利用者ガイド: Windows イベントログを Yagura へ転送する](../guides/forward-windows-eventlog.md)/ Issue #130(改訂履歴 2 の契機——管理 UI からの MSI アップロード)

## 文脈と課題

Windows 端末のイベントログを Yagura へ転送する手段として、Fluent Bit(Apache-2.0)の
プリコンフィグ配布キット(サイレント導入対応)を提供している(PR #99・#101。
Windows Server 2025 実機で MSI 無人導入 → サービス登録・遅延自動起動 → OS 再起動後の
転送再開・送信フレームの実測までの全経路を検証済み)。

静的キットの運用では、配布する管理者が **Yagura サーバの宛先(IP / ホスト名・ポート)を
自分で指定**する必要がある。企業配布基盤(Intune / SCCM / GPO)で数十〜数百台へ push する
場面では、この「宛先指定」が唯一の手作業であり、誤記した場合は**全端末が誤った宛先へ
送信し続ける**(UDP のため送信側にエラーも出ない)という失敗様式を持つ。

Yagura 管理 UI が「自機を指す設定を焼き込んだキット」を生成できれば、**手入力起因の**
宛先誤記を減らせる。ただしこれは失敗様式の**軽減であって除去ではない**——宛先候補から
どれを選ぶかの判断責任は管理者に残り、候補検出の誤りという新しい失敗様式も持ち込みうる
(「決定」の設計条件で対処)。また、配布後の運用中に宛先へ到達できなくなる事態
(経路変更・ファイアウォール変更・Yagura 停止等)は本 ADR の範囲外であり、
転送の死活監視は別途必要である(委任事項参照)。

なお、キットが導入する Fluent Bit サービスの実行アカウント(LocalSystem)は
第三者ツール側の既定であり、Yagura 本体のサービス実行権限方針(ADR-0004)の
対象外である。本 ADR はこの前提を変えない。

## 検討した選択肢

### 選択肢 A: 静的キットのみ(現状維持)

- 利点: 追加実装ゼロ。Yagura 本体の攻撃面・保守面が増えない
- 欠点: 宛先の手入力が残る(上記の失敗様式)。ガイド文書への往復が必須で、
  「Windows 管理者が深く意識せず導入できる」という体験目標に届かない
- 補足: 静的キットの試用フィードバックはまだ無く、「手入力誤記による導入失敗」の
  実測例も無い。それでも B へ進む判断は、実害の実測ではなく製品の体験目標
  (オーナー方針。v0.2 スコープ決定 2026-07-07)に基づくものであることを明記しておく

### 選択肢 B: 管理 UI での動的生成(採用案)

管理 UI(TCP 8515・loopback 固定 = ADR-0004)に「フォワーダキット生成」画面を追加する。
管理者は宛先・ポート・収集チャネルを指定し、置換済み設定一式を ZIP でダウンロードする。

- 利点: 手入力起因の宛先誤記を軽減。管理 UI 上の操作 1 回で「宛先設定込みの配布物」が
  得られる(**MSI の取得と配布基盤へのパッケージ化は引き続き別途必要**——全自動には
  ならないことを誇張しない)
- 欠点(コストの内訳):
  1. 宛先候補検出(NIC 列挙・除外・表示)の実装とテスト
  2. ZIP 生成(テンプレート置換・梱包)の実装とテスト
  3. テンプレートをビルド時に埋め込む工程がビルドパイプラインに増える
  4. Fluent Bit の版追従時、静的キットと生成 UI の**両方**を同時に更新する変更半径になる
     (テンプレート単一ソース化で緩和するが、検証責任は残る)
- Fluent Bit の MSI 本体は**既定では同梱しない**(取得手順の案内に留める。静的キットと
  同じ判断)。ただし管理者が明示的に配置した MSI に限り**オプトインで同梱できる**
  (2026-07-07 改訂。設計条件 9 と非同梱の判断根拠は改訂履歴 1 に一本化)

### 選択肢 C: 閲覧 UI(TCP 8514)での生成

- 却下: 閲覧 UI は認証なしで LAN に公開される(ADR-0004)。キット生成は読み取り専用操作
  ではあるが、「サーバの構成を反映した導入物の作成」は管理領域の操作であり、
  パラメータ付き生成エンドポイントを非認証面に増やすことは防御既定(閲覧面は読み取りのみ)
  と整合しない

## 決定

選択肢 B を採用する。管理 UI に「フォワーダキット生成」画面を追加し、
自機を指す設定入りの Fluent Bit 配布キット(ZIP)を生成・ダウンロードできるようにする。
MSI は既定で非同梱とし、管理者が明示配置した場合に限りオプトイン同梱を許す(設計条件 9)。

### スコープ境界の原則(ADR-0001 との整理)

ADR-0001 は「Windows イベントログ転送エージェントの自作」をスコープ外とし、その理由を
「既存の実績あるツールの設定例を案内する方が利用者の利益になる」とした。本 ADR は
この境界線を次の原則で明文化する:

> **Yagura が生成・保守するのは、既存テンプレートの実行時パラメータ化(置換)と梱包までとする。
> 転送ロジック本体・エージェント実行体(バイナリ・プラグイン・スクリプトの新規ロジック)には
> 関与しない。**

キット生成はこの原則の範囲内(= 設定例の案内の発展形)であり、原則を超える提案
(例: 転送エージェントの改造・独自プラグイン)は従来どおりスコープ外である。
この原則は将来の類似判断(他ツールの設定生成等)にも同じ基準で適用する。
境界判定の目安: **生成物の構造・ロジック(分岐)が入力パラメータの値によって変わらないこと**
——値の置換に留まれば「パラメータ化」、条件によって処理内容を組み替えるなら「ロジック生成」
であり原則の外(クリス round-2 の指摘を受けた判定基準)。MSI のオプトイン同梱は、
管理者が配置したファイルを**そのまま梱包する**だけで生成物のロジックを変えないため、
この原則の範囲内である。

### 設計条件

1. **宛先候補の提示**: サーバの NIC から検出した候補の選択式 + 手入力。候補からは
   ループバック・リンクローカル(APIPA 含む)・無効化された NIC を除外し、
   各候補に NIC の説明名を併記する。**既定選択は置かず、管理者に明示的に選択させる**
   (候補提示は誤記の軽減策であり、到達可能性の判断責任は管理者に残ることを画面上でも示す)
2. **収集チャネルの既定**: 静的キットと同じ `System,Application`。Security チャネルは
   **初期オフのオプトイン**とし、有効化時は機微情報・量の注意を表示する
3. **生成 ZIP の内容要件**: 置換済み `fluent-bit-yagura.conf` + `install.ps1` + Lua フィルタ +
   README + 生成メタデータ(`GENERATED.txt`: 生成日時・宛先・チャネル・Yagura バージョン)。
   README には最低限「検証済み Fluent Bit 版」「MSI は別途取得(取得元 URL と SHA256 検証手順)」
   「MSI 上書き更新後は install.ps1 を再実行」「宛先(サーバ IP)変更時は再生成・再配布が必要」を
   含める(静的キットガイドの注意事項を生成物側でも欠落させない)
4. **テンプレートの単一ソース**: 生成テンプレートは `forwarder/fluent-bit/` 配下のファイルを
   正とし、ビルド時に埋め込む(コピーを別に持たない)。具体機構は全体設計書へ委任
5. **置換値検証の単一ソース**: 宛先・ポート・チャネルの検証制約は `install.ps1` の
   `ValidatePattern` 定義を正とし、サーバ側実装との同期を自動テストで固定する
   (設定ファイルへの任意文字列注入を作らない)
6. **監査**: キット生成操作は ADR-0004 決定 7 の管理操作監査の対象に含める
   (生成日時・宛先値・チャネルを記録する。MSI 同梱時は設計条件 9 の追加項目も記録)
7. **生成の閉域性**: 生成はテンプレート置換と ZIP 化のみで、外部ネットワークへアクセスしない。
   ZIP は**メモリ上で組み立て、ディスクに一時ファイルを書かない**(ADR-0004 決定 5 の
   一時保管領域の論点を発生させない。MSI 同梱時は配置済みファイルを読み取って封入するのみ)
8. **秘密情報なし**: 生成物に秘密情報を含めない(現行の転送は平文 UDP・認証なしであり、
   キットには宛先とチャネル名しか入らない)
9. **MSI のオプトイン同梱**(2026-07-07 改訂。非同梱を既定とする判断根拠は改訂履歴 1。
   配置経路の追加は 2026-07-09 改訂。判断根拠は改訂履歴 2):
   既定は非同梱を維持する。管理者が Fluent Bit の MSI を所定フォルダへ**配置**した
   場合に限り、生成画面に「MSI を同梱する」チェックボックスをオプトインで表示し、生成 ZIP に
   MSI を含められる(ZIP 単体で自己完結する配布物になる)。配置の経路は次の 2 つを認める:
   **(a) サーバのファイルシステムへの手動配置**(RDP・エクスプローラ等。従来からの経路)、
   **(b) 管理 UI 経由のアップロード**(2026-07-09 改訂で追加。詳細は本条件末尾の
   「配置経路(b): 管理 UI アップロード」を参照)。次を満たすこと:
   - **配置フォルダと権限**: 配置先は `%ProgramData%\Yagura\forwarder\`(データルート配下)。
     **既定 ACL は Administrators のみ書き込み可**とする(ADR-0004 決定 5 の ACL 方針に
     準拠——ここに書ける者は「全端末に配る MSI を差し替えられる者」であり、防御上最も価値の
     高い書き込み対象になるため)。**この既定 ACL 自体は緩めない**(Users / Authenticated
     Users への付与は行わない)。ただし経路 (b) を成立させるため、サービスアカウントに
     **本フォルダに限定した「変更」権限(フルコントロールは付与しない)** を例外として追加する
     ——これは元の「緩めるインストーラ・手順は提供しない」という無条件文からの後退であり、
     その是非とトレードオフは改訂履歴 2 で明示的に扱う。生成画面は**このパスを明示表示**し、
     フォルダに MSI が無い場合も「未検出であること + 配置先パス + 期待ファイル名」を
     案内する(「チェックボックスが出ない」だけの沈黙にしない)
   - **検出と版判定**: ファイル名パターン `fluent-bit-*-win64.msi` で検出し、版は **MSI の
     ProductVersion を優先**して判定する(ファイル名は補助。リネームで欺かれないため)。
     複数の MSI がフォルダにある場合は**既定でエラー停止**(生成を進めない)——安全側に倒し、
     管理者に単一化させる(自動選択の緩和は委任事項での再検討事項)。**経路 (b) はこの
     多重状態を構造的に作らない**(常に単一の正式ファイルへの新規配置・置換として扱う。
     詳細は後述)
   - **版整合**: 同梱 MSI の版が「検証済み版」と異なる場合は生成画面で警告する。警告時の同梱は
     **単なる表示に留めず二段階確認**(管理者が明示的に承認する操作)を要し、警告の有無と承認の
     事実を監査記録に構造化して残す(「言った・言わない」を作らない)。**経路 (b) では同じ
     二段階確認を配置(アップロード)時点にも前倒しで適用する**(後述)
   - **改ざんの事後確認(検知ではなく記録)**: 生成時にサーバが配置 MSI の SHA256 を計算し、
     `GENERATED.txt` と監査に記録する(生成画面にも表示)。これは配置**後**の同一性を管理者が
     事後確認するための記録であって、配置**前**に改ざんされた MSI を Yagura が検知する機構では
     ない。加えて、検証済み版に対応する公式配布の SHA256 を Yagura に静的に保持し、一致すれば
     「公式ハッシュと一致」・不一致なら明示警告する(自動照合の実現範囲は実装 PR で確定)。
     ZIP 全体の SHA256 も `GENERATED.txt` に併記し、配布経路での破損・改ざんを ZIP 単位でも
     確認できるようにする。**経路 (b) では配置(アップロード)確定の時点で同じ SHA256 計算・
     公式ハッシュ照合を実行し、結果を配置操作の監査記録にも残す**(生成時点まで結果が
     分からない従来経路より早いタイミングでフィードバックできる)
   - **識別性**: 生成 ZIP のファイル名に同梱有無を機械可読に含める(例 `..._with-msi.zip` /
     `..._no-msi.zip`)。開かずに棚卸しできるようにする。想定 ZIP サイズが大きくなる(20 MB 超)
     ことは生成前に画面で予告する
   - **責任の所在(免責)**: README・`GENERATED.txt` に「同梱 MSI について Yagura は取得元・
     真正性・脆弱性対応の責任を負わない。Yagura は管理者が配置したファイルを梱包し、その来歴を
     記録するのみ」を明記する(**経路 (b) でアップロードされた MSI も対象は同じ**——Yagura が
     アップロードを受け付けたという事実は、Yagura が MSI の真正性を保証することを意味しない)
   - **配置経路 (b): 管理 UI アップロード**(2026-07-09 改訂。改訂履歴 2 で追加。トレードオフの
     全文は改訂履歴 2 を参照):
     - **対象操作**: 新規配置・既存 MSI の置換・削除。テンプレート一式(`forwarder/fluent-bit/`)
       は対象外(設計条件 4 のとおりビルド時単一ソースのまま。本改訂が対象とするのは MSI のみ)
     - **経路制限**: 管理リスナ(TCP 8515・loopback 固定。ADR-0004 決定 2)限定。他の管理操作と
       同じ画面(フォワーダキット生成画面)からのみ到達できる。閲覧リスナ(8514)には一切公開しない
       (選択肢 C の却下理由と同じ原則)
     - **入力検証**: アップロードされたファイルの**クライアント指定ファイル名は信用しない**
       (パストラバーサル対策として、保存先パスの構成に使わない)。拡張子は `.msi` のみ許可し、
       サイズ上限(既定 100 MiB。実測に基づく確定値は委任事項 10)を超える要求は本文を読み切る
       前に拒否する。最終的な格納ファイル名は、検証済み MSI から読み取った ProductVersion を
       もとに Yagura が生成する(既存の検出パターン `fluent-bit-*-win64.msi` に一致させる)
     - **書き込み手順(設計条件 7「生成の閉域性」との整合)**: 設計条件 7 の「ディスクに一時
       ファイルを書かない」は**生成(ZIP 組み立て)処理の閉域性**を定めたものであり、本条件が
       新設する**配置(アップロード)処理には適用されない**——配置は元々ディスクへの永続書き込み
       を目的とする操作であり、両者は別の処理である(この区分を本改訂で明文化する)。ただし
       配置処理自体の閉域性は別途次のとおり定める: アップロード本文は `%TEMP%` 等の広い ACL の
       領域を経由せず、**配置フォルダ(`forwarder\`)内のステージング名**(例
       `.uploading-<GUID>.msi`。既存の検出パターンに一致しないため `Lookup()` から見えない)へ
       直接ストリーム書き込みする。書き込み完了後にファイルとして開けること・ProductVersion
       読み取り・SHA256 計算のすべてを通過した場合のみ、正式ファイル名へ**アトミックにリネーム**
       する。検証に失敗した場合はステージングファイルを削除し、正式ファイルは一切変更しない
       (中間状態を `Lookup()` に晒さない)
     - **単一化**: アップロードは常に「単一の正式ファイルへの新規配置または置換」として扱う
       (アップロード経路は複数 MSI 併存状態を構造的に作らない)。既存ファイルがある場合は
       旧 SHA256・新 SHA256 を並べて表示したうえで「置換する」の明示確認を必須とする
     - **SHA256 検証・二段階確認**: 検証済み版に対応する公式ハッシュと一致する場合は単一確認で
       配置できる。一致しない場合(未知の版・検証済み版と異なる版を含む)は、設計条件 9 の
       「版整合」と同じ**二段階確認**(型を統一し UI コンポーネントを共通化する)を要する
     - **削除**: 配置済み MSI の削除も同じ画面から行える(同梱運用をやめたい場合に、FS アクセス
       なしで完結させるため)。削除は配置と同じ重みの操作として扱い、削除前の SHA256 を監査に残す
     - **ACL の維持**: 配置フォルダの ACL 適用・維持は引き続きインストーラの責務とする
       (security.md §5.1 の既存整理を変えない)。本改訂が追加するのはサービスアカウントへの
       本フォルダ限定の書き込み権限のみであり、Administrators 以外の他プリンシパルへの権限拡大
       は行わない
     - **監査**: 配置(アップロード)・削除のそれぞれを 2000 番台の新規イベントとして記録する
       (採番候補 2006・2007。最終採番は委任事項 10 で security.md に固定)。記録内容:
       操作者の接続元・UTC 時刻・格納ファイル名・SHA256・サイズ・ProductVersion・
       officialHashMatch・(アップロード時)versionMismatchAcknowledged・(削除時)削除前 SHA256
   - 既定(非同梱)の全条件(README の取得手順・SHA256 検証手順の記載等)は同梱時も維持する

## 帰結

- 良くなること: 企業配布での導入手数が「MSI を置く → ZIP を展開して push」まで縮む。
  手入力起因の宛先誤記が減る。MSI オプトイン同梱で ZIP が自己完結し、MSI 別取得の一手間が消える
- **自己完結の効能の射程**: ZIP 単体で完結する利点が実質的に効くのは**手動・少数端末導入**の
  シナリオである。Intune / SCCM 等の企業配布基盤では最終的に `.intunewin` 等へ再パッケージ
  するため MSI 同梱の有無は自己完結体験にほぼ寄与しない(この点でオプトイン同梱の価値は
  限定的であり、既定を非同梱に据える理由の一つでもある)
- 悪くなること(受け入れるトレードオフ): 選択肢 B の欠点 1〜4(候補検出・ZIP 生成の
  実装テスト、ビルド工程追加、版追従の変更半径拡大)。「検証済み Fluent Bit 版」の表明を
  Yagura 側が保守する責任を持つ——版の検証を伴わない案内更新はしない(conventions.md の
  実体検証原則)。MSI 同梱を選ぶと、検証済み版更新のたびに同梱運用側は MSI 再配置 → 再生成 →
  再配布が必要になる(非同梱なら MSI 更新は生成 ZIP に影響しない。この簡便性の喪失は
  同梱を選ぶ側のトレードオフとして明記する)
- リスク:
  - Fluent Bit 側の設定仕様変更(5.x 系等)への追従が遅れると、生成キットが最新 MSI と
    組み合わせられない期間が生じうる。「検証済み版」の明記で緩和する
  - Fluent Bit の脆弱性(CVE)公表を検知する体制を持たない場合、脆弱な版を「検証済み」と
    表明し続けるリスクがある(検知体制の整理は委任事項)。MSI 同梱時はこのリスクが
    配布物に焼き込まれるため、免責の明記と来歴記録で管理者の判断を支援するに留める
  - ダウンロード済み ZIP はその時点のサーバ設定の焼き込みであり、サーバの IP 変更後も
    古い宛先のまま使い回されうる。失効機構は設けず(複雑さがスコープに見合わない)、
    README の再生成注意と `GENERATED.txt` の来歴で対処する
- 本 ADR は転送の**観測性を改善しない**: 配布した端末からの転送が止まったことの検知
  (端末別の最終受信確認等)は範囲外であり、ADR-0001 決定 4(損失の観測可能性)との
  接続は委任事項として別途扱う

## 先送りにする場合の再評価トリガ

本 ADR が却下(選択肢 A 維持)となった場合、以下のいずれかで再評価する:

- 静的キットの試用フィードバックで宛先設定に起因する導入失敗が報告されたとき
- ネイティブ・エージェントレス収集(リモートイベントログ pull)の ADR を起案するとき

- Fluent Bit に重大な脆弱性が公表され、「検証済み版」の更新体制の必要性が顕在化したとき

   ### なぜ設計条件 9 の例外を設けるのか(利便性 vs 防御価値のトレードオフ)

   配置フォルダの既定 ACL(Administrators のみ書き込み可)は、「全端末に配る MSI を差し替え
   られる者」を OS レベルの管理者に限定する、本 ADR で最も防御価値の高い制約である。管理 UI
   からのアップロードを許すことは、この制約に**アプリ経由の書き込み口を新設する**ことを意味する
   ——管理リスナは loopback 固定(ADR-0004 決定 2)だが、v0.1 時点では認証 opt-in 前であり、
   「loopback へ到達できる者」は必ずしも Windows の Administrators グループに属するとは限らない
   (サーバへ RDP・コンソールログオンできる非管理者アカウントも含みうる)。**これは元の設計条件 9
   の「緩めるインストーラ・手順は提供しない」という無条件文に対する後退であり、両立させる話では
   なく、明示的なトレードオフとして扱う**:

   - **得るもの(利便性)**: サーバのファイルシステムアクセス(RDP・共有フォルダ経由のファイル
     コピー等)なしに、管理 UI の操作だけで MSI 同梱運用が完結する。Server Core 等 GUI ログオン
     を統制で禁止する環境でも(管理リスナへ到達できる別クライアントから)配置できる。
   - **失うもの(防御価値)**: 「MSI を差し替えられる者 = 真に Windows Administrator である者」
     という OS レベルの身元保証が、「管理リスナへ到達できる者」という ADR-0004 の一般的な管理
     操作の信頼境界まで引き下げられる。これは他の管理操作(設定変更・DB 切替等)と同じ信頼水準
     へ揃える動きであり、設計条件 9 が当初あえてこれより厳しい境界を敷いていた判断(供給網の
     severity の高さゆえの特別扱い)を部分的に手放す。
   - **埋め合わせの設計(補完統制)**: 上記の後退を無条件に許容するのではなく、次の統制の総体で
     リスクを縮小する。ADR-0004 決定 7 の「軽減されるが消えない」という限界明示の流儀に倣い、
     これらの統制は**リスクをゼロにはしない**ことを明記する:
     - 経路制限: 管理リスナ(8515・loopback)限定。閲覧リスナには一切公開しない
     - 入力検証: クライアント指定ファイル名を信用しない・拡張子/サイズ上限を強制・格納ファイル名
       は Yagura が MSI の ProductVersion から生成する
     - 二段階確認: 公式ハッシュと一致しない MSI(未知の版を含む)は、単なる警告表示に留めず
       管理者の明示的な承認操作を要求する(設計条件 9 の版不一致確認と統一)
     - SHA256 検証: 配置(アップロード)確定時点で計算・公式ハッシュ照合し、結果を監査に残す
       (生成時点まで待たない早期フィードバック)
     - 単一化: アップロードは常に単一の正式ファイルへの新規配置・置換として扱い、多重 MSI 状態を
       構造的に作らない
     - 監査: 配置・削除をそれぞれ新規イベントとして記録し、「誰が・いつ・何を・結果はどうだったか」
       を証跡に残す(§4.3 の枠組みに準拠)
     - ACL の維持: 追加するのはサービスアカウントへの本フォルダ限定の書き込み権限のみであり、
       Administrators 以外の他プリンシパルへの拡大は行わない。フォルダ ACL の適用・維持は
       引き続きインストーラの責務のまま変えない
   - **結論**: 「軽減されるが消えない」——真の OS Administrator であることの身元保証は失われる
     が、経路の限定・入力検証・二段階確認・監査の総体で、悪用の障壁と事後追跡可能性を確保する。
     この結論の妥当性(埋め合わせが十分か)は、供給網統治の観点から田中の、決定の強度の観点から
     クリスのペルソナレビューで特に精査を求める論点として明示しておく。

   ### 設計条件 7(生成の閉域性)との整合

   設計条件 7 は「生成(ZIP 組み立て)処理はディスクに一時ファイルを書かない」ことを定めたもので
   あり、**配置(MSI の永続書き込み)処理はそもそも射程に含まれない**——配置は最初から永続化を
   目的とする操作であり、生成処理とは別工程である。この区分は改訂履歴 1 の時点でも暗黙に成立して
   いた(手動配置は生成処理の外で行われるため)が、本改訂で初めてアプリが配置処理そのものを
   実装するため、区分を本文に明文化した。**配置処理自体の閉域性**(アップロード本文が `%TEMP%`
   等の広い ACL の領域を経由しないこと)は別途、配置フォルダ内でのステージング→アトミック
   リネームという形で満たす(本文参照)。

   ### 却下した代替案

   - **現状維持(経路 (a) のみ)**: Issue #130 が指摘する「UI の外に残る唯一の手作業」を解消
     できない。却下(ただし経路 (a) 自体は廃止しない。(b) は追加の選択肢)。
   - **サーバ側ヘルパー経由の間接書き込み**(タスクスケジューラ等で Administrators 権限の

     (b) ADR-0004 の管理操作の信頼境界(loopback 到達 = 管理操作可)は他の管理操作(設定変更・

     一貫性の説明が弱い、という 2 点で見送った。将来、経路 (b) の悪用が実測された場合の代替案
     として記録に残す。
   - **閲覧 UI(8514)からのアップロード**: 選択肢 C の却下理由(閲覧面は読み取り専用)と同じ理由
     で却下。検討の俎上にも載せていない。
   - **公式ハッシュ不一致を無条件でブロック(自動可否判定のみ・二段階確認なし)**: 検証済み版以外
     の MSI(将来版の先行検証運用等、正当な理由で異なる版を配置したい場合)を一律に拒むと利便性を
     過度に損なう。設計条件 9 の既存方針(版不一致は警告 + 二段階確認)と一貫させ、アップロードも
     同じ扱いとした。
   - **アップロード機能を認証 opt-in(AD 認証)の提供まで待つ**: 認証があれば「loopback 到達 =
     真の Administrator」に近い保証を回復できるが、AD 認証の提供は v1.0 まで(ADR-0004 決定 3)
     であり、それまで経路 (b) を封じる根拠として待つのは v0.2 スコープのオーナー判断(Issue #130)
     に沿わない。認証 opt-in が先に実装された場合、経路 (b) の到達条件に認証済み・管理役割を
     追加することは自然な強化であり、委任事項 10 の実装 PR での検討事項として残す。

   ### なぜ supersession でなく amendment か

   README.md の運用ルールは「既定を反転させるなら supersession、既定を保ったまま例外を足す
   だけなら amendment」とし、疑わしい場合は最小の単位で反転の有無を検証するよう求めている
   (改訂履歴 1 の自己吟味と同じ基準)。本改訂を複数の粒度で検証する:
   - **最も大きい粒度(MSI 同梱の既定)**: 「既定は非同梱」は不変。反転なし。
   - **中間の粒度(配置の主体)**: 「配置は管理者の意図的な操作によってのみ行われる」も不変
     ——経路が増えても、無人・自動での配置は導入していない。反転なし。
   - **最も細かい粒度(配置に必要な身元保証の水準)**: ここは**反転している**。従来は「OS レベルの
     Administrators グループに属する者のみが書き込める」という無条件の保証があったが、本改訂は
     これを「管理リスナへ到達できる者」という(v0.1 時点では認証を伴わない)より広い境界まで
     引き下げる、条件付きの経路を追加する。この単位で見れば、設計条件 9 の「緩めるインストーラ・
     手順は提供しない」という文は文字どおりには成立しなくなる。
   - **それでも amendment と判断する理由**: 反転しているのは「配置の身元保証の水準」という
     実装手段の性質であり、**ADR-0008 の中核決定(選択肢 B の採用・既定非同梱・供給網統治の核)
     はいずれも不変**である。加えて、この後退は無条件の緩和ではなく、経路制限・二段階確認・
     SHA256 検証・監査という補完統制を伴う**条件付きの例外**として追加している(改訂履歴 1 が
     「既定を保ったまま例外を足すだけ」と整理した構造そのものを踏襲している)。**この判断は
     境界事例であることを明示する**——「防御上最も価値の高い書き込み対象」という設計条件 9 の
     核心的な文言に触れる変更であるため、疑わしければ supersession に倒すという原則に照らし、
     ペルソナレビュー(特にクリス・田中)でこの amendment / supersession の判定自体への異論を
     明示的に募る。**レビューで「反転の実質は既定の反転に相当する」との指摘が採用された場合、
     本改訂は supersession として起こし直す**(起案側は判定を最終としない)。

   ### 効果測定と再評価

   委任事項 11 の効果測定で、経路 (b) の追加が実際に導入手数・所要時間を短縮したと確認できない
   場合、または経路 (b) を通じた不正・誤配置の実例が観測された場合は、本 amendment(経路 (b))
   の縮小・撤回を含めて再評価する。「進める」の判断は測定結果でフィードバックされうる暫定判断
   である(改訂履歴 1 と同じ流儀)。

   ### 申し送り

   改訂履歴 1 の申し送り(設計条件 9 の実装細部を委任事項 7 の実装完了時に security.md へ
   一本化する)は本改訂でさらに厚みが増した。委任事項 10 の実装完了時には、設計条件 9 全体
   (経路 (a)(b) 双方の実装細部)をまとめて security.md 側へ移し、本文は方針レベル(配置経路の
   種類・ACL 例外の存在・補完統制の種類)まで圧縮する整理を行う。
