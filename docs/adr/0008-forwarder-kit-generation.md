# ADR-0008: フォワーダ配布キットの動的生成(管理 UI)

- 状態: proposed
- 日付: 2026-07-06
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md)(スコープ)/ [ADR-0004](0004-security-model.md)(管理面の分離)/ PR #99(静的キット)/ [利用者ガイド: Windows イベントログを Yagura へ転送する](../guides/forward-windows-eventlog.md)

## 文脈と課題

Windows 端末のイベントログを Yagura へ転送する手段として、Fluent Bit(Apache-2.0)の
プリコンフィグ配布キット(サイレント導入対応)を提供している(PR #99。
Windows Server 2025 実機で MSI 無人導入 → サービス登録・遅延自動起動 → OS 再起動後の
転送再開までの全経路を検証済み)。

静的キットの運用では、配布する管理者が **Yagura サーバの宛先(IP / ホスト名・ポート)を
自分で指定**する必要がある。企業配布基盤(Intune / SCCM / GPO)で数十〜数百台へ push する
場面では、この「宛先指定」が唯一の手作業であり、誤記した場合は**全端末が誤った宛先へ
送信し続ける**(UDP のため送信側にエラーも出ない)という失敗様式を持つ。

Yagura 管理 UI が「**自機を指す設定を焼き込んだキット**」を生成できれば、この手作業と
失敗様式を除去できる。ADR-0001 はスコープ外として「Windows イベントログ転送エージェントの
自作」を挙げるが、その却下理由は「既存の実績あるツールの設定例を案内する方が利用者の
利益になる」であり、本件は**エージェント自作ではなく「設定例の案内」を生成へ発展させる**
ものである。この整理の妥当性自体もレビュー対象とする。

## 検討した選択肢

### 選択肢 A: 静的キットのみ(現状維持)

- 利点: 追加実装ゼロ。Yagura 本体の攻撃面・保守面が増えない
- 欠点: 宛先の手入力が残る(上記の失敗様式)。ガイド文書への往復が必須で、
  「Windows 管理者が深く意識せず導入できる」という体験目標に届かない

### 選択肢 B: 管理 UI での動的生成(採用案)

管理 UI(TCP 8515・loopback 固定 = ADR-0004)に「フォワーダキット生成」画面を追加する。

- 管理者は宛先アドレス(サーバの NIC から検出した候補の選択式 + 手入力)・ポート・
  収集チャネルを指定 → 置換済み `fluent-bit-yagura.conf` + `install.ps1` + Lua フィルタ +
  README を **ZIP でダウンロード**
- Fluent Bit の MSI 本体は同梱しない(Apache-2.0 のため同梱は合法だが、①生成 ZIP が
  20 MB 超になる ②Yagura リリースと Fluent Bit の版更新が結合する ③改ざん・脆弱性対応の
  責任境界が曖昧になる——ため取得 URL の案内に留める。静的キットと同じ判断)
- 利点: 宛先誤記の失敗様式を除去。管理 UI 上の操作 1 回で配布物が完成する
- 欠点: 管理 UI に生成機能の保守が乗る。テンプレートと静的キット(forwarder/fluent-bit/)の
  二重管理になる——**生成テンプレートは静的キットのファイルを正とし、ビルド時に埋め込む**
  (コピーを別に持たない)ことを実装条件とする

### 選択肢 C: 閲覧 UI(TCP 8514)での生成

- 却下: 閲覧 UI は認証なしで LAN に公開される(ADR-0004)。キット生成は読み取り専用操作
  ではあるが、「サーバの構成を反映した導入物の作成」は管理領域の操作であり、
  パラメータ付き生成エンドポイントを非認証面に増やすことは防御既定(閲覧面は読み取りのみ)
  と整合しない

## 決定

(proposed)選択肢 B を採用する。管理 UI に「フォワーダキット生成」画面を追加し、
自機を指す設定入りの Fluent Bit 配布キット(ZIP)を生成・ダウンロードできるようにする。

セキュリティ上の設計条件:

- 生成物に秘密情報を含めない(現行の転送は平文 UDP・認証なしであり、キットには宛先と
  チャネル名しか入らない)
- 置換値(宛先・ポート・チャネル)はサーバ側で検証してから埋め込む(`install.ps1` の
  `ValidatePattern` と同等の制約。設定ファイルへの任意文字列注入を作らない)
- 生成はテンプレート置換と ZIP 化のみで、外部ネットワークへのアクセスを行わない

## 帰結

- 良くなること: 企業配布での導入手数が「MSI を置く → ZIP を展開して push」まで縮む。
  宛先誤記による無音の送信先間違いがなくなる
- 悪くなること(受け入れるトレードオフ): 管理 UI の実装・テスト面積が増える。
  生成キットの「検証済み Fluent Bit 版」の表明(現在 4.0.14)を Yagura 側が保守する
  責任を持つ——版の検証を伴わない案内更新はしない(conventions.md の実体検証原則)
- リスク: Fluent Bit 側の設定仕様変更(5.x 系等)への追従が遅れると、生成キットが
  最新 MSI と組み合わせられない期間が生じうる。ガイドに「検証済み版」を明記し続ける
  ことで緩和する

## 先送りにする場合の再評価トリガ

本 ADR が却下(A 維持)となった場合、以下のいずれかで再評価する:

- 静的キットの試用フィードバックで宛先設定に起因する導入失敗が報告されたとき
- ネイティブ・エージェントレス収集(リモートイベントログ pull)の ADR を起案するとき
  (フォワーダ戦略全体を再整理する機会となるため)
