# ADR-0019: TLS 受信の証明書選択の UI 設定化（ADR-0012 の再利用）

- 状態: proposed
- 日付: 2026-07-18
- 決定者: YANAI Taketo
- 関連: [ADR-0012](0012-admin-https-cert-ui.md)（管理リモート HTTPS の証明書選択 UI——本 ADR の再利用元）/ [security.md](../design/security.md) §6（TLS 受信）/ [configuration.md](../design/configuration.md) §8（`Ingestion:Tls:*`）/ CF-4（Issue #262。リスナ再構成）/ Issue #276 / PR #341

## 文脈と課題

TLS 受信（RFC 5425。opt-in。Issue #137 で実装済み）の証明書は、現状 **`yagura.json` の `Ingestion:Tls:CertificateThumbprint` を手編集する以外に設定手段がない**。管理リモート HTTPS 側は ADR-0012 で証明書ストア列挙・選択 UI・保存前 fail-closed 検証が実装済みであり、**「異常は知らせる（期限接近 1017・使用不能 1018 の能動通知は実装済み）が、UI から直せない」**という非対称が残っている。

本件は ADR-0012 のスコープから分離済み（オーナー裁定 2026-07-11・論点 4）で、起案予定への登録済み（Issue #276）。**主論点は「ADR-0012 で確立した設計をどこまで再利用できるか」**——新しい設計判断を増やさず、確立済みの決定への差分だけを書く。

### スコープ（非目標）

- 証明書の取得・生成（ACME・自己署名生成等）は扱わない（ADR-0012 と同じ）
- TLS 受信リスナの無瞬断再構成は扱わない（CF-4 層2 が「TLS 受信キーは対象外——宣言どおり再起動」と確定済み。本 ADR はその宣言を変えない——再評価トリガ参照）
- クライアント証明書（相互 TLS）は扱わない（意図的先送りのまま）

## 検討した選択肢

### ADR-0012 設計の再利用範囲

- **(a) 全面再利用（採用）**: 列挙契約（serverAuth EKU + 秘密鍵あり・最小フィールド・read-only）・選択 UI・保存前 fail-closed 検証・「UI 事前検証と起動時検証の乖離ゼロ」・秘密鍵の読取検証 + 誘導（付与は試みない——lab 実測 2026-07-11 で `NT SERVICE\Yagura` は `WRITE_DAC` 不可と確定済み）・テスト戦略（Abstractions 契約 + 偽実装）を**そのまま流用**し、対象キーと起動時検証経路（1016〜1018）だけ差し替える
- (b) TLS 受信専用の簡易版（拇印手貼り欄のみ UI 化）: 実装は薄いが、ADR-0012 が撤廃した拇印手貼りを新設面で復活させることになり、設計の一貫性を壊す。**却下**
- (c) ADR-0012 の画面に統合（1 画面で管理 HTTPS と TLS 受信の両方）: **反映方式は両者同一である**（管理 HTTPS 4 キーも `RestartRequired` = サービス再起動 = 全受信断——ADR-0012 決定 6 が自ら明記。ここを差分と書いた当初案はペルソナレビュー——クリス——で事実誤認と指摘され訂正した）。実際に異なるのは、①**稼働中の失敗時挙動**（管理 HTTPS: 期限切れでハンドシェイク拒否・復旧は loopback へ逃がせる / TLS 受信: 提示を継続し送信元がサイレント脱落し得る）②それに由来する**期限切れ証明書の保存可否ポリシー**（決定 2）③**管理側のみに存在する認証・RemoteBinding との fail-closed 組み合わせ検証**、の 3 点であり、これらの説明・検証条件が 1 画面で混線する。**却下**（画面は分け、列挙・検証の実装は共有する。バッチ適用は決定 4 のとおり画面分割でも成立する）

## 決定

### 1. TLS 受信の証明書設定画面を管理面に追加する（ADR-0012 決定 1・2 の流用）

- 対象キー: `Ingestion:Tls:Enabled`・`Ingestion:Tls:Port`・`Ingestion:Tls:CertificateThumbprint`（**すべて configuration.md §8 の確定済み表に既出——本 ADR は設定キーを 1 つも追加しない**。ADR-0012 決定 6 と同じ低リスク主論拠）。**`Ingestion:Tls:BindAddress` は意図的に対象外とする**（受信系の bind 先 UI は UDP/TCP 受信の BindAddress も含めて未対応であり、bind 先の UI 化は本 ADR のスコープ外——手編集のまま残す）
- 証明書選択は ADR-0012 決定 2 の列挙 UI を流用する（serverAuth EKU + 秘密鍵あり・最小フィールド・空一覧の案内文）。ただし**期限切れ証明書は一覧から除外せず、警告表示のうえ選択可能とする**（ADR-0012 の「除外または警告」の選択を TLS 受信側では警告側に固定——決定 2 の「期限切れは警告付きで保存を通す」と整合させる。管理 HTTPS 画面と挙動が分かれてよい）。**列挙契約・実装は管理 HTTPS 側と共有し、二重実装しない**
- 画面は管理 HTTPS 設定とは別画面とし（選択肢 (c) の却下理由）、相互リンクを置く。**両画面の見出し・説明文で用途を平易に書き分ける**（「syslog 受信〔送信元機器 → Yagura〕用」/「管理画面の閲覧〔ブラウザ → Yagura〕用」——同じ証明書一覧が出る 2 画面の取り違えを防ぐ）

### 2. 保存前 fail-closed 検証と起動時検証の乖離ゼロ（ADR-0012 決定 4 の流用）

- 保存前検証の対象を**拒否する項目と警告して通す項目に分けて宣言する**（同じ列挙に混ぜない——実装 PR で拒否側へ倒される事故を防ぐ）:
  - **拒否**: 「`Enabled=true` なのに拇印未設定」「証明書が見つからない」「秘密鍵にアクセスできない」「serverAuth EKU 不適合」（EKU は管理側と同じ拒否で確定。手編集で現に動いている非 serverAuth 証明書の構成を UI から再保存できなくなる副作用は受け入れ、帰結に記す）
  - **警告して通す**: 「期限切れ」
- **検証は起動時の TLS 証明書解決へ寄せる**: 共有単位は `AdminCertificateProvider.Load` であり、起動時 1016 経路（Program.cs）・周期監視（`StoreIngestionTlsCertificateStatusProbe`）・本 ADR の保存前検証の 3 者がこれを呼ぶ。「UI 事前検証が緑なら起動後に想定外の縮小継続（1016）が起きない」ことを回帰テストで固定する（D-6 と同型）
- **期限切れの扱いは既存の設計判断に従う**: TLS 受信は期限切れでも受信を止めない（security.md §6——送信側の検証ポリシー次第で接続が拒否されるのは送信側の判断）。したがって UI は期限切れ証明書の保存を**警告付きで通す**。**管理 HTTPS 側（期限切れは拒否側）と挙動が異なることは確定事項**であり、その差分と理由を画面文言で説明する（条件文にしない——security.md §6 が非対称を確定済み。差分説明の表示は受け入れ基準に含める）

### 3. 秘密鍵は読取検証 + 誘導に留める（ADR-0012 決定 3 の流用）

- UI は付与を試みず、サービスアカウントが秘密鍵を読めるかを検証・表示し、読めない場合は `certlm.msc`（CF-D2）手動付与へ誘導する。起動時の best-effort 付与（監査 2010）は既存のまま
- lab 実測（2026-07-11。PR #244）の縮退確定を TLS 受信側にもそのまま適用する——実測の変数（実行主体 `NT SERVICE\Yagura`・ストア `LocalMachine\My`・CNG 鍵の既定 DACL）はどれも変わらないため再検証は不要。**この推論は TLS 受信の設定 UI が管理 HTTPS UI と同一プロセス・同一アカウントで動く限りで成立する**（将来 UI の実行主体を分ける変更があれば前提が崩れる）

### 4. 反映方式は再起動（受信断）の明示とバッチ適用（ADR-0012 決定 6 の流用）

- `Ingestion:Tls:*` は §8 宣言どおり **RestartRequired**（TLS リスナ構築は起動時処理。CF-4 層2 の対象外宣言を維持）。**反映方式は管理 HTTPS 側と同一**（あちらも再起動 = 全受信断——選択肢 (c) の訂正参照）
- UI には「保存だけでは反映されない・反映は次回サービス再起動時・その間 TLS だけでなく全受信が停止する」ことを、**推定断時間・想定ロス量（直近受信レートからの概算——ADR-0012 決定 6 と同じ 3 点セット）**・再起動待ちキーの常設表示（#286）への合流とあわせて明示する
- 保存された未反映変更は既存の再起動待ちキー機構（1020・/admin 常設カード）にそのまま乗る——新しい表示機構は作らない。**この累積機構により、管理 HTTPS 画面と本画面の変更をまとめて 1 回の再起動で反映できる**（同一証明書を両キーへ流用する更新——security.md §6 が認める形態——も 2 画面 1 再起動で完結する。「1 変更 = 1 再起動」を誘発しない UX——ADR-0012 決定 6——は画面分割でも成立する）
- 保存が再起動待ちカードへ計上される経路（#287 の「保存後に再読み込みを自動で呼ぶ」チェックボックスとの整合を含む）は委任 2 で確定する
- **拇印固定参照は意図的な確定**とする（サブジェクト等による更新後証明書への自動追従はしない——選択の明示性を優先。証明書更新のたびに UI 再選択 + 再起動が要る運用負担は帰結と再評価トリガで扱う）

### 5. 監査（ADR-0012 決定 7 の流用）

- **決定 1 の対象 3 キー（`Enabled`・`Port`・`CertificateThumbprint`）の変更**を、独立イベント ID で記録する（2000 番台 additive。採番は実装 PR——security.md §4.3 SEC-5 のレール。管理 HTTPS 側の 2012 と対になる TLS 受信版。`Port` は到達面の変更でありむしろ監査価値が高い——2012 も 3 キーすべてを対象としている）
- 記録内容は 2012 と同型: **変更キーと新値（拇印は公開識別子であり値を残す）+ 操作者（2011/2012 と同じ命名空間付き利用者名・認証方式）**

### 5b. 観測性への導線（ADR-0012 委任の引き取り）

- ADR-0012 の委任・未解決が本 ADR へ送った「TLS 受信面の証明書健全性の可視化」を引き取る: **証明書設定画面から、TLS ハンドシェイク失敗の計器（`RecordTlsHandshakeFailure` 系）と送信元別受信状況（無音化検出一覧）への導線（リンク）を置く**。証明書差し替え + 再起動の直後に運用者が行う「全送信元が再接続できたかの確認」を、設定画面から 1 クリックで始められるようにする（新しい観測機構は作らない——既存計器・既存一覧への導線のみ）

### 6. 残存リスクの明記（ADR-0012 決定 8 の流用）

- 既定無認証 loopback で列挙 API が到達可能である点は ADR-0012 決定 8 の記述がそのまま当てはまる。ただし射程を正確に書く: **増えないのは列挙（読み出し）面**（同一 API の第 2 の呼び出し画面が増えるだけ）。一方で**書き込み面は増える**——従来 `Ingestion:Tls:*` の変更にはファイル ACL を突破する手編集が必要だったが、本 ADR 後は既定無認証 loopback 到達者（ADR-0010 が脅威モデルに明記した任意のローカル実行主体）が TLS 受信の無効化・ポート変更・証明書差し替えを保存できる。反映は再起動時であり独立監査 ID（決定 5）が残り、loopback 認証 opt-in で守れる——ADR-0012 の設定 API と同一の緩和構造に収まるが、TLS 受信の事故モードは「送信元のサイレント脱落」であり気づかれにくい面である点を含めて利用者向けドキュメントに追記する

## 受け入れ基準

- 通常運用で拇印の手入力・JSON 手編集なしに TLS 受信証明書を選択・保存できる（`BindAddress` は対象外——決定 1）
- UI 事前検証が緑なら、再起動後に想定外の縮小継続（1016）が起きない（回帰テストで固定）
- 期限切れ証明書は一覧で警告表示のうえ選択可能で、保存は警告付きで通り、起動後も受信が止まらない（security.md §6 との一致）。**管理 HTTPS 側との挙動差分の説明が画面に表示される**
- EKU 不適合は保存前に拒否される（管理側と同一）
- 保存時に受信断の予告（再起動が必要・全受信が止まる・**推定断時間・想定ロス量の概算**）が表示され、未反映変更が再起動待ちカードに現れる。**管理 HTTPS 画面と本画面の変更が 1 回の再起動でまとめて反映できる**
- 列挙・検証の実装が管理 HTTPS 側と共有されている（共有単位 = `AdminCertificateProvider.Load`。二重実装がないことをレビューで確認）
- 変更の監査記録（新 ID・対象 3 キー・新値・**操作者**を含む）が残る
- 証明書設定画面からハンドシェイク失敗計器・送信元別受信状況への導線がある（決定 5b）

## 帰結

- 良くなること: 「異常は知らせるが UI から直せない」非対称が解消し、TLS 受信の導入・証明書更新が管理 HTTPS と同じ操作感で完結する。ADR-0012 の設計・実装・lab 実測を再利用するため、新規の設計リスクがほぼない
- 悪くなること・受け入れるリスク:
  - 反映は再起動 = 受信断のまま（CF-4 層2 の宣言を維持）。証明書の定期更新のたびに計画受信断 + UI 再選択（拇印固定参照——決定 4）が要る——緩和は再評価トリガへ
  - EKU 不適合を拒否に固定したため、手編集で現に動いている非 serverAuth 証明書の構成は UI から再保存できない（手編集は引き続き可能。UI が厳しい側 = D-6 の方向を優先）
  - 画面が 1 つ増える（管理 HTTPS と別画面）。統合しない理由（失敗時挙動・保存ポリシー・検証条件の差）は検討した選択肢に記録済み

## 先送りにする場合の再評価トリガ

- **証明書更新の受信断なし反映**: 利用者報告を待たない**能動条件**で再評価する（運用者は計画受信断を報告せず夜間再起動のスクリプト化で黙って自己解決しがちであり、報告駆動では信号が届かない——ペルソナレビュー クリスの指摘）。トリガは次のいずれか早い方: **(a) 有効期間の短い証明書（期限接近警告 1017 が年 2 回以上発火する水準——概ね 180 日未満）が TLS 受信に設定される事象の検出**、**(b) 公的 CA の有効期間短縮スケジュール（CA/Browser Forum の段階的短縮）で 100 日級が標準化した時点（2027 年目安）**。再評価時の候補は CF-4 層2 のリスナ再構成全体ではなく、**拇印キーのみの証明書ホットスワップ**（`TlsSyslogListener` はハンドシェイクごとに証明書を提示する構造であり、差し替えはソケット再 bind を伴わない——限定緩和で足りる可能性が高い。コストを不当に重く見積もらない）
- **相互 TLS（クライアント証明書）**: 既存の意図的先送りのまま（独立利用者 3 件以上）
