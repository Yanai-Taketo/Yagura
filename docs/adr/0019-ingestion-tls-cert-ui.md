# ADR-0019: TLS 受信の証明書選択の UI 設定化（ADR-0012 の再利用）

- 状態: proposed
- 日付: 2026-07-18
- 決定者: YANAI Taketo
- 関連: [ADR-0012](0012-admin-https-cert-ui.md)（管理リモート HTTPS の証明書選択 UI——本 ADR の再利用元）/ [security.md](../design/security.md) §6（TLS 受信）/ [configuration.md](../design/configuration.md) §8（`Ingestion:Tls:*`）/ CF-4（Issue #262。リスナ再構成）/ Issue #276 / PR #341

## 文脈と課題

TLS 受信（RFC 5425。opt-in。Issue #137 で実装済み）の証明書は、現状 **`yagura.json` の `Ingestion:Tls:CertificateThumbprint` を手編集する以外に設定手段がない**。管理リモート HTTPS 側は ADR-0012 で証明書ストア列挙・選択 UI・保存前 fail-closed 検証が実装済みであり、**「異常は知らせる（期限接近 1017・使用不能 1018 の能動通知は実装済み）が、UI から直せない」**という非対称が残っている。

本件は ADR-0012 のスコープから分離済み（オーナー裁定 2026-07-11・論点 4）で、起案予定への登録済み（Issue #276）。**主論点は「ADR-0012 で確立した設計をどこまで再利用できるか」**——新しい設計判断を増やさず、確立済みの決定への差分だけを書く。

### スコープ（非目標）

- 証明書の取得・生成（ACME・自己署名生成等）は扱わない（ADR-0012 と同じ）
- TLS 受信リスナの無瞬断再構成は扱わない（CF-4 層2 が「TLS 受信キーは対象外——宣言どおり再起動」と確定済み。本 ADR はその宣言を変えない——再評価トリガ参照）
- クライアント証明書（相互 TLS）は扱わない（意図的先送りのまま）

## 検討した選択肢

### ADR-0012 設計の再利用範囲

- **(a) 全面再利用（採用）**: 列挙契約（serverAuth EKU + 秘密鍵あり・最小フィールド・read-only）・選択 UI・保存前 fail-closed 検証・「UI 事前検証と起動時検証の乖離ゼロ」・秘密鍵の読取検証 + 誘導（付与は試みない——lab 実測 2026-07-11 で `NT SERVICE\Yagura` は `WRITE_DAC` 不可と確定済み）・テスト戦略（Abstractions 契約 + 偽実装）を**そのまま流用**し、対象キーと起動時検証経路（1016〜1018）だけ差し替える
- (b) TLS 受信専用の簡易版（拇印手貼り欄のみ UI 化）: 実装は薄いが、ADR-0012 が撤廃した拇印手貼りを新設面で復活させることになり、設計の一貫性を壊す。**却下**
- (c) ADR-0012 の画面に統合（1 画面で管理 HTTPS と TLS 受信の両方）: 対象リスナ・反映方式（管理 HTTPS = 管理リスナ再起動系 / TLS 受信 = **受信断**を伴う再起動）が異なり、「保存 = 何が止まるか」の説明が 1 画面で混線する。**却下**（画面は分け、列挙・検証の実装は共有する）

## 決定

### 1. TLS 受信の証明書設定画面を管理面に追加する（ADR-0012 決定 1・2 の流用）

- 対象キー: `Ingestion:Tls:Enabled`・`Ingestion:Tls:Port`・`Ingestion:Tls:CertificateThumbprint`（**すべて configuration.md §8 の確定済み表に既出——本 ADR は設定キーを 1 つも追加しない**。ADR-0012 決定 6 と同じ低リスク主論拠）
- 証明書選択は ADR-0012 決定 2 の列挙 UI を流用する（serverAuth EKU + 秘密鍵あり・最小フィールド・期限切れ/EKU 不適合/秘密鍵なしの除外または警告・空一覧の案内文）。**列挙契約・実装は管理 HTTPS 側と共有し、二重実装しない**
- 画面は管理 HTTPS 設定とは別画面とし（選択肢 (c) の却下理由）、相互リンクを置く

### 2. 保存前 fail-closed 検証と起動時検証の乖離ゼロ（ADR-0012 決定 4 の流用）

- 保存前検証の対象: 「`Enabled=true` なのに拇印未設定」「証明書が見つからない／秘密鍵にアクセスできない／期限切れ／EKU 不適合」。**検証は起動時の TLS 証明書解決（イベント ID 1016〜1018 経路。`ActiveNotificationMonitor.EvaluateIngestionTlsCertificate` が使う同一の解決コード）へ寄せ、「UI 事前検証が緑なら起動後に想定外の縮小継続（1016）が起きない」ことを回帰テストで固定する**（D-6 と同型）
- **期限切れの扱いは既存の設計判断に従う**: TLS 受信は期限切れでも受信を止めない（security.md §6——送信側の検証ポリシー次第で接続が拒否されるのは送信側の判断）。したがって UI は期限切れ証明書の保存を**警告付きで通す**（管理 HTTPS 側と挙動が異なる場合は、その差分と理由を画面文言・受け入れ基準に明示する）

### 3. 秘密鍵は読取検証 + 誘導に留める（ADR-0012 決定 3 の流用）

- UI は付与を試みず、サービスアカウントが秘密鍵を読めるかを検証・表示し、読めない場合は `certlm.msc`（CF-D2）手動付与へ誘導する。起動時の best-effort 付与（監査 2010）は既存のまま
- lab 実測（2026-07-11。PR #244）の縮退確定を TLS 受信側にもそのまま適用する——同じ `NT SERVICE\Yagura`・同じ CNG 鍵 DACL 制約であり、再検証は不要

### 4. 反映方式は再起動（受信断）の明示（ADR-0012 決定 6 の流用 + 受信断の強調）

- `Ingestion:Tls:*` は §8 宣言どおり **RestartRequired**（TLS リスナ構築は起動時処理。CF-4 層2 の対象外宣言を維持）
- **管理 HTTPS 側との決定的な差分**: 反映のための再起動は **syslog 受信断**を意味する。UI には「保存だけでは反映されない・反映は次回サービス再起動時・その間 TLS だけでなく全受信が停止する」ことを、推定断時間・再起動待ちキーの常設表示（#286）への合流とあわせて明示する
- 保存された未反映変更は既存の再起動待ちキー機構（1020・/admin 常設カード）にそのまま乗る——新しい表示機構は作らない

### 5. 監査（ADR-0012 決定 7 の流用）

- TLS 受信の証明書拇印・有効化の変更は、独立イベント ID で記録する（2000 番台 additive。採番は実装 PR——security.md §4.3 SEC-5 のレール。管理 HTTPS 側の 2012 と対になる TLS 受信版）
- 記録内容は 2012 と同型（変更キーと新値——拇印は公開識別子であり値を残す）

### 6. 残存リスクの明記（ADR-0012 決定 8 の流用）

- 既定無認証 loopback で列挙 API が到達可能である点は ADR-0012 決定 8 の記述がそのまま当てはまる（本 ADR で新しい列挙面は増えない——同一 API の第 2 の呼び出し画面が増えるだけ）。利用者向けドキュメントの該当記述に TLS 受信画面を追記する

## 受け入れ基準

- 通常運用で拇印の手入力・JSON 手編集なしに TLS 受信証明書を選択・保存できる
- UI 事前検証が緑なら、再起動後に想定外の縮小継続（1016）が起きない（回帰テストで固定）
- 期限切れ証明書の保存は警告付きで通り、起動後も受信が止まらない（security.md §6 との一致）
- 保存時に受信断の予告（再起動が必要・全受信が止まる）が表示され、未反映変更が再起動待ちカードに現れる
- 列挙・検証の実装が管理 HTTPS 側と共有されている（二重実装がないことをレビューで確認）
- 変更の監査記録（新 ID・拇印新値を含む）が残る

## 帰結

- 良くなること: 「異常は知らせるが UI から直せない」非対称が解消し、TLS 受信の導入・証明書更新が管理 HTTPS と同じ操作感で完結する。ADR-0012 の設計・実装・lab 実測を再利用するため、新規の設計リスクがほぼない
- 悪くなること・受け入れるリスク:
  - 反映は再起動 = 受信断のまま（CF-4 層2 の宣言を維持）。証明書の定期更新のたびに計画受信断が要る——TLS リスナの無瞬断再構成は再評価トリガへ
  - 画面が 1 つ増える（管理 HTTPS と別画面）。統合しない理由（受信断の説明の混線防止）は決定 1 に記録済み

## 先送りにする場合の再評価トリガ

- **TLS 受信リスナの無瞬断再構成**（証明書更新を受信断なしで反映）: 証明書の定期更新運用で計画受信断が負担であるという利用者報告があった時点で、CF-4 層2 の対象外宣言の見直しとして独立に再評価する（本 ADR の画面はそのまま使える——反映方式だけが変わる）
- **相互 TLS（クライアント証明書）**: 既存の意図的先送りのまま（独立利用者 3 件以上）
