## 文脈と課題

2026-07-11 に実施された別セッションのセキュリティレビューが **M-1（ロックアウト DoS）** を提起した: ADR-0010 決定 3 が採用したアプリ独自認証の保護機構は「一定回数の失敗試行で一定時間の完全ロックアウト」であり、これは**単一アカウントモデル**（Phase 1。`IAdminAccountStore` は最初の管理者アカウント 1 件のみを保持する）と、**ADR-0010 Phase 2 で実装済みのリモートバインド解禁**（PR #224。管理リスナがインターネット/LAN から到達可能になり得る）が組み合わさると、**推測容易なユーザー名へ数回の誤試行を送るだけで、唯一の正規管理者を締め出せる可用性攻撃の梃子になる**、という指摘である。

この指摘は #227 で amendment として起案されたが、オーナーは 2026-07-11 に **supersession（新規 ADR）として起案し直すため #227 をクローズ**した。クローズ判断の要旨(全文は #227 のクローズコメント参照):

1. ADR-0010 決定 3 の「一定回数の失敗で一時ロックアウトする」は**無条件文**であり、SEC-12・委任事項 4 が確定を委ねているのは「閾値・期間」という**パラメータ**であって機構の選択そのものではない。本変更は「試行は一定期間**完全に拒否**されるか、否か」という最小単位で見て Yes→No の**反転**であり、[docs/adr/README.md](README.md) の判定基準（「既定を反転させるなら supersession」「疑わしければ supersession 側に倒す」）と [ADR-0002](0002-architecture-principles.md) 改訂履歴の先例（「Decision を変える場合は supersession」）に照らして supersession が既定になる。
2. #227 でコミットされた amendment 本文にはレビューが参照するセクション構造が存在せず、5 ペルソナの指摘と反映の対応が PR 上で追えなかった——[docs/development/persona-review.md](../development/persona-review.md) の「対話はすべて PR 上に残す」原則を実質的に満たしていなかった（ADR 本文をセクション構造を持つ完全な形でコミットしていなかったことが原因）。

本 ADR は、**#227 の分析と 5 ペルソナ（田中・鈴木・クリス・佐藤・リサ）の指摘一式を出発点として**、supersession として起案し直すものである。オーナーのクローズ判断は、加えて #227 のレビューで残った未解消論点（IP レート制限/グローバルトークンバケット側の loopback 除外の明記・`security.md` L86 の用語不整合・NAT 配下の巻き込み対策の IP 窓/上限）も本 ADR で扱うことを明示的に求めている。

## 検討した選択肢

### 失敗試行対策の機構

- (a) **現行のハードロックアウト維持**: 一定回数の失敗で一定時間の完全拒否。単純だが、M-1 の核心（唯一の正規管理者を締め出せる）を放置する。単一アカウントモデル + リモート公開という ADR-0010 Phase 2 が作った利用像の下では、この設計そのものが可用性攻撃の梃子になる。却下
- **(a') ハードロックアウトを維持しつつ、ロックアウト状態を (アカウント × 到達リスナ種別〔loopback / remote〕) で分離する中間案**（クリスの再レビュー指摘 1 で提起された、最も安い対案）: 「リモート経由の失敗はリモート側のロックアウト状態のみを進め、loopback 経由の試行には一切影響しない」とすれば、M-1 が名指しする害（唯一の正規管理者を **loopback からの復旧経路を含めて完全に** 締め出す）は構造的に消える。かつ完全拒否が持つ「確実な総当たり上限」（N 回で必ず止まる）を残せる。決定 3 が採用する複合キー分離（`IsUnauthenticatedLoopbackBypassAllowed` と同型）をそのまま流用でき、波及範囲も小さい。**この安い対案を先に殺してからでなければ、三層防御という波及の大きい変更に進むべきではない**（クリスの指摘）——以下、正面から評価する:
  - **却下の決め手**: この中間案は M-1 の害を「完全な締め出し」から「**リモート管理能力の、攻撃者が随意に発動できる恒常的な否認**」へ縮小するにとどまる。ロックアウト状態を loopback/remote で分離しても、**リモート側のロックアウトは残る**——外部の未認証攻撃者が N 回の誤試行を T 間隔で送り続けるだけで、リモート側を恒久的にロックアウト状態に保てる。その間、正規管理者は「リモートから入れず、物理到達（RDP/コンソール）に退避させられる」。これは ADR-0010 Phase 2 が解禁したリモート管理能力そのもの（= 佐藤ペルソナが原初に訴えた「RDP が唯一の管理経路」という不便の解消）を、攻撃者が任意のタイミングで無効化できることを意味する。**「loopback には必ず退避できる」は事実だが、それは Phase 2 が売った価値を攻撃者が随意に奪える状態を許容することと引き換えである**。バックオフはこの点で構造的に優れる——正しいパスワードを持つ管理者は、攻撃継続中も（最大 cap の遅延を負うだけで）**リモートから必ず入れる**。リモート管理能力が未認証攻撃者によって否認されることがない
  - **失う「確実な総当たり上限」の評価**（正面から）: ハードロックアウトの「N 回で必ず止まる」上限は、まさに M-1 の梃子そのものである——**攻撃者を止める同じ性質が、正規利用者をも止める**。この性質を「listener で分離すれば正規利用者だけ守れる」と見えるのが中間案の魅力だが、上記のとおり分離は「完全な締め出し」を「リモート否認」へ格下げするだけで、攻撃者が可用性を制御できる状態は残る。**「確実な上限」と「M-1 耐性」は同一機構内では両立しない**（上限を課す＝どこかで正規利用者も止まる）。**この命題は count-based な拒否機構（試行回数を数えて上限で止めるロックアウト系全般）に限った命題である**（クリスの再レビュー指摘への射程明示）——攻撃者のカウントと正規利用者のカウントを区別できない以上、必ずどこかで正規利用者も止まる。**カウント以外の識別信号を持つ機構（CAPTCHA 等の「人間性の証明」を挟む方式）はこの命題の外側にある**（自動化された攻撃だけをカウントに依らず止め得る）。本 ADR が CAPTCHA 系を検討した選択肢に含めないのは、Yagura のスコープ——Windows-shop 向け・信頼ネットワーク前提・管理 UI の単純さ（外部 CAPTCHA サービスへの依存・オフライン運用との不整合を避ける）——に照らした意図的な除外である。バックオフはこの緊張を「崖をなくす」ことで解消し、失った上限を **パスワード強度（決定 7）+ IP レート制限 + グローバルトークンバケット + 能動通知（決定 6）** で多層に補う。これは「一撃の確実な上限」を「複数の確率的・観測的な抑止」へ置き換える設計判断であり、M-1（可用性）を優先する本 ADR の目的に整合する
- (b) **バックオフのみ**（アカウント単位の指数遅延、完全拒否なし）: 「正しいパスワードなら待てば必ず通る」を実現し M-1 を構造的に消すが、**単一アカウントモデル固有の脅威——そもそも実在ユーザー名が何かを当てる探索攻撃——には無力**（クリスの指摘。非実在ユーザー名にバックオフを課さない設計を選ぶ限り、探索速度を落とす機構が存在しない）。単独では不十分。却下（他と併用）
- (c) **送信元 IP レート制限のみ**: 単一送信元からの高速試行は防げるが、**送信元 IP を分散すれば回避できる**分散ブルートフォースに無力。単独では不十分。却下（他と併用）
- (e) **IP allowlist（管理リスナへの到達元 IP を許可リストで絞る）**（クリスの再レビュー指摘 1 で言及された「攻撃面を減らす」方向の対案）: 到達できる送信元を事前に制限すれば試行の母集団自体が縮む。しかし **失敗試行対策の代替にはならない**——許可された IP の内側でも何らかの失敗試行対策は依然必要であり（許可 IP が NAT ゲートウェイなら背後に多数の主体が居る）、本 ADR が選ぶ機構と直交する。加えて、本 ADR の主対象層（ひとり情シス・動的 IP・NAT・出張先からの管理〔ADR-0010 が解禁したリモート管理の想定利用像〕）では静的な allowlist の保守が現実的でない場合が多く、loopback 脅威モデル（同一ホスト上の悪意あるローカル主体）も覆えない。**補完的な opt-in としては将来有用**（決定 4 の loopback 脅威や NAT 巻き込みを別軸で縮小し得る）だが、本 ADR の失敗試行対策の選択そのものを代替しないため、ここでは採用機構の代替としては却下し、将来の opt-in 候補として再評価トリガに記録する
- **(d) バックオフ（アカウント単位）+ IP レート制限（送信元単位）+ グローバルトークンバケット（プロセス全体・単一アカウントモデル向けの探索対策）の三層防御（採用）**: 各層が異なる脅威軸をカバーする——バックオフは「正規アカウントへの高速総当たり」、IP レート制限は「単一送信元からの高速試行」、グローバルトークンバケットは「多数の送信元 IP に分散したユーザー名探索」（クリスの指摘 4 への直接の応答）。三層はいずれも「完全拒否」を行わない（トークン涸渇時も待てば再充填される、または有限の Retry-After 後に再試行できる）ため、M-1 の核心（締め出し）を構造的に持ち込まない。中間案 (a') に対する優位は上記のとおり「リモート管理能力を未認証攻撃者に否認させない」点にある

### amendment か supersession か

- (a) **amendment として `security.md`・ADR-0010 改訂履歴のみで処理する**（#227 の当初案）: 変更差分は小さく見えるが、クリスが指摘したとおり「完全拒否をやめるか否か」は最小単位で見て決定の**反転**であり、README の判定基準（「疑わしければ supersession」）に反する。加えて、実装済み資産（DB 列・原子的 UPDATE・監査 ID 3005）の扱いという波及範囲の大きい設計判断を伴い、実質は supersession 水準に近い。**オーナーが 2026-07-11 に却下（#227 クローズ判断）**
- **(b) supersession（新規 ADR）として起案し直す（採用。本 ADR）**: 決定の反転を ADR 単位で明示し、5 ペルソナレビューを通しやすいセクション構造を持つ独立文書として起案する。ADR-0010 側には supersession された旨を追記する（決定 1）

## 決定

### 1. supersession の宣言と範囲

- **本 ADR は [ADR-0010](0010-admin-ui-authentication.md) 決定 3 のうち「ロックアウト」の記述（「一定回数の失敗試行で一時ロックアウトする」から始まる段落、およびその帰結として §2.4 が定めた実装参照記述）を supersede する**。ADR-0010 決定 3 の他の内容（最小実装の採用・`PasswordHasher<TUser>` の利用・単一アカウントモデル・ロックアウト/パスワード失念時の復旧経路が物理到達 → loopback 管理 UI であること・初期パスワード設定の流れ・両認証方式の共存の設計方針・ブレークグラス推奨）は**本 ADR の対象外であり、そのまま有効**とする
- ADR-0010 全体を superseded にするのではなく、**決定 3 のロックアウト部分のみ**を対象とする部分 supersession である。ADR-0010 の状態は `accepted` のまま維持し、決定 3 冒頭・決定 6 の該当箇所に本 ADR への参照注記を追加する（本 PR の時点で反映済み。決定 9 の ADR-0010 本文整合の項を参照）。**部分 supersession の記述作法**（旧 ADR の status は `accepted` のまま維持し、該当決定の冒頭に supersession 注記を加える）は本プロジェクト初の実例であり、`docs/adr/README.md` の運用ルールにこの慣例を明文化した（リサ・田中の指摘。本 PR で反映）

### 2. 機構の転換: バックオフ + IP レート制限 + グローバルトークンバケットの三層防御

- **完全拒否を廃止し、指数バックオフへ置き換える**: 連続失敗が閾値 k 回を超えた時点から、試行ごとの遅延を指数的に増加させる（`delay = min(base * 2^(n-k), cap)`。n は連続失敗回数）。**cap に達しても拒否はしない**——正しいパスワードであれば cap の待ち時間の後に必ずログインできる。これが M-1（唯一の正規管理者の締め出し）を構造的に消す核心である
- **送信元 IP 単位のレート制限を併設する**（単一送信元からの高速試行対策。アカウントの実在有無に関わらず一律に適用——存在しないユーザー名への探索試行もこの層で遅くなる）
- **プロセス全体のグローバルトークンバケットを併設する**（クリスの指摘 4 への応答。単一アカウントモデルでは「そもそも実在ユーザー名が何か」を当てる探索が費用対効果の高い攻撃経路になるため、送信元 IP を分散されても総試行数そのものを絞る layer を置く）
- **Windows 統合認証（Negotiate）は対象外のまま維持する**（ADR-0010 決定 3 の既存の非対称——Negotiate は OS/AD のロックアウト・パスワードポリシーに依拠し本製品は関与しない——を継承。決定 9 参照）
- **三層の評価順序とレイヤー間の状態更新の因果（リサの指摘 2、鈴木の指摘 3 への応答。実装解釈の割れを防ぐため本文で固定する）**:
  - **評価順序は ①IP レート制限（送信元単位・最も安価・loopback 除外——決定 4） → ②グローバルトークンバケット（プロセス全体・loopback 除外——決定 4） → ③アカウント単位バックオフ + パスワード検証**、とする。前段（①②）で拒否された試行は 429 + Retry-After で即座に返し、**パスワード検証まで到達しない**
  - **連続失敗回数 n は「パスワード検証まで到達して失敗した試行」だけが進める**。①②のレート制限・トークンバケットによる拒否は認証前の拒否であり **n を進めない**（二重計上しない）。この因果により、決定 10 の仮値表の n は「実際にパスワードを誤った回数」を意味し、レート制限に阻まれた試行で意図せず膨らむことがない
  - この順序は「安価な層で母集団を絞ってから高価な PBKDF2 検証に進む」という防御・計算資源の両面で合理的な並びである（クリスの指摘した『各層が異なる脅威軸をカバーする』の実装上の帰結）

### 3. バックオフの詳細設計（田中の指摘 1・2、鈴木の指摘 1 への応答）

- **状態化する対象は実在アカウントのユーザー名に限る。非実在ユーザー名には個別のバックオフ状態を持たせない**（田中の指摘 1 への応答）。理由: Phase 1 は単一アカウントモデルであり、実在アカウントの集合は攻撃者ではなく管理者が制御する（将来複数アカウントに拡張しても、アカウント数は運用者が作成した数に限られる）。**状態を持つキー空間の上限が攻撃者の入力ではなく管理者の操作によって決まる**ため、LRU・リングバッファ等の追加の上限機構なしに、設計そのものでメモリ枯渇 DoS を回避できる。非実在ユーザー名への試行は、既存の列挙耐性（ダミーハッシュ検証によるコスト平準化。`AppAdminAuthenticationService` 既存実装）をそのまま維持したうえで、決定 2 の IP レート制限・グローバルトークンバケットのみで絞る——**「非実在名には個別状態を持たせず IP レート制限側でまとめて絞る」という田中の提案の後半を採用する**
- **バックオフ状態のキーは (アカウント正規化ユーザー名 × 到達リスナ種別〔loopback / remote〕) の複合キーとする**（鈴木の指摘 1 への直接の応答）。リモート HTTPS 経由の攻撃者の失敗はリモート側のカウンタのみを押し上げ、loopback 経由の正規ログイン試行（RDP/コンソール到達後の復旧操作）の待ち時間には一切影響しない。これは ADR-0010 決定 1 が繰り返し明記してきた「loopback 経由の管理リスナは影響を受けない」「RDP + loopback からの復旧操作は常に可能」という設計原則を、バックオフ機構でも維持するための直接の帰結である。**送信元 IP はキーに含めない**（複数送信元へ分散しても同一アカウント・同一リスナ種別ならバックオフは迂回されない——これは #227 の設計をそのまま維持する。良い点として全ペルソナが支持済み）
- **状態更新は原子的に行う**（田中の指摘 2 への応答）。`ConcurrentDictionary<TKey, BackoffState>.AddOrUpdate` のアトミックな更新デリゲート（またはアカウント単位の軽量ロック）を用い、現行 SQLite 実装（`SqliteAdminAccountStore.RecordFailedLoginAsync`。PR #217 で `UPDATE...RETURNING`/`OUTPUT` による単一原子文へ改修済み）が持っていた「分散送信元からの同時失敗による lost update 防止」という保証を、インメモリ実装でも明示的に引き継ぐ。**実装 PR の受け入れ条件として「同時多発失敗でも取りこぼしがないこと」を回帰テストで固定する**（委任事項 1）
- **「正しいパスワードなら待てば必ず通る」の正確な射程を明記する**（鈴木の指摘 3 への応答）: 攻撃が継続中は、n が高止まりし続けるため、正規管理者の入力も常に cap 近辺の待ちに巻き込まれる。「待てば必ず通る」が指すのは「完全拒否（詰み）が発生しない」ことであり、「常に短時間で通る」ことではない。この区別を UI 文言・利用者向けドキュメント（決定 6・SEC-D6）に明記する
- **連続失敗回数 n のリセット・減衰条件を明記する**（佐藤の指摘 1・質問、鈴木の指摘、リサの指摘 3 への応答。#227 で未記載だった論点）:
  - **ログイン成功時に、当該キー（アカウント × リスナ種別）の n を 0 にリセットする**（指数バックオフの標準的前提だが、ADR 単体で実装できるよう明記する）
  - **一定時間そのキーへの失敗試行が途絶えたら n を自然減衰（リセット）する**（アイドル減衰。仮値: 30 分無失敗で n を 0 に戻す——決定 10）。これがないと、佐藤ペルソナが指摘した「金曜夜の数回の誤入力が月曜朝の自分を待ち構える」ケースが発生する。減衰窓は SEC-12 の一項目として実運用で確定する
  - **プロセス再起動では全キーの状態が失われる**（インメモリ保持の帰結）。これは「正規の復旧手段」ではなく攻撃者にも作用する副作用であり、SEC-D6 で「バックオフ・レート制限の待ちを解消する目的でサービスを再起動しないこと（= syslog 受信断を招くだけで、待てば cap で抜けられる）」を明記する（鈴木の指摘 4・帰結）
- **反復試行によるユーザー列挙耐性の扱い（田中の指摘 2〔最重要〕への応答。明示的な決定として閉じる）**:
  - **現行実装の非対称を確認した**（`src/Yagura.Host/Administration/AdminAuthentication/AppAdminAuthenticationService.cs` 81-84 行）: 旧ロックアウト方式は、ロックアウト中に実パスワード検証（PBKDF2）をスキップして `LockedOut` を即返す最適化を持つ。非実在ユーザー名は絶対に `LockedOut` にならない（常にダミーハッシュ検証 + `InvalidCredentials`）ため、**「同一ユーザー名を閾値回以上叩くと挙動が変わるか」という反復軸の観測点が、実在/非実在を暴く**。バックオフ移行後も、素朴に実装すれば実在アカウントは k 回目以降に遅延が生じ、非実在名は何回叩いても即時 `InvalidCredentials` のまま、という同型の非対称が残る
  - **決定（応答形状・UI 文言の統一）**: 利用者への **応答の形（クライアントが観測する結果種別・HTTP 応答）と、ログイン画面の待機表示の文言の両方を、実在アカウントのバックオフ待機中・非実在ユーザー名・IP レート制限/グローバルバケット拒否のいずれでも区別しない**。バックオフ待機中でも `LockedOut` 相当の「待たされている理由を暴く結果種別」をクライアントへ返さず、汎用の `InvalidCredentials` 相当に留める。**UI の待機表示も、原因の別（連続失敗 / アクセス集中）を明かさない単一の一般文言に統一する**（決定 6 で文言を規定——「連続失敗のため」のような原因を告げるテキストは使わない。田中の再レビュー指摘への応答）。失敗理由・待機理由の種別は監査記録にのみ残す（既存の `security.md` §4.3「失敗理由の種別は監査記録にのみ残し、利用者への応答では区別しない」を反復軸・UI 文言軸にも拡張）
  - **決定（本決定と決定 6 の射程の切り分け）**: 本決定 3 が規定するのは「応答種別・UI 文言で原因を暴かない」こと、決定 6 が規定するのは「その統一文言の具体（カウントダウンの見せ方等）」である——両者は矛盾せず、決定 6 の文言は本決定 3 の非開示要件に従う。**残る差はカウントダウンの秒数値そのものである**（実在名のバックオフ遅延と非実在名の Retry-After は必ずしも一致しない）が、これは下項の**タイミング非対称**（帰結に明記した Phase 1 の受け入れるトレードオフ）に収斂する——文言という「計測不要の直接的な告白」は本決定で消し、残るのは「計測して初めて分かる推論」だけになる
  - **決定（タイミング非対称は Phase 1 の受け入れるトレードオフ）**: それでも、実在アカウントは account-keyed バックオフの遅延を、非実在名は IP レート制限・グローバルトークンバケットの遅延のみを負う——**両者の遅延時間は完全には一致しない**（決定 3 のメモリ枯渇回避のため非実在名に個別状態を持たせない以上、per-name のタイミング平準化は構造的に不可能）。この **タイミング軸の非対称は Phase 1（単一アカウントモデル）では実害が限定的**（実在ユーザー名は運用者が決めた 1 件のみで、それを暴いても攻撃者は既にそのアカウントを攻撃している）として **受け入れるトレードオフに明記**する（帰結）。**複数アカウントモデルへの拡張時**（決定 10 の再評価トリガ）は、まさにグローバルトークンバケットが対策する「ユーザー名探索」をこの非対称が容易にするため、そのタイミングで列挙耐性を維持する再設計（例: 非実在名にも上限付きで擬似的な遅延を課す）を必須の検討事項とする（再評価トリガに登録）

### 4. loopback 無条件復旧経路の原則をレート制限側にも拡張する（オーナー指摘の未解決論点への応答）

#227 のクローズ判断が名指しで残した論点。ADR-0010 決定 1 の「loopback は認証手段が全滅しても無条件の最終復旧経路」という原則は、決定 3（バックオフ）だけでなく IP レート制限・グローバルトークンバケットにも及ぶべきであり、本 ADR で明示する。**#227 稿では IP レート制限を「自然な分離」に留め、層ごとに loopback の扱いが不揃いだった（鈴木・田中・クリスの指摘）。本稿はこれを是正し、三層すべてで loopback の扱いを一貫させる**:

- **原則（三層一貫）**: **loopback（`127.0.0.1` および `::1` の両方）経由の試行は、IP レート制限・グローバルトークンバケットのいずれによっても拒否（429）されない**。loopback に作用する失敗試行対策は、**決定 3 のアカウント単位バックオフ（cap で頭打ちの有限遅延・完全拒否なし）のみ**とする。これにより「loopback は認証手段が全滅しても無条件の最終復旧経路」（決定 1）が、三層防御のどの層によっても否認されないことを保証する
- **IP レート制限の loopback 明示除外（鈴木の指摘 1 への直接の応答。#227 の『自然な分離』からの是正）**: #227 は「送信元アドレスで判定する設計上 loopback は他の送信元と別バケットになる（自然な分離）」と述べていたが、これは **loopback 自身の試行回数で loopback 自身が 429 を食らう経路を残す**（深夜の焦りで復旧オペレータが `127.0.0.1` から短時間に何度も試すと同一バケットに積み上がる——鈴木の指摘）。加えて `127.0.0.1` と `::1` を別送信元として扱うと実効枠が 2 倍になる非対称も生じる（田中の指摘 3）。**したがって「別バケットに割り当たるだけ」ではなく、loopback（両アドレス）を IP レート制限の明示的な除外対象とする**——loopback からの試行はカウントも 429 拒否もしない。復旧オペレータが焦って連打しても IP レート制限に阻まれない
- **グローバルトークンバケットの loopback 除外**: プロセス全体で単一状態を持つ設計上、自然な分離を持たない——これが #227 で未検討のまま残っていた穴である。**loopback 束縛ポート経由の試行はトークンを消費せず、涸渇状態からも影響を受けない**。リモート攻撃者がバケットを涸渇させても、loopback 経由の正規ログインはバケットの状態に関わらず決定 3 のバックオフ判定のみに従う
- **判定方式**: いずれの層の loopback 判定も、ADR-0010 Phase 2 が既に確立した「接続の実ローカルポートから loopback/remote を判定する」実装（`IsUnauthenticatedLoopbackBypassAllowed` と同型の判定点。`security.md` §2.5）にそのまま乗せる。**三層とも同一の判定点を共有**し、層ごとに loopback の定義がずれないことをテストで固定する（委任事項 9・10）
- **この扱いは「loopback は認証を免除される」という意味ではない**（loopback 認証 opt-in が有効な環境では引き続き認証を要求する。ADR-0010 決定 1）。ここで免除するのはあくまで「失敗試行対策としての追加の遅延・拒否」であり、認証そのものの要否とは独立した軸である
- **田中の「loopback 専用の緩いが有限なバケット」案の評価と採否（田中の指摘 3、クリスの指摘 3 への応答）**: 田中は「グローバルバケットに一切カウントしない（完全除外）」ではなく「loopback 専用の、他送信元より緩いが有限のバケットを別途持たせる」設計を提案した——将来の複数アカウントモデルで、侵害されたローカル主体が loopback からユーザー名探索を摩擦ゼロで行える穴を塞ぐため。**評価**: この懸念は複数アカウントモデルで初めて実在する（Phase 1 は単一アカウントで探索対象のユーザー名が事実上存在しない）。一方、「loopback 専用バケット」は loopback に **有限の拒否経路を再導入**するため、決定 1 の「無条件の最終復旧経路」を Phase 1 の時点で弱める。**したがって Phase 1 では完全除外を採る**（信頼ネットワーク前提〔ADR-0004〕+ 単一アカウント + 決定 1 の最優先性が根拠。クリスが求めた「trusted network 内なので許容する」の理由づけをここで明示する）。**ただし田中案は複数アカウント化のタイミングで採用を再検討する設計として再評価トリガに登録する**（そのときこそ loopback 経由のユーザー名探索が実在の脅威になり、決定 1 との緊張を「十分寛容だが有限」の値で調停する意味が生じる）

### 5. NAT 配下の正規ユーザー巻き込み対策の方針（オーナー指摘の未解決論点への応答）

- **IP レート制限の窓・上限は、単一の NAT ゲートウェイ配下に複数の正規利用者が同居し得ることを前提に、寛容な値を選ぶ方針とする**。具体的な数値は SEC-x（決定 10）で確定するが、方針として次を確定する:
  - **アカウント単位のバックオフ（決定 3）を主たる防御とし、IP レート制限・グローバルトークンバケットは補助的な第二・第三軸に位置づける**。バックオフは実在アカウント数（管理者が制御する集合）に依存し NAT の影響を受けないため、IP レート制限側を強く絞り込まなくても全体としての防御水準は落ちない。この主従関係により、**IP レート制限の窓・上限を「多少緩めても安全側に倒れる」設計にできる**——NAT 配下の正規ユーザーを巻き込むリスクと、IP レート制限自体の防御力への要求を両立させる
  - IP レート制限は「通常の人間が同一 NAT 配下から短時間に生成し得る試行数」を明確に上回る値を下限とする（後述の仮値は 60 秒窓 10 回。数名が同時にタイプミスを重ねても抵触しにくい水準を狙う——実運用のフィードバックで SEC-x として確定する）
  - **429（レート制限拒否）の応答にも待機理由を明記し**（決定 6）、巻き込まれた正規利用者が「攻撃と誤認されて拒否されている」ことを識別できるようにする

### 5.1. レート制限・トークンバケット層は「待たせず即拒否」とし、待ち時間に有限の上限を持つ（鈴木の指摘 2 への応答）

- **IP レート制限・グローバルトークンバケットは、要求を待ち行列に保持せず、超過時は即座に 429 + `Retry-After` で拒否する**（決定 2 の評価順序 ①②）。バックオフ（決定 3）が「要求を遅延させて最終的に通す」のに対し、レート制限層は「今は通さない・有限時間後に再試行せよ」を返す。**この設計により、待ち行列末尾での無限待ちが構造的に発生しない**——鈴木が指摘した「複数オペレータの同時ログインで 6 人目・7 人目が 1/秒の補充を延々待つ」事態を、各要求が有限の `Retry-After` を受け取って自律的に再試行する形に置き換える
- **グローバルトークンバケットのバースト値は、Phase 2 の複数オペレータ同時対応の利用像に合わせて見直す**（鈴木の指摘 2）。§2.2 が circuit 上限を 5→20 に引き上げた根拠（「管理リスナ = 実質 1 人」の前提が Phase 2 で崩れた）はトークンバケットにも当てはまる——#227 稿のバースト 5 は「数名が 1 回ずつ誤入力しただけで枯渇」する過小値であった。**バーストは想定同時オペレータ数を上回る値に設定する**（仮値をバースト 5 → 20 へ引き上げ、`Retry-After` の上限も設ける。決定 10）。確定は SEC-12（想定同時オペレータ数の根拠つき）で行う
- **loopback は本層の対象外**（決定 4）——複数オペレータが物理到達（RDP/コンソール）経由で loopback へ同時ログインする局面では、そもそも 429 を受けない

### 6. UI 表示・能動通知への昇格（佐藤の指摘 1・3、田中の指摘 4、鈴木の指摘 2 への応答）

- **ログイン画面に待機表示を出す（ただし原因は明かさない——決定 3 の列挙耐性と両立させる。田中の再レビュー指摘への応答）**: バックオフ・IP レート制限・グローバルトークンバケットのいずれによる待機でも、**原因を区別しない単一の一般文言 + カウントダウン**を出す（例: 「しばらくお待ちください。あと X 秒で再試行できます。」）。**「連続失敗のため」のような、そのユーザー名にバックオフ状態が存在する＝アカウントが実在することを暴く原因テキストは使わない**。**非実在ユーザー名に対しても同一の文言・体裁で表示する**（決定 3 の応答形状・UI 文言の統一）。この「壊れた／固まった、と誤解させない」役割（佐藤の指摘 1）は、原因を明かさなくても果たせる——利用者に必要なのは「待てば通る・あと何秒」であって「なぜ待たされているか」ではない
- **同一アカウントで cap 到達状態が一定時間（仮値 15 分。決定 10）以上継続、またはレート制限拒否が同水準で継続する場合、`ActiveNotificationMonitor`（architecture.md §4.6 の既存の周期監視基盤）を用いた 1000 番台の能動通知へ昇格する**（田中の指摘 4・鈴木の指摘 2 への応答）。既存パターン（スプール逼迫・証明書期限接近と同じ周期監視 + 抑制窓の仕組み）をそのまま流用する
- **能動通知の届け先チャネルを明記する**（佐藤の質問、鈴木の指摘 4 への応答）: 届け先は **`ActiveNotificationMonitor` の既存チャネル = Windows イベントログ（ソース `Yagura`・1000 番台・警告レベル）** である。本 ADR はメール送信等の新しい通知チャネルを追加しない（既存の観測基盤に乗る——`security.md` §4.3・SEC-D4「ソース Yagura の警告以上を通知」の最小監視構成で拾える）。「メールで飛ぶのか」という佐藤の疑問への答えは「既定ではイベントログ。イベントログ転送・監視を組んでいれば拾える」であり、これを SEC-D6 に明記する
- **能動通知の本文要件（鈴木の指摘 4 への応答）**: 通知本文には、対応判断に足る情報として **①発生源の主因層（バックオフ cap 到達 / IP レート制限 / グローバルトークンバケット涸渇のいずれか）、②上位の送信元 IP（複数あれば上位数件）** を含める。深夜に通知だけ受けた運用者が次の一手（該当 IP を FW で遮断・NAT ゲートウェイ担当へ連絡等）に使えるようにする。この本文要件を委任事項 7 に明記する
- **利用者向けドキュメント（SEC-D6）に反映する**（佐藤の指摘 3・質問、鈴木の指摘 4）: ①バックオフ・レート制限は仕様であり、正しいパスワードであれば待てば必ずログインできること（**数式ではなく「5 回連続で間違えると徐々に待ち時間が延びて最大 30 秒になる」等の一般向けの言葉で説明する**——佐藤の修正提案。SEC-D6 は非エンジニア管理者が読む場所である）、②単一アカウントモデルでは同僚の誤入力が自分のログインの遅延として波及し得ること（佐藤の指摘 2。設計で解消する性質のものではなく、共有アカウントモデルの構造的な帰結として明記に留める）、③攻撃が継続している間は待ち時間が常に cap 近辺になり得ること（決定 3 の射程の正確化）、④アプリ独自認証を有効化した時点で追加設定なしにバックオフ・レート制限の両方が既定で有効になること（佐藤の指摘 5）、⑤**バックオフ・レート制限の待ちを解消する目的でサービスを再起動しないこと**（再起動は syslog 受信断を招くだけで、待てば cap で必ず抜けられる——鈴木の指摘 4・決定 3 の n リセット条項）、⑥能動通知の届け先が既定でイベントログであること（佐藤の質問）

### 7. パスワード強度要件を本 ADR で確定する（田中の指摘 1・3、クリスの指摘 2、佐藤の指摘 4・質問への応答）

#227 稿・本稿初版は最小長のみを決め、複雑性要件を「実装 PR で検討」と先送りしていた。田中・クリスの指摘（cap=30 秒 ≒ 年間約 100 万試行オーダーの試行予算があり、`password12345` 等の辞書上位パターンはこの予算内に入る——「最小長だけ足して数値上対応した」体裁になっている）、佐藤の要求（何を満たせば良いパスワードなのかを実装前に決めてほしい）を受け、**本 ADR の時点で強度要件を確定する**:

- **最小長 12 文字以上を強制する**（`AppAdminAuthenticationService.SetAccountAsync` にチェックを追加。仮値は決定 10）
- **既知漏洩パスワード辞書・頻出パターンとの突合（ブロックリスト方式）を必須要件として確定する**（田中の指摘 1 の核心への応答。「要否を実装 PR で検討」という開いた書き方をやめ、**必須**に固定する）。`password12345`・`Password1234` 等の辞書上位・パターン上位のパスワードは、cap=30 秒の試行予算（年間約 100 万試行）の内側に入るため、長さだけでは塞げない——これを塞ぐのがブロックリストである。突合対象辞書の具体（同梱する既知漏洩リストの規模・出典・更新方針、または禁止パターン規則）は実装 PR で確定するが、**「ブロックリストによる辞書・パターン排除を行う」という設計判断そのものは本 ADR で確定**とする（委任事項 5）
- **辞書は実行時にネットワーク経由で取得しない（本 ADR で確定。クリスの再レビュー指摘への応答）**: ブロックリストの辞書は **リリース時点で静的に同梱し、更新はソフトウェア更新でのみ行う**。実行時にオンラインの漏洩パスワード DB へ問い合わせる設計は採らない——Yagura の「信頼ネットワーク前提・オフライン運用可能」の原則（CLAUDE.md 方針 7）と衝突し、新しい外部通信経路を生むためである。静的同梱は「辞書が徐々に陳腐化する」代償を負うが、これはソフトウェア更新の周期で受容する（外部通信を増やさない側を優先する）
- **同梱辞書の最小規模の目安（田中の非ブロッカー指摘への応答）**: 年間約 100 万試行オーダーの予算に対して意味のある防御にするため、**同梱辞書は少なくとも上位数万〜数十万語級を下限の目安とする**（数百語級の「やってる感」だけのリストにしない）。正確な規模・出典は実装 PR（委任事項 5）で確定するが、この下限の目安を本 ADR で示しておくことで、規模不明のまま形骸化する事故を防ぐ
- **文字種別（記号・大文字必須等）の複雑性ルールは課さない**（佐藤の質問への明確な回答 = 課さない）。理由: 文字種別ルールは利用者を予測可能なパターン（`Password1!` 等）へ誘導し、実効的な強度をむしろ下げることが知られている（現代のパスワードガイダンスは「長さの下限 + 既知漏洩/頻出のブロックリスト」を推奨し、文字種別強制を非推奨とする方向にある）。Yagura は後者を採る——**長さ下限 + ブロックリスト**で `password12345` 級を確実に弾き、記号必須のような覚えにくさは課さない（佐藤ペルソナの「厳しすぎて覚えられない域ではない」という評価を保つ）
- この強度要件は cap=30 秒を「最終防御」と呼ぶ主張の前提であり（決定 2・帰結）、両者をセットで確定することで田中の「強度側に何の強制もない現状と切り離して評価できない」という指摘を閉じる

### 8. DB 列の扱い（クリスの指摘 2・3 への応答）

- **`FailedAttemptCount`/`LockoutUntilUtc` の DB 列は「将来の DB 永続への戻しやすさ」を理由に温存しない。実装 PR で削除マイグレーションを行う**（クリスの指摘への応答。「列名を流用できても、バックオフが必要とする状態（n 回目ごとに再計算される次回許可時刻・最終失敗時刻）は既存列の書き込みモデルと異なり、実利は薄い」という指摘を採用し、「列を残すことの実利は移行コスト低減ではなく単なる削除の先送りである」という誠実な整理に立つ）
- **判定はインメモリのバックオフ状態に一本化し、DB への書き込み経路（`RecordFailedLoginAsync`/`RecordSuccessfulLoginAsync` のロックアウト関連引数）自体を呼ばなくする**——「書くが判定には使わない」という中間状態は選ばない（リサの指摘 1 が懸念した「運用者が DB を直接調査してロックアウト中と誤認する」事故の温床を残さない）
- **DB 列削除は監査証跡の連続性を損なわない**（田中の指摘 4 への応答。監査影響評価を読み手が自分で再構成しなくて済むよう明文化する）: 個別のログイン失敗試行は、DB 列（`FailedAttemptCount`/`LockoutUntilUtc`）とは独立に、append-only の監査ログへ既に記録されている（監査 ID 3004「アプリ独自認証のログインに失敗」。`security.md` 現行表・ADR-0010 決定 6）。DB 列は「現在の失敗カウント・ロックアウト期限」という**揮発的な判定用状態**を保持していたにすぎず、事後追跡の一次資料ではない。したがって DB 列を削除しても、侵害後調査に使う一次資料（append-only 監査ログ）は失われない。むしろ判定状態がインメモリへ一本化されることで、DB 上に「実際の状態と乖離した古い値」が残る事故（決定 8 冒頭）も消える
- `IAdminAccountStore.RecordFailedLoginAsync` の署名変更・削除マイグレーションの具体的な移行手順は実装 PR で確定する（委任事項 2）。**将来複数アカウント管理へ拡張する際の永続化要否は、本 ADR とは独立に再評価する**（バックオフ状態を DB へ戻す積極的な理由は本 ADR の時点では存在しない——インメモリのままで単一プロセス構成の前提と整合する）

### 9. 監査記録・`security.md` 記述の是正方針（リサの指摘 1・2・4・5 への応答）

- **監査 ID 3005（「管理者アカウントをロックアウト」）は additive-only 規約に従い凍結する**（本機構の採用以降は発火しなくなる。ID の意味・レベルは変更しない）。**`security.md` §4.3 のイベント ID 一覧表そのものの 3005 行「意味」列に凍結の事実を追記する**——散文の説明だけに頼らない（リサの指摘 2）。新規事象（バックオフ cap 到達・IP レート制限拒否・グローバルトークンバケット涸渇・能動通知への昇格）は 3000 番台の次番（3006 以降）で additive-only に採番する（実装 PR。委任事項 3）

### 10. SEC-12 の統合（仮値一覧）

`security.md` §7 SEC-12（現行: 「アプリ独自認証のロックアウト閾値・期間。仮値: 閾値 5 回 / 期間 15 分」）は、本 ADR の方式に合わせて実装 PR で次のとおり書き換える（項目名も「ロックアウト閾値・期間」から「バックオフ・レート制限仮値」に改める）:

| 項目 | 仮値 | 決め方（確定は引き続き SEC-12） |
|---|---|---|
| バックオフ猶予閾値 k | 3 回（k 回目までは遅延なし） | 実利用（誤入力の反復頻度）を踏まえ確定 |
| バックオフ基数 base | 1 秒 | 同上 |
| バックオフ上限 cap | 30 秒（`delay = min(base * 2^(n-k), cap)`） | UI 表示（決定 6）ありを前提に 30 秒を仮値とする。表示なしのまま実装される事態になった場合は佐藤の指摘 4 のとおり短縮を再検討する |
| n のアイドル減衰窓 | 30 分（無失敗が継続したら n を 0 にリセット。成功時は即 0——決定 3） | 「翌朝の自分が金曜夜の誤入力に巻き込まれない」窓として仮置き。実利用の失敗間隔分布で確定 |
| IP レート制限窓/上限 | 60 秒 / 10 回（loopback〔`127.0.0.1`・`::1`〕は明示除外——決定 4） | 想定同時利用者数の根拠を明示して確定する。**想定値の起点 = 単一グローバル IP の背後に数名〜十数名（ひとり情シス小規模オフィス。リサ・佐藤の想定像）がタイプミスを重ねても抵触しにくい水準**。**数百台規模の障害対応で複数の正規オペレータが同一 VPN 出口/NAT 経由で入る場面（鈴木の再レビュー指摘）では、悪意なく 60 秒/10 回に達し 429 が連発し得るため、初回リリースの仮値が辛めに振れていないかを実装 PR で一度実測する**。実運用フィードバックで確定 |
| グローバルトークンバケット | 定常 1 トークン/秒・バースト 20（プロセス全体のアプリ独自認証ログイン試行が対象。loopback 経由は対象外——決定 4。超過時は待たせず 429 + `Retry-After`——決定 5.1） | バースト 5→20 へ引き上げ（Phase 2 の複数オペレータ同時対応。§2.2 の circuit 上限 5→20 と同根——鈴木の指摘 2）。**単一アカウントモデルでの防御価値の射程を明記**（下記 †）。複数アカウント運用が一般化した段階で再評価 |
| レート制限層の `Retry-After` 上限 | 30 秒（バックオフ cap と揃える。無限待ちを構造的に排除——決定 5.1） | 複数オペレータ同時ログインの体感待ち時間を実運用で確認して確定 |
| 能動通知への昇格閾値 | 同一アカウントで cap 到達状態が 15 分以上継続、またはレート制限拒否が同水準で継続 | 実運用のアラート疲れ・見落としのバランスで確定 |
| パスワード最小長 | 12 文字以上 | 委任事項 5 のスコープ。実運用の覚えやすさとの兼ね合いで確定 |
| パスワード ブロックリスト | 既知漏洩辞書・頻出パターンとの突合を**必須**（辞書の規模・出典は実装 PR で確定。文字種別の複雑性ルールは課さない——決定 7） | 同梱辞書の規模・更新方針を委任事項 5 で確定 |

† **グローバルトークンバケットの Phase 1 における防御価値の射程**（クリスの指摘 2 への応答）: Phase 1 は単一アカウントモデルであり、実在ユーザー名の候補は事実上「admin」等ごく少数に絞られる——**その 1〜2 候補を当てるだけなら 1 トークン/秒はほぼ防御にならない**。本層が Phase 1 で意味を持つのは「候補を絞り込むための広域なユーザー名探索（多数の候補名を多数の送信元 IP から試す）を、送信元分散に依らずプロセス全体の総試行量で頭打ちにする」場面に限られる。単一の実在名への総当たりはバックオフ（決定 3）が、単一送信元からの高速試行は IP レート制限が主に受け持ち、**本層は「分散した広域探索」という残りの軸を埋める補完層**である。この射程を正直に記す（「三層防御の第三層」という体裁が Phase 1 の実脅威に対して過大評価にならないようにする——クリスの指摘）。単一アカウントモデルの間の暫定値であり、複数アカウント化で候補空間が広がった段階に本層の主眼が移る（再評価トリガ）

## 帰結

- 良くなること:
  - **M-1（正規管理者の締め出し DoS）が構造的に消える**: 完全拒否を行う経路がなくなるため、正しいパスワードを持つ管理者は待てば必ずログインできる
  - **loopback の無条件復旧経路という ADR-0010 の一貫した価値が、バックオフ・レート制限の両面で維持される**（決定 3・4）——#227 で見過ごされていたグローバルトークンバケット側の穴も塞ぐ
  - 非実在ユーザー名を個別状態化しない設計（決定 3）により、バックオフ機構自体が新たなメモリ枯渇 DoS の入口にならない
  - 単一アカウントモデル固有のユーザー名探索攻撃に対して、グローバルトークンバケットという専用の防御層を持つ（クリスの指摘への応答）
  - 攻撃の継続を能動通知へ昇格する経路ができ、無人時間帯の持続的な攻撃が「ログに残っているだけ」で埋もれることを防ぐ
- 悪くなること（受け入れるトレードオフ）:
  - **旧ロックアウト方式が(皮肉にも)持っていた「管理者本人に強制的に気づかせる」副作用が消える**（田中の指摘 4）。能動通知への昇格（決定 6）で補うが、通知を監視していない環境では気づきが遅れる余地は残る
  - **持続的な攻撃が続く間は、正規管理者も cap 近辺の待ち時間を都度負う**（鈴木の指摘 3）。「待てば必ず通る」であって「常に速く通る」ではないという射程を利用者に正確に伝える必要がある
  - **共有アカウントモデルの構造的な副作用**: 単一アカウントを複数人で使う小規模環境では、同僚の誤入力が自分のログイン遅延として波及する（佐藤の指摘 2）。この副作用は本 ADR の方式固有ではなく単一アカウントモデル自体に起因するため、設計変更ではなくドキュメント明記（SEC-D6）で対処する
  - **プロセス再起動が攻撃者にとってのリセット手段になる**（田中の指摘 5）。インメモリ保持である以上、いかなる理由の再起動でもバックオフ・レート制限の状態がゼロへ戻る——他のクラッシュ系不具合の深刻度評価にこの含意（可用性攻撃ではなく認証保護解除の手段になり得る）を波及させる必要がある。この限界自体は #227 が既に明記していたが、記述を「頻繁な再起動で緩む」という受動的な表現から「再起動を誘発できる攻撃者は能動的に解除できる」という前提へ改める
  - **IP レート制限の寛容な仮値（決定 5）は、NAT を伴わない単一送信元からの持続的攻撃に対する制限力をやや弱める**——このトレードオフは、アカウント単位のバックオフを主防御に据える設計（決定 3・5）で補う
  - **反復試行によるユーザー列挙のタイミング非対称を受け入れる（Phase 1 限定。田中の指摘 2）**: 応答の形は実在/非実在で統一する（決定 3）が、実在アカウントは account-keyed バックオフ遅延を、非実在名は IP レート制限・グローバルバケット遅延のみを負うため、**遅延時間の差から反復試行で実在/非実在を推定され得る**。Phase 1 は単一アカウント（実在名は運用者が決めた 1 件）のため実害は限定的だが、複数アカウント化のタイミングで列挙耐性を維持する再設計を要する（再評価トリガに登録）。旧ロックアウト方式も同型の非対称（`LockedOut` は実在名にしか返らない）を既に持っており、本 ADR がこれを新規に悪化させるわけではない
  - **持続的な分散攻撃の最中は、正しいパスワードを持つ *リモート* 管理者も、最初の一回からグローバルバケット涸渇の待ち（有限の `Retry-After`。決定 5.1）に巻き込まれ得る**（リサの指摘 4 への応答）。グローバルトークンバケットは決定 2 の評価順序上パスワード検証より前に消費されるため、**成功・失敗を問わずバケットを通過する全試行がトークンを消費する**——攻撃でバケットが涸れていれば、正規リモート管理者も 429 + `Retry-After` を受けてから通る。loopback は除外される（決定 4）ため物理到達経路は影響を受けないが、「M-1（締め出し）が構造的に消える」という帰結の射程には、この「リモートでは有限時間の待ちが挟まり得る（詰みはしない）」ことが含まれる点を正確に記す
  - `security.md`・ADR-0010 の複数箇所（§2.4 原子的 UPDATE 記述・「ロックアウトは方式ごとに独立」の一文・§4.3 イベント ID 表の 3005 行）が、本 ADR の accepted 後、実装 PR での追従が完了するまで一時的に古い記述を残す（決定 9 で対象を名指し済み。PR #167/#173 の教訓を踏まえ、リサの指摘 3 のとおり `security.md` 改訂を機構実装と同一 PR に閉じ込めることで乖離期間を最小化する。ADR-0010 本文自身のロックアウト依存注記は本 PR で反映済み——決定 9）

## 却下した代替案

- **現行のハードロックアウト維持**: M-1 の核心（正規管理者の締め出し）を放置するため却下
- **ハードロックアウトを (アカウント × リスナ種別) で分離する中間案（(a')）**: M-1 の「完全な締め出し」は消せるが、リモート側のロックアウトが残り、**未認証攻撃者が随意にリモート管理能力を否認できる**（正規管理者を物理到達へ退避させ続けられる）状態を許す。ADR-0010 Phase 2 が売った価値そのものを攻撃者に握らせるため却下（検討した選択肢 (a') の評価のとおり。クリスの再レビュー指摘 1 への正面応答）
- **IP allowlist を失敗試行対策の代替とする案（(e)）**: 到達元を絞る「攻撃面削減」は有用だが、許可 IP 内側の失敗試行対策を代替せず、本 ADR の主対象層（動的 IP・NAT・出張先管理）で静的リスト保守が非現実的、loopback 脅威も覆えないため、**代替としては却下**（補完的 opt-in としては再評価トリガに記録）
- **バックオフのみ（IP レート制限・グローバルトークンバケットなし）**: 単一アカウントモデル固有のユーザー名探索攻撃に無力なため却下（クリスの指摘 4）
- **IP レート制限のみ（バックオフなし）**: 送信元 IP を分散する攻撃に無力なため却下
- **amendment としての起案**: 決定の反転を「詳細化」の粒度に矮小化することになり、README の判定基準・ADR-0002 の先例に反するため却下（オーナー #227 クローズ判断）
- **DB 列を「未使用」のまま温存する設計**: 二重の真実源を生み、既存の実装済み資産（原子的 UPDATE）を無言で死蔵させるため却下（クリスの指摘 2・3。決定 8）
- **非実在ユーザー名にも個別バックオフ状態を持たせる設計**: メモリ枯渇 DoS を新規に生むため却下（田中の指摘 1。決定 3）
- **文字種別（記号・大文字必須）の複雑性ルール**: 予測可能なパターンへ誘導し実効強度をむしろ下げるため却下。長さ下限 + ブロックリストを採る（決定 7。佐藤の質問への回答）
- **loopback 専用の緩いが有限なトークンバケット（Phase 1 で導入）**: 決定 1 の「無条件の最終復旧経路」を Phase 1 の時点で弱めるため、Phase 1 では採らない（完全除外を採る）。複数アカウント化のタイミングで再検討する（田中の指摘 3。決定 4・再評価トリガ）

## 先送りにする場合の再評価トリガ

- **バックオフ・IP レート制限・グローバルトークンバケットの仮値（決定 10）**: 実運用のフィードバック（誤入力の反復頻度・NAT 配下からの正規ログイン試行が 429 に抵触した報告等）が得られた時点で SEC-12 として確定する
- **複数アカウントモデルへの拡張時にまとめて再評価する事項**（ADR-0010 決定 3 が Phase 1 のスコープ外とした複数アカウントモデルへ拡張する段階が共通トリガ）:
  - **グローバルトークンバケットの要否・パラメータ**: 単一アカウントモデル固有の脅威（候補空間が極小）を前提にした本層の主眼が、候補空間の広がりとともに移る（決定 10 †）
  - **反復試行によるユーザー列挙のタイミング非対称の解消**（田中の指摘 2。決定 3・帰結）: 複数アカウントでは非対称がユーザー名探索を容易にするため、列挙耐性を維持する再設計（非実在名への上限付き擬似遅延等）を必須の検討事項とする
  - **loopback 専用の緩いが有限なバケットの導入**（田中の指摘 3。決定 4）: loopback 経由のユーザー名探索が実在の脅威になるため、決定 1 との緊張を「十分寛容だが有限」の値で調停する意味が生じる
- **IP allowlist（到達元制限）の opt-in 追加**（クリスの指摘 1 で言及。検討した選択肢 (e)）: 静的な allowlist を保守できる環境（固定 IP・限られた管理拠点）から要望が反復した場合（目安: 独立利用者から 3 件以上）、失敗試行対策の代替ではなく攻撃面削減の補完的 opt-in として別途設計する
- **能動通知への昇格閾値**: 実運用でアラート疲れ（過敏すぎる）または見落とし（鈍すぎる）のいずれかが観測された場合に閾値を見直す
- **DB 削除マイグレーション後の永続化要否**: 将来、監査・フォレンジック目的でバックオフ状態の永続化が必要という要望が独立した利用者から反復した場合（目安: 3 件以上。既存の再評価トリガと同じ運用）、再度の設計判断（DB 永続化への回帰）を検討する
