# ADR-0006: v1.0 公開基準

- 状態: proposed
- 日付: 2026-07-04
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md) 決定 8（バージョン戦略の委任元）、[ADR-0002](0002-architecture-principles.md)（基準 1 の実装範囲の根拠）、[ADR-0004](0004-security-model.md) 決定 3（opt-in 強化の提供時期）、[ADR-0005](0005-oss-packaging.md)（脆弱性報告窓口）

## 文脈と課題

ADR-0001 決定 8 は「v0.1 から開始し v1.0 公開を目指す。公開基準は別 ADR で定義する」と本 ADR に委任した。「v1.0」は利用者に対する「本番利用を推奨できる」という宣言であり、作者の主観ではなく、**第三者が検証できる証拠**で判定できる基準として先に固定する。基準を後から決めると、達成しやすい形に基準の方が動く。

各基準には**判定の証拠形式**を付す。作者の自己申告に閉じる判定（チェックリストに ✓ を付けるだけ）を排し、リンク可能な記録で充足を示す。

なお ADR-0001 決定 8 が予告した観点名「機能凍結」は、本 ADR では「機能の充足（基準 1）」と「後方互換性の凍結（基準 4）」に分解して整理した（ADR-0001 の改訂履歴 2 に対応を記載）。

## 決定

v1.0 は次の 5 基準を**すべて**満たしたときに公開できる。判定はリリース準備時に行い、各基準の充足根拠（リンク・記録）を公開の場（Issue / Discussion / リリースノート）に残す。

### 基準 1: 機能 — 2 つの原則と約束済み opt-in の充足

- ADR-0001 の**導入体験の原則**（インストール → 最初のログ閲覧、追加サーバ製品・手編集なし）が動作している。**証拠**: この動線を検証する E2E テストの CI 実行記録
- ADR-0001 の**品質の原則**の実装群——ADR-0002 決定 2 の発生箇所別ドロップ計測（**全体設計書で宿題化した OS レベル取りこぼし観測とスプール経路の平常時検証の決着を含む**）と、決定 4 のベンチ CI 合否判定——が動作している。**証拠**: CI のベンチジョブの合格記録。合否は回帰検知にとどめず、**全体設計書で実測により確定した性能目標（公称値）を満たすこと**を含む
- ADR-0002 で v0.1 必須とした範囲（組み込み DB・SQL Server provider・スプール・バックプレッシャ・箇所別計測）に加え、挿入点どまりで出発した**流量制御が完動**している。**証拠**: 対応する受け入れテストの CI 記録
- ADR-0004 決定 3 の **opt-in 強化 3 点（TLS 受信・Web UI の HTTPS・AD 連携認証——AD 環境向けの gMSA 対応を含む）がすべて提供済み**である。**証拠**: 各機能の受け入れテスト記録 + 実環境での動作確認記録（最低 1 環境。作者環境で可、基準 2 の協力環境ならなお良い）

### 基準 2: 実運用実績

- **運用主体が作者および相互に独立した 2 環境以上**で、**連続 30 日以上**の実運用が行われていること。「独立」は運用する組織（人）の独立を指す（同一人物の 2 拠点は 1 環境と数える）
- **試用は既存のログ基盤との並行受信（ミラー）で可**とする。協力者の本番の可用性を Yagura に依存させる必要はない（v0.x が本番推奨でない以上、単独受信を求めるのは矛盾するため）
- 「実運用」の質の下限: **実機器由来のトラフィックを複数送信元から継続的に受信**していること（具体的な下限流量の目安は全体設計書で確定する）
- **判定は沈黙ではなく観測記録で行う**: 運用者は期間の開始・終了時に、発生箇所別ドロップカウンタの値（ゼロ、または説明可能な仕様内ドロップ）を含む報告を残す。報告様式（開始報告 + 完了報告 + 重大事象発生時のみ随時、の計 2〜3 回を想定した最小フォーマット）は試用協力の呼びかけ時に提供する。**証拠**: Issue / Discussion 上の報告記録（非公開環境は運用者の報告で可）
- **重大不具合の定義**: ①観測不能なログ喪失（発生箇所別カウンタに計上されない喪失）、②設計上の想定（上限・流量制御の既定等）を逸脱した喪失、③起動不能、④データ破損（タイムスタンプ・送信元の誤記録を含む）、⑤ログを取り出せない持続的な閲覧不能。**カウンタに正しく計上された仕様内のドロップは重大不具合としない**。期間中に重大不具合が発生した場合、修正後に当該環境の 30 日を再起算する（**不具合が発生していない環境が修正版へ更新すること自体は再起算事由としない**——版をまたいだ継続を認める）
- **全域条件**: 上記 2 環境の内外を問わず、リリース判定時点で**未解決の重大不具合（上記定義）が存在しない**こと（2 環境が無事でも、第 3 の環境で既知のデータ破損が未解決なら公開できない）

### 基準 3: 利用者向け文書の完備

- 導入手順・運用手順（バックアップ・復旧・アップグレード・**手動 DB 移行手順（ADR-0002 決定 3 (iii)）**を含む）・[ADR-0004 委任事項](0004-security-model.md)の利用者向けチェックリスト類が、実際の製品の挙動と一致した状態で揃っている
- **判定**: 作者と利害が独立した**人間** 1 名以上（AI エージェントは判定材料として不可。基準 2 の協力者と兼任可）が、**文書だけを頼りに**導入から運用（アップグレードは v0.x 中の版間更新での確認で可）まで到達できることを実地確認する。**証拠**: 項目別のチェック結果と、確認時点の文書コミット SHA を Issue / Discussion に残す。確認後に該当文書へ実質的な変更が入った場合、変更部分は再確認の対象とする

### 基準 4: 後方互換性の凍結

- 設定スキーマ・DB スキーマ・保存データ形式について、**v1.0 以降は後方互換を破る変更を MAJOR バージョンでのみ行う**運用を開始できる状態にある。**証拠**: 凍結対象の一覧が全体設計書で確定している（リンク）+ v0.x で導入した deprecation の階段がすべて清算済みである（清算 PR のリンク一覧）
- **v0.x → v1.0 のアップグレードで、蓄積データと設定が実際に保全されることを実地に実証する**（v1.0 の最初の利用者が体験するのはこの更新である）。**証拠**: 直前の v0.x からの更新検証の記録

### 基準 5: リリースの完全性 — コード署名を含む

- リリース資産（インストーラ）に **Authenticode コード署名（タイムスタンプ付き）**が付与されている（SmartScreen 警告への対処 + 証明書期限後も署名が有効であり続けるため。証明書の入手・管理はリリース準備の課題として別途扱う）
- リリース資産に checksum（.sha256）が添付され、リリース手順が再現可能な形で文書化されている
- **SECURITY.md が実態と一致している**: リリース提供の開始を反映し、**v1.0 以降の修正提供対象（サポート対象バージョン。最低限「最新リリース + main」の明記）が確定**している
- **証拠**: リリースノートから署名の検証手順（signtool verify 等）と .sha256 が辿れること、SECURITY.md の該当節へのリンク

## 運用

- **v0.x の間**: MINOR で機能追加し、後方互換を破る変更も許容する（ただし conventions.md の deprecation 階段は v0.x でも守る）。「v0.x は本番推奨ではない」ことは README の状態表示で維持する
- 基準は**満たすまで下げない**。変更（緩和を含む）は本 ADR の supersession によってのみ行う。なお、決定者と supersession の承認者が同一人物である以上、この拘束は手続きと公開記録による自己拘束であることを自覚的に記す——だからこそ変更は必ず公開の場（PR + ペルソナレビュー）を通す
- 基準 2 の実績集めは、v0.x 後半で「試用協力の呼びかけ」（報告様式・並行受信可の明示を含む）を README 等で行う。呼びかけでは **SQL Server provider 構成の環境を優先的に募る**（判定基準ではなく運用上の努力目標）
- **量的な床も自己拘束の内側に置く**: v1.0 判定に用いる性能公称値（基準 1）と実運用の下限流量（基準 2）は全体設計書で確定するが、**確定後の引き下げは本 ADR の変更と同等に扱い、supersession に準ずる手続き（PR + ペルソナレビュー + 変更理由の公開）を要する**（基準の骨格に鍵を掛けても量的な床が動けば合格ラインが動く、という抜け道の封鎖）

## 検討した選択肢

### 基準の性格

- **(a) 証拠形式付きの 5 基準（採用）**: 判定の恣意性を排し、利用者への宣言として第三者が検証可能にする
- (b) オーナーの総合判断: 基準が動く。作者の「完成した感」は利用者の「本番で使える」と一致しない。却下
- (c) 期日ベース（例: 20XX 年に v1.0）: 品質と無関係に版番号が進む。却下

### 実運用実績の水準

- **(a) 独立 2 環境 × 30 日 + 観測記録による判定（採用）**: 個人運用の OSS が現実に集められる水準で、かつ「作者の環境でしか動かない」「見ていなければ合格」の両方を排除できる最小線
- (b) より多く（例: 5 環境 × 90 日）: 理想だが、利用者ゼロから始まる OSS には到達困難な基準であり、実質的に v1.0 を無期限化する。却下
- (c) 作者環境のみ: 「独立した環境で動く」ことの検証にならない。却下

## 帰結

- 良くなること: 「v1.0 = 本番推奨」の意味が検証可能になり、利用者・貢献者・作者の期待値が揃う。v0.x の破壊的変更の自由も明確になる
- 悪くなること: 基準 2（外部の実運用実績）と基準 5（コード署名証明書の入手）は作者の実装努力だけでは満たせず、v1.0 の時期が外部要因に依存する。証拠形式の要求により、リリース準備の記録コストが増える
- 受け入れるリスク: 試用協力者が現れない場合、v1.0 が長期化する。まず呼びかけの方法を工夫する側で対処し、**12 か月経過時は再評価トリガに従う**（基準の変更はその場合も supersession による）

## 先送りにする場合の再評価トリガ

- 基準 5 のコード署名: 証明書の入手性・費用が個人 OSS に対して非現実的と判明した場合（目安: 年間費用が継続不能な水準、または個人への発行要件を満たせない）、変更を supersession で検討する。**その場合の歯止め条件**: ①無償・低コストの代替（Sigstore 系の署名・attestation、Azure Trusted Signing の個人利用可否等）の検討記録を残すこと、②署名なしを選ぶ場合は「v1.0 の延期」を対等な選択肢として併記・比較すること、③それでも署名なしで公開するなら README・リリースノートに「署名なし」の事実と検証手順を明示すること（**署名なしへの変更は代替手段の差し替えではなく「本番推奨」の意味の縮小であり、利用者への明示義務を伴う**）
- 基準 2 の水準: v0.x 公開から 12 か月を経ても試用環境が現れない場合、呼びかけ方法と基準の現実性を再評価する（基準を変更する場合も supersession による）
