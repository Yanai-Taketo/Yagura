# ADR-0006: v1.0 公開基準

- 状態: proposed
- 日付: 2026-07-04
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md) 決定 8（バージョン戦略の委任元）、[ADR-0004](0004-security-model.md) 決定 3（opt-in 強化の提供時期）

## 文脈と課題

ADR-0001 決定 8 は「v0.1 から開始し v1.0 公開を目指す。公開基準は別 ADR で定義する」と本 ADR に委任した。「v1.0」は利用者に対する「本番利用を推奨できる」という宣言であり、主観（作者が完成したと感じた）ではなく、**満たしたか否かを機械的に判定できる基準**として先に固定する。基準を後から決めると、達成しやすい形に基準の方が動く。

## 決定

v1.0 は次の 5 基準を**すべて**満たしたときに公開できる。判定はリリース準備時のチェックリストで行い、各基準の充足根拠（リンク・記録）を残す。

### 基準 1: 機能 — 2 つの原則と約束済み opt-in の充足

- ADR-0001 の**導入体験の原則**（インストール → 最初のログ閲覧、追加サーバ製品・手編集なし）と**品質の原則**（発生箇所別のドロップ計測 + ベンチの CI 合否判定）が、実装として動作している
- ADR-0002 で v0.1 必須とした範囲（組み込み DB・SQL Server provider・スプール・バックプレッシャ・箇所別計測）に加え、挿入点どまりで出発した**流量制御が完動**している
- ADR-0004 決定 3 の **opt-in 強化 3 点（TLS 受信・Web UI の HTTPS・AD 連携認証）がすべて提供済み**である

### 基準 2: 実運用実績

- **作者以外の独立した 2 環境以上**で、**連続 30 日以上**の実運用（実トラフィックの受信・閲覧）が行われ、その間に**未解決の重大不具合（ログ喪失・起動不能・データ破損）がない**こと
- 実績は Issue / Discussion 等の公開記録で確認できる形とする（非公開環境の場合は運用者の報告で可）

### 基準 3: 利用者向け文書の完備

- 導入手順・運用手順（バックアップ・復旧・アップグレード含む）・[ADR-0004 委任事項](0004-security-model.md)の利用者向けチェックリスト類が、**実際の製品の挙動と一致した状態で**揃っている
- 判定は「文書だけを見て第三者が導入から運用まで到達できるか」を実地に確認した記録による

### 基準 4: 後方互換性の凍結

- 設定スキーマ・DB スキーマ・保存データ形式について、**v1.0 以降は後方互換を破る変更を MAJOR バージョンでのみ行う**運用を開始できる状態にある（凍結対象の一覧が全体設計書で確定しており、v0.x で導入した deprecation の階段がすべて清算済み）

### 基準 5: リリースの完全性 — コード署名を含む

- リリース資産（インストーラ）に **Authenticode コード署名**が付与されている（SmartScreen 警告への対処。証明書の入手・管理はリリース準備の課題として別途扱う）
- リリース資産に checksum（.sha256）が添付され、リリース手順が再現可能な形で文書化されている

## 運用

- **v0.x の間**: MINOR で機能追加し、後方互換を破る変更も許容する（ただし conventions.md の deprecation 階段は v0.x でも守る）。「v0.x は本番推奨ではない」ことは README の状態表示で維持する
- 基準は**満たすまで下げない**。どうしても変更が必要な場合は本 ADR の supersession による（なし崩しの緩和をしない）
- 基準 2 の実績集めは、v0.x 後半で「試用協力の呼びかけ」を README 等で行う（実績ゼロのまま v1.0 直前に慌てない）

## 検討した選択肢

### 基準の性格

- **(a) 機械的に判定できる 5 基準（採用）**: 判定の恣意性を排し、利用者への宣言として検証可能にする
- (b) オーナーの総合判断: 基準が動く。作者の「完成した感」は利用者の「本番で使える」と一致しない。却下
- (c) 期日ベース（例: 20XX 年に v1.0）: 品質と無関係に版番号が進む。却下

### 実運用実績の水準

- **(a) 独立 2 環境 × 30 日（採用）**: 個人運用の OSS が現実に集められる水準で、かつ「作者の環境でしか動かない」を排除できる最小線
- (b) より多く（例: 5 環境 × 90 日）: 理想だが、利用者ゼロから始まる OSS には到達困難な基準であり、実質的に v1.0 を無期限化する。却下
- (c) 作者環境のみ: 「独立した環境で動く」ことの検証にならない。却下

## 帰結

- 良くなること: 「v1.0 = 本番推奨」の意味が検証可能になり、利用者・貢献者・作者の期待値が揃う。v0.x の破壊的変更の自由も明確になる
- 悪くなること: 基準 2（外部の実運用実績）と基準 5（コード署名証明書の入手）は作者の実装努力だけでは満たせず、v1.0 の時期が外部要因に依存する
- 受け入れるリスク: 試用協力者が現れない場合、v1.0 が長期化する。その場合も基準を下げず、呼びかけの方法を工夫する側で対処する

## 先送りにする場合の再評価トリガ

- 基準 5 のコード署名: 証明書の入手性・費用が個人 OSS に対して非現実的と判明した場合（目安: 年間費用が継続不能な水準、または個人への発行要件を満たせない）、署名の代替（実行ファイルの checksum 検証手順の強化等）への変更を supersession で検討する
- 基準 2 の水準: v0.x 公開から 12 か月を経ても試用環境が現れない場合、呼びかけ方法と基準の現実性を再評価する（基準を下げる場合も supersession による）
