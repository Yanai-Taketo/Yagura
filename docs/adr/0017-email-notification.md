# ADR-0017: 能動通知のメール送信チャネル（SMTP。opt-in）

- 状態: proposed
- 日付: 2026-07-18
- 決定者: YANAI Taketo
- 関連: architecture.md §4.6（能動通知）/ ADR-0001（スコープ制御）/ ADR-0004 決定 5（資格情報の保管）/ security.md §4.3（イベント ID 採番）/ configuration.md §2・§3 / Issue #273

## 文脈と課題

architecture.md §4.6 の能動通知は「UI を見ていない夜間でも対応が必要な事象に気づける」ことを存在意義とするが、書き出し先は **Windows イベントログのみ**である。イベントログ転送・監視基盤（SCOM・Zabbix・イベントログ転送 + 集約等）を持つ環境ではこれで実効性があるが、本プロジェクトの主対象である中小規模の Windows-shop（ADR-0001。ひとり情シス〜小規模チーム）では、**イベントログを常時監視する仕組み自体が存在しない**ことが多い。その現場では現状の能動通知は実質「誰も気づかない」に近い（Issue #273。2026-07-16 の棚卸しで空白と認定）。

採用自体は 2026-07-16 のオーナー裁定で確定済み: SMTP によるメール通知は相関分析・インシデント管理のような商用 SIEM 領域（ADR-0001 の明示 NG）ではなく、**syslog サーバの基本機能の範囲**である（Kiwi Syslog Server 等の商用の同種製品もメール通知を基本機能として持つ）。本 ADR はその実現方式——opt-in の形・SMTP-AUTH / STARTTLS の扱い・認証情報の保管・送信失敗時の縮退・通知トリガ集合——を決める。

### スコープ（非目標）

商用 SIEM / アラート管理領域に踏み込まない線引きを最初に固定する:

- **ログ内容に基づくアラートルールは作らない**（「特定の文字列を含む syslog が来たらメール」等）。通知対象はあくまで architecture.md §4.6 の能動通知（= Yagura 自身の運用状態の異常）である
- **エスカレーション・オンコール輪番・確認応答（acknowledge）は作らない**
- **メール以外のチャネル（Slack / Teams / 汎用 webhook）は本 ADR では作らない**（再評価トリガは末尾）
- **トリガ単位の宛先振り分け・通知ルール編集 UI は作らない**（送るか送らないかの一括 opt-in のみ）

## 検討した選択肢

### 1. チャネルの構造 — 独自トリガ体系 vs 既存能動通知の第 2 の書き出し先

- **(a) メール専用の独自トリガ体系を新設する**: メール用に「何を通知するか」の条件・閾値を別に定義する。柔軟だが、トリガ定義が二重管理になり、閾値・抑制窓のチューニング面も倍になる。「通知ルールの編集」という SIEM 的 UI へ滑り出す入口にもなる。**却下**
- **(b) 既存の能動通知事象をそのまま第 2 の書き出し先（メール）へも送る**: トリガの定義・評価・抑制窓は既存の能動通知機構（`ActiveNotificationMonitor` の周期監視 + 発生箇所からの即時通知）を唯一の正とし、メールはイベントログと並ぶ**書き出し先の追加**に徹する。定義の二重化がなく、イベントログとメールで「見える事象」が食い違わない。**採用**
- **(c) 作らない（イベントログ転送・OS のタスクスケジューラのイベントトリガ起動に委ねる）**: 追加コード 0 だが、対象現場（監視基盤なし）では「イベントトリガ + PowerShell で Send-MailMessage」を各自組むことになり、導入体験の思想（ADR-0004 決定 2）に反する。オーナー裁定（2026-07-16）で採用が確定しており、却下の経緯として記録する

### 2. 送信の実装 — 依存ライブラリ

- **(a) `System.Net.Mail.SmtpClient`（BCL 標準）**: 追加依存 0。ただし公式ドキュメントが新規コードでの使用を推奨せず（品質・近代プロトコル対応の理由で MailKit を明示推奨）、廃止予告付きの API に新機能を載せるのは v1.0 を目指す本体には不適。**却下**
- **(b) MailKit**: .NET のメール送信のデファクト。SMTP-AUTH（PLAIN / LOGIN / CRAM-MD5 / OAuth 等）・STARTTLS・implicit TLS を網羅し、Microsoft 公式ドキュメントが `SmtpClient` の代替として名指しする。**採用**。依存追加の作法として、**実装 PR の時点で当日の最新版をライブ検証してから採用版を確定する**（CLAUDE.md の執筆スタイル規約。本 ADR ではバージョンを固定しない）。供給網統治は既存の枠組み（Dependabot + `Directory.Packages.props` の集中管理 + dependency-review）に乗せる

### 3. 暗号化（STARTTLS）の既定

対象現場の SMTP サーバは「社内 Exchange / 内部リレーの 25 番・無認証・STARTTLS 任意」から「submission 587・AUTH 必須・STARTTLS 必須」まで幅がある。

- **(a) STARTTLS 必須固定**: 最も安全だが、STARTTLS を提供しない内部リレーで機能自体が使えない。信頼ネットワーク前提 + opt-in 強化（ADR-0004）の既定思想とも整合しない。**却下**
- **(b) 3 値の明示設定**: `none`（使わない）/ `auto`（サーバが対応すれば使う = opportunistic。**既定**）/ `required`（必須。確立できなければ送信失敗として扱う）。既定 `auto` は「内部リレーでもそのまま動き、対応サーバなら自動で暗号化される」——平文受信が既定の syslog 本体と同じ「既定は動くこと・強化は opt-in」の整理。認証情報を設定した場合の降格リスク（後述の帰結）は文書で明示する。**採用**
- implicit TLS（SMTPS 465）は初版では対象外とする（対象現場での必要度が低く、`required` + 587 で実用は満たせる。要望が出た時点で `Security` の第 4 値として追加できる）

### 4. SMTP-AUTH と認証情報の保管

- **認証は任意**とする（ユーザー名・パスワードが設定されていれば AUTH を行い、未設定なら匿名で送る）。無認証の内部リレーが対象現場の多数派である現実に合わせる
- **パスワードの保管は ADR-0004 決定 5 の確立様式を踏襲する**: UI（管理画面）経由の入力は常に DPAPI で暗号化して `dpapi:<Base64>` 形式で保存し、手編集の平文は受理しつつ警告する（`DpapiConnectionStringProtector` の再利用。SQL Server 接続文字列で確立済み——configuration.md §2）。専用の秘密ストアを新設しない
- パスワード値はログ・監査・UI 再表示のいずれにも出さない（configuration.md §2 の「パスワード値そのものは記録しない」に従う）

### 5. 送信失敗時の縮退

- **送信は受信・保存経路から完全に独立させる**: 通知イベントは有界のメモリキュー（上限は仮値。超過時は古い側を保全して新規を破棄し、破棄数を計器に計上——監査の SEC-10 と同じ流儀）へ積み、専用の背景処理が送る。SMTP サーバ停止・応答遅延が受信スループット・UI に影響する経路を構造的に作らない
- **メール送信の失敗はメールで通知しない**（ループの構造的排除）。失敗はイベントログの警告（新イベント ID・抑制窓付き）+ ライブ計器で観測可能にする——「通知チャネル自体の障害」はイベントログ・ゲージという既存の独立チャネルが受け持つ（architecture.md §4.6 の多段の整理と同じ）
- 再試行は有界（仮値: 1 通あたり 1 回）。それでも失敗した通知は破棄する——メールは「気づくきっかけ」であり記録の正本ではない（正本はイベントログと監査記録）。破棄しても事象自体はイベントログに残っているため情報は失われない

### 6. 通知トリガ集合

- **既存の能動通知事象の全部を対象とする（トリガ選別 UI は作らない）**: 対象は architecture.md §4.6 の能動通知としてイベントログへ警告以上で書き出される事象（スプール系・縮退起動・自己検証失敗・証明書期限接近・稼働中の証明書使用不能・circuit 上限・認証防御の昇格等）。「どれを送るか」を選ばせる UI は、通知ルール編集（非目標）の入口になるため作らない
- 抑制窓は既存トリガごとの窓をそのまま共有する（イベントログに 1 件ならメールも最大 1 通）。加えてメール独自の**全体流量上限**（仮値: 1 時間あたり 10 通。超過分は送らずイベントログへ「抑制した」事実を 1 回だけ警告）を置く——障害の連鎖でメールボックスを溢れさせない（証跡の希釈対策 security.md §4.4 と同じ向き）

## 決定

### 1. 能動通知の第 2 の書き出し先としてメール送信を追加する（opt-in・既定無効）

- 新設キー `Notification:Email:*`（既定 `Enabled=false`）。ゼロ設定ファーストランには一切現れない
- トリガの定義・評価は既存の能動通知機構を唯一の正とし、メールは書き出し先の追加に徹する（選択肢 1-(b)）
- 通知メールの本文は、イベントログに書くのと同じ情報（事象の説明・発生時刻・対象・イベント ID）のプレーンテキストとする。HTML テンプレート・カスタマイズは作らない

### 2. 接続と暗号化

- `Notification:Email:SmtpHost` / `SmtpPort`（既定 25）/ `Security` = `none | auto | required`（既定 `auto`）
- `required` で STARTTLS を確立できない場合は**送信失敗として扱う**（平文へ降格しない——fail-closed）。`auto` の降格リスクは利用者向けドキュメントに明示する

### 3. SMTP-AUTH と資格情報

- `Notification:Email:Username` / `Password`（任意。両方設定されたときのみ AUTH）
- `Password` は ADR-0004 決定 5 の様式（UI 経由は `dpapi:<Base64>`、手編集平文は受理 + 警告、値は一切記録しない）を踏襲する

### 4. 宛先・差出人

- `Notification:Email:From` / `Notification:Email:To`（配列。複数宛先可）。宛先ごとの振り分けはしない
- 設定変更は既存監査（ウィザード外の設定面のため 2016 = 再読み込み、または管理画面の設定保存経路）にキー名粒度で残る。メール専用の監査 ID は新設しない

### 5. 送信の独立性と縮退

- 有界キュー + 背景送信。受信・保存・UI への影響経路を作らない（選択肢 5）
- 失敗はイベントログ警告（新 1000 番台 ID。抑制窓付き）+ ライブ計器。**メール失敗はメールで通知しない**
- 再試行は有界（仮値 1 回）、失敗通知は破棄（正本はイベントログ・監査記録）

### 6. 反映方式とテスト送信

- `Notification:Email:*` は**即時反映**（CF-4 の `ImmediateConfigurationApplier` に登録。送信クライアントは接続を保持しないため参照交換で足りる）
- 管理画面に**テスト送信**（現在の設定で 1 通送る）を置く。結果（成功 / 失敗と理由）を画面に即時表示する——「夜間に初めて失敗に気づく」を設定時に潰す。テスト送信は状態を変えない操作のため監査対象にしない（実行はログに情報レベルで残す）

### 7. 実装の位置

- 送信の実体は Host（`Yagura.Host.Observability.ActiveNotification` 配下）。既存の能動通知の書き出し点（イベントログへ書く箇所）に書き出し先抽象を挟むのではなく、**通知事象を受け取る小さな内部フック**を足す（イベントログ書き出しは `ILogger` レールのまま変えない——既存の観測経路を実装し直すリスクを取らない）
- UI（設定画面・テスト送信）は既存の管理画面パターン（`Yagura.Abstractions` 契約 + Host 実体 + `Yagura.Web` 画面）に従う

## 帰結

- 良くなること: 監視基盤を持たない現場でも、夜間のスプール成長・証明書期限・認証攻撃の予兆に**人が届く経路**で気づける。能動通知の存在意義（§4.6）が対象現場で初めて実効化する
- 悪くなること・受け入れるリスク:
  - 外部依存（MailKit)が 1 つ増える（供給網統治は既存の枠組みに乗せる）
  - `Security=auto` の既定は STARTTLS ストリップ（中間者による平文降格）に対して無防備である。信頼ネットワーク前提（ADR-0004)の範囲内と整理し、敵対的経路を挟む構成では `required` を使うことを文書で案内する
  - パスワードを設定ファイルに置く形式（DPAPI 暗号化）はサーバ束縛であり、構成配布運用と両立しない——既知の限界（configuration.md §2 CF-D3）をメールにも延長する
  - メールは at-most-once（有界再試行 + 破棄）。「メールが来ない = 正常」とは限らない——正本はイベントログである旨を利用者向けドキュメントに明示する

## 先送りにする場合の再評価トリガ

- **メール以外のチャネル（Slack / Teams / webhook）**: 独立した利用者からの要望 3 件以上で個別 ADR として再評価（本 ADR の書き出し先追加の構造——選択肢 1-(b)——はチャネル追加の挿入点を既に持つ）
- **トリガ選別・宛先振り分け**: 同じく要望 3 件以上で再評価。その際も「ログ内容に基づくルール」は NG 線（ADR-0001）のまま
- **implicit TLS（SMTPS 465）**: 対応が必要な利用者が現れた時点で `Security` の第 4 値として追加（構造変更を伴わない）
- **OAuth 2.0（Microsoft 365 の基本認証廃止方向）**: 対象現場で M365 経由送信の要望が反復した時点で再評価。初版は SMTP-AUTH（Basic）+ 社内リレーを主経路とする
