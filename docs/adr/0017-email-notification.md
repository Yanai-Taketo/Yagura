# ADR-0017: 能動通知のメール送信チャネル（SMTP。opt-in）

- 状態: accepted（2026-07-18。PR #337 マージ）
- 日付: 2026-07-18
- 決定者: YANAI Taketo
- 関連: [architecture.md](../design/architecture.md) §4.6（能動通知）/ [ADR-0001](0001-project-founding.md)（スコープ制御）/ [ADR-0004](0004-security-model.md) 決定 5（資格情報の保管）・決定 7（監査）/ [security.md](../design/security.md) §1.1（外向き通信台帳）・§4.3（イベント ID 採番）/ [configuration.md](../design/configuration.md) §1・§2・§3・§8 / Issue #273 / PR #337

## 文脈と課題

architecture.md §4.6 の能動通知は「UI を見ていない夜間でも対応が必要な事象に気づける」ことを存在意義とするが、書き出し先は **Windows イベントログのみ**である。イベントログ転送・監視基盤（SCOM・Zabbix・イベントログ転送 + 集約等）を持つ環境ではこれで実効性があるが、本プロジェクトの主対象である中小規模の Windows-shop（ADR-0001。ひとり情シス〜小規模チーム）では、イベントログを常時監視する仕組みを持たないことが多いと想定される。その現場では現状の能動通知は実質「誰も気づかない」に近い（Issue #273。2026-07-16 の棚卸しで空白と認定）。

採用自体は 2026-07-16 のオーナー裁定で確定済み: SMTP によるメール通知は相関分析・インシデント管理のような商用 SIEM 領域（ADR-0001 の明示 NG）ではなく、**syslog サーバの基本機能の範囲**である（Kiwi Syslog Server 等の商用の同種製品もメール通知を基本機能として持つ）。本 ADR はその実現方式——opt-in の形・SMTP-AUTH / STARTTLS の扱い・認証情報の保管・送信失敗時の縮退・通知トリガ集合——を決める。

なお本機能は、逆引き DNS（ADR-0007）に続く**製品として 2 例目の自発的外向き通信であり、利用者が宛先を設定する型としては初**である。この性質（外向き通信台帳・監査・信頼ネットワーク境界の外への情報流出）への統制を決定に含める（ペルソナレビュー 田中の指摘）。

### スコープ（非目標）

商用 SIEM / アラート管理領域に踏み込まない線引きを最初に固定する:

- **ログ内容に基づくアラートルールは作らない**（「特定の文字列を含む syslog が来たらメール」等）。通知対象はあくまで architecture.md §4.6 の能動通知（= Yagura 自身の運用状態の異常）である
- **エスカレーション・オンコール輪番・確認応答（acknowledge）は作らない**
- **メール以外のチャネル（Slack / Teams / 汎用 webhook）は本 ADR では作らない**（再評価トリガは末尾）
- **トリガ単位の宛先振り分け・通知ルール編集 UI は作らない**（送るか送らないかの一括 opt-in のみ）

## 検討した選択肢

### 1. チャネルの構造 — 独自トリガ体系 vs 既存能動通知の第 2 の書き出し先

- **(a) メール専用の独自トリガ体系を新設する**: メール用に「何を通知するか」の条件・閾値を別に定義する。柔軟だが、トリガ定義が二重管理になり、閾値・抑制窓のチューニング面も倍になる。「通知ルールの編集」という SIEM 的 UI へ滑り出す入口にもなる。**却下**
- **(b) 既存の能動通知事象をそのまま第 2 の書き出し先（メール）へも送る**: トリガの定義・評価・抑制窓は既存の能動通知機構（`ActiveNotificationMonitor` の周期監視 + 発生箇所からの即時通知）を唯一の正とし、メールはイベントログと並ぶ**書き出し先の追加**に徹する。定義の二重化がなく、イベントログとメールで「見える事象」が食い違わない。**採用**
- **(c) 作らない（イベントログ転送・OS のタスクスケジューラのイベントトリガ起動に委ねる）**: 追加コード 0。ただし対象現場（監視基盤なし）が自前で組む場合、①イベントトリガのタスクを対象イベント ID ごとに設定する必要があり（十数件）、②抑制・流量制御がなく、③送信スクリプトに SMTP 資格情報が平文で置かれがちで、④送信の標準手段だった `Send-MailMessage` は Microsoft 自身が非推奨化済み（本 ADR が `SmtpClient` を却下したのと同じ系統の理由）——「各自組む」の実質難度は低くない。オーナー裁定（2026-07-16）で採用が確定しており、却下の経緯として記録する。なお**監視基盤を持つ現場では、メールを有効化せずイベントログ転送のレールに乗せることが引き続き第一選択**である（利用者向けドキュメントに明記する——申し送り D-4）

### 2. 送信の実装 — 依存ライブラリ

- **(a) `System.Net.Mail.SmtpClient`（BCL 標準）**: 追加依存 0。ただし公式ドキュメントが新規開発での使用を明示的に非推奨とする——「We don't recommend that you use the `SmtpClient` class for new development because `SmtpClient` doesn't support many modern protocols. Use MailKit or other libraries instead.」（learn.microsoft.com の SmtpClient クラス Remarks。2026-07-18 確認）。廃止予告付きの API に新機能を載せるのは v1.0 を目指す本体には不適。**却下**
- **(b) MailKit**: .NET のメール送信のデファクトであり、上記の公式ドキュメントが代替として名指しする。SMTP-AUTH（PLAIN / LOGIN / CRAM-MD5 / OAuth 等）・STARTTLS を網羅する。**採用**。依存追加の作法として、**実装 PR の時点で当日の最新版をライブ検証してから採用版を確定する**（CLAUDE.md の執筆スタイル規約。本 ADR ではバージョンを固定しない）。推移的依存（MimeKit・暗号系）を含む依存ツリー全体を実装 PR に記録し、`Directory.Packages.props` の集中管理 + Dependabot + dependency-review の既存供給網統治に乗せる

### 3. 暗号化（STARTTLS）の既定

対象現場の SMTP サーバは「社内 Exchange / 内部リレーの 25 番・無認証・STARTTLS 任意」から「submission 587・AUTH 必須・STARTTLS 必須」まで幅がある。

- **(a) STARTTLS 必須固定**: 最も安全だが、STARTTLS を提供しない内部リレーで機能自体が使えない。信頼ネットワーク前提 + opt-in 強化（ADR-0004）の既定思想とも整合しない。**却下**
- **(b) 3 値の明示設定**: `none`（使わない）/ `auto`（サーバが対応すれば使う = opportunistic。**既定**）/ `required`（必須。確立できなければ送信失敗として扱う）。既定 `auto` は「内部リレーでもそのまま動き、対応サーバなら自動で暗号化される」——平文受信が既定の syslog 本体と同じ「既定は動くこと・強化は opt-in」の整理。**採用**。ただし資格情報を設定した構成での `auto` は STARTTLS ストリップにより**資格情報自体**が平文で流れ得るため、能動的な警告を課す（決定 3）
- implicit TLS（SMTPS 465）は初版では対象外とする（`required` + 587 で実用は満たせる。要望が出た時点で `Security` の第 4 値として追加できる）

### 4. SMTP-AUTH の必須性

- **(a) AUTH 必須**: 無認証の内部リレー（Exchange の受信コネクタ等、Windows 環境で広く使われる想定）で使えない。**却下**
- **(b) 任意**（ユーザー名・パスワードが設定されていれば AUTH、未設定なら匿名）: 内部リレー・submission の両構成に 1 つのキー体系で対応できる。**採用**

### 5. 送信失敗時の扱い

- **(a) 永続キュー + 無制限再試行**: 「必ず届ける」に寄せる案。メールを記録の正本に近づけてしまい、ディスク管理・順序・重複の複雑さを持ち込む。メールは「気づくきっかけ」であり、正本はイベントログと監査記録である。**却下**
- **(b) 有界メモリキュー + 有界再試行 + 破棄（at-most-once）**: 送信を受信・保存経路から完全に独立させ、失敗の限界を明示する。**採用**（詳細は決定 5。破棄しても事象自体はイベントログに残る）

### 6. 通知トリガ集合

- **(a) トリガ選別 UI**: 「どれを送るか」を選ばせる。通知ルール編集（非目標）の入口になる。syslog 導入初心者には選別の判断材料もない。**却下**
- **(b) 対象イベント ID の明示リスト（allowlist）で「全部」を定義する**: 選別 UI は作らず、製品として対象を固定する。**採用**。「全部」を「1000 番台」「警告以上」のような性質で定義すると、能動通知でない運用警告（1020 再起動待ち残存・1021 再読み込み拒否等）や**メール送信失敗の新 ID 自身**が混入し、ループ排除が言葉の解釈に依存してしまう（ペルソナレビュー クリス・リサの指摘）——membership は明示リストで固定し、メール送信失敗 ID をリストに含めないことで**ループを定義レベルで排除**する（決定 6）

## 決定

### 1. 能動通知の第 2 の書き出し先としてメール送信を追加する（opt-in・既定無効）

- 新設キー `Notification:Email:*`（既定 `Enabled=false`）。ゼロ設定ファーストランには一切現れない
- トリガの定義・評価は既存の能動通知機構を唯一の正とし、メールは書き出し先の追加に徹する（選択肢 1-(b)）
- 通知メールの本文は、イベントログに書くのと同じ情報（事象の説明・発生時刻・イベント ID・本文メッセージ）の**日本語プレーンテキスト**とする（イベントログ本文の日本語化と同じ整理。多言語化は意図的先送りに従う）。HTML テンプレート・カスタマイズは作らない。本文に載せるフィールドの上限列挙は実装 PR で固定し、以後 additive にのみ増やす（受信 syslog の本文・資格情報・接続文字列は構造的に載らない——載せる情報源をイベントログ書き出しと同一にすることで担保する）

### 2. 接続と暗号化

- 新設キーは `Notification:Email:{Enabled, From, To}` + `Notification:Email:Smtp:{Host, Port, Security, Username, Password}` の 8 キー（SMTP 接続属性は `Smtp` 階層へまとめ、既存体系の階層切りに合わせる）
- `Port` 既定 25、`Security` = `none | auto | required`（既定 `auto`）
- `required`: サーバ証明書の検証は**既定有効**。STARTTLS を確立できない・証明書検証に失敗した場合は**送信失敗として扱う**（平文へ降格しない——fail-closed）。社内 CA・自己署名証明書への緩和を設ける場合は独立した opt-in キー（明示設定 + 警告付き）とし、方式は実装設計で確定する（委任 3）
- `auto`: STARTTLS を試行し、確立できない場合（サーバ非対応・証明書検証失敗を含む）は**平文で送信を継続する**（opportunistic の定義どおり。この降格リスクは帰結と利用者向けドキュメントに明示する）
- **不正時挙動（configuration.md §1 の 3 分類の宣言）**: `Notification:Email:*` 配下のキーの不正値（`Security` の未知値・`Port` の範囲外・`From`/`To` の形式不正）、および `Enabled=true` で必須項目（`Smtp:Host`・`From`・`To`）が欠落した構成は、**メール通知機能を無効化して警告する**（縮小側で継続。受信・保存には一切影響しない。緩い側の `auto` へも黙って倒さず、送信が黙って死ぬ `required` へも倒さない——「設定したのに送られていない」を警告で必ず見せる）

### 3. SMTP-AUTH と資格情報

- `Smtp:Username` / `Smtp:Password`（任意。両方設定されたときのみ AUTH）。**片方のみ設定された構成は不正構成として決定 2 の規則（機能無効化 + 警告）で扱う**——黙って匿名送信へ落とさない（「設定したのに使われていない」を作らない）
- `Password` は ADR-0004 決定 5 の様式を踏襲する: UI 経由の入力は常に DPAPI で暗号化して `dpapi:<Base64>` 形式で保存し、手編集の平文は受理しつつ警告する。DPAPI entropy は SQL Server 用と分離した専用名前空間 **`Yagura:Notification:Email:Smtp:Password:v1`** とする（版管理付き。`DpapiConnectionStringProtector` の機構を一般化して再利用）
- **復号失敗時（改ざん・破損・別マシンへの構成コピー）はメール送信を停止して警告する**（fail-notify）。無認証送信への黙示フォールバックはしない（誤送信・送信元アカウントのロック誘発を避ける）
- **`Password` 設定済み + `Security` ≠ `required` の構成は、設定保存時・起動時・再読み込み時に警告する**（configuration.md §1 の「能動的に見せる警告」の系。STARTTLS ストリップで漏れるのは通知内容ではなく SMTP 資格情報——多くの環境では AD アカウント——である旨を警告文言に含める）
- パスワード値はログ・監査・UI 再表示のいずれにも出さない。送信失敗イベントに SMTP サーバ応答を含める場合、AUTH 交換部分は記録しない（実装要件）

### 4. 宛先・差出人と監査

- `From` / `To`（`To` は JSON 配列。既存の配列キー——`Admin:Authentication:Windows:AdminGroups` 等——と**同じ JSON 表現**。ただし既存の配列キーは手編集のみで、**配列キーを UI から保存する経路は本 ADR が初めて要求する**——実装は委任 9）。宛先ごとの振り分けはしない
- **メール通知設定の変更は専用の監査 ID（2000 番台新設）で、変更キーと新値を含めて記録する**（`Password` は変更の事実のみ）。宛先・接続先は「通知がどこへ向かうか」= 流出経路そのものの定義であり、キー名粒度（2016 の「前後値を含めない」）では事後に追えない——値を残す前例は ADR-0012 決定 7（2012 = 証明書拇印・ポートの新値を記録）に既にある。手編集 + 再読み込み経路は従来どおり 2016（キー名粒度）に合流し、手編集 + 再起動は起動時差分照合 2019（Issue #329）が変更キー名を残す（この経路の限界は security.md §4.1 の受容のまま）
- **外向き通信台帳（security.md §1.1）を拡張し、SMTP 送信を登録する**（宛先 = 利用者設定の SMTP サーバ・契機 = 能動通知/テスト送信・無効化手段 = `Enabled=false`）。台帳の現行スコープ（閲覧層）を製品全体へ広げるか Host 側の相当欄を新設するかは実装 PR で確定する（委任 4）

### 5. 送信の独立性と縮退

- **有界メモリキュー（上限 仮値 64 件）+ 背景送信**。受信・保存・UI への影響経路を作らない。キュー溢れ時は**古い側を破棄して新しい側を保全**する（破棄数はライブ計器に計上）。監査の SEC-10（障害起点の保全 = 古い側保全）とは**意図的に逆**——監査は証跡だから発生順が正だが、メールは「今の異常に気づくきっかけ」であり、SMTP 長時間停止からの復旧時に届くべきは最新の状態である（ペルソナレビュー 鈴木の指摘）
- **SMTP タイムアウト**: 接続 10 秒・送信 30 秒（仮値）。**再試行は 1 通あたり 1 回、間隔を空けて行う（仮値 5 分後）**——即時再試行は落ちているサーバ・greylisting（初回配送の意図的一時拒否）に対して無力である。それでも失敗した通知は破棄する（at-most-once。正本はイベントログと監査記録）。**再試行待ちの通知はキュー枠を占有したまま後続の送信を塞がない**（待ち時刻付きで保持し、待ちのない通知を先に送る——head-of-line blocking を作らない）
- **複数宛先の一部 RCPT 拒否は「メッセージとしては送信成功・拒否宛先を警告ログ」とし、再送しない**（成功済み宛先への二重送信を作らない。実装での固定とテストは委任 7）
- 送信失敗はイベントログの警告（新 1000 番台 ID・抑制窓付き）+ ライブ計器で観測可能にする。**メール送信の失敗はメールで通知しない**——ループは決定 6 の allowlist にメール失敗 ID を含めないことで定義レベルで排除する
- **チャネル健全性の常設表示**: 認証済み管理面のダッシュボードに、最終送信成功時刻・直近の失敗理由・キュー深度・累積破棄数・**流量上限による抑制の発生（累積回数を含む——閾値を設定キー化しない判断の再評価材料になるため。再評価トリガ参照）・設定不正による機能無効化の状態（決定 2）**の常設カードを置く（#286/#318 の再起動待ちキー常設表示と同じパターン）。無効化警告の一次到達先がイベントログであることと「イベントログを見ない現場」の前提の循環は、このカードが閉じる。「イベントログを見ない現場」という本 ADR 自身の前提の下で、チャネルの静かな死（パスワード失効・リレー廃止等）に日常動線で気づける経路を確保する——at-most-once + 破棄の正当化はこの常設表示があって成立する（ペルソナレビュー 佐藤・鈴木・クリスの指摘）
- `Enabled=false` への即時反映時、キュー内の未送信通知は破棄する（送り切りを待たない——無効化の意図を優先）

### 6. トリガ集合と流量制御

- **対象はイベント ID の明示リスト（allowlist）で定義する**。初版の対象は実装済みの能動通知事象: スプール系（1001〜1007）・自己検証失敗（1009・1010）・周期評価失敗（1008）・証明書期限接近/使用不能（1014・1015・1017・1018）・認証防御の昇格（1019）・**受信リスナの縮小継続（1022）**を基本とし、確定は実装 PR で行う。**メール送信失敗の新 ID はリストに含めない**（ループの構造的排除）。リストは security.md §4.3 の ID 表に「メール通知対象」マーカーとして記録し、**allowlist と実際の配線の一致を検証するテストを実装 PR に含める**（将来 ID を追加する PR が「メールに乗るか」を明示的に判断することを表の更新義務で強制する——既定は対象外）
- **同一トリガのメール再送間隔はイベントログの抑制窓より長くとる（仮値 60 分）**。イベントログに 1 件ならメールも最大 1 通、の上限関係は維持したまま、持続条件（スプール上限接近等）が一晩で数十通を積む事態を防ぐ
- **メール層の同一性判定は EventId 粒度とする**（イベントログ側の抑制キー——送信元 IP 別・ボリューム別のトリガキー文字列——より意図的に粗い。ILoggerProvider に渡る情報で判定できる粒度であり、発火点のコードに触らないという決定 7 の採用理由を守る）。帰結として、**同一 EventId 内の別インスタンス（1019 の別攻撃者・1006 の別ボリューム等）は再送間隔内は 1 通に畳まれる**——上限関係（メール ≤ イベントログ）は保たれ、詳細の全量はイベントログが持つため、「気づくきっかけ」の役割には足りると整理して受容する（インスタンス粒度が必要になった場合の手段——構造化ログ state でのトリガキー搬送——は再評価トリガへ）
- **全体流量上限（仮値 10 通/時）**を置く。超過時の選好: ①エラーレベルの事象（1008・1009 等）は上限の対象外とする ②警告レベルでは**未送信トリガ種の初報を、送信済みトリガ種の再送より優先する**——同一障害の重複通知が枠を食い尽くして希少で重要な初報（認証防御の昇格等）を抑制する優先度逆転を防ぐ（固定 2 段。設定・編集 UI は作らない）。抑制の発生はイベントログへ 1 回だけ警告し、常設カード（決定 5）にも表示する
- 上記の仮値 6 点（キュー上限 64・再試行間隔 5 分・再送間隔 60 分・流量上限 10 通/時・SMTP 接続タイムアウト 10 秒・送信タイムアウト 30 秒）は configuration.md §9 の確定待ち一覧（CF-x）へ登録し、実測で確定する（設定キーとしては公開しない）

### 7. 実装の位置 — ILogger レール上の第 2 のシンク

能動通知の発火点は 1 箇所ではない: `ActiveNotificationMonitor`（Host。周期系）・`PersistenceWriter`（**Yagura.Ingestion**。即時系 1005）・起動時経路（Program.cs。1001 等）・`AdminAuthFailureDefense`（1019）。受信機構は 2 案を比較した:

- **(a) 発火点直呼びのフック**: 散在する全発火点に二重書き（ILogger + フック呼び出し）を足すことになり、フック契約を `Yagura.Ingestion` から参照できる層（`Yagura.Abstractions`）へ公開する必要が生じる。配線漏れが構造的に起こり得る（選択肢 1-(b) の「食い違わない」が実装規律頼みになる）。**却下**
- **(b) カスタム `ILoggerProvider` を第 2 のシンクとして Host の合成ルートで登録する**: 挿入点は 1 つ・発火点のコードは一切触らない。EventLog プロバイダと同じ `ILogger` レールに乗るため、**Ingestion 発の即時通知・起動時イベント（DI ロガー経由で Build 後に発火するもの）も自動的に捕捉できる**。対象の選別は決定 6 の EventId allowlist で行う。**採用**

- プロバイダの契約: **有界キューへの投入のみを行い、ブロックせず、例外を呼び出し元へ漏らさない**（ロギング呼び出しが受信ホットパス上にあるため）。送信の実体（キュー消化・SMTP 接続）は Host の背景コンポーネント（`Yagura.Host.Observability.ActiveNotification` 配下）が担う
- **利用者の `Logging:*` フィルタ設定から独立させる**: 本プロバイダはプロバイダ固有のコード側フィルタで全カテゴリ・全レベルを受け取り、選別は決定 6 の allowlist のみで行う。イベントログのノイズ調整のつもりの `Logging:LogLevel` 変更が**黙ってメールを止める**経路を作らない（決定 2 の「設定したのに送られていない状態を警告で必ず見せる」との一貫。独立性の検証は委任 6 のテストに含める）
- **起動時経路の allowlist ID（1001・1022 等）がプロバイダ登録後に発火することを実装 PR で確認する**。登録前に発火し捕捉できない ID が見つかった場合は、発火の遅延か allowlist からの除外（+ 帰結への記録）を個別に決める（委任 6）
- 将来のチャネル追加（Slack / webhook 等）は、同じレールに別のシンク（`ILoggerProvider` またはキューの別コンシューマ）を足す形で実現できる——`Yagura.Abstractions` への公開契約の新設は初版では不要（要望が出た時点で再評価）
- UI（設定画面・テスト送信・常設カード）は既存の管理画面パターン（`Yagura.Abstractions` 契約 + Host 実体 + `Yagura.Web` 画面）に従う

### 8. テスト送信

- 管理画面に**テスト送信**を置く。**画面に入力中の（未保存の）値で送信できる**——保存前の検証こそがテスト送信の価値であり、誤設定を保存しないと試せない作りにしない。結果（成功 / 失敗と理由）を画面に即時表示する。テスト送信は**キュー・流量上限を経由しない直送**とする（テストが実通知の枠を消費しない）。「直送」はキュー非経由の意であり、**UI 操作としては非同期実行 + 同時実行 1 件 + キャンセル可能**とする（タイムアウト上限——接続 10 秒 + 送信 30 秒——の間、管理画面の操作を塞がない。受け入れ基準「UI 応答が影響を受けない」との整合）
- **パスワード欄が空欄のままのテスト送信は、保存済みの `Smtp:Password`（DPAPI 復号値）を使用する**（パスワードは UI に再表示しないため、空欄 = 「変更しない」の意味に固定する）。この形態は「画面上の任意ホストへ保存済み資格情報で AUTH を試行できる」操作になるため、**監査レコードに「保存済み資格情報を使用」の別を含める**（値は含めない）
- **テスト送信は監査対象とする**（2000 番台の新 ID。接続先・宛先・成否・操作者を記録し、資格情報値は記録しない）。「状態を変えない操作」だが、①資格情報の使用 + 外向き実トラフィックを伴う点で本番昇格の接続検証（2002）と同型であり、②未保存値で任意ホスト・ポートへ接続を試せる操作は内部ネットワークの到達性探査に転用し得るため、「誰がいつどこへ試したか」を証跡に残す（ペルソナレビュー 田中の指摘。監査対象外とした当初案を撤回）

### 9. 反映方式

- `Notification:Email:*` は**即時反映**（CF-4 の `ImmediateConfigurationApplier` に登録。送信クライアントは接続を保持しないため参照交換で足りる）。`Password` の DPAPI 復号の再読み込み経路への配線・configuration.md §8 表への全キー記録を実装 PR に含める

## 受け入れ基準

- 設定画面から保存 → テスト送信成功 → 実際の能動通知（例: スプール上限接近を人工的に発生）がメールで届くエンドツーエンドが成立する
- SMTP サーバ停止・応答遅延・TCP ブラックホールのいずれでも、受信スループット・UI 応答・監視ループの周期評価が影響を受けない
- `required` で STARTTLS 不成立の場合に平文送信が発生しない（パケットレベルで確認）
- allowlist と配線の一致検証テスト・流量上限とエラー優先の単体テスト・`Password` 復号失敗時の fail-notify テストが存在する

## 帰結

- 良くなること: 監視基盤を持たない現場でも、夜間のスプール成長・証明書期限・認証攻撃の予兆に**人が届く経路**で気づける。能動通知の存在意義（§4.6）が対象現場で初めて実効化する
- **受理時の波及**: architecture.md §4.6 に「第 2 の書き出し先（メール。opt-in）」を追記し、「イベントログは既存監視のレールに乗る」という前提と「そのレールを持たない現場がある」という本 ADR の前提の関係（前者はレールを持つ現場の整理・後者への補完が本機能）を §4.6 側でも整合させる（実装 PR で実施——委任 5）
- 悪くなること・受け入れるリスク:
  - 外部依存（MailKit + 推移的依存）が増える（供給網統治は既存の枠組みに乗せる）
  - `Security=auto` の既定は STARTTLS ストリップ（中間者による平文降格）に対して無防備である。信頼ネットワーク前提（ADR-0004）の範囲内と整理し、敵対的経路を挟む構成では `required` を使うことを文書で案内する。資格情報を設定した構成の降格リスクは決定 3 の能動警告で補う
  - パスワードを設定ファイルに置く形式（DPAPI 暗号化）はサーバ束縛であり、構成配布運用と両立しない——既知の限界（configuration.md §2 CF-D3）をメールにも延長する
  - メールは at-most-once（有界再試行 + 破棄）。特に**起動時の一度しか発火しない事象（1001 縮退起動等）は、復電直後などメールリレー側が未収束のタイミングでは再試行 1 回の後に恒久喪失し得る**。「メールが来ない = 正常」とは限らない——正本はイベントログであり、チャネル健全性は常設カード（決定 5）で確認する
  - **通知メールの配送経路・宛先メールボックスは信頼ネットワーク前提の外にある**。宛先がクラウド（M365 等）の構成では、イベントログ同等の運用・セキュリティ情報（攻撃元 IP・狙われたアカウント名・内部ホスト名等）が管理セグメント外へ恒常的に出ていく。機微性の高い環境では宛先を管理内メールシステムに限定することを利用者向けドキュメントで案内する（申し送り D-3）
  - **Exchange Online（M365）の SMTP AUTH（Basic）は廃止方向**であり、社内リレーを持たず M365 直送しか経路がない現場では初版が使えない可能性がある。再評価トリガ（OAuth）参照

## 先送りにする場合の再評価トリガ

- **メール以外のチャネル（Slack / Teams / webhook）**: 独立した利用者からの要望 3 件以上で個別 ADR として再評価（決定 7 の ILogger レール上のシンク追加として実現できる。`Yagura.Abstractions` への公開契約の要否もその時点で判断）
- **トリガ選別・宛先振り分け**: 同じく要望 3 件以上で再評価。その際も「ログ内容に基づくルール」は NG 線（ADR-0001）のまま
- **implicit TLS（SMTPS 465）**: 対応が必要な利用者が現れた時点で `Security` の第 4 値として追加（構造変更を伴わない）
- **OAuth 2.0（Microsoft 365）**: 委任 8 のライブ検証で Exchange Online の SMTP AUTH（Basic）が廃止済みと判明した場合は**初版スコープ入りを再判断**する（「要望 3 件」を待たない——監視基盤を持たない現場ほど M365 直送しか経路がない可能性が高く、主対象層が初版で置き去りになるため）。提供中であれば、廃止アナウンスの具体化または要望の反復で再評価
  - **委任 8 の実施結果（2026-07-19。Issue #350）**: SMTP AUTH（Basic）は**提供中**のため初版スコープ外を維持する。ただし **2026 年 12 月末に既存テナントで既定オフ**（管理者が再有効化可能）、**恒久削除日は未発表**（発表自体が 2027 年後半予定）。廃止スケジュールは**過去 2 回延期されている**ため、**日付を設計・コード・文書の前提にハードコードしない**（撤回済みの旧情報「2026 年 3 月開始 / 4 月末 100% 拒否」が流通している点に注意）。あわせて判明した重要事項:
    - **すでに効いている制約**の方が実務上は重い——2020 年 1 月以降作成のテナントは既定オフ・Security Defaults 有効時は無効・条件付きアクセスが上位で遮断。**本 ADR の主対象層ほど該当しやすい**（新しいテナント + Security Defaults 既定オン）
    - **技術的障壁は消えた**: `SMTP.SendAsApp` により対話ユーザー不要の client credentials フローが正式サポートされている。障壁は**管理者側のセットアップ負荷**へ移った（Exchange 側の `New-ServicePrincipal` / `Add-MailboxPermission` は **PowerShell 専用で Web UI から代行できず** ADR-0001 の導入容易性と衝突する。`FullAccess` は当該メールボックスの全メール読み取り権限であり送信専用の通知には過剰）
    - **後継経路は OAuth だけではない**: Microsoft 自身が外部宛の推奨として案内する **Azure Communication Services Email は、Entra のクライアントシークレットをそのまま SMTP パスワードとして渡す**仕様であり、**本 ADR の汎用 SMTP 実装がそのまま対応になる**（追加コード・追加依存・追加設定キーのいずれも不要）。したがって「Basic 認証の廃止に備えていない」状態にはならない
    - 本件は **M365 固有**であり業界全体の傾向ではない（オンプレ Exchange Server SE・ISP・SendGrid/SES・Gmail のアプリパスワードはいずれも SMTP AUTH + STARTTLS が現役）
- **流量制御の閾値（決定 6 の仮値 6 点）の設定キー化**: 初版は**設定キーとして公開しない**（決定 6）。理由は数値の自信ではなく、**設定項目を増やすこと自体が「Windows 管理者が既に持つスキルだけで導入できる」（ADR-0001）・「ゼロ設定ファーストラン」（CLAUDE.md 方針 2）を損なう**ためである——閾値 6 つを並べた設定画面は「調整すべき項目がある画面」に見え、導入の敷居を上げる。加えて `0` や過大値を受理すると、本 ADR が最も警戒する「設定したのにメールが来ない / 深夜に大量送信される」事故の経路を利用者自身が作れてしまう。
  - **再評価の条件**: **流量上限による抑制が実際に発生したという報告**（常設カード——決定 5——に抑制の発生が表示されるため、利用者は抑制されたことを認識できる）。「抑制されたが調整できない」という声が出た時点で、**全体流量上限のみ**を手編集キーとして公開する案から再評価する（下限・上限を製品側で固めて `0` を受理しない形。UI へは出さない）。**将来的に対応できる余地を残すこと自体は意図であり、初版で閉じているのは公開範囲であって設計ではない**
  - 実測による確定は configuration.md §9（CF-x）で行う。本トリガは「確定した値が現場に合わなかった場合」の受け皿であり、確定作業とは別のもの
