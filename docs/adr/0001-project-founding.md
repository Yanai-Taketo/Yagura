# ADR-0001: プロジェクト創設 — 目的・スコープ・開発の原則

- 状態: proposed
- 日付: 2026-07-03
- 決定者: YANAI Taketo
- 関連: [docs/README.md](../README.md)（ドキュメント体系）、[docs/development/persona-review.md](../development/persona-review.md)（設計レビュー制度）

## 文脈と課題

Windows 環境での syslog 集約には空白地帯がある。商用製品はライセンス費用が導入の壁になり、Linux ベースの OSS（Graylog 等）は Linux サーバの構築・運用スキルという別の壁を持ち込む。**Windows 管理者が既に持っているスキル（SQL Server・Active Directory・Windows サービスの運用）だけで導入・運用できる OSS の syslog サーバ**は、調査した範囲で見当たらない。

Yagura はこの空白を埋めるプロジェクトである。本 ADR は、プロジェクト創設にあたり目的・スコープ・運営の原則を固定する。

## 検討した選択肢

### スコープの切り方

- **(a) syslog 集約に特化する（採用）**: 受信・保存・閲覧・通知という syslog サーバの本分に集中する
- (b) 汎用ログ基盤・SIEM を志向する: 分析エンジンや相関検知まで踏み込む。中小 Windows 環境には過剰で、商用 SIEM と正面競合し、開発リソースとも釣り合わない。却下

### ライセンス

- **Apache License 2.0（採用）**: 特許条項が明示されており企業の法務審査を通しやすい。商用環境への組み込みも許容し、導入障壁を作らない
- GPL 系: コピーレフトが企業導入審査の障壁になりやすい。却下
- MIT: 簡潔だが特許リスクが未手当て。却下

## 決定

1. **目的**: 「Windows に syslog ツールを簡単に構築できる仕組みを作る」を最上位の目的とする
2. **ターゲットユーザー**: 中小企業の Windows 管理者（ひとり情シスを含む。当面の主対象は日本語圏）。既存スキル（SQL Server・AD・Windows サービス運用）の活用を設計の主軸とする
3. **導入体験の原則**: インストールから最初のログ閲覧まで、追加のサーバ製品の導入や設定ファイルの手編集なしで到達できることを、設計の受け入れ条件とする。**SQL Server もここでの「追加のサーバ製品」に含まれる** — 最初のログ閲覧までは同梱の組み込みデータベースで完結させ、SQL Server 等の外部 DB は本番運用への昇格時に利用者が選択する（導入方式の詳細はアーキテクチャ ADR で定める）
4. **品質の原則**: 「ログを失わないこと。失った場合に必ず観測できること」を初版からの品質原則とする。取りこぼしは発生箇所を区別して計測可能にし、性能の実測（ベンチマーク）を初期バージョンの開発範囲に含める
5. **セキュリティ姿勢**: 信頼ネットワーク前提とし、TLS・認証等の強化は利用者が任意に有効化する方式（opt-in）で提供する。この前提が成立する条件と限界は、設計フェーズのセキュリティモデル ADR で定義する（起案予定は [docs/adr/README.md](README.md) で管理し、起案時に本 ADR と相互リンクする）
6. **ライセンス**: Apache License 2.0
7. **言語ポリシー**: ドキュメントは日本語を優先する（主ターゲットが日本語圏の Windows 管理者のため）。コード・コミットメッセージは英語とする。国際的な利用者向けの英語 README は将来整備する
8. **バージョン戦略**: Semantic Versioning 2.0。v0.1 から開始し、v1.0 公開を目指す。v1.0 の公開基準（機能凍結・実運用実績・文書整備・後方互換性の凍結・コード署名の 5 観点）は別 ADR で定義する（起案予定一覧に登録済み）
9. **開発体制**: すべての変更は PR 経由とする。設計文書はペルソナレビュー制度によるレビューとオーナー（YANAI Taketo）の承認を経る。ドキュメントは「常に現在形の全体設計書」と「ADR（決定の履歴）」の 2 層構造で維持する

## スコープ外（明示 NG）

以下は商用 SIEM・SaaS 監視基盤の領域として明示的にスコープ外とする。機能提案のたびに都度判断するのではなく、ADR として固定することで守備範囲の膨張を防ぐ。

| 項目 | 理由 |
|---|---|
| ログ加工パイプライン（GROK・処理ルールエンジン） | 加工は送信側ツールの領分。Yagura は受けて保存する側に徹する |
| クラウドへのアーカイブ階層（S3 / Azure Blob） | 中小企業のログ量では効果が薄く、DB 一段保管 + 保持期間管理で足りる |
| ログからのメトリクス派生 | メトリクス監視は Zabbix / PRTG 等の領分 |
| 監視プラットフォーム化（マルチクラウド転送等） | syslog 集約特化を守る |
| UEBA・機械学習分析 | 大規模基盤前提であり、中小オンプレミスに不適合 |
| Microsoft 365 Defender / XDR 連携 | 高額ライセンス前提の環境向けであり、対象ユーザー像と矛盾 |
| SOAR（ワークフローエンジン） | 別種の製品領域 |
| 大規模分析基盤（専用クエリ言語・サマリインデックス等） | 数 TB/日規模の分析前提であり、対象環境には過剰 |
| RELP 受信・Kafka 連携 | データセンター向けであり、Windows 中小 LAN が対象の本製品では扱わない |
| カーネルチューニングの設定画面露出 | 設定画面の複雑化はターゲットユーザーのスキル前提を逸脱する |
| Windows イベントログ転送エージェントの自作 | 既存の実績あるツール（NXLog 等）の設定例を案内する方が利用者の利益になる |

## 帰結

- 良くなること: 立ち位置が明確になり、機能提案の採否判断が速くなる。ライセンスと意思決定証跡が最初から整い、企業の導入審査・外部コントリビュータの参加に耐える体裁を持つ
- 悪くなること: 特化ゆえに断る要望が発生する。日本語優先は国際的なコントリビュータへの障壁になる（英語 README で将来緩和する）
- 受け入れるリスク: 「簡単に構築できる」という体験目標は、実装の簡単さと相反することがある（ゼロ設定で動く仕組みは内部の作り込みを要する）。体験を優先する

## 先送りにする場合の再評価トリガ

スコープ外の各項目は、**利用者からの具体的な要望が反復して挙がった場合**（目安: 独立した利用者から 3 件以上の Issue）に、個別 ADR で再評価してよい。その場合も「syslog 集約特化」の核と矛盾しない形に限る。
