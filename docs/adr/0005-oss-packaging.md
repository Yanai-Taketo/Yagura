# ADR-0005: OSS 体裁 — README・貢献導線・脆弱性報告窓口

- 状態: accepted
- 日付: 2026-07-04
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md) 決定 6・7（ライセンス・言語ポリシー）、[ADR-0004](0004-security-model.md) 委任事項（脆弱性報告窓口）

## 文脈と課題

リポジトリは公開されているが、外部の訪問者が最初に見る 3 点——「これは何か（README）」「どう関われるか（CONTRIBUTING）」「脆弱性をどこに報告するか（SECURITY）」——が存在しない。ライセンスがあっても入口の文書がなければ、fork・貢献・責任ある脆弱性報告のいずれも始まらない。コードの実装より先にこれらを整備する（レビュー指摘: LICENSE がない期間・入口がない期間は外部者の離脱点になる）。

本 ADR は入口文書の方針を定め、初版の実体（README.md・CONTRIBUTING.md・SECURITY.md）を同じ PR で提供する。

## 決定

### 1. ルート README（日本語 + 英語の導入段落）

- ルート README.md は**日本語を正文**とし、**冒頭に英語 1 段落の紹介**を置く（国際的な訪問者がプロジェクトの正体とステータスを判断できる最小限の導線。ADR-0001 決定 7 の「英語 README は将来整備」への中間解）
- 内容は**現状を正直に**書く: 開発初期でありリリースは未提供であること、目指すもの（目的・導入体験の原則・品質の原則）、設計文書（ADR）への案内、ライセンス
- リリース未提供の間、**機能一覧は「予定」として書かず、確定した設計原則のみを書く**（作っていないものを並べない。リリース後に実装済み機能へ置き換える）

### 2. CONTRIBUTING（貢献導線）

- 貢献の流れ（Issue → PR、main 直接 push 禁止、squash merge）、開発規約（[conventions.md](../development/conventions.md)）への参照、言語ポリシー（Issue・PR は日本語・英語どちらでも可 / コード・コミットは英語 / ドキュメントは日本語優先）を明記する
- **設計文書へ提案する外部コントリビュータにペルソナレビューの実行義務はない**（オーナー側の責務。persona-review.md の運用ノートの再掲）
- AI エージェントとの協働開発であることを明示する（コミット・PR の Co-Authored-By 表記の意味を含む。透明性のため）

### 3. SECURITY（脆弱性報告窓口）

- **報告経路は GitHub の非公開脆弱性報告（Private Vulnerability Reporting）を第一**とする（公開 Issue に書かないことの明示。メールアドレスの公開による窓口私書箱化・スパムを避ける）。**代替経路は意図的に置かない**（報告には GitHub アカウントが必要である旨を SECURITY.md に明記する）
- **受け入れ条件**: この機能はリポジトリ設定での有効化が必要であり、**有効化と報告導線の表示を確認してから本 PR を merge する**（2026-07-04 に `gh api repos/{owner}/{repo}/private-vulnerability-reporting` で `enabled: true` を確認済み）
- **応答目標を明記する**: 受領確認まで 7 日以内、初期評価まで 14 日以内。個人運用の OSS として誠実な範囲の約束に留め、修正期限は約束しない（重大度により最優先で対応する旨のみ記す）
- **開示ポリシーを明記する**: 修正提供までの公開自粛のお願い（協調的開示）、**報告から 90 日応答がない場合は報告者自身の判断による公開を尊重する**（応答が止まった場合の報告者側の出口）、修正時の報告者クレジット（希望により匿名可）
- 対象範囲: リリース版（提供開始後）と main ブランチ。リリース提供前の現段階では「main のみ」であることを明記する

### 4. 文書の保守

- README・CONTRIBUTING・SECURITY は**全体設計書と同じ「常に現在形」の層**として扱い、実態と乖離させない（機能・状態の記述が変わる PR は README の該当箇所も同じ PR で更新する。docs/README.md の 2 層構造の原則に従う）。**この更新義務は conventions.md の PR 要件にも反映する**（規約側に書かなければレビュー時に参照されず形骸化するため）
- NOTICE ファイル（同梱する第三者ソフトウェアの帰属表示）は、最初の外部依存を同梱するリリース準備時に整備する（現時点では対象がない）

## 検討した選択肢

### README の言語構成

- **(a) 日本語正文 + 英語導入段落（採用）**: 主対象（日本語圏の Windows 管理者）への最適化と、国際的な訪問者への最低限の説明責任を両立する
- (b) 日本語のみ: GitHub の公開リポジトリとして、英語話者が正体を判断する手がかりがゼロになる。却下
- (c) 英語正文 + 日本語版併設: 主対象が日本語圏である以上、正文は日本語であるべき（ADR-0001 決定 7）。二重管理のコストも大きい。却下

### 脆弱性報告窓口

- **(a) GitHub 非公開脆弱性報告（採用）**: リポジトリに統合され、報告者・受領者双方の身元と経路が明確。追加のインフラ不要
- (b) 専用メールアドレス: 公開アドレスはスパムの的になり、個人運用では管理が続かない。却下
- (c) 公開 Issue で受け付け: 修正前の脆弱性が公開される。却下

## 帰結

- 良くなること: 外部の訪問者・貢献者・報告者それぞれに入口ができる。「正直な README」により、開発初期であることを含めて期待値が正しく設定される
- 悪くなること: README が「常に現在形」の層に加わるため、状態が変わる PR の更新義務の対象が 1 つ増える
- 受け入れるリスク: 応答目標（7 日 / 14 日）は個人運用の約束であり、達成できない期間が生じ得る。その場合は SECURITY.md 側の目標値を amendment で実態に合わせる（黙って破らない）

## 先送りにする場合の再評価トリガ

- 英語の完全版 README（README.en.md）: 英語話者からの Issue・質問が反復した場合（目安: 独立した利用者から 3 件以上）に整備する（ADR-0003 の英語 UI トリガと同じ基準）
- NOTICE: 最初の外部依存同梱時（上記決定 4 のとおり）
