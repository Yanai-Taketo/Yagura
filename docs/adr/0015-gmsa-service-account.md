# ADR-0015: AD 環境向け gMSA によるサービス実行（opt-in）

- 状態: accepted（2026-07-18 マージ）
- 日付: 2026-07-16
- 決定者: YANAI Taketo
- 関連: [ADR-0004](0004-security-model.md) 決定 4（仮想サービスアカウント既定・gMSA は AD 認証 opt-in とあわせて対応）/ [ADR-0006](0006-v1-release-criteria.md) 基準 1（v1.0 ブロッカー）/ [ADR-0010](0010-admin-ui-authentication.md) 委任事項（SPN と gMSA の接続）/ [security.md](../design/security.md) §4・§5 / Issue #263・#228

## 文脈と課題

Yagura のサービス実行アカウントは仮想サービスアカウント `NT SERVICE\Yagura`（ADR-0004 決定 4。LocalSystem では動かさない）である。これはスタンドアロン機・ワークグループ機を含むあらゆる Windows で追加準備なしに成立し、ゼロ設定ファーストランを支えている。

一方、AD 環境では仮想サービスアカウントに固有の限界がある:

1. **ネットワーク上の識別がマシンアカウント（`DOMAIN\HOST$`）に畳まれる**: リモート SQL Server へ Windows 統合認証で接続すると、DB 側から見えるのは「そのホスト」であって「Yagura というサービス」ではない。同一ホストの他サービスと識別を共有するため、DB 側の権限を Yagura に限定できない（最小権限の原則——ADR-0004——がネットワーク越しに崩れる）
2. **保存データ・監査の主体をドメインの管理体系（グループ・監査ポリシー）に載せられない**
3. 専用ドメインアカウントを手動で作る運用は、パスワードの手動管理という新しい事故源（無期限パスワード・平文控え）を持ち込む

gMSA（グループ管理サービスアカウント）は、パスワードを AD が自動生成・自動ローテーションするサービス専用のドメインアカウントであり、Windows 管理者の既存スキル（AD 運用）にそのまま乗る。ADR-0004 決定 4 は gMSA 対応を予告済みであり、ADR-0006 基準 1 が v1.0 公開の必須条件として名指ししている。本 ADR はその実現方式を確定する。

**前提として重要な実測（と、その記録の未着地）**: Issue #228 の AD lab 実機検証（2026-07-11）で、仮想サービスアカウント構成のまま**正規の SPNEGO-Kerberos・Kerberos AP-REQ が誤遮断されず疎通する**ことを確認済み（security.md §2.4 に記録あり——Kerberos の発券が成立していた、つまりマシンアカウント SPN 経由の Kerberos が動作していたことを示す）。したがって gMSA は「Kerberos を成立させるための必須手段」ではなく、上記 1〜3 を解く **AD 環境の運用正道としての opt-in** である。ただし security.md §2 の「SPN の成立可否は Phase 3 まで未検証」・SEC-D7 ①の文言は #228 の結果を反映しないまま残っており（レビュー指摘——クリス懸念 3）、**この着地（security.md §2/SEC-D7 の更新 + ADR-0010 Phase 3・委任事項側への決着注記）を本 ADR の実装フェーズの必須作業に含める**（決定 8）。ADR-0010 委任事項の「仮想サービスアカウントへの SPN 登録が成立しない場合、gMSA との接続で再評価する」は、成立が確認されたため gMSA は SPN 起因では要求されない、という形で決着する。

## 検討した選択肢

### そもそも v1.0 でやるか（レビュー指摘——クリス質問 1——を受けて追記）

- **v1.0 スコープを維持（採用）**: ADR-0006 基準 1 の名指しに加え、2026-07-14 の棚卸し（Issue #277）でオーナーが着手順を承認済み。実利用者の要望は未リリースゆえ 0 件だが、対象利用者（AD 前提の Windows ショップ）にとって「サービスの専用識別 + パスワード自動管理」は導入判断（社内セキュリティレビュー）で問われる基本項目であり、後送りは v1.0 の「AD 環境で恥ずかしくない」線を下げる
- 案: v1.0 から切り出す（ADR-0006 の部分 supersession）: 却下——SPN 動機の消滅（#228）後も文脈 1〜2（DB 最小権限・監査主体）は残り、これは基準 1 が gMSA を含めた当初理由そのもの。基準の変更は本 ADR の管轄外であり、必要なら独立に提案されるべき

### 実行アカウントの指定方法

- **案 A: MSI プロパティ + セットアップ画面の入力欄（採用）**: `YAGURA_SERVICE_ACCOUNT`（既定 `NT SERVICE\Yagura`）をインストーラのプロパティとし、対話インストールではセットアップ画面に gMSA 名（`DOMAIN\name$`）の入力欄を出す。パスワード欄は設けない（gMSA はパスワード入力が不要——空パスワードでのサービス登録が仕様。パスワード欄の不在自体が「手動管理アカウントを想定しない」という設計意図の表明になる）
- 案 B: インストール後の切替スクリプトのみ: 却下——AD 環境の新規導入で最初から gMSA を使いたいだけの利用者に 2 段階の手順を強い、GPO/サイレント配布（`msiexec /qn YAGURA_SERVICE_ACCOUNT=...`）の一発構成もできない
- 案 C: 管理 UI からの切替: 却下——SCM 操作・ACL 付替・再起動を跨ぐ失敗時の中途状態が重く、v1.0 の対象状況（導入時に決める）に対して過剰

### 前提条件（gMSA の作成・取得許可）の扱い

- **案 A: AD 管理者の責務とし、製品はガイドと診断まで（採用）**: gMSA の作成（`New-ADServiceAccount`）・ホストへのパスワード取得許可（`PrincipalsAllowedToRetrieveManagedPassword`）は AD 側の操作であり製品外。Yagura 側はガイド（前提条件・確認コマンド `Test-ADServiceAccount`）と、サービス起動失敗時の切り分け情報を提供する
- 案 B: インストーラが gMSA を自動作成: 却下——AD への書き込みはドメイン管理者権限を要し、インストール実行者の権限前提が跳ね上がる。gMSA の配置（OU・命名・取得許可グループ）は組織のポリシー領域であり、製品が代行すべきでない（ADR-0001 の「Windows 管理者の既存スキルで運用できる」は、AD 管理の代行ではなく AD 管理への接続を意味する）

### 既存インストール（仮想 SA 稼働中）からの切替

- **案 A: 再インストール/アップグレード時の指定のみを製品スコープとする（採用。2026-07-16 オーナー裁定）**: 稼働中の切替（サービス停止 → `sc config` → 保存データ ACL 付替 → 起動）は operations ガイドの手順書として記載し、スクリプト・管理 UI 化はしない
- 案 B: 切替スクリプト同梱: 却下（再評価トリガ付き）——手順自体は単純で、ガイドの手順書で足りる。スクリプト化は検証面（ACL 付替の全域性・失敗時の巻き戻し）のコストが先行する。**再評価トリガ: 利用者から切替自動化の具体的要望が挙がったとき**。なお採用案（インストーラ経由の切替）にも同じ全域性・失敗時着地の問題は存在する（レビュー指摘——クリス懸念 4）ため、これを決定 7 の受け入れ条件（既存データありのアカウント変更 E2E）で検証する——「案 B を却下した検証コストを採用案では払わない」という二重基準にしない

## 決定

1. **gMSA によるサービス実行を opt-in として提供する**。既定は仮想サービスアカウント `NT SERVICE\Yagura` のまま変えない（ゼロ設定ファーストラン・非 AD 環境の体験は不変）
2. **指定はインストーラで行う**: MSI プロパティ `YAGURA_SERVICE_ACCOUNT`（既定 `NT SERVICE\Yagura`）+ セットアップ画面の入力欄（パスワード欄なし）
   - **remember property パターン（Issue #203 と同じ）を適用**し、アップグレード時は前回値を継承する——指定漏れで gMSA 構成が黙って仮想 SA へ戻り、リモート SQL 統合認証だけが静かに壊れる退行（レビュー指摘——鈴木・リサ）を構造的に塞ぐ
   - **検証は fail-closed**: 受理する値は `NT SERVICE\Yagura` または gMSA 形式（`DOMAIN\name$`）のみ。それ以外（`LocalSystem`・`NetworkService`・`LocalService` の well-known、`$` なしの一般アカウント形式等）は**対話・サイレントを問わずインストールを失敗させる**（警告付き続行は設けない——ADR-0004 決定 4 の LocalSystem 禁止、および本プロジェクトの fail-closed 規律〔ADR-0010 決定 1・イベント ID 1011/1012〕との一貫。レビュー指摘——田中・クリス・佐藤）。**形式検証の限界も明示する**: `DOMAIN\name$` 形式ではコンピュータアカウント・sMSA を gMSA と判別できない（判別は AD への照会を要しインストーラの責務を超える）——形式不正の混入はサービス起動失敗（1069 系）として決定 3 の切り分けに合流する
   - **実現方式は条件付き決定**: 第一候補 = `Package.wxs` の `ServiceInstall/@Account` プロパティ間接参照（`[YaguraFwProfile]` と同じ手法）。**gMSA 指定での成立可否は実機依存のため、accepted 後・実装最初期に AD lab でスパイク検証する**（ADR-0010 のライブ検証作法）。不成立の場合はカスタムアクションでの `sc config` 後追いへフォールバックし、その場合の ACL 付与タイミング・失敗時ロールバック要件・検証結果を**本 ADR の改訂履歴（amendment）に記録する**（レビュー指摘——クリス・リサ。「実装時判断」のまま文書とコードを乖離させない）
   - **SeServiceLogonRight（サービスとしてログオン）**: MSI の `ServiceInstall` はこの権利を付与しない前提で扱い、付与の成立主体（SCM の自動付与の有無を含む）を lab スパイクの検証項目とする。付与が必要な場合はガイドの前提条件（ローカルポリシー/GPO 手順）に含める（レビュー指摘——クリス懸念 2）
3. **前提条件は AD 管理者の責務**: gMSA の作成・取得許可・KDS ルートキーを含む AD 側の準備は製品外。利用者ガイドには以下を**必須要件**として含める（レビュー指摘——佐藤・田中）:
   - KDS ルートキー未整備時の初回作成と**既定 10 時間の待ち時間**（単一 DC 環境での即時化 `-EffectiveTime ((Get-Date).AddHours(-10))` を含む）——初回導入者が全員通る道であり例外扱いにしない
   - **取得許可の最小化をセキュリティ要件として明記**: `PrincipalsAllowedToRetrieveManagedPassword` はインストール先ホストのコンピューターアカウント単位（またはそのホストのみを含む専用グループ）に限定する。広域グループ（Domain Computers 等）の指定は、グループ内の全ホストに Yagura の識別の騙り（gMSA パスワード取得 → DB 接続）を許し、決定 5 の最小権限の価値を AD 側の一設定で崩す
   - 起動失敗（エラー 1069 系）の **3 段チェックリスト**: ①ホストで `Test-ADServiceAccount` が `True` か → ②`False` なら取得許可に当該ホストが含まれるか → ③グループ経由で許可した場合はホスト再起動（グループメンバーシップ反映）を行ったか
   - **gMSA 構成ではサービス起動が DC 到達性に依存する**こと（停電復旧等で syslog サーバが DC より先に起動すると 1069 で失敗する）。SCM の失敗時回復（インストーラが既定設定する再起動 5 秒 × 3 回——ADR-0004 実装済み）では DC 復旧まで届かない場合があるため、gMSA 構成向けに回復設定の見直し（再起動間隔・回数）を lab 実測を踏まえて確定し、ガイドに記載する（レビュー指摘——鈴木）
4. **実行アカウントへの権限付与・検証の全域を `YAGURA_SERVICE_ACCOUNT` に追随させる**。追随対象を列挙する（レビュー指摘——田中・リサ・鈴木）:
   - インストーラの ACL 付与（データルート・フォワーダ配置・監査領域——security.md §5）。**実装機構の留意**: 現行の SDDL 静的埋め込み（`PermissionEx`）は SID をビルド時に要するため、ドメイン採番の gMSA SID には適用できない——実装方式（SID 解決カスタムアクション / `icacls` 後追い等)は lab 検証で確定し security.md §5 に記録する
   - **証明書秘密鍵の実行時付与（監査 ID 2009/2010）の付与先**
   - **起動時 ACL 検証（security.md §5）の期待値導出**（付与先を固定名でなく実行アカウントから導出する——gMSA 環境での常時誤警報・ドリフト見逃しの両方を防ぐ）
   - **アカウント切替時、旧アカウントの ACE は除去する**（権限の残骸を残さない）。除去できないケースが lab で判明した場合は、残置の事実と理由を operations 手順書・§5 に明示する
   - 切替後の初回起動で**旧アカウント時代のスプール退避分が drain できる**こと（既存ファイルへの継承反映を含む——読めなければ退避済みログのロスに直結する）
5. **DB 接続**: gMSA 実行時、SQL Server への Windows 統合認証はサービス固有のドメイン識別で成立する
   - database.md §6.1 に「Windows 統合認証の実行主体は実行アカウント（仮想 SA = マシンアカウント `DOMAIN\HOST$` / gMSA = `DOMAIN\name$`）」を明記し、**昇格ウィザードは DB 側で `CREATE LOGIN` すべきアカウント名を画面に表示する**（レビュー指摘——佐藤）
   - **推奨の主語を限定する**: gMSA を推奨するのは「リモート SQL Server + DB 側の最小権限付与を要する AD 環境」。SQLite のまま・SQL Server 同居・マシンアカウント許可で足りる環境では仮想 SA のままでよい——この判断フローをガイドに含める（レビュー指摘——クリス・佐藤。全 AD 環境への無条件推奨にしない）
   - **観測性**: 統合認証の接続失敗時、イベントログ警告に実行主体（アカウント名）・失敗種別を含め、DC 起因か SQL Server 起因かの一次切り分けがサーバ単体でつくようにする（レビュー指摘——鈴木）
6. **切替は再インストール/アップグレード時の指定のみ**（検討した選択肢のとおり。稼働中切替は operations ガイドの手順書——DPAPI 保護値は machine スコープのため同一マシン内の切替で影響しないことを手順書に明記する〔佐藤質問 3〕）。**切替経路でも保存データ・設定は保持される**（変わるのは ACL の付与先——決定 4——のみ）ことを利用者ガイドに明記する
7. **受け入れ条件は AD lab の実機検証**（ADR-0006 基準 1 の要求。yagura.test DC）。E2E シナリオ（レビュー指摘を反映して拡張）:
   - (a) 新規: gMSA 作成 → インストール（プロパティ指定）→ 受信 → 閲覧 → リモート SQL Server への統合認証接続
   - (b) **強制パスワードローテーション（`Reset-ADServiceAccountPassword`）を跨ぐ受信・DB 書き込みの継続**
   - (c) **DC 停止状態でのサービス再起動**（1069 の実挙動・System ログ証跡・SCM 回復設定の実効）
   - (d) **既存データを持つアップグレードでのアカウント変更**（ACL 付替の全域性・旧 ACE 除去・スプール退避分の drain・失敗時の着地状態）
   - (e) **監査証跡の確認**（決定 8 の転記・変化検出、2009/2010 の付与先、起動時 ACL 検証が誤警報しないこと）
   - (f) SeServiceLogonRight の成立主体・SPN 付け替え（リモート管理 HTTPS 利用時は gMSA への HTTP SPN 登録がほぼ確実に必要になる想定で、**手順を operations ガイドとして確定する**——「要否確認」ではなく手順の確定を成果物とする〔クリス質問 3〕）
   - **CI（AD なし）で継続検知する範囲も明示する**（レビュー指摘——リサ）: アカウント形式検証ロジックの単体テスト・MSI テーブル照合（プロパティ既定値・間接参照の展開）・仮想 SA 経路の非退行（installer-e2e）。lab でしか検証できない項目は installer/README の検証マトリクスに記載する
8. **実行アカウントの監査証跡**（レビュー指摘——田中。新設）: `YAGURA_SERVICE_ACCOUNT` の構成値を初回起動時にイベントログへ転記する（security.md §4.1 のインストーラ由来転記レール・2000 番台 additive-only）。さらに**起動時に実効実行アカウントを前回起動時の記録と照合し、変化を監査事象として記録する**——製品外の `sc config` による切替も次回起動で必ず証跡化され、「いつからこの識別で動いているか」に監査記録で答えられる。あわせて #228 の SPN 実測の設計書への着地（文脈の項参照——security.md §2/SEC-D7 の更新・ADR-0010 側への決着注記）を実装フェーズの必須作業とする

## 帰結

- **良くなること**: AD 環境で「サービス固有のドメイン識別・パスワード自動管理・DB 側の最小権限付与」が成立する。GPO/サイレント配布で最初から gMSA 構成にできる。v1.0 基準 1 の gMSA 要件を満たす
- **悪くなること・受け入れるリスク**:
  1. インストーラの分岐（アカウント検証・ACL 付与先の切替・remember property）が増え、lab 検証項目が大きく増える（決定 7 の (a)〜(f)）
  2. **gMSA 構成ではサービス起動が DC 到達性に依存する**（仮想 SA は外部依存ゼロ）。「受信を止めない」原則の適用範囲が SCM の外側（サービス起動前）で狭まることを、ガイドの明示と SCM 回復設定（決定 3）で緩和する——構造的には解消できない、gMSA 採用の本質的対価
  3. gMSA の準備不足（取得許可漏れ・KDS 未整備）によるつまずき点が生まれる——ガイド（決定 3 の必須要件）と診断情報で緩和するが、AD 側の設定ミスを製品が検出することはできない。AD 側の事後変化（取得許可からの除外等）は次回起動の 1069 まで顕在化しない「時限」性を持つ（田中質問 4）——定期確認（`Test-ADServiceAccount`）を operations ガイドの推奨手順に含め、能動通知化は実需を見て再評価する
  4. `ServiceInstall/@Account` の gMSA 成立・ACL 実装方式は実機依存であり、lab スパイク（決定 2）まで確定しない
- **スコープ外の明示**: sMSA（gMSA に置き換えられた旧世代）・一般ドメインアカウントでの実行・複数サービスインスタンスでの gMSA 共有は対象外

## 先送りにする場合の再評価トリガ

- 切替スクリプトの同梱（決定 6 の却下案 B）: 利用者から切替自動化の具体的要望が挙がったとき
- 一般ドメインアカウント対応（スコープ外）: gMSA を建てられない環境（KDS ルートキー未整備等）からの具体的要望が反復（3 件以上）したとき——その場合もパスワード管理の事故源の議論を必須とする
- AD 側事後変化の能動通知化（帰結 3）: 「再起動して初めて 1069 で気づいた」という報告が実利用で挙がったとき

## 改訂履歴

### 1. 決定 2 が要求する実装最初期 lab スパイクの結果（2026-07-18）

**分類**: amendment（決定は不変。条件付き決定の分岐確定と実装要件の追加）
**実施環境**: `WIN-H31KVKTHCKU`（Windows Server 2025 10.0.26100 / yagura.test の DC 兼検証機）/ Yagura 0.4.0 / WixToolset.Sdk 7.0.0
**位置づけ**: 決定 7 の E2E 受け入れ (a)〜(f) は実装完了後の話であり、本記録は決定 2 が要求する**実装最初期のスパイク**のみである。

#### AD 側の前提条件（ガイドの素材。実施手順そのもの）

```powershell
Add-KdsRootKey -EffectiveTime ((Get-Date).AddHours(-10))   # lab 用の即時有効化。本番は 10 時間待つ
New-ADServiceAccount -Name gmsaYagura -DNSHostName gmsaYagura.yagura.test `
  -PrincipalsAllowedToRetrieveManagedPassword "WIN-H31KVKTHCKU$"
Install-ADServiceAccount -Identity gmsaYagura
Test-ADServiceAccount -Identity gmsaYagura      # → True
```

採番された gMSA: `SamAccountName = gmsaYagura$` / `SID = S-1-5-21-1618155702-1212114865-3694278133-1111`（**ドメイン採番**であることに注意——下記 (iii)）。

#### (i) `ServiceInstall/@Account` のプロパティ間接参照による gMSA 指定 → **第一候補が成立。フォールバック不要**

`Package.wxs` を一時パッチし（`<Property Id="SERVICEACCOUNT" Value="NT SERVICE\Yagura" Secure="yes" />` + `Account="[SERVICEACCOUNT]"`）、`msiexec /i ... SERVICEACCOUNT=YAGURA\gmsaYagura$` で検証した。MSI ログに次のとおり記録され、**サービスは gMSA アカウントで登録された**:

```
Executing op: ServiceInstall(Name=Yagura,...,StartName=YAGURA\gmsaYagura$,Password=**********,...)
```

すなわち **`ServiceInstall/@Account` はプロパティ間接参照を受理し、SCM は `DOMAIN\gmsa$` 形式（パスワードなし）を受理する**。カスタムアクションによる `sc config` 後追いへのフォールバックは**不要**である（決定 2 の第一候補で確定）。

#### (ii) `SeServiceLogonRight` の付与主体 → **誰も付与しない。明示的な付与が必須**

権利を付与しない状態では、サービス登録自体は成功するが**起動に失敗**し、インストールが 1603 でロールバックする:

```
製品: Yagura -- エラー 1920。 サービス 'Yagura' (Yagura) を開始できませんでした。
システム サービスを開始する特権を持っていることを確認してください。
```

**MSI の `ServiceInstall` も SCM も gMSA へ本権利を自動付与しない**ことを実測で確定した。`secedit` で gMSA へ `SeServiceLogonRight` を付与すると、同じ MSI でサービスプロセスの起動まで到達する（下記 (iii) の別要因で落ちるまで進む）。

参考として、既定の仮想サービスアカウントで本問題が起きないのは、`SeServiceLogonRight` に `*S-1-5-80-0`（`NT SERVICE\ALL SERVICES`）が既定で含まれ仮想アカウントがこれに包含されるためである（実測: 既定値は `*S-1-5-80-0,*S-1-5-83-0,*S-1-5-90-0`）。gMSA はドメインアカウントでありこの包含に与らない。

**実装要件**: gMSA 構成では `SeServiceLogonRight` の付与をガイドの前提条件（ローカルポリシー/GPO 手順）に含めるか、インストーラのカスタムアクションで付与する。付与しない限り**サービスは一度も起動できない**ため、これは任意の推奨ではなく必須手順である。

> **訂正注記（2026-07-24。SEC-14 受け入れ検証の過程で判明）**: 上記 (ii) の「未付与だと 1920 で必ず失敗＝明示付与が必須」は、その後の AD lab 実機検証（**Windows Server 2025 / Windows 10 `10.0.26100`**、PR #419 の MSI）で**再現しなかった**。gold-standard 実測——`LsaEnumerateAccountRights` で gMSA の権利がゼロ、プロセストークンは真正の `DOMAIN\gmsaYagura$`、`SeServiceLogonRight` を**明示付与→明示除去しても**再起動でログオンが成立（7038/1069 なし）——により、**当該ビルドの SCM は gMSA の service logon に本権利を強制しない**ことが確定した（SCM の暗黙付与ではない＝除去後も LSA は権利ゼロ）。したがって本スパイクの 1920 の真因は logon right ではなく別要因（最有力候補: gMSA パスワード取得失敗——ホスト未認可・KDS 未伝播——を secedit 付与操作と時間的に取り違えた confound）である疑いが濃厚だが、**スパイク環境を再現できないため断定はしない**。ガイド §2.4 は本注記を反映し「必須」→「推奨（強制は環境依存）」へ更新済み。SEC-14 (f) の 1920 ロールバック観測は成立不能として再スコープした（証跡: [PR #419 コメント](https://github.com/Yanai-Taketo/Yagura/pull/419)。lab 実機検証 2026-07-24）。

#### (iii) ACL 付与方式 → **SDDL 静的埋め込みは適用不可を実証。後追い付与が必須**

決定 2 の「現行の SDDL 静的埋め込み（`PermissionEx`）はドメイン採番 SID の gMSA に適用できない」を実機で確証した。gMSA でサービスを起動すると、プロセスは起動するがアプリが**データルートを開けずクラッシュ**する:

```
Microsoft.Data.Sqlite.SqliteException (0x80004005): SQLite Error 14: 'unable to open database file'.
```

このときのデータルート ACL（`icacls` 実出力）——**ビルド時に埋め込まれた仮想アカウントの SID がそのまま残り、名前解決もされない**:

```
C:\ProgramData\Yagura NT AUTHORITY\SYSTEM:(OI)(CI)(F)
                      BUILTIN\Administrators:(OI)(CI)(F)
                      S-1-5-80-2906685102-3215373564-3806320331-1391897514-1145735297:(OI)(CI)(M)
```

gMSA（`S-1-5-21-...-1111`）はこの ACE のいずれにも該当しないため、データルートへ一切アクセスできない。**したがって SID 解決カスタムアクション / `icacls` 後追いのいずれかが必須**である。当たりとしては、(i)(ii) の結果から**インストール時に実行アカウントが確定している**（`SERVICEACCOUNT` プロパティとして既知）ため、そのアカウント名を `icacls` 相当で後追い付与する方式が最も単純に成立する見込みである（SID の事前解決を要しない）。**対象は security.md §5 のデータルートに加え、フォワーダ配置フォルダ・監査記録領域の全 `PermissionEx` 適用先**である。

#### 決定への影響

決定 2 の条件付き決定は**第一候補（プロパティ間接参照）で確定**し、フォールバック（`sc config` 後追い）は発動しない。一方、決定 2 が「lab 検証で確定する」としていた 2 点に実装要件が確定した——**`SeServiceLogonRight` の明示付与**と **ACL の後追い付与**であり、いずれも「あると良い」ではなく**これ無しでは gMSA 構成が全く機能しない**必須要件である。実装 PR はこの 2 点を含めること。security.md §5 への ACE 記録は実装方式の確定後に行う。
