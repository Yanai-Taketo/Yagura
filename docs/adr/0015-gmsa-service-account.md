# ADR-0015: AD 環境向け gMSA によるサービス実行（opt-in）

- 状態: proposed
- 日付: 2026-07-16
- 決定者: YANAI Taketo
- 関連: [ADR-0004](0004-security-model.md) 決定 4（仮想サービスアカウント既定・gMSA は AD 認証 opt-in とあわせて対応）/ [ADR-0006](0006-v1-release-criteria.md) 基準 1（v1.0 ブロッカー）/ [ADR-0010](0010-admin-ui-authentication.md) 委任事項（SPN と gMSA の接続）/ [security.md](../design/security.md) §5 / Issue #263・#228

## 文脈と課題

Yagura のサービス実行アカウントは仮想サービスアカウント `NT SERVICE\Yagura`（ADR-0004 決定 4。LocalSystem では動かさない）である。これはスタンドアロン機・ワークグループ機を含むあらゆる Windows で追加準備なしに成立し、ゼロ設定ファーストランを支えている。

一方、AD 環境では仮想サービスアカウントに固有の限界がある:

1. **ネットワーク上の識別がマシンアカウント（`DOMAIN\HOST$`）に畳まれる**: リモート SQL Server へ Windows 統合認証で接続すると、DB 側から見えるのは「そのホスト」であって「Yagura というサービス」ではない。同一ホストの他サービスと識別を共有するため、DB 側の権限を Yagura に限定できない（最小権限の原則——ADR-0004——がネットワーク越しに崩れる）
2. **保存データ・監査の主体をドメインの管理体系（グループ・監査ポリシー）に載せられない**
3. 専用ドメインアカウントを手動で作る運用は、パスワードの手動管理という新しい事故源（無期限パスワード・平文控え）を持ち込む

gMSA（グループ管理サービスアカウント）は、パスワードを AD が自動生成・自動ローテーションするサービス専用のドメインアカウントであり、Windows 管理者の既存スキル（AD 運用）にそのまま乗る。ADR-0004 決定 4 は gMSA 対応を予告済みであり、ADR-0006 基準 1 が v1.0 公開の必須条件として名指ししている。本 ADR はその実現方式を確定する。

**前提として重要な実測**: Kerberos SPN については、Issue #228 の AD lab 実機検証で**仮想サービスアカウントのまま（マシンアカウント SPN で）リモート管理 HTTPS の Kerberos SSO が成立する**ことを確認済み。したがって gMSA は「Kerberos を成立させるための必須手段」ではなく、上記 1〜3 を解く **AD 環境の運用正道としての opt-in** である（この位置づけの確定が ADR-0010 委任事項——「仮想サービスアカウントへの SPN 登録が成立しない場合、gMSA との接続で再評価する」——の決着になる: SPN は成立したため、gMSA は SPN 起因では要求されない）。

## 検討した選択肢

### 実行アカウントの指定方法

- **案 A: MSI プロパティ + セットアップ画面の入力欄（採用）**: `YAGURA_SERVICE_ACCOUNT`（既定 `NT SERVICE\Yagura`）をインストーラのプロパティとし、対話インストールではセットアップ画面に gMSA 名（`DOMAIN\name$`）の入力欄を出す。パスワード欄は設けない（gMSA はパスワード入力が不要——空パスワードでのサービス登録が仕様。パスワード欄の不在自体が「手動管理アカウントを想定しない」という設計意図の表明になる）
- 案 B: インストール後の切替スクリプトのみ: インストーラは常に仮想 SA で入れ、gMSA へは同梱スクリプトで切り替える。却下——AD 環境の新規導入で最初から gMSA を使いたいだけの利用者に 2 段階の手順を強い、GPO/サイレント配布（`msiexec /qn YAGURA_SERVICE_ACCOUNT=...`）の一発構成もできない
- 案 C: 管理 UI からの切替: サービスが自分自身の実行アカウントを変更して自己再起動する方式。却下——SCM 操作・ACL 付替・再起動を跨ぐ失敗時の中途状態が重く、v1.0 の対象状況（導入時に決める）に対して過剰

### 前提条件（gMSA の作成・取得許可）の扱い

- **案 A: AD 管理者の責務とし、製品はガイドと診断まで（採用）**: gMSA の作成（`New-ADServiceAccount`）・ホストへのパスワード取得許可（`PrincipalsAllowedToRetrieveManagedPassword`）は AD 側の操作であり製品外。Yagura 側はガイド（前提条件・確認コマンド `Test-ADServiceAccount`）と、サービス起動失敗時の切り分け情報を提供する
- 案 B: インストーラが gMSA を自動作成: 却下——AD への書き込みはドメイン管理者権限を要し、インストール実行者の権限前提が跳ね上がる。gMSA の配置（OU・命名・取得許可グループ）は組織のポリシー領域であり、製品が代行すべきでない（ADR-0001 の「Windows 管理者の既存スキルで運用できる」は、AD 管理の代行ではなく AD 管理への接続を意味する）

### 既存インストール（仮想 SA 稼働中）からの切替

- **案 A: 再インストール/アップグレード時の指定のみを製品スコープとする（採用。2026-07-16 オーナー裁定）**: 稼働中の切替（サービス停止 → `sc config` → 保存データ ACL 付替 → 起動）は operations ガイドの手順書として記載し、スクリプト・管理 UI 化はしない
- 案 B: 切替スクリプト同梱: 却下（再評価トリガ付き）——手順自体は単純で、ガイドの手順書で足りる。スクリプト化は検証面（ACL 付替の全域性・失敗時の巻き戻し）のコストが先行する。**再評価トリガ: 利用者から切替自動化の具体的要望が挙がったとき**

## 決定

1. **gMSA によるサービス実行を opt-in として提供する**。既定は仮想サービスアカウント `NT SERVICE\Yagura` のまま変えない（ゼロ設定ファーストラン・非 AD 環境の体験は不変）
2. **指定はインストーラで行う**: MSI プロパティ `YAGURA_SERVICE_ACCOUNT`（既定 `NT SERVICE\Yagura`）+ セットアップ画面の入力欄（パスワード欄なし）。`Package.wxs` の `ServiceInstall/@Account` をプロパティ間接参照にする（`[YaguraFwProfile]` と同じ手法）。値の形式検証（`DOMAIN\name$` または `NT SERVICE\Yagura`）はインストーラ側で行い、想定外の形式（一般ユーザーアカウント等）は警告する——**gMSA と仮想 SA 以外での実行は動作対象外**（LocalSystem 禁止——ADR-0004——を含む）
3. **前提条件は AD 管理者の責務**: gMSA の作成・取得許可・（KDS ルートキーを含む）AD 側の準備は製品外。利用者ガイドに前提条件・確認コマンド・起動失敗時（パスワード取得不可 = サービス起動エラー 1069 系）の切り分けを記載する
4. **保存データ ACL の付与先を実行アカウントに追随させる**: インストーラの ACL 付与（データルート・フォワーダ配置・監査領域——security.md §5）は `YAGURA_SERVICE_ACCOUNT` の値を主体として適用する。gMSA 版の具体 ACE は AD lab の実機検証で確定し、security.md §5 に `icacls` 出力の形で記録する（SEC-3 と同じ作法）
5. **DB 接続**: gMSA 実行時、SQL Server への Windows 統合認証はサービス固有のドメイン識別で成立する。database.md §6.1（昇格ウィザードの接続組み立て）に「Windows 統合認証の実行主体は実行アカウント（仮想 SA = マシンアカウント / gMSA = 専用識別）」を明記し、gMSA を推奨構成として案内する
6. **切替は再インストール/アップグレード時の指定のみ**（検討した選択肢のとおり。稼働中切替は operations ガイドの手順書）
7. **受け入れ条件は AD lab の実機検証**（ADR-0006 基準 1 の要求）: yagura.test DC で「gMSA 作成 → インストール（プロパティ指定）→ 受信 → 閲覧 → リモート SQL Server への統合認証接続」の E2E を通し、記録を残す。SPN: リモート管理 HTTPS（Negotiate）利用環境で gMSA へ移行する場合の SPN 付け替え要否も同じ lab で確認する

## 帰結

- **良くなること**: AD 環境で「サービス固有のドメイン識別・パスワード自動管理・DB 側の最小権限付与」が成立する。GPO/サイレント配布で最初から gMSA 構成にできる。v1.0 基準 1 の gMSA 要件を満たす
- **悪くなること・受け入れるリスク**: ①インストーラの分岐（アカウント検証・ACL 付与先の切替）が増え、lab 検証項目が増える ②gMSA の準備不足（取得許可漏れ）によるサービス起動失敗という新しいつまずき点が生まれる——ガイドと診断情報で緩和するが、AD 側の設定ミスを製品が検出することはできない ③MSI の `ServiceInstall/@Account` への gMSA 指定が全環境で成立するかは実機依存であり、lab 検証までは確定しない（成立しない場合はカスタムアクションでの `sc config` 後追い設定へフォールバックする——実装時判断）
- **スコープ外の明示**: sMSA（単独管理サービスアカウント。gMSA に置き換えられた旧世代）・一般ドメインアカウントでの実行・複数サービスインスタンスでの gMSA 共有は対象外

## 先送りにする場合の再評価トリガ

- 切替スクリプトの同梱（決定 6 の却下案 B）: 利用者から切替自動化の具体的要望が挙がったとき
- 一般ドメインアカウント対応（スコープ外）: gMSA を建てられない環境（KDS ルートキー未整備等）からの具体的要望が反復（3 件以上）したとき——その場合もパスワード管理の事故源の議論を必須とする
