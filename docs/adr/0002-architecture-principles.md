# ADR-0002: アーキテクチャ原則

- 状態: accepted
- 日付: 2026-07-03
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md)（導入体験の原則・品質の原則の実装方針）、[全体設計書](../design/README.md)

## 文脈と課題

ADR-0001 は「インストールから最初のログ閲覧まで追加のサーバ製品なしで到達できること」（導入体験の原則）と「ログを失わないこと。失った場合に必ず観測できること」（品質の原則）を設計の受け入れ条件として定めた。個別機能の設計に入る前に、この 2 つの原則を成立させるための全体アーキテクチャの原則を固定する。

後から変更するコストが最も高い横断的関心事——プロセス構成・永続化の抽象・信頼性機構の挿入点・観測性——を対象とし、個別機能の詳細（画面設計・プロトコル仕様・数値）は全体設計書と後続文書に委ねる。

## 決定

### 1. ランタイムとプロセス構成

- **C# / .NET（現行 LTS。開始時点は .NET 10）** で実装する（選定理由と却下案は「検討した選択肢」参照）
- **Windows 専用**とする。.NET 自体はクロスプラットフォームだが、Linux 対応は ADR-0001 のターゲット定義（Windows 管理者のスキル活用）から意図的にスコープ外とする
- **単一の Windows サービスプロセス**に受信・永続化・Web UI を同居させ、**1 つのインストーラ**で配布する。導入の単純さを最優先する
- 内部はアセンブリ境界（受信・永続化・UI）で分離し、将来の別プロセス化の可能性を閉じない。ただしアセンブリ境界は保守性の境界であってセキュリティ境界ではない（同居の代償は「帰結」参照）

### 2. 受信パイプラインと信頼性機構

受信から保存までを「受信 → 解析 → バッファ → 永続化」の非同期パイプラインとして構成し、以下を**初期設計からの標準装備**とする（後からの増築にしない）。各項目には v0.1 での到達水準を明記する。

- **バックプレッシャ**（v0.1: 完動）: 各段は容量制限付きで接続する。溢れの処理は段ごとにブロックかドロップかを設計時に明示し、**受信段では、OS 側の計測不能なロスに転化するブロックを選ばない**。ドロップは必ず発生箇所別カウンタに計上する
- **ディスクスプール**（v0.1: 完動）: 保存先 DB に書き込めない間、受信済みログをローカルディスクへ退避し、復旧後に自動で追いつく。発動条件は保存先の停止に限らず、**持続的な速度不足（保存先が生きているが受信に追いつかない状態）も対象とする**。**既定で有効**とする（「ログを失わない」原則に対し、知らないと失う opt-in ではなく、不要な環境が切る opt-out が正しい向き）。ディスク使用量には上限を設け、上限到達時の挙動（新規破棄か古い順の回転か）は全体設計書で定めるが、**いずれの場合も破棄は発生箇所別カウンタに計上する**。スプール領域は**ログ本体と同等の保護対象**とする（ACL・削除保証等の具体はセキュリティモデル ADR と全体設計書で定める）
- **流量制御**（v0.1: 挿入点のみ。機能実装は初期バージョン群で提供）: 暴走した送信元が全体を巻き込むことを防ぐ送信元単位の制限。実装時の既定は品質の原則との整合上 **opt-out（既定有効）を基本線**とし、既定閾値は「通常運用で発火しない安全側」に実測で定める。発火は必ず計測される
- **発生箇所別のドロップ計測**（v0.1: 完動）: 取りこぼしは「OS ソケット / 内部バッファ / 流量制御 / 永続化失敗」を区別してカウンタ計測する。**OS レベルの取りこぼしも観測対象とする**（アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る、という前提で設計する）。OS 統計の実現手段と粒度は実機検証（2026-07-05、M4-4）を経て v0.1 の方式を確定した: 取得手段は .NET 標準 API（`IPGlobalProperties` の UDP 統計）、**粒度はシステム全体統計**（同居プロセスの UDP 破棄も含む値になる。**検証範囲は .NET 標準 API に限定しており、非標準手段——ETW・パフォーマンスカウンタ等——は検証対象外とした**スコープ判断である。詳細・前提条件・代替手段の採否は [architecture.md §4.2](../design/architecture.md)）。原則——OS 側のロスの存在を無視しない——は成立するが、**この値単独では取りこぼしの原因が本製品由来かを切り分けられない**（§改訂履歴 1）

配送の意味論は **at-least-once**（重複より欠損を避ける）とする。スプールからの復旧時等に重複保存が起こり得ることを仕様として許容し、重複排除は約束しない。

### 3. 永続化: provider 抽象

- 永続化層は**初日から provider 抽象**とし、SQL 方言・機能差は provider 層に封じ込める（導入時期の対抗案と却下理由は「検討した選択肢」参照）
- **組み込み既定: SQLite**（v0.1 必須）。インストール直後、DB の準備なしで受信・閲覧を成立させる。**組み込み DB のまま運用し続ける選択も正式サポートとする**（小規模環境では昇格が不要なまま使い続けられることを想定する）
- **本番の第一推奨: SQL Server**（v0.1 必須。これは本 ADR の決定である——昇格先が存在しない状態で v0.1 を出すと、ADR-0001 決定 3 の「外部 DB は本番昇格時に選択する」が言葉だけになるため）。**無償の SQL Server Express を含めて動作対象**とし、SQL Server を保有しない環境の現実的な昇格先とする
- **PostgreSQL / MySQL はサポート対象への追加を意図する**（採否と時期は抽象の実証後に個別に確定する。約束ではなく意図である）。provider が満たすべき契約（インターフェースと適合テスト）は全体設計書で定義し、外部コントリビュータによる provider 追加の入口とする
- **本番昇格は製品として支援する**。約束の範囲を分解すると: (i) 組み込み DB の実用限界の観測と昇格の案内（v0.1。**案内は利用者が抑制でき**、組み込み DB のまま使い続ける判断を尊重する）、(ii) ウィザードからの接続切替（v0.1。切替以降の新規ログは新 DB へ）、(iii) 蓄積済みログの移行機能（後続バージョン。それ単体で提案書 1 本分の規模があるため、時期と方式は個別に確定する。提供までの間は手動移行手順を文書で提供する）

### 4. 観測性と性能

- メトリクスは .NET 標準の計測 API（`System.Diagnostics.Metrics`）で実装し、出力先（ログ・Web UI・外部監視）は差し替え可能にする。**メトリクスを含むあらゆるデータの外部送信は既定で無効とし、テレメトリの自動送信は行わない**
- **スループットベンチハーネスを v0.1 の開発範囲に含める**。ベンチ結果は「測って終わり」にせず、**回帰の合否基準として CI で判定する**（判定方式——絶対値か基準比か——は実行環境の揺らぎを踏まえ全体設計書で定める）
- 容量・バッチサイズ・性能目標などの数値は推測で固定せず、**ベンチ実測で確定してから**全体設計書に記録する

### 5. 設定

- 初期設定は Web UI のセットアップウィザードで生成し、**設定ファイルの手編集を前提としない**（手編集は引き続き可能だが、必須にしない）
- 設定スキーマは additive-only（[conventions.md](../development/conventions.md) の deprecation 階段に従う）

### 6. セキュリティ境界の挿入点

- 既定は信頼ネットワーク前提（ADR-0001 決定 5）。本 ADR での原則は、opt-in 強化が「後付けの改造」ではなく「挿入点への装着」になる構造を最初から持つこと
- 挿入点として確保する対象: **通信の保護**（TLS 受信）、**利用者の認証・管理操作の認可**、**保存データの保護**（スプール領域・組み込み DB ファイル・接続資格情報）、**設定変更の監査記録**。いずれも具体的な範囲・既定・提供時期はセキュリティモデル ADR（起案予定 0004）で定める

## 検討した選択肢

### ランタイム

- **(a) C# / .NET（採用）**: Windows サービス・AD・SQL Server 接続の純正サポートが揃い、ターゲットユーザーの環境と親和する。オーナーのスキルセットとも一致し、単一言語で UI（Blazor）まで書ける
- (b) Go / Rust（単一バイナリ配布）: 配布は軽いが、AD 認証・Windows サービス統合・SQL Server 接続をそれぞれサードパーティ実装に頼ることになり、本製品の中核領域でエコシステムの薄い側を選ぶことになる。却下
- (c) Java 系: ランタイム導入が「追加のサーバ製品」に近い運用負担になり、Windows 統合も弱い。却下

### プロセス構成

- **(a) 単一サービス + 1 インストーラ（採用）**: 導入体験の原則に直結。ターゲット環境（1 台の Windows サーバ）に適合
- (b) 受信と UI を別プロセス・別配布に分離: 大規模環境では利点があるが、導入手数が増え、v0.1 のターゲットには過剰。アセンブリ境界の維持で将来の分離余地は残るため却下

### 永続化抽象の導入時期

- **(a) 初日から抽象 + SQLite / SQL Server の 2 provider を v0.1 必須（採用）**: ゼロ設定ファーストラン（組み込み）と本番昇格先（外部）の両方が v0.1 の約束に含まれるため、2 つの provider は投機ではなく必要要件。抽象の縫い目は 2 実装があって初めて検証される
- (b) v0.1 は SQLite 単独で開始し、外部 DB は後続で追加: 実装量は減るが、昇格先が存在しない期間は ADR-0001 決定 3 の「本番昇格時に選択」が言葉だけになる。また単一実装に合わせて作った抽象は後続 provider で作り直しになりやすい。却下
- (c) 抽象なしで 2 実装を条件分岐で持つ: 短期は速いが、分岐が全域に散り、3 つ目の provider 追加時に全面改修になる。却下

なお、2 実装での検証が第 3・第 4 の provider への適合を保証するわけではない（抽象は手元の実装に過学習し得る）。これが PostgreSQL / MySQL を「約束」でなく「意図」に留める理由である。

### ゼロ設定ファーストランの組み込み DB

- **(a) SQLite（採用）**: 追加インストール不要・単一ファイル・実績豊富。SQL Server とはアーキテクチャ（組み込み / サーバ型）と方言の両面で離れており、抽象の検証相手として適する——ただし方言差だけで縫い目の検証に「十分」とは断定しない。抽象の漏れは運用特性（一括挿入・保持期間管理の実装手段等）にも現れるため、十分性は provider 適合テストの設計で担保する
- (b) SQL Server Express LocalDB: SQL Server と方言が同じで昇格は容易だが、ランタイムの追加インストールが必要になり「追加のサーバ製品なし」と緊張する。却下
- (c) ゼロ設定ファーストランを諦め、初回から SQL Server を要求: ADR-0001 決定 3 に反する。却下

### ディスクスプールの既定と方式

- **(a) 障害時退避方式・既定で有効・opt-out（採用）**: 「ログを失わない」を既定の挙動にする。ディスク上限と観測可能性を必須条件として添える
- (b) 既定で無効・opt-in: 障害時にしか動かないコードパスが増えない、無人環境でディスクを消費しない、組み込み DB と同一ディスクの I/O 競合がない——という実利はある。しかし品質の原則に照らすと「仕組みを知っている運用者しか守られない」既定であり、却下。なお (b) の利点が示すコストは実在するため、採用案には上限・観測可能性・保護対象化を必須条件として課している
- (c) 常時 WAL 経由（全受信をまずディスクジャーナルに書き、そこから DB へ流す）: スプール経路が常時本番トラフィックで検証される（障害時専用経路は最悪の瞬間に最少テストのコードが走る、という懸念への最強の回答）。しかし全構成に常時のディスク書き込みコストを課し、ゼロ設定構成では組み込み DB と同一ディスクで I/O が倍加する。導入体験の原則（小規模環境で軽く動く）を優先して却下。ただし懸念自体は正当なので、**スプール経路を平常時にも定期実行して検証する仕組み**を全体設計書の宿題とする

## 帰結

- 良くなること: 導入体験と信頼性という 2 つの原則が、機能の後付けではなくアーキテクチャの形として担保される。provider 抽象が最初から本物の 2 実装で検証される
- 悪くなること:
  - provider 抽象の維持コストが常に掛かる（新機能は複数 DB での動作を考慮する）
  - 単一プロセス同居の障害波及: UI の重い操作（大量ログの検索等）が受信パイプラインと資源（スレッド・GC・ディスク I/O）を奪い合う。また**サービス再起動は受信断であり、syslog は再送されないため再起動時間はそのままロス時間になる**。UI 側の資源制限と再起動時間の最小化を全体設計書の宿題とする
  - Web UI の脆弱性が受信・永続化と同一プロセス権限で影響する。緩和の中核はサービスアカウントの権限最小化であり、セキュリティモデル ADR で定める
- 受け入れるリスク: SQLite の書き込み性能には限界があり、ゼロ設定のまま大量ログを受けると飽和し得る。緩和策として、限界値の実測・飽和の観測可能化（スプールの速度不足発動を含む）・昇格の案内を v0.1 の設計に含める（放置して「静かに落ちる」ことを許さない）

## 先送りにする場合の再評価トリガ

- PostgreSQL / MySQL の追加時期: provider 抽象の実証（SQLite + SQL Server の 2 実装が本番品質で動作）完了後に着手判断。利用者からの要望が先行して反復した場合は前倒しを検討する。**実証完了前に外部から provider 追加の PR が届いた場合は、適合テストの整備を優先課題に繰り上げて受け入れ可否を判断する**（放置しない）
- ベンチ実測で SQLite がゼロ設定ファーストランの想定負荷を満たさない場合、組み込み DB の選定を再評価する（その場合も「追加のサーバ製品なし」の原則は維持する）
- OS レベルの取りこぼし観測が実機検証で意図した粒度で実現できないと判明した場合、決定 2 の該当箇所を amendment で実態に合わせる（システム全体統計への縮退等）。原則（OS 側のロスを無視しない）自体は維持する。**→ 2026-07-05 発動**（.NET 標準 API の範囲でソケット/ポート単位の粒度が得られないことを確認。§改訂履歴 1 で v0.1 の方式を確定）
- **OS 統計の粒度の再挑戦**（改訂履歴 1 の縮退は恒久確定ではない）: 次のいずれかが生じた場合、非標準手段（ETW・パフォーマンスカウンタ等）を含めた per-socket/per-port 粒度の実現可能性を再評価する——(a) 利用者から同居環境での誤読・誤診断の報告が反復した場合、(b) .NET の将来バージョンで標準 API にソケット/ポート単位の UDP 統計が追加された場合、(c) 流量制御の本実装（初期バージョン群）で送信元単位の破棄観測との統合を設計する際

## 改訂履歴

本節は Decision を変えない amendment（状態: accepted を維持）の記録である。Decision を変える場合は supersession（新規 ADR）による。

1. **2026-07-05 — 決定 2「OS レベル取りこぼし観測」の v0.1 粒度をシステム全体統計に確定**: 上記再評価トリガの事前定義どおりの発動。M4-4（マイルストーン M4 の 4 番目の作業単位。実装計画 Issue #19 / PR #43）の実機検証（Windows ARM64 / .NET 10）で、取得手段は `IPGlobalProperties.GetUdpIPv4Statistics()` / `GetUdpIPv6Statistics()` の `IncomingDatagramsDiscarded` として確定した（負荷に応じて増加する実値を確認）
   - **検証範囲の明示**: 本判断の検証範囲は .NET 標準 API に限定した。非標準手段（ETW・Windows パフォーマンスカウンタ・WinSock 拡張 API 等）は「技術的に不可能」と検証したのではなく、**P/Invoke・OS バージョン依存の保守コストを理由に v0.1 では検証対象外とした**スコープ判断である（採否の内訳は [architecture.md §4.2](../design/architecture.md)）
   - **失われたもの（証拠能力の縮退）**: システム全体統計は同居プロセスの UDP 破棄を含むため、**この値の増加から「本製品が取りこぼした」と断定することはできなくなった**。本製品が実質的に UDP 受信を専有する環境でのみ有効な近似値であり、同居 UDP サービス（DNS・SNMP・監視エージェント等）がある環境では偽陽性が生じ得る
   - 緩和（プロセス起動時からの差分公開・受信バッファ設定・ベンチでの突合検証）と前提条件・監査記録との区別は [architecture.md §4.2](../design/architecture.md) に記載。粒度の限界と値の解釈ガイドは利用者向けドキュメントに明記し（同 §10 D-6）、**UI にこのゲージを表示する際は「システム全体値であり同居プロセスの破棄を含む」旨の常時可視の注記を必須とする**（M8 の状態画面実装時に適用）
