# ADR-0002: アーキテクチャ原則

- 状態: proposed
- 日付: 2026-07-03
- 決定者: YANAI Taketo
- 関連: [ADR-0001](0001-project-founding.md)（導入体験の原則・品質の原則の実装方針）

## 文脈と課題

ADR-0001 は「インストールから最初のログ閲覧まで追加のサーバ製品なしで到達できること」（導入体験の原則）と「ログを失わないこと。失った場合に必ず観測できること」（品質の原則）を設計の受け入れ条件として定めた。個別機能の設計に入る前に、この 2 つの原則を成立させるための全体アーキテクチャの原則を固定する。

後から変更するコストが最も高い横断的関心事——プロセス構成・永続化の抽象・信頼性機構の挿入点・観測性——を対象とし、個別機能の詳細（画面設計・プロトコル仕様等）は全体設計書と後続 ADR に委ねる。

## 決定

### 1. ランタイムとプロセス構成

- **C# / .NET（現行 LTS。開始時点は .NET 10）** で実装する。ターゲットユーザーの環境（Windows Server）に対する純正サポート（Windows サービス・AD・SQL Server 接続）が揃っており、オーナーのスキルセットとも一致する
- **単一の Windows サービスプロセス**に受信・永続化・Web UI を同居させ、**1 つのインストーラ**で配布する。導入の単純さを最優先する
- ただし内部はアセンブリ境界（受信・永続化・UI）で分離し、将来の別プロセス化・別ホスト化の可能性を閉じない

### 2. 受信パイプラインと信頼性機構

受信から保存までを「受信 → 解析 → バッファ → 永続化」の非同期パイプラインとして構成し、以下を**初期設計からの標準装備**とする（後からの増築にしない）:

- **バックプレッシャ**: 各段は容量制限付きで接続し、溢れは制御された方法で処理する
- **ディスクスプール**: 保存先 DB に書き込めない間、受信済みログをローカルディスクへ退避し、復旧後に自動で追いつく。**既定で有効**とする（「ログを失わない」原則に対し、知らないと失う opt-in ではなく、不要な環境が切る opt-out が正しい向き）。ディスク使用量には上限を設け、上限到達も観測可能にする
- **流量制御**: 暴走した送信元が全体を巻き込むことを防ぐ仕組み（送信元単位の制限）を、パイプラインの挿入点として最初から確保する（既定の有効/無効と閾値は実測を踏まえ全体設計書で定める）
- **発生箇所別のドロップ計測**: 取りこぼしは「OS ソケット / 内部バッファ / 流量制御 / 永続化失敗」を区別してカウンタ計測する。**OS レベルの取りこぼしも観測対象とする**（アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る、という前提で設計する）

### 3. 永続化: provider 抽象

- 永続化層は**初日から provider 抽象**とし、SQL 方言・機能差は provider 層に封じ込める。この抽象は投機ではない——ゼロ設定ファーストラン（組み込み DB）と本番運用（外部 DB）という **2 つの provider が v0.1 の時点で必須**であり、抽象の縫い目は最初から要る
- **組み込み既定: SQLite**。インストール直後、DB の準備なしで受信・閲覧を成立させる
- **本番の第一推奨: SQL Server**（ターゲットユーザーのスキル活用。ADR-0001 決定 2）
- **PostgreSQL / MySQL を正式サポート対象**とする。ただし v0.1 の必須範囲は SQLite + SQL Server とし、PostgreSQL / MySQL は抽象の実証後に追加する（対応順は利用者要望で調整）
- **本番昇格を製品機能として提供する**: 組み込み DB から外部 DB への移行（蓄積済みログと設定の引き継ぎ）を手順書ではなく機能として実装する。組み込み DB の実用限界（目安値は実測で確定）に近づいたら、Web UI 上で昇格を促す

### 4. 観測性と性能

- メトリクスは .NET 標準の計測 API（`System.Diagnostics.Metrics`）で実装し、出力先（ログ・Web UI・外部監視）は差し替え可能にする
- **スループットベンチハーネスを v0.1 の開発範囲に含める**。ベンチ結果は「測って終わり」にせず、**回帰の合否基準として CI で判定する**（基準値を下回ったら劣化として検出）
- 容量・バッチサイズ・性能目標などの数値は推測で固定せず、**ベンチ実測で確定してから**全体設計書に記録する

### 5. 設定

- 初期設定は Web UI のセットアップウィザードで生成し、**設定ファイルの手編集を前提としない**（手編集は引き続き可能だが、必須にしない）
- 設定スキーマは additive-only（[conventions.md](../development/conventions.md) の deprecation 階段に従う）

### 6. セキュリティ境界の挿入点

- 既定は信頼ネットワーク前提（ADR-0001 決定 5）。本 ADR での原則は、**TLS 受信・認証などの opt-in 強化が「後付けの改造」ではなく「挿入点への装着」になる構造**を最初から持つこと。具体的な範囲と提供時期はセキュリティモデル ADR（起案予定 0004）で定める

## 検討した選択肢

### プロセス構成

- **(a) 単一サービス + 1 インストーラ（採用）**: 導入体験の原則に直結。ターゲット環境（1 台の Windows サーバ）に適合
- (b) 受信と UI を別プロセス・別配布に分離: 大規模環境では利点があるが、導入手数が増え、v0.1 のターゲットには過剰。アセンブリ境界の維持で将来の分離余地は残るため却下

### ゼロ設定ファーストランの組み込み DB

- **(a) SQLite（採用）**: 追加インストール不要・単一ファイル・実績豊富。provider 抽象の実証相手としても、SQL Server と方言が十分に異なり縫い目の検証になる
- (b) SQL Server Express LocalDB: SQL Server と方言が同じで昇格は容易だが、ランタイムの追加インストールが必要になり「追加のサーバ製品なし」と緊張する。却下
- (c) ゼロ設定ファーストランを諦め、初回から SQL Server を要求: ADR-0001 決定 3 に反する。却下

### ディスクスプールの既定

- **(a) 既定で有効・opt-out（採用）**: 「ログを失わない」を既定の挙動にする。ディスク上限と観測可能性を必須条件として添える
- (b) 既定で無効・opt-in: 仕組みを知っている運用者しか守られない。品質の原則（ADR-0001 決定 4）と整合しないため却下

## 帰結

- 良くなること: 導入体験と信頼性という 2 つの原則が、機能の後付けではなくアーキテクチャの形として担保される。provider 抽象が最初から本物の 2 実装で検証される
- 悪くなること: provider 抽象の維持コストが常に掛かる（新機能は複数 DB での動作を考慮する）。単一プロセス構成は超大規模環境への訴求力を持たない
- 受け入れるリスク: SQLite の書き込み性能には限界があり、ゼロ設定のまま大量ログを受けると飽和し得る。緩和策として、限界値の実測・飽和の観測可能化・本番昇格の促しを v0.1 の設計に含める（放置して「静かに落ちる」ことを許さない）

## 先送りにする場合の再評価トリガ

- PostgreSQL / MySQL の追加時期: provider 抽象の実証（SQLite + SQL Server の 2 実装が本番品質で動作）完了後に着手判断。利用者からの要望が先行して反復した場合は前倒しを検討する
- ベンチ実測で SQLite がゼロ設定ファーストランの想定負荷を満たさない場合、組み込み DB の選定を再評価する（その場合も「追加のサーバ製品なし」の原則は維持する）
