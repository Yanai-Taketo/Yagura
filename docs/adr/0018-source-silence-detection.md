# ADR-0018: 送信元の途絶検知（ウォッチリスト方式。opt-in）

- 状態: proposed
- 日付: 2026-07-18
- 決定者: YANAI Taketo
- 関連: [ADR-0008](0008-forwarder-kit-generation.md) 委任事項 4 / [ADR-0016](0016-os-drop-gauge-mechanism.md)（無音化検出への一本化）/ [architecture.md](../design/architecture.md) §4.6（能動通知）/ [ui.md](../design/ui.md) §4（送信元別受信状況・UI-4）/ [database.md](../design/database.md) §1.2（`QuerySourceActivityAsync`）/ ADR-0017（メール通知——proposed・並行起案 PR #337）/ Issue #275 / PR #339

## 文脈と課題

「送信元が黙った」ことは、syslog 集約サーバにとって最も気づきにくい障害である。送信側の設定消失・機器故障・経路遮断・ファイアウォール変更のいずれでも、サーバ側には**何のエラーも現れず、ただログが来なくなるだけ**——現行の観測性（カウンタ・ゲージ・能動通知）はすべて「サーバ自身の異常」を対象としており、「来るはずのものが来ていない」を検出する機構がない。なお送信停止は運用障害だけでなく、**侵害されたホストによる証跡遮断（攻撃者がログ送信を止める）の兆候**でもある——サーバ側から両者は区別できないため通知は運用警告として出すが、この二面性は利用者向けドキュメントに明記する（申し送り D-1）。

現状で存在するのは**受動的な可視化のみ**である: ダッシュボードの送信元別受信状況（UI-4。`QuerySourceActivityAsync` による最終受信時刻の古い順の全送信元一覧）は「いつも静かな装置が黙った」を上位 N 表示に埋もれさせない設計として確立済みだが、**運用者が画面を見に行かない限り機能しない**。ADR-0016 も OS レベル観測の撤回に際し「無音化検出」への一本化を打ち出したが、その無音化検出自体は受動のままである。

ADR-0008 は委任事項 4 として「転送の観測性（配布端末別の最終受信・**途絶検知**）」を別 ADR へ委任した。本 ADR はその引き取りである（2026-07-16 オーナー裁定で起案予定へ登録済み。Issue #275）。主論点は、①監視対象の指定方式 ②途絶判定（閾値・窓）③能動通知への接続 ④送信元数がスケールしたときの状態管理コスト。

### スコープ（非目標）

- **能動的な死活監視はしない**（ping・SNMP・エージェント heartbeat 等）。判定材料は「syslog が届いた」事実のみ——Yagura は syslog サーバであり、監視システム（商用 SIEM / NMS 領域。ADR-0001）にならない。ただし逆に、**NMS の死活監視は本機能を代替できない**——「装置は生きているが syslog 送信設定が消えた」は受信側でしか検出できず、これが本機能の固有価値である
- **ログ内容に基づく判定はしない**（「特定メッセージが一定時間ない」等のルール）
- **期待受信間隔の自動学習はしない**（検討した選択肢 1-(a) 参照）
- 送信側（フォワーダ）への heartbeat 送出機能の追加はしない（キット側の変更はスコープ外）

## 検討した選択肢

### 1. 監視対象の指定方式

- **(a) 全送信元の自動学習**: 送信元ごとの受信間隔を学習し、逸脱を検知する。登録の手間はないが、①DHCP アドレス変動・機器退役のたびに誤検知が発生し続ける ②「正常な間隔」の推定はログ種別・夜間バッチ・週次機器で本質的に多峰であり、誤検知を抑える閾値チューニング UI という SIEM 的装置が必要になる ③全送信元分の状態を持つため、送信元数の暴走（詐称を含む）に対して非有界——本プロジェクトが監査（#313〜#317）で繰り返し潰してきた「送信元 IP 起点の無制限辞書」の再導入になる。**却下**
- **(b) 明示的なウォッチリスト（opt-in）**: 運用者が「途絶を知りたい送信元」を明示登録する（上限つき）。登録の手間はあるが、①検知対象 = 運用者が守ると決めた送信元であり誤検知の責任構造が明確 ②状態は登録数で有界。**採用**。なお「重要な装置は数えるほど」という現場像は**想定であり、独立した利用者からの報告に基づくものではない**（ADR-0008 改訂履歴 1 と同じ透明性の作法で明記する）——上限は数百台の配布現場の全数登録も収まる値に置く（決定 1）
- **(c) 作らない（受動可視化で足りる）**: UI-4 の一覧は既にあり、Top talkers（#288）同様に「見れば分かる」。しかし「UI を開いていない夜間に気づけない」という能動通知の存在意義（architecture.md §4.6）がここにも当てはまり、ADR-0008 委任 4 は「本 ADR の採否に関わらず必要性は残る」と明記している。**却下**（受動可視化は本機能の土台として残る）

### 2. 途絶判定のデータ源

- **(a) DB への周期集計クエリ**: `QuerySourceActivityAsync` を周期実行して最終受信時刻を得る。実装は薄いが、SourceAddress 列に索引がなく（Issue #145 の整理）、窓なし集計は大規模時にフルスキャンへ劣化し得る——1 分周期の監視ループに載せる負荷として不適。**却下**（起動時の初期値取得 1 回のみに限定して使う）
- **(b) 受信経路でのインメモリ追跡（ウォッチリスト限定）**: 解析段でウォッチリスト所属の送信元のみ最終受信時刻を更新する（有界辞書——登録上限で cap）。ホットパスへの追加コストは「凍結済みセットの membership 判定 + 時刻代入」のみ。**採用**

### 3. 判定主体と通知

- `ActiveNotificationMonitor` の周期評価（1 分間隔）に評価メソッドを 1 つ追加する——自己検証（Issue #152）・証明書期限（1014/1017）と同じ確立パターン。新しい常駐機構は作らない

## 決定

### 1. ウォッチリスト方式の途絶検知を opt-in で追加する

- 新設キー `Notification:SourceSilence:*`（既定 = リスト未設定 = 機能無効）。ゼロ設定ファーストランには現れない
- **登録単位は送信元アドレス**とし、エントリは `{Address, Label(任意), ThresholdMinutes(任意)}` の構造化配列キー `Notification:SourceSilence:Watchlist` で保持する（**本プロジェクト初のオブジェクト構造化配列キーであり、既存の配列キー（`*Groups` = KnownArrayKeys 系統: 手編集のみ・再起動固定・比較対象外）とは別系統の新機構になる**——設定機構との整合は委任 3 で確定する）
- **登録上限は仮値 1000 件**（有界化はこの上限が担う。当初案 100 は「数百台の配布現場・通信事業者規模で初日から不足する」というレビュー指摘——鈴木——を受けて引き上げた。エントリ実体は数十バイト・評価は O(n) の時刻比較のみで、技術コストは 100 と変わらない）
- **上限の適用規則**: UI 経由は上限到達で追加を拒否し理由を表示する。手編集で上限超過のリストを読んだ場合は**ファイル順で上限までを有効とし、超過分を無効化して警告する（警告に無効化した対象アドレスを列挙する——「監視されているつもりで監視されていない」検知ギャップを黙らせない）**。UI と手編集の非対称は意図的である（UI は入力時点で防げる・手編集は読み込み時点では縮小継続しかできない——configuration.md §1 の縮小原則）
- 既定閾値は `Notification:SourceSilence:DefaultThresholdMinutes`（仮値 **1440 分 = 24 時間**）。ただし**これは手編集で閾値を省略したエントリの補完値であり、UI 経由の登録では閾値の明示確定を必須とする**（決定 4——「登録した = すぐ気づける」という期待と 24 時間既定のズレを、既定値の変更ではなく登録動線で解消する。レビュー指摘——佐藤・クリス）。`ThresholdMinutes` の受理範囲は**下限 10 分（評価周期 1 分 + 送信ジッタを考慮した仮値）・上限 43200 分（30 日）**とし、範囲外は当該エントリ無効化 + 警告
- **不正時挙動（configuration.md §1 の 3 分類 + エントリ単位の縮小）**: エントリの `Address` 形式不正・`ThresholdMinutes` 範囲外は**当該エントリのみ無効化して警告**する（リスト全体は殺さない——1 エントリのタイポで他の監視まで止めない。警告に対象エントリを明示する）。キー全体が解釈不能（配列として読めない等）な場合は機能無効化 + 警告。**空配列 `[]` は「正常な空リスト = 機能無効（警告なし）」とし、§1 の「空配列 = 不正値」規定の例外とする**（あの規定は「空 + 機能有効 = 誰も対象にならない」文脈のもの。本キーは空 = 意図的な無効が自然な意味論。§1/§8 への規約反映——エントリ単位無効化という第 4 の挙動を含む——は委任 3）

### 2. 「配布端末別」の委任は、途絶検知の中核を引き取り、全数自動観測はスコープ外化する

ADR-0008 委任 4 の原文シナリオ（数十〜数百台へ配布したキットの転送の全数把握）に対する本 ADR の立場を正確に書く:

- **引き取るもの**: 送信元単位の最終受信保持と途絶検知（委任の中核）。上限 1000 件は数百台規模の全数登録を物理的に収容する
- **スコープ外化するもの**: 配布端末とキットの機械的対応付け・自動登録。キット ID の RFC 5424 SD-ELEMENT 埋め込み等による端末識別は、①転送フォーマットへの侵襲 ②Fluent Bit テンプレートの改変（検証済み版の表明——ADR-0008 委任 3——への影響）③キットを使わない送信元（機器直送）に適用不能、の 3 点で**却下**。登録・維持は運用者の手作業（+ 決定 4 の候補選択・手編集一括投入）に残る——この残余は再評価トリガに独立項として登録する
- ウォッチリストの `Label`（表示名）が「どの端末・装置か」の対応を運用者の語彙で保持する。配布端末の資産管理は運用者の領分であり、製品は送信元アドレスという観測可能な単位に徹する

### 3. 判定と状態管理

- **「受信した」の定義（一文で固定する）**: **解析段（`ParsingStage`）に到達した当該送信元のレコードの存在**をもって受信とする（解析の成否は問わない——解析失敗の raw 保存も受信。流量制御ゲートを通過した後・保存の成否は問わない）。この定義の帰結として、①流量制御でほぼ全量破棄される送信元は途絶に見え得る ②保存に失敗し続けている送信元は健在に見える ③**当該アドレスを詐称したパケットも受信に数える**（帰結の限界参照）
- **追跡の実装**: `ParsingStage` 内でウォッチリスト所属送信元の最終受信時刻をインメモリ更新する（ウォッチリストの参照交換は `UpdateDefaultRfc3164TimeZone` と同じパターン。凍結済みルックアップ + 上限 1000 の有界辞書。IPv4-mapped IPv6 の正規化は流量制御・Top talkers と同じ既存規約）。**加えて、スプール drain の合流点でも該当送信元の最終受信時刻を `max()` 更新する**——深いスプール滞留 + 再起動の組で「直前まで送っていた装置」が途絶に見える偽陽性（レビュー指摘——鈴木・クリス。§3.2.2 は滞留を正常状態と明記しており「短い窓」ではない）を塞ぐ。追跡は**単調クロック（`TimeProvider.GetTimestamp`）基準**とし、壁時計は表示・記録にのみ使う（NTP ステップでの一斉誤発火を防ぐ。`TokenBucketIngressGate` の前例を踏襲）
- **起動時の初期値（seed）**: `QuerySourceActivityAsync` を起動時に 1 回照会し（limit はウォッチリスト上限以上）、該当エントリを seed する。結果に現れないエントリ（同契約は古い順のため、切り捨てられるのは最近まで活発な送信元側）と照会失敗時は起動時刻を仮基準とする（活発な送信元は直後の実受信で更新されるため実害はない）
- **起動時グレース（仮値 30 分）**: seed 時点で既に閾値超過のエントリは即発火せず、**起動後グレースの間に受信がなければ発火**する——サーバ自身が閾値超の期間停止していた場合（週末メンテ等）に、健在な装置まで初回評価で一斉発火する偽陽性を防ぐ（レビュー指摘——クリス・鈴木）
- **稼働中に追加されたエントリ・一度も受信のない送信元**: **登録時点を仮の最終受信時刻とし、閾値経過で通常どおり発火する**。これは仕様である——「先回りで登録したが機器側の設定が済んでいない／経路が開通していない」を検出する数少ない機会を提供する（レビュー指摘——佐藤）
- **判定**: `ActiveNotificationMonitor` の周期評価で「単調時計での経過 > 閾値」のエントリを途絶と判定し、**新イベント ID（1000 番台・警告。採番は実装 PR——並行起案中の再採番と衝突させない）**で警告する。途絶中はエントリごとの状態フラグで再発火を抑え（状態遷移で 1 回）、受信再開でフラグを解除する（次の途絶で再発火）。**同一エントリの再発火は既存の抑制窓（仮値 15 分）でも律速する**——短い個別閾値 + 不安定な装置のフラッピングで発火が反復する事態を防ぐ（状態遷移と抑制窓の併用。1009/1010 のラッチ設計と同じ向き）。閾値の変更で途絶条件を満たさなくなったエントリはフラグを解除する
- **一斉途絶の集約**: 同一評価周期に**仮値 5 件以上**が途絶へ遷移する場合は、個別警告ではなく **1 件の集約警告**（件数 + エントリ一覧 + 「共通上流障害の疑い」の明示）にする——集約スイッチ障害等で 50 台が同時に黙ったとき、個別 50 件（メール接続時は 50 通）は診断情報としても劣化している（レビュー指摘——鈴木）
- **サーバ都合の受信断との区別**: **全受信リスナが受信不能な間（bind 失敗継続等）は途絶判定を保留し、受信経路の回復後はグレースを挟んで再開する**。途絶警告の Detail にはサーバ側受信経路の状態を併記する——真因がサーバ側なのに運用者を装置側の調査へ誘導しない（レビュー指摘——鈴木・クリス。実装粒度は委任 6）
- **復帰の扱い**: 復帰は**能動通知しない**（対応を要する事象ではない——Issue #132 前例）が、**情報レベルのイベントログ記録を 1 件残す**——途絶警告（始端）と対で「ログが欠けていた期間」の終端を証跡に残す（外部送信元のログ欠落期間は監査上の関心事であり、サーバ内部の自己回復とは監査対象性が異なる——レビュー指摘——田中）
- 通知の Detail には送信元アドレス・`Label`・最終受信時刻（壁時計表示）・経過時間・サーバ側受信経路の状態を含める（ログ本文は含めない）

### 4. UI

- **ダッシュボードの送信元別受信状況（既存 UI-4 一覧）に、ウォッチリスト登録済み送信元の途絶状態を表示する**（途絶中の強調 + 登録済みマーク）。新しい画面は作らず既存カードを拡張する
- 管理面にウォッチリストの設定画面（追加・編集・削除）を置く。**登録は既知送信元からの候補選択を主経路とする**（`QuerySourceActivityAsync` の一覧から選択——IP の転記ミスによる「形式は正しいが違うアドレス」の黙った空振りを構造的に減らす。手入力は「まだ受信のない先回り登録」用に残す。レビュー指摘——佐藤）。**候補選択時は当該送信元の最終受信時刻・直近の受信ペースを添えて表示し、閾値の明示確定を必須とする**（提案値の初期表示は可。既定閾値 24 時間の意味——「24 時間受信がないと通知」——を画面ヘルプに明記する）。閾値入力の単位表現（分直打ちにしない）は委任 5
- 手編集による一括投入（構造化配列キー + 即時反映）は大量登録の公式経路として利用者向けドキュメントに記載する（手編集は監査対象外という既存の整理——security.md §4.1——の注意書きを含む。レビュー指摘——鈴木）
- 閲覧面（UI-4 一覧）からの登録導線は、閲覧/管理の分離（登録は管理操作）を跨ぐため置かない（#286 の「管理系情報を閲覧面に露出しない」裁定と同じ向き）
- **ADR-0017 の帰趨に関わらず本 ADR は単独成立する**: 配列キーの UI 保存経路は、ADR-0017（委任 9）が accepted なら共通機構として実装し、そうでなければ**本 ADR の実装 PR で新設する**（後から 0017 が accepted になった時点で共通化）

### 5. 監査・通知チャネル

- **ウォッチリスト変更の監査記録は、ADR-0017 の採否に関わらず、追加・削除・変更されたエントリ（アドレスと Label）を Detail に含める**（本 ADR の決定として固定する。ウォッチリストは検知範囲そのものの定義であり、「管理権限を得た攻撃者が証跡遮断の前にエントリを外す」を事後に再構成できる粒度が要る——キー名だけの既存粒度では答えられない。レビュー指摘——田中。値は秘密情報ではない）
- 途絶警告は architecture.md §4.6 の能動通知としてイベントログへ書き出す。**ADR-0017（メール通知）が accepted の場合、新イベント ID をメール allowlist へ登録する**（集約警告——決定 3——も同様）。accepted でない間の到達経路はイベントログのみであり、実効性はイベントログ監視レール（転送・タスクスケジューラのイベントトリガ等）の有無に依存する——この順序依存は帰結に明記する

### 6. 反映方式

- `Notification:SourceSilence:*` は**即時反映**（ウォッチリストの参照交換。CF-4 の `ImmediateConfigurationApplier` に登録）。リスト変更時、既存エントリの追跡状態（最終受信時刻・途絶フラグ）は保持し、削除エントリの状態は破棄する

## 受け入れ基準

- ウォッチリスト登録 → 該当送信元の送信停止 → 閾値経過後にイベントログ警告（1 回のみ）→ 送信再開（復帰の情報記録）→ 再度停止 → 再度警告、の状態遷移が成立する
- 短い個別閾値での途絶・復帰の反復時、発火頻度が抑制窓で律速される（フラッピング検証）
- seed 時点で閾値超過のエントリが起動後グレース内の受信で発火しない／グレース経過まで受信がなければ発火する
- 同一周期に閾値件数以上が途絶へ遷移した場合、個別ではなく 1 件の集約警告になる
- 未登録送信元（watchlist 外）の受信がインメモリ追跡に一切影響しない（有界性のテスト）
- 上限 1000 件・手編集超過分の無効化（対象列挙警告）・不正エントリの個別無効化・既定/個別閾値の優先関係・受理範囲の単体テスト
- 受信ホットパスへの追加コストの計測は、**ウォッチリスト上限まで登録済み + 登録外送信元が混在する負荷**で行う（空リストの素通し計測では検証にならない——レビュー指摘——リサ）

## 帰結

- 良くなること: 「送信元が黙った」が受動可視化（見に行けば分かる）から能動通知（夜間でも届く）へ昇格する。ADR-0008 委任 4 の宙に浮いた委任が決着し（決定 2 の範囲で）、ADR-0016 の「無音化検出への一本化」が能動側でも完結する
- 悪くなること・受け入れるリスク:
  - **詐称による途絶隠蔽（限界の明示）**: 生存判定は「その送信元アドレスから何かが届いた」事実のみに依るため、**送信元アドレスを詐称できる攻撃者は、実機を沈黙させたまま途絶検知を空振りさせられる**（平文 UDP syslog の本質的限界。TLS 受信 opt-in でもクライアント証明書検証（相互 TLS）は意図的先送りのままであり、この限界は解けない）。本機能は**改ざん・隠蔽に耐えるセキュリティ統制ではなく運用検知**である（ADR-0004 決定 7 の限界明示の流儀。D-1 にも記載）
  - ウォッチリストは手動登録であり、**登録していない送信元の途絶は従来どおり検知されない**（受動一覧のみ。D-2）
  - **ADR-0017 非成立の間、通知の到達経路はイベントログのみ**であり、「夜間でも届く」の実効性はイベントログ監視レールの有無に依存する。それでも NMS の死活監視では代替できない（「装置は生きているが送信設定が消えた」は受信側でしか検出できない——本機能の固有価値）
  - 送信元アドレスの変更（DHCP・機器リプレース）でウォッチリストは黙って空振りする——アドレス固定（予約）の推奨と、旧アドレスの途絶警告自体が空振りの検出器を兼ねることを D-1 に含める
  - **再起動時、途絶中フラグは失われる**——既知の長期途絶エントリは起動後グレース経過で再警告される（月例再起動の運用では、退役済み装置のエントリ削除を促すリマインダとして働く。煩わしければエントリを消すのが正しい対処——D-1）
  - 流量制御でほぼ全量破棄される送信元は途絶に見え、保存に失敗し続けている送信元は健在に見える（「受信した」の定義——決定 3——の帰結。前者は流量制御の発火可視化——#288——が、後者は保存系の既存能動通知が併走して観測を担う）

## 先送りにする場合の再評価トリガ

- **配布端末の全数自動観測**（キット連携・自動登録——決定 2 でスコープ外化した残余）: ウォッチリストの手動登録・維持では賄えない規模・頻度の配布現場からの報告があった時点で、エージェントレス収集・フォワーダ戦略全体の再整理 ADR（ADR-0008 再評価トリガ）の文脈で再検討する
- **期待間隔の自動学習・全送信元監視**: 独立した利用者からの要望 3 件以上で再評価（その場合も ADR-0001 の SIEM 非目標との整合を先に確認する）
- **送信元グループ単位の閾値・通知集約の高度化**: 上限 1000 件が実運用で不足した報告、または一斉集約（仮値 5 件）の粒度が実態と合わない報告があった時点で再評価
