# ADR-0018: 送信元の途絶検知（ウォッチリスト方式。opt-in）

- 状態: proposed
- 日付: 2026-07-18
- 決定者: YANAI Taketo
- 関連: [ADR-0008](0008-forwarder-kit-generation.md) 委任事項 4 / [ADR-0016](0016-os-drop-gauge-mechanism.md)（無音化検出への一本化）/ [architecture.md](../design/architecture.md) §4.6（能動通知）/ [ui.md](../design/ui.md) §4（送信元別受信状況・UI-4）/ [database.md](../design/database.md) §1.2（`QuerySourceActivityAsync`）/ ADR-0017（メール通知——proposed・並行起案）/ Issue #275 / PR #338

## 文脈と課題

「送信元が黙った」ことは、syslog 集約サーバにとって最も気づきにくい障害である。送信側の設定消失・機器故障・経路遮断・ファイアウォール変更のいずれでも、サーバ側には**何のエラーも現れず、ただログが来なくなるだけ**——現行の観測性(カウンタ・ゲージ・能動通知)はすべて「サーバ自身の異常」を対象としており、「来るはずのものが来ていない」を検出する機構がない。

現状で存在するのは**受動的な可視化のみ**である: ダッシュボードの送信元別受信状況(UI-4。`QuerySourceActivityAsync` による最終受信時刻の古い順の全送信元一覧)は「いつも静かな装置が黙った」を上位 N 表示に埋もれさせない設計として確立済みだが、**運用者が画面を見に行かない限り機能しない**。ADR-0016 も OS レベル観測の撤回に際し「無音化検出」への一本化を打ち出したが、その無音化検出自体は受動のままである。

ADR-0008 は委任事項 4 として「転送の観測性(配布端末別の最終受信・**途絶検知**)」を別 ADR へ委任した。本 ADR はその引き取りである(2026-07-16 オーナー裁定で起案予定へ登録済み。Issue #275)。主論点は、①監視対象の指定方式 ②途絶判定(閾値・窓)③能動通知への接続 ④送信元数がスケールしたときの状態管理コスト。

### スコープ（非目標）

- **能動的な死活監視はしない**(ping・SNMP・エージェント heartbeat 等)。判定材料は「syslog が届いた」事実のみ——Yagura は syslog サーバであり、監視システム(商用 SIEM / NMS 領域。ADR-0001)にならない
- **ログ内容に基づく判定はしない**(「特定メッセージが一定時間ない」等のルール)
- **期待受信間隔の自動学習はしない**(検討した選択肢 1-(a) 参照)
- 送信側(フォワーダ)への heartbeat 送出機能の追加はしない(キット側の変更はスコープ外)

## 検討した選択肢

### 1. 監視対象の指定方式

- **(a) 全送信元の自動学習**: 送信元ごとの受信間隔を学習し、逸脱を検知する。登録の手間はないが、①DHCP アドレス変動・機器退役のたびに誤検知が発生し続ける ②「正常な間隔」の推定はログ種別・夜間バッチ・週次機器で本質的に多峰であり、誤検知を抑える閾値チューニング UI という SIEM 的装置が必要になる ③全送信元分の状態を持つため、送信元数の暴走(詐称を含む)に対して非有界——本プロジェクトが監査(#313〜#317)で繰り返し潰してきた「送信元 IP 起点の無制限辞書」の再導入になる。**却下**
- **(b) 明示的なウォッチリスト(opt-in)**: 運用者が「途絶を知りたい送信元」を明示登録する(上限つき)。登録の手間はあるが、①検知対象 = 運用者が守ると決めた送信元であり誤検知の責任構造が明確 ②状態は登録数で有界 ③「重要な装置は数えるほど」という対象現場の実態(全送信元の網羅監視が必要な規模は商用 SIEM の領分)に合う。**採用**
- **(c) 作らない(受動可視化で足りる)**: UI-4 の一覧は既にあり、Top talkers(#288)同様に「見れば分かる」。しかし「UI を開いていない夜間に気づけない」という能動通知の存在意義(architecture.md §4.6)がここにも当てはまり、ADR-0008 委任 4 は「本 ADR の採否に関わらず必要性は残る」と明記している。**却下**(受動可視化は本機能の土台として残る)

### 2. 途絶判定のデータ源

- **(a) DB への周期集計クエリ**: `QuerySourceActivityAsync` を周期実行して最終受信時刻を得る。実装は薄いが、SourceAddress 列に索引がなく(Issue #145 の整理)、窓なし集計は大規模時にフルスキャンへ劣化し得る——1 分周期の監視ループに載せる負荷として不適。**却下**(起動時の初期値取得 1 回のみに限定して使う)
- **(b) 受信経路でのインメモリ追跡(ウォッチリスト限定)**: 解析段の後段でウォッチリスト所属の送信元のみ最終受信時刻を更新する(有界辞書——登録上限で сап)。ホットパスへの追加コストは「凍結済みセットの membership 判定 + 時刻代入」のみ。**採用**

### 3. 判定主体と通知

- `ActiveNotificationMonitor` の周期評価(1 分間隔・トリガ別抑制窓)に評価メソッドを 1 つ追加する——自己検証(Issue #152)・証明書期限(1014/1017)と同じ確立パターン。新しい常駐機構は作らない

## 決定

### 1. ウォッチリスト方式の途絶検知を opt-in で追加する

- 新設キー `Notification:SourceSilence:*`(既定 = リスト空 = 機能無効)。ゼロ設定ファーストランには現れない
- **登録単位は送信元アドレス**とし、エントリは `{Address, Label(任意), ThresholdMinutes(任意)}` の構造化配列キー `Notification:SourceSilence:Watchlist` で保持する
- **登録上限は仮値 100 件**(有界化。上限到達時は保存を拒否して理由を表示)。既定閾値は `Notification:SourceSilence:DefaultThresholdMinutes`(仮値 **1440 分 = 24 時間**——誤検知しない側に倒した保守的既定。ポーリング的に静かな装置を既定でカバーし、高頻度装置はエントリ個別の `ThresholdMinutes` で短縮する)
- **不正時挙動(configuration.md §1 の 3 分類)**: エントリの `Address` 形式不正・`ThresholdMinutes` 範囲外・上限超過のリストは**当該エントリのみ無効化して警告**する(リスト全体は殺さない——1 エントリのタイポで他の監視まで止めない。機能全体のキー不正——配列として解釈不能等——は機能無効化 + 警告)

### 2. 「配布端末別」は送信元アドレス単位への読み替えとして決着する

ADR-0008 委任 4 の「配布端末別」を、**配布キットと送信元の機械的対応付けなしに**成立させる:

- キット ID の RFC 5424 SD-ELEMENT 埋め込み等による端末識別は、①転送フォーマットへの侵襲 ②Fluent Bit テンプレートの改変(検証済み版の表明——ADR-0008 委任 3——への影響)③キットを使わない送信元(機器直送)に適用不能、の 3 点で**却下**
- 代わりに、**ウォッチリストの `Label`(表示名)が「どの端末・装置か」の対応を運用者の語彙で保持する**。配布端末の管理は運用者の資産管理の領分であり、製品は送信元アドレスという観測可能な単位に徹する
- フォワーダキット生成 UI からの「この端末をウォッチリストへ登録」導線は実装 PR で軽く接続できる場合のみ(委任 4)

### 3. 判定と状態管理

- **受信経路での追跡**: 解析段の後段で、ウォッチリスト所属送信元の最終受信時刻をインメモリ更新する(選択肢 2-(b)。凍結済みルックアップ構造 + 上限 100 の有界辞書。IPv4-mapped IPv6 の正規化は流量制御・Top talkers と同じ既存規約に従う)
- **起動時の初期値**: `QuerySourceActivityAsync` を 1 回だけ照会してウォッチリスト該当分を seed する(再起動のたびに「起動からの経過」で計り直して誤検知・検知漏れを作らない)。照会失敗時は起動時刻を仮の基準とし、その旨をログに残す
- **判定**: `ActiveNotificationMonitor` の周期評価で「現在時刻 − 最終受信時刻 > 閾値」のエントリを途絶と判定し、**新イベント ID(1000 番台・警告。採番は実装 PR——並行起案中の再採番と衝突させない)**で 1 回警告する。途絶中はエントリごとの状態フラグで再発火を抑え(抑制窓の反復ではなく状態遷移で 1 回)、**受信が再開したらフラグを解除する**(次の途絶で再度 1 回発火)
- **復帰はイベントログへ通知しない**(退避復帰を通知しない前例——architecture.md §4.6 の Issue #132 判断——と同じ整理: 自己回復は対応を要しない。復帰の確認は UI の無音化一覧・状態表示が担う)
- 通知の Detail には送信元アドレス・`Label`・最終受信時刻・経過時間を含める(ログ本文は含めない)

### 4. UI

- **ダッシュボードの送信元別受信状況(既存 UI-4 一覧)に、ウォッチリスト登録済み送信元の途絶状態を表示する**(途絶中の強調 + 登録済みマーク)。新しい画面は作らず既存カードを拡張する
- 管理面にウォッチリストの設定画面(追加・編集・削除。配列キーの UI 保存経路は ADR-0017 委任 9 と同じ新機構に乗る——両 ADR が accepted の場合は実装を共通化する)
- 閲覧面(UI-4 一覧)からの「この送信元をウォッチリストへ登録」導線は、閲覧/管理の分離(登録は管理操作)を跨ぐため、**閲覧面には出さず管理面の設定画面で完結させる**(#286 の「管理系情報を閲覧面に露出しない」裁定と同じ向き)

### 5. 監査・通知チャネル

- ウォッチリストの変更は設定変更として既存監査に乗る(ウィザード外の管理画面保存。ADR-0017 決定 4 と同型の「変更キーと新値」記録——ただし ADR-0017 が accepted でない場合は 2016/2001 の既存粒度に従う)
- 途絶警告は architecture.md §4.6 の能動通知としてイベントログへ書き出す。**ADR-0017(メール通知)が accepted の場合、新イベント ID をメール allowlist へ登録する**(途絶検知はまさに「夜間に人が届く経路で気づきたい」事象の典型)

### 6. 反映方式

- `Notification:SourceSilence:*` は**即時反映**(ウォッチリストの参照交換。CF-4 の `ImmediateConfigurationApplier` に登録)。リスト変更時、既存エントリの追跡状態は保持し、削除エントリの状態は破棄する

## 受け入れ基準

- ウォッチリスト登録 → 該当送信元の送信停止 → 閾値経過後にイベントログ警告(1 回のみ)→ 送信再開 → 再度停止 → 再度警告、の状態遷移が成立する
- 未登録送信元(watchlist 外)の受信がインメモリ追跡に一切影響しない(有界性のテスト)
- 上限 100 件・不正エントリの個別無効化・既定/個別閾値の優先関係の単体テスト
- 受信ホットパスへの追加コストが計測可能な劣化を生まない(既存ベンチへの影響確認)

## 帰結

- 良くなること: 「送信元が黙った」が受動可視化(見に行けば分かる)から能動通知(夜間でも届く)へ昇格する。ADR-0008 委任 4 の宙に浮いた委任が決着し、ADR-0016 の「無音化検出への一本化」が能動側でも完結する
- 悪くなること・受け入れるリスク:
  - ウォッチリストは手動登録であり、**登録していない送信元の途絶は従来どおり検知されない**(受動一覧のみ)。「全部を自動で見張ってほしい」要望には応えない(スコープ非目標・再評価トリガ参照)
  - 既定閾値 24 時間は保守的であり、高頻度装置の途絶検知には個別閾値の設定が要る——既定のまま「1 時間で気づけると思っていた」の期待ズレは利用者向けドキュメントで防ぐ(申し送り D-1)
  - 送信元アドレスの変更(DHCP・機器リプレース)でウォッチリストは黙って空振りする——アドレス固定(予約)の推奨と、途絶警告自体が空振りの検出器を兼ねる(旧アドレスが途絶として発火する)ことを D-1 に含める
  - インメモリ状態は再起動で失われるが、起動時 seed(決定 3)により実用上の連続性は保たれる。DB に届いていない受信(スプール滞留中)は最終受信時刻に反映されない短い窓がある——seed の一度きりの近似として受容する

## 先送りにする場合の再評価トリガ

- **期待間隔の自動学習・全送信元監視**: 独立した利用者からの要望 3 件以上で再評価(その場合も ADR-0001 の SIEM 非目標との整合を先に確認する)
- **送信元グループ単位の閾値・通知集約**: ウォッチリスト上限(100 件)が実運用で不足した報告があった時点で、上限引き上げと合わせて再評価
- **キット ID による端末識別(決定 2 で却下)**: エージェントレス収集・フォワーダ戦略全体を再整理する ADR(ADR-0008 再評価トリガ)を起案するとき、その文脈で再検討する
