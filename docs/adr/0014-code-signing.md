# ADR-0014: リリース成果物のコード署名 — 証明書の調達方式の選択

- 日付: 2026-07-14
- 決定者: YANAI Taketo
- 関連: [Issue #264](https://github.com/Yanai-Taketo/Yagura/issues/264)（本 ADR がこの Issue の「証明書の調達方針を決める」「ADR を起案するか判断する」に応える）、[ADR-0006](0006-v1-release-criteria.md)（基準 5「リリースの完全性」の実現手段を確定する）、[ADR-0009](0009-architecture-support.md)（委任事項 8「コード署名方針」を閉じる）、[ADR-0005](0005-oss-packaging.md)（入口文書の体裁）、[ADR-0004](0004-security-model.md)（セキュリティモデル）、[SECURITY.md](../../SECURITY.md)、[conventions.md](../development/conventions.md)（リリース）

## 文脈と課題

### 現在地

Yagura のリリース資産（`Yagura-<版>-x64.msi` / `Yagura-<版>-arm64.msi`）は **Authenticode 署名されていない**。利用者は MSI を実行すると SmartScreen の「Windows によって PC が保護されました」警告に遭い、「発行元: 不明」と表示される。回避には「詳細情報 → 実行」の 2 クリックが要る。

改ざん検知の手段としては `.sha256` チェックサムを添付しているが、これは**配布経路が信頼できることを前提にした完全性チェック**であって、真正性（誰が作ったか）を証明しない。チェックサムと成果物が同じ GitHub Release に並んでいる以上、Release を掌握した攻撃者は両方を差し替えられる。

### 決定を要求している既存の合意

コード署名は新規の思いつきではなく、すでに 2 つの ADR が宿題として登録している。

- **ADR-0006 基準 5「リリースの完全性 — コード署名を含む」**: v1.0 公開の要件として「リリース資産（インストーラ）に **Authenticode コード署名（タイムスタンプ付き）**が付与されている」ことを要求し、証拠として「リリースノートから署名の検証手順（signtool verify 等）と `.sha256` が辿れること」を求めている。同 ADR は「証明書の入手・管理はリリース準備の課題として別途扱う」と明記して手段の決定を先送りした
- **ADR-0009 委任事項 8**: 「リリース成果物のコード署名（Authenticode）方針 / 別 ADR または ADR-0005 の amendment。署名の要否・証明書の入手と鍵管理・リリースワークフローへの組み込み。アーキ数に依存しない独立論点だが、導入時は全アーキ成果物への適用を設計に含める」

**本 ADR はこの 2 つを閉じる。** ADR-0006 は「署名する」という要否をすでに決めているため、本 ADR が決めるのは**手段**（どの証明書を、どう入手し、どう鍵を守り、どうワークフローへ組み込むか）である。

### 課題の核心 — OSS 個人プロジェクトは証明書を買えない

コード署名証明書は、**法的主体（legal entity）に対してのみ発行される**。OSS プロジェクトそれ自体は法人格を持たないため、CA から見た受取人になれない。さらに 2 つの制約が個人開発者を締め出している。

1. **秘密鍵のハードウェア保管が必須**: CA/Browser Forum の Code Signing Baseline Requirements により、**2023-06-01 以降、EV・非 EV を問わずコード署名証明書の鍵ペアは FIPS 140-2 Level 2 または Common Criteria EAL 4+ 以上のハードウェア暗号モジュール（HSM／ハードウェアトークン）で生成・保管することが必須**となった（[CA/Browser Forum Code Signing Requirements](https://cabforum.org/working-groups/code-signing/requirements/)、確認日 2026-07-14）。ソフトウェア鍵での運用は不可能になり、個人が負担する初期費用・運用手間が上がった
2. **CI との相性の悪さ**: ハードウェアトークンは物理的に 1 か所にしか挿さらない。GitHub Actions のようなクラウド CI から署名するには、クラウド HSM か署名サービスを別途調達する必要がある。「手元のマシンで手動署名してアップロード」はビルドの再現性・来歴（provenance）を壊し、ADR-0006 が求める「検証可能な完全性」の趣旨に反する

つまり Yagura が署名を実現するには「**個人が法人格なしに、公的に信頼される証明書へアクセスでき、かつ CI から自動で署名できる**」経路が要る。この 3 条件を同時に満たす選択肢は多くない。

### この決定を Yagura 単独で閉じられない事情 — もう 1 つのプロジェクトの存在

オーナー（YANAI Taketo）は Yagura のほかに **[Open DUMP Viewer](https://github.com/Open-DUMP-Viewer/Open-DUMP-Viewer)**（以下 ODV）を公開している。ODV の性質は本 ADR の選択肢を強く制約するため、事実として記録する（確認日 2026-07-14）。

- ODV は **OSS ではない**。`LICENSE` は "Open DUMP Viewer for Oracle database Proprietary License" と題され、`All rights reserved.` で始まり、本文に「The source code is made available for reference purposes only ("Source-Available") and is **NOT open-source software**」と明記されている（ソース公開の理由は透明性・監査可能性と知識継承であると同ファイルが述べる）
- ODV は `.msixbundle` と Inno Setup 製の `.exe` インストーラを配布している。**MSIX は署名されていなければインストールできない形式**であり、ODV は遅かれ早かれ証明書を必要とする

ここから 2 つの帰結が出る。

1. **無償の OSS 向け証明書は、原理的に ODV を署名できない**。SignPath Foundation は「OSI 承認ライセンス・プロプライエタリ要素なし」を適格要件とし、Certum Open Source は OSS 専用で商用・非 OSS への使用を失効事由とする。ODV はどちらの要件も満たさない
2. **したがって「1 枚の証明書で自分の全プロジェクトを署名する」構成を取れるのは、通常の有料証明書だけ**である。そして**その有料証明書を ODV のために購入するなら、同じ 1 枚で Yagura も署名でき、Yagura 側の追加費用はゼロになる**

この事実により、本 ADR の選択は「無償 vs 有料」という単純な費用比較ではなくなる。**ODV に有料証明書を買うか否かによって、Yagura にとっての SignPath の価値が変わる**——買うなら、SignPath を選ぶ理由から「費用」が消え、「秘密鍵を CI に置かないこと・origin verification・実名を出さないこと」だけが残る。買わないなら、Yagura のためだけに年 3〜5 万円を負担するかどうかの問題になる。

**この相互依存を ADR の中で正面から扱う。** Yagura の ADR が ODV の意思決定を縛ることはできないが、選択肢の評価に ODV の存在を織り込まないのは事実の隠蔽にあたる。

### (A) SignPath Foundation の無償 OSS 証明書 + SignPath.io の署名基盤 【推奨案】

[SignPath Foundation](https://signpath.org/) は OSS プロジェクトに無償のコード署名を提供する非営利団体。証明書は **SignPath Foundation 名義で発行され、それを OSS プロジェクトに使わせる**（[terms](https://signpath.org/terms.html)、確認日 2026-07-14）。プロジェクトが法人格を持つ必要がない。署名は SignPath.io のクラウド HSM 上で行われ、GitHub Actions から `signpath/github-action-submit-signing-request` で要求する。

- 利点:
  - **費用ゼロ**。個人 OSS の継続性（ADR-0001 の運営原則）と両立する
  - **鍵を一切持たない**。秘密鍵は SignPath.io の HSM にあり、Yagura のリポジトリにも CI にも鍵が存在しない。漏洩させる鍵がないことが最大の防御になる（ADR-0004 の最小権限の思想と同型）
  - **来歴の技術的強制**: SignPath は署名要求のたびに origin verification を行い、「その成果物が、指定されたリポジトリのソースから、GitHub ホストランナー上の実際のワークフロー実行で生成されたこと」を検証する。この検証はビルドスクリプトの自己申告ではなく **GitHub 側が提供するメタデータ**に基づく。署名は「Yagura が作った」だけでなく「**このコミットから CI が自動生成した**」ことの証明になり、ADR-0006 が求める完全性の水準を、単なる Authenticode 署名より高い位置で満たす
  - リリースごとに**人間の手動承認**を要求する（terms「Every release needs manual approval for signing」）。CI が単独でリリース資産を署名・公開できない構造は、リポジトリ侵害時の被害を限定する
- 欠点・リスク:
  - **審査に通る保証がない**（後述「受け入れるリスク」）。terms は実行可能プログラムについて「a certain verifiable reputation」を要求すると明記する
  - **第三者への依存**が増える。SignPath Foundation がサブスクリプションを停止・証明書を失効させれば、以後のリリースは署名できなくなる（terms は「pause or terminate the subscription without prior notice」「revoke the certificate effective immediately or retroactively」の権利を留保している）
  - **発行元表示は "SignPath Foundation"** になる。Yagura の名前は証明書の Subject に出ない。利用者から見た発行元は Yagura ではなく SignPath Foundation である
  - terms が課す運用制約（コード署名ポリシーの掲示、全メンバーの MFA、メタデータの統一、役割分担の明文化）を恒久的に守る義務を負う
  - **Yagura しか署名できない。** 証明書は SignPath Foundation 名義でプロジェクト単位に貸与される（terms「Sign your own projects only」）。ODV は OSS ではないため対象外であり、**オーナーが ODV の署名手段を別途調達するなら、その手段は Yagura にも使える**——このとき (A) の最大の利点である「無償」は、Yagura にとっての**限界費用ゼロという相対優位を失う**（前掲「もう 1 つのプロジェクトの存在」）

### (B) Azure Artifact Signing（旧 Trusted Signing）【却下】

Microsoft のマネージド署名サービス。月額約 $10 と安価で、鍵は Azure 側の HSM に置かれ、CI 連携も公式アクションがある。個人開発者（self-employed）にも 2026 年時点で開放されており、旧来の「3 年の事業履歴」要件は個人には課されない。

- **却下理由（実体検証済み）**: **提供地域が限定されており、日本の個人開発者は対象外**。Microsoft Learn の [Quickstart: Set up Artifact Signing](https://learn.microsoft.com/en-us/azure/artifact-signing/quickstart) は次のように明記する（確認日 2026-07-14、原文）:

  > For Public Trust certificates, Artifact Signing is currently available to organizations in the USA, Canada, the European Union, and the United Kingdom, as well as individual developers in the USA and Canada.

  Yagura のオーナーは日本の個人であり、組織（米国・カナダ・EU・英国）にも個人開発者（米国・カナダ）にも該当しない。技術的な優劣ではなく**資格要件で門前払いになる**ため、現時点では選択肢にならない。

  なお Azure のリージョンとしては Japan East が Artifact Signing をサポートしており（同ページのリージョン表）、リージョンの話と**証明書発行の資格地域の話は別**である点に注意する。この 2 つを混同すると「日本でも使える」と誤読しかねない。
- **再評価トリガ**: 提供地域に日本が追加された場合（後述）

### (C) 有料の商用証明書を個人名義で購入する 【対抗案】

「1 枚の証明書で、オーナーが持つ全プロジェクト（Yagura・ODV・将来のもの）を、実名で、審査なしに署名する」構成。個人名義で発行され、鍵はクラウド HSM に置き、GitHub Actions から署名する。

**共通の利点**:

- **ODV を含む全プロジェクトを 1 枚でカバーできる**。証明書は主体（個人）に紐づき、プロジェクト識別子を含まない。DigiCert は FAQ で「署名できるアプリ数に上限なし」、GlobalSign は「署名対象アプリケーション数: 無制限」と明記する。**ODV のために証明書を買うなら、Yagura の署名は限界費用ゼロで手に入る**
- **審査に落ちるリスクがない**。本人確認（政府発行 ID 等）を通れば発行される。評判要件は存在しない。**Yagura が star 0 の新しいプロジェクトであることが一切不利にならない**
- **発行元が実名になる**。「誰が作ったか」の答えが直接的になる（プライバシーとのトレードオフは後述）
- **リリースごとの手動承認が強制されない**。緊急のセキュリティ修正を承認待ちなしに出せる
- 第三者の審査・運営方針に継続的に依存しない（CA の失効権限は残るが、terms 違反による裁量的なサブスクリプション停止という経路はない）

**共通の欠点・リスク**:

- **継続費用**（年 3〜5 万円規模）。個人が無報酬で運営する OSS に、リリースを続ける限り毎年かかるコストを載せる
- **実名・所在地が証明書に載り、署名済みバイナリを通じて恒久的に公開される**。個人 OSS 開発者にとって無視できない帰結
- **署名の資格情報を GitHub Secrets に置くことになる**（eSigner の認証情報 + TOTP シークレット、または GCP のサービスアカウント鍵）。**これが本 ADR で最も重い論点である**——リポジトリまたは CI を掌握した攻撃者は、**任意のバイナリにオーナーの実名で署名できる**。鍵そのものは HSM から出ないが、「署名させる権限」は CI にある。(A) の origin verification（署名対象が指定リポジトリのコミットから CI で生成されたことを SignPath 側が検証する）と手動承認は、まさにこの経路を塞ぐための仕組みであり、(C) にはそれがない。**単一メンテナが実名で署名する構成において、この差は金額では測れない**
- 署名回数の従量上限がある方式がある（eSigner Tier1 は月 20〜240 署名、DigiCert KeyLocker は年 1,000 署名）。Yagura のリリース頻度では問題にならないが、CI で試験署名を回す設計にすると効いてくる

#### (C1) SSL.com IV Code Signing + eSigner 【対抗案の本命】

- 証明書 **$129/年**（5 年契約で $96.75/年）+ **eSigner 別課金 Tier1 = 年 $180**（月 20〜240 署名、超過 $1/回）。実質 **年 $309 前後**。新規証明書は最初の 30 日 eSigner 無料
- **IV（Individual Validation）は「事業体不要・個人名を検証」する商品**であり、独立開発者・OSS メンテナ向けと公式に位置づけられている。地域制限の記載はない（日本可の明示も確認できていない）
- **公式 GitHub Action がある**（`SSLcom/esigner-codesign`。最新 v1.3.2 = 2025-12-22）。Ubuntu ランナーで動作する。CSC 準拠の REST API もある
- 本人確認: 政府発行 ID の表裏 + ID を顔の横に持った写真。標準 3〜5 営業日
- **注意すべき材料**: eSigner の課金運用について第三者から強い批判が出ている（MetaBrainz の公式ブログ記事等）。真偽は検証していないが、契約前に課金条件を精読すべき

#### (C2) Sectigo（リセラー）+ Google Cloud KMS + jsign 【対抗案の代替】

- 証明書 **約 $220/年**（リセラー実売。Sectigo 公式は「5 年契約で $536.25/年から」としか出しておらず、一次価格が公開されていない）+ Google Cloud KMS の HSM 鍵の月額（実践記事では $2.50/月。**正確な料金表は確認できず**）
- **Sectigo 公式 KB が個人発行を明記**する: "OV Code Signing certificates may be issued to: **Individuals**, Private organizations, …"
- **Google Cloud HSM の attestation を Sectigo が受理する**（公式 KB に手順あり）。**Azure Key Vault は鍵 attestation に対応しないため Sectigo では使えない**
- 公式 GitHub Action はなく、`jsign` で自作する。**日本の開発者による GitHub Actions 実装事例がある**（ただし当該事例の証明書は法人審査を経ており、**個人審査 + GCP KMS の組み合わせの実績は確認できなかった**）
- 署名回数の従量課金が事実上ゼロで、長期的には最安クラス。引き換えに構築・保守を自分で負う

#### (C3) Certum Open Source Code Signing 【却下 — CI から署名できない】

- **€49〜69/年**と最も安く、**個人専用・OSS 専用**で、Subject は `Open Source Developer, <実名>`。一見この用途に最も適した商品に見える
- **却下理由**: **GitHub ホストランナーから署名できない**。クラウド版（SimplySign）は SimplySign Desktop という GUI アプリが仮想カードリーダーをエミュレートし、モバイルアプリの OTP で対話ログインする方式で、セッションは 2 時間で切れる。**公式のクラウド署名 REST API が存在しない**（確認日 2026-07-14）。有志の回避策（TOTP シークレット抽出 + SendKeys、Xvnc + p11-kit コンテナ等）は存在するが、いずれもデスクトップ環境を要求し GitHub ホストランナーでは動かない
- GUI 付きの self-hosted ランナーを常時運用するなら選択肢になるが、**それは「ビルドを GitHub ホストランナーで行う」という現行の前提（および SignPath の origin verification が要求する条件）を捨てることを意味し、供給網の保証を弱める方向の変更**である。安さのために保証を下げる取引は本 ADR では選ばない
- **ODV も署名できない**（OSS 専用。商用配布ソフトへの使用は公式に失効事由）

#### (C4) DigiCert + KeyLocker 【却下 — 個人発行の可否が確認できない】

- CI 連携は最も洗練されている（公式アクション `digicert/code-signing-software-trust-action`）が、**個人（自然人）への発行可否が公式情報で確認できなかった**。公式 FAQ は "a Code Signing certificate is tied to your Organization Name only" と組織を前提に書かれている。価格も公開されていない（リセラー掲載で OV $400〜）
- KeyLocker は **年 1,000 署名の従量上限**つき
- 個人で買える確証が取れない以上、対抗案の候補から外す。**(C1)/(C2) が両方とも駄目だった場合に、可否を直接問い合わせる先として残す**

#### (C5) GlobalSign / GMO グローバルサイン 【却下 — 法人のみ】

- 日本語・円建て・国内サポートという利点があり、日本在住者にとって最も摩擦が少なそうに見えるが、**個人は申請できない**。GlobalSign 公式ショップは "Code Signing Certificates can be issued in the name of **a legally registered organization only**" と明記し、GMO グローバルサイン日本公式も「発行対象: 法人」と明記する（¥60,000/年 税抜、EV ¥78,000/年）
- **法人化すれば選択肢に入る**が、コード署名のために法人を作るのは本末転倒であり、本 ADR の範囲外

### (D) Sigstore / cosign による署名【却下 — 問題を解決しない】

キーレス署名。OIDC で身元を証明し、証明書は短命。透明性ログ（Rekor）に記録される。ADR-0006 の 82 行が「無償・低コストの代替」の例として名指ししている。

- **却下理由（実体検証済み）**: **Windows の Authenticode 検証を満たさない**。Authenticode は Microsoft Trusted Root Program に参加する CA が発行した証明書チェーンを要求するが、Sigstore の CA（Fulcio）はこのプログラムに参加していない。したがって Sigstore のみで署名した Windows バイナリは、依然として SmartScreen で「発行元不明」と表示される（確認日 2026-07-14）。

  Sigstore は「配布物の来歴を検証可能にする」目的には有効だが、**ADR-0006 基準 5 が解決しようとしている問題（SmartScreen 警告への対処）そのものを解決しない**。ADR-0006 が代替候補として挙げた項目だが、実体を確認した結果、代替になっていないことがここで判明した。これは ADR-0006 の歯止め条件①「代替の検討記録を残すこと」に対する記録である
- **併用の可能性は残す**: GitHub Actions の artifact attestation（SLSA provenance）との併用は Authenticode 署名と排他ではない。ただし本 ADR のスコープ外とし、委任事項として登録する

### (E) 署名しない（現状維持）【却下】

- **却下理由**: ADR-0006 基準 5 が v1.0 の公開要件として署名を明示的に要求している。署名しないことを選ぶには、ADR-0006 の supersession が必要であり、その場合は同 ADR の歯止め条件（82 行）により「**v1.0 の延期を対等な選択肢として併記・比較する**」ことと「README・リリースノートへの署名なしの明示」が義務づけられる。無償の実現手段（(A)）が存在し、まだ試してもいない段階でこの道を選ぶ理由はない

## 決定

### 決定 1: 証明書の調達方式 —— **両論併記。ペルソナレビューを経てオーナーが裁定する**

**この決定だけは起案側で閉じない。** 決め手が技術的優劣ではなく、**費用・プライバシー・供給網の安全性・別プロジェクト（ODV）との関係**という、オーナー個人の事情と価値判断に属する軸で決まるためである。起案側が一方に倒して「決定」として提示するのは、判断の実質をオーナーから奪うことになる。

#### 推奨案: (A) SignPath Foundation

起案側の推奨は (A) である。根拠:

1. **秘密鍵も「署名させる権限」も Yagura 側に存在しない**。(C) は鍵こそ HSM から出ないが、**署名を発火させる資格情報は GitHub Secrets に載る**。リポジトリ・CI の侵害が「オーナーの実名で任意のバイナリに署名できる状態」に直結する。単一メンテナ・実名という条件下で、この差は年 3〜5 万円という金額と釣り合わない
2. **origin verification が「この成果物は、このリポジトリのこのコミットから、GitHub の CI が生成した」ことを技術的に強制する**。署名が「誰が」だけでなく「何から」を証明する。ADR-0006 が「リリースの完全性」と呼んだものの本質に、単なる Authenticode 署名より深く応える
3. **リリースごとの人間の承認が構造として強制される**。不可逆な公開行為に人間の確認が挟まる
4. **実名・所在地を公開しなくて済む**
5. 費用ゼロ（ただし ODV に有料証明書を買う場合、この優位は相対的に消える。後述）

#### 対抗案: (C1) SSL.com IV Code Signing + eSigner（または (C2) Sectigo + GCP KMS）

対抗案の根拠は弱くない。むしろ**オーナーの実情によっては、こちらが正解である**:

1. **ODV に証明書が要るなら、Yagura の署名は限界費用ゼロで手に入る**。ODV は MSIX を配布しており、MSIX は署名なしにインストールできない。ODV がいずれ有料証明書を買うのなら、Yagura のために SignPath の審査・運用制約（ポリシー掲示・メタデータ強制・手動承認・terms 追従）を別途背負う理由は薄くなる
2. **審査に落ちるリスクがない**。(A) は star 0・作成 11 日の Yagura が「verifiable reputation」要件を通る保証がなく、**却下されれば v1.0 が ADR-0006 基準 5 を満たせず延期になる**（決定 8-1）。(C) にこのリスクは存在しない
3. **緊急セキュリティリリースが承認待ちで律速されない**
4. **発行元が実名になる**ことは、プライバシーの代償であると同時に、「誰が作ったか」を利用者に直接示すという**利点**でもある

#### 裁定に必要な情報（オーナーへの問い）

1. **ODV に有料証明書を買う予定があるか。** 買うなら (C) の限界費用はゼロであり、推奨案の根拠 5 は消え、根拠 1〜4 だけで (A) を選ぶかどうかの判断になる
2. **実名・所在地を証明書に載せることを受け入れられるか。** これは取り消せない（署名済みバイナリは回収できない）
3. **「CI を掌握されたら実名で任意署名される」というリスクを、どう評価するか。** これが (A) と (C) の最も本質的な差である

#### ペルソナレビューへの明示的な付託

- **田中（セキュリティ・監査）**: (C) が GitHub Secrets に署名資格情報を置くことの供給網リスクと、(A) の origin verification + 手動承認が提供する保証を比較評価してほしい。年 3〜5 万円で買えるものと引き換えに失うものは何か
- **クリス（批判的アーキテクト）**: 起案側の推奨 (A) は「無償だから」という慣性で選ばれていないか。ODV の存在を踏まえてなお (A) が正しいか。逆に (C) の「審査リスクなし・全プロジェクト対応」という単純さを過小評価していないか
- **佐藤（ひとり情シス）**: 発行元が「SignPath Foundation」と出るのと「YANAI Taketo」と出るのとで、社内での説明のしやすさは変わるか
- **鈴木（NW 運用者）**: 緊急セキュリティリリースが承認 1 人に律速されることを許容できるか
- **リサ（外部コントリビュータ）**: どちらの方式が、外部貢献者にとって参入障壁が低いか

#### この決定に依存する範囲・しない範囲

- **依存しない（どちらを選んでも同じ）**: 決定 2（署名対象の範囲）、決定 3（ワークフローの構造）、決定 4（全アーキ適用）、決定 7（メタデータの一元化）。**これらは先に確定でき、実装に着手できる**
- **依存する**: 決定 5（コード署名ポリシーの掲示内容——(A) は terms が文言を指定する。(C) なら任意）、決定 6（役割の明文化——(A) の terms 要件。(C) でも良い実践ではある）、決定 8（退避経路）
- **(C) を選んだ場合の追加の決定**: 署名資格情報を GitHub Secrets に置くことのリスク低減策（Environments による承認ゲート、tag 限定の権限、署名ジョブの分離）を設計に含めること。**(A) が構造的に提供する保証を、(C) では自力で作らねばならない**という事実を設計要件として引き継ぐ

### 決定 2: 署名対象は「自前バイナリのみ」とし、上流バイナリは署名しない

SignPath Foundation の terms「Sign your own binaries only」に従い、署名対象を次のとおり限定する。

| 対象 | 署名 | 根拠 |
|---|---|---|
| `Yagura-<版>-<arch>.msi`（インストーラ本体） | **する** | Yagura のビルドスクリプトが生成する自前成果物。利用者が実行する入口であり、SmartScreen が評価する対象 |
| `Yagura.Host.exe`・`Yagura.*.dll`（Yagura が書いたアセンブリ） | **する** | 自前ソースからのビルド成果物。MSI 内部の deep signing |
| .NET ランタイム一式（self-contained publish に同梱される `coreclr.dll` 等） | **しない** | Microsoft が既に署名している上流バイナリ。terms は自プロジェクトのソースから作っていないものへの署名を禁じている |
| MudBlazor 等の第三者アセンブリ（NOTICE に掲載） | **しない** | 同上（上流 OSS の成果物）。terms は「未署名の上流バイナリを署名済みパッケージに同梱すること」を明示的に許可している |

**「MSI は署名されているが、中の第三者 DLL の一部は未署名」という状態を正とする。** これは terms が想定・許容する構成であり、隠すべき欠陥ではない。

### 決定 3: リリースワークフローは「未署名で機能を検証し、署名済みで署名を検証し、署名済みのみを公開する」

`release.yml` を次の構造へ改める。

```
build (x64 / arm64)
  → 未署名 MSI を GitHub artifact としてアップロード
  → [既存] Full E2E スモーク（x64 / ARM64 実機）を未署名 MSI に対して実施
  → SignPath へ署名要求（submit-signing-request）
  → 【人間による手動承認】
  → 署名済み MSI を取得
  → 署名検証ゲート（Authenticode 検証 + 署名済み MSI のインストール〜起動スモーク）
  → SHA256 を【署名済み MSI に対して】算出
  → GitHub Release を公開（署名済み MSI + その .sha256 のみ）
```

判断の内訳:

- **機能 E2E は未署名 MSI に対して行う**。署名は手動承認を挟むため、E2E の前段に置くと「承認待ちで CI が止まったまま、機能の破損がまだ判明していない」状態が生まれる。壊れた MSI に人間が承認印を押す事故を避けるため、**機能検証を承認より前に置く**
- **ただし「テストしたものと出荷するものが違う」問題を放置しない**。署名後に**署名検証 + インストール〜起動のスモーク**を必ず通す（署名は PE / MSI の構造を変更するため、「署名したら壊れた」は理論上ありうる）。フル E2E の再実行はしない——署名が機能を壊す経路は限られており、コストに見合わないと判断する。この判断が誤りであった場合（署名済み MSI 固有の不具合が実際に出た場合）は、フル E2E の再実行へ格上げする
- **SHA256 は必ず署名済み MSI に対して算出する**。署名はファイルを変更するため、未署名 MSI のチェックサムを公開すると利用者の検証が必ず失敗する。現行 `release.yml` はビルドジョブ内でチェックサムを算出しており、**この移設は必須の変更**である
- **未署名 MSI を GitHub Release へ公開しない**。署名要求が失敗・却下された場合はリリースを中断する（署名済みと未署名が混在する Release を作らない）

### 決定 4: 署名は全アーキ（x64・ARM64）の成果物に等しく適用する

ADR-0009 委任事項 8 の「導入時は全アーキ成果物への適用を設計に含める」に従う。ARM64 の検証水準が「試験的」であること（ADR-0009 決定 2）と、**署名の有無は別の軸**である。試験的だから署名しない、という扱いはしない——署名は「誰が作ったか」の証明であって「どれだけテストしたか」の証明ではなく、両者を混同すると利用者に誤ったメッセージを送る。

### 決定 5: コード署名ポリシーを公開し、入口文書に検証手順を書く

terms「Specify a code signing policy」は掲示内容を具体的に指定している。次を新設・更新する。

- **`docs/code-signing-policy.md`（新設）**: 見出しに「コード署名ポリシー」を用いる。必須の記載事項は terms が定めるとおり:
  - 定型文言「Free code signing provided by SignPath.io, certificate by SignPath Foundation」（terms が原文での掲示を指定しているため、**この 1 文は英語のまま掲載する**。日本語の説明文を併記する）
  - チームの役割（Authors / Reviewers / Approvers）とそのメンバー
  - プライバシーポリシー（後述）
- **README.md**: 「コード署名ポリシー」への導線をインストール節に置く（terms は「home page and download/release pages」での掲示を求める）
- **SECURITY.md**: 署名の検証手順（`signtool verify` / `Get-AuthenticodeSignature`）を追加する。ADR-0006 基準 5 の「証拠」要件が求める内容である
- **リリースノート**: 署名済みである旨と検証手順への導線を定型文に追加する（現行の「アーキテクチャ対応」表と同じ枠組みで運用する）

**プライバシーポリシーの記載**: Yagura はテレメトリを持たず、利用者が設定した宛先以外へ情報を送信しない（SECURITY.md の既定構成の姿勢と一致する）。terms が用意する定型文「This program will not transfer any information to other networked systems unless specifically requested by the user or the person installing or operating it」を採用する。ただし**逆引き（PTR）の DNS 問い合わせ（ADR-0007）とフォワーダ配布キットの取得は、利用者が明示的に有効化・実行する操作である**旨を日本語で補足し、定型文の「unless specifically requested by the user」の内実を利用者に対して具体的に示す。

### 決定 6: チームの役割を「単一メンテナ + 外部貢献は必ずレビュー」として明文化する

terms「Assign code signing roles」への回答。Yagura は現在オーナー 1 名の体制であり、これを偽らずそのまま宣言する。

| 役割 | 現在の担当 | 意味 |
|---|---|---|
| Authors | YANAI Taketo（リポジトリオーナー） | 追加レビューなしにソースを変更できる者 |
| Reviewers | YANAI Taketo | 外部からの PR は必ずレビューを経てマージされる（`main` は PR 必須 + 承認 1 件必須のブランチ保護下にある） |
| Approvers | YANAI Taketo | 署名要求を承認する者 |

- **単一メンテナであることの帰結を隠さない**: 役割分離が形式的にしか成立していない（同一人物が全役割を兼ねる）。これは体制の実態であり、コントリビュータが増えた時点で役割を分ける。ポリシーページにこの事実をそのまま書く
- **全メンバーの MFA は terms の必須要件**。オーナーの GitHub・SignPath アカウントで MFA を有効化していることを申請前に確認する。将来コミット権を持つメンバーを迎える場合、MFA の有効化を権限付与の条件とする

### 決定 7: 製品メタデータを一元化し、CI で一致を強制する

terms「SignPath configuration requirements」は、署名対象バイナリに**製品名・製品バージョンのメタデータが設定され、file metadata restrictions で強制されている**ことを要求する。

現状、`Directory.Build.props` にも各 `.csproj` にも `Product` / `Company` / `Version` / `Copyright` が**一切設定されていない**。結果として生成されるアセンブリは SDK 既定値（`AssemblyCompany` = アセンブリ名、`AssemblyFileVersion` = `1.0.0.0`）を持ち、バージョンの正は `installer/Package.wxs` の `Version` **のみ**である。この状態では SignPath のメタデータ制約を満たせない。

- `Directory.Build.props` に製品メタデータを追加する（`Product` = `Yagura`、`Company` = `Yagura Project`（MSI の `Manufacturer` と一致させる）、`Copyright`（NOTICE と一致させる）、`Version` 系）
- **バージョンの単一の正（single source of truth）を決め、`Package.wxs` の `Version` とアセンブリのバージョンが食い違ったら CI で失敗させる**。現行 `release.yml` は「git タグ vs `Package.wxs`」の一致検証を既に持っている（74-99 行）。この検証を「タグ vs `Package.wxs` vs アセンブリ」の三者一致へ拡張する
- 正をどちらのファイルに置くか（`Directory.Build.props` から `Package.wxs` へ流し込むか、その逆か）は実装上の選択であり、**委任事項 1** とする

### 決定 8: 署名基盤が使えなくなった場合の退避経路をあらかじめ決めておく

**この決定は決定 1 の裁定によらず成立する**（どちらを選んでも、第三者に依存することに変わりはない）。切れたときの手順を決めずに依存しない。

**共通の設計要件**: **署名基盤への依存を `release.yml` の 1 ジョブに閉じ込める**（委任事項 6）。署名の実行主体を差し替えても、決定 3 のワークフロー構造（未署名で機能検証 → 署名 → 署名検証 → 署名済みのみ公開）が維持できるようにする。これは (A) と (C) の相互乗り換えを可能にするための保険でもある。

**(A) SignPath を選んだ場合**:

1. **申請が却下された場合**: 却下理由が「評判不足」であれば、**整備した内容（ポリシーページ・メタデータ・ワークフローの署名対応）はそのまま維持し、署名ステップだけを無効化して**リリースを継続する。実績を積んでから再申請するか、(C) へ移行する。**署名なしのままなら v1.0 は ADR-0006 基準 5 を満たせないため延期する**（署名なしでの v1.0 公開は ADR-0006 の supersession を要する別の決定であり、本 ADR では選ばない）
2. **サブスクリプションが停止・証明書が失効した場合**: (C1) または (C2) へ移行する
3. **失効が遡及適用された場合**（terms は「retroactively」の可能性を明記）: 既公開の署名済み MSI が「失効した証明書で署名されたもの」になる。タイムスタンプ付き署名は証明書の**期限切れ**には耐えるが、**失効**には耐えない場合がある。この事象が起きた際は、SECURITY.md でその事実と影響範囲を告知する（隠さない）

**(C) 有料証明書を選んだ場合**:

1. **署名資格情報（GitHub Secrets）が漏洩した疑いが生じた場合**: 直ちに Secrets をローテートし、CA に証明書の失効を要求する。**失効すると、その証明書で署名した既公開バイナリすべてが影響を受ける**（Yagura だけでなく ODV も。1 枚で全プロジェクトを署名することの裏面である）。この波及範囲を受け入れた上で (C) を選ぶこと
2. **費用の支払いが継続できなくなった場合**: (A) への移行を検討する（Yagura は OSS なので (A) の適格要件は満たす。ODV は満たさない）
3. **CA が個人発行を停止した場合**: 別 CA へ移行する

## 帰結

**決定 1 が未確定のため、帰結も両案について書く。** 決定 2〜7 に由来する帰結は共通である。

### 共通して良くなること（どちらを選んでも）

- 利用者が MSI を実行したときの SmartScreen 警告が軽減され、発行元が「不明」ではなくなる。ADR-0001 が掲げる「Windows 管理者が既に持つスキルだけで導入できる」という思想において、**警告を無視する手順を利用者に教えなくて済む**ことの価値は大きい（「警告を無視して進む」習慣を育てない、という `configuration.md` §6 の自己署名証明書に関する判断と同じ原則）
- リリース資産の真正性が、配布経路への信頼に依存せず検証可能になる
- 製品メタデータが一元化され、git タグ・MSI・アセンブリのバージョン不整合が CI で検知される（決定 7。**現状は三者がバラバラで、アセンブリは `1.0.0.0` のまま**）
- ADR-0006 基準 5 と ADR-0009 委任事項 8、Issue #264 が閉じる

### 共通して悪くなること（どちらを選んでも）

- CI のワークフローが複雑になる（ジョブ数の増加、artifact の受け渡し、署名待ちのタイムアウト設計）
- 署名基盤という第三者依存が 1 つ増える。可用性・継続性は Yagura の管理外にある
- **SHA256 の算出位置を移設しなければならない**（署名はファイルを変える）。移設漏れは「公開したチェックサムが必ず一致しない」という形で利用者に直撃する

### (A) SignPath を選んだ場合に固有の帰結

- 良くなること: **秘密鍵も署名権限も Yagura 側に存在しない**。origin verification が「このコミットから CI が生成した」ことを技術的に強制する。リリースごとに人間の承認が入る。費用ゼロ。実名を公開しなくて済む
- 悪くなること: **リリースが完全自動でなくなる**（タグを push したら Release ができる、という運用は失われる）。**緊急セキュリティ修正でも承認が要る**（SECURITY.md の SLA と矛盾しないか——委任事項 5）。**発行元表示が Yagura ではなく SignPath Foundation になる**。terms の運用制約（ポリシー掲示・MFA・メタデータ統一・役割分担）を恒久的に守る義務を負い、terms が変われば追従する
- **受け入れるリスク（審査で却下される）**: terms は「For executable programs that may be downloaded and executed based on our signature, we require a certain verifiable reputation」と明記する。2026-07-14 時点の Yagura は **star 0・fork 0・リポジトリ作成から 11 日・リリース資産のダウンロード数は各 0〜4 件**であり、この要件に照らして強い立場にはない。技術要件（OSI ライセンス・リリース済み・GitHub ホストランナーでのビルド・アンインストール提供）はすべて満たせる見込みだが、**評判要件は満たしているとは言えない**
  - この事実を隠して申請しない。体制（単一メンテナ・新しいプロジェクト）をそのまま申告する
  - **整備作業（ポリシー・メタデータ・ワークフロー）は却下されても無駄にならない**——(C) へ移行する場合にも再申請する場合にもそのまま使える

### (C) 有料証明書を選んだ場合に固有の帰結

- 良くなること: **ODV を含む全プロジェクトを 1 枚で署名できる**。審査に落ちるリスクがない。緊急リリースが承認待ちで律速されない。発行元が実名になり「誰が作ったか」が直接示される
- 悪くなること: **年 3〜5 万円の継続費用**。**実名・所在地が署名済みバイナリを通じて恒久的に公開される**（取り消せない）
- **受け入れるリスク（署名権限が CI に載る）**: 鍵は HSM から出ないが、**署名を発火させる資格情報は GitHub Secrets に置かれる**。リポジトリまたは CI を掌握した攻撃者は、**オーナーの実名で任意のバイナリに署名できる**。(A) が origin verification と手動承認で構造的に塞いでいる経路を、(C) では自力で塞ぐ必要がある（GitHub Environments の承認ゲート、タグ限定の権限、署名ジョブの分離）。**この対策を設計要件として引き継ぐことを (C) 採用の条件とする**
- **失効の波及範囲が広い**: 1 枚で全プロジェクトを署名する以上、失効は全プロジェクトに波及する（決定 8-(C)-1）

### どちらでも約束できないこと

**SmartScreen の警告が「消える」とは約束できない。** SmartScreen の評価は証明書ごとの評判に基づく。(A) の SignPath Foundation 証明書は多くの OSS が共用するため評判が蓄積していると**期待される**が実測していない。(C) の新規発行証明書は**評判ゼロから始まる**（EV でなければ即座の評判は付かない）——この点はむしろ (A) が有利な可能性がある。したがって:

- 入口文書に「SmartScreen 警告が出なくなる」とは書かない。「署名され、発行元が検証可能である」という事実のみを書く
- **署名済み MSI での SmartScreen の実挙動を lab で検証し、その結果を記録する**（conventions.md「実環境依存の機能は lab 検証を受け入れ条件に含める」）。**委任事項 4**
