- 日付: 2026-07-14（決定 1 の裁定: 2026-07-15）
- 決定者: YANAI Taketo
- 関連: [Issue #264](https://github.com/Yanai-Taketo/Yagura/issues/264)（本 ADR がこの Issue の「証明書の調達方針を決める」「ADR を起案するか判断する」に応える）、[ADR-0006](0006-v1-release-criteria.md)（基準 5「リリースの完全性」の実現手段を確定する）、[ADR-0009](0009-architecture-support.md)（委任事項 8「コード署名方針」を閉じる）、[ADR-0005](0005-oss-packaging.md)（入口文書の体裁）、[ADR-0004](0004-security-model.md)（セキュリティモデル）、[SECURITY.md](../../SECURITY.md)、[conventions.md](../development/conventions.md)（リリース）

## 文脈と課題

### 現在地

Yagura のリリース資産（`Yagura-<版>-x64.msi` / `Yagura-<版>-arm64.msi`）は **Authenticode 署名されていない**。利用者は MSI を実行すると SmartScreen の「Windows によって PC が保護されました」警告に遭い、「発行元: 不明」と表示される。回避には「詳細情報 → 実行」の 2 クリックが要る。

改ざん検知の手段としては `.sha256` チェックサムを添付しているが、これは**配布経路が信頼できることを前提にした完全性チェック**であって、真正性（誰が作ったか）を証明しない。チェックサムと成果物が同じ GitHub Release に並んでいる以上、Release を掌握した攻撃者は両方を差し替えられる。

### 決定を要求している既存の合意

コード署名は新規の思いつきではなく、すでに 2 つの ADR が宿題として登録している。

- **ADR-0006 基準 5「リリースの完全性 — コード署名を含む」**: v1.0 公開の要件として「リリース資産（インストーラ）に **Authenticode コード署名（タイムスタンプ付き）**が付与されている」ことを要求し、証拠として「リリースノートから署名の検証手順（signtool verify 等）と `.sha256` が辿れること」を求めている。同 ADR は「証明書の入手・管理はリリース準備の課題として別途扱う」と明記して手段の決定を先送りした
- **ADR-0009 委任事項 8**: 「リリース成果物のコード署名（Authenticode）方針 / 別 ADR または ADR-0005 の amendment。署名の要否・証明書の入手と鍵管理・リリースワークフローへの組み込み。アーキ数に依存しない独立論点だが、導入時は全アーキ成果物への適用を設計に含める」

**本 ADR はこの 2 つを閉じる。** ADR-0006 は「署名する」という要否をすでに決めているため、本 ADR が決めるのは**手段**（どの証明書を、どう入手し、どう鍵を守り、どうワークフローへ組み込むか）である。

### 課題の核心 — OSS 個人プロジェクトは証明書を買えない

コード署名証明書は、**法的主体（legal entity）に対してのみ発行される**。OSS プロジェクトそれ自体は法人格を持たないため、CA から見た受取人になれない。さらに 2 つの制約が個人開発者を締め出している。

1. **秘密鍵のハードウェア保管が必須**: CA/Browser Forum の Code Signing Baseline Requirements により、**2023-06-01 以降、EV・非 EV を問わずコード署名証明書の鍵ペアは FIPS 140-2 Level 2 または Common Criteria EAL 4+ 以上のハードウェア暗号モジュール（HSM／ハードウェアトークン）で生成・保管することが必須**となった（[CA/Browser Forum Code Signing Requirements](https://cabforum.org/working-groups/code-signing/requirements/)、確認日 2026-07-14）。ソフトウェア鍵での運用は不可能になり、個人が負担する初期費用・運用手間が上がった
2. **CI との相性の悪さ**: ハードウェアトークンは物理的に 1 か所にしか挿さらない。GitHub Actions のようなクラウド CI から署名するには、クラウド HSM か署名サービスを別途調達する必要がある。「手元のマシンで手動署名してアップロード」はビルドの再現性・来歴（provenance）を壊し、ADR-0006 が求める「検証可能な完全性」の趣旨に反する

つまり Yagura が署名を実現するには「**個人が法人格なしに、公的に信頼される証明書へアクセスでき、かつ CI から自動で署名できる**」経路が要る。この 3 条件を同時に満たす選択肢は多くない。

### この決定を Yagura 単独で閉じられない事情 — もう 1 つのプロジェクトの存在

オーナー（YANAI Taketo）は Yagura のほかに **[Open DUMP Viewer](https://github.com/Open-DUMP-Viewer/Open-DUMP-Viewer)**（以下 ODV）を公開している。ODV の性質は本 ADR の選択肢を強く制約するため、事実として記録する（確認日 2026-07-14）。

- ODV は **OSS ではない**。`LICENSE` は "Open DUMP Viewer for Oracle database Proprietary License" と題され、`All rights reserved.` で始まり、本文に「The source code is made available for reference purposes only ("Source-Available") and is **NOT open-source software**」と明記されている（ソース公開の理由は透明性・監査可能性と知識継承であると同ファイルが述べる）
- ODV は `.msixbundle` と Inno Setup 製の `.exe` インストーラを配布しているが、**現時点でコード署名証明書を持たない**（リリースワークフローに `signtool` は 0 件、MSIX は Microsoft Store 提出用の未署名構成。README がスポンサー資金の使途に "code-signing certificates" を挙げている。2026-07-14 実測）

ここから 2 つの帰結が出る。

1. **無償の OSS 向け証明書は、原理的に ODV を署名できない**。SignPath Foundation は「OSI 承認ライセンス・プロプライエタリ要素なし」を適格要件とし、Certum Open Source は OSS 専用で商用・非 OSS への使用を失効事由とする。ODV はどちらの要件も満たさない
2. **したがって「1 枚の証明書で自分の全プロジェクトを署名する」構成を取れるのは、通常の有料証明書だけ**である。ODV が証明書を必要としており、その有料証明書を取得するなら、**同じ 1 枚で Yagura も署名できる**

**この相互依存が決定 1 の裁定を導いた。** Yagura の ADR が ODV の意思決定を縛ることはできないが、選択肢の評価に ODV の存在を織り込まないのは事実の隠蔽にあたる。ODV と Yagura の双方が証明書を必要とし、ODV は無償証明書を使えない——この構図から、**有料証明書 1 枚を両プロジェクトで共用する (C2) が選ばれた**（決定 1）。

### (A) SignPath Foundation の無償 OSS 証明書 + SignPath.io の署名基盤 【次点 — ODV を賄えないため不採用】

[SignPath Foundation](https://signpath.org/) は OSS プロジェクトに無償のコード署名を提供する非営利団体。証明書は **SignPath Foundation 名義で発行され、それを OSS プロジェクトに使わせる**（[terms](https://signpath.org/terms.html)、確認日 2026-07-14）。プロジェクトが法人格を持つ必要がない。署名は SignPath.io のクラウド HSM 上で行われ、GitHub Actions から `signpath/github-action-submit-signing-request` で要求する。

- 利点:
  - **費用ゼロ**。個人 OSS の継続性（ADR-0001 の運営原則）と両立する
  - **鍵を一切持たない**。秘密鍵は SignPath.io の HSM にあり、Yagura のリポジトリにも CI にも鍵が存在しない。漏洩させる鍵がないことが最大の防御になる（ADR-0004 の最小権限の思想と同型）
  - **来歴の技術的強制**: SignPath は署名要求のたびに origin verification を行い、「その成果物が、指定されたリポジトリのソースから、GitHub ホストランナー上の実際のワークフロー実行で生成されたこと」を検証する。この検証はビルドスクリプトの自己申告ではなく **GitHub 側が提供するメタデータ**に基づく。署名は「Yagura が作った」だけでなく「**このコミットから CI が自動生成した**」ことの証明になり、ADR-0006 が求める完全性の水準を、単なる Authenticode 署名より高い位置で満たす
  - リリースごとに**人間の手動承認**を要求する（terms「Every release needs manual approval for signing」）。CI が単独でリリース資産を署名・公開できない構造は、リポジトリ侵害時の被害を限定する
- 欠点・リスク:
  - **審査に通る保証がない**（後述「受け入れるリスク」）。terms は実行可能プログラムについて「a certain verifiable reputation」を要求すると明記する
  - **第三者への依存**が増える。SignPath Foundation がサブスクリプションを停止・証明書を失効させれば、以後のリリースは署名できなくなる（terms は「pause or terminate the subscription without prior notice」「revoke the certificate effective immediately or retroactively」の権利を留保している）
  - **発行元表示は "SignPath Foundation"** になる。Yagura の名前は証明書の Subject に出ない。利用者から見た発行元は Yagura ではなく SignPath Foundation である
  - terms が課す運用制約（コード署名ポリシーの掲示、全メンバーの MFA、メタデータの統一、役割分担の明文化）を恒久的に守る義務を負う
  - **Yagura しか署名できない。** 証明書は SignPath Foundation 名義でプロジェクト単位に貸与される（terms「Sign your own projects only」）。ODV は OSS ではないため対象外であり、**オーナーが ODV の署名手段を別途調達するなら、その手段は Yagura にも使える**——このとき (A) の最大の利点である「無償」は、Yagura にとっての**限界費用ゼロという相対優位を失う**（前掲「もう 1 つのプロジェクトの存在」）

### (B) Azure Artifact Signing（旧 Trusted Signing）【却下】

Microsoft のマネージド署名サービス。月額約 $10 と安価で、鍵は Azure 側の HSM に置かれ、CI 連携も公式アクションがある。個人開発者（self-employed）にも 2026 年時点で開放されており、旧来の「3 年の事業履歴」要件は個人には課されない。

- **却下理由（実体検証済み）**: **提供地域が限定されており、日本の個人開発者は対象外**。Microsoft Learn の [Quickstart: Set up Artifact Signing](https://learn.microsoft.com/en-us/azure/artifact-signing/quickstart) は次のように明記する（確認日 2026-07-14、原文）:

  > For Public Trust certificates, Artifact Signing is currently available to organizations in the USA, Canada, the European Union, and the United Kingdom, as well as individual developers in the USA and Canada.

  Yagura のオーナーは日本の個人であり、組織（米国・カナダ・EU・英国）にも個人開発者（米国・カナダ）にも該当しない。技術的な優劣ではなく**資格要件で門前払いになる**ため、現時点では選択肢にならない。

  なお Azure のリージョンとしては Japan East が Artifact Signing をサポートしており（同ページのリージョン表）、リージョンの話と**証明書発行の資格地域の話は別**である点に注意する。この 2 つを混同すると「日本でも使える」と誤読しかねない。
- **再評価トリガ**: 提供地域に日本が追加された場合（後述）

### (C) 有料の商用証明書を個人名義で購入する 【採用系統】

「1 枚の証明書で、オーナーが持つ全プロジェクト（Yagura・ODV・将来のもの）を、実名で、審査なしに署名する」構成。個人名義で発行され、鍵はクラウド HSM に置き、GitHub Actions から署名する。

**共通の利点**:

- **ODV を含む全プロジェクトを 1 枚でカバーできる**。証明書は主体（個人）に紐づき、プロジェクト識別子を含まない。DigiCert は FAQ で「署名できるアプリ数に上限なし」、GlobalSign は「署名対象アプリケーション数: 無制限」と明記する。**ODV のために証明書を買うなら、Yagura の署名は限界費用ゼロで手に入る**
- **審査に落ちるリスクがない**。本人確認（政府発行 ID 等）を通れば発行される。評判要件は存在しない。**Yagura が star 0 の新しいプロジェクトであることが一切不利にならない**
- **発行元が実名になる**。「誰が作ったか」の答えが直接的になる（プライバシーとのトレードオフは後述）
- **リリースごとの手動承認が強制されない**。緊急のセキュリティ修正を承認待ちなしに出せる
- 第三者の審査・運営方針に継続的に依存しない（CA の失効権限は残るが、terms 違反による裁量的なサブスクリプション停止という経路はない）

**共通の欠点・リスク**:

- **継続費用**（年 3〜5 万円規模）。個人が無報酬で運営する OSS に、リリースを続ける限り毎年かかるコストを載せる
- **実名・所在地が証明書に載り、署名済みバイナリを通じて恒久的に公開される**。個人 OSS 開発者にとって無視できない帰結
- **署名の資格情報を GitHub Secrets に置くことになる**（eSigner の認証情報 + TOTP シークレット、または GCP のサービスアカウント鍵）。**これが本 ADR で最も重い論点である**——リポジトリまたは CI を掌握した攻撃者は、**任意のバイナリにオーナーの実名で署名できる**。鍵そのものは HSM から出ないが、「署名させる権限」は CI にある。(A) の origin verification（署名対象が指定リポジトリのコミットから CI で生成されたことを SignPath 側が検証する）と手動承認は、まさにこの経路を塞ぐための仕組みであり、(C) にはそれがない。**単一メンテナが実名で署名する構成において、この差は金額では測れない**
- 署名回数の従量上限がある方式がある（eSigner Tier1 は月 20〜240 署名、DigiCert KeyLocker は年 1,000 署名）。Yagura のリリース頻度では問題にならないが、CI で試験署名を回す設計にすると効いてくる

#### (C1) SSL.com IV Code Signing + eSigner 【不採用 — MFA を設計上無効化する】

- 証明書 **$129/年**（5 年契約で $96.75/年）+ **eSigner 別課金 Tier1 = 年 $180**（月 20〜240 署名、超過 $1/回）。実質 **年 $309 前後**。新規証明書は最初の 30 日 eSigner 無料
- **IV（Individual Validation）は「事業体不要・個人名を検証」する商品**であり、独立開発者・OSS メンテナ向けと公式に位置づけられている。地域制限の記載はない（日本可の明示も確認できていない）
- **公式 GitHub Action がある**（`SSLcom/esigner-codesign`。最新 v1.3.2 = 2025-12-22）。Ubuntu ランナーで動作する。CSC 準拠の REST API もある
- 本人確認: 政府発行 ID の表裏 + ID を顔の横に持った写真。標準 3〜5 営業日
- **注意すべき材料**: eSigner の課金運用について第三者から強い批判が出ている（MetaBrainz の公式ブログ記事等）。真偽は検証していないが、契約前に課金条件を精読すべき

#### (C2) Sectigo（リセラー）+ Google Cloud KMS + jsign 【採用】

- 証明書 **約 $220/年**（リセラー実売。Sectigo 公式は「5 年契約で $536.25/年から」としか出しておらず、一次価格が公開されていない）+ Google Cloud KMS の HSM 鍵の月額（実践記事では $2.50/月 + $0.15/1 万署名。**正確な料金表は確認できず——委任事項 A で発注前に確認する**）
- **Sectigo 公式 KB が個人発行を明記**する: "OV Code Signing certificates may be issued to: **Individuals**, Private organizations, …"
- **Google Cloud HSM の attestation を Sectigo が受理する**（公式 KB に手順あり）。**Azure Key Vault は鍵 attestation に対応しないため Sectigo では使えない**
- **署名ツールは 2 通りとも成立する**（実装 PR で確定＝委任事項 2）。①`jsign`（`--storetype GOOGLECLOUD` + `--certfile` + `--tsaurl <TSA> --tsmode RFC3161`。Linux ランナー可、MSI/EXE/MSIX 対応、複数 TSA フォールバック内蔵）を公式ドキュメントで確認（2026-07-15）。②実装事例（[zenn.dev/inop](https://zenn.dev/inop/articles/7db6acf976c519)）は **Windows の SignTool + Google Cloud KMS CNG Provider** を使用（Windows ランナー）。いずれも `--storepass` 相当に **Workload Identity Federation の短命トークン**を渡せ、長期の秘密鍵 JSON を GitHub Secrets に置かない
- **GitHub Actions からの署名パイプライン自体は実例で成立が確認できている**（上記 zenn 記事。Sectigo + GCP KMS HSM + WIF + `roles/cloudkms.signerVerifier`、RSA 4096/PKCS#1 v1.5/SHA256、attestation は KMS UI から取得、CSR は PKCS#11 経由。**Sectigo とのやり取りが日本語でできた**とも記載）。**ただし当該事例は法人（Organization）審査**であり、**個人（Individual）審査 + GCP KMS 外部鍵の組み合わせだけは実績を確認できていない**——採用にあたっての唯一の未検証点であり、**委任事項 A**（発注前に Sectigo リセラーへ「個人 + 外部 HSM(GCP KMS)」の可否を確認する）とする
- 署名回数の従量課金が事実上ゼロで、長期的には最安クラス。引き換えに構築・保守を自分で負う
- 署名回数の従量課金が事実上ゼロで、長期的には最安クラス。引き換えに構築・保守を自分で負う
- **採用の決め手（裁定）**: (C1) と違い **TOTP シークレットを GitHub Secrets に置かない**（WIF で代替）。1 枚で ODV も署名できる。個人発行が公式に裏づけられている。(A) の唯一の本質的優位（署名承認・証跡が GitHub と独立した認証境界にある）を、最小権限 IAM + Cloud Audit Logs + GitHub Environments 承認ゲートで**かなりの程度まで模倣できる**（田中の指摘。決定 1 参照）

#### (C3) Certum Open Source Code Signing 【却下 — CI から署名できない】

- **€49〜69/年**と最も安く、**個人専用・OSS 専用**で、Subject は `Open Source Developer, <実名>`。一見この用途に最も適した商品に見える
- **却下理由**: **GitHub ホストランナーから署名できない**。クラウド版（SimplySign）は SimplySign Desktop という GUI アプリが仮想カードリーダーをエミュレートし、モバイルアプリの OTP で対話ログインする方式で、セッションは 2 時間で切れる。**公式のクラウド署名 REST API が存在しない**（確認日 2026-07-14）。有志の回避策（TOTP シークレット抽出 + SendKeys、Xvnc + p11-kit コンテナ等）は存在するが、いずれもデスクトップ環境を要求し GitHub ホストランナーでは動かない
- GUI 付きの self-hosted ランナーを常時運用するなら選択肢になるが、**それは「ビルドを GitHub ホストランナーで行う」という現行の前提（および SignPath の origin verification が要求する条件）を捨てることを意味し、供給網の保証を弱める方向の変更**である。安さのために保証を下げる取引は本 ADR では選ばない
- **ODV も署名できない**（OSS 専用。商用配布ソフトへの使用は公式に失効事由）

#### (C4) DigiCert + KeyLocker 【却下 — 個人発行の可否が確認できない】

- CI 連携は最も洗練されている（公式アクション `digicert/code-signing-software-trust-action`）が、**個人（自然人）への発行可否が公式情報で確認できなかった**。公式 FAQ は "a Code Signing certificate is tied to your Organization Name only" と組織を前提に書かれている。価格も公開されていない（リセラー掲載で OV $400〜）
- KeyLocker は **年 1,000 署名の従量上限**つき
- 個人で買える確証が取れない以上、対抗案の候補から外す。**(C1)/(C2) が両方とも駄目だった場合に、可否を直接問い合わせる先として残す**

#### (C5) GlobalSign / GMO グローバルサイン 【却下 — 法人のみ】

- 日本語・円建て・国内サポートという利点があり、日本在住者にとって最も摩擦が少なそうに見えるが、**個人は申請できない**。GlobalSign 公式ショップは "Code Signing Certificates can be issued in the name of **a legally registered organization only**" と明記し、GMO グローバルサイン日本公式も「発行対象: 法人」と明記する（¥60,000/年 税抜、EV ¥78,000/年）
- **法人化すれば選択肢に入る**が、コード署名のために法人を作るのは本末転倒であり、本 ADR の範囲外

### (D) Sigstore / cosign による署名【却下 — 問題を解決しない】

キーレス署名。OIDC で身元を証明し、証明書は短命。透明性ログ（Rekor）に記録される。ADR-0006 の 82 行が「無償・低コストの代替」の例として名指ししている。

- **却下理由（実体検証済み）**: **Windows の Authenticode 検証を満たさない**。Authenticode は Microsoft Trusted Root Program に参加する CA が発行した証明書チェーンを要求するが、Sigstore の CA（Fulcio）はこのプログラムに参加していない。したがって Sigstore のみで署名した Windows バイナリは、依然として SmartScreen で「発行元不明」と表示される（確認日 2026-07-14）。

  Sigstore は「配布物の来歴を検証可能にする」目的には有効だが、**ADR-0006 基準 5 が解決しようとしている問題（SmartScreen 警告への対処）そのものを解決しない**。ADR-0006 が代替候補として挙げた項目だが、実体を確認した結果、代替になっていないことがここで判明した。これは ADR-0006 の歯止め条件①「代替の検討記録を残すこと」に対する記録である
- **併用の可能性は残す**: GitHub Actions の artifact attestation（SLSA provenance）との併用は Authenticode 署名と排他ではない。ただし本 ADR のスコープ外とし、委任事項として登録する

### (E) 署名しない（現状維持）【却下】

- **却下理由**: ADR-0006 基準 5 が v1.0 の公開要件として署名を明示的に要求している。署名しないことを選ぶには、ADR-0006 の supersession が必要であり、その場合は同 ADR の歯止め条件（82 行）により「**v1.0 の延期を対等な選択肢として併記・比較する**」ことと「README・リリースノートへの署名なしの明示」が義務づけられる。無償の実現手段（(A)）が存在し、まだ試してもいない段階でこの道を選ぶ理由はない

## 決定

### 決定 1: (C2) Sectigo の商用証明書（個人名義）+ Google Cloud KMS を採用し、証明書 1 枚を Open DUMP Viewer と共用する

**オーナー裁定 2026-07-15。** ペルソナレビュー（PR #278）で両論併記した (A) SignPath Foundation と (C) 有料証明書のうち、**(C2) Sectigo + Google Cloud KMS + jsign** を採る。証明書は個人（YANAI Taketo）名義で 1 枚取得し、**Open DUMP Viewer（ODV）と Yagura の両方の署名に用いる**。GitHub Actions から署名する。

#### 駆動因を確定する（クリスの再レビュー指摘）

**この裁定の駆動因は「オーナーが、自分の全プロジェクトを 1 枚の有料証明書で実名署名する、と決めたこと」である。** これはオーナーが下す価値・選好の判断（有料でよい・実名でよい・1 枚共用でよい）であって、**「ODV がどうせ証明書を買うだろう」という予測ではない**。

クリスの再レビューは、初版の言い回しが「ODV があるから (C)」という予測に読め、**Yagura 単独の最善（無償の (A) で足りたかもしれない）と、2 プロジェクト合算の最適（1 枚共用）とをすり替えている**と突いた。これを正す。実測のとおり ODV は現に証明書を持たず調達も未確定であり、「ODV がいずれ買う」は根拠にできない。**採用の根拠を予測に置かず、オーナーの調達意思決定そのものに置く**。オーナーは ODV と Yagura を同一証明書で運用すると明示的に決めた（2026-07-15）——本裁定はその意思決定を前提に、有料系統の中で最善の実装（(C2)）を選ぶものである。

#### なぜ有料系統の中で (C2) か

- **(C1) SSL.com ではない**: (C1) eSigner は TOTP シークレットを GitHub Secrets に置き、**多要素認証を設計上無効化する**（決定 6 の MFA 要件と正面から矛盾する）。(C2) は Workload Identity Federation で**長期の秘密を GitHub Secrets に置かずに済む**——GCP KMS 署名は OAuth アクセストークンで行えるため、WIF が発行する短命トークンをそのまま使える（jsign / SignTool+CNG のいずれでも成立することを実装事例で確認。2026-07-15）
- **(C3) Certum は CI から署名できない、(C5) GlobalSign は法人のみ、(C4) DigiCert は個人可否が不明**（検討した選択肢を参照）。日本の個人が CI から署名でき、外部 HSM(GCP KMS) を使える実現手段として (C2) が残る

#### なぜ (A) SignPath ではなく有料系統か

オーナーが「1 枚で全プロジェクトを実名署名する」と決めた以上、**(A) は ODV を賄えない**（非 OSS のため適格要件を満たさない）ため、そもそも駆動因を満たさない。加えて審査リスク（(A) は "verifiable reputation" を要求し star 0 の Yagura が通る保証がない。却下なら v1.0 延期）を負わずに済む。

（(A) には無償で**唯一の本質的優位**があった——署名の承認と証跡が GitHub とは独立した認証境界の向こう側にあり、GitHub を掌握した攻撃者には消せない。ADR-0004 決定 7「監査記録を Windows イベントログにも併記する」と同型の論法である。初版が (A) の根拠に挙げた「origin verification が技術的に強制する」「リリースごとの手動承認」は、**汚染コミットは origin verification を定義上パスする・単一メンテナでは承認者＝攻撃者**という理由でペルソナレビューで撤回した。この撤回は記録として残す。）

**(C2) はこの (A) の優位を "部分的に" 模倣する**——最小権限 IAM・**明示的に有効化した Cloud Audit Logs**（既定は無効。田中の指摘。委任事項 4）・GitHub Environments の承認ゲート・WIF の厳格な信頼条件（委任事項 3）による。**ただし「かなりの程度」という定量化されていない評価は取り下げる**（クリスの指摘）。正確には次のとおりである:

- **模倣が成立するのは、監査ログを実際に有効化し（委任事項 4）、WIF の信頼条件を特定のリポジトリ・ref・Environment に絞った（委任事項 3）場合に限る**。どちらも既定では効いておらず、実装で明示的に組んで初めて成立する
- **それでも GitHub アカウントの乗っ取りが CI 経由の署名につながる経路は残る**（(A) では起きない）。(C2) の署名証跡は「GitHub の外」ではなく「GCP の中」にあり、GCP へのアクセスが GitHub の OIDC にフェデレートされている以上、境界は (A) ほど独立していない
- したがって裁定は「(C2) が (A) と同等の保証を持つ」とは主張しない。**「オーナーが 1 枚共用・実名・有料を選んだ以上、有料系統の中では (C2) が最も安全側に寄せられる」**という限定的な主張にとどめる

#### 受け入れるリスク（裁定にあたって明示的に引き受ける）

1. **署名を発火させる権限が CI に載る。** 鍵は GCP KMS の HSM から出ないが、リポジトリまたは CI を掌握した攻撃者は**オーナーの実名で任意のバイナリに署名できる**。(A) が独立した認証境界で構造的に塞いでいた経路を、(C2) では自力で塞ぐ——WIF（長期秘密の排除）、最小権限 IAM、Cloud Audit Logs、GitHub Environments の承認ゲート、タグ限定の権限、署名ジョブの分離（**決定 8 の設計要件・委任事項 5**）。これらは (A) の保証の**完全な代替ではない**（GitHub アカウントの乗っ取りが CI 経由の署名につながる経路は残る）ことを承知の上で引き受ける
2. **実名・所在地が証明書に載り、署名済みバイナリを通じて恒久的に公開される**（取り消せない）
3. **Yagura の署名の真正性が ODV の CI の安全性にも従属する**（田中・クリスの指摘。**受け入れる主体（オーナー）と負担する主体（利用者）が異なる論点**）。1 枚の証明書を共用する以上、ODV の CI を掌握した攻撃者は **Yagura として正当に検証される署名**を作れる。攻撃面が両リポジトリの和集合になる。**この緩和策の限界を正確に書く**（初版は緩和策を予防のように書いていた。田中・クリスの指摘で訂正）:
   - リポジトリごとに別のサービスアカウント/WIF プールを設けても、**両者は同一の KMS 鍵に対する署名権限を持つ**（暗号的な境界にはならない。田中）。**証明書が同一である以上、ODV 経路で作られた署名は、利用者が決定 5 の期待値（Subject・thumbprint）で照合しても Yagura の正規署名と区別できない**（クリス）
   - したがって「リポジトリ別サービスアカウント + Cloud Audit Logs」は**予防ではなく事後の検知・帰属**である（誰が署名したかを後から追える。署名させないことはできない）。ポリシーにもこの限界を正確に開示する（「区別できない」ことを隠さない）
   - **別証明書 2 枚に分ければ相互汚染を暗号的に断てる**（ODV 侵害が Yagura の thumbprint を作れなくなる）。これは費用が 2 倍になる。**オーナーは費用と引き換えに 1 枚共用を選んだ**——この却下した代替（2 枚）を記録として残す。**相互汚染リスクが許容できないと判明したら 2 枚へ分ける**（再評価トリガに登録）
4. **継続費用**（Sectigo 約 $220/年 + GCP KMS 月額。**発注前に総額を確認する——委任事項 A**）
5. **失効の波及範囲が広い**: 失効は 1 枚を共用する全プロジェクト（Yagura・ODV）に波及する。この被害を限定する唯一の制御が**タイムスタンプ**である（決定 9）

#### 委任 A（個人審査 + GCP KMS 外部鍵の可否）が不可だった場合の分岐（クリスの指摘）

**(C2) には唯一の未検証点がある**——「個人（自然人）審査で、GCP KMS の外部鍵（attestation 経由）に発行できるか」。実装事例（zenn.dev/inop）は技術パイプラインの成立を示すが**法人名義**であり、個人名義でのこの組み合わせは確認できていない（**委任事項 A で発注前に Sectigo へ確認する**）。この 1 点が裁定全体の荷重を負うため、**不可だった場合の分岐を裁定に接続しておく**（事故の最中に考えない）:

1. **別 CA で「個人 + 外部 HSM(GCP KMS or Azure Key Vault)」が可能なものを探す**（(C4) DigiCert の個人可否を直接確認する等。Azure Key Vault が使える CA なら KMS の代わりに使える）
2. それも不可なら、**Yagura のみ (A) SignPath（無償・OSS 適格）に退避し、ODV は別途手当てする**（個人 + ハードウェアトークン + self-hosted ランナー等）。この場合 1 枚共用は諦める
3. **いずれの分岐でも、決定 2〜9 の設計（署名対象・ワークフロー・タイムスタンプ・退避弁）は署名基盤に依存しないため再利用できる**（委任事項 6 の「1 ジョブに閉じ込める」設計の効用）

#### この決定に依存する範囲・しない範囲

- **決定 1 の裁定に依存せず既に確定していた**: 決定 2（署名対象の範囲）、決定 3（ワークフローの構造）、決定 4（全アーキ適用）、決定 7（メタデータの一元化）、決定 9（タイムスタンプ）——**これらは裁定前から実装に着手できる状態にあった**
- **裁定により (C2) 形に確定した**: 決定 5（ポリシーの掲示内容）、決定 6（役割の明文化）、決定 8（退避経路・署名資格情報の保護）

### 決定 2: 署名対象は「MSI + 自前アセンブリ」とし、上流バイナリは署名しない

署名対象を次のとおり限定する。(C2) の自前証明書では上流バイナリも法的には署名できるが、以下の理由でしない。

| 対象 | 署名 | 根拠 |
|---|---|---|
| `Yagura-<版>-<arch>.msi`（インストーラ本体） | **する** | Yagura のビルドスクリプトが生成する自前成果物。利用者が実行する入口であり、SmartScreen が評価する対象 |
| `Yagura.Host.exe`・`Yagura.*.dll`（Yagura が書いたアセンブリ） | **する** | 自前ソースからのビルド成果物。MSI 内部の deep signing |
| .NET ランタイム一式（self-contained publish に同梱される `coreclr.dll` 等） | **しない** | **Microsoft が既に署名している**。自分の証明書で再署名すると Microsoft の署名を上書きすることになり、かえって信頼の連鎖を損なう |
| MudBlazor 等の第三者アセンブリ（NOTICE に掲載） | **しない** | 上流 OSS が配布する形のまま同梱する。自分が書いていないコードに自分の名前で署名しない（来歴の明確さを優先） |

**「MSI は署名されているが、中の第三者 DLL の一部は未署名」という状態を正とする。** これは Authenticode の一般的な構成であり（未署名バイナリを署名済み MSI に内包することは許容される）、隠すべき欠陥ではない。（MudBlazor のアセンブリが実際に未署名であることは `Get-AuthenticodeSignature` で確認済み——田中の検証による。推測ではない）

**ただし帰結を明記する**（田中の指摘。初版はこれを書いていなかった）: **未署名バイナリが同梱される以上、利用者は WDAC / AppLocker の「発行元（publisher）ルール」だけで Yagura の実行を許可できない**。厳格なアプリケーション制御を敷いている環境では、未署名 DLL に対してハッシュルールやパスルールの併用が必要になる。この制約を SECURITY.md に記載する（ADR-0004 の想定読者——閉域・統制環境の管理者——にとっては実務上の情報である）。

### 決定 3: リリースワークフローは「未署名で機能を検証し、署名済みで署名を検証し、署名済みのみを公開する」

`release.yml` を次の構造へ改める。

```
build (x64 / arm64)
  → 未署名 MSI を GitHub artifact としてアップロード
  → [既存] Full E2E スモーク（x64 / ARM64 実機）を未署名 MSI に対して実施
  → 署名ジョブ（GitHub Environment の承認ゲート = 人間の承認）
      · Workload Identity Federation で GCP の短命トークンを取得（長期秘密なし）
      · jsign が GCP KMS の鍵で MSI を署名（--tsaurl でタイムスタンプ付与）
  → 署名済み MSI を取得
  → 署名済み MSI に対して Full E2E を再実行 + 署名/Subject/タイムスタンプ検証
  → SHA256 を【署名済み MSI に対して】算出
  → GitHub Release を公開（署名済み MSI + その .sha256 のみ）
  → 公開後検証ジョブ（公開資産を取り直して署名・Subject・.sha256 を突合、証跡記録）
```

判断の内訳:

- **機能 E2E は未署名 MSI に対して行う**。署名ジョブは GitHub Environment の承認ゲート（人間の承認）を挟むため、E2E の前段に置くと「承認待ちで CI が止まったまま、機能の破損がまだ判明していない」状態が生まれる。壊れた MSI に人間が承認印を押す事故を避けるため、**機能検証を承認より前に置く**。（(A) SignPath なら承認は SignPath.io 側で行われるが、(C2) では **GitHub Environments の保護ルール**で承認を実現する——決定 1 の受け入れリスク 1 の緩和策の 1 つ）
- **署名済み MSI に対しても、フル E2E を再実行する**（**ペルソナレビューを受けて初版から変更**。鈴木・クリスが独立に同じ指摘）。
  - 初版は「署名検証 + 起動スモークのみ。フル E2E の再実行は署名が機能を壊す経路が限られており、コストに見合わない」と書いた。**これは根拠のない仮説であり、実装も見ずに書いた**:
    - 決定 2 の deep signing は **MSI の展開・内部バイナリの署名・再パッケージ**を伴う。影響半径は「限られて」いない（クリス）
    - 現行の `installer/e2e/Invoke-YaguraInstallerE2E.ps1` は **MSI のパスと出力先を引数に取るだけの再利用可能なスクリプト**である。署名済みへの再実行は「ジョブを複製して引数を差し替える」だけで済む（鈴木）。無料ランナー・約 30 分・年数回のコストであり、「コストに見合わない」は定量的に成立しない
  - したがって **出荷するもの（署名済み MSI）そのものに対して、x64・ARM64 とも Full E2E を通す**。「テストしたものと出荷するものが違う」状態を作らない
- **SHA256 は必ず署名済み MSI に対して算出する**。署名はファイルを変更するため、未署名 MSI のチェックサムを公開すると利用者の検証が必ず失敗する。現行 `release.yml` はビルドジョブ内でチェックサムを算出しており（141-143 行）、**この移設は必須の変更**である
- **公開後検証ジョブを設ける**（鈴木の指摘。移設漏れの検知手段が皆無だったため）。`create-release` の完了後に、**公開された Release 資産を `gh release download` で取り直し**、①Authenticode 署名が有効であること、②署名者の Subject が期待値と一致すること、③`.sha256` が実ファイルと突合すること、を検証して失敗させる。あわせて**証明書の thumbprint と署名日時をリリースの証跡として記録する**（遡及失効が起きた際に影響範囲を告知するには、どのリリースがどの証明書で署名されたかの記録が要る——決定 8）
- **未署名 MSI を GitHub Release へ公開しない**。署名要求が失敗・却下された場合はリリースを中断する（署名済みと未署名が混在する Release を作らない）。**ただし「署名基盤が落ちている最中の緊急リリース」だけは例外とする**——決定 8 の退避弁を参照
- **全アーキ（x64・ARM64）が揃って署名されない限り Release を公開しない**（鈴木の指摘）。片方のアーキだけ署名成功・他方が失敗した状態で公開すると、「同じバージョンなのに一方だけ署名あり」という利用者を混乱させる資産構成になる。アーキ単位ではなくリリース単位で all-or-nothing にする
- **署名フローが途中で止まった場合の復旧手順を決めておく**（鈴木・リサの指摘。初版・改訂版とも、この復旧経路が設計されていなかった穴）。承認ゲートで長時間放置された・署名ジョブがタイムアウトした・承認者が不在——これらで「タグは push 済みだが Release が未作成」の状態が生じうる。**同一 run の re-run で回復できるか、artifact の再利用が効くか、再ビルドが要るか**（公開済みタグの再 push 禁止＝conventions.md との整合を含む）を **委任事項 11** で確定する
- **署名ジョブは、署名の資格情報が利用できない実行では skip する**（リサ・田中の指摘）。`release.yml` は `pull_request`（自ファイルの変更時）でも発火し、fork からの PR にも secrets は渡らない。署名ジョブをイベント分岐なしに直列挿入すると、**リリースワークフロー自体を修正する PR の CI が必ず落ちる**。skip 時は「署名なしのビルド・E2E の確認まで」を行い、Release は作らない（`create-release` の既存の if 条件と同じ思想）。**これは実装の詳細ではなく設計要件として ADR に置く**。**外部貢献者は自分の fork で「署名なしビルド + フル E2E」まで再現できる**（署名が必須ステップになって fork のビルドが通らなくなる設計にはしない。リサ）
- **署名アクション/ツールはコミット SHA でピン留めする**（田中。`cla.yml` に既存の前例がある）。署名を担う外部アクションをタグ参照にすると、タグの差し替えが署名工程への任意コード注入経路になる

### 決定 4: 署名は全アーキ（x64・ARM64）の成果物に等しく適用する

ADR-0009 委任事項 8 の「導入時は全アーキ成果物への適用を設計に含める」に従う。ARM64 の検証水準が「試験的」であること（ADR-0009 決定 2）と、**署名の有無は別の軸**である。試験的だから署名しない、という扱いはしない——署名は「誰が作ったか」の証明であって「どれだけテストしたか」の証明ではなく、両者を混同すると利用者に誤ったメッセージを送る。

### 決定 5: コード署名ポリシーを公開し、入口文書に検証手順を書く

(C2) では SignPath terms のような外部から課される掲示義務はないが、**透明性のためにコード署名ポリシーを自発的に公開する**（ADR-0005 の OSS 体裁の思想と一致する）。次を新設・更新する。

- **`docs/code-signing-policy.md`（新設）**: 見出しに「コード署名ポリシー」を用いる。記載事項:
  - **署名の仕組みの開示**: 証明書は個人（YANAI Taketo）名義の Sectigo OV 証明書であること、秘密鍵は Google Cloud KMS の HSM にあり手元にも CI にも存在しないこと、署名は GitHub Actions から Workload Identity Federation 経由で行われること
  - **同一証明書を Open DUMP Viewer と共用する事実と、その限界を正確に開示する**（決定 1 の受け入れリスク 3。田中・クリスの指摘）。「ODV の CI を掌握した攻撃者は、Yagura の正規署名と**利用者からは区別できない**署名を作れる」「リポジトリ別サービスアカウント + Cloud Audit Logs は**事後の検知・帰属**であって予防ではない」ことを、予防と誤読されない書き方で明記する。利用者向けに 1 行の要約も置く（佐藤の指摘）
  - チームの役割（Authors / Reviewers / Approvers）とそのメンバー。**実態を偽らずに書く**（決定 6）
  - プライバシーポリシー（後述）
  - **署名者の期待値**（下記）
  - **社内説明用の日本語段落**（佐藤の指摘）。発行元が実名で表示されることについて、利用者が上司や社内セキュリティ担当へ**そのまま転送・提示できる**日本語の説明（「これは個人 OSS 開発者 YANAI Taketo が署名した正規のソフトである」旨と、真正性の確認方法）を必須で置く。実名表示は第三者団体名より社内説明のハードルが上がる（佐藤）ため、その手当てをポリシー側で用意する
- **README.md**: 「コード署名ポリシー」への導線をインストール節に置く。**現在の README はインストール節で SmartScreen 警告に一切触れていない**（佐藤の指摘）。署名の有無にかかわらず、利用者が最初に遭遇する画面について書いていないのは不備であり、**決定 1 の裁定を待たずに直せる**。**署名導入後は「警告画面で発行元が実名で出たら、この Subject 文字列と一致することを確認して実行してよい／一致しなければ実行しない」という判断基準**を書く（佐藤——警告画面で実名が出たときに利用者が無準備で判断を迫られないようにする）
- **SECURITY.md**: 署名の検証手順を追加する。ADR-0006 基準 5 の「証拠」要件が求める内容である。あわせて決定 2 の未署名 DLL 混在（WDAC/AppLocker の発行元ルールが使えない）も記載する
- **CONTRIBUTING.md**: conventions.md は入口文書を「README・SECURITY・CONTRIBUTING」と定めており、**初版はここから CONTRIBUTING.md を落としていた**（リサの指摘。ADR 自身が規約違反を犯していた）。記載内容: ①CI 設定への変更は特に厳格にレビューされること、②コミット権を持つ場合の MFA 必須、③**ローカル/fork でのビルドは署名の影響を受けない**（署名は資格情報のある本番リリースでのみ走り、手元ビルドは従来どおり動く）という安心材料（リサ）
- **リリースノート**: 署名済みである旨と検証手順への導線を定型文に追加する（現行の「アーキテクチャ対応」表と同じ枠組みで運用する）

**検証手順には必ず「期待値」を書く**（佐藤の指摘。これは本 ADR で最も実質的な修正の 1 つである）:

> `Get-AuthenticodeSignature` が `Valid` を返しても、**攻撃者が自分の正規の証明書で署名した偽物も `Valid` を返す**。「署名が有効か」だけを確認する手順は、検証として機能しない。

したがって手順には**署名者の Subject（および証明書の thumbprint）の期待値そのもの**を明記し、「この文字列と一致することを確認せよ」という形にする。手順の掲載順は、実行のしやすさの順とする（佐藤）:

1. **GUI**（ファイルのプロパティ → デジタル署名タブ）——追加のツール導入が不要で、本番サーバでも実行できる
2. **PowerShell**（`Get-AuthenticodeSignature`）——標準搭載
3. **`signtool verify`**——**Windows SDK の導入が必要であり、本番サーバに入れられないことが多い**。最後に置く（初版は `signtool verify` を先頭に挙げていた。ADR-0006 の文言をそのまま引き写したためだが、利用者の実行可能性を考えていなかった）

**プライバシーポリシーの記載**: Yagura はテレメトリを持たず、利用者が設定した宛先以外へ情報を送信しない（SECURITY.md の既定構成の姿勢と一致する）。この事実を日本語で明記し、あわせて**逆引き（PTR）の DNS 問い合わせ（ADR-0007）とフォワーダ配布キットの取得は、利用者が明示的に有効化・実行する操作である**旨を補足する（利用者が「何が外部へ出るか」を具体的に把握できるようにする）。

### 決定 6: チームの役割を「単一メンテナ + 外部貢献は必ずレビュー」として明文化する

コード署名の責任分界を明示する。Yagura は現在オーナー 1 名の体制であり、これを偽らずそのまま宣言する。

| 役割 | 現在の担当 | 意味 |
|---|---|---|
| Authors | YANAI Taketo（リポジトリオーナー） | 追加レビューなしにソースを変更できる者 |
| Reviewers | YANAI Taketo | **外部からの PR** は必ずレビューを経てマージされる |
| Approvers | YANAI Taketo | 署名（GitHub Environment の承認）を承認する者 |

**初版の記述は事実に反していた**（田中が実測で指摘。**訂正する**）。初版は Reviewers の欄に「`main` は PR 必須 + 承認 1 件必須のブランチ保護下にある」と書いたが、`gh api repos/Yanai-Taketo/Yagura/branches/main/protection` で確認した実際の設定は次のとおりである（確認日 2026-07-14）:

```
enforce_admins: false
required_signatures: false
required_approving_review_count: 1
```

**`enforce_admins: false` である以上、ブランチ保護はオーナー（管理者）には適用されない。** オーナーは PR もレビューも経ずに `main` へ直接 push できる。これは conventions.md が明示的に選んだ運用である（「個人運用期のブランチ保護は `enforce_admins: false` + admin merge（GitHub は自己 approve 不可のため）。contributor 参加時に `true` へ格上げする」）。

**この事実を伏せたまま「承認 1 件必須のブランチ保護下にある」とコード署名ポリシーページに掲示すれば、利用者に対する虚偽の掲示になる**（(C2) には SignPath の terms のような掲示義務はないが、公開するポリシーが実態と食い違ってはならないという原則は変わらない）。ポリシーページには次のとおり実態を書く:

- 役割分離は**形式的にしか成立していない**（同一人物が全役割を兼ねる）
- **ブランチ保護はオーナーには適用されていない**（`enforce_admins: false`）。外部からの PR にはレビューが必須だが、オーナー自身の変更にはレビューがない
- コントリビュータが増えた時点で役割を分け、`enforce_admins: true` へ格上げする

**「掲載・MFA が要るのはコミット権を持つ者だけ」であることを明記する**（リサの指摘。参入障壁の線引き）。**単に PR を送るだけの外部貢献者は、ポリシーページへの氏名掲載も MFA も不要**である（レビューを経てマージされる側であり、署名権限を持たない）。掲載と MFA の対象は Authors/Approvers（コミット権・署名承認権を持つ者）に限る。

**全メンバーの MFA を自前の必須要件とする**。署名を発火させる権限が CI に載る (C2) では、GitHub アカウントの乗っ取りが署名につながりうる（決定 1 の受け入れリスク 1）。**オーナーの GitHub アカウントと Google Cloud アカウントで MFA を有効化していることを、証明書発注前に確認する**（**委任事項 B**）。将来コミット権を持つメンバーを迎える場合、MFA の有効化を権限付与の条件とする。

**なお (C1) SSL.com eSigner を採らなかった理由の 1 つがこの MFA 要件との矛盾である**（田中）——TOTP シークレットを GitHub Secrets に置く方式であり、2 要素が同じ金庫に入る。(C2) は Workload Identity Federation により長期の秘密を置かず、この矛盾を避ける。

### 決定 7: 製品メタデータを一元化し、CI で一致を強制する

署名対象バイナリには**製品名・製品バージョンのメタデータが正しく設定されている**べきである。署名は「発行元」を保証するが、利用者がプロパティ画面で見る製品名・バージョンが空や既定値のままでは、署名の価値が削がれる（「発行元は検証できるのに、製品名が不明」という不整合を避ける）。

現状、`Directory.Build.props` にも各 `.csproj` にも `Product` / `Company` / `Version` / `Copyright` が**一切設定されていない**。結果として生成されるアセンブリは SDK 既定値（`AssemblyCompany` = アセンブリ名、`AssemblyFileVersion` = `1.0.0.0`）を持ち、バージョンの正は `installer/Package.wxs` の `Version` **のみ**である。署名の導入と独立に、この状態は是正すべきである。

- `Directory.Build.props` に製品メタデータを追加する（`Product` = `Yagura`、`Company` = `Yagura Project`（MSI の `Manufacturer` と一致させる）、`Copyright`（NOTICE と一致させる）、`Version` 系）
- **バージョンの単一の正（single source of truth）を決め、`Package.wxs` の `Version` とアセンブリのバージョンが食い違ったら CI で失敗させる**。現行 `release.yml` は「git タグ vs `Package.wxs`」の一致検証を既に持っている（74-99 行）。この検証を「タグ vs `Package.wxs` vs アセンブリ」の三者一致へ拡張する
- 正をどちらのファイルに置くか（`Directory.Build.props` から `Package.wxs` へ流し込むか、その逆か）は実装上の選択であり、**委任事項 1** とする
- **どこを直せばバージョンが変わるのかを、外部貢献者が読んで分かる形にする**（リサ）。正を conventions.md の「リリース」節に 1 行で明記し、不一致を検知する CI のエラーメッセージには**直すべきファイル名を含める**（現行 `release.yml` 95 行の「chore(release) PR で Package.wxs を先に更新すること」が良い手本になっている）

### 決定 8: 署名基盤が使えなくなった場合の退避経路をあらかじめ決めておく

**この決定は決定 1 の裁定によらず成立する**（どちらを選んでも、第三者に依存することに変わりはない）。切れたときの手順を決めずに依存しない。

**共通の設計要件**: **署名基盤への依存を `release.yml` の 1 ジョブに閉じ込める**（委任事項 6）。署名の実行主体を差し替えても、決定 3 のワークフロー構造（未署名で機能検証 → 署名 → 署名済みでフル E2E → 公開 → 公開後検証）が維持できるようにする。これは (A) と (C) の相互乗り換えを可能にするための保険でもある。

#### 退避弁: 署名基盤が落ちている最中の緊急リリース（鈴木の指摘。初版の最大の穴）

**初版の決定 8 は「恒久的な乗り換え手順」しか書いておらず、「署名基盤が落ちている、まさにその最中に Critical な脆弱性修正を出す」手順が存在しなかった。**

これを決定 3 の fail-closed（署名できなければリリース中断）と SECURITY.md（バックポートなし = 利用者は最新版へ上がるしかない）と突き合わせると、**署名基盤の可用性が、脆弱性修正の配布の単一障害点になる**。署名という「安全のための仕組み」が、安全のための修正の配布を止める——本末転倒である。

したがって**平時のうちに退避弁を明文化しておく**（事故の最中に設計しない）:

- **発動条件は「署名基盤の連鎖のいずれかの断点で、Critical 修正の署名済み公開が達成できない状態」**とする（鈴木の指摘で拡張）。署名基盤の障害（Sectigo/GCP KMS/TSA のいずれか）だけでなく、**承認者が長時間不在で承認ゲートを通せない場合**も含める（初版の「署名基盤が利用不能」という文言では承認者不在が漏れていた）
- **Critical な脆弱性修正に限り、上記の場合に「署名なしリリース」を許可する**
- その場合、**必ず**次を伴う: ①リリースノートの冒頭に「このリリースは署名工程を通せなかったため未署名である」ことを明示、②`.sha256` の添付（従来どおり）、③SECURITY.md / Advisory での明示、④復旧後に**同一バージョンを署名付きで再リリースしない**（公開済みタグの再 push 禁止——conventions.md——と衝突するため、次のパッチバージョンで署名を回復する）
- **この退避弁を使う判断はオーナーのみが行う**。CI が自動的に署名を飛ばすことは許さない（`workflow_dispatch` の明示的な入力とする）

**採用した (C2) の退避・運用経路**:

1. **署名資格情報が漏洩した疑いが生じた場合**: 直ちに **WIF プールバインディングと IAM を失効・ローテートする**（田中の指摘で訂正——WIF 前提では「サービスアカウント鍵をローテート」は矛盾する。長期鍵を置かないのが WIF の要点であり、失効すべきは WIF の信頼バインディングと IAM 付与である）。**Cloud Audit Logs で不正な署名操作の有無を確認する**（そのためにも監査ログの有効化＝委任事項 4 が前提になる）。証明書自体の失効が必要なら Sectigo に要求する。**失効すると、その証明書で署名した既公開バイナリすべてが影響を受ける**（Yagura だけでなく ODV も。決定 1 の受け入れリスク 5）。**タイムスタンプ（決定 9）が失効前の署名を保護する**ため、被害範囲は「失効時点以降に発行された署名」に限定されうる
2. **異常な署名操作を検知・アラートする**（田中の指摘。事後追跡を「気づける」ものにする）。**Cloud Audit Logs 上で、想定外のタイミング・想定外の principal・想定外のリポジトリからの KMS 署名操作にアラートを設定する**（委任事項 13）。これがないと決定 1 の「事後の検知・帰属」は「後から掘れば分かる」に留まり、能動的な検知にならない
3. **費用の支払いが継続できなくなった場合**: Yagura のみ (A) SignPath への移行を検討する（Yagura は OSS なので (A) の適格要件を満たす。ODV は満たさないため別途の手当てが要る）。決定 3 のワークフローは署名ジョブを 1 つに閉じ込めてあるため、署名基盤の差し替えで対応できる（委任事項 6）
4. **Sectigo が個人発行を停止した場合**: 別 CA へ移行する（決定 1「委任 A が不可だった場合の分岐」と同じ選択肢）
5. **証明書の有効期限切れ**: タイムスタンプ付き署名は期限切れに耐えるため、既公開バイナリは影響を受けない。次回リリースまでに証明書を更新する

### 決定 9: 署名には必ずタイムスタンプ（RFC 3161 TSA）を付与し、その存在を CI で検証する

**初版はこの決定を欠いていた**（田中の指摘）。ADR-0006 基準 5 は「Authenticode コード署名（**タイムスタンプ付き**）」を v1.0 の要件として明示しているのに、初版の ADR-0014 には TSA の決定も、検証ゲートでのタイムスタンプ存在確認も書かれていなかった。**要件を閉じるはずの ADR が、要件の半分を落としていた。**

- **すべての署名に RFC 3161 タイムスタンプを付与する**。タイムスタンプがなければ、**証明書の有効期限が切れた時点で既公開バイナリの署名が無効になる**（利用者の手元で、ある日突然「発行元不明」に戻る）
- **決定 3 の公開後検証ジョブで、タイムスタンプの存在と時刻を検証する**（署名が有効なだけでなく、タイムスタンプが付いていることを確認する）
- **これは (C2) では特に重要である**（田中）: タイムスタンプは、**証明書が失効したときの被害範囲を限定する唯一の制御**である（失効前に発行されたタイムスタンプ付き署名は、失効の影響を受けない扱いになりうる）。1 枚の証明書を Yagura と ODV で共用する構成では、失効が両プロジェクトに波及するため、この制御の有無が波及範囲を左右する
- **TSA を明示指定し、フォールバック TSA を用意する**（`--tsaurl <primary> <secondary> --tsmode RFC3161`。jsign は複数 TSA へのフォールバックとリトライに対応する）。**単一の TSA は新たな単一障害点になる**（鈴木の指摘）——主 TSA（例 Sectigo）が応答しないとき副 TSA（別提供者）に切り替える。使用する TSA 群とリトライ設定は **委任事項 7**
- **TSA が全滅した場合は fail-closed とする**（鈴木）。タイムスタンプなしの署名を公開しない（決定 3 の「未署名を公開しない」と同じ思想——タイムスタンプは失効時の唯一の保護であり、欠けた署名を出すのは署名の意味を損なう）。TSA 障害が長引く緊急時は決定 8 の退避弁に合流する

## 帰結

### 良くなること

- 利用者が MSI を実行したときの SmartScreen 警告が軽減され、発行元が「不明」ではなく**実名（YANAI Taketo）**になる。ADR-0001 が掲げる「Windows 管理者が既に持つスキルだけで導入できる」という思想において、**警告を無視する手順を利用者に教えなくて済む**ことの価値は大きい（「警告を無視して進む」習慣を育てない、という `configuration.md` §6 の自己署名証明書に関する判断と同じ原則）
- リリース資産の真正性が、配布経路への信頼に依存せず検証可能になる
- **証明書 1 枚で Yagura と Open DUMP Viewer の両方を署名できる**。ODV も同時に「発行元不明」問題を解決できる
- **審査に落ちるリスクがない**。star 0 の新しいプロジェクトであることが一切不利にならない
- **鍵は Google Cloud KMS の HSM にあり、手元にも CI にも秘密鍵が存在しない**。WIF により長期の秘密を GitHub Secrets に置かない
- 製品メタデータが一元化され、git タグ・MSI・アセンブリのバージョン不整合が CI で検知される（決定 7。**現状は三者がバラバラで、アセンブリは `1.0.0.0` のまま**）
- ADR-0006 基準 5 と ADR-0009 委任事項 8、Issue #264 が閉じる

### 悪くなること・トレードオフ

- **年間の継続費用**（Sectigo 約 $220/年 + GCP KMS 月額。委任事項 A で総額確認）。個人が無報酬で運営する OSS に、リリースを続ける限り毎年かかるコストを載せる
- **実名・所在地が証明書に載り、署名済みバイナリを通じて恒久的に公開される**（取り消せない）
- CI のワークフローが複雑になる（署名ジョブ・GCP 連携・GitHub Environment 承認・公開後検証）
- **SHA256 の算出位置を移設しなければならない**（署名はファイルを変える）。移設漏れは「公開したチェックサムが必ず一致しない」という形で利用者に直撃する
- 署名基盤（Sectigo・GCP）という第三者依存が増える。可用性・継続性は Yagura の管理外にある
- 構築・保守を自分で負う（(C2) には公式 GitHub Action がなく jsign で自作する）

### 受け入れるリスク（決定 1 で明示的に引き受けた）

- **署名を発火させる権限が CI に載る。** リポジトリ/CI を掌握した攻撃者は実名で任意署名できる。緩和策（WIF・最小権限 IAM・Cloud Audit Logs・GitHub Environment 承認・タグ限定・署名ジョブ分離・リポジトリ別サービスアカウント）を設計要件とするが、GitHub アカウントの乗っ取りが署名につながる経路は残る（(A) の独立した認証境界の**完全な代替ではない**ことを承知で引き受ける）
- **Yagura の署名の真正性が ODV の CI の安全性にも従属する**（1 枚共用の帰結）。この事実と緩和策をコード署名ポリシーに明記する
- **失効は Yagura・ODV の両方に波及する**。タイムスタンプ（決定 9）が失効前の署名を保護する

### 約束できないこと

**SmartScreen の警告が「消える」とは約束できない。** SmartScreen の評価は証明書ごとの評判に基づき、**新規発行の OV 証明書は評判ゼロから始まる**。さらに **EV 証明書ですら 2024 年 3 月の Windows 仕様変更で SmartScreen を即時バイパスできなくなった**（zenn 記事で確認）——OV でも EV でも、評判は時間をかけて積むしかない点は変わらない。したがって:

- 入口文書に「SmartScreen 警告が出なくなる」とは書かない。「署名され、発行元が検証可能である」という事実のみを書く
- **署名済み MSI での SmartScreen の実挙動を lab で検証し、その結果を記録する**（conventions.md「実環境依存の機能は lab 検証を受け入れ条件に含める」）。**委任事項 9**

## 改訂履歴

### 1. 2026-07-24（amendment）: 委任事項 9（署名済み MSI の SmartScreen 実挙動）の lab 検証結果

**契機**: Issue #264 / 委任事項 9。署名済み MSI（訂正版証明書チェーンで署名・Actions run **30017908696** の `release-msi-signed-x64`）を実機で MOTW 付与のうえ実行し、SmartScreen の見え方を記録した。

**環境**: `WIN-DV7G9OMOF4U`（Windows 11 系デスクトップ・`10.0.26100`）/ 2026-07-24。両 MSI に `Zone.Identifier`（ZoneId=3 = インターネット由来）を付与して実行。

**署名の事前確認（プログラム検証）**: `Get-AuthenticodeSignature` で **`Status=Valid`**・**`Subject=CN=柳井建人, O=柳井建人, S=Yamaguchi, C=JP`**・拇印 **`36257D1DC06F7CDECA61B9727A88320DA24D27B6`**・**RFC 3161 タイムスタンプ有**（Sectigo Public Time Stamping Signer R37）を確認。証明書・署名は正常。

**SmartScreen ダイアログ実挙動**（スクリーンショットは Issue #264 に添付）:

| | 署名済み MSI | 未署名 MSI（v0.4.0 Release） |
|---|---|---|
| 見出し | 「Windows によって PC が保護されました」（`認識されないアプリ` = 評判蓄積前。**署名しても評判ゼロの間は警告自体は出る**——委任 9 の想定どおり） | 同左 |
| **発行元** | **`JP, Yamaguchi, ???縺?I, ???縺?I`**（証明書 subject の DN が表示される＝**署名が認識されている**。未署名との判別点は成立） | **`不明な発行元`** |

**判定**: 委任 9 の判定点（**発行元表示の有無で署名済み／未署名を判別できるか**）は**成立**——署名済みは発行元フィールドが populated（`JP, Yamaguchi` = C=JP・S=Yamaguchi）で、未署名の「不明な発行元」と明確に区別できる。

**想定外の発見**: 証明書の **日本語部分（O=`柳井建人`・CN=`柳井建人`）が SmartScreen ダイアログ上で文字化けする**（`???縺?I` と表示。`縺` は UTF-8 バイト列を CP932 として解釈したときの典型的な化け方）。`Get-AuthenticodeSignature` では `柳井建人` と正しく取得できているため**証明書自体は正常**であり、これは **SmartScreen ダイアログが証明書 subject の非 ASCII 文字を誤エンコードで描画している**現象である。結果として、利用者が見るダイアログでは発行元が「**柳井建人**」と読める形では出ない（ASCII の `JP, Yamaguchi` は正常表示）。**UAC 昇格ダイアログの発行元表示は本検証では未採取**（SmartScreen で「実行しない」を選び導入自体は行わなかったため）。
- **原因の実体確認（2026-07-24）**: 証明書 subject の `柳井建人`（O・CN）は **UTF8String（ASN.1 tag `0x0C`）** で符号化されている（`SignerCertificate.SubjectName.RawData` を実測——`柳井建人` の UTF-8 バイト列 `E6 9F B3 …` が UTF8String タグ + 長さ 0x0C で格納。BMPString〈tag 0x1E〉ではない）。`Get-AuthenticodeSignature`（.NET）は正しく `柳井建人` と復号するが、**日本語 Windows の SmartScreen ダイアログはこの UTF8String を ANSI コードページ（CP932）経路で描画するため化ける**（`縺` の化け方が UTF-8→CP932 誤解釈の典型）。**Yagura のコード側では修正できない**（証明書 subject の中身と SmartScreen の描画の問題であり、いずれも製品の管轄外）。
- **裁定候補**: (i) 現状受容（判別点は満たすため v0.5.0 は進める）／**(ii) 証明書 subject をローマ字表記（例 `CN=YANAI Taketo`・`O=YANAI Taketo`）で再発行する——推奨**。日本語圏の発行者の多くがこの SmartScreen/UAC の非 ASCII 表示問題を避けるためローマ字で証明書を取得しており、個人 IV 証明書のため検証名のローマ字表記は一般に可能（Sectigo 手続き＝委任 A で指定）。ローマ字なら SmartScreen・UAC とも読める発行元名になる。／(iii) SmartScreen 側の既知の非 ASCII 表示制限として文書化のみ（署名の信頼シグナルを損なうため非推奨）。代替として「CA に BMPString 符号化を依頼」は、現代の CA が RFC 5280 準拠で UTF8String を既定とし符号化をこちらで選べず全ダイアログでの表示保証もないため確実でない。README の SmartScreen 節（署名後に更新予定。PR #412）で「発行元が『柳井建人』と表示される」旨を書く場合は、この文字化けと齟齬するため**表現に注意**が要る（現行 main の README は未署名前提の記述のため直接の矛盾はない）。

**オーナー裁定（2026-07-24）: (i) 現状受容で確定**。理由: 本証明書は**マイナンバーカードによる本人確認（IV）で取得しており、検証名がカード記載の漢字氏名になる。マイナンバーカードにローマ字表記が無いため、ローマ字 subject の証明書は本人確認経路の制約上そもそも発行できない**（(ii) は選択肢として成立しない——将来「ローマ字にすればよい」の蒸し返しを防ぐため理由を明記する）。したがって v0.5.0 以降も漢字 subject のまま進める。**緩和と正しい発行者名の確認手段**: ①SmartScreen の「認識されないアプリ」警告自体は評判蓄積で解消していく（決定 1・本 amendment 上表の想定どおり——警告ダイアログが出なくなれば化けた発行元行も表に出なくなる）。②署名の真正性と正しい発行者名（`柳井建人`）は `Get-AuthenticodeSignature` およびファイルのプロパティ→デジタル署名の詳細で正しく確認できる（SmartScreen とは別の Unicode 対応経路。本 lab で `Get-AuthenticodeSignature` = `CN=柳井建人`・`Status=Valid` を実測）。README／SECURITY.md には「SmartScreen ダイアログ上の発行者名が一部の日本語 Windows で文字化けし得るが、署名は有効であり正しい発行者はファイルのデジタル署名詳細で確認できる」旨を注記するのが望ましい（利用者の不信を招かないため。文言化は別 PR）。
