## 文脈と課題

v0.1 の管理 UI(TCP 8515)は loopback 束縛 + 認証なしを既定とする(ADR-0004 決定 2)。この決定自体が、将来の拡張を条件文であらかじめ予告していた:

> リモートからの管理操作は、認証 + HTTPS(決定 3)を有効化した場合に開放される

しかし、この条件をどう満たすか(認証方式・監査への反映・段階導入・受け入れ基準)はこれまで具体化されていなかった。この空白が、直近で 2 つの形で顕在化した:

1. **PR #167(Issue #130: フォワーダ MSI アップロード機能)の見送り**。ペルソナレビューで田中(セキュリティ・監査担当)・クリス(批判的アーキテクト)がそれぞれ独立に「管理リスナが無認証 loopback である現状では、身元保証が構造的に成立しない」(監査の「誰が」欄が常に `127.0.0.1` にしかならない)「低頻度操作の利便性が、サービスアカウントへの恒常的な書き込み権限付与という常在コストに見合わない」と指摘し、オーナーは 2026-07-09 に本改訂の見送り・クローズを決定した。**再評価トリガとして「管理 UI に身元保証を伴う認証(AD 認証 opt-in 等)が導入された時点で、supersession として再起案する」ことを明記**し、5 ペルソナの指摘(受け入れ基準・操作者特定・拒否監査・SignalR 転送制約・ステージング後始末等)を再起案時の出発点として保存した。
2. **日常運用での不便**。RDP でしかサーバへ到達できない構成(踏み台経由の RDP・Server Core 検証機)では、管理 UI を開くこと自体が手間になる。佐藤(ひとり情シス)ペルソナが PR #167 レビューで述べた「RDP しなくて済むのは素直に嬉しい」という声は、MSI アップロード個別機能への評価であると同時に、管理面全体がリモートから触れないことへの一般的な不満の表れでもある。

Issue #172 でオーナーは、この 2 つを同時に解く設計として次の方向性を決定済みである(2026-07-09):

1. 主目的は**リモート管理の解禁**であり、認証はその前提条件(手段)である
2. 認証方式は **Windows 統合認証(Negotiate: Kerberos/NTLM)と、アプリ独自 ID/パスワードの両方を opt-in で提供**する(いずれかの選択を強制しない)
3. 適用範囲は**管理 UI(8515)を先行**させる。閲覧 UI(8514)への拡張可否は本 ADR で両論併記し、ペルソナレビューの論点とする

本 ADR はこの方向性を具体化する。実装の詳細(監査イベント ID の最終確定・パスワードハッシュの実装・icacls 相当の ACL 実機検証)は、ADR-0008 が踏んだ手順に倣い、委任事項として実装 PR + `security.md`/`configuration.md` へ送る——**本 PR(ADR のみ)は `docs/design/` の同時更新を含めない**。理由: PR #167 のペルソナレビュー(リサ)が、ADR 単独更新と設計書更新のタイミングがずれると「常に現在形」であるはずの設計書が一時的に矛盾した記述を残す、と指摘した教訓を踏まえ、**そのズレが生じることをここで先に明記し、実装 PR での追従を約束する**(暗黙の先送りにしない)。

## ライブ検証記録(2026-07-09。conventions.md の技術的主張の検証原則に従う)

推測を避けるため、Windows 統合認証(Negotiate)の .NET 10 / Kestrel / Blazor Interactive Server での成立可否を公式ドキュメントで当日確認した。

### 検証 1: Kestrel での Negotiate 認証(.NET 10)

[Microsoft Learn "Configure Windows Authentication in ASP.NET Core"](https://learn.microsoft.com/en-us/aspnet/core/security/authentication/windowsauth?view=aspnetcore-10.0)(`aspnetcore-10.0` モニカー、最終更新 2026-04-30・確認日 2026-07-09)より:

- `Microsoft.AspNetCore.Authentication.Negotiate` NuGet パッケージ(NuGet.org 確認: 最新安定版 **10.0.9**、確認日 2026-07-09)を Kestrel に組み込むと、**Windows・Linux・macOS で Negotiate(Kerberos/NTLM)認証**が有効になる。構成は `AddAuthentication(NegotiateDefaults.AuthenticationScheme).AddNegotiate()` の 1 行
- **Kestrel は `WWW-Authenticate: Negotiate` ヘッダーのみを返す**(NTLM を直接のスキーム名として指定することはできない)。ただし **NTLM は Negotiate に包まれて透過的にサポートされる**——Kestrel 上で Kerberos と NTLM のどちらが使われたかは、ヘッダーの Base64 デコード結果(`HTTP` = Kerberos、`NTLM` = NTLM)でしか判別できない
- **プロキシ経由では使用禁止**(1:1 の接続アフィニティ(持続接続)をプロキシが維持しない限り、認証情報が接続をまたいで誤って永続化されうる警告が明記されている)。これは ADR-0004 決定 2 が既に「リバースプロキシ前置構成は既定ではサポートしない」と定めていることと整合する——**本 ADR の認証機構が入っても、プロキシ前置を推奨構成にはしない**
- **HTTP/2 非対応**: 認証ハンドシェイク自体は HTTP/1.1 へのダウングレードを要する(認証後の通常通信は HTTP/2 に復帰できる)。管理リスナの Kestrel 設定が HTTP/1.1 を許容していることを実装 PR で確認する必要がある(Blazor Interactive Server の SignalR circuit は WebSocket 昇格を伴うため、実務上の追加制約になる可能性は低いと見るが未検証)
- **SPN(Service Principal Name)は「サービスを実行するユーザーアカウント」に登録する**(マシンアカウントではない。User Mode 認証のため)。例: `setspn -S HTTP/myservername.mydomain.com myuser`。**この一文が Yagura にとって固有の宿題を生む**: Yagura のサービスは仮想サービスアカウント(`NT SERVICE\Yagura`。ADR-0004 決定 4)で動作するが、仮想サービスアカウントに対する `setspn` の可否・挙動は一般的なドメインユーザーアカウントを前提にした公式手順が想定していない構成であり、**推測で「動く」と書かない**(委任事項 3 の実機検証事項とする——検証 3 参照)

### 検証 2: Blazor Web App(Interactive Server)と Windows 認証の併用

[Microsoft Learn "Secure an ASP.NET Core Blazor Web App with Windows Authentication"](https://learn.microsoft.com/en-us/aspnet/core/blazor/security/blazor-web-app-with-windows-authentication?view=aspnetcore-10.0)(`aspnetcore-10.0` モニカー、更新日 2025-11-11・確認日 2026-07-09)より:

- 公式サンプル `BlazorWebAppWinAuthServer`(.NET 9 以降向け)が、**Interactive Server render mode(全画面対話化)+ Windows 統合認証**の組み合わせをそのまま実証している——ADR-0003 決定 1 の「全画面 Interactive Server 単一モード」との併用に構造的な障害はない
- 構成は Kestrel と同じ `AddNegotiate()` に加え、`AddAuthorization` の `FallbackPolicy`(既定で認証必須)、`AddCascadingAuthenticationState()`(認証状態を コンポーネントツリーへカスケード)を追加する
- **Windows SID ベースの認可ポリシー例が明記されている**: well-known SID(`S-1-5-113` = ローカルアカウントによるネットワークサインイン)を `RequireClaim("...claims/groupsid", "S-1-5-113")` で判定するパターンが公式に示されている——同じ手法で **`S-1-5-32-544`(`BUILTIN\Administrators`)を判定に使える**ことが確認できる(決定 5 で採用)

### 検証 3: circuit(SignalR 接続)確立後の認証状態の扱い

[Microsoft Learn "ASP.NET Core server-side and Blazor Web App additional security scenarios"](https://learn.microsoft.com/en-us/aspnet/core/blazor/security/additional-scenarios?view=aspnetcore-10.0)(`aspnetcore-10.0` モニカー、更新日 2026-05-29・確認日 2026-07-09)より:

- 公式ドキュメントは「**Circuit handler to capture users for custom services**」という節で、`CircuitHandler.OnConnectionUpAsync`(circuit の再接続のたびに呼ばれ、接続の生存期間の "user" を設定する)と `OnCircuitOpenedAsync` + `AuthenticationStateProvider.AuthenticationStateChanged` ハンドラ(認証状態の変化をカスタムサービスへ反映する)を組み合わせるパターンを明示している
- この構造が要る理由(dotnet/aspnetcore issue #15180・#12051・#44820 で報告されている既知の課題。GitHub 上の一次情報であり公式ドキュメントの節そのものではないため参考情報として区別する): Blazor Server 既定の `FixedAuthenticationStateProvider` は**認証状態を circuit の生存期間中固定**として扱う設計だが、`WindowsPrincipal` は元の HTTP リクエストに紐づく OS ハンドルであり、リクエスト完了後もそのまま使い続けられる保証がない。Negotiate は接続(TCP コネクション)単位で認証情報を持続させる設計(検証 1 の「プロキシ経由禁止」の警告と同根)であり、SignalR の再接続で新しい接続に切り替わるたびに認証状態を明示的に汲み直す必要がある
- **本 ADR にとっての意味**: `security.md` §2.3 が既に一般則として定めている「**書き込み・管理操作は失効を即時反映する。操作のたびに現在の認証状態で認可する**」という設計は、Windows 統合認証の circuit 特性とも整合する——ただし実装は「接続確立時に固定された `FixedAuthenticationStateProvider` の初期値」に頼ってはならず、公式パターンの `OnConnectionUpAsync` による明示的な汲み直しを実装要件として明記する必要がある(決定 2 で反映)

### 検証 4: 非ドメイン機・NTLM フォールバック

検証 1 の「Kerberos vs NTLM」節の記述から導かれる帰結: `AddNegotiate()` は Windows 上で **Kerberos を優先して試行し、成立しない場合(SPN 未登録・非ドメイン参加機・ローカルアカウントでのアクセス等)は NTLM に自動的にフォールバックする**——これは同一の 1 コード経路で起こり、アプリ側の追加分岐を要さない。したがって:

- **非ドメイン参加機(ワークグループ構成)でも、ローカル Windows アカウントに対する NTLM 認証がそのまま機能する**(Windows OS の基本機能であり Kerberos のようにドメイン参加を前提にしない)
- **ホスト名でのリモートアクセス時に SPN が未登録・不正な場合、Kerberos SSO(パスワード再入力なしの透過認証)は成立せず NTLM へ自動的に後退する**——これは失敗ではなく既定の縮退であり、資格情報の入力ダイアログが出る形で利用者に見える。IP アドレスでのアクセスは SPN の一致対象(ホスト名ベース)を持たないため、実務上ほぼ常に NTLM 経路になる
- **NTLM 縮退の含意はリスクとして別途評価する**(検証だけでは「機能する」ことしか言えない): NTLM にはリレー攻撃・pass-the-hash・相互認証の欠如という既知の弱点があり、Microsoft 自身が NTLM の段階的廃止を進めている。仮想サービスアカウントでの SPN 成立可否が未検証(検証 1)である以上、**Phase 2 時点の Windows 統合認証は「NTLM が事実上の常用経路」になる可能性が高い**——この明示的なリスク受容は「帰結」と決定 8 に記す(クリスの PR #173 レビュー指摘)

## 検討した選択肢

### 認証方式

- (a) Windows 統合認証(Negotiate)のみ: AD 環境と親和性が高く CLAUDE.md 方針 7(AD 認証は当初から想定)と一致するが、**AD を持たない小規模環境・ワークグループ構成を締め出す**(検証 4 のとおり NTLM フォールバックはあるが、ローカルアカウントの棚卸し・パスワードポリシーは OS 管理に依存し、Yagura 導入のためだけにローカルアカウントを作る運用は「Windows 管理者の既存スキルに乗る」という原則(CLAUDE.md 冒頭)からはやや外れる)。加えて **AD 単独依存はブレークグラス(非常口)を持たない**——DC 障害時に RDP 経路(踏み台・VPN)も同じ AD に依存している構成では、管理面全体が AD 障害 1 点で不能になる(鈴木の PR #173 レビュー指摘)。却下(単独では不十分)
- (b) アプリ独自 ID/パスワードのみ: 環境を問わず動くが、AD 環境での「また別の資格情報」という重複投資になる(ADR-0004 検討した選択肢「管理用パスワード」が既に却下した理由と同じ)。却下(単独では不十分)
- **(c) 両方を opt-in で提供する(採用。Issue #172 のオーナー決定)**: AD/Windows 環境では統合認証でパスワード管理を増やさず、AD を持たない・単一ローカル運用の環境ではアプリ独自認証で自己完結する。**両方式の実装・テスト・脆弱性対応という恒常コストを 2 系統分負う**トレードオフは「帰結」に明記する。共存時の設計方針(優先順位・ロックアウトの相互作用・監査の命名空間)は実装詳細ではなく本決定の成立条件であるため、決定 3 で本文レベルの方針を固定する(クリスの PR #173 レビュー指摘を受けた昇格)

### アプリ独自認証の実装方式

- (a) ASP.NET Core Identity のフル導入: ロール・クレーム・アカウントロックアウト・2FA 等の機能を持つ成熟した実装だが、**EF Core ベースのユーザーストア(`Microsoft.AspNetCore.Identity.EntityFrameworkCore`)という新規 NuGet 依存を持ち込み**、CLAUDE.md 確定方針 1(DB provider 抽象——SQL Server/PostgreSQL/MySQL/SQLite)全系列にスキーマを持つ必要がある。Yagura が必要とするのは「閲覧/管理の 2 段」(security.md §3)のみであり、Identity が提供する機能の大半(メール確認・外部 IdP 連携・2FA 等)を使わない。**conventions.md の「依存を増やさない」判断規準**(L-5 の参照検査で `NetArchTest.Rules` を過剰と判断し標準リフレクションを採用した実例)と同じ理由で、規模に見合わない
- **(b) 最小実装(採用)**: 単一の管理者テーブル(ユーザー名・パスワードハッシュ・失敗試行カウンタ・ロックアウト期限)を既存のデータ provider 抽象に載せ、パスワードハッシュには ASP.NET Core が提供する `PasswordHasher<TUser>`(PBKDF2 実装)の利用を検討する——**この API が Yagura の参照する共有フレームワーク(`Microsoft.AspNetCore.App`)経由で追加の NuGet 参照なしに使えるかは未確認であり、実装 PR で通常のライブ検証手順(conventions.md)を踏んで確定する**(推測で「依存ゼロ」と断定しない)。ロール(閲覧/管理)は security.md §3 の 2 段モデルにそのまま載せる
- 決め手: 供給網統治(田中の PR #167 観点)を重視する本プロジェクトの一貫した判断規準に沿う。Identity のフル導入は将来ロール粒度が本格的に増える段階(security.md §3 が「独立利用者から 3 件以上」を再評価トリガとして明記済み)で再検討する余地を残す
- **最小実装の対価を明記する**(クリスの PR #173 レビュー指摘): Identity なら標準機能でカバーされる領域——ユーザー列挙耐性(存在しないユーザー名と誤パスワードで応答を区別しない)・分散ブルートフォース対策(単一送信元ロックアウトでは防げない低速試行)・ログイン POST の CSRF 対策・レート制限——を**自前で設計・実装・テストする責務**が生じる。これらは委任事項 4 に明示的に含め、実装 PR で手薄になることを防ぐ

### リモートバインド時の通信保護

- (a) 認証さえあれば平文 HTTP でも可: Negotiate のハンドシェイク自体は資格情報を平文で載せない(チャレンジ・レスポンス方式)が、**認証成功後の circuit 上のやり取り(設定変更の内容・監査記録の閲覧内容等)は平文で流れる**。アプリ独自 ID/パスワードは初回の資格情報そのものが平文で流れる。いずれも LAN 外からの盗聴・中間者攻撃に対して無防備。却下
- **(b) HTTPS 必須(採用)**: ADR-0004 決定 3 が既に「認証によるリモート管理の開放は、HTTPS の有効化を前提条件とする」と定めている。本 ADR はこれを管理リスナに対して具体化するのみで、新しい決定ではない

### 管理を許可する主体の判定(認可)

- **(a) 既定は `BUILTIN\Administrators`(well-known SID `S-1-5-32-544`)、AD グループ指定は opt-in の追加(採用)**: 検証 2 で確認した SID クレーム判定パターンをそのまま使える。「Windows 管理者が既に持つスキル(SQL Server・AD・Windows サービス運用)だけで導入・運用できる」という CLAUDE.md 冒頭の思想に最も一致し、AD が無い最小構成でも動く(ローカル Administrators は AD なしで既定に存在する)。AD 環境では security.md §3 が既に定める「AD グループへのマッピング」による拡張(または置換)を選べる
- (b) 個別ユーザーの列挙: 管理対象が増える。security.md §3 が既に「個別ユーザーの列挙は管理対象を増やすため既定にしない」と判断済み。却下
- (c) 認証済みユーザーを一律「管理」役割にする: 「認証されている」ことと「管理権限を持つ」ことを混同する設計であり、Windows 統合認証で言えば「ドメインの誰でも Yagura の設定を変更できる」に等しい。田中ペルソナの観点(権限最小化)に明確に反する。却下

### loopback アクセスの扱い(認証有効化後)

- (a) loopback は常に無認証(本 ADR の初稿案): `configuration.md` §6 が前提とする復旧経路(証明書事故時に loopback 経由でなら常に復旧操作ができる)を最も単純に守れるが、**同一ホスト上の「管理リスナへネットワーク到達できる任意のローカル実行主体」(対話的ログオンユーザーに限らず、低権限で動く別サービス・スケジュールタスク・侵害されたプロセスを含む——PR #167 クリスの脅威モデル指摘の表現を踏襲する)に対する操作者特定・アクセス制御が構造的に永遠に成立しない**。共有サーバ・多人数ログオン環境の利用者に選択肢を与えられない。撤回(ペルソナレビュー(田中・クリス)の指摘とオーナー決定 2026-07-09 による)
- **(b) 既定は loopback 無認証を維持し、opt-in で loopback にも認証を課せる(採用。2026-07-09 オーナー決定)**: 既定の無認証維持は、ロックアウト・DC 障害・証明書事故時の**単一依存に引きずられない最終復旧経路**として佐藤・鈴木の両ペルソナが最重要視した価値をそのまま守る。その上で、loopback 到達者の操作者特定・アクセス制御が必要な環境(共有端末・多人数ログオンサーバ・将来の Issue #130 再起案の前提条件——決定 9)向けに、**設定で loopback アクセスにも認証を課す opt-in** を提供する。opt-in を有効化した環境でのロックアウト等の最終復旧手段は、設定ファイルの手編集(ADR-0002 決定 5 のとおり手編集は常に可能)による opt-in の解除である(GUI 経路が全滅しても物理到達 + ファイルシステムがあれば復旧できる)
- (c) 認証有効化後は loopback にも一律で認証を課す: 「操作者特定」の一貫性は最も上がるが、①復旧経路(証明書事故・ロックアウト・DC 障害)を認証の巻き込みで複雑にする、②RDP でサーバコンソールに到達できる既存の身元保証(Windows のログオン監査)との二重要求の費用対効果が低い環境が多い、③既定を変えることは「アップデートしたら急にログイン画面が出る」事故になる(佐藤)。却下(opt-in としては (b) が同じ価値を選択制で提供する)

### ホスティング方式

- **(a) Kestrel + `Microsoft.AspNetCore.Authentication.Negotiate`(採用)**: 検証 1〜3 のとおり .NET 10 / Blazor Interactive Server との併用実績が公式に確認できる。Yagura は既に Kestrel を前提にホストしている(ADR-0002)
- (b) HTTP.sys への差し替え: HTTP.sys は Kernel Mode で Negotiate/NTLM/Basic をサポートし、SPN をマシンアカウントで扱える簡便さがあるが、**Yagura のリスナ実装全体を Kestrel から HTTP.sys へ切り替えることになり、影響範囲がリスナ 1 本の認証方式選定を大きく超える**(ADR-0002 のホスティング方針そのものの見直しに波及する)。加えて HTTP.sys は Windows 専用であり、`Microsoft.AspNetCore.Authentication.Negotiate` パッケージ自体が確認済みのクロスプラットフォーム性(Windows/Linux/macOS)を放棄することになる。却下
- (c) リバースプロキシ前提(TLS 終端・認証をプロキシに委譲): ADR-0004 決定 2 が既に「既定構成ではサポートしない、認証 opt-in が必須」と定義済みであり、本 ADR の対象外。加えて検証 1 のとおり Negotiate はプロキシとの相性が悪い(1:1 接続アフィニティを要する)ため、**本 ADR 実装後もプロキシ前置構成を推奨構成にはしない**(アプリ独自 ID/パスワード + プロキシ側 TLS 終端の組み合わせであれば選択肢として残る、という限定を明記するに留める)。却下(既定にしない)

## 決定

### 1. opt-in 認証の基本方針とリモートバインド解禁の条件

- **既定は現状維持**: 認証なし・管理リスナは loopback 束縛のみ。本 ADR は新しい既定を導入しない
- **認証を有効化して初めて、管理リスナのリモートバインドを選べるようになる**: 「認証なしでリモートバインドする」設定の組み合わせは**起動時に fail-closed で拒否**する——`security.md` §1 の L-4(「設定がどう壊れていても管理リスナが loopback 以外へ束縛されない」)を、**「認証が有効でない限り」という条件を追加したうえで維持する新しい不変条件として整理し直す**(既存 L-4 の対象を拡張する形。番号・CI 回帰テスト化は実装 PR + `security.md` で確定)。これはクリスが PR #167 で指摘した「他の設計条件が持つのと同水準の受け入れ基準・テスト要件」を最初から要求として明記する狙いを兼ねる。**「認証が有効」の判定は、Windows 統合認証・アプリ独自認証のいずれか一方でも有効なら真とする単純な OR** とし、委任事項 9(共存の詳細)と独立に確定できる形にする(リサの PR #173 レビュー質問への回答を仕様化)。**fail-closed の拒否は「なぜ起動しないか・何を直せばよいか」が一目で分かるエラーメッセージ・イベントログを伴うことを受け入れ条件に含める**(佐藤の指摘。委任事項 5)
- **loopback アクセスの扱い(2026-07-09 オーナー決定による改訂)**: **既定は現状(認証なし)を維持する**——ロックアウト・DC 障害・証明書事故のいずれが起きても単一依存に引きずられない最終復旧経路としての価値(佐藤・鈴木)を守る。**その上で、loopback アクセスにも認証を課す opt-in を設定として提供する**。無認証のままの loopback には次の限界が残ることを明記する(田中・クリスの指摘): ①操作者は接続元(`127.0.0.1`)としか記録されず個人を特定できない(ADR-0004 決定 7 既記の限界)、②到達できる主体は対話的ログオンユーザーに限らず、**管理リスナへネットワーク到達できる任意のローカル実行主体**(低権限で動く別サービス・スケジュールタスク・マルウェア・侵害されたプロセスを含む)である。loopback 認証 opt-in を有効化した環境では、loopback 経由の操作にも「誰が」欄の利用者名記録(決定 6)が及ぶ
- **loopback 認証 opt-in の誤設定に対する fail-closed(リモートバインドの fail-closed と対称)**: 「loopback 認証 opt-in が有効、かつ認証方式(Windows 統合・アプリ独自)が一つも有効に構成されていない」設定の組み合わせは、**有効化を受け付けない(設定ファイル手編集で作られた場合は起動時に fail-closed で拒否する)**——この組み合わせを許すと、認証手段が存在しないのに loopback にも認証が要求され、佐藤・鈴木の両ペルソナが最重要視した「最終復旧経路」そのものを誤設定 1 つで自壊させてしまう(クリスの再レビュー指摘 B)。リモートバインドの fail-closed(本決定 1 冒頭)と同じ扱い(エラーメッセージ・イベントログの誘導・CI 回帰テスト化)とし、Phase 1 の受け入れ条件に含める(委任事項 5)
- **loopback 認証 opt-in 有効時の最終復旧手段(前提と代償を正確に記す)**: 認証手段が全滅した場合(Windows 統合認証側の AD アカウントロックアウト〔決定 3 が明記する「Negotiate は OS/AD のロックアウト・パスワードポリシーに依拠する」もの。**アプリ独自認証の失敗試行対策は [ADR-0011](0011-app-auth-failure-backoff.md) でバックオフ化され「ロックアウト」ではなくなった**——本項の「ロックアウト」は AD 側を指す〕 + DC 障害等)の復旧は、**物理到達(RDP/コンソール)+ 設定ファイルの手編集による opt-in の解除**である(ADR-0002 決定 5 の手編集経路は常に残る)。ただしこの経路には次の前提・代償があり、いずれも利用者向けドキュメントに明記する(委任事項 11):
  - **反映には現状サービス再起動(= syslog 受信断)を要する**(鈴木の再レビュー指摘): 手編集の反映手段は「UI からの設定再読み込み」または「サービス再起動」のみ(`configuration.md` §3。ファイル監視による自動反映はしない)だが、**UI 再読み込みはまさに解除したい認証で守られており、この局面では使えない**(循環)。UI を経由しない再読み込み手段(コマンドライン等)は `configuration.md` §9 の **CF-5 として未確定**であり、CF-5 が確定するまでこの復旧は事実上サービス再起動——単一プロセス構成の帰結として**再起動時間はそのままログのロス時間になる**(ADR-0002)——を意味する。**CF-5 は本復旧経路の実効性を左右する事実上の前提条件**であり、この依存関係を委任事項 15 として本 ADR からも追跡する。ブレークグラス推奨(決定 3)に従わずに opt-in を有効化した場合の「詰んだときのコスト」は、設定ファイルを直す手間ではなく**受信断**である——この因果関係が opt-in 有効化の判断前に利用者へ伝わる記述を要件とする
  - **手編集には設定ファイルへの書き込み ACL を満たす必要がある**(クリスの再レビュー指摘 A・田中の再レビュー指摘): この復旧経路の安全性は `security.md` §5 の ACL 構成(設定ファイルを含むデータルート配下は `SYSTEM`/`Administrators`/サービスの仮想サービスアカウントのみ書き込み可。`Users`/`Authenticated Users` の ACE なし)に依存している。決定 1 で拡張した脅威モデル(任意のローカル実行主体)をこの経路にも当てはめると、**書き込みは ACL により Administrators 等に限定されており、opt-in の防御が設定ファイル書き換えで骨抜きになることはない**(ADR 単体でこの確認を追えるよう、依存を明示する)。残る余地として、**サービスアカウント自身は設定ファイルへの変更権限を持つため、Yagura プロセス自体が侵害された場合は、侵害プロセスが自分にかかる opt-in を設定ファイル経由で自己解除できる**——ただしこれは RCE 水準の侵害であり、その時点で認証機構全体が無意味化しているため、本 ADR 固有の新規リスクとしては扱わない(限界の明示に留める)
  - **裏返しとして、ローカル Administrators 権限を持たない者はこの経路で自己復旧できない**(田中の再レビュー指摘): 決定 5 は AD グループマッピングで「管理」役割をローカル Administrators 非所属の AD ユーザーへ付与することを認めている。この構成の管理者は、物理到達(RDP/コンソール)ができても設定ファイルの書き込み ACL を満たさず、**復旧には別途ローカル Administrator 権限を持つ者への依頼が必要になる**。この隠れた前提(手編集復旧の実行可能者 = ローカル Administrators)を利用者向けドキュメントに明記する(委任事項 11)

### 2. Windows 統合認証(Negotiate)の実現性と設計反映

ライブ検証 1〜4 の結果に基づき、次を実装要件として確定する:

- Kestrel への組み込みは `Microsoft.AspNetCore.Authentication.Negotiate`(NuGet、確認日時点の最新安定版 10.0.9)の `AddNegotiate()` を用いる。実装 PR で conventions.md の「外部依存を追加・更新する PR」手順(当日ライブ検証・`Vendored:` 行)を踏む。**採用後の継続的な脆弱性監視は、conventions.md の既存規約(依存パッケージの脆弱性スキャンを CI に組み込み、検出時は PR をブロックする)がそのまま覆う**——認証基盤の依存もこの CI スキャンの対象であることをここで明示参照する(田中の指摘)
- Blazor 側は `AddCascadingAuthenticationState()` を追加し、認証状態をコンポーネントツリーへカスケードする(公式サンプル `BlazorWebAppWinAuthServer` と同型)
- **circuit の認証状態は接続確立時のスナップショットに頼らない**: 検証 3 の公式パターン(`CircuitHandler.OnConnectionUpAsync` + `AuthenticationStateChanged`)を管理リスナの circuit に適用し、`security.md` §2.3 が既に定める「操作のたびに現在の認証状態で認可する」原則を、Windows 統合認証の技術的制約(`WindowsPrincipal` は元のリクエストに紐づく)の下でも満たす実装とする
- **NTLM フォールバックは追加実装なしで機能する**(検証 4)。非ドメイン参加機・ローカルアカウントでも Windows 統合認証を選べることを利用者向けドキュメントに明記する(ドメイン必須という誤解を避ける)。**あわせて、NTLM を無効化し Kerberos のみを要求する opt-in(Kerberos-only モード)を提供する**(田中の指摘。NTLM の既知の弱点——リレー攻撃・pass-the-hash・相互認証の欠如——を受容できない環境向け。実装方式は委任事項 12)
- **DC 到達不能時の Negotiate の挙動(即時失敗か・タイムアウト待ちのハングか・UI 上のフィードバック)を実装 PR の受け入れ条件として実機検証する**(鈴木の指摘。委任事項 13——委任事項 3 の SPN 検証とは別軸の検証項目として独立させる)。障害対応の現場で「管理画面を開いたら固まった」を作らないため、検証結果に基づく利用者向けの警告文言も確定する
- **Kerberos SPN の扱いは未解決事項として明示する**: サービスが仮想サービスアカウント(`NT SERVICE\Yagura`)で動作する前提で、リモートアクセス時にホスト名を使う構成での SPN 登録の可否・手順は**実機検証(lab)を要する**(conventions.md「実環境依存の機能は lab 検証を受け入れ条件に含める」に従う。委任事項 3)。**登録できない・失敗する場合でも NTLM への自動フォールバックにより機能自体は止まらない**(検証 4)が、これは「意図した保護水準(Kerberos)を満たす」こととは別である——Phase 2 のリスク受容(決定 8・帰結)を参照
- HTTP/2 非対応の制約(検証 1)は管理リスナの Kestrel プロトコル設定で HTTP/1.1 を許容していることの確認を実装 PR に含める(Phase 1 受け入れ条件 (i) の「検証 1〜4 で洗い出した実装要件」に含まれる)

### 3. アプリ独自 ID/パスワード方式の設計方針

- **最小実装を採用する**(検討した選択肢のとおり)。管理者アカウントは既存のデータ provider 抽象に載る単一テーブルとし、全 DB provider(SQLite/SQL Server/将来の PostgreSQL・MySQL)で同一スキーマを持つ
- **パスワード保管**: `PasswordHasher<TUser>` 相当の PBKDF2 実装の利用可否(追加 NuGet 依存の要否を含む)を実装 PR で確定する。独自のハッシュ実装は行わない(枯れた実装への相乗り)。**方向性として、`PasswordHasher<TUser>` が利用可能な場合はそのハッシュ形式との互換を優先する**——将来 ASP.NET Core Identity へ移行する場合に既存アカウントのパスワードハッシュがそのまま持ち越せる(全アカウント強制リセットを避ける)ため(リサの指摘)
- **ロックアウト(2026-07-11 supersession により本段落は無効。[ADR-0011](0011-app-auth-failure-backoff.md) 参照)**: ~~一定回数の失敗試行で一時ロックアウトする(閾値・期間は仮値で実装し実装 PR で確定——`architecture.md`/`security.md` の SEC-x/CF-x と同じ運用に倣う)。ロックアウト発生・解除は監査対象とする(決定 6)。~~ **2026-07-11、単一アカウントモデル + Phase 2 リモートバインド解禁の組み合わせが「唯一の正規管理者を締め出せる DoS の梃子になる」というセキュリティレビュー指摘(M-1)を受け、オーナーは本段落の「一定時間の完全ロックアウト」機構を supersession 対象と判断した。ADR-0011 が、完全拒否を行わない指数バックオフ + 送信元 IP レート制限 + グローバルトークンバケットへ置き換える(loopback の無条件復旧経路という決定 1 の前提はレート制限側にも拡張する)。ロックアウト発生・解除の監査(決定 6)は ADR-0011 決定 9 のとおり新設 ID へ引き継ぐ。**ロックアウトだけに頼らない**という設計思想自体(ユーザー列挙耐性・分散送信元対策・CSRF 対策を委任事項 4 に含める——田中・クリスの指摘)は ADR-0011 でも維持される
- **ロックアウト・パスワード失念時の復旧経路を明記する**(佐藤の指摘。**2026-07-11 注記: 見出しの「ロックアウト」は [ADR-0011](0011-app-auth-failure-backoff.md) 適用後は「バックオフ待機・レート制限中」と読み替える——復旧経路そのもの〔下記〕は不変**): **物理到達(RDP/コンソール)→ loopback 管理 UI からの解除・再設定が基本経路**である(既定では loopback は認証なしのため常に到達できる)。loopback 認証 opt-in(決定 1)を有効化した環境では、決定 1 に記した設定ファイル手編集が最終手段になる。なお ADR-0011 のバックオフでは loopback は IP レート制限・グローバルトークンバケットの対象外(ADR-0011 決定 4)であり、この復旧経路は失敗試行対策のどの層にも阻まれない
- **初期パスワード設定の流れ**: 「ゼロ設定ファーストラン」(CLAUDE.md 確定方針 2)との整合を次のように取る——**アプリ独自認証を有効化する操作自体が、既に loopback 限定の管理 UI(認証なしの現状の管理リスナ)上で行われる**。つまり最初の管理者アカウント(ユーザー名・パスワード)は、物理到達性による信頼が既に効いている状態(RDP・コンソールログオン。ADR-0004 決定 2)でオペレーターが対話的に設定する。ランダム生成した初期パスワードを別経路で配布する、という「初期パスワード漏洩・失念」のリスクの高い設計は採らない——**リモートバインドは、認証(と最初の管理者アカウント)が既に構成済みの状態でのみ次の opt-in として選べる**(決定 1 の順序がそのままこの問題を回避する構造になっている)
- **両認証方式の共存の設計方針(本文レベルで固定する。クリスの指摘による委任事項 9 からの昇格)**:
  - **共存は ASP.NET Core の複数認証スキームとして構成する**(Negotiate スキームとアプリ独自スキームの並立)。利用者から見える優先順位・ログイン画面の振る舞い(選択式か・自動試行か)の詳細は委任事項 9 で確定する
  - **ロックアウトは方式ごとに独立とし、相互のバイパス経路にしない**(**2026-07-11 注記: [ADR-0011](0011-app-auth-failure-backoff.md) 適用後は「ロックアウト」を「バックオフ・レート制限」と読み替える。方式ごとに独立という原則自体は不変——`security.md` §2.4 側の用語更新は ADR-0011 委任事項 4**): アプリ独自アカウントのロックアウトは Windows 統合認証の可否に影響せず、その逆もない。これは「バイパス」ではない——Windows 統合認証は OS/AD の身元保証(独自のロックアウト・パスワードポリシーを持つ)に依拠しており、アプリ独自認証のロックアウトが守ろうとする対象(アプリ内アカウントの資格情報)とは別の信頼基盤である。一方の方式の侵害がもう一方の保護されたセッションへ波及しない分離(セッション・トークンを方式間で共有しない)を委任事項 9 の受け入れ条件に含める(**2026-07-12 [ADR-0013](0013-admin-winauth-session.md) により本文中「セッション・トークンを方式間で共有しない」の部分を supersede**: 認証成立後は方式に依らない単一の認証セッション Cookie を発行する方式へ改める。分離の実体は資格情報とバックオフ／レート制限状態の独立として保存する——ADR-0013 決定参照。この共存セッションモデルの初期実装は Windows 認証でセッションを一切発行せず 401 ループする欠陥を持っていた〔Issue #252〕)
  - **監査の「誰が」欄は認証方式を併記する命名空間つき表記とする**(例: 方式識別子 + 利用者名。具体形式は委任事項 6)——`DOMAIN\user`(Windows)とアプリ独自ユーザー名の衝突・なりすましの余地を構造的に防ぐ(クリスの指摘)
- **ブレークグラス(非常口)の強い推奨**(鈴木の指摘): **リモート管理を Windows 統合認証で運用する場合も、アプリ独自 ID/パスワードの管理者アカウントを最低 1 つ保持することを強く推奨する**——AD/DC 障害時に RDP 経路(踏み台・VPN)も同じ AD に依存している構成では、Negotiate 単独運用は AD 障害 1 点で管理面全体を失う。必須にはしない(AD 単独で運用する自由は残す——強制はそれ自体が新しい資格情報管理の押しつけになる)が、リモートバインド有効化ウィザードでの推奨表示と利用者向けドキュメントへの明記を実装要件とする(委任事項 11)
- **ASP.NET Core Identity フル導入との比較**は検討した選択肢のとおり——供給網統治(依存追加・DB provider 横断のスキーマ増加)を理由に最小実装を選ぶ。将来ロールが増える場合の再検討の余地は残す

### 4. 平文 HTTP でのリモート認証の危険性への対処

- **リモートバインドされた管理リスナは HTTPS を必須とする**(ADR-0004 決定 3 の具体化)。認証(Windows 統合認証・アプリ独自 ID/パスワードのいずれも)を有効化しただけでは、リモートバインドは開放されない(決定 1)——HTTPS 未構成のままリモートバインドを試みる設定は fail-closed で拒否する
- **最低 TLS バージョンを明示する**(田中の指摘): リモートバインドされた管理リスナは **TLS 1.2 以上を最低要件とし、TLS 1.3 を優先する**。OS 既定(schannel ポリシー)に暗黙に委ねず、Kestrel のリスナ構成で明示的に固定する(Windows Server の版が混在する導入先で TLS 1.0/1.1 が意図せず有効のまま露出することを防ぐ)。暗号スイートの個別制約の要否は実装 PR で確定する(委任事項 7)
- **証明書の参照方式は `configuration.md` §6(閲覧リスナの HTTPS)と同型を用いる**: Windows 証明書ストア(ローカルコンピューター)からの選択、秘密鍵の読み取り権限のみをウィザードが付与(付与は監査対象)、自己署名証明書の生成支援は提供しない——これらはいずれも `configuration.md` §6 が既に確定した設計であり、本 ADR は方式を継承する。**ただし継承だけで済ませない**(クリスの指摘): 本 ADR が解禁するリモート管理の主対象層(AD はあるが AD CS 未導入の環境が最多想定——`configuration.md` §6 自身の記述)にとって証明書取得が最初の障壁になることを認め、①CF-D2(operations.md への申し送り)の「選択指針」に**ひとり情シス向けの最短経路の明示**(どの選択肢が最少手数か)を必須内容として追加する(佐藤の指摘。委任事項 7)、②自己署名証明書を利用者が持ち込んだ場合の Negotiate 認証フローの成立可否(証明書警告のインタースティシャル後にチャレンジが正しく発火するか等)を lab 検証する(委任事項 14)
- **設定キーは閲覧リスナの HTTPS と独立させる**(`security.md` §6 が TLS 受信証明書と Web UI HTTPS の関係で既に採用している「設定キーは独立とし、同一証明書の流用は両方のキーに指定することで実現する。暗黙の連動をしない」という規約を管理リスナにも適用する)。**同じ証明書を使い回すことは妨げないが、片方の変更が他方へ黙って波及しない構造を保つ**
- **`configuration.md` の複数箇所が本 ADR の実装完了後は正確でなくなる**(リサの指摘により対象を拡充): ①**§6** の「管理リスナは HTTPS 化の対象外——防御の主体は loopback 束縛」(リモートバインド opt-in 時は正しくない)、②**§4.2** の「bind 先(loopback)を変える設定キーは設けない」(本 ADR はまさにこのキーを条件付きで導入する)、③**§4.3** の「管理リスナは loopback 束縛のため受信許可規則(ファイアウォール)を要しない」(リモートバインド時は FW 規則が必要になる)。**この 3 箇所の矛盾の解消(qualify する追記)は実装 PR で `configuration.md` に反映する**——本 PR の時点で対象箇所を名指しで明示しておくことで、PR #167 でリサが指摘した「ADR と設計書のズレ」問題を「認識済みで追従対象が特定されている」状態にする
- **証明書の期限切れ時の挙動**: `configuration.md` §6 の既存方針(HTTPS リスナは停止し HTTP へフォールバックしない)を管理リスナのリモートバインド面にも適用する。**ただし loopback 経由の管理リスナは HTTPS の対象外のまま残る**(決定 1)ため、リモート HTTPS が証明書事故で止まっても、サーバコンソール(RDP)からの復旧操作は引き続き可能——`configuration.md` §6 が既に確立した「復旧の動線は HTTPS 停止に巻き込まれない」という設計をそのまま活かせる。期限接近の事前警告は `configuration.md` §6 既存の能動通知(証明書期限の接近)がそのまま管理リスナ用証明書にも適用される(佐藤の質問 4 への回答: 本 ADR の範囲内であり、既存機構への追加登録として実装 PR で扱う)

### 5. 認可(誰を管理者とするか)

- **既定は `BUILTIN\Administrators`(well-known SID `S-1-5-32-544`)**。検証 2 で確認した SID クレームベースの認可ポリシーで判定する。AD 環境でのグループマッピング拡張は `security.md` §3 の既存設計(AD グループへのマッピング。指定形式・ネストグループ解決は SEC-9 で認証 opt-in 実装時に確定予定)にそのまま乗せる
- **アプリ独自 ID/パスワード認証で作成したアカウントは、常に「管理」役割のみを持つ**(v1.0 までの役割は閲覧/管理の 2 段。`security.md` §3)。アプリ独自認証は管理リスナ専用の認証手段として設計するため、閲覧役割の概念を持たない
- **役割の粒度は増やさない**(`security.md` §3 の既存の再評価トリガ「独立利用者から 3 件以上の要望」をそのまま適用する。本 ADR は新しい粒度を持ち込まない)

### 6. 監査

- **「誰が」欄の実効化(射程を正確に記す)**: `security.md` §4.1 は記録内容を「誰が(認証有効時は利用者名、無効時は接続元)」と既に定義済みだが、v0.1 は認証機構自体が存在しないため常に「無効時」の枝(接続元)しか実現できなかった。本 ADR の実装により、**認証を経由した操作については** 2000 番台の全管理操作記録(2001〜2005 等)の「誰が」欄に、認証済み利用者名(決定 3 の命名空間つき表記——方式識別子 + `DOMAIN\user` またはアプリ独自ユーザー名)を記録する。**射程の限定を明記する**(田中の指摘): これが及ぶのは①リモート経由の操作(常に認証必須)と、②loopback 認証 opt-in(決定 1)を有効化した環境の loopback 経由の操作、である。**既定(loopback 認証 opt-in 無効)の loopback 経由の操作は、従来どおり接続元(`127.0.0.1`)しか記録されない**——PR #167 で田中が指摘した「操作者特定が構造的に成立しない」問題のうち、本 ADR が解消するのはリモート面(および opt-in 有効時の loopback 面)であり、既定の loopback 面の限界は残る
- **新規の拒否・セキュリティ事象(3000 番台)を追加する**(`security.md` §4.3 の採番方針に従い、実装 PR で次番から確定。本 ADR では意味と記録フィールドを先に定義する):
  - Windows 統合認証のハンドシェイク失敗・拒否
  - アプリ独自認証のログイン失敗(試行されたユーザー名を保持する。§4.4 の集約規約——事象種別が変われば個別記録・利用者名の次元を保持——をそのまま適用する)
  - ~~アカウントロックアウトの発生・解除~~ (**2026-07-11 注記: [ADR-0011](0011-app-auth-failure-backoff.md) 決定 9 により、この監査事象は 3005〔ロックアウト〕の凍結 + 3006 以降〔バックオフ cap 到達・レート制限拒否等〕の新設へ引き継がれた。本行の「ロックアウトの発生・解除」は ADR-0011 適用後は発火しない**)
  - **記録フィールドの要件**(田中の指摘): 上記の各事象は §4.1 の一般内容(接続元・UTC 時刻)に加え、**送信元 IP・失敗理由の種別(資格情報誤り / アカウント不在 / ロックアウト中 / SPN・プロトコル失敗等)・使用された認証方式とプロトコル(アプリ独自 / Negotiate、Negotiate の場合は Kerberos か NTLM か)**を構造化して記録する(侵害後調査で「どのプロトコルで・なぜ失敗したか」を追跡できるようにする)。**失敗理由の種別は監査記録にのみ残し、利用者への応答(画面・HTTP レスポンス)では区別しない**(決定 3 のユーザー列挙耐性と両立させる——攻撃者に理由を教えず、監査には残す)
  - **認証なしでのリモートバインド試行の起動時拒否**(決定 1 の fail-closed): これは外部からの要求への拒否ではなく設定検証の失敗であり、**起動失敗に直結する重大事象として 1000 番台(運用警告)のエラーレベル相当で扱う方向を先に決める**(鈴木の指摘——3000 番台の警告レベルに置くと他の拒否試行と同列に埋もれ、深夜の起動失敗の発見が遅れる。最終採番は実装 PR)
- **`security.md` §4.4 の拒否試行の流量制御をログイン失敗にも適用する**——ブルートフォース的な試行が単発の拒否を希釈しない設計をそのまま踏襲する
- **既存の限界の明記を維持する**: 認証を経由しない loopback 操作は引き続き「誰が」欄が接続元のみになる。この限界は ADR-0004 決定 7 が既に明記済みであり、利用者向けドキュメントの既存の注記(「接続元を記録しているから個人を特定できる、という誤解を与えない」)はそのまま有効である

### 7. 閲覧 UI(8514)への拡張——opt-in 対象に含める(2026-07-09 オーナー決定で確定)

ペルソナレビューに両論併記で付託した本論点は、オーナー決定(2026-07-09)により**「閲覧 UI も opt-in 対象に含める」で確定**した。

**決定**:

- **管理 UI と同じ認証基盤(認証方式・fail-closed の設計)を、閲覧 UI(8514)にも opt-in で提供する**。既定は現状どおり認証なし・LAN 公開のまま(信頼ネットワーク前提 + opt-in 強化——ADR-0004 決定 2・CLAUDE.md 方針 7 の維持)。閲覧 UI の認証 opt-in を有効化しない環境では、本決定による体験の変化は一切ない
- 役割判定は `security.md` §3 の既存の 2 段モデル(閲覧/管理)にそのまま乗せる: **閲覧 UI の認証は Windows 統合認証(AD グループマッピング → 「閲覧」役割)を主経路**とし、これまで使われる場面のなかった「閲覧」役割がここで初めて実効化する。アプリ独自アカウントは決定 5 のとおり「管理」役割のみであり(管理 ⊇ 閲覧として閲覧面へもアクセスできる)、**アプリ独自の「閲覧専用アカウント」は導入しない**(決定 5 の「閲覧役割の概念を持たない」を維持——閲覧者の個別アカウント管理という新しい管理対象を作らない)
- 認証有効時の circuit 失効の反映は `security.md` §2.3 の既存設計(読み取り専用表示は猶予付き・掲示ダッシュボードを不意に切らない・猶予上限は SEC-6)がそのまま適用される——閲覧 UI 認証はこの設計が最初から想定していた適用先そのものである
- **閲覧リスナ固有のエンドポイントとの関係は次のとおり整理する**(詳細は委任事項 16): ①`security.md` §1 の L-5(閲覧リスナの全ルート・全ハブの許可リスト突合)は認証導入後も不変条件として維持する——認証は「誰が到達できるか」を絞る機構であり、「どの経路が存在するか」を検証する L-5 と直交する(認証を理由に許可リストを緩めない)。②**liveness エンドポイント `/health`(Issue #126 で採用済み。`security.md` §1——認証なし・固定文字列 `OK` のみ・GET/HEAD のみ・内部情報を一切持たない)は、閲覧 UI 認証 opt-in 有効時も認証の対象外のまま維持する**——監視システムは資格情報を持たない前提が一般的であり、認証を課すと「認証基盤の障害 = 監視の盲目化」という依存を作る。対象外にできる根拠は同エンドポイントの既存設計(内部情報ゼロの固定レスポンス限定)そのものであり、**この「非機微・固定・読み取りのみ」という制約を認証除外の継続条件とする**(制約を緩める変更は除外根拠を失わせるため、認証要否の再判定を伴う)。③CSV エクスポート等、ログ本体をまとめて取り出す口を閲覧リスナに設ける場合は、通常の閲覧画面と同じ認証判定の対象とする(画面で読めるものはエクスポートでも読める、以上の緩和をしない)
- **実装は管理 UI 先行の Phase 構成の後段(Phase 4)に位置づける**(決定 8)。管理 UI(Phase 1〜2)の認証基盤・fail-closed・監査の実装と検証が済んだ土台の上に展開する

**検討経緯の記録**(ペルソナレビューに付託した両論と判断理由):

- 採用(拡張する)を支えた材料: ①閲覧 UI は機微情報を含みうるログを LAN 内の誰でも読める(ADR-0004 決定 2 が「受け入れるリスク」として明記済み——このリスクを opt-in で縮小する手段の提供)、②管理面の認証基盤構築のついでに展開すれば投資対効果が上がる、③`security.md` §3 の「閲覧」役割がここで初めて活きる、④具体的なユーザー像(PR #173 佐藤レビューの実声): 機微ログ(社内の認証失敗・IP アドレス等)を扱うひとり情シス環境では「誰でも見て良いとは言い切れない」という需要が現実にある。機微情報への接触の統制という田中(PR #167・#173)の一貫した観点とも整合する
- 却下案(管理 UI のみに留める)の論拠と、それを退けた理由: ①「スコープ拡大で実装・レビューが肥大化する」→ Phase 4 として後段に分離することで、主目的(リモート管理の解禁 = Phase 1〜2)の速度は犠牲にしない。②「閲覧 UI は認証なしでも安全に設計されている(書き込みエンドポイントを持たない)」→ 書き込みがないことは「読まれてよい」ことを意味しない(機微ログの閲覧統制は書き込み防御とは別の要求)。③「ADR-0004 の既存の再評価トリガ(要望 3 件以上)に委ねる方が一貫」→ そのトリガが守る対象は「既定を認証必須へ変えるか」であり、既定を変えない opt-in の提供とは衝突しない——トリガを先取りして既定を変えるのではなく、選択肢を先に用意するだけである。「30 分で動かせるか」(佐藤)の体験も既定が変わらないため影響を受けない

### 8. 段階導入計画と受け入れ条件

- **Phase 1: 認証基盤の実装(loopback 限定のまま)**。Windows 統合認証・アプリ独自 ID/パスワードの両方を実装し、管理者は loopback 内で認証を任意に有効化できる状態にする(loopback 認証 opt-in——決定 1——もこの Phase に含む)。リモートバインドはまだ許可しない。受け入れ条件: (i) 検証 1〜4 で洗い出した実装要件(circuit 認証状態の明示的な汲み直し・HTTP/1.1 許容確認を含む)がテストで固定されている、(ii) 決定 6 の監査イベント(記録フィールド要件を含む)が実装され「誰が」欄が実効化されている、(iii) アプリ独自認証のロックアウト・パスワードハッシュの実装がテストで固定されている、(iv) **決定 3 の共存設計方針(スキーム並立・ロックアウト独立・~~セッション非共有~~〔2026-07-12 [ADR-0013](0013-admin-winauth-session.md) により「資格情報・バックオフ/レート制限状態の独立」へ読み替え。Issue #252〕・監査命名空間)が実装に反映され、テストで固定されている**(クリスの指摘による追加)、(v) **loopback 認証 opt-in の fail-closed 不変条件(認証方式が一つも構成されていない状態での有効化の拒否——決定 1)がリモートバインドの fail-closed と同水準の CI 回帰テストで固定されている**(クリスの再レビュー指摘 B による追加)
- **Phase 2: リモートバインドの解禁 + HTTPS**。決定 1 の fail-closed 不変条件(認証なしでのリモートバインド拒否)を CI 回帰テストとして固定する(`security.md` L-4 系の拡張。クリスが PR #167 で要求した「他の設計条件と同水準の受け入れ基準」に応える)。決定 4 の HTTPS 証明書実装(独立キー・秘密鍵権限付与・最低 TLS バージョンの明示固定)を完了する。受け入れ条件: (i) 上記 CI 回帰テストの合格、(ii) HTTPS 証明書の秘密鍵権限付与の実機確認、(iii) **管理リスナの circuit 上限(SEC-1 仮値: 管理 5)が「リモートから複数オペレーターが同時対応する」利用像で見直されていること**(鈴木の指摘——現行仮値は「loopback 限定・実質 1 人」前提であり、リモート解禁はこの前提を変える)、(iv) **管理リスナへの認証負荷(大量のログイン試行・接続集中)が syslog 受信スループットへ悪影響を及ぼさないことの負荷試験による確認**(鈴木の指摘——「ロス 0」を掲げる製品の新しい攻撃対象面である以上、推測で済ませない)、(v) **Kerberos SPN が未検証の時点でのリモート解禁は「NTLM が事実上の常用認証経路になり得る」ことの明示的なリスク受容を伴う**——この受容と NTLM の既知の弱点をリリースノート・利用者向けドキュメントで明示する(クリス・リサの指摘。Phase 3 任せにしない)
- **Phase 3: Kerberos SPN の実機検証**。仮想サービスアカウントに対する SPN 登録の可否を lab 環境で確認し、`operations.md` へ手順(またはできないことの明記と NTLM フォールバックへの誘導)を追記する。この Phase は Phase 2 の受け入れ条件ではない(検証 4 のとおり、失敗しても NTLM フォールバックで機能は止まらないため)——**ただしこの順序は「NTLM 実質専用でのリモート解禁を暫定的に受容する」という意思決定であり、Phase 2 受け入れ条件 (v) でその受容を明示する**
- **Phase 4: 閲覧 UI(8514)への opt-in 認証の拡張**(決定 7)。管理 UI で実装・検証済みの認証基盤を閲覧リスナへ展開する。既定(認証なし・LAN 公開)は変えない。受け入れ条件: (i) L-5 許可リスト・circuit 統治(SEC-1/SEC-8)等の既存の閲覧リスナ不変条件が認証導入後も維持されることのテスト固定、(ii) 認証除外エンドポイント(`/health`——Issue #126 実装済み)の除外の実装と、除外面が「非機微・固定・読み取りのみ」に限られ続けることの確認(委任事項 16)、(iii) `security.md` §2.3 の失効猶予(SEC-6)・掲示用途配慮の値の確定(同節が「認証 opt-in 実装時に確定」と予告済みの事項をこの Phase で引き受ける)。Phase 3(Kerberos SPN 検証)との順序関係は固定しない(独立に進められる)
- 各 Phase の完了ごとにペルソナレビューを再度受けるかは、`docs/development/persona-review.md` の「指摘対応で起案文書の内容が大幅に変わった場合」の運用ルールに従う(実装 PR は通常のコードレビューだが、`security.md`/`configuration.md` の追従更新は設計文書の変更として扱う)

### 9. Issue #130(MSI アップロード)再起案への接続

Issue #130 の再評価トリガ(PR #167 オーナー判断)は「管理 UI に身元保証を伴う認証が導入された時点」である。本 ADR との接続を正確に記す(田中の PR #173 レビュー指摘による明確化):

- **本 ADR の実装(Phase 1・2)だけでは、#167 が実際に問題にした脅威モデル(loopback 到達者による MSI 差し替え)は解消されない**——既定の loopback は無認証のまま残るため(決定 1)
- **#130 を再起案する際は、「loopback 認証 opt-in(決定 1)の有効化」を MSI アップロード機能の利用の前提条件とすることを強く推奨する**(クリスの再レビュー指摘 C を受け、任意形「採れる」から決定 3 のブレークグラス推奨と同じ温度に強めた)——これにより「MSI を差し替えられる者」の全経路(リモート・loopback とも)に利用者名の記録(決定 6)が及び、#167 の田中の指摘(操作者特定)・クリスの指摘(任意のローカル実行主体)に正面から応える構成になる。最終判断は再起案時の設計(とそのペルソナレビュー)に委ねるが、**前提条件としない設計を選ぶ場合はその理由の明示をペルソナレビューの必須論点とする**。あわせて、**「認証が導入されたから MSI アップロードも自動的に安全になった」という誤読をここで明示的に否定しておく**
- 再起案は本 ADR のスコープに含めない。実装完了後に別 Issue として起票し、PR #167 に保存された 5 ペルソナの指摘を出発点とする supersession として進める(決定を混ぜない)

## 却下した代替案

- **HTTP.sys への委譲**: 検討した選択肢のとおり、ホスティング方式全体の見直しに波及するため却下
- **リバースプロキシ前提**: ADR-0004 決定 2 で既に却下・非対応と定義済み。本 ADR の認証機構が入っても既定構成として推奨しない(検討した選択肢のとおり)
- **認証なしでのリモートバインド**: Issue #172・PR #167 の指摘の核心そのものであり、却下

## 帰結

- 良くなること:
  - リモート管理が可能になり、RDP が唯一の管理経路という制約(佐藤ペルソナの不満)が解消する
  - 管理操作の監査記録の「誰が」欄が、認証を経由した操作(リモート経由の全操作 + loopback 認証 opt-in 有効時の loopback 操作)について実効性を持つようになる。**射程の限定に注意**: PR #167 で田中が指摘した loopback 面の身元保証の欠落は、既定(opt-in 無効)では残る——解消されるのは opt-in を有効化した環境に限る(決定 6・9)
  - Issue #130(MSI アップロード)の再評価トリガが本 ADR の実装完了(少なくとも Phase 1・2)によって満たされる。再起案時の設計指針(loopback 認証 opt-in の前提条件化)は決定 9 のとおり
  - 機微ログを扱う環境が、閲覧 UI の認証を opt-in で有効化できるようになる(決定 7・Phase 4)。ADR-0004 決定 2 が「受け入れるリスク」としてきた「LAN 内の全員がログを読める」既定に、既定を変えずに選択肢が加わる
- 悪くなること(受け入れるトレードオフ):
  - **NTLM の既知の弱点を持ち込む**: NTLM にはリレー攻撃・pass-the-hash・相互認証の欠如という具体的な弱点があり、Microsoft 自身が段階的廃止を進めている。仮想サービスアカウントでの Kerberos SPN 成立が未検証のまま Phase 2 でリモート解禁する場合、**Windows 統合認証は NTLM が事実上の常用経路になる可能性が高い**——これは「例外的な縮退」ではなく Phase 2 時点の実態になり得ることを明示的に受容する(決定 8 (v)。緩和策: Kerberos-only モードの opt-in——決定 2・委任事項 12、および HTTPS 必須による経路の暗号化)。アプリ独自認証はパスワードブルートフォース・列挙攻撃の面を持ち込む(緩和策: 決定 3・委任事項 4)
  - **認証方式 2 系統の恒常コスト**: 実装・テスト・脆弱性対応を 2 系統分負い続ける(検討した選択肢 (c) のトレードオフ。共存設計——決定 3——の複雑さを含む)
  - 既定の loopback 面の限界(操作者特定不能・任意のローカル実行主体の到達可能性)は、opt-in を有効化しない限り残る(決定 1・6 で誠実に線引き)
  - **loopback 認証 opt-in は、「無条件の最終手段」だった loopback アクセスに初めて「詰む」余地を持ち込む**(鈴木の再レビュー指摘の要約): 誤設定による自壊は fail-closed(決定 1)で防ぐが、正しく設定された opt-in でも、認証手段の全滅時の復旧は CF-5 確定まで**サービス再起動 = syslog 受信断**を伴う。この代償はブレークグラス推奨(決定 3)と委任事項 11・15 で利用者に事前に伝える
  - `configuration.md` §4.2・§4.3・§6 の記述が本 ADR の実装完了まで一時的に古くなる(決定 4 で対象を名指しし、実装 PR での追従を予告)
  - アプリ独自認証の最小実装は、ASP.NET Core Identity が提供する将来の拡張性(2FA・外部 IdP 連携等)を持たない。役割粒度やより強い認証手段が必要になった場合は再設計を要する(パスワードハッシュ形式の互換優先——決定 3——により移行コストの一部は緩和される)
- 受け入れるリスク: Kerberos SPN の仮想サービスアカウントでの成立可否が本 ADR 起案時点で未検証であり、Phase 3 まで不確実性が残る。ただし検証 4 のとおり NTLM フォールバックにより認証機構自体は機能する(そのリスク受容は上記のとおり明示済み)

6. **2026-07-12 (supersession 注記の追加。[ADR-0013](0013-admin-winauth-session.md) 起案。[Issue #252](https://github.com/Yanai-Taketo/Yagura/issues/252))**。lab 実機(v0.3.2)で、`Admin:Authentication:Windows:Enabled=true` + `Admin:Authentication:RequireForLoopback=true` の構成において**正しい資格情報でも管理 UI にログインできない**(認証ダイアログが繰り返し戻り `/admin` へ到達できない)P0 回帰を発見した。監査には `AdminLoginSucceeded`(2008・`scheme=windows`)が記録され**認証は成功するが identity が後続の認可判定に伝播しない**症状。根本原因は、選択式ログイン(決定 3・委任事項 9)の Windows 経路(`/admin/login/windows`)が Negotiate 成功後に**認証セッションを一切発行していなかった**こと——Negotiate はステートレスで Cookie を発行せず、既定認証スキームはアプリ独自認証の Cookie スキームを指す(かつアプリ独自認証有効時のみ登録)ため、後続の管理画面要求が匿名判定 → Negotiate 401 チャレンジ → また匿名、の無限ループになる。委任事項 9 の「セッション非共有」を「Cookie を方式ごとに別立て」と過剰解釈したことが設計上の一因。ADR-0012 のリモートバインド(HTTPS)経由の Windows 認証にも同じ欠陥が波及する。

   オーナーは 2026-07-12、**認証成立後は方式に依らない単一の認証セッション Cookie を発行する方式(方式 B)**を採用し、これを ADR-0010 決定 3 の「セッション・トークンを方式間で共有しない」共存セッションモデルの**部分 supersession** として [ADR-0013](0013-admin-winauth-session.md) で扱うことを決定した(状態行・決定 3 本文・委任事項 9・Phase 1 受け入れ条件 (iv) に取消線 + 注記を追加)。これは決定 3 の明文「セッション・トークンを方式間で共有しない」を最小単位で反転する変更であり([README](README.md) の判定基準「疑わしければ supersession 側」、および改訂履歴 5 = ADR-0011 が明文の決定語反転を supersession とした先例に倣う)、amendment ではなく supersession として扱う(いったん amendment=本改訂履歴 6 の初稿として起案したが、PR #253 のペルソナレビュー〔リサ・クリス〕とオーナー判断で ADR-0013 へ格上げした)。決定 3 の他の内容(最小実装・単一アカウントモデル・スキーム並立・ロックアウト独立=ADR-0011・監査命名空間・ブレークグラス推奨)、および決定 1〜9 のそれ以外の決定・既定は影響を受けず、そのまま有効。
