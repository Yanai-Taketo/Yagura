# ADR-0001: 旧リポジトリからの再出発（フルリライト）の決定と旧資産の扱い

- 状態: proposed
- 日付: 2026-07-03
- 決定者: YANAI Taketo
- 関連: [docs/lessons/legacy-lessons.md](../lessons/legacy-lessons.md)、[docs/lessons/legacy-feature-inventory.md](../lessons/legacy-feature-inventory.md)、旧リポジトリ Yanai-Taketo/yagura-dev

## 文脈と課題

Yagura は旧リポジトリ（yagura-dev）で v0.5.0 まで到達したが、v0.6 開発途中（2026 年 5 月下旬、PR #154）で活動が途絶えた。約 1 か月の中断を経て再開するにあたり、以下の課題が確認された。

1. **文脈の喪失**: 中断により「どこまで何が決まっていたか」の把握コストが大きくなっていた。再開時に旧リポジトリ全体（PR 138 本 / Issue 16 本 / SPEC 26 本 / ADR 11 本）の再調査が必要だった
2. **構造的な負債**: DB 抽象化（旧 SPEC-023）は SQL Server 直書きが全域に染み込んだ後では改修不能となり、唯一の未実装 SPEC として残った。信頼性機構（スプール・流量制御・キュー容量）は v0.6 での後付け増築になった
3. **UI の品質問題**: デザインシステム不在のままページ単位で画面と JavaScript を継ぎ足した結果、見た目・一貫性ともに品質が低かった（オーナー評価:「かなり醜いデザイン」）
4. **ドキュメントの継ぎ足し体質**: 仕様が差分 SPEC の積み重ねになり、「現在の全体仕様」を示す文書が存在しなかった。ドキュメント追従だけの PR が全体の約 2 割を占めた

## 検討した選択肢

### (a) yagura-dev で継続開発する

- 利点: 工数最小。実装済み機能・テスト約 420 件・CI をそのまま使える
- 欠点: 上記の構造的負債（DB 直書き・UI・ドキュメント体質）はそのまま残る。特に DB 抽象化は「後付け改修が高コストすぎて着手されなかった」という実績があり、継続しても解消の見込みが薄い

### (b) 部分移植（コア実装を持ち込み、周辺を書き直す）

- 利点: 実装コストと刷新のバランス
- 欠点: 「どこまで持ち込むか」の線引き判断が全域で発生し、持ち込んだコードが旧設計の制約を新設計に持ち込む。UI と永続化層という書き直し対象がコードの大半を占めるため、実際に残せる部分は少ない

### (c) フルリライト（採用）

- 利点: DB provider 抽象・信頼性機構・デザインシステム・ドキュメント体系を初期設計に組み込める。「既設に縛られず新しい思想を取り入れる」というオーナー方針に合致
- 欠点: 実装済み機能の再構築コスト。旧実装で踏んだ技術的トラップ（15 件累積）を再び踏むリスク

## 決定

**新リポジトリ（Yanai-Taketo/Yagura）で 1 からフルリライトする。**

旧リポジトリの扱い:

- **コードは持ち込まない**。設計は白紙から行い、旧設計の踏襲を既定にしない
- 旧リポジトリは**教訓データベース + 機能チェックリスト**として参照する。教訓は [docs/lessons/legacy-lessons.md](../lessons/legacy-lessons.md) に、機能一覧は [docs/lessons/legacy-feature-inventory.md](../lessons/legacy-feature-inventory.md) に蒸留済みであり、個別調査の前にまずこれらを参照する
- yagura-dev はアーカイブとして保全する（削除しない）

継承する思想（不変の核）:

1. 「**Windows に syslog ツールを簡単に構築できる仕組みを作る**」が最上位の目的
2. Windows 管理者の既存スキル（SQL Server・AD・Windows サービス運用）の活用
3. 信頼ネットワーク前提 + opt-in セキュリティ強化
4. ADR による意思決定の永続化と、オーナー承認ゲート付きの AI 協働
5. スコープ制御（商用 SIEM 領域に踏み込まない。旧 ADR-0011 の NG 10 項目を継承）

バージョンは v0.1 から始め、v1.0 公開を目指す（v1.0 基準は旧 ADR-0011 の 5 件を土台に別 ADR で再定義する）。

## 帰結

- 良くなること: 構造的負債 4 点（DB 直書き / 信頼性の増築体質 / UI 品質 / ドキュメント体質)を初期設計で解消する機会を得る。プロジェクト全体が再び把握可能なサイズに戻る
- 悪くなること: 旧 v0.5 相当の機能に再到達するまでの期間、リリース可能な成果物が存在しない
- 受け入れるリスク: 技術的トラップの再踏。緩和策として、equivalence claim の実体検証・外部依存のライブ検証・実環境依存機能の lab 検証を開発規約（[docs/development/conventions.md](../development/conventions.md)）に組み込み済み

## 先送りにする場合の再評価トリガ

本 ADR に deferral はない。ただし再実装が旧 v0.2 相当（受信 + 永続化 + サービス化 + 最小 UI + セットアップ）に到達する前に開発が 2 か月以上停滞した場合、フルリライト方針自体を再評価する（選択肢 (a) への回帰を含む）。
