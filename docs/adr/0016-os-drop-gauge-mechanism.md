## 文脈と課題

[ADR-0002](0002-architecture-principles.md) 決定 2「発生箇所別のドロップ計測」は、取りこぼしを「OS ソケット / 内部バッファ / 流量制御 / 永続化失敗」に区別して計測することを求め、そのうち **OS レベルの取りこぼしの観測手段として .NET 標準 API（`System.Net.NetworkInformation.IPGlobalProperties` の UDP 統計）を v0.1 で採用**した。「アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る」という前提を、この標準 API のゲージが埋めるという設計だった。

この観測手段は、2 段階の実機検証で**当初の目的を果たさないことが確定した**（一次資料は architecture.md §4.2 と `tools/Yagura.Bench/results/2026-07-05-*`）:

1. **粒度の縮退（M4-4）**: 本 API は「本製品のソケット/ポート単位」ではなく**システム全体統計**であり、同居プロセスの UDP 破棄を含む。「本製品が取りこぼした」と断定できない近似値に留まる。
2. **観測手段としての機能不全（M7-2）**: さらに根本的に、**Windows 10.0.26200 + 本製品のワークロード（syslog UDP 受信）では、本 API は受信・破棄のいずれの変化も一切反映しない**。自己宛 UDP はカウント経路をバイパスし、クロスマシン検証では外部送信元のトラフィックすら計上されない（8,601 通の配送・100,000 通の溢れ送出でも増分 0。独立再実行で再現）。M4-4 で観測した「単調増加する生きた値」は、本ワークロードと独立に稀に発生する背景増分を見ていたものと再解釈する。

**これらの発見は当初、ADR-0002 §改訂履歴 1・2 に amendment（状態 accepted 維持）として記録された。** しかし M-8 の残作業（Issue #272）としてこの分類を [docs/adr/README.md](README.md) の判定基準に照らして再吟味した結果、**supersession が正しい**と判断した（根拠は次節の自己吟味）。本 ADR は、その再分類を実行し、OS レベル取りこぼし観測の実現手段を正式に置き換えるものである。

## amendment か supersession かの自己吟味（README 判定基準の適用）

[docs/adr/README.md](README.md) は「**既定を反転させるなら supersession、既定を保ったまま例外を足すだけなら amendment**」「疑わしければ supersession 側に倒す」「**最小の単位で反転の有無を検証**する」と定める。これを本件に適用する:

- **原則の軸で見れば**: ADR-0002 決定 2 の原則「OS 側のロスを無視しない」は維持されている（本 ADR も維持する）。この軸だけを見れば「手段の適用条件を限定しただけ」＝ amendment に見え、§改訂履歴 2 はこの見方を採った。
- **最小の単位で見れば（README が要求する検証）**: 決定 2 は OS レベルの取りこぼしの**観測手段として標準 API のゲージを提供する**ことを到達水準に含めていた。M7-2 の確定を受け、v0.1 の UI は**このゲージの値を表示せず**、状態画面の常時可視の説明のみに留める判断になった（ui.md §5.5）。すなわち「OS レベルを標準 API のゲージで観測・提示する」→「提示しない（動く環境での補助シグナルに降格）」という、**この観測手段の提供有無という最小単位で見れば Yes→No の反転**である。
- **先例との整合**: これは [ADR-0011](0011-app-auth-failure-backoff.md) が「一定期間**完全に拒否**するか否か」という最小単位の Yes→No 反転を根拠に amendment を supersession へ改めた判断、および ADR-0002 §改訂履歴ヘッダー自身の「Decision を変える場合は supersession」と同型である。

したがって「疑わしければ supersession 側に倒す」の原則に従い、**本件は amendment ではなく（部分）supersession とする**。原則そのものは覆らないため、supersede の対象は決定 2 のうち**実現手段（標準 API ゲージ）を採用した部分に限定**する（全面 supersession ではない）。

## 検討した選択肢

- **(a) amendment のまま据え置く（§改訂履歴 2 の分類を維持）**: 変更は最小で済む。しかし上記の最小単位検証で反転が確認され、README の「疑わしければ supersession」に反する。観測手段の提供有無という利用者・運用者から見える到達水準が変わっているのに、決定の履歴が「補足」の粒度に留まると、後続の設計者が「OS ゲージは決定 2 が要求する観測手段として有効」と誤読するリスクが残る。却下。
- **(b) 決定 2 の OS ゲージ観測部分のみを部分 supersession する（採用）**: [docs/adr/README.md](README.md) の部分 supersession 作法に従い、ADR-0002 の status は accepted のまま、一覧・状態行・該当決定に本 ADR への参照注記を加える。原則と決定 2 の他の到達水準は無傷で残す。反転した最小単位（観測手段の提供）だけを正確に置き換える。
- **(c) 決定 2 全体を supersession する**: 決定 2 のバックプレッシャ・ディスクスプール・流量制御・アプリ内カウンタ・at-least-once はいずれも有効であり、これらまで superseded 扱いにするのは不正確。過剰。却下。

## 決定

**ADR-0002 決定 2 のうち、「OS レベルの取りこぼしの観測手段として .NET 標準 API（`IPGlobalProperties` の UDP 統計）のゲージを v0.1 の到達水準として採用する」部分を supersede する。** 置き換え後の到達水準は次のとおり:

1. **原則は維持する**: 「アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る、という前提で設計する」は引き続き v0.1 の設計原則である。本 ADR は原則を撤回しない——原則を満たす**手段**を差し替える。

2. **OS レベル破棄の実効的な観測・緩和は、次の 3 手段が担う**（いずれも実装済み。architecture.md §4.2）:
   - **① 受信バッファ拡大（第一の緩和。M-2）**: ソケット受信バッファの既定を 1 MiB へ拡大し、OS 側ロスの大半を発生箇所別カウンタで計上できる形（アプリ内の観測可能な境界）へ転化する。M-2 実測で、64 KiB では総損失の相当割合（バースト 86.7%）が OS レベルの無計測ロスだったのに対し、1 MiB 以上で全実測条件でこれがゼロになることを確認済み。
   - **② ベンチの閉じた系での引き算導出（測定時）**: ベンチは送信数を自前で正確に計上し、OS バッファ以降は全カウンタで観測済みであることを利用して、残差を「OS ソケットバッファでの破棄（導出値）」として引き算で得る（`tools/Yagura.Bench` の `CounterReconciler`）。
   - **③ 送信元別の無音化検出（運用時。UI-4）**: 「本来届くはずの送信元からの受信が途絶した」ことの検出で、OS 以前のロスの運用時シグナルとする。

3. **標準 API のゲージは撤去せず、「動く環境では追加のシグナル」へ降格して残置する**: 実装（`IngestionMetrics` の ObservableGauge）は削除しない——将来の Windows ビルド／環境で機能する可能性（再評価トリガ参照）に備え、動く環境では補助シグナルとして働くため。ただし **v0.1 の UI はこのゲージの値を表示しない**（ui.md §5.5。常時可視の説明のみ）。UI に将来表示する場合は「システム全体値であり同居プロセスの破棄を含む」旨の常時可視の注記を必須とする。

4. **証拠能力の限界を明示する**: 本 ADR 以降、「OS 統計ゲージの値」を本製品の取りこぼしの証拠として引用してはならない（Windows 10.0.26200 + 本ワークロードでは機能しないことが確定しているため）。損失の証拠能力は上記 3 手段が担う。

## 帰結

- **良くなること**: 決定の履歴が実態と一致し、後続設計者が「OS ゲージは決定 2 が要求する有効な観測手段」と誤読しなくなる。原則（OS 側のロスを無視しない）を満たす実効的な手段（受信バッファ拡大・引き算導出・無音化検出）が、正式な到達水準として ADR に明記される。
- **悪くなること／受け入れるリスク**: OS バッファ**以前**（NIC ドライバ・OS プロトコルスタックの手前）のロスは、依然としてアプリから直接カウンタ計上できない——これは ADR-0002 決定 2 制定時からの限界（architecture.md §4.1 に明記済み）であり、本 ADR で新たに悪化するものではない。標準 API による「OS 統計との突合」という補完手段を失うが、その補完手段は M7-2 で機能しないことが確定しているため、失うのは名目上の手段であって実効的な観測能力ではない。
- **ADR-0006 基準 2「ロスは必ずカウンタに計上される」との整合**: 基準 2 の約束はアプリが観測可能な境界（Q1 以降のカウンタは全て健在）に対するものであり、OS バッファ以前のロスがアプリから直接計上できないことは当初からの明示的限界である。本 ADR は補完手段の実効性を訂正するものであって、基準 2 の約束の範囲を変えない。

## 先送りにする場合の再評価トリガ

ADR-0002 の再評価トリガ（OS 統計の再挑戦。(a)〜(d)）を本 ADR が引き継ぐ。特に **(d) M9 の lab 検証（Windows Server 実機）での本 API の動作確認**は未実施であり、本 ADR の検証範囲（単一ビルド Windows 10.0.26200）が v0.1 の実ターゲット（Windows Server 系列）を覆っていない限界を埋める予定の検証として残る。

- **(d) の結果が「Windows Server では本 API が機能する」だった場合**: 本 ADR の決定 3（UI 非表示・補助降格）を、当該 OS 系列に限って見直す amendment を起票する（ゲージの UI 表示の条件付き復活等）。原則・手段①〜③は維持する。
- **(a) 利用者からの誤読・誤診断報告の反復、(b) .NET 標準 API へのソケット/ポート単位 UDP 統計の追加、(c) 流量制御本実装での送信元別破棄観測との統合**: いずれも本 ADR の決定を見直す契機とする（ADR-0002 の該当トリガと同一）。

なお **(d) の実測結果の記録（Issue #272 checkbox 3）は M9 lab の実施を待つ**——本 ADR 起案時点では未実施のため、結果はトリガ発動時に追記する。
