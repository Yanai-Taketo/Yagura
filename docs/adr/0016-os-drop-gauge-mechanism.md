## 文脈と課題

> **用語注記**: 本 ADR および関連文書の「v0.1」は、ADR-0002 以来の設計文書が用いる**初期世代の到達水準ラベル**（実質「v1.0 公開前の現行世代」の意）であり、リリース番号 v0.1.0 そのものを指さない（本 ADR 起草時点の最新リリースは v0.4.0）。

[ADR-0002](0002-architecture-principles.md) 決定 2「発生箇所別のドロップ計測」は、取りこぼしを「OS ソケット / 内部バッファ / 流量制御 / 永続化失敗」に区別して計測することを求め、そのうち **OS レベルの取りこぼしの観測手段として .NET 標準 API（`System.Net.NetworkInformation.IPGlobalProperties` の UDP 統計）を v0.1 で採用**した。「アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る」という前提を、この標準 API のゲージが埋めるという設計だった。

この観測手段は、2 段階の実機検証で**当初の目的を果たさないことが確定した**（一次資料は architecture.md §4.2 と `tools/Yagura.Bench/results/2026-07-05-*`）:

1. **粒度の縮退（M4-4）**: 本 API は「本製品のソケット/ポート単位」ではなく**システム全体統計**であり、同居プロセスの UDP 破棄を含む。「本製品が取りこぼした」と断定できない近似値に留まる。
2. **観測手段としての機能不全（M7-2）**: さらに根本的に、**Windows 10.0.26200 + 本製品のワークロード（syslog UDP 受信）では、本 API は受信・破棄のいずれの変化も一切反映しない**。自己宛 UDP はカウント経路をバイパスし、クロスマシン検証では外部送信元のトラフィックすら計上されない（8,601 通の配送・100,000 通の溢れ送出でも増分 0。独立再実行で再現）。M4-4 で観測した「単調増加する生きた値」は、本ワークロードと独立に稀に発生する背景増分を見ていたものと再解釈する。

**これらの発見は当初、ADR-0002 §改訂履歴 1・2 に amendment（状態 accepted 維持）として記録された。** M-8 の残作業（Issue #272）としてこの分類を [docs/adr/README.md](README.md) の判定基準に照らして再吟味した結果、**supersession が正しい**と判断した（根拠は次節の自己吟味）。本 ADR は、その再分類を実行し、OS レベル取りこぼし観測の実現手段を正式に置き換えるものである。

**本 ADR の価値の正直な位置づけ**（ペルソナレビュー・クリスの指摘を受けた明確化）: §改訂履歴 2 は機能不全の実測事実を既に詳細に記録しており、本再分類が「進行中の実害」を止めるわけではない。本 ADR の価値は、①README 分類規約との一貫性の回復（決定の履歴が「補足」の粒度に留まると、決定変更の追跡可能性という ADR 制度自体の信頼が損なわれる）、②撤回により生じた**到達水準の再定義**（決定 2）と**運用時に残る観測の空白の明示**（帰結）を、amendment の散文ではなく決定として固定すること、の 2 点である。

## amendment か supersession かの自己吟味（README 判定基準の適用）

[docs/adr/README.md](README.md) は「**既定を反転させるなら supersession、既定を保ったまま例外を足すだけなら amendment**」「疑わしければ supersession 側に倒す」「**最小の単位で反転の有無を検証**する」と定める。これを本件に適用する:

- **原則の軸で見れば**: ADR-0002 決定 2 の原則「OS 側のロスを無視しない」は維持されている（本 ADR も維持する）。この軸だけを見れば「手段の適用条件を限定しただけ」＝ amendment に見え、§改訂履歴 2 はこの見方を採った。
- **最小の単位で見れば（README が要求する検証）**: 決定 2 の到達水準は「**v0.1 で OS 側ロスの実行時（runtime）観測手段を提供する**」ことを含んでいた——起案時の想定では標準 API のゲージがこれを担う。M7-2 の確定を受けた現在の実態は、**本番稼働中の OS 側ロスを観測する手段は提供されない**（後述のとおり、置き換え後の 3 手段はいずれも予防・測定時導出・別現象の検出であり、runtime 観測ではない）。すなわち「**v0.1 で OS 側ロスの runtime 観測手段を提供する: Yes → No**」という、決定 2 自身の到達水準の最小単位での反転である。
- **先例との整合と非対称の明示**: [ADR-0011](0011-app-auth-failure-backoff.md) も「一定期間完全に拒否するか否か」という最小単位の Yes→No 反転を根拠に amendment を supersession へ改めた。本 ADR も機構の実処置（ゲージ撤去 → 予防・測定時導出・無音化検出の 3 手段への置き換え。決定 2・3）を伴い、この点でも ADR-0011 と同型である。ただし本 ADR の supersession 判定は先例への類推ではなく、上記の**到達水準の反転それ自体**を根拠とする（ADR-0002 §改訂履歴ヘッダー自身の「Decision を変える場合は supersession」にも合致する）。

したがって「疑わしければ supersession 側に倒す」の原則に従い、**本件は amendment ではなく（部分）supersession とする**。原則そのものは覆らないため、supersede の対象は決定 2 のうち**実現手段（標準 API ゲージ）を採用した部分に限定**する（全面 supersession ではない）。

## 検討した選択肢

- **(a) amendment のまま据え置く（§改訂履歴 2 の分類を維持）**: 変更は最小で済む。§改訂履歴 2 は実測事実を詳細に記録済みであり、それを読んだ設計者の誤読リスクは実質的に低い。しかし上記の最小単位検証で「runtime 観測手段の提供」という到達水準の反転が確認され、README の「疑わしければ supersession」に反する。分類規約の一貫性（決定変更が supersession として追跡できること）は ADR 制度の信頼の基盤であり、明確な反転を「補足」に分類したままにする前例を作らない。却下。
- **(b) 決定 2 の OS ゲージ観測部分のみを部分 supersession する（採用）**: [docs/adr/README.md](README.md) の部分 supersession 作法に従い、ADR-0002 の status は accepted のまま、一覧・状態行・該当決定に本 ADR への参照注記を加える。原則と決定 2 の他の到達水準は無傷で残す。反転した最小単位（runtime 観測手段の提供）だけを正確に置き換える。
- **(c) 決定 2 全体を supersession する**: 決定 2 のバックプレッシャ・ディスクスプール・流量制御・アプリ内カウンタ・at-least-once はいずれも有効であり、これらまで superseded 扱いにするのは不正確。過剰。却下。

## 決定

**ADR-0002 決定 2 のうち、「OS レベルの取りこぼしの観測手段として .NET 標準 API（`IPGlobalProperties` の UDP 統計）のゲージを v0.1 の到達水準として採用する」部分を supersede する。** 置き換え後の到達水準は次のとおり:

1. **原則の生存範囲を精密に維持する**: 「アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る、という前提で設計する」は引き続き v0.1 の設計原則である。ただしその生存範囲を正直に書く——原則は**予防（受信バッファサイジングで OS 側ロスをアプリ内の観測可能な境界へ転化する）とオフライン測定（ベンチの引き算導出）として生存**し、**本番稼働中の OS 側残余ロスの runtime 観測は v0.1 では提供されない**。「観測対象とする」という当初の文言を無条件に維持するとの誤読を許さない。

2. **OS レベル破棄への実効的な対処は、次の 3 手段が担う**（いずれも実装済み。architecture.md §4.2。**各手段の可視性の別**——運用時に画面で見えるか・測定時のみか・自動で効くだけか——を併記する）:
   - **① 受信バッファ拡大（第一の緩和。M-2。自動で効く予防——運用者が値を見る画面はない）**: ソケット受信バッファの既定を 1 MiB へ拡大し、OS 側ロスの大半を発生箇所別カウンタで計上できる形（アプリ内の観測可能な境界）へ転化する。M-2 実測で、64 KiB では総損失の相当割合（バースト 86.7%）が OS レベルの無計測ロスだったのに対し、1 MiB 以上で全実測条件でこれがゼロになることを確認済み。**本番での OS 側ロス対策はこのサイジングで転化し切ることに依存する**（下記の帰結参照）。
   - **② ベンチの閉じた系での引き算導出（測定時のみ——本番では使えない）**: ベンチは送信数を自前で正確に計上し、OS バッファ以降は全カウンタで観測済みであることを利用して、残差を「OS ソケットバッファでの破棄（導出値）」として引き算で得る（`tools/Yagura.Bench` の `CounterReconciler`）。送信数を制御できない本番トラフィックには適用できない。
   - **③ 送信元別の無音化検出（運用時に画面で見える——ただし検出対象は送信元の途絶）**: 「本来届くはずの送信元からの受信が途絶した」ことの検出。**稼働中の送信元からの部分ロス（ストリームは続くが一部が落ちる）は検出対象外**である（下記の帰結参照）。

3. **標準 API のゲージは製品コードから撤去する**（機構処遇の裁定経緯: 初稿①ライブ残置 + description 刻印 → 第 2 回改稿②opt-in gate 化〔オーナー裁定 2026-07-18 前半〕→ **最終裁定③撤去〔オーナー裁定 2026-07-18 後半。オーナー自身の提案〕**。①②を退けた実質根拠は後述）:
   - **撤去対象**: `IngestionMetrics` の ObservableGauge（Meter `Yagura` の `yagura.os.udp.ipv4/ipv6.datagrams_discarded`）と、その登録・起動時ベースライン取得・取得不可環境への耐性の各経路、および専用テスト（`IngestionMetricsOsUdpStatsTests`）。**ベンチ側は無傷で維持する**——`tools/Yagura.Bench` の `OsUdpStatsProbe` は `IPGlobalProperties` をベンチ自身が読む独立実装であり、製品ゲージに依存しない（M7-2 のクロスマシン実測で受信側が実際に使用したのは、これと同種の独立読み取りである同梱スクリプト `stats-check.ps1`——下記 (i)）。`CounterReconciler` の引き算導出（決定 2 ②）も同様に無傷。
   - **①ライブ残置・②gate 化を退けた実質根拠**（第 2 回レビュー・クリスの「撤去の再評価」指摘を受けた実体検証の結果）: コードを残す理由として立てられていた 2 点は、いずれも検証で崩れた。(i)「トリガ (d) の lab 実測に製品コードが必要」——不要。M7-2 のクロスマシン実測で受信側が実際に使用したのは同梱スクリプト `stats-check.ps1`（`IPGlobalProperties` を直接読む独立 PowerShell。受信ソケット・一時 FW 規則込み）であり、**製品コードなしで統計の読み取り・判定が成立することが実証済み**（一次資料: `tools/Yagura.Bench/results/2026-07-05-cross-machine-udp-stats/`。ベンチの `OsUdpStatsProbe` は同種の独立読み取り器だが破棄合計のみを公開する——第 3 回レビュー・リサの実体確認によるツール同定の訂正）。(d) の lab 実測は同スクリプト（または受信総数観測を加えた probe 拡張）で行い、**判定手順**としての成立条件——溢れの誘発・受信総数側の観測——はチェックリスト⑦で固定する。(ii)「M-15（既定の仮想サービスアカウント権限で本 API 呼び出しが完結するかの検証）は製品の実行文脈を要する」——M-15 は製品が本 API を呼ぶから存在する要件であり、呼び出しを撤去すれば要件ごと消滅する（循環論法だった）。残る gate の便益（opt-in の検証的利用・トリガ (d) 陽性時の再有効化の速さ）は、前者がベンチツールで代替でき、後者は「陽性時はどのみち amendment 起票 + 実装が必要」なため実益がない。**期待値も非対称**——(d) 陰性なら撤去が完全に優位（gate は「キーと死んだ両経路の撤去裁定」という後始末まで残す）、陽性でも撤去の追加コストは小規模な再実装のみで、それは陽性時に必ず発生する amendment 実装に自然に含まれる。公開設定キーの異物性・有効/無効両経路の保守という gate の恒常コストは丸ごと回避される。
   - **再導入の条件と設計記録**: トリガ (d) が「Windows Server では機能する」を示した場合の amendment（下記）で再導入を検討する。その実装 PR では、製品の実行文脈での動作・権限確認（旧 M-15 相当）を併せて行う。撤去されたコードの完全な設計記録は git 履歴と本 ADR（および architecture.md §4.2 の実測記録）が保持する。
   - **v0.1 世代の UI はこのゲージの値を表示しない**（ui.md §5.5）——撤去後は「表示しない」が「計器が存在しない」へ強まる。状態画面の常時可視説明（「OS レベルのロスが画面に無い」ことへの説明）自体は運用者の導線として残すが、理由文は撤去ベース（予防①・測定時導出②・無音化検出③が対処を担う）へ更新する。
   - **実装申し送り**: 現行実装は常時ライブ（v0.1.0 以降の全リリース版に含まれる）であるため、本 ADR の accepted を受けて撤去を実装する（Issue 化する。docs のみの本 PR には含めない）。要件:
     - **撤去は accepted 後の次版リリースに含める**（オーナー裁定 2026-07-18。決定 4 の機構的強制の不在期間を最短で閉じる）。**機構的担保として、撤去 Issue には次版リリースのマイルストーン（リリースブロッカー相当）を設定する**（田中・クリスの指摘——決定 4 の時限注記の受容はこの拘束を前提とする）
     - 削除範囲は上記「撤去対象」のとおり。あわせて `UiText.OsUdpGaugeExplanation`/`Supplement` と SystemStatus の説明文言・`installer/lab/m9-3-lab-procedure.md`（チェックリスト⑦の独立読み取り手順へ）・`OsUdpStatsProbe.cs` の doc comment（チェックリスト⑧）・`IngestionCounterSnapshot.cs` の OS 統計ゲージ言及コメントを撤去ベースに更新する。**`UiText` の当該定数はリネームする**（撤去後に実体のない「Gauge」を含む名前を残さない——参照する `ViewerPageRenderTests` のアサーションも追従する。リサの質問への裁定）
     - 回帰テストは「`Yagura` Meter に `yagura.os.udp.*` 計器が存在しないこと」を対象とする（再導入の際に無自覚な復活を防ぐ。置き場所は `Yagura.Ingestion.Tests` の IngestionMetrics テストとして新設——削除する専用テスト `IngestionMetricsOsUdpStatsTests` の受け皿。リサの指摘）
     - 第 2 回レビューで条件・申し送りとされた **gate 前提の要件（有効化時失敗の警告ログ・診断系プレフィックス命名・手編集切替の証跡確認・M9 lab との順序制約）は、opt-in 機構の消滅により対象ごと解消**したことを記録する（田中・リサの各指摘への対応はこの解消をもって満たす）
   - **accepted 時の関連文書反映チェックリスト**（部分 supersession は反映先が散るため本 ADR 側に固定する——リサの指摘）: ① ADR-0002 の注記確定（状態行・決定 2 冒頭を「提案中」から superseded 断定形へ。**決定 2 本文の「動く環境では追加のシグナルという補助的な位置づけに留め」の一文も superseded の範囲に含まれることが分かる文言にする**） ② [docs/adr/README.md](README.md) 一覧の注記確定 ③ architecture.md §4.2 の「本ゲージは『動く環境では追加のシグナル』という位置づけへ降格する」ほかゲージ現在形記述を撤去ベース（計器は存在しない・実測記録と撤去の経緯は残す）へ書き換え ④ architecture.md §4.1.1 計器一覧から `yagura.os.udp.*` 2 行を削除 ⑤ architecture.md §4.2「権限要件」（M-15）の解消——実測待ち一覧からの**無言の削除ではなく「ADR-0016 決定 3 により要件消滅（(d) 陽性時の再導入実装に吸収）」の解消記録を残す形**で行う（一覧は「約束した検証の台帳」であり、理由の記帳なく項目が消えると実施/消滅を一覧単体で追跡できなくなるため——田中の指摘） ⑥ ui.md §5.5 の理由文（「計器自体を廃止しない理由」）を撤去ベースへ更新（常時可視説明の存置は維持） ⑦ `installer/lab/m9-3-lab-procedure.md` のトリガ (d) 実測手順を独立読み取り（`stats-check.ps1` / `OsUdpStatsProbe` 拡張）ベースへ更新——**更新後の手順は次の 3 点を必ず含む**（クリスの指摘: discarded 読み値単独では「陰性」と「溢れ未誘発による測定不成立」を区別できないため。判定可能性の成立条件は本 ADR が固定する）: (i) OS バッファ溢れの誘発手段（例: 受信バッファ最小構成 + 外部からのバースト送出）、(ii) 受信総数（`IncomingDatagrams`）側の観測（`netstat -s -p udp` 前後比較、または probe の `IncomingDatagrams` 読み対応拡張——M7-2 の決定的証拠は「配送成立なのに受信総数の増分 0」だった）、(iii) discarded 差分の観測。あわせて **E-1（dotnet-counters による製品ゲージ存在確認）は削除する**（撤去後は実施不能。(d) 陽性で再導入した場合の実装 PR で復活——リサの指摘） ⑧ `OsUdpStatsProbe` の doc comment 更新——remarks の「製品側 §4.2 のゲージの有効性はこの発見と両立する」（クロスマシン実測以前の認識・撤去後は存在しないゲージへの言及になる）と §4.2 参照を、撤去ベース + クロスマシン結果反映へ正す（(d) の実測手段系に属するファイルである以上、自己記述の正確性は判定の信頼性に直結する——田中・クリスの指摘） ⑨ architecture.md §4.1 発生箇所別カウンタ表の「OS 受信破棄（突合値）」行の撤去ベース化——**行の存廃を裁定し、同表の「全 18 種はすべて実装済み」の数え上げと整合させる**（行削除なら 17 種へ更新。リサの指摘） ⑩ ui.md §5.5「再評価の条件」の参照先更新（M-15 の lab 検証 → 本 ADR トリガ (d)。⑥と同じ更新で実施。リサの指摘）。①②は本 PR のマージ時または直後のフォローアップで、③〜⑩は撤去実装 PR で行う。

4. **証拠能力の限界を明示する**: 「OS 統計ゲージの値」を本製品の取りこぼしの証拠として引用してはならない。この禁止の射程は検証済みエンベロープ（**Windows 10.0.26200 + 本ワークロード**——機能しないことが確定している範囲）に基づくが、未検証環境でも粒度限界（システム全体値）により単独の証拠能力はない。再評価トリガ (d) が「機能する」を示した場合、当該環境系列に限って本項を見直す（自動では解けない——amendment を要する）。なお本項は損失の証拠能力に関する規範であり、architecture.md §4.2 の既存記述「本ゲージは監査記録ではない（監査要件の充足根拠として引用しない）」——security.md §4 の監査証跡との区別——とは**射程が異なる別個の禁止**である（両方が併存する。本項が既存記述を上書きするものではない）。**強制の実体**: 第一の強制は決定 3 の撤去が機構として担う——計器がコードごと存在しないため、誤読・誤引用の対象そのものが存在しない（gate 案の「既定では現れない」よりさらに強い、最終形の強制である）。ui.md §5.5 の常時可視説明が「なぜ画面に無いか」への運用者導線を補完する。**時限注記（クリスの指摘）**: 現行実装は常時ライブであり、撤去実装（決定 3 の実装申し送り。accepted 後の次版リリースに含める）が完了するまでの間、本項の機構的強制は存在せず UI 非表示と本文書の規範のみが働く——この不在期間を短く保つことが次版リリース要件の根拠である。

## 帰結

- **良くなること**: 決定の履歴が実態と一致し、後続設計者が「OS ゲージは決定 2 が要求する有効な観測手段」と誤読しなくなる。README 分類規約との一貫性が回復する。原則の生存範囲（予防 + オフライン測定）と観測の空白（下記）が、amendment の散文ではなく決定として固定される。また撤去（決定 3）により、機能不全の計器がコードごと消え、誤読・誤引用（決定 4）の対象そのものが存在しなくなる。公開設定面・保守対象コードも増えない（gate 案との比較は決定 3 参照）。
- **悪くなること／受け入れるリスク（運用時と測定時を分けて明示する）**:
  - **測定時**: 引き算導出（決定 2 ②）により OS ソケットバッファ破棄の定量観測は維持される。失うものはない。
  - **運用時**: **本番稼働中の OS ソケットバッファ破棄（受信バッファ超過分）を直接指すシグナルは存在しなくなる**。稼働中の送信元からの部分ロス（デバイスは送り続けているが一部が落ちる）は、①でバッファ内に収まれば Q1 以降のカウンタに現れるが、バッファを超えて OS 側で落ちた分は計上されない——③は完全途絶のみ検出し、②は本番では使えないため。**第一の防御は①のバッファサイジングで OS 側ロスを Q1 へ転化し切ることに依存する**。インシデント調査で「OS レベルで落ちたか」を事後に問われたとき、v0.1 は定量の答えを持たない（この限界は元ゲージが名目上担っていた能力の喪失として正直に記録する——粒度限界により実質的な証拠能力は当初から無かったとはいえ）。
  - OS バッファ**以前**（NIC ドライバ・OS プロトコルスタックの手前）のロスは、依然としてアプリから直接カウンタ計上できない——これは ADR-0002 決定 2 制定時からの限界（architecture.md §4.1 に明記済み）であり、本 ADR で新たに悪化するものではない。
  - **未検証環境での潜在的シグナルの放棄**: Windows Server 系列で本 API が機能する可能性は残っている（未検証）が、撤去によりその恩恵を受ける経路は製品から無くなる（検証したい場合はベンチの `OsUdpStatsProbe` を使う）。恩恵の解放は再評価トリガ (d) の lab 実測 → amendment（ゲージ再導入・UI 表示等の見直し）と再実装を経る。「万一動く環境での追加シグナルを残す」より「機能しないと確定した計器を持たない単純さ」を優先する判断である（オーナー裁定 2026-07-18）。
- **運用者向けの導線（申し送り）**: 運用時に OS レベルのロスを疑ったときの確認経路は「①受信バッファ設定値の妥当性確認 → Q1 破棄カウンタ（`yagura.ingestion.internal_buffer.dropped`）の監視 → ③送信元別の受信状況」である。operations.md 起筆時に、この確認手順と「OS ゲージの値が画面に無いのは正常である」旨の運用者向けの平易な 1 節を含めること（ui.md §5.5 の掲示だけでは状態画面を開かない運用者に届かない——ペルソナレビュー・佐藤の指摘）。
- **ADR-0006 基準 2「ロスは必ずカウンタに計上される」との整合**: 基準 2 の約束はアプリが観測可能な境界（Q1 以降のカウンタは全て健在）に対するものであり、OS バッファ以前のロスがアプリから直接計上できないことは当初からの明示的限界である。本 ADR は補完手段の実効性を訂正するものであって、基準 2 の約束の範囲を変えない。

## 先送りにする場合の再評価トリガ

ADR-0002 の再評価トリガ（OS 統計の再挑戦）を、本 ADR の実態（計器撤去・証拠引用禁止）に合わせて次のとおり引き継ぐ:

- **(d) M9 の lab 検証（Windows Server 実機）での本 API の動作確認**（主トリガ。未実施）: 本 ADR の検証範囲（単一ビルド Windows 10.0.26200）が v0.1 の実ターゲット（Windows Server 系列）を覆っていない限界を埋める予定の検証。M9 lab で**独立実装の統計読み取り（M7-2 実測で使用した `stats-check.ps1`、または受信総数観測を加えた `OsUdpStatsProbe` 拡張。決定 3・チェックリスト⑦参照）により**実測し、**結果は本 ADR の改訂履歴へ追記する**（結果の記録先を Issue に依存させない——Issue #272 クローズ後も本 ADR だけで追跡できるようにする）。**両側の分岐を対称に定める**:
  - **「Windows Server では機能する」だった場合**: 決定 3（撤去）・決定 4（証拠引用禁止）を当該 OS 系列に限って見直す amendment を起票する（ゲージの再導入・UI 表示の条件付き復活等。再導入の実装 PR では製品の実行文脈での動作・権限確認〔旧 M-15 相当〕を併せて行う——決定 3 参照）。原則・手段①〜③は維持する。
  - **「Windows Server でも機能しない」だった場合**: 「v0.1 世代の本番ターゲット OS において OS レベルの runtime 観測は恒久的に持たない」ことの正式受容を、本 ADR の改訂履歴に amendment として記録する（受容自体の裁定はその時点のオーナー判断。帰結の「運用時」の限界が恒久化する）。撤去は決定 3 で実施済みのため、追加の後始末は発生しない。
- **(a) メトリクスを直接購読する利用者からの誤読・誤診断の報告が反復した場合**: 撤去（決定 3）の完了後は計器自体が存在しないため、本トリガの発火源は消滅する——**撤去実装が次版リリースに載るまでの暫定期間に限り有効**とし、撤去完了をもって解消済みとして引き継がない。ただし解消が及ぶのは**撤去版以降のリリースのみ**であり、旧版（v0.1.0〜撤去前の版）を稼働させ続ける利用者の環境にはゲージが残る——旧版由来の誤読報告への標準回答は「撤去版への更新の案内」とする（クリスの指摘）。
- **(b) .NET の将来バージョンで標準 API にソケット/ポート単位の UDP 統計が追加された場合、(c) 流量制御の送信元単位の破棄観測との統合を設計する際**: いずれも ADR-0002 の該当トリガと同一のまま引き継ぐ。
