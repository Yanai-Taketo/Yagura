## 1. 設定の原則

- **初期設定はセットアップウィザードが生成する**。設定ファイルの手編集を前提としない——ただし手編集は引き続き正式に可能とする（ADR-0002 決定 5。Server Core 等、サーバ上でブラウザを使えない構成の管理手段でもある——ADR-0004 決定 2）
- **設定スキーマは additive-only**。キーの削除は「1 バージョン警告 → 次バージョン削除」の deprecation 階段を踏む（conventions.md。既存環境の設定ファイルを一方的に起動不能にしない）
- **不正値の扱いは設定キーごとの属性として宣言する**（反映方式 §3 と同じ機構に乗せる）。3 分類:
  - **起動失敗**: 受信の成立に不可欠なキー（受信ポートの値が不正等）。分類の判定はスキーマ定義時にキーごとに宣言する（§8 の規約）
  - **既定値で継続**: 上記以外の一般のキー。ただし数値項目の既定への差し替えは「ログを失わない方向」を第一義とし、フォールバックの発生時は**キー名・検出した不正値・適用した値の 3 点**をイベントログ警告（architecture.md §4.6）と UI の両方に明示する（「保持期間を 90 日にしたつもりが既定 30 日で動いていた」を実害の前に気づけるようにする）
  - **縮小側で継続**: **公開範囲・bind 先・認証関連のセキュリティ項目は、不正値のとき製品既定（開放側）へ落とさない**——より狭い側（localhost / loopback / 機能無効）で継続する。特に**管理リスナは、設定がどう壊れていても loopback 以外へ束縛されない**ことを不変条件とし、loopback 束縛の CI 回帰テストの検証対象に含める（[security.md](security.md) §1 L-4 として仕様化済み）
- **未知のキーは警告して無視する**（起動失敗にしない。deprecation 階段の途中版や新旧版の混在で起動不能を作らないため）。ただしタイポしたキーが「設定したつもり」で放置される事故に対し、**未知キーの一覧は起動時・再読み込み時の結果として UI とイベントログの両方に明示する**（警告を能動的に見せる——§3 の再読み込み結果表示と同じ場所）

## 2. 設定ファイルとデータ配置

- **既定のデータルートは `%ProgramData%\Yagura`** とし、配下に設定ファイル・組み込み DB・スプール・メタデータ領域を置く（プログラム本体は `Program Files` 配下。データとバイナリを分離する Windows の標準慣行に従う）。スプールと組み込み DB の置き場所はそれぞれ設定で変更できる（同一ディスクの I/O 競合を避けたい環境向け。architecture.md M-12 の「置き場所」はこれで確定し、上限値のみ実測待ちとして残る。database.md DB-2 も本節で確定）
- 設定ファイルは**データルート直下の `yagura.json`**（2026-07-05 オーナー確定）、形式は **JSON**（.NET 構成システムの標準）とし、**ウィザードと手編集は同一ファイルを対象とする**——二重の設定源（ウィザード用 DB と手編集用ファイル等）を作らない
- データルート配下の ACL は ADR-0004 決定 5 に従う（サービスアカウント + Administrators のみ。具体値はセキュリティ設計文書）
- **DB 接続資格情報は設定ファイルに平文で置かない**（ADR-0004 決定 5）: ウィザード経由の入力は常に DPAPI で暗号化して保存する（2026-07-06 実装済み）:
  - **暗号化表現**: `dpapi:<Base64>`（接頭辞 `dpapi:` により平文と機械判別できる。M5-3 で予約した形式をそのまま確定した）
  - **スコープ**: `DataProtectionScope.LocalMachine` + 固有 entropy（`Yagura:Storage:SqlServer:ConnectionString:v1` の UTF-8 バイト列）。暗号化（昇格ウィザード）・復号（起動時の設定読み込み）はいずれもサービスプロセス内で行われるが、管理者が手動でサービスアカウントを変更しても復号が壊れない machine スコープを採る（CurrentUser スコープはアカウント変更・プロファイル削除で復号不能になる）。**entropy は OSS のソースコードに含まれるため秘密ではない**——目的は同一マシン上の無関係なプロセスによる entropy なし復号の機械的成功を防ぐ名前空間の分離と、表現形式の版管理（`:v1`）であり、アクセス制御の主体はデータルートの ACL である（ADR-0004 決定 5）
  - **帰結（マシン束縛）**: 暗号化データは当該マシンに束縛される——**`yagura.json` を他マシンへコピーしても復号できない**（ADR-0004 決定 5 が明記する制約。構成バックアップの別サーバ復元・サーバ交換時は資格情報の再入力を要する）
  - **復号失敗（改ざん・破損・別マシンへのコピー）**: §8 の「接続文字列不備」と同じ「**SQLite へ縮小 + 強い警告**」で継続する（§1 の縮小側継続——起動を止めない。復旧は昇格ウィザードでの再入力）
  - **手編集の平文は受理し、自動書き換え（平文 → 暗号化への書き戻し）はしない**（2026-07-06 オーナー決定。本節の旧設計「起動時・再読み込み時に検出して暗号化表現へ書き換える」をこの決定で置き換えた——利用者のファイルを勝手に変更しない）: 手編集で平文パスワードが書かれた場合、起動時および設定の再読み込み時（§3）に検出して強い警告を出し続けるが、値はそのまま使用する（手編集ユーザーを壊さない）。警告は対象キー名のみを含み、**パスワード値そのものは記録しない**。暗号化表現への移行は昇格ウィザードでの再入力による
  - **限界の明示**: 平文のまま運用される手編集環境では、**平文が存在する期間に取得された構成バックアップには平文が残る**。ストレージ上の残留への対抗（上書き消去）も行わない（architecture.md §3.2.4 と同じ扱い）。利用者向けドキュメントに明記する（[§10](#10-利用者向けドキュメントへの申し送り) CF-D3）
  - **構成管理でファイルを配布する環境への注意**: DPAPI 暗号文はサーバに束縛されるため配布できない。資格情報キーを配布対象から外し、各マシンでウィザードにより投入する運用が必要である（CF-D3）。なお自動書き換えをしない決定により、旧設計で懸念した「配布 → 書き換え → 差分検知 → 平文再配布」のループは発生しなくなった
  - **実装参照**: 暗号化・復号は `DpapiConnectionStringProtector`、平文検出は `SqlServerConnectionStringCredentialGuard`（M5-3 / Issue #47 の枠組みを本実装で読み込み経路へ配線）、復号と警告は `YaguraConfigurationLoader`、暗号化保存は `PromotionWizardService`（いずれも `Yagura.Host`）。依存パッケージは `System.Security.Cryptography.ProtectedData` 10.0.9（当日ライブ検証の記録は `Directory.Packages.props`）

## 3. ウィザードと手編集の関係

- 本文書では、初期セットアップ・昇格・導入後の設定変更を担う管理画面群を総称して「ウィザード（管理画面）」と呼ぶ（画面の分割・名称は ui.md の管轄。§5 の資格情報の保持単位もこの総称単位で適用する）
- **競合の扱い**: 設定の保存は「読み込み → 変更 → 検証 → 保存」の全体書き換えとする。書き手は 2 者——ウィザード・手編集——であり（旧設計にあった第 3 の書き手「自動書き換え（平文の暗号化書き戻し）」は 2026-07-06 オーナー決定で行わないことにした——§2）、ウィザードは保存時にファイルの外部変更を検知した場合、上書きせずに再読み込みを促す（楽観的な競合検出。管理操作は localhost 限定で同時実行者は実質 1 人だが、「手編集した内容が画面操作で黙って消えた」事故は防ぐ）
- **実装参照（M8-4 / Issue #71）**: 初期セットアップウィザード（管理画面 `/admin/setup`。契約 `ISetupWizardService` / 実体 `Yagura.Host.Administration.SetupWizardService`）と本番昇格ウィザード（`/admin/promotion`。`IPromotionWizardService` / `PromotionWizardService`）の骨格を実装した。保存は既存の `YaguraConfigurationWriter`（全体書き換え + 原子的置換 + バージョントークンによる楽観競合検出）に合流し、競合時は上書きせず確認ステップからのやり直しを促す。反映方式の表示（§3）は `ConfigurationChangePlanner` の計算結果を適用結果として画面に出す。設定保存・切替実行は監査記録（security.md §4.3 の 2001/2003）の対象
- **反映方式を項目の属性として持つ**: 各設定キーは「即時反映 / リスナ再構成（接続の瞬断を伴う）/ サービス再起動（受信断を伴う）」のいずれかの反映方式をスキーマ上で宣言し、UI は変更時にそれを利用者へ表示する。**再起動を要する項目は最小に保つ**ことを設計目標とする
- **リスナ再構成の瞬断も観測対象とする**: 再構成中に届いた UDP は失われ得るため、**再構成の区間（時刻・対象リスナ・契機となった設定変更）を architecture.md §4.4 の受信断区間と同等の仕組みで記録・可視化する**（設定変更由来の断は唯一予告できる断であり、クラッシュより観測が弱くてはならない）。TCP 側の切断は TCP 接続断カウンタ（architecture.md §4.1）に計上する。無瞬断で適用できる項目（既存ソケットへの適用が可能な場合等）の特定と再構成の所要時間は実装設計で検証する（[CF-4](#9-確定待ち一覧)）
- **手編集の反映**は、サービス再起動または UI からの「設定の再読み込み」操作による（ファイル監視による自動反映はしない——編集途中の中間状態を読む事故を避ける）:
  - 再読み込みは**差分適用**とする——変更されたキーの反映方式に該当する処理のみを実行し、無関係なリスナを再構成しない
  - 変更に「サービス再起動」の項目が含まれる場合、**未反映のまま残る項目を再読み込みの結果として UI とイベントログに明示する**（§7 の「半分だけ適用を作らない」の手編集経路への拡張——気づかない中途状態を作らない）
  - **再読み込みは管理操作であり、監査記録の対象とする**。記録には旧設定との差分要約（変更キーと前後値。秘密情報キーは値をマスク）を含める——UI 経由・手編集経由が同じ証跡に合流する（ADR-0004 決定 7 の「管理操作全般」を手編集経路にも成立させる）
  - **UI を経由しない再読み込み手段（CF-5。✅ 確定 2026-07-16 オーナー裁定）**: **Windows サービスのカスタム制御コード方式を採用**する——`sc control Yagura 128` で設定の再読み込みを起動できる。新規の待受面（名前付きパイプ・追加ポート等）を作らず、制御の送信権限（SERVICE_USER_DEFINED_CONTROL。既定で管理者）を OS に委ねられ、Server Core・管理画面に入れない状況でも手編集を受信断なしで反映できる。SCM 経由は結果を直接返せないため、結果の確認はイベントログ（監査 2016 / 警告 1020・1021）で行う（実装: `YaguraWindowsServiceLifetime`）
- **実装参照（CF-4 層1。Issue #262）**: 再読み込みの実体は `ConfigurationReloadService`（UI = 管理画面「設定の再読み込み」/admin/reload と SCM カスタム制御の両方が合流する）。①ファイルを読み直して検証（**検証失敗——§1 の「起動失敗」分類の不正値——は何も適用せず旧設定のまま継続する**。起動時の fail-fast と異なる意図的な非対称——稼働中は「受信を止めない」を優先。イベント ID 1021）②前回適用時点との差分を計算（`ConfigurationChangePlanner`）③即時反映の口（`ImmediateConfigurationApplier`。合成ルートで登録——現時点: 流量制御 3 キー・`Retention:Days`/`ExecutionTimeOfDay`・`Audit:RetentionDays`・`Viewer:ReverseDns:Enabled`・`Ingestion:Rfc3164:DefaultTimeZone`・`Spool:QuotaBytes`）を持つ変更キーのみ適用④それ以外の変更キーは**再起動待ちとして再起動まで累積表示**する（イベント ID 1020 の警告 + UI。§3「未反映のまま残る項目の明示」）⑤実行を監査記録 2016 に残す
- **実装参照（CF-4 層2。Issue #262）**: 受信リスナ（UDP/TCP）の再構成は `IngestionPipeline.ReconfigureListenersAsync`。①options に変更のないリスナには一切触れない（瞬断なし）②変更されたリスナは「旧を graceful に停止（TCP は進行中フレームを Q1 へ吐き切る）→ 新 options で起動」の順で入れ替える——Q1・解析段・永続化段・スプール・計器・流量制御ゲートは共有のまま継続し、受信段のインスタンスだけが替わる③同一ポートの再構成は短い瞬断を伴い、**瞬断区間は受信断のシステムイベント（`downtime.listener-reconfigure`）として記録する**（書き込みは書き込みゲートで直列化——Issue #151 の単一 writer 契約）④**新構成の bind 失敗は旧構成で復旧**（旧構成の維持）⑤復旧も失敗した場合はリスナ停止のまま **CF-6 の定期再試行**（仮値 30 秒間隔）へ移行する。**TLS 受信（`Ingestion:Tls:*`）は対象外**（証明書ストア参照・秘密鍵アクセス権付与を伴うため宣言どおり再起動——§8 表）

## 4. 既定ポート・bind 先・ファイアウォール

### 4.1 受信（syslog）

- **UDP 514・TCP 514 を既定**とする（syslog の標準ポート。多くのネットワーク機器の既定と合わせ、送信元の設定変更なしで届くことを優先する——ADR-0004 決定 2 の「導入体験を守る」の具体化）
- TLS 受信（syslog over TLS。RFC 5425。opt-in・既定無効）は **TCP 6514**（RFC 5425 の標準ポート）を既定とする（実装済み。Issue #137。設定キー `Ingestion:Tls:*`——§8 参照）
- **bind 先は既定で全インターフェース・IPv4/IPv6 両対応**とする（Issue #133）。既定値 `BindAddress = "::"`（IPv6 ワイルドカード）を `Socket.DualMode = true` の単一ソケットで bind し、IPv4・IPv6 双方の送信元から受信する（Windows は DualMode を標準サポート——閲覧・管理リスナの Kestrel 側と同じ仕組み。実装は `Yagura.Ingestion.Net.DualStackBindAddress` / `UdpSyslogListener` / `TcpSyslogListener`）。設定で特定アドレス（IPv4 または IPv6 のいずれか一方）に縮小できる
  - **`0.0.0.0` 明示指定 = IPv4 専用への後方互換の逃げ道**: `BindAddress` に `0.0.0.0`（IPv4 ワイルドカード）を明示指定した場合のみ、IPv4 単独ソケットで bind し IPv6 を受けない。v0.1〜v0.2（既定が `0.0.0.0` だった時期）からの手編集設定を持つ環境が、アップグレード後も意図せず IPv6 を新たに受け始めないようにするための設計判断——**縮小方向の変更を伴わない設定は、アップグレードで挙動を変えない**（§1 の additive-only の精神を bind 先の意味づけにも適用した）
  - **送信元アドレスの表現の正規化**: DualMode ソケットが受ける IPv4 送信元は、OS レベルでは IPv4-mapped IPv6（`::ffff:x.x.x.x`）として現れる。これをそのまま記録すると、同一の IPv4 送信元が「`0.0.0.0` 時代のログ」と「DualMode 化後のログ」で異なる文字列表現に分裂し、検索・送信元別集計・逆引き（[ADR-0007](../adr/0007-reverse-dns-display.md)）の一致判定が壊れる。受信段は `ReverseDnsResolver`（ADR-0007 決定 2）と同じ規約——IPv4-mapped IPv6 は `IPAddress.MapToIPv4()` で純粋な IPv4 表記へ正規化してから記録する——を適用し、アップグレード前後で IPv4 送信元の表示形式を変えない
  - **IPv6 スタックが無効な環境での挙動（既定と明示で分ける）**: Windows の IPv6 スタックが使えない環境（レジストリ `DisabledComponents`（KB929852）による無効化・NIC の IPv6 コンポーネント無効化等——一部の統制要件で行われる構成）では `::` の DualMode ソケット作成が失敗する。`Socket.OSSupportsIPv6` の事前チェックと bind 実試行での `AddressFamilyNotSupported` の両方で検出し、次のとおり分岐する:
    - **既定値（`BindAddress` キー未設定）**: **IPv4 ワイルドカード（`0.0.0.0`）へ自動で縮小して起動を継続**し、警告（イベントログに届くレベル）で「IPv6 が無効のため IPv4 のみで受信する」ことを明示する——既定構成の受信可用性を IPv6 スタックの有無に依存させない（IPv6 無効機で Yagura 全体が起動不能になることを許さない）
    - **明示指定（利用者が `::` を設定ファイルに書いた場合）**: **縮小せず、原因と復旧手順（IPv6 の有効化 / キーの削除 / `0.0.0.0` への変更）を含むエラーで起動を失敗させる**——利用者の明示意図（IPv6 でも受信する）を黙って裏切らない（fail-fast）
    - **この自動縮小の条件は「IPv6 が使えない」に限る**: ポート競合等の別要因の bind 失敗は縮小せずそのまま失敗させる——ポート事故が黙って「IPv4 のみ受信」へ化けることを防ぐ
- **指定した bind 先が使用できない場合**（NIC 交換・IP 変更後の起動等）: そのリスナは**開かずに縮小側で継続する**（全インターフェースへフォールバックしない——§1 の縮小側原則。意図せぬ公開拡大を設定事故で起こさない）。開けなかったリスナは能動通知と UI で強く可視化する
- **起動失敗との境界**: §1 の「起動失敗」は値の不正（静的検証で判定できるもの）に限る。**値は正しいが環境要因で bind できない場合は、全リスナが開けない場合を含めて縮小継続とする**——管理リスナは loopback ゆえ常に開けるため、UI からの復旧動線が残る（architecture.md §1.2 のスプールなし縮退と同じ「可視化された縮退 > 起動失敗」の判断）。**bind 失敗後は定期的に再試行する**（サービス起動が NIC のアドレス確立より先行する起動時競合で「再起動のたびに手動回復が要る」状態を作らない。再試行間隔は [CF-6](#9-確定待ち一覧)）

### 4.2 閲覧 UI と管理

- **閲覧リスナ**: 既定で LAN に公開する（認証なし・読み取りのみ。ADR-0004 決定 2 の決定事項）。公開範囲は設定キー `Viewer:PublicAccess`（`Lan` 既定 / `LocalhostOnly`）で選択できる。**既定ポートは 8514**（[CF-1](#9-確定待ち一覧) 確定済み。Issue #51・2026-07-05 オーナー決定）
- **管理リスナ**: 書き込み系（ウィザード・設定変更・昇格等）は既定で **loopback 専用リスナ**に束縛する（`127.0.0.1` / `::1` の両系。ADR-0004 決定 2 の第一候補方式）。**既定ポートは 8515**（CF-1 確定済み）。ポートは設定キー `Admin:HttpPort` で変更できるが、**loopback bind 先自体（127.0.0.1/::1）を変える設定キーは設けない**——設定がどう壊れていても loopback bind 先そのものへ束縛されない（§1 の不変条件は維持。loopback 束縛の CI 回帰テストは [security.md](security.md) §1 で仕様化済み）
- **管理リスナのリモートバインド（[ADR-0010](../adr/0010-admin-ui-authentication.md) Phase 2。opt-in。既定 `false`）**: 上記の loopback bind は既定・常時のまま**置き換えず**、設定キー `Admin:RemoteBinding:Enabled` を `true` にすると、**別ポート**（既定 8516。設定キー `Admin:Https:Port`）へ **HTTPS 必須**の追加 bind エントリ（全インターフェース）を張る。同一ポートに loopback（平文）と全インターフェース（ワイルドカード bind）を共存させないのは、OS の bind 制約（ワイルドカード bind と特定アドレス bind は同一ポートで共存できない）と、ADR-0010 決定 4「loopback 経由の管理リスナは HTTPS の対象外のまま残る」（証明書事故時も loopback からの復旧を維持する）の両方による設計判断——別ポートにすることで、loopback は常に平文 HTTP のまま、リモート面のみ独立して HTTPS 必須にできる
  - **fail-closed 不変条件**: `Admin:RemoteBinding:Enabled = true` かつ、①認証（`Admin:Authentication:Windows:Enabled` / `Admin:Authentication:App:Enabled` のいずれも無効）または②HTTPS（`Admin:Https:Enabled = false`、または `Admin:Https:CertificateThumbprint` が未設定・不正形式）のいずれかが未構成の組み合わせは、§1「起動失敗」として起動時に拒否する（[security.md](security.md) §2.4 の fail-closed 不変条件・イベント ID 1012）。**「loopback bind 先を変える設定キーは設けない」という §1 の不変条件そのものは変わっていない**——本 opt-in が追加するのは新しい bind 先（別ポートの全インターフェース）であり、既存の loopback bind の意味を変えるものではない
  - 拇印が有効な形式で証明書ストアに実在するかは環境依存（実機のみで判定できる）ため、fail-closed の対象外とし、**縮小継続**（リモート bind エントリのみ開かず、loopback は影響を受けない。起動時警告イベント ID 1013）として扱う（§4.1「指定した bind 先が使用できない場合」と同型の扱い）。詳細は [security.md](security.md) §2.4・§2.5
- **CF-1 決定記録（2026-07-05 オーナー選択。Issue #51）**: 閲覧リスナ既定ポート = **8514** / 管理リスナ既定ポート = **8515** に確定した。選定基準は「口頭で伝えやすい番号」（「syslog の 514 の頭に 8」で電話伝達可能）。衝突調査（2026-07-05 IANA レジストリ当日照合）: 8514/8515 とも IANA 未割当。8514 を代替 syslog 受信ポートに使う他製品の構成例（Splunk/Graylog）は存在するが、Yagura 自身が syslog サーバであり同居衝突は実質起きにくいと判断。5140/5141 は Graylog 公式ガイドの非特権 syslog 受信ポートの定番のため「受信ポート」の連想が強く却下。8140 は Puppet 割当済みで却下
- **ルートのリスナ帰属の実装方式（M6-1）**: 閲覧系ルート（`MapYaguraWebViewer`）と管理系ルート（`MapYaguraAdmin`）はいずれも同一 Kestrel プロセス内の別ポートへの `Listen` で受信し、`HttpContext.Connection.LocalPort`（接続を実際に受け付けたローカルポート。クライアントが偽装できない）をミドルウェアで判定して、管理系エンドポイントが管理ポート以外から実行されないよう強制する。ASP.NET Core の `RequireHost` は HTTP `Host` ヘッダで判定するため、非 loopback 接続が `Host: x:8515` 相当のヘッダを送るだけで判定を回避でき、防御として採用できない（公式ドキュメントも偽装可能性を明記している）。**閲覧系ルートは管理リスナからも到達できる**（ui.md §4 の不変条件は「閲覧リスナに書き込み系を置かない」であり、管理リスナは loopback 限定のため閲覧系の同居は安全側——管理者がローカルで全部見られることはむしろ自然）

### 4.3 ファイアウォール規則 — インストーラの責務

「入れたのにログが来ない」の第一原因は Windows ファイアウォールである。インストーラの責務を次のとおり定める:

- インストーラは**受信ポート（UDP/TCP 514）と閲覧ポートの受信許可規則を作成する**（規則名は「Yagura」で始まる名前空間に統一し、利用者が一覧で識別・監査できるようにする）
- 規則の**ネットワークプロファイルは Domain + Private 限定**とする（Public は含まない。信頼ネットワーク前提の製品思想（ADR-0004 決定 2）に沿い、ノート PC 等が公衆ネットワークへ接続した際に受信・閲覧ポートが露出しないようにする。Issue #125）
- **インストーラは完了時に閲覧 URL を利用者へ案内する**（完了画面での表示とスタートメニューへのリンク作成を責務とする。表示の詳細は実装設計・ui.md）——ファイアウォールを開け、既定ポートで受信し、**最後に「どこを開けばログが見えるか」まで案内して 30 分動線を閉じる**
- 規則の作成はインストーラ上で**オプトアウトできる**（ファイアウォールを集中管理（GPO 等）している環境向け。オプトアウトした場合に必要な手動設定は利用者向けドキュメントに記載する——[§10](#10-利用者向けドキュメントへの申し送り) CF-D1）。**オプトアウトの選択と、作成した規則の一覧はインストール記録として残し、初回起動時にイベントログへ転記する**（「なぜこのサーバには規則がないのか」に証跡で答える）
- 管理リスナは既定（loopback 束縛のみ）では受信許可規則を要しない（作らない——不要な穴を開けない）。**リモートバインド opt-in（`Admin:RemoteBinding:Enabled`。[ADR-0010](../adr/0010-admin-ui-authentication.md) Phase 2）を有効化した場合のみ**、リモート HTTPS 用ポート（既定 8516。`Admin:Https:Port`）の受信許可規則が必要になる——ただしインストーラは opt-in の有効化時点（インストール後の設定変更）を検知できないため、**この規則はインストーラが自動作成しない**。有効化する運用者が手動でファイアウォール規則を作成する（規則名は「Yagura」始まりの名前空間に合わせる。手順は[§10](#10-利用者向けドキュメントへの申し送り) CF-D7）。管理リスナのネットワークプロファイルも受信ポート・閲覧ポートと同じ Domain + Private 限定を推奨する（Public は含めない——信頼ネットワーク前提の製品思想に沿う）
- アンインストール時に作成した規則を削除する
- **リスナの実ポートと Yagura 名前空間の規則のポートの不一致は、起動時とポート変更の適用時に検出して警告する**。ファイアウォールでの drop はアプリのカウンタにも OS のソケット統計にも現れない観測の完全な死角であり、「ポートを変えたのに届かない」を無音で固定化させないため、この照合は規則追従の方式（自動 / 手動——[CF-2](#9-確定待ち一覧)）にかかわらず必須とする
- **実装参照（M9-1。Issue #74）**: 本節の責務のうち規則作成（UDP/TCP 514・閲覧 8514。規則名は「Yagura」始まり）・オプトアウト（MSI プロパティ `YAGURA_FIREWALL` + セットアップ画面のチェックボックス）・インストール記録（データルート直下の `firewall-rules.ini` に選択と規則一覧を記録）・完了画面の閲覧 URL 案内とスタートメニューリンク・アンインストール時の規則削除を installer/（WiX 7 MSI）として実装した。初回起動時のイベントログへの転記と、不一致検出・警告（上記）は Host 側の後続 Issue で実装する。実機での適用挙動の検証は M9-3（lab）
- **実装参照（Issue #125）**: Domain + Private 限定は installer/Firewall.wxs で実装済み。各系統 1 規則（計 3 規則）のまま、`Profile="[YaguraFwProfile]"` のプロパティ間接参照でビットマスク整数 `3`（Domain 0x1 | Private 0x2）を Firewall 拡張の custom action へ渡し、複合プロファイルの単一規則を作る。名前指定の列挙値（`domain` 等）は単一値のみで、同名規則の 2 本分割は custom action の Name 衝突上書きにより成立しない——方式選定の実体検証の詳細は Firewall.wxs 冒頭コメントを参照
- **実装参照（Issue #131）**: 管理画面（8515・loopback 固定）への入口もインストーラが用意する。スタートメニューに「Yagura 管理」（`http://localhost:8515/admin`。「Yagura ログ閲覧」と並ぶ・無条件）、デスクトップに「Yagura」（閲覧 UI。セットアップ画面のチェックボックスまたは MSI プロパティ `YAGURA_DESKTOP_SHORTCUT` で選択可・既定 ON）を作成し、完了画面の案内には閲覧・管理の両 URL を併記する。デスクトップショートカットの作成先は perMachine インストール（ALLUSERS=1）のため **`%PUBLIC%\Desktop`（全ユーザー共有のパブリック デスクトップ）**であり、インストール実行者個人のデスクトップではない——サーバ機を複数の管理者アカウントで運用する環境（RDP 等）では全員のデスクトップに現れる。この性質はセットアップ画面の注記で利用者にも明示する。管理向けのファイアウォール規則は引き続き作らない（loopback 束縛の不変条件はそのまま——本入口は「サーバ自身から管理画面を開く動線」のみを足す）

## 5. 管理者資格情報の扱い — 長時間処理での再入力方針

ADR-0004 決定 4（都度入力・使用後破棄・保持しない）の「長時間処理」への適用を定める:

- **保持の単位は「ウィザードの 1 実行」**とする: 資格情報は処理の開始時に入力させ、その処理（複数ステップを含む）が完了または失敗するまでメモリ内でのみ保持し、終了時に破棄する。ステップごとの再入力は求めない
- **資格情報はサーバ側の永続的なセッション状態（§7）に含めない**——進行状態の永続化と資格情報の非永続を明確に分離する。circuit 喪失・タイムアウト後の再開では、確定済みステップは残るが**資格情報は再入力を要する**
- **無操作タイムアウト**: 入力途中で放置されたウィザードセッションは、一定時間の無操作で資格情報を破棄し、再開時に再入力を求める（時間は仮値: 15 分。[CF-3](#9-確定待ち一覧)）
- **昇格の準備フェーズ（database.md §6.1）のように日をまたぐ中断・再開を許す処理では、再開のたびに再入力する**のが仕様である（「1 実行」は連続した操作セッションを指し、数日にわたる保持はしない）
- **長時間バッチ（将来の蓄積ログ移行等）は「実行中の再入力」を設計しない**: 必要な権限・接続の検証を開始前に完了させ（準備フェーズと同じ思想）、実行開始後は資格情報の追加入力なしで完走できる構成とする——深夜バッチが資格情報の入力待ちで朝まで停止する事故を防ぐ
- 資格情報をディスク・設定ファイル・ログ・監査記録のいずれにも書かない（監査記録は「管理者資格情報を使用した」事実のみを残す）
- **実装参照（M8-4 / Issue #71）**: 本番昇格ウィザードの接続文字列は `PromotionWizardService` のメモリにのみ保持し、切替の完了で破棄する。無操作タイムアウト（CF-3 仮値 15 分 = `WizardSessionDefaults.InactivityTimeout`）で資格情報のみを破棄し、確定済みの進行状態（処分の選択等）は保持する——再開時は再入力を求め、その旨を画面に明示する。接続検証の実施は「使用した事実 + 成否」のみを監査記録（security.md §4.3 の 2002）に残す。切替実行時の設定ファイルへの保存は §2 の DPAPI 暗号化表現（`dpapi:<Base64>`）で常に行う（2026-07-06 実装。ウィザードには平文が設定ファイルへ書かれる経路が存在しない。暗号化自体が失敗した場合は切替を失敗させ、平文保存へはフォールバックしない）
- **資格情報の粒度（昇格ウィザードの接続組み立て化に伴う精密化。database.md §6.1）**: 本節が破棄の対象とする「資格情報」は**パスワードのみ**とする。サーバ名・データベース名・認証方式・ユーザー名・サーバ証明書の信頼、およびパスワードを含まない接続文字列（直接入力方式——`Password`/`Pwd` 等のパスワード系キーの記載は正規化パースの上で拒否される）は秘密ではなく、**進行状態として保持する**（無操作タイムアウトで消えない——再開のたびにサーバ名から打ち直させない）。タイムアウトはパスワードとともに検証済み状態・実行トークンも無効化する（検証は「現に保持している入力」に対してのみ有効、の従来原則を維持。Windows 統合認証ではパスワードが存在しないため、タイムアウト後の再開は再入力なしの再検証のみで足りる）

**管理 UI 認証（[ADR-0010](../adr/0010-admin-ui-authentication.md) Phase 1）のアプリ独自 ID/パスワードとの違い**: 本節が扱う「長時間処理での再入力方針」は本番昇格ウィザードの SQL Server 接続資格情報を対象とする。管理 UI 認証のアプリ独自認証アカウントのパスワードは性質が異なる（サーバ側に永続化するハッシュ値であり、都度入力・使用後破棄の対象ではない）——保管方式は `PasswordHasher<TUser>`（PBKDF2）によるハッシュ化のみで、平文はメモリ上の入力値受け渡し以外に一切保持しない。詳細は security.md §2.4 を参照。

## 6. HTTPS 証明書の取り扱い（opt-in 強化。v1.0 までに提供）

本節の射程は **Web UI（閲覧リスナ）の HTTPS 証明書**である。TLS 受信（§4.1 の TCP 6514）用の証明書の参照方式・期限切れ挙動は [security.md](security.md) §6 で確定済み（HTTPS = 停止 / TLS 受信 = 継続、という非対称の根拠を含む）。

- **証明書は利用者の持ち込みを基本とし、Windows 証明書ストア（ローカルコンピューター）からの参照を唯一の指定方式とする**。ウィザードはストア内の証明書一覧から選択させる（拇印を直接入力させない）。Windows 管理者の既存スキル（証明書スナップイン・AD CS）にそのまま乗る方式であり、PFX ファイルパス + パスワードを設定に持つ方式は採らない（ファイルとパスワードの管理という新しい漏洩面を作らないため。ストアへの取り込み手順は operations.md——[§10](#10-利用者向けドキュメントへの申し送り) CF-D2）
- **秘密鍵へのアクセス権**: サービスは仮想サービスアカウントで動作する（ADR-0004 決定 4）ため、証明書を選択しただけでは秘密鍵を読めない。**ウィザードは証明書の選択時に、サービスアカウントへ当該証明書の秘密鍵の読み取り権限のみを付与する**（広い権限へ逃げない）。付与は監査記録の対象とする。確認手順は CF-D2 に含める
- **自己署名証明書の生成支援は提供しない**（ADR-0004 決定 3 が検討を委任した項の結論）:
  - 主理由: 自己署名はブラウザ警告を常態化させ、「警告を無視して進む」習慣を利用者と閲覧者に育てる——TLS を opt-in した動機（通信の真正性）を自ら損なう。製品の機能として支援することは、この運用を製品が推奨する形になる
  - **主理由の限界も正直に記す**: 参考手順は文書で提供する（CF-D2）ため、抑止の実効は「機能のワンクリック」と「手順書のコマンド」の手間の差にとどまる。それでも「製品の推奨」と「文書に注意書き付きで載る参考手順」の位置づけの差に意味がある、という判断である
  - 補助理由: AD CS を**導入済みの**環境では証明書の発行・配布の正道が既にあり、生成支援は不要。**AD はあるが AD CS を建てていない環境（本製品のターゲットで最多と想定される）には、この理由は当てはまらない**——この環境向けの選択指針（商用 CA / 社内 CA / 自己署名の使いどころと注意）を CF-D2 の必須内容とする
  - 却下した対案——生成支援を内蔵する: 導入は楽になるが、上記の害が「製品が推奨した」形になる。提供しないことで生じる導入の壁は、手順の文書化で緩和する
- **証明書の期限切れ・失効時は HTTPS リスナを停止し、HTTP へは落とさない**: TLS を opt-in した環境の閲覧が証明書事故で黙って平文に戻ることを許さない（AD 認証 opt-in 時の認証通信・リモート管理も HTTPS 前提——ADR-0004 決定 3——のため、自動的に閉じる）。平文の syslog 受信は影響を受けず継続する。閲覧の停止は能動通知で可視化し、**期限の接近を事前に警告する**（architecture.md §4.6 の能動通知に「証明書期限の接近」を追加する）。復旧手段は証明書の更新（または利用者の明示操作による HTTPS 無効化 = opt-in の解除）である
- **復旧の動線は HTTPS 停止に巻き込まれない**: **管理リスナの loopback 面は HTTPS 化の対象外のまま残る**（防御の主体は loopback 束縛——ADR-0004 決定 2。ADR-0004 決定 3 の「HTTPS 前提」はリモートからの管理開放の条件であり、loopback 内の管理通信には適用されない）。したがって証明書の更新・HTTPS 無効化の操作は、（リモート）HTTPS リスナが止まっていても管理リスナの loopback 面経由で常に実行できる。**帰結**: リモート管理 opt-in 環境では、期限切れ時の復旧はサーバ上の操作（RDP + loopback）が前提になる——この帰結は利用者向けドキュメントに明記する（[§10](#10-利用者向けドキュメントへの申し送り) CF-D2）
- **管理リスナのリモート HTTPS（[ADR-0010](../adr/0010-admin-ui-authentication.md) Phase 2）は本節の射程外**: 上記はいずれも閲覧リスナの HTTPS を対象とした記述である。管理リスナのリモートバインド面（`Admin:RemoteBinding:Enabled` opt-in）に要求される HTTPS は、**証明書ストアからの選択・秘密鍵アクセス権の付与・自己署名の生成非提供という参照方式こそ本節と同型**だが、設定キー（`Admin:Https:*`）・最低 TLS バージョンの明示固定（TLS 1.2 以上・1.3 優先）・期限切れ時の挙動（リモート HTTPS bind のみ縮小継続——プロセス全体は止めない）の詳細は §4.2 と [security.md](security.md) §2.4・§2.5 が管轄する（設定キーの独立化——決定 4「片方の変更が他方へ黙って波及しない構造を保つ」——により、閲覧 UI の HTTPS を本節が単独で確定できる構造は崩れていない）

## 7. 書き込み系画面のネットワーク瞬断時挙動

Blazor Interactive Server の circuit は瞬断で失われ得る（ADR-0003 受け入れリスク）。ウィザード・設定変更など書き込み系画面の挙動を定める:

- **ステップ完了ごとにサーバ側へ確定する**: ウィザードの進行状態は circuit のメモリではなくサーバ側の永続的なセッション状態に置き、再接続・circuit 喪失後の再入で**確定済みステップから再開できる**。入力途中（未確定フォーム）の内容は失われることを許容し、再開時に「どこから再開しているか」を明示する（黙って空フォームに戻さない）。**資格情報はこの永続状態に含めない**（§5）
- **確定操作は一回性を保証する**: 設定の適用・昇格の実行などの確定操作は冪等トークン（同一操作の再送を検出する識別子）を伴い、瞬断 → 再送で二重適用されない
- **「半分だけ適用」を作らない**: 設定変更の適用は「検証 → 適用 → 記録」を単一の確定単位として実行する。瞬断が適用の途中状態を残さない
- v0.x の管理操作は localhost 限定（RDP 内のループバック接続）のため瞬断は稀だが、認証 opt-in 後のリモート管理では常態的なリスクになる——設計は最初から上記を前提とする
- **実装参照（M8-4 / Issue #71）**: ウィザードの進行状態はサーバ側（Host のシングルトンサービス）に保持し、circuit 喪失後の再入は確定済みステップから再開する（再開位置は画面に明示する——黙って空フォームに戻さない）。確定操作（設定の適用・切替の実行）は確認ステップで発行する冪等トークンを要求し、同一トークンの再送では再適用しない（一回性の保証はテスト `SetupWizardServiceTests` / `PromotionWizardServiceTests` が固定する）。保存自体は原子的置換 1 回のため「半分だけ適用」は生じない。サービス再起動をまたぐ進行状態のファイル永続化は骨格に含めない（初期セットアップは分レベルで完了する操作であり保持すべき資格情報もない。日をまたぐ中断・再開の要否は CF-3 と合わせて実利用で評価する）

**設定キー追加の規約**（外部貢献を含む。CONTRIBUTING からも参照する）:
