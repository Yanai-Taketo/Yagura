## 1. 設定の原則

- **初期設定はセットアップウィザードが生成する**。設定ファイルの手編集を前提としない——ただし手編集は引き続き正式に可能とする（ADR-0002 決定 5。Server Core 等、サーバ上でブラウザを使えない構成の管理手段でもある——ADR-0004 決定 2）
- **設定スキーマは additive-only**。キーの削除は「1 バージョン警告 → 次バージョン削除」の deprecation 階段を踏む（conventions.md。既存環境の設定ファイルを一方的に起動不能にしない）
- **設定エラーで受信を止めない**: 起動時の設定検証で不正を検出した場合、その項目が受信の成立に不可欠でない限り、**安全側の既定値で継続し、イベントログ警告（architecture.md §4.6）とUI で強く可視化する**。受信自体を成立させられない不正（受信ポートの値が不正等）のみ起動失敗とする。「黙って既定に落ちる」ことと「設定ミス 1 つで受信断が固定化する」ことの両方を避ける
- **未知のキーは警告して無視する**（起動失敗にしない）。deprecation 階段の途中版や、新旧版の設定ファイルが混在する更新時に、起動不能を作らないため

## 2. 設定ファイルとデータ配置

- **既定のデータルートは `%ProgramData%\Yagura`** とし、配下に設定ファイル・組み込み DB・スプール・メタデータ領域を置く（プログラム本体は `Program Files` 配下。データとバイナリを分離する Windows の標準慣行に従う）。スプールと組み込み DB の置き場所はそれぞれ設定で変更できる（同一ディスクの I/O 競合を避けたい環境向け。architecture.md M-12 の「置き場所」はこれで確定し、上限値のみ実測待ちとして残る。database.md DB-2 も本節で確定）
- 設定ファイルの形式は **JSON**（.NET 構成システムの標準）とし、**ウィザードと手編集は同一ファイルを対象とする**——二重の設定源（ウィザード用 DB と手編集用ファイル等）を作らない
- データルート配下の ACL は ADR-0004 決定 5 に従う（サービスアカウント + Administrators のみ。具体値はセキュリティ設計文書）
- **DB 接続資格情報は設定ファイルに平文で置かない**（ADR-0004 決定 5）: ウィザード経由の入力は DPAPI machine スコープで暗号化して保存する。**手編集で平文パスワードが書かれた場合、起動時に検出して暗号化表現へ書き換え、書き換えた事実をイベントログに記録する**（手編集経路でも平文を残置しない。DPAPI のサーバ束縛の帰結——バックアップ復元時の再入力——は operations.md の管轄）

## 3. ウィザードと手編集の関係

- ウィザードが正規の設定経路であり、ゼロ設定ファーストラン（設定なしで受信・閲覧が成立）からの「必要になったときに設定する」動線を担う。手編集は同じファイルへの直接操作として常に可能
- **競合の扱い**: 設定の保存は「読み込み → 変更 → 検証 → 保存」の全体書き換えとする。ウィザード（管理画面）は保存時にファイルの外部変更（手編集）を検知した場合、上書きせずに再読み込みを促す（楽観的な競合検出。管理操作は localhost 限定で同時実行者は実質 1 人だが、「手編集した内容が画面操作で黙って消えた」事故は防ぐ）
- **反映方式を項目の属性として持つ**: 各設定キーは「即時反映 / リスナ再構成（接続の瞬断を伴う）/ サービス再起動（受信断を伴う——architecture.md §4.4 で可視化される）」のいずれかの反映方式をスキーマ上で宣言し、UI は変更時にそれを利用者へ表示する。**再起動を要する項目は最小に保つ**ことを設計目標とする（割当は実装設計で確定し、本文書のスキーマ一覧 §8 に記録する）
- 手編集の反映は、再起動またはUI からの「設定の再読み込み」操作による（ファイル監視による自動反映はしない——編集途中の中間状態を読む事故を避ける）

## 4. 既定ポート・bind 先・ファイアウォール

### 4.1 受信（syslog）

- **UDP 514・TCP 514 を既定**とする（syslog の標準ポート。多くのネットワーク機器の既定と合わせ、送信元の設定変更なしで届くことを優先する——ADR-0004 決定 2 の「導入体験を守る」の具体化）
- TLS 受信（opt-in。v1.0 までに提供）は **TCP 6514**（RFC 5425 の標準ポート）を既定とする
- bind 先は既定で全インターフェース。設定で特定アドレスに縮小できる

### 4.2 閲覧 UI と管理

- **閲覧リスナ**: 既定で LAN に公開する（認証なし・読み取りのみ。ADR-0004 決定 2 の決定事項）。公開範囲は設定で「LAN 公開 / localhost のみ」を選択できる。既定ポートは Web 用の慣用値から衝突調査のうえ確定する（[CF-1](#9-確定待ち一覧)）
- **管理リスナ**: 書き込み系（ウィザード・設定変更・昇格等）は **loopback 専用リスナ**に束縛する（`127.0.0.1` / `::1` の両系。ADR-0004 決定 2 の第一候補方式）。ポートは閲覧リスナと別とし、既定値は CF-1 と併せて確定する。loopback 束縛の CI 回帰テスト（v0.1 受け入れ条件）はセキュリティ設計文書で仕様化する

### 4.3 ファイアウォール規則 — インストーラの責務

「入れたのにログが来ない」の第一原因は Windows ファイアウォールである。インストーラの責務を次のとおり定める:

- インストーラは**受信ポート（UDP/TCP 514）と閲覧ポートの受信許可規則を作成する**（規則名は「Yagura」で始まる名前空間に統一し、利用者が一覧で識別・監査できるようにする）
- 規則の作成はインストーラ上で**オプトアウトできる**（ファイアウォールを集中管理（GPO 等）している環境向け。オプトアウトした場合に必要な手動設定は利用者向けドキュメントに記載する——[§10](#10-利用者向けドキュメントへの申し送り) CF-D1）
- 管理リスナは loopback 束縛のため受信許可規則を要しない（作らない——不要な穴を開けない）
- アンインストール時に作成した規則を削除する
- 設定でポートを変更した場合の規則追従（自動更新か手動案内か）は実装設計で確定する（[CF-2](#9-確定待ち一覧)）

## 5. 管理者資格情報の扱い — 長時間処理での再入力方針

ADR-0004 決定 4（都度入力・使用後破棄・保持しない）の「長時間処理」への適用を定める:

- **保持の単位は「ウィザードの 1 実行」**とする: 資格情報は処理の開始時に入力させ、その処理（複数ステップを含む）が完了または失敗するまでメモリ内でのみ保持し、終了時に破棄する。ステップごとの再入力は求めない
- **無操作タイムアウト**: 入力途中で放置されたウィザードセッションは、一定時間の無操作で資格情報を破棄し、再開時に再入力を求める（時間は仮値: 15 分。[CF-3](#9-確定待ち一覧)）
- **長時間バッチ（将来の蓄積ログ移行等）は「実行中の再入力」を設計しない**: 必要な権限・接続の検証を開始前に完了させ（database.md §6.1 の準備フェーズと同じ思想）、実行開始後は資格情報の追加入力なしで完走できる構成とする——深夜バッチが資格情報の入力待ちで朝まで停止する事故を防ぐ
- 資格情報をディスク・設定ファイル・ログ・監査記録のいずれにも書かない（監査記録は「管理者資格情報を使用した」事実のみを残す）

## 6. HTTPS 証明書の取り扱い（opt-in 強化。v1.0 までに提供）

- **証明書は利用者の持ち込みを基本とし、Windows 証明書ストア（ローカルコンピューター）からの参照を唯一の指定方式とする**。ウィザードはストア内の証明書一覧から選択させる（拇印を直接入力させない）。Windows 管理者の既存スキル（証明書スナップイン・AD CS）にそのまま乗る方式であり、PFX ファイルパス + パスワードを設定に持つ方式は採らない（ファイルとパスワードの管理という新しい漏洩面を作らないため。ストアへの取り込み手順は operations.md——[§10](#10-利用者向けドキュメントへの申し送り) CF-D2）
- **自己署名証明書の生成支援は提供しない**（ADR-0004 決定 3 が検討を委任した項の結論）:
  - 理由: ①自己署名はブラウザ警告を常態化させ、「警告を無視して進む」習慣を利用者と閲覧者に育てる——TLS を opt-in した動機（通信の真正性）を自ら損なう。②AD 環境には AD CS という正道があり、AD 連携認証（同じ opt-in 群）を使う環境なら証明書配布の基盤も既にある。③どうしても自己署名が必要な環境では OS 標準の手段で生成できるため、製品が生成機能を持つ必要がない（参考手順は operations.md 起筆時に実機確認のうえ記載する）
  - 却下した対案——生成支援を内蔵する: 導入は楽になるが、上記①の害が「製品が推奨した」形になる。提供しないことで生じる導入の壁は、手順の文書化で緩和する
- 証明書の期限切れ・失効は起動失敗にせず、HTTPS リスナのみ停止して HTTP（既定構成）で継続し、イベントログ警告で可視化する（証明書事故で閲覧・受信まで止めない。ただし**認証 opt-in が有効な場合、リモート管理は HTTPS 前提のため自動的に閉じる**——ADR-0004 決定 3 の前提条件を維持する）

## 7. 書き込み系画面のネットワーク瞬断時挙動

Blazor Interactive Server の circuit は瞬断で失われ得る（ADR-0003 受け入れリスク）。ウィザード・設定変更など書き込み系画面の挙動を定める:

- **ステップ完了ごとにサーバ側へ確定する**: ウィザードの進行状態は circuit のメモリではなくサーバ側の永続的なセッション状態に置き、再接続・circuit 喪失後の再入で**確定済みステップから再開できる**。入力途中（未確定フォーム）の内容は失われることを許容し、再開時に「どこから再開しているか」を明示する（黙って空フォームに戻さない）
- **確定操作は一回性を保証する**: 設定の適用・昇格の実行などの確定操作は冪等トークン（同一操作の再送を検出する識別子）を伴い、瞬断 → 再送で二重適用されない
- **「半分だけ適用」を作らない**: 設定変更の適用は「検証 → 適用 → 記録」を単一の確定単位として実行する。瞬断が適用の途中状態を残さない
- v0.x の管理操作は localhost 限定（RDP 内のループバック接続）のため瞬断は稀だが、認証 opt-in 後のリモート管理では常態的なリスクになる——設計は最初から上記を前提とする

## 8. 設定スキーマの構成（一覧）

スキーマの詳細キーは実装設計で確定し、確定したものから本節に記録する（**本節が database.md §7 凍結対象一覧の「設定スキーマ」の定義場所である**。v1.0 で凍結）。区分の構成:

| 区分 | 主な内容 | 反映方式（目標） |
|---|---|---|
| 受信 | ポート・bind 先・受信バッファ（M-2）・TLS（opt-in） | リスナ再構成 |
| 流量制御 | 有効/無効・送信元別閾値（M-4。v0.1 は挿入点のみ） | 即時 |
| スプール | 有効/無効（opt-out）・置き場所・上限（M-12） | 置き場所のみ再起動 |
| 永続化 | provider 選択・接続情報（資格情報は暗号化表現 §2） | ウィザード経由（昇格 = database.md §6.1） |
| 保持期間 | 日数（DB-1）・実行時間帯 | 即時 |
| UI | 閲覧ポート・公開範囲（LAN / localhost）・管理ポート | リスナ再構成 |
| 通知 | イベントログ警告の有効範囲（architecture.md §4.6） | 即時 |
