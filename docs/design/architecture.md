## 1. プロセス構成

### 1.1 単一サービスと内部境界

- **単一の Windows サービスプロセス**に受信・永続化・Web UI を同居させ、1 つのインストーラで配布する（ADR-0002 決定 1）
- 内部は次の 4 つのアセンブリ境界で分離する。ADR-0002 決定 1 の 3 境界（受信・永続化・UI）に、どこにも属さない結線・サービス統合の置き場として**ホスト**を加えた 4 分割である。境界は保守性のためのものであり、セキュリティ境界ではない

| モジュール | 責務 | 担当するパイプラインの段（§2.1） |
|---|---|---|
| ホスト | Windows サービス統合（実行アカウントは仮想サービスアカウント既定。ADR-0004 決定 4）、起動・停止順序の制御、設定の読込、メタデータ領域（§4.3）、各モジュールの結線 | —（段を持たない） |
| 受信 | ソケットの listen、syslog メッセージの解析、流量制御の挿入点 | 受信段・解析段 |
| 永続化 | provider 抽象（database.md）、書き込みバッチ、ディスクスプール | 永続化段・スプール |
| UI | Blazor Web UI（ADR-0003）、検索・ダッシュボード・セットアップウィザード | —（読み取り側） |

- モジュール間の参照は「ホスト → 各モジュール」「受信・UI → 永続化の抽象インターフェース」のみとし、受信と UI は互いを参照しない
- **モジュール横断の契約の置き場として `Yagura.Abstractions` を置く**（M6-4。Issue #54 の PR レビューでオーナー決定・2026-07-05）: どの Yagura プロジェクトにも依存しない最下層プロジェクトで、UI が「ホスト管轄の実装を持つサービス」を型として参照するための抽象を置く。**依存を持たないことがこのプロジェクトの成立条件**であり、`ProjectReference` の追加は参照構造の設計変更として扱う。**M8-4（Issue #71）で M6-4 の申し送り 2 件を決着した**: ①管理画面の書き込み系サービスの契約群（`ISetupWizardService`・`IPromotionWizardService`・`ICircuitManagementService`。いずれもマーカー `IYaguraWriteService` を実装——security.md §1 L-5 の参照分離検査の規約）を `Yagura.Abstractions.Administration` に配置し、実体はウィザード系 = `Yagura.Host.Administration`（設定ファイル・DB 接続の管轄が Host のため）/ circuit 管理 = `Yagura.Web.Administration`（circuit 台帳が Web 層のため）が持つ。②監査記録の書き込み口（`IAuditRecorder`・`AuditEvent`・`AuditEventKind`）を `Yagura.Storage.Auditing` から `Yagura.Abstractions.Auditing` へ**移設した**——監査記録の実体はログ本体の永続化（provider 抽象 = Storage の管轄）とは独立した「ホスト管轄のローカルファイル + イベントログ併記」（security.md §4.2）であり、Web・Host の両方が使う横断契約は最下層に置くのが参照構造として正しいため（実体 `FileAuditRecorder` の結線は従来どおり Host）

### 1.2 起動順序 — 受信を最初に開く

サービス再起動は受信断であり、syslog は再送されない（ADR-0002 帰結）。再起動によるロスをプロセス停止時間そのものまで縮めるため、起動順序を次のとおり固定する:

1. スプール領域を開く（前回退避分の存在確認を含む）
2. **受信ソケットを開き、受信を開始する**（この時点から取りこぼしが止まる）
3. DB provider を初期化する。完了までの間は DB への書き込みが確定できないため、受信済みログは §3.2.1 の発動条件に該当し、通常のパイプライン経路でスプールへ退避される
4. Web UI を開始する

DB や UI の初期化失敗・遅延が受信開始を遅らせない。これが「ゼロ設定ファーストラン」（SQLite の初期化は軽い）と「本番構成」（SQL Server への接続確立を待たない）の両方で成立する。

**手順 1 が失敗した場合**（ディスク障害・ACL 破損等でスプールを開けない）: 受信は開始し、**スプールなしの縮退運転**として続行する。縮退はイベントログへの警告（§4.6）とメトリクスで強く可視化し、黙って opt-out 相当に落ちることを許さない。起動失敗（= 受信断の固定化）よりも、可視化された縮退の方が「ログを失わない」原則への害が小さい、という判断である。縮退中に Q2 が溢れた分・書き込みに失敗した分は退避先がないため破棄となり、永続化失敗カウンタに計上する。

### 1.3 停止順序 — メモリ上のログをスプールへ逃がす

停止要求を受けたら:

1. 受信ソケットを閉じ、その時点のカウンタをメタデータ領域へ書く（以降の到着分はプロセス停止と同じくロスであり、§4.4 の受信断として可視化される）
2. **メモリ上の未永続化ログを DB へ書き切るのを待たず、スプールへ追記して退避する**（退避分は次回起動時の drain で保存される）。退避中に発生した破棄は**メタデータ領域のカウンタへ逐次（一定件数ごと）反映する**——強制終了された場合でも退避中の破棄が計上されないまま消えないようにするため
3. カウンタを最終値で永続化し（§4.3）、正常停止イベントを記録して終了する

- 手順 2 の設計意図は「ディスクへの追記は DB への確定書き込みより短時間で終わる」ことにあるが、これは仮説であり、退避方式が書き切り方式より実際に速いことの検証を停止時間の実測（[M-9](#9-実測待ち一覧)）に含める
- **手順 2 が失敗する場合**（スプール上限到達・書込失敗）: 当該分は破棄し、スプール破棄またはスプール書込失敗カウンタに計上したうえで手順 3 に進む（手順 1 で一度カウンタを書いているため、退避中の破棄が計上されないまま落ちる窓を最終値の書き込み失敗時のみに限定できる）
- 停止処理全体がサービスコントロールマネージャの停止待機時間に収まらず強制終了された場合は、クラッシュと同じ扱いになる（§4.4 の近似断点・§4.1 の計上限界）。停止時間の目標値は実測で確定し（[M-9](#9-実測待ち一覧)）、停止待機時間の既定に収まることを受け入れ条件とする

### 1.4 フォワーダ配布キットのテンプレート埋め込み（ADR-0008 委任 #1）

管理 UI のフォワーダキット生成機能（`/admin/forwarder-kit`。ui.md §4）は、リポジトリの `forwarder/fluent-bit/`（静的な手動配布キット）配下のファイルを**単一ソース**として使う。コピーを別に持たず、`src/Yagura.Web/Yagura.Web.csproj` の `EmbeddedResource`（`Link` + `LogicalName` 明示）でビルド時にアセンブリへ埋め込む。実装参照:

- 埋め込み対象: `fluent-bit-yagura.conf`・`winevt-severity.lua`・`install.ps1`・`uninstall.ps1`・`README.generated.md`（生成専用。静的キットには含まれない）
- 読み出し: `Yagura.Web.ForwarderKit.ForwarderKitBuilder` が `Assembly.GetManifestResourceStream` で読み、置換後に **メモリ上の `ZipArchive`** で梱包する（ディスクへの一時ファイルを書かない——ADR-0008 設計条件 7）
- 置換値（宛先ホスト・ポート・チャネル）の検証制約は `install.ps1` の `ValidatePattern` を正とし、`Yagura.Web.ForwarderKit.ForwarderKitConstraints` へ同一の正規表現を定数化した上で、同期テスト（`Yagura.Web.Tests.ForwarderKit.ForwarderKitConstraintsSyncTests`）がリポジトリの `install.ps1` を読み込んで機械的に突合する（ADR-0008 設計条件 5）
- `install.ps1` 自体は静的キット・生成キットの両方に対応する（`@@YAGURA_HOST@@` プレースホルダの有無で分岐——プレースホルダが残っていれば `-YaguraHost` 必須の静的キット、置換済みなら宛先設定済みの生成キットとして無引数実行できる）

### 1.5 フォワーダ MSI のオプトイン同梱（ADR-0008 設計条件 9・委任 #7）

MSI 同梱の要否判定に必要な「配置フォルダ（データルート配下 `forwarder`）の実パス」は、Web 層が直接知らない（§1.1 の参照構造——データルートの解決は `Yagura.Host.Configuration.YaguraConfigurationLoader` の管轄）。`INicCandidateSource`（1.4 の宛先候補検出）とは異なり外部設定（データルート）に依存するため、`ISetupWizardService` / `IPromotionWizardService` と同じ結線パターンを使う:

- 契約: `Yagura.Web.ForwarderKit.IForwarderMsiSource`（配置フォルダの列挙・単一検出時の詳細取得）。実体 `SystemForwarderMsiSource` は判別可能な型 `ForwarderMsiLookup`（`NotFound` / `Multiple` / `Single`）を返す
- 結線: `Yagura.Host.Program` が `Path.Combine(dataRoot, ForwarderMsiConstraints.PlacementSubPath)` で実パスを組み立て、`SystemForwarderMsiSource` へ渡して DI 登録する（`AddYaguraAdmin` 内ではなく `Program.cs` で明示登録——`INicCandidateSource` の `TryAddSingleton` 既定とは異なる）
- 判定の分離: ファイル名パターン一致・版解決（ProductVersion 優先・ファイル名補助）・公式 SHA256 照合は `ForwarderMsiFilter`（純粋関数）に切り出し、列挙 I/O（`SystemForwarderMsiSource`）から独立してテストできるようにする（`NicCandidateFilter` / `SystemNicCandidateSource` と同じ設計）
- MSI の版取得: `msi.dll` の `MsiGetFileVersion`（P/Invoke 1 関数）で MSI 自身の ProductVersion を取得する。`MsiOpenDatabase` 系のハンドル管理を避けるための選択。取得不能時はファイル名から抽出した版を補助的に使う
- 同梱時の ZIP 組み立て: `ForwarderKitBuilder.Build` が MSI バイト列をメモリ上で読み込み ZIP に封入する（設計条件 7 のディスク非使用原則を維持）。`GENERATED.txt` の `Kit-SHA256`（ZIP 全体のハッシュ）は自己参照を避けるため「`GENERATED.txt` を含まない全エントリのハッシュ」と定義し、2 段階（仮組み立て → ハッシュ算出 → `GENERATED.txt` を追加した本組み立て）で算出する

### 1.6 フォワーダ MSI のアーキ別検出（ADR-0009 決定7・委任 #4）

収集対象端末（Fluent Bit を導入する側）のアーキ対応を x64 に加えて ARM64（`winarm64`）へ拡張する。Yagura サーバ本体のアーキ（§1.1 の RID 分岐等）とは独立の論点であり、`IForwarderMsiSource` はアーキ引数を取る形へ変更した:

- 契約変更: `IForwarderMsiSource.Lookup()` → `Lookup(ForwarderMsiArchitecture architecture)`（`ForwarderMsiArchitecture` は `Win64` / `WinArm64` の 2 値。x86 は対象外——サーバ本体の x86 不採用（ADR-0009 決定1）と同じ理由）。呼び出し時に指定したアーキのファイル名パターン（`fluent-bit-*-win64.msi` / `fluent-bit-*-winarm64.msi`）のみを候補とする
- 誤配布への牽制: 「配置フォルダに win64・ARM64 の MSI が混在すると誤って別アーキを同梱してしまう」という失敗様式(ADR-0008 委任 #4 の検討事項)は、`Lookup` のアーキ絞り込みそのものが構造的に防ぐ——選ばなかったアーキの MSI は候補にすら入らないため、画面上の追加警告は実装していない。同一フォルダに win64・ARM64 が 1 つずつ存在する状態(混在フレット配布)は「複数検出」エラーにならない(アーキごとに独立判定するため)
- 管理 UI（`ForwarderKitScreen`）は「収集対象端末のアーキテクチャ」（既定 x64）を選択させ、選択に応じて `Lookup` を呼び直す。ダウンロードエンドポイント（`YaguraAdminExtensions.MapForwarderKitDownload`）は `architecture` クエリパラメータ（既定 `x64`。未指定・不正値の扱いはエンドポイント実装参照）で同じアーキを受け取る
- `install.ps1` 側（静的キット・MSI 自動検出時）は `Get-LocalMsiFilenamePattern` が実行端末のプロセッサアーキを判定し、対応するファイル名パターンの MSI のみを検出する。判定の第一情報源は WMI/CIM（`Win32_Processor.Architecture`——ネイティブの WMI サービスが応答するため呼び出しプロセスのエミュレーション状態に左右されない）で、環境変数（`PROCESSOR_ARCHITECTURE` / `PROCESSOR_ARCHITEW6432`）は CIM 不可時のフォールバック（環境変数のみの判定は「ARM64 上の x64 エミュレーションでは両変数とも AMD64 を返す」既知の限界があり実 OS を反映しないため——PR #222 レビュー）。これにより 1 つのキットフォルダに win64・winarm64 の MSI を共存させ、x64/ARM64 混在フレットへ同一の配布パッケージ(Intune/SCCM/GPO)を使い回せる。x86(32bit)上で実行した場合は明確なエラーで停止する
- 検証水準の非対称性: Yagura サーバ本体の ARM64 対応（ADR-0009 決定2）と同じく、フォワーダキットの ARM64 も「試験的」の位置づけとする。ARM64 の Fluent Bit MSI 取得・SHA256 実測は 2026-07-10 に確認済みだが、ARM64 実機での導入・転送の実機検証は本機能の実装時点で未実施(検証環境が x64 のため)——[利用者ガイド](../guides/forward-windows-eventlog.md)の該当節に明記する

## 2. 受信パイプライン

### 2.1 段の構成

受信から保存までを容量制限付きの非同期パイプラインとして構成する（ADR-0002 決定 2）:

```
送信元 ──UDP/TCP──> [OS ソケットバッファ]
                        │ 読み取り専念ループ
                        v
                    [受信段] ──(流量制御 挿入点)──> [解析段]
                        │ 有限キュー Q1                │ 有限キュー Q2
                        v                              v
                    （溢れ: UDP 破棄 /             [永続化段] ──バッチ書込──> DB provider
                      TCP 読み取り停止）               │ Q2 溢れ・書き込み失敗/タイムアウト
                                                       v
                                                  [ディスクスプール] ──drain──> 永続化段へ合流
```

- **受信段**: ソケットからの読み取りに専念する。解析や書き込みを同じループで行わない（読み取りの停滞は OS ソケットバッファ溢れ = 計測困難なロスに直結するため）
- **解析段**: RFC 3164 / RFC 5424 の解析。**解析に失敗したメッセージは破棄しない**——受信した生データのまま、解析失敗の印を付けて保存する（「ログを失わない」原則。不正形式の頻発はそれ自体が観測対象であり、カウンタにも計上する）
- **永続化段**: provider へのバッチ書き込み。**書き込みには時間上限（タイムアウト）を設け、応答のないハングを打ち切る**（§3.2.1）。バッチサイズ・書込間隔は実測で確定する（[M-1](#9-実測待ち一覧)）、タイムアウト値は [M-13](#9-実測待ち一覧)
- **DB への書き込みは単一の書き込みゲートで直列化する**: provider 契約（database.md §1.2）は「書き込みは呼び出し側が直列化する」ことを前提とするが、実際の書き込み経路はライブ書き込み（永続化段）・スプール drain（§3.2.2）・保持期間削除（database.md §3）の 3 つが独立したタスクで並行し得る。ホストはこの 3 経路が共有する単一の書き込みゲート（実装 = `Yagura.Storage.LogStoreWriteGate`。結線 = `Yagura.Host.Program`）を構築し、契約と実配線を一致させる。**ゲート待ちの時間予算は書き込みタイムアウト（上記）とは独立**とし、ゲート待ちが「速度不足」のシグナル（§5.3）に化けることを防ぐ。ゲートを取得できないまま予算を使い切ったライブ・drain のバッチは、書き込みタイムアウトと同じ退避経路（スプール退避 / セグメント未消化のまま次周期へ）に合流する——ログを失わない設計は変わらない。なお、起動処理の受信断記録（§4.4。起動時 1 回の直接書き込み）はゲートを通らない第 4 の経路だが、3 経路のいずれもまだ開始されていない時点で実行されるため、非同時実行は §1.2 の起動順序により保証される（`ILogStore` の doc コメントに明記）

### 2.2 保存の意味論

- **at-least-once**（ADR-0002 決定 2）。スプールからの復旧時等に重複保存が起こり得ることを仕様として許容し、重複排除は約束しない
- **保存順序は保証しない**。ライブ経路とスプール drain 経路が並行するため、DB への挿入順は受信順と一致しないことがある。閲覧・検索は受信時刻列で並べ替えて提供する（時刻の契約——UTC 保存・表示層で変換——は ui.md / database.md で定める）

## 3. 信頼性機構

### 3.1 バックプレッシャ — 境界ごとの溢れ挙動

各境界の溢れ時挙動をブロックか破棄か明示する（ADR-0002 決定 2。受信段では、OS 側の計測不能なロスに転化するブロックを選ばない）:

| 境界 | 容量 | 溢れ時の挙動 | 計上先カウンタ |
|---|---|---|---|
| OS ソケットバッファ | 受信バッファサイズ（設定項目 `Ingestion:Udp:ReceiveBufferBytes`。既定 **1 MiB**——[M-2](#9-実測待ち一覧) 実測確定 2026-07-05: 開発機の持続 20k〜60k msg/sec・バースト 20,000 通の全条件で、1 MiB 以上なら OS バッファ破棄（導出値）がゼロになることを確認。実測記録: `tools/Yagura.Bench/results/2026-07-05-dev-machine-rcvbuf/`） | OS が破棄（アプリからは直接見えない） | OS 統計との突合（§4.2） |

- **Q1 溢れ（UDP）を破棄とする理由**: Q1 が溢れるのは解析段が追いつかない瞬間であり、このときスプール追記（チェックサム付きディスク I/O）が解析より速く受け止められるという保証はない。また解析前の生データを退避対象にするとスプールに第 2 のレコード形式を持ち込むことになる。単純さを優先して破棄 + 計上とする。**この判断は「Q1 溢れは持続せず、実質的にバースト時に限られる」という前提に依存する**——バースト負荷時の Q1 破棄の発生有無をベンチの計測対象に含め（[M-1](#9-実測待ち一覧)）、平常運用で恒常的に発生することが実測で判明した場合は、Q1 溢れ分の生データ退避を再評価する
- 破棄はすべて発生箇所別カウンタに計上する。「カウンタに計上されない喪失」は重大不具合である（ADR-0006 基準 2 の定義①。クラッシュ時のメモリ内喪失の限界は §4.4 に明記する）
- **受信エラー（SocketException）時の扱い（Issue #142）**: 個々の `ReceiveAsync` 失敗（Windows の UDP では、直前の送信に対する ICMP ポート到達不能の反映等、環境依存の一過性エラーが発生し得る）は読み取りループを止めないが、無言では握り潰さない——SocketErrorCode 付きでログ出力（5 秒間隔で抑制・集約し、持続的エラー時のログ溢れを防ぐ）し、UDP 受信エラーカウンタ（§4.1）へ計上する。さらに連続して失敗する間は指数的な backoff（10ms 起点・1000ms 上限。単発エラーは backoff しない）を挟み、持続的エラー状態での密ループによる CPU 浪費を防ぐ。実装は `Yagura.Ingestion.Udp.UdpSyslogListener`

### 3.2 ディスクスプール

#### 3.2.1 発動条件と構造

- 発動条件は「**永続化段が、受け取ったログを所定時間内に DB へ確定できないこと**」に統一する。具体的には次の 2 形態が同じ退避動作に合流する:
  - **容量**: Q2 が満杯で新規投入分を受け取れない（持続的な速度不足はこの形で現れる）
  - **時間**: バッチ書き込みが失敗した、またはタイムアウト（[M-13](#9-実測待ち一覧)）で打ち切られた——打ち切られたバッチはスプールへ退避し、宙吊りのまま失われる in-flight を作らない（タイムアウト後に DB 側で書き込みが成立していた場合の重複は at-least-once の範囲内）
- 保存先の死活を監視する専用のヘルスチェック経路は持たない。停止・ハング・速度不足のいずれも、上記 2 形態のどちらかとして現れる（低流量環境で Q2 が溢れないまま保存先が死んでいる状態は、タイムアウトが打ち切る）
- 既定で有効（opt-out。ADR-0002 決定 2）
- 構造要件: 追記専用のセグメントファイル群とし、レコード単位の破損検出（チェックサム）と**レコード種別の識別**（通常ログ / 自己検証用の合成レコード。§3.2.5）を持つ。セグメントは DB へのコミット確認後に削除する（クラッシュ時は未確認セグメントを再 drain する——重複は at-least-once の範囲内）。破損検出により読み捨てた末尾分は drain がセグメント削除の確定時に「スプール末尾破損破棄」カウンタ（§4.1。Issue #201）へバイト数で計上する——破損検出そのものは無償の読み捨てにしない
- ディスク使用量には上限があり、スプールがディスクを無制限に消費することはない。上限の既定値は [M-12](#9-実測待ち一覧) で確定する（置き場所の既定——データルート配下・設定で変更可——は [configuration.md](configuration.md) §2 で確定済み）
- スプール領域はログ本体と同等の保護対象とし、ACL は ADR-0004 決定 5 に従う

#### 3.2.2 drain — 復旧後の追いつきと振動防止

復旧直後に drain がライブ流入と資源を奪い合うと、二次飽和 → 再スプールの振動が起こり得る。これを次の 3 点で防ぐ:

- **ライブ優先**: drain は Q2 の使用率が低水位を下回っている間だけ進める。ライブ流入が増えて Q2 が埋まれば drain は自動的に停止する（ヒステリシス。低水位・再開水位は実測で確定 [M-3](#9-実測待ち一覧)）
- **速度上限**: drain の投入速度に上限を設ける（既定値は実測で確定 [M-3](#9-実測待ち一覧)）
- **保存先障害中は進めない**: 直近の書き込みが失敗・タイムアウトしている間は drain を停止し、バックオフ後に再試行する（§3.2.1 と同じシグナルの再利用であり、排した専用ヘルスチェックの新設ではない）。また **drain 由来のバッチが書き込みに失敗・タイムアウトした場合はスプールへ再追記しない**——元セグメントを未消化のまま残して drain を停止する（スプールへの退避はライブ経路由来のバッチに限る）。タイムアウトの恒常化時に「退避 → drain → 再タイムアウト → 再退避」で同一レコードがスプール内に複製され続け、実流入と無関係に上限到達（= 実ログの破棄）を早めるループを閉じるためである

この設計の帰結として、**持続的な速度不足の間は drain が進まず、スプールは上限へ向かって片方向に成長する**。これは隠れた欠陥ではなく「保存先の増強か流入の削減が必要」という状態そのものであり、進行を §4.6 のゲージ（使用量・最古滞留時間）と能動通知（上限接近の警告）で運用者に見せる。運用者の行動手順は operations.md に委ねる。

#### 3.2.3 上限到達時の挙動 — 新規破棄を採用

ディスク使用量が上限に達したら、**新規到着分を破棄する**（破棄はスプール破棄カウンタに計上）。

- **主たる採用理由は次の 2 点**: ①「退避に成功したログは（重複はあっても）失われない」という単純な約束を守れる——回転は退避済みデータの事後破棄であり、この予測可能性を失う。②満杯状態での回転（削除と追記の並行）という最も苦しい局面の複雑な I/O を持たず、実装と障害解析が単純になる
- **対価を明示する**: DB の長時間停止では、新規破棄は復旧時点に近い——つまり障害調査で最も参照したい——ログを失う側の選択である。この対価は認識のうえで受け入れる。上限到達は「上限の見積もりか保存先の復旧が破綻している」状態であり、どちらの破棄方式を選んでも損失は損失として発生する。本設計は残すログの新旧よりも、約束の単純さと機構の単純さを優先する
- 補助的な理由: 新規破棄では損失区間が「上限到達から空きができるまで」の 1 区間にまとまる。回転では drain・回転削除・追記が並行する時間帯に、保存済み・破棄済みが時間軸上で交錯し得る
- 上限到達の発生は能動通知（§4.6）で運用者へ知らせ、取るべき行動は operations.md に委ねる

#### 3.2.4 削除保証

- 消化（DB コミット確認）済みセグメントと破棄されたセグメントは、**通常のファイル削除で速やかに削除する**。消化済みデータを削除待ちのまま滞留させない
- **ストレージ上の残留痕への対抗（上書き消去等）は行わない**。この層の保護は ACL（ADR-0004 決定 5）とディスク全体の暗号化（利用環境側の統制。operations.md で案内）が担い、アプリによる確実消去は約束しない——この限界は利用者向けドキュメントに明記する（[§10](#10-利用者向けドキュメントへの申し送り)）

#### 3.2.5 平常時の経路検証

障害時専用経路は最悪の瞬間に最少テストのコードが走る（ADR-0002 却下案 (c) の正当な懸念）。代償措置として:

- **定期自己検証**: 一定周期（仮値: 1 日 1 回。[M-11](#9-実測待ち一覧)）で合成レコードをスプールへ書き込み、**drain の実機構に読ませて**照合する。合成レコードはレコード種別（§3.2.1 の構造要件）で識別され、**drain は種別を見て DB 書き込みの直前で合成レコードを破棄する**——この識別・破棄は定期検証時だけでなく drain の常時動作であり、クラッシュ後の再 drain でも合成レコードが通常ログとして DB へ混入する経路はない（証跡 DB への偽レコード混入の防止）。検証の失敗は能動通知（§4.6）の対象とする
- **検証の覆域を明示する**: 自己検証が覆うのは「スプール書込 → セグメント読出 → 逆直列化 → drain 合流判定」まで、ライブトラフィックが常時検証するのは共有された DB 書込コード（drain 専用の書込実装を持たない）である。**「drain が読んだレコードを実際に DB へコミットしてセグメントを削除する」結合部そのものは、どちらにも常時は覆われない**——この区間は受け入れテスト（スプール発動 → 復旧 → 件数突合）を CI に置いて担保する（§5.1 のベンチとは別の機能テスト）
- **実装状況（Issue #152）**: 投入と照合を橋渡しする状態管理として `Yagura.Storage.Spool.SpoolSelfTestTracker` を置いた——未照合マーカー（高々 1 件）と投入時刻を保持するだけの小さな共有オブジェクトであり、投入側・照合側のどちらのプロジェクトからも参照できる Yagura.Storage 層に置くことで、Host → Ingestion の既存の参照方向を保ったまま両者が同一インスタンスを共有できる（`Yagura.Host.Program` が生成し、両側へ渡す）。**投入・タイムアウト判定**は `Yagura.Host.Observability.ActiveNotification.ActiveNotificationMonitor.EvaluateOnceAsync` に評価メソッドを 1 つ追加する形で乗せた（想定どおりの拡張。§4.6 実装状況の記述を参照）——同クラスの周期監視（仮値 1 分）のたびに、①直前に投入したマーカーが期待時間（仮値 10 分。[M-16](#9-実測待ち一覧)）以内に drain へ合流判定されたかを判定し、②自己検証の周期（仮値 1 日）に達していれば新しい合成レコードを投入する。**照合**は `Yagura.Ingestion.Persistence.SpoolDrainCoordinator` が、合成レコードを DB 書き込み直前で破棄する既存の常時動作の中でトラッカーへ通知する形で実装した（専用の検証用読出経路は設けていない——「drain の実機構に読ませて照合する」という本節の要求どおり）。**スプール無効（opt-out）・縮退運転中（スプール領域を開けなかった。§1.2）はいずれも自己検証を行わない**（投入対象自体が存在しない）——縮退運転は別の通知（EventId 1001）で既にカバー済みのため、自己検証側からは重ねて警告しない。**投入自体の失敗（書込失敗・上限到達）はタイムアウトを待たず即座に警告し、失敗したマーカーの未照合登録は取り消す**（書込失敗はそれ自体が経路破損の一次シグナルであり、次の判定機会まで待つ理由がないため。登録を取り消すのは、ディスクへ書かれなかったマーカーは drain レビュー対応）。`DeleteSegment` は drain が DB 書き込み確定後にのみ呼ぶため、このカウンタの増分は drain 消化の直接証拠になる。タイムアウト判定時、**直近（自己検証の期待時間 10 分と同じ窓）に進捗を観測していれば「未消化バックログの滞留」（EventId 1010。警告。現在のスプール使用率を含む）**、**観測していなければ「経路障害の疑い」（EventId 1009。エラー。従来どおり）**とする。drain は 1 セグメントの書込失敗で以降のセグメントの消化を止める設計（`SpoolDrainCoordinator` の FIFO 1009・1010（security.md §4.3）。テストは `SpoolSelfTestTrackerTests`（トラッカー単体の投入・照合・タイムアウト判定・失敗時の登録取消）・`SpoolDrainCoordinatorTests`（drain がトラッカーへ実際に通知すること）・`ActiveNotificationMonitorTests`（`FakeTimeProvider` ラッチ後の単発進捗で 1010 へ揺り戻らないこと（次マーカー投入での解除を含む）、の 4 シナリオ）に置いた。

### 3.3 流量制御

- 挿入点は**受信段の直後・解析段の前**に置く（送信元の判別に解析は不要——送信元アドレスはソケット層で得られる。解析コストを浪費させないため解析より前）
- 送信元単位の制限とし、方式は **token bucket**（Issue #260 の実装設計で確定。v0.1 は挿入点とカウンタの枠のみだった——ADR-0002 決定 2 の到達水準）
- **実装状況（Issue #260）**: `Yagura.Ingestion.FlowControl.TokenBucketIngressGate` が送信元 IP アドレスごとに token bucket を持ち、持続速度（`Ingestion:FlowControl:MessagesPerSecond`）・バーストサイズ（`Ingestion:FlowControl:BurstSize`）で判定・破棄する。設計判断:
  - **判定単位は件数**（バイト数ではない）——破棄カウンタの単位 `{datagram}` と揃え、運用者が「何件落ちたか」を直接読める
  - **補充は遅延計算・時刻源は単調クロック**（`TimeProvider.GetTimestamp`）——システム時刻の変更に影響されない。拒否時もトークンの端数は失われない
  - **状態空間の有界化**: 送信元アドレス（UDP では偽装可能）をキーに状態が無制限成長しないよう、①満杯まで回復したバケットの周期削除（削除は判定結果を変えない——満杯のバケットは次の到着時に満杯で再生成されるため）②追跡数上限（仮値 100,000）到達時は新規送信元を**追跡なしで通す**（fail-open——偽装による飽和が正規の新規送信元の受信断へ転化することを避け、「受信を止めない」を優先。追跡済み送信元の制限は維持される）
  - 破棄は挿入点の呼び出し元（UDP/TCP/TLS の各リスナ）が必ず計上する（§4.1「流量制御破棄」。「発火は必ず計測される」の原則どおり）
  - **発火状況の送信元別可視化（Issue #288。2026-07-18 オーナー裁定——案 (a) カード型）**: 「どの送信元が制限に達しているか」を、ゲート内部の送信元別バケットに**拒否カウントを併せて保持**して観測する（読み取り契約 `IFlowControlRejectionReader`。ダッシュボードのカード表示は ui.md §4）。既存の有界化設計（上記——スイープ・追跡上限）に相乗りし、可視化のために新たな無限成長点を作らない。帰結として**カウントの寿命はバケットと同じ**——制限なく受信できる状態が続いた送信元はスイープでカウントごと消え、設定変更のゲート差し替えでもリセットされる（起動からの破棄総数は計器「流量制御破棄」が持ち続ける）。送信元別の抑制窓付きイベントログ（案 (b)）は見送り（抑制窓の設計コストとログ騒音化リスクに対し、カードで運用上の主目的「どの装置か」に到達できるため——裁定）
- 既定は opt-out（既定有効。`Ingestion:FlowControl:Enabled`。ADR-0002 決定 2）。既定閾値は「通常運用で発火しない安全側」に実測で定める（[M-4](#9-実測待ち一覧)）——現在の仮値は持続 10,000 件/秒・バースト 20,000 件（CI 回帰ベンチの単一送信元実測スループット SustainedZeroDrop ≒ 5,000 件/秒の 2 倍を持続速度とし、その 2 秒ぶんをバーストとした安全側）

#### 4.1.1 計器の命名規則（M4-4 で確定）

**単一 Meter `Yagura` + 計器名 `yagura.<領域>.<事象>`** 方式を採用する。対抗案「発生箇所ごとにタグ付けした単一カウンタ（例: `yagura.dropped.total` + `site` タグで区別）」は、OpenTelemetry の計装ガイドライン（opentelemetry.io/docs/specs/semconv/general/metrics/、確認日 2026-07-05。「同種の事象の次元展開はタグで表現し、計器の乱立を避ける」旨の記載）を踏まえて比較検討したが、本表の 8 種は「同じ事象の異なる次元」ではなく**意味論そのものが異なる別々の事象**であるため、(i) 「損失は必ずどれかのカウンタに計上される」の検証が「タグ値の網羅」という間接的な確認になり事故りやすい、(ii) [UI-7](ui.md#12-確定待ち一覧) の試用報告様式が要求する「カウンタ識別子（開発用語側のキー）との 1 対 1 対応」をタグ付き単一カウンタは崩す、の 2 点で不採用とした。Meter 名は M2〜M4-3 の `Yagura.Ingestion`（アセンブリ名スコープ）から、メタデータ領域・OS 統計ゲージ（Yagura.Host 側にまたがる。ゲージはのち [ADR-0016](../adr/0016-os-drop-gauge-mechanism.md) で撤去——統合判断自体は有効なまま）を機に **`Yagura`** へ統合した——観測点をアセンブリ境界と無関係な単一の計測空間として扱うことで、試用報告・M8 状態画面が「1 つの Meter を購読すれば全カウンタ・ゲージが揃う」ことを前提にできる。

**TLS ハンドシェイク失敗（Issue #137）は送信元アドレス（`source_address`）をタグに持つ**。本節冒頭の「タグ付き単一カウンタは採らない」判断（意味論の異なる別々の事象をタグで畳まない）とは矛盾しない: 本カウンタは「TLS ハンドシェイクの失敗」という**単一の事象**を送信元次元で展開したものであり、複数の異なる事象をタグで束ねているのではない。送信元別の展開が必要な理由は security.md §6 のとおり——証明書期限切れ中に「どの送信元が脱落しているか」が運用者の初動解析に直結するため。実装は `Yagura.Ingestion.Diagnostics.IngestionMetrics.RecordTlsHandshakeFailure`（`Yagura.Ingestion.Tls.TlsSyslogListener` から計上）。UDP 受信エラー・逆引き解決カウンタと同じ「診断用カウンタ」の扱いで、プロセス内累積のみ（再起動をまたぐ永続化——§4.3——の対象外）。**タグの distinct 値は上限（`IngestionMetrics.MaxTlsHandshakeFailureSourceCardinality`。既定 256）で有界化し、超過分は `(other)` へ畳む**——送信元アドレスは TLS 認証成立前に計上される攻撃者制御の次元であり、無制限のタグ基数が集約 exporter のメモリを圧迫し得るため。「ほぼ全送信元が失敗」の観測は overflow バケット（`(other)`）の増加として現れるため、送信元別脱落確認（security.md §6・SEC-D3）の目的は損なわれない。

**スプール退避は `reason` タグを持つ（Issue #271 / M-7）**——退避契機を `q2_overflow`（容量: Q2 溢れ。§3.2.1）・`write_timeout`（時間: 永続化バッチの書き込み失敗/タイムアウト。書き込みゲート取得タイムアウトを含む）・`shutdown`（停止時ドレイン: §1.3 手順 2）で判別できる。TLS ハンドシェイク失敗と同じ「単一の事象（スプール退避）を契機次元で展開したもの」であり、複数の異なる事象を束ねているのではない。**この展開は可観測性（契機分布の把握）のためであり、§5.3 の昇格案内の誤発火防止の判定を変えない**（同判定は退避形態ではなく「同一観測窓内に成功した書き込みが存在するか」で行う設計を据え置く）。累積総数（`SnapshotCumulativeCounters` が返す §4.3 永続化対象）は契機によらず単一の総和のまま——タグは実行時計器の次元展開のみで、損失台帳の「1 事象 = 1 カウンタ」対応は崩さない。実装は `RecordSpoolEvacuated(SpoolEvacuationReason)`（`Yagura.Ingestion.Parsing.ParsingStage`＝容量/停止時、`Yagura.Ingestion.Persistence.PersistenceWriter`＝時間/停止時 から計上）。

**解析失敗（保存済み）・TCP 不完全メッセージ（Issue #270）**は、いずれも解析段が生データのまま保存し**損失ではない**（§2.1「解析に失敗したメッセージは破棄しない」・database.md §2.1 の排他 3 値）。それでも「破棄・異常は必ず計上する」原則の穴を埋めるため計器化した。実装は `Yagura.Ingestion.Parsing.ParsingStage.ParseAndCount`——解析直後の単一点で `LogRecord.ParseStatus` を見て計上し、後段（Q2 投入・停止時のスプール退避）の経路によらず二重計上・計上漏れを避ける。UDP 受信エラー・TLS ハンドシェイク失敗と同じ「診断用カウンタ」の扱いで、プロセス内累積のみ（再起動をまたぐ永続化——§4.3——の対象外。損失台帳に載せない＝保存済みで失われていないため）。**状態画面/ダッシュボードへの露出の判断（Issue #270）**: 他の 16 計器と同じく単一 Meter `Yagura` に載るため、試用報告様式（UI-7）・M8 状態画面が「1 つの Meter を購読すれば全カウンタが揃う」経路で自然に取得できる。**専用のウィジェットや強調表示は設けない**——両者は損失ではなく「送信元の設定・実装品質のシグナル」であり、損失系カウンタ（永続化失敗・スプール破棄等）と同格に扱うと運用者の注意を誤誘導するため、他の診断用カウンタと同じ一覧内の 1 項目として出すに留める。

OS 受信破棄の 2 ゲージは「プロセス起動時からの増分」を返す（システム全体の累積値をそのまま出すと本製品起動前からの値が混入するため）。実装は `Yagura.Ingestion.Diagnostics.IngestionMetrics`。

Web 層（閲覧・管理リスナ）の 2 カウンタは security.md §1 L-3b（拒否試行の計上）・§4.2（記録失敗の観測可能化——アプリ記録ファイルと Windows イベントログの両方が失敗した場合のみ計上）に対応する。実装は `Yagura.Web.Diagnostics.WebGuardMetrics`——`Yagura.Ingestion` とは別インスタンスだが同じ Meter 名 `Yagura` を使い、単一の計測空間に統合する（Yagura.Web は Yagura.Ingestion を参照しない参照構造のため、インスタンスではなく Meter 名の一致で統合する）。

逆引き解決の 4 カウンタ（ADR-0007 決定 6。ui.md §9 が管轄）は「意味論の異なる別々の事象」の扱い（本節の判断と同じ——タグ付き単一カウンタにしない）で独立の計器名を持つ。実装は `Yagura.Web.Diagnostics.ReverseDnsMetrics`（Meter 名は同じく `Yagura`）。**本 4 種は表示補助の観測でありログの損失計上ではない**（「損失は必ずどれかのカウンタに計上される」の対象外。状態帯の異常判定の入力にもならない）。メタデータ領域への永続化（§4.3）は行わず、プロセス起動からの累積のみを状態画面に表示する。

UDP 受信エラー（Issue #142）は、`ReceiveAsync` の個々の失敗が必ずしもデータグラム損失と 1 対 1 対応するとは限らない診断用カウンタであるため、逆引き解決の 4 カウンタと同じ扱いとする——メタデータ領域への永続化（§4.3）・状態画面の健康状態判定（`Yagura.Host.Observability.SystemStatusReader`）の入力には含めず、プロセス内累積のみを保持する。実装は `Yagura.Ingestion.Diagnostics.IngestionMetrics`（`Yagura.Ingestion.Udp.UdpSyslogListener` から計上）。

**スプール末尾破損破棄（Issue #201）**: `SpoolDrainCoordinator.DrainSegmentAsync` がスプールセグメント末尾の破損（`corruptTailDetected`）を検出した際、従来はその破棄分がどのカウンタにも計上されないままセグメントが削除されていた——「カウンタに計上されない喪失は重大」（§3.1）に反する観測ギャップだった。本カウンタで計上する。他の発生箇所別カウンタと異なり**単位はレコード数ではなくバイト数**にした——破損した末尾はフレーム境界が保証されない（`SpoolSegmentReader` の実装ノート参照）ため、そこに何件のレコードがあったはずかを数える手段が原理的に存在しない。単位差の扱いは次のとおり判断した:

- **状態画面の健康状態判定（`SystemStatusReader.Assess`）**: 観測窓内の増分を他 5 種と合算した「> 0 か」の判定にのみ使うため、単位混在の影響を受けない。バイト単位の増分であっても取りこぼしが発生した事実に変わりはなく、異常判定（`LossObserved`）を発火させる。Issue #199 で確立した「一時保管ゲージが 0 に戻っても一部喪失があれば正常表示に戻さない」原則を、末尾破損の喪失（一時保管ゲージには表れない）にも同様に適用する
- **Dashboard の「取りこぼし（累計）」集計**: `IsLoss` な全カウンタの単純合計（本カウンタ追加まで単位は全種「件」で統一されていた）に含める。単位混在は本来望ましくないが、(a) 本カウンタが非ゼロになるのは torn write（クラッシュ・強制終了時の末尾破損）発生時のみで 1 回の発生量はフレーム 1 件分程度に収まり実運用で「件」総数を大きく歪める規模にはならない、(b) 除外すると本 Issue の目的（観測ギャップの解消）を閲覧面で再現することになり本末転倒、の 2 点から単純合計へ含める判断とした（詳細な内訳確認は状態画面のカウンタ一覧で行う——ラベルに「単位はバイト」と明記する）
- **ベンチの厳密な送信数突合（`tools/Yagura.Bench` の `CounterReconciler`）**: 送信メッセージ数との等式比較（`difference == 0`）を行うため、バイト値を含めると単位ズレで誤判定し得る。本カウンタは除外する（正常系のベンチシナリオでは常に 0 のため実害はない）
- **計上タイミング（重複計上の防止）**: `SpoolDrainCoordinator` は破損検出のたびに計上せず、**セグメント削除（= 当該セグメントの drain 完了が確定するタイミング）の直前でのみ**計上する。書き込み失敗により同じ破損セグメントを次周期で再読み込みする場合でも、削除が確定するまでは計上しないため二重計上しない

実装は `Yagura.Ingestion.Diagnostics.IngestionMetrics`（`Yagura.Ingestion.Persistence.SpoolDrainCoordinator` から計上）。メタデータ領域への永続化（§4.3）は他 6 種の発生箇所別カウンタと同じく additive-only で対象に含める（`MetadataStoreFileFormat.CountersFileFormat.SpoolCorruptTailDiscardedBytes`）。

### 4.2 OS レベル取りこぼしの観測

アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る、という前提で設計する（ADR-0002 決定 2）。

**現在の姿（[ADR-0016](../adr/0016-os-drop-gauge-mechanism.md) 決定 3。2026-07-18 accepted・撤去実装 = Issue #308）**: 当初採用した標準 API（`IPGlobalProperties` の UDP 統計）のゲージは、検証済み環境で受信・破棄のいずれも反映しないことが実測確定し（下記「覆域の限界」）、**製品コードから撤去済み**である——Meter `Yagura` に `yagura.os.udp.*` は存在しない（不在は回帰テスト `IngestionMetricsOsUdpAbsenceTests` で固定。再導入は同 ADR 再評価トリガ (d) 陽性時の amendment を要する）。OS レベル破棄への実効的な対処は、①受信バッファ拡大（下記。第一の緩和・自動で効く予防）、②ベンチの閉じた系での引き算導出（測定時のみ）、③送信元別の無音化検出（UI-4。運用時——検出対象は途絶のみ）の 3 手段が担い、**本番稼働中の OS 側残余ロスを直接指す runtime シグナルは存在しない**（この空白の正直な受容が ADR-0016 の帰結である）。以下の各項は、現在も有効な設計（受信バッファ・ベンチ導出）と、採用から撤去に至る検証記録である:

- **取得手段（当初採用——M4-4。のち撤去）**: `System.Net.NetworkInformation.IPGlobalProperties.GetUdpIPv4Statistics()` / `GetUdpIPv6Statistics()` の `IncomingDatagramsDiscarded` を採用していた。Windows ARM64・.NET SDK 10.0.301（2026-07-05 確認）で、実際に非ゼロの値が取得でき、UDP トラフィックの発生に応じて値が単調増加する「生きた」カウンタであることを実機で確認した（IPv4・IPv6 の両方）——ただしこの観測は後に M7-2 で「本ワークロードと独立に稀に発生する背景増分」と再解釈された（下記「覆域の限界」）
- **粒度の限界（v0.1 の検証範囲での結論）**: 上記 API は**システム全体統計**であり、意図していた「本製品のソケット単位・ポート単位」の粒度は .NET 標準 API では得られない（同居プロセスの UDP 破棄も含む値になる）。**本判断の検証範囲は .NET 標準 API に限定した**——「あらゆる手段で不可能」の証明ではない。ADR-0002 の再評価トリガの発動として、v0.1 の方式をシステム全体統計に確定した（[ADR-0002 §改訂履歴 1](../adr/0002-architecture-principles.md) で確定済み。再挑戦の条件は同 ADR の再評価トリガに新設）
- **検証対象外とした代替手段（採否の記録）**: 次の手段は per-socket/per-port 粒度を実現し得る可能性があるが、v0.1 では**検証せずに見送った**（不可能と判定したのではない）——(a) **ETW**（`Microsoft-Windows-TCPIP` 等のプロバイダ購読）: リアルタイムセッション管理・イベントスキーマの OS バージョン間差異の保守コストが v0.1 の価値に見合わない、(b) **Windows パフォーマンスカウンタ**: カテゴリ/インスタンス名のローカライズ・存在有無の環境差の吸収コスト、(c) **WinSock 拡張 API**: P/Invoke 依存と受信経路への侵襲性。いずれも将来の再評価対象（ADR-0002 の再評価トリガ参照）
- **同居環境での偽陽性（検証記録——撤去後も証拠能力の議論に有効）**: 本統計は**システム全体値**であり、仮に機能する環境でも「本製品が取りこぼした」と断定できる粒度を持たない（DNS・SNMP トラップ受信・監視エージェント等の同居環境では無関係な破棄が混入する）。撤去前の UI 判断（値を表示せず常時可視の説明のみ——M8-3 / Issue #70）の記録と確定文言・再評価条件は ui.md §5.5（計器撤去後も説明の掲示は運用者導線として維持している）
- **証拠能力の禁止（ADR-0016 決定 4）**: 旧ゲージの値を本製品の取りこぼしの証拠として引用してはならない（撤去前の版を稼働させている環境の読み値にも適用する）。また本統計は §4.1 系のメトリクスであり、security.md §4 の監査記録（管理操作・拒否試行の証跡）ではない——監査要件の充足根拠として引用しないこと（両者は射程の異なる別個の禁止）
- **権限要件（解消）**: 旧ゲージの API 呼び出しが仮想サービスアカウント権限で完結するかの確認（M-15）は、ADR-0016 決定 3 の撤去により**要件消滅**した（トリガ (d) 陽性で再導入する場合の実装 PR に吸収。[§9 M-15](#9-実測待ち一覧) に解消記録）
- **撤去前実装の記録**: 旧ゲージは「プロセス起動時からの差分」として公開し（累積値の混入による誤読防止）、取得不可環境（`NetworkInformationException` / `PlatformNotSupportedException`）では登録しない「枠のみ」へ縮退する設計だった。詳細は git 履歴（`IngestionMetrics` の 2026-07-18 以前）と ADR-0016 決定 3 が保持する
- **受信バッファサイズを設定項目にする**: OS 側ロスの第一の緩和はソケット受信バッファの拡大である。既定値は **1 MiB** で確定（[M-2](#9-実測待ち一覧) 実測 2026-07-05: 64 KiB（OS 既定）では持続負荷・バーストとも総損失の相当割合（バーストでは 86.7%）が OS レベルの無計測ロスだったのに対し、1 MiB 以上では全実測条件でこのロスがゼロになり、損失が発生箇所別カウンタに計上される形へ転化した——本節の設計原則が機能することの一次データ。1 MiB 超への拡大に追加の恩恵は確認されなかった。指定可能範囲は 64 KiB〜256 MiB（上限は誤入力を弾く安全弁であり実測対象ではない）。設定時に実効値を読み戻し、丸め・失敗があれば警告ログへ記録する（開発機では丸め・失敗とも非再現だが防御的に維持）。実測記録: `tools/Yagura.Bench/results/2026-07-05-dev-machine-rcvbuf/`）
- **覆域の限界（M7-2 実機検証で確定。2026-07-05。理由は 2026-07-18 の Windows Server 実機検証で訂正・特定）**: OS バッファ破棄が実際に発生している状況（ベンチのバースト実測: 送信 2000・アプリ受信 1006・Q1 破棄 0）でも、`IncomingDatagramsDiscarded` の増分は 0 である。**M7-2 のクロスマシン実機検証（2026-07-05。受信 = YAGURA-STG・送信 = 開発機、いずれも Windows 10.0.26200）でも、読み取り停止ソケットへの 100,000 通送出（受信バッファ溢れ確実）で `IncomingDatagramsDiscarded`/`IncomingDatagramsWithErrors` の増分 0** を確認し、独立した再実行でも再現した。一次資料: `tools/Yagura.Bench/results/2026-07-05-cross-machine-udp-stats/`。

  **理由（2026-07-18 の Windows Server 2025 / .NET 10 実機検証で特定。一次資料: `tools/Yagura.Bench/results/2026-07-18-server-udp-stats-trigger-d/`）**: `IncomingDatagramsDiscarded` は `netstat -s -p udp` の **`No Ports`**（宛先ポートに listener が無く破棄された数）に対応しており、**Windows の UDP MIB（`MIB_UDPSTATS`）にはソケット受信バッファ満杯による破棄を表すカウンタがそもそも存在しない**。したがって本ゲージがバッファ破棄を観測できないのは、OS バージョンや送信元の位置に依存する事象ではなく**カウンタ体系の構造的な帰結**である（実測: 49,994 通が実際にバッファ溢れで失われた場面で discarded 増分 0。同時に受信総数 `DatagramsReceived` は +50,001 と全 datagram を計上しており、datagram が計上経路を通過したうえで破棄が記録されないことの直接証拠）。

  > **訂正（2026-07-18）**: 本節は当初、限界の理由を「**自己宛 UDP は Windows の UDP 統計のカウント経路をバイパスする**」「受信総数 `IncomingDatagrams` の増分も 0」と説明していたが、**いずれも誤りであり撤回する**。受信総数は自己宛でも正確に計上される（実測で送信数と完全一致）。当時の「受信総数の増分 0」は、検証スクリプトが**存在しないプロパティ名** `IncomingDatagrams` を読んでいた測定アーティファクトである（正しくは `DatagramsReceived`。PowerShell は存在しないプロパティを黙って `$null` と返す）。破棄側の観測は実在プロパティを読んでおり有効で、上記の構造的理由により結論（ゲージは破棄観測に使えない）は変わらない。経緯: [ADR-0016](../adr/0016-os-drop-gauge-mechanism.md) 改訂履歴 1。**本発見は [ADR-0016](../adr/0016-os-drop-gauge-mechanism.md) として accepted（2026-07-18）**（ADR-0002 決定 2 の OS ゲージ観測手段の部分 supersession。Issue #272。当初 ADR-0002 §改訂履歴 2 に amendment として記録したものを、README 判定基準に照らし supersession へ再分類した）。**同 ADR 決定 3 によりゲージは製品コードから撤去済み（Issue #308）**。OS レベル破棄の実効的な観測・緩和は、①受信バッファ拡大（M-2。第一の緩和）、②ベンチの閉じた系での引き算導出（測定時）、③送信元別の無音化検出（UI-4。運用時）が担う
- ベンチ（§5）の性能検証は「送った数 = 保存された数 + 全カウンタの合計」の突合で行う。ただし上記の覆域の限界により、**自己宛送信であるベンチのトラフィックの OS バッファ破棄は本統計では観測できない**——ベンチは閉じた系（送信数はベンチ自前の正確な計上・OS バッファ以降は全カウンタで観測済み）であることを利用し、**残差を「OS ソケットバッファでの破棄（導出値）」として引き算で観測する**（M7-2 で確定した方式。`tools/Yagura.Bench` の `CounterReconciler` 参照）。OS 統計の読み値はベンチ側の独立実装（`OsUdpStatsProbe`——製品コードに依存しない）で参考記録に留める（非ゼロ = 他プロセス宛の外部トラフィックの破棄が同時に起きていた、の意味）

### 4.3 メタデータ領域とカウンタの再起動をまたぐ永続化

- **メタデータ領域**は、カウンタ累積値・停止イベント・生存時刻を保存する**ホスト管轄の独立ローカルファイル**である。DB provider には依存しない——DB 障害中・provider 切替中にこそ記録が必要になるため、DB 内には置かない。したがって本番昇格（SQLite → SQL Server）の移行対象にも含まれない
- メタデータ領域は「ログが無い期間の説明責任」を担う証跡であり、**ログ本体と同等の保護対象として ACL は ADR-0004 決定 5 に従う**（スプールと同じ扱い）
- カウンタは累積値として**一定間隔でメタデータ領域に永続化し、正常停止時には必ず書き切る**（§1.3）。起動時に復元して継続する
- クラッシュ時は最終永続化以降の増分が失われ得る。損失窓を永続化間隔（仮値: 10 秒。[M-11](#9-実測待ち一覧)）以下に限定し、クラッシュ発生自体を §4.4 で可視化することで補う

#### 4.3.1 ファイル形式と原子性（M4-4 実装判断）

- **形式**: JSON の**全体書き換え + 原子的置換**（一時ファイルへ書き込んだうえで `File.Replace`、初回保存のみ `File.Move`）を採用した。configuration.md §3 の設定ファイル保存（`YaguraConfigurationWriter`）と同じ方式であり、新規の追記型フォーマット（スプールのようなセグメント形式）は設けない。判断理由: (i) 書き込みは「一定間隔 + 停止時」の低頻度であり、スプールほどの書き込み性能は要求されない、(ii) 内容（カウンタ 12 個——Issue #143・#140 で TCP 接続断・TCP 接続アイドルタイムアウト・TCP メッセージ破棄（上限超過）の 3 個、オーナー決定 2026-07-09 で TCP 接続再同期上限・TCP フレーミング進捗タイムアウトの 2 個を追加——・停止イベント・生存時刻）は数百バイト程度で全体書き換えのコストが無視できる、(iii) 本リポジトリに既に検証済みの原子的置換パターンが存在し、同じ判断枠組みを流用できる
- **破損時のフォールバック**: 原子的置換により「対象ファイルへ書き込み途中の内容が見える瞬間」自体が発生しないため、対象ファイルが破損する経路は原理的にない。読み込み時に検出し得るのは「ファイルは存在するが JSON として不正」（手編集・別要因のディスク破損等）のみであり、この場合は例外を投げず**ゼロから再開（既定値）+ 警告ログ**にフォールバックする
- **文字コード**: UTF-8 BOM なし（設定ファイルと同じ理由）
- 実装: `Yagura.Host.Observability.MetadataStore`

#### 4.3.2 カウンタ累積値の合成（前回までの累積 + 今回プロセス分）

`System.Diagnostics.Metrics.Counter<T>` は加算専用の書き込み API のみを公開し、現在の累積値を読み戻す標準 API を持たない。`MeterListener` で購読する経路はあるが、購読の生存期間管理が定期永続化タイマーと二重の状態管理になり複雑化するため採らなかった。そのため `IngestionMetrics` 自身が各 `Record*()` 呼び出しと同時に `Interlocked` でプロセス内累積値（Counter とは別に保持する `long` フィールド）を持ち、メタデータ領域はこのスナップショットを読み書きする設計にした。起動時は、メタデータ領域から読み込んだ前回値を「種」としてこのプロセス内累積値へ設定（`SeedCumulativeCounters`。加算ではなく上書き）してから受信を開始することで、以後の加算が前回分の上に積み増される——「前回までの累積 + 今回プロセス分」を実現する。実装: `Yagura.Ingestion.Diagnostics.IngestionMetrics` / `IngestionCounterSnapshot`。

### 4.4 受信断（再起動・クラッシュ）の可視化

再起動時間はロス時間である（ADR-0002 帰結）。計画再起動を含め、受信できなかった区間を記録として残す:

- **正常停止**: 停止イベント（時刻）を記録してから終了する。次回起動時に「停止〜起動」を受信断区間として保存する
- **クラッシュ**: 稼働中は一定間隔（仮値: 10 秒。カウンタ永続化と同一周期とするかは [M-11](#9-実測待ち一覧) で確定）で生存時刻をメタデータ領域に更新する。起動時に前回の正常停止イベントがなければ、最終生存時刻を近似の断点として受信断区間を保存する（近似であることを区間に印す）
- **受信断区間の保存**: 区間の確定は起動時に行い、保存は通常のパイプライン（スプールを含む耐障害経路）を通す——DB 障害中の起動でも記録が失われない
- **計上の限界を明記する**: クラッシュ（強制終了を含む）の瞬間に Q1・Q2 上およびスプール未書込だったログは、失われたうえ**件数をどのカウンタにも計上できない**（計上する主体ごと消えるため）。この喪失は件数ではなく、クラッシュ由来の受信断区間（近似断点の印付き）として**区間で観測可能にする**。「観測不能なログ喪失を作らない」（ADR-0006 基準 2 定義①）に対する本設計の回答は「件数計上が原理的に不可能な局面では、発生の事実と時間範囲を記録する」である
- **設定変更によるリスナ再構成の計画瞬断も、同じ区間記録の仕組みで記録・可視化する**（時刻・対象リスナ・契機となった設定変更。configuration.md §3）
- 受信断区間は検索・ダッシュボードで通常のログと同じ時間軸上に可視化する（「この時間帯はログが無いのではなく、受けられなかった」を見分けられるようにする。表示は ui.md で定める）

### 4.5 受信プロトコル別のロス特性

ロスの現れ方はプロトコルで異なる。カウンタの解釈を誤らせないため、特性を仕様として明記する:

- **UDP**: 到達しなかったこと自体をサーバ側は観測できない（ネットワーク上のロスは §4.1 のどのカウンタにも現れない）。サーバで観測できるのは OS バッファ以降のみ。「カウンタ全部 0 = ロス 0」ではなく「サーバ到達後のロス 0」である——この解釈は利用者向けドキュメントにも明記する（[§10](#10-利用者向けドキュメントへの申し送り)）
- **TCP**: 転送自体は信頼できるが、**接続断のときに送信側が未送信分をどう扱うか（保持・再送・破棄）は送信側実装に依存し、サーバからは制御も観測もできない**。サーバ側で観測・記録できる事象は「接続の確立・切断（相手・時刻）」「切断時の不完全メッセージ」であり、§4.1 のカウンタに計上する。サーバ側は Q1 満杯時に読み取りを止めて TCP のフロー制御に委ねる（§3.1）ことで、送信側の保持・再送能力を活かす設計とする。読み取り停止中も接続は保持されるため、**同時接続数に上限を設けて資源を有限化する**（設定項目。既定値は [M-14](#9-実測待ち一覧)。上限到達時は新規接続を拒否し、拒否は §4.1 のカウンタに計上する）。受信断（§4.4）中の TCP 送信元は接続拒否を検知できるため、送信側の再送機能が働き得る——これは UDP に対する利点として利用者向けドキュメントで案内する（[§10](#10-利用者向けドキュメントへの申し送り)）
  - **1 メッセージの逸脱への耐性（Issue #143）**: RFC 6587 の厳密仕様違反（octet-counting のフレーム間に紛れた LF/CR・空行、1 メッセージのサイズ上限超過）を理由に接続を丸ごと切断し、その持続接続の以後の正常ログまで全損させる設計は過剰反応と判断した。**octet-counting のフレーム間に紛れた LF/CR は寛容にスキップして再同期する**（相互運用で余分な改行を挟む送信元が現実に存在するため）。**1 メッセージのサイズ上限超過（octet-counting・non-transparent-framing とも）は当該メッセージのみを破棄し、接続は維持する**（破棄は §4.1「TCP メッセージ破棄（上限超過）」に計上し、警告ログにも記録する）。接続を切断するのは、MSG-LEN の桁の途中に数字以外のバイトが混入する等、**再同期できない深刻な破損の場合に限る**（実装: `Yagura.Ingestion.Tcp.TcpFrameDecoder` / `TcpFrameSizeExceededException`）
  - **アイドルタイムアウト（Issue #140）**: 同時接続数上限による有限化だけでは、無言・低速な接続（何も送らない、または 1 バイトだけ送って黙る）がその枠を占有し続け、正常な送信元の新規接続が拒否され続ける経路（slowloris 型の資源枯渇）を防げない。**接続ごとにアイドルタイムアウト（設定項目。既定 5 分——一般的な syslog 送信元の送信間隔に対して十分な余裕を持たせつつ、無言接続を有限時間で回収できる値としての暫定値。M-14 の同時接続数上限の実測確定と合わせて再評価する候補）を設け、無通信が続く接続を切断して枠を返す**。切断は §4.1「TCP 接続断」「TCP 接続アイドルタイムアウト」の両カウンタに計上する（実装: `Yagura.Ingestion.Tcp.TcpSyslogListenerOptions.IdleTimeout`）
  - **寛容化の天井——再同期バイト数上限とフレーミング進捗タイムアウト（オーナー決定 2026-07-09）**: 他社実装の調査（rsyslog / syslog-ng / Fluent Bit / Vector / NXLog / Graylog。2026-07-09）では、**フレーミングエラー即切断という一次防御が業界主流**であり（truncate 継続型は rsyslog で CVE-2022-24903 の前歴があり非推奨）、Issue #143 の寛容化はこの一次防御を外した状態だった——寛容化のままでは「LF を送らない・再同期不能な破損を起こさない範囲のゴミデータ」を送り続ける接続に強制切断の天井がなく、しかも読み取りが起き続けるためアイドルタイムアウトでも回収できない（slowloris 型資源枯渇の亜種）。**Yagura は寛容化と次の 2 つの天井の組で、業界主流の即切断と同等の防御水準を保つ**: ①**再同期バイト数上限**——有効なメッセージが 1 件も確定しないまま読み捨てたバイト数（フレーム間 LF/CR のスキップ・上限超過メッセージ本体の読み飛ばし・non-transparent-framing の破棄行）の上限。既定 **128 KiB**（仮値。実利用で確定。既定の 1 メッセージ上限 64 KiB を超えたメッセージ 1 通分の読み飛ばし + 余裕を許容しつつ、調査で得た参考値——Vector max_length = 100 KiB / rsyslog MaxFrameSize 既定 = 200,000 バイト——の間に収まる 2 の冪として選定）。②**フレーミング進捗タイムアウト**——バイトは届いているのに有効なメッセージが 1 件も確定しないまま経過した時間の上限（低速トリクル対策。アイドルタイムアウトとは別軸）。既定 **60 秒**（仮値。実利用で確定。オーナー指定範囲 30〜60 秒の上端——既定の 1 メッセージ上限 64 KiB を 60 秒かけても確定できないのは 9 kbit/s 未満の病的な低速であり、正常な低速送信元の巻き込みを最小化する側に倒した）。**どちらのカウント/タイマーも有効なメッセージが 1 件確定するたびにリセットされ、正常な送信元は巻き込まれない**。超過による切断は §4.1「TCP 接続断」に加えて「TCP 接続再同期上限」「TCP フレーミング進捗タイムアウト」の内訳カウンタと警告ログに計上する（実装: `TcpFrameDecoderOptions.MaxResyncBytes` / `TcpSyslogListenerOptions.FramingProgressTimeout`）

### 4.6 ゲージと能動通知

カウンタ（累積値）は「起きたこと」の記録であり、「今どうなっているか」と「悪化しつつあること」は別の手段で見せる:

- **ゲージ**（現在値のメトリクス）: Q1・Q2 の使用率 / TCP 現在接続数（対上限比）・読み取り停止中の接続数 / スプール使用量（対上限比）/ スプール内最古レコードの滞留時間 / drain の未消化バックログ / UI の circuit 現在数（対上限比）・再接続回数・circuit 喪失回数（ui.md §9）。これらは §3.2.2 の「振動していないこと」「速度不足が進行していること」を運用者が現在形で確認する手段である
- **能動通知**: UI を開いていない夜間にスプールが満ちていくのが最悪シナリオであり、pull 型の観測だけでは実効性がない。次の事象は **Windows イベントログへの警告書き出し（既定）**の対象とする: スプール退避の継続 / スプール上限接近・到達 / スプール書込失敗 / スプールなし縮退での起動（§1.2）/ 自己検証の失敗（§3.2.5）/ HTTPS 証明書の期限接近（configuration.md §6）/ circuit 上限の到達（security.md §2.2）。イベントログは Windows 管理者の既存監視（イベントログ転送等）のレールに乗る。通知の書き出し自体が失敗しても、ゲージとカウンタは独立した観測チャネルとして残る（通知失敗時の扱いの詳細は委任先で定める）。通知の詳細（イベント ID 採番はセキュリティ設計文書の監査系と統一する）と、通知を受けた運用者の行動手順は operations.md・セキュリティ設計文書に委ねる
  - **実装状況（Issue #149。M4-6）**: 上記のうちスプール系 4 種（退避の継続・上限接近・上限到達・書込失敗）とスプールなし縮退起動、および database.md §5.3 の SQL Server Express DB 容量接近を配線済み。**周期監視 4 種**（スプール使用率の上限接近・到達 / 退避の継続 / 監視対象ボリュームの空き容量（DriveInfo。M-16）/ Express 容量接近）は `Yagura.Host.Observability.ActiveNotification.ActiveNotificationMonitor` が仮値の周期（1 分。M-16）で評価し、トリガごとの抑制窓（仮値 15 分。M-16）で連発を抑える。**周期評価はループ全体が包括的な例外保護を持つ**——評価中の未捕捉例外はエラーとしてイベントログへ記録（抑制窓付き）し、ループは次周期で再試行する（監視自身が無警告で恒久停止する経路を残さない。PR #188 レビュー指摘への対応）。**空き容量の監視対象はデータルートに加えて、スプール有効時はスプール置き場所のボリュームを含む**（`Spool:Directory` が別ドライブに向いた構成でも「夜間にスプールが満ちていく」現場のボリュームを見逃さない。同一ボリュームなら読み取り時に 1 件へ重複排除する——同レビュー指摘への対応）。**即時通知 1 種**（スプール書込失敗）は発生箇所 `Yagura.Ingestion.Persistence.PersistenceWriter.EvacuateSingleRecordAsync` から即座に警告する（抑制窓は恒久障害警告と同じ仮値 5 分）。イベント ID は 1001〜1007（security.md §4.3 の 1000 番台区画。同節に一覧を記録）。**自己検証の失敗（§3.2.5）は Issue #152 で実装済み**——想定どおり `ActiveNotificationMonitor.EvaluateOnceAsync` に評価メソッドを 1 つ追加する形で乗せた（ループ側の例外保護があるため、追加メソッドの例外安全性がループの生存を左右しない、という当初の想定どおりの構造で実装できた）。投入周期は仮値 1 日 1 回（M-11）、投入したマーカーが drain へ合流判定されるまでの期待時間（タイムアウト）は仮値 10 分（M-16）。**タイムアウトのバックログ起因判別（Issue #202）も実装済み**——drain の進捗（消化済みセグメント削除の累積カウンタ `DiskSpool.DeletedSegmentsTotal` の増分）を直近の期待時間内に観測していれば未消化バックログの滞留（EventId 1010。警告）、観測していなければ経路障害の疑い（EventId 1009。エラー）とする（一度 1009 と判定したら当該マーカーの追跡が終わるまで 1010 へ戻さないラッチ付き）。判別機構の設計は §3.2.5 実装状況を参照。**HTTPS 証明書の期限接近・circuit 上限の到達は本 Issue のスコープ外**（該当機能側の実装時に配線する）
  - **退避からの復帰（消化完了）はイベントログへ通知しない（Issue #132 の判断）**: 一時保管への退避が解消（`DiskSpool` の使用量が 0 に戻る＝退避データの DB 格納が完了）した際、閲覧画面（状態帯・ダッシュボード）は `Yagura.Host.Observability.SystemStatusReader` のリアルタイムゲージ判定により直ちに復帰表示へ切り替わる（ui.md §5.1「消化完了」）が、**この復帰そのものは 1000 番台の新規イベント ID を割り当てて通知しない**。理由: (1) 本節冒頭の能動通知の存在意義は「UI を見ていない夜間でも対応が必要な事象に気づけること」であり、自己回復（消化完了）は運用者の対応を要しない、(2) 退避 → drain の反復が短時間に多発する局面（§3.2.2 の振動）では復帰通知も同じ頻度で発生し、抑制窓（本節の連発抑制）の設計意図（ノイズ削減）に反する、(3) 1004（スプール退避の継続）の警告は既存どおり残り、日中の状態把握は状態帯・ダッシュボードのリアルタイム表示（更新間隔 10 秒）が担うため、イベントログでの重複通知の必要性が薄い。以降の番号（1009 は Issue #152 の自己検証失敗、1010 は Issue #202 のバックログ起因判別で採番済みのため 1011 以降）は将来の新規事象のために温存する
- 監査記録（管理操作・拒否試行）の書き込み失敗の観測（ADR-0004 決定 7）は、本節の能動通知と同じ機構を使うが、要件の定義は**セキュリティ設計文書に委任する**
- **第 2 の書き出し先: メール（[ADR-0017](../adr/0017-email-notification.md)。opt-in・既定無効。Issue #350）**: 上記の「Windows イベントログへの警告書き出し（既定）」は、**イベントログを既存の監視基盤に取り込んでいる現場**では十分に機能する。一方、本製品の主対象層には**そのレールを持たない現場**が現に存在し、そこではイベントログへの書き出しは「誰も見ないファイルに書く」ことと変わらず、能動通知の存在意義そのものが実効化しない。メール送信はこの層への補完であり、**イベントログ方式を置き換えるものではない**（両者は前提の異なる現場をそれぞれ担当する）。
  - **実装位置**: 発火点のコードには一切触れず、カスタム `ILoggerProvider` を Host の合成ルートで第 2 のシンクとして登録する（ADR-0017 決定 7）。EventLog プロバイダと同じ `ILogger` レールに乗るため、`Yagura.Ingestion` 発の即時通知（1005）や起動時経路も自動的に捕捉される。対象の選別はイベント ID の allowlist（security.md §4.3 の「メール通知」列）のみで行い、**利用者の `Logging:*` フィルタからは独立させる**（イベントログのノイズ調整のつもりの設定変更が黙ってメールを止める経路を作らない）
  - **正本はイベントログのまま**: メールは有界キュー + 有界再試行（1 通あたり 1 回）+ 破棄という at-most-once であり、**「メールが来ない ＝ 正常」とは限らない**。とくに起動時に一度しか発火しない事象（1001 等）は、復電直後などリレー側が未収束のタイミングで恒久的に失われ得る。この限界と引き換えに受信・保存・UI への影響経路を一切作らない設計であり、チャネル自体の健全性は管理画面の常設カード（ADR-0017 決定 5）で確認する
  - **流量制御**: 同一イベント ID は仮値 60 分に 1 通、全体で仮値 10 通/時（エラーレベルは全体上限の対象外）。閾値は設定キーとして公開しない（configuration.md §8。2026-07-18 オーナー決定）
- **送信元の途絶検知（[ADR-0018](../adr/0018-source-silence-detection.md)。opt-in・既定無効。Issue #351）**: 上記の能動通知が「サーバ側の異常」を対象とするのに対し、本機能は**「来るはずのものが来ていない」**——ウォッチリストに明示登録した送信元からの受信が閾値を超えて途絶した事象——を検出する（イベント ID 1027〔個別警告〕・1028〔一斉集約警告〕・1029〔復帰の情報記録〕。1027・1028 はメール通知の対象）。
  - **実装位置**: 新しい常駐機構は作らず、`ActiveNotificationMonitor` の既存の周期評価（1 分）が判定器（`SourceSilenceDetector`）を駆動する。「受信した」の記録は共有追跡コンポーネント（`Yagura.Storage.Observability.ISourceActivityTracker`。実体は Host 側）へ、解析段の消費ループとスプール drain の合流点の 2 系統が単調クロック基準で書き込む（NTP ステップでの一斉誤発火を防ぐ。深いスプール滞留 + 再起動で「直前まで送っていた装置」が途絶に見える偽陽性は drain 側の遅延反映が塞ぐ）。起動時は `QuerySourceActivityAsync` を 1 回照会して登録済みエントリの基準を DB の最終受信時刻へ seed する（決定 3。閾値超過のエントリと照会失敗時は起動時刻仮基準——Issue #381 で実装）
  - **サーバ都合の受信断との区別（決定 3）**: 構成済みの全受信リスナが受信不能な間（bind 失敗の縮小継続等）は途絶判定を保留し、回復時に**保留中に閾値超過となったエントリ**を回復時点で再アームする（各エントリの再検知は当該エントリの閾値で律速——起動時の再アームと同一規則。対象外のエントリは追跡時計を保ち、本来の「最終受信 + 閾値」で判定される——全エントリを再アームすると短い受信断の反復で検知が恒久先送りされ得る。Issue #381 で ADR の規定へ訂正）。判定源は `IngestionPipeline.ListenerAvailability`（起動 Outcome・再構成 Outcome・CF-6 復旧の 3 系統を現在状態へ畳んだスナップショット）で、監視ループが周期ごとにポーリングする。部分受信断（UDP のみ down 等）は保留せず、警告本文にサーバ側受信経路の状態を併記して対応する
  - **UI**: ダッシュボードの送信元別受信状況（ui.md §4 UI-4）に登録済みマークと途絶中の強調を表示し、ウォッチリストの管理は認証済み管理面の設定画面（候補選択主経路・閾値の明示確定必須。ADR-0018 決定 4）で行う。変更は監査 2023 にエントリ列挙つきで残る（決定 5）

### 5.1 ベンチハーネス（v0.1 の開発範囲）

- 構成: 負荷生成器（送信数を正確に把握できる自前実装）→ 本体 → 検証器（保存件数 + 全カウンタ + OS 統計の突合）
- 計測対象の最小セット: 受信スループット（UDP/TCP 別）、破棄ゼロで維持できる持続流量、**バースト負荷時の Q1 破棄の発生有無**（§3.1 の前提検証）、スプール発動 → 追いつきの所要、SQLite / SQL Server 各 provider の書き込み上限
- 容量・バッチサイズ等の設計値（[M-1](#9-実測待ち一覧)〜[M-3](#9-実測待ち一覧)）はこのハーネスの実測で確定し、本文書に記録する
- ベンチとは別に、スプールの結合部（drain → DB コミット → セグメント削除）を検証する機能テスト（発動 → 復旧 → 件数突合）を CI に置く（§3.2.5）

### 5.2 合否判定 — 回帰は基準比、公称は絶対値

判定方式は 2 層に分ける（ADR-0002 決定 4 の委任と ADR-0006 基準 1 の要求を、実行環境の揺らぎと両立させる）:

- **CI の回帰判定（毎 PR）は基準比とする**: リポジトリに記録した基準値との比を判定し、許容帯を超えて劣化したら不合格とする。CI 実行環境の揺らぎで絶対値は安定しないため、絶対値の合否は CI では行わない。許容帯は CI 環境の揺らぎを実測してから確定する（[M-5](#9-実測待ち一覧)）。**実装（M7-3。Issue #62）**: `Yagura.Bench` の `--compare-baseline` オプションが基準値ファイル `tools/Yagura.Bench/baselines/ci-baseline.json` との基準比判定を行い、`.github/workflows/ci.yml` がテスト後に代表シナリオの短縮版 2 本（SustainedZeroDrop・ProviderWriteCeiling）で実行する。突合成立（IsReconciled）は比ではなく不変条件として無条件に要求する。基準値の更新手続きは [conventions.md](../development/conventions.md)「CI 回帰ベンチの基準値更新」節に定める
- **基準値の更新はなし崩しを防ぐ手続きで行う**: 更新は PR として明示的に行い、**引き下げ方向の更新には理由の分類（実行環境の変更 / 機能追加による意図的なトレードオフ）を必須とする**。性能公称値（M-6）の確定後は、更新 PR に基準値と公称値の余裕率を明示し、余裕率が閾値を割る引き下げはリリース判定と同じ重さ（オーナー承認 + 公開の場での記録）で扱う——「環境変更への追従」名目の小さな引き下げの積み重ねでリリース時に初めて公称値割れが発覚する事態を防ぐ
- **公称値の判定（リリース時）は絶対値とする**: 定義済みの基準環境上で実測し、全体設計書に記録した性能公称値を満たすことを確認する。公称値と基準環境の定義は実測を経て確定する（[M-6](#9-実測待ち一覧)）。**確定後の公称値の引き下げは ADR-0006 の supersession に準ずる手続き**（PR + ペルソナレビュー + 変更理由の公開）を要する

### 5.3 SQLite の実用限界と昇格案内

ゼロ設定のまま大量ログを受けると SQLite は飽和し得る（ADR-0002 受け入れリスク）。「静かに落ちる」ことを許さないため:

- **飽和のシグナルはスプール退避カウンタ**とする。保存先が生きているのに退避が続く状態は「持続的な速度不足」そのものであり、専用の検知機構を持たずに §3.2.1 の機構から得られる
- **誤発火の防止**: 退避を「速度不足（発火に数える）」と「保存先の障害（数えない）」に分類する。判定は退避の形態（Q2 溢れかタイムアウトか）ではなく、**同一観測窓内に成功した書き込みが存在するか**で行う——成功が混在していれば保存先は生きており速度不足（飽和した SQLite の書き込みが Q2 溢れではなくタイムアウトとして現れても沈黙しない）、成功がなければ障害であり誤案内しない。SQLite の飽和が実際にどちらの形態で現れるかはベンチで確認する（[M-7](#9-実測待ち一覧)・[M-13](#9-実測待ち一覧) の決め方に含める）。**退避形態そのものは `yagura.ingestion.spool.evacuated` の `reason` タグ（`q2_overflow` / `write_timeout` / `shutdown`）で判別できる**（Issue #271 / M-7。§4.1.1）——この可視化は形態分布の把握のためであり、上記の誤発火防止の判定基準（成功書き込みの有無）を置き換えるものではない。**保持期間削除の実行中に発生した退避も速度不足に数えない**（メンテナンス由来の自己飽和を昇格の根拠にしない。database.md §3）
- **昇格案内の発火条件**: 速度不足由来のスプール退避が一定期間内に一定回数を超えたとき、Web UI で SQL Server への本番昇格を案内する。閾値は SQLite の実用限界の実測（[M-7](#9-実測待ち一覧)）を踏まえて確定する
- 案内は利用者が抑制でき、組み込み DB のまま使い続ける判断を尊重する（ADR-0002 決定 3）。**案内の文言は内部用語（スプール・カウンタ等）ではなく利用者の言葉で、次に取る手順とともに示す**（ADR-0003 決定 5 の平易化原則。具体的な文言・画面は ui.md で定める）

### 5.4 アーキテクチャ別の検証水準 — ARM64 スモークの受け入れ基準

対応アーキテクチャは x64（対応 = Supported）と ARM64（試験的 = Experimental）の 2 種で、検証の裏付けが非対称である（[ADR-0009](../adr/0009-architecture-support.md) 決定 1・2）。x64 の検証水準は本章 §5.1〜5.3・§9 の管轄（実機 lab E2E・CI 回帰ベンチ）。本節は ARM64 の検証水準——リリースごとのスモーク——の受け入れ基準・合否判定・記録様式を定める（ADR-0009 委任事項 5 の受け皿）。

- **実行環境**: `windows-11-arm` GitHub ホストランナー（ARM64 実アーキ）。ビルド自体は x64 ランナー上のクロス publish で行ってよいが、以下のスモークは必ず実アーキ上で実行する
- **実行タイミング**: リリースワークフロー（`.github/workflows/release.yml` の `smoke-arm64` ジョブ）。スモークの不合格はリリース公開（`create-release`）をブロックする
- **スモーク項目と合否基準**（すべて必須。1 つでも不合格ならジョブ失敗）:
  1. **実 MSI インストール経由の起動・受信・閲覧**: ARM64 版 MSI をサイレントインストールし、サービス `Yagura` の Running 到達 → UDP 514 への syslog 送出 → 閲覧リスナ `/search` の HTML に RunId トークンが出現することを確認する（`installer/e2e/Invoke-YaguraInstallerE2E.ps1` の Full モード。x64 の installer-e2e と同一スクリプト・同一判定——検証項目をアーキ別に分岐させない）
  2. **観測性カウンタの出力**（§4.1）: `/status` に `yagura.ingestion.internal_buffer.dropped`・`yagura.ingestion.spool.evacuated`・`yagura.ingestion.persistence.failed` の計器名が出現すること（`yagura.ingestion.udp.receive_error` は §4.1.1 の設計判断で状態画面の対象外のため照合しない）
  3. **スプール発動 → drain 追いつきの 1 サイクル**: `Yagura.Bench SpoolActivationRecovery`（スプール容量を縮小して発動を誘発）を実アーキ上で実行し、(a) スプール退避カウンタ > 0（実際に発動したこと）、(b) drain 完了（スプール使用量が 0 に復帰）、(c) 突合成立（送信数 = 保存件数 + 全カウンタ。§5.1 の不変条件と同じ）の 3 点を満たすこと。バックプレッシャ経路は高負荷でしか発火せず、静的スモークだけでは「未検証のまま埋め込まれた欠陥」と「副ゆえ許容する劣化」を区別できないため、この動的サイクルを必須に含める（ADR-0009 決定 6 Phase 1 完了条件 ②）
- **数値はブロッキング判定に使わない**: ARM64 の持続流量・保存率などの数値は参考記録に留める（判定は上記の成立/不成立のみ）。x64 の CI 回帰ベンチ基準値（`tools/Yagura.Bench/baselines/ci-baseline.json`）は ARM64 へ流用しない——基準値は x64 ランナーの分布で較正されており、他アーキに同じ帯を持ち込む根拠がない（ADR-0009 決定 5・委任事項 3）。ARM64 の参考実測（SustainedZeroDrop 縮小版）は同ジョブで採取し artifact に残す
- **記録様式**: 各リリースの run URL と artifact（E2E の `*.summary.json`/`*.log.txt`・ベンチの `*.json`/`*.summary.txt`）を証拠とする。E2E サマリは機械可読（ADR-0006 基準 1 と同じ形式）であり、後から来る貢献者が「そのリリースの ARM64 スモークが何をどこまで通したか」を run 履歴から機械的に判別できる
- **不合格時の扱い**: ARM64 スモークの不合格はそのリリースの ARM64 配布物の公開を止めるが、x64 の公開可否とは独立に判断してよい（緊急パッチ時の x64 先行リリースは ADR-0009 決定 5 で明文化済み）

## 6. UI と受信の資源分離

単一プロセス同居の障害波及（UI の重い操作が受信と資源を奪い合う。ADR-0002 帰結）への防御:

- **絞り込みの強制はしない**: 時間範囲だけの検索・全件表示を拒否しない（Windows のイベントビューアと同じ操作感を基準にする）。防御は入力の制限ではなく出力の制限で行う
- **結果上限**: 1 回の検索が返す件数に上限を設ける（仮値: 10,000 件。[M-10](#9-実測待ち一覧)）。上限到達は「打ち切られた」ことを画面に明示する
- **一覧は軽量列のみ**: 一覧表示はメッセージ先頭の一定文字数までの射影とし、全文は詳細表示で個別取得する（転送量・メモリの両方を防御する。射影の文字数は仮値: 200 文字。[M-10](#9-実測待ち一覧)）
- **クエリタイムアウト**: 検索クエリに実行時間上限を設ける（仮値: 30 秒。[M-10](#9-実測待ち一覧)）。タイムアウトは「条件を絞る」案内とともに画面へ返す
- 検索の読み取りが受信の書き込みを妨げない分離（接続・ロックの設計）は provider ごとの性質に依存するため database.md で定める
- SignalR circuit 数の上限・到達時挙動はセキュリティ設計文書の管轄（ADR-0004 決定 6）

## 7. 永続化 provider 抽象（概要）

- 永続化層は初日から provider 抽象とし、v0.1 は SQLite（組み込み既定）と SQL Server（本番第一推奨）の 2 provider を必須とする（ADR-0002 決定 3）
- provider が満たすべき契約（インターフェース・適合テストスイート）、スキーマ、保持期間管理（自動削除）、昇格（接続切替・蓄積ログ移行）の設計は **database.md** で定める
- 本文書からの要求は次の 4 点: ①バッチ書き込みの提供（§2.1）、②書き込み失敗の区別可能な報告（一時障害か恒久障害か——リトライ・退避判断の入力。応答しないハングは永続化段側のタイムアウトが打ち切る）、③読み書き分離の性質の明示（§6）、④保持期間管理（自動削除）の提供（ログを失わない設計の裏面として「いつ・どう消すか」は初版から定義する。設計は database.md）
- カウンタ等のメタデータ領域（§4.3）は provider 管轄外（ホスト管轄のローカルファイル）であり、provider 契約には含まれない
