## 1. プロセス構成

### 1.1 単一サービスと内部境界

- **単一の Windows サービスプロセス**に受信・永続化・Web UI を同居させ、1 つのインストーラで配布する（ADR-0002 決定 1）
- 内部は次の 4 つのアセンブリ境界で分離する。ADR-0002 決定 1 の 3 境界（受信・永続化・UI）に、どこにも属さない結線・サービス統合の置き場として**ホスト**を加えた 4 分割である。境界は保守性のためのものであり、セキュリティ境界ではない

| モジュール | 責務 | 担当するパイプラインの段（§2.1） |
|---|---|---|
| ホスト | Windows サービス統合（実行アカウントは仮想サービスアカウント既定。ADR-0004 決定 4）、起動・停止順序の制御、設定の読込、メタデータ領域（§4.3）、各モジュールの結線 | —（段を持たない） |
| 受信 | ソケットの listen、syslog メッセージの解析、流量制御の挿入点 | 受信段・解析段 |
| 永続化 | provider 抽象（database.md）、書き込みバッチ、ディスクスプール | 永続化段・スプール |
| UI | Blazor Web UI（ADR-0003）、検索・ダッシュボード・セットアップウィザード | —（読み取り側） |

- モジュール間の参照は「ホスト → 各モジュール」「受信・UI → 永続化の抽象インターフェース」のみとし、受信と UI は互いを参照しない

### 1.2 起動順序 — 受信を最初に開く

サービス再起動は受信断であり、syslog は再送されない（ADR-0002 帰結）。再起動によるロスをプロセス停止時間そのものまで縮めるため、起動順序を次のとおり固定する:

1. スプール領域を開く（前回退避分の存在確認を含む）
2. **受信ソケットを開き、受信を開始する**（この時点から取りこぼしが止まる）
3. DB provider を初期化する。完了までの間は DB への書き込みが確定できないため、受信済みログは §3.2.1 の発動条件に該当し、通常のパイプライン経路でスプールへ退避される
4. Web UI を開始する

DB や UI の初期化失敗・遅延が受信開始を遅らせない。これが「ゼロ設定ファーストラン」（SQLite の初期化は軽い）と「本番構成」（SQL Server への接続確立を待たない）の両方で成立する。

**手順 1 が失敗した場合**（ディスク障害・ACL 破損等でスプールを開けない）: 受信は開始し、**スプールなしの縮退運転**として続行する。縮退はイベントログへの警告（§4.6）とメトリクスで強く可視化し、黙って opt-out 相当に落ちることを許さない。起動失敗（= 受信断の固定化）よりも、可視化された縮退の方が「ログを失わない」原則への害が小さい、という判断である。縮退中に Q2 が溢れた分・書き込みに失敗した分は退避先がないため破棄となり、永続化失敗カウンタに計上する。

### 1.3 停止順序 — メモリ上のログをスプールへ逃がす

停止要求を受けたら:

1. 受信ソケットを閉じ、その時点のカウンタをメタデータ領域へ書く（以降の到着分はプロセス停止と同じくロスであり、§4.4 の受信断として可視化される）
2. **メモリ上の未永続化ログを DB へ書き切るのを待たず、スプールへ追記して退避する**（退避分は次回起動時の drain で保存される）。退避中に発生した破棄は**メタデータ領域のカウンタへ逐次（一定件数ごと）反映する**——強制終了された場合でも退避中の破棄が計上されないまま消えないようにするため
3. カウンタを最終値で永続化し（§4.3）、正常停止イベントを記録して終了する

- 手順 2 の設計意図は「ディスクへの追記は DB への確定書き込みより短時間で終わる」ことにあるが、これは仮説であり、退避方式が書き切り方式より実際に速いことの検証を停止時間の実測（[M-9](#9-実測待ち一覧)）に含める
- **手順 2 が失敗する場合**（スプール上限到達・書込失敗）: 当該分は破棄し、スプール破棄またはスプール書込失敗カウンタに計上したうえで手順 3 に進む（手順 1 で一度カウンタを書いているため、退避中の破棄が計上されないまま落ちる窓を最終値の書き込み失敗時のみに限定できる）
- 停止処理全体がサービスコントロールマネージャの停止待機時間に収まらず強制終了された場合は、クラッシュと同じ扱いになる（§4.4 の近似断点・§4.1 の計上限界）。停止時間の目標値は実測で確定し（[M-9](#9-実測待ち一覧)）、停止待機時間の既定に収まることを受け入れ条件とする

## 2. 受信パイプライン

### 2.1 段の構成

受信から保存までを容量制限付きの非同期パイプラインとして構成する（ADR-0002 決定 2）:

```
送信元 ──UDP/TCP──> [OS ソケットバッファ]
                        │ 読み取り専念ループ
                        v
                    [受信段] ──(流量制御 挿入点)──> [解析段]
                        │ 有限キュー Q1                │ 有限キュー Q2
                        v                              v
                    （溢れ: UDP 破棄 /             [永続化段] ──バッチ書込──> DB provider
                      TCP 読み取り停止）               │ Q2 溢れ・書き込み失敗/タイムアウト
                                                       v
                                                  [ディスクスプール] ──drain──> 永続化段へ合流
```

- **受信段**: ソケットからの読み取りに専念する。解析や書き込みを同じループで行わない（読み取りの停滞は OS ソケットバッファ溢れ = 計測困難なロスに直結するため）
- **解析段**: RFC 3164 / RFC 5424 の解析。**解析に失敗したメッセージは破棄しない**——受信した生データのまま、解析失敗の印を付けて保存する（「ログを失わない」原則。不正形式の頻発はそれ自体が観測対象であり、カウンタにも計上する）
- **永続化段**: provider へのバッチ書き込み。**書き込みには時間上限（タイムアウト）を設け、応答のないハングを打ち切る**（§3.2.1）。バッチサイズ・書込間隔は実測で確定する（[M-1](#9-実測待ち一覧)）、タイムアウト値は [M-13](#9-実測待ち一覧)

### 2.2 保存の意味論

- **at-least-once**（ADR-0002 決定 2）。スプールからの復旧時等に重複保存が起こり得ることを仕様として許容し、重複排除は約束しない
- **保存順序は保証しない**。ライブ経路とスプール drain 経路が並行するため、DB への挿入順は受信順と一致しないことがある。閲覧・検索は受信時刻列で並べ替えて提供する（時刻の契約——UTC 保存・表示層で変換——は ui.md / database.md で定める）

## 3. 信頼性機構

### 3.1 バックプレッシャ — 境界ごとの溢れ挙動

各境界の溢れ時挙動をブロックか破棄か明示する（ADR-0002 決定 2。受信段では、OS 側の計測不能なロスに転化するブロックを選ばない）:

| 境界 | 容量 | 溢れ時の挙動 | 計上先カウンタ |
|---|---|---|---|
| OS ソケットバッファ | 受信バッファサイズ（設定項目。既定値は実機検証で確定 [M-2](#9-実測待ち一覧)） | OS が破棄（アプリからは直接見えない） | OS 統計との突合（§4.2） |
| 受信段 → 解析段（Q1・UDP 由来） | 有限（実測で確定 [M-1](#9-実測待ち一覧)） | **破棄**（ブロックすると読み取りが停滞し OS 側ロスに転化するため） | 内部バッファ破棄 |
| 受信段 → 解析段（Q1・TCP 由来） | 同上 | **読み取り停止**（接続からの読み取りを止め、TCP のフロー制御を通じて送信側に伝搬させる。送信側の保持・再送能力（§4.5）を活かすため破棄しない） | —（破棄しない。停止中であることはゲージ §4.6 で観測） |
| 解析段 → 永続化段（Q2） | 有限（実測で確定 [M-1](#9-実測待ち一覧)） | **新規投入分をスプールへ退避**（破棄でもブロックでもない第 3 の経路。Q2 内の滞留分は順次 DB へ進める。§3.2） | スプール退避（退避は損失ではないが観測する） |
| TCP 同時接続数 | 上限（設定項目。既定値は [M-14](#9-実測待ち一覧)） | **新規接続を拒否**（既存接続を守る。読み取り停止で接続が非有限に蓄積することを防ぐ有限化） | TCP 接続拒否 |
| スプール書き込み | —（機構の失敗） | リトライ後に**破棄** | スプール書込失敗 |
| スプール容量 | ディスク使用量上限（設定項目。既定値は [M-12](#9-実測待ち一覧)、置き場所の既定は [configuration.md](configuration.md) §2 で確定済み） | **新規破棄**（§3.2.3） | スプール破棄 |
| 流量制御 | —（挿入点。§3.3） | 破棄（実装後） | 流量制御破棄 |

- **Q1 溢れ（UDP）を破棄とする理由**: Q1 が溢れるのは解析段が追いつかない瞬間であり、このときスプール追記（チェックサム付きディスク I/O）が解析より速く受け止められるという保証はない。また解析前の生データを退避対象にするとスプールに第 2 のレコード形式を持ち込むことになる。単純さを優先して破棄 + 計上とする。**この判断は「Q1 溢れは持続せず、実質的にバースト時に限られる」という前提に依存する**——バースト負荷時の Q1 破棄の発生有無をベンチの計測対象に含め（[M-1](#9-実測待ち一覧)）、平常運用で恒常的に発生することが実測で判明した場合は、Q1 溢れ分の生データ退避を再評価する
- 破棄はすべて発生箇所別カウンタに計上する。「カウンタに計上されない喪失」は重大不具合である（ADR-0006 基準 2 の定義①。クラッシュ時のメモリ内喪失の限界は §4.4 に明記する）

### 3.2 ディスクスプール

#### 3.2.1 発動条件と構造

- 発動条件は「**永続化段が、受け取ったログを所定時間内に DB へ確定できないこと**」に統一する。具体的には次の 2 形態が同じ退避動作に合流する:
  - **容量**: Q2 が満杯で新規投入分を受け取れない（持続的な速度不足はこの形で現れる）
  - **時間**: バッチ書き込みが失敗した、またはタイムアウト（[M-13](#9-実測待ち一覧)）で打ち切られた——打ち切られたバッチはスプールへ退避し、宙吊りのまま失われる in-flight を作らない（タイムアウト後に DB 側で書き込みが成立していた場合の重複は at-least-once の範囲内）
- 保存先の死活を監視する専用のヘルスチェック経路は持たない。停止・ハング・速度不足のいずれも、上記 2 形態のどちらかとして現れる（低流量環境で Q2 が溢れないまま保存先が死んでいる状態は、タイムアウトが打ち切る）
- 既定で有効（opt-out。ADR-0002 決定 2）
- 構造要件: 追記専用のセグメントファイル群とし、レコード単位の破損検出（チェックサム）と**レコード種別の識別**（通常ログ / 自己検証用の合成レコード。§3.2.5）を持つ。セグメントは DB へのコミット確認後に削除する（クラッシュ時は未確認セグメントを再 drain する——重複は at-least-once の範囲内）
- ディスク使用量には上限があり、スプールがディスクを無制限に消費することはない。上限の既定値は [M-12](#9-実測待ち一覧) で確定する（置き場所の既定——データルート配下・設定で変更可——は [configuration.md](configuration.md) §2 で確定済み）
- スプール領域はログ本体と同等の保護対象とし、ACL は ADR-0004 決定 5 に従う

#### 3.2.2 drain — 復旧後の追いつきと振動防止

復旧直後に drain がライブ流入と資源を奪い合うと、二次飽和 → 再スプールの振動が起こり得る。これを次の 3 点で防ぐ:

- **ライブ優先**: drain は Q2 の使用率が低水位を下回っている間だけ進める。ライブ流入が増えて Q2 が埋まれば drain は自動的に停止する（ヒステリシス。低水位・再開水位は実測で確定 [M-3](#9-実測待ち一覧)）
- **速度上限**: drain の投入速度に上限を設ける（既定値は実測で確定 [M-3](#9-実測待ち一覧)）
- **保存先障害中は進めない**: 直近の書き込みが失敗・タイムアウトしている間は drain を停止し、バックオフ後に再試行する（§3.2.1 と同じシグナルの再利用であり、排した専用ヘルスチェックの新設ではない）。また **drain 由来のバッチが書き込みに失敗・タイムアウトした場合はスプールへ再追記しない**——元セグメントを未消化のまま残して drain を停止する（スプールへの退避はライブ経路由来のバッチに限る）。タイムアウトの恒常化時に「退避 → drain → 再タイムアウト → 再退避」で同一レコードがスプール内に複製され続け、実流入と無関係に上限到達（= 実ログの破棄）を早めるループを閉じるためである

この設計の帰結として、**持続的な速度不足の間は drain が進まず、スプールは上限へ向かって片方向に成長する**。これは隠れた欠陥ではなく「保存先の増強か流入の削減が必要」という状態そのものであり、進行を §4.6 のゲージ（使用量・最古滞留時間）と能動通知（上限接近の警告）で運用者に見せる。運用者の行動手順は operations.md に委ねる。

#### 3.2.3 上限到達時の挙動 — 新規破棄を採用

ディスク使用量が上限に達したら、**新規到着分を破棄する**（破棄はスプール破棄カウンタに計上）。

- **主たる採用理由は次の 2 点**: ①「退避に成功したログは（重複はあっても）失われない」という単純な約束を守れる——回転は退避済みデータの事後破棄であり、この予測可能性を失う。②満杯状態での回転（削除と追記の並行）という最も苦しい局面の複雑な I/O を持たず、実装と障害解析が単純になる
- **対価を明示する**: DB の長時間停止では、新規破棄は復旧時点に近い——つまり障害調査で最も参照したい——ログを失う側の選択である。この対価は認識のうえで受け入れる。上限到達は「上限の見積もりか保存先の復旧が破綻している」状態であり、どちらの破棄方式を選んでも損失は損失として発生する。本設計は残すログの新旧よりも、約束の単純さと機構の単純さを優先する
- 補助的な理由: 新規破棄では損失区間が「上限到達から空きができるまで」の 1 区間にまとまる。回転では drain・回転削除・追記が並行する時間帯に、保存済み・破棄済みが時間軸上で交錯し得る
- 上限到達の発生は能動通知（§4.6）で運用者へ知らせ、取るべき行動は operations.md に委ねる

#### 3.2.4 削除保証

- 消化（DB コミット確認）済みセグメントと破棄されたセグメントは、**通常のファイル削除で速やかに削除する**。消化済みデータを削除待ちのまま滞留させない
- **ストレージ上の残留痕への対抗（上書き消去等）は行わない**。この層の保護は ACL（ADR-0004 決定 5）とディスク全体の暗号化（利用環境側の統制。operations.md で案内）が担い、アプリによる確実消去は約束しない——この限界は利用者向けドキュメントに明記する（[§10](#10-利用者向けドキュメントへの申し送り)）

#### 3.2.5 平常時の経路検証

障害時専用経路は最悪の瞬間に最少テストのコードが走る（ADR-0002 却下案 (c) の正当な懸念）。代償措置として:

- **定期自己検証**: 一定周期（仮値: 1 日 1 回。[M-11](#9-実測待ち一覧)）で合成レコードをスプールへ書き込み、**drain の実機構に読ませて**照合する。合成レコードはレコード種別（§3.2.1 の構造要件）で識別され、**drain は種別を見て DB 書き込みの直前で合成レコードを破棄する**——この識別・破棄は定期検証時だけでなく drain の常時動作であり、クラッシュ後の再 drain でも合成レコードが通常ログとして DB へ混入する経路はない（証跡 DB への偽レコード混入の防止）。検証の失敗は能動通知（§4.6）の対象とする
- **検証の覆域を明示する**: 自己検証が覆うのは「スプール書込 → セグメント読出 → 逆直列化 → drain 合流判定」まで、ライブトラフィックが常時検証するのは共有された DB 書込コード（drain 専用の書込実装を持たない）である。**「drain が読んだレコードを実際に DB へコミットしてセグメントを削除する」結合部そのものは、どちらにも常時は覆われない**——この区間は受け入れテスト（スプール発動 → 復旧 → 件数突合）を CI に置いて担保する（§5.1 のベンチとは別の機能テスト）

### 3.3 流量制御（v0.1: 挿入点のみ）

- 挿入点は**受信段の直後・解析段の前**に置く（送信元の判別に解析は不要——送信元アドレスはソケット層で得られる。解析コストを浪費させないため解析より前）
- 送信元単位の制限とし、実装時の方式は token bucket を想定する（確定は実装設計時）
- v0.1 では判定・破棄は行わず、**挿入点のインターフェースと流量制御破棄カウンタの枠だけを設ける**（ADR-0002 決定 2 の到達水準どおり）
- 実装時の既定は opt-out（既定有効）を基本線とし、既定閾値は「通常運用で発火しない安全側」に実測で定める（[M-4](#9-実測待ち一覧)）。発火は必ず計測される

## 4. 観測性

### 4.1 発生箇所別ドロップカウンタ

計測は .NET 標準の計測 API（`System.Diagnostics.Metrics`）で実装し、出力先（ログ・Web UI・外部監視）は差し替え可能にする。あらゆるデータの外部送信は既定で無効（ADR-0002 決定 4）。

| カウンタ | 発生箇所 | 意味 |
|---|---|---|
| OS 受信破棄（突合値） | OS ソケットバッファ | アプリが読み取る前に OS が破棄した数（§4.2 の手段で観測） |
| 内部バッファ破棄 | Q1 溢れ（UDP 由来） | 受信したが解析段に渡せず破棄した数 |
| 流量制御破棄 | 流量制御 | 制限により破棄した数（v0.1 では常に 0） |
| スプール退避 | Q2 溢れ・書き込み失敗/タイムアウト | スプールへ退避した数（損失ではない。飽和の予兆シグナル） |
| スプール書込失敗 | スプール追記 | 退避を試みたが書き込めず破棄した数（ディスク障害等） |
| スプール破棄 | スプール上限 | 上限到達により破棄した数 |
| 永続化失敗 | DB 書き込み | リトライとスプール退避でも救えず失われた数（スプールなし縮退中の喪失を含む） |
| 解析失敗（保存済み） | 解析段 | 解析に失敗し生データのまま保存した数（損失ではない） |
| TCP 接続拒否 | 受信段 | 同時接続数上限により新規接続を拒否した数（拒否された送信元の保持・再送は送信側実装に依存 §4.5） |
| TCP 接続断 | 受信段 | TCP 接続の切断回数（§4.5。損失ではなく解釈の手がかり） |
| TCP 不完全メッセージ | 受信段 | 切断時に解析途中だった不完全メッセージの数（生データのまま保存し、印を付ける） |

原則: **損失は必ずどれかのカウンタに計上される**（クラッシュ時のメモリ内喪失の限界は §4.4）。カウンタ値は UI から 1 回のコピー操作で取得できるようにする（試用報告の様式が要求する。ADR-0006 基準 2）。

### 4.2 OS レベル取りこぼしの観測

アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る、という前提で設計する（ADR-0002 決定 2）:

- **第一候補**: OS が提供するシステム全体の UDP 統計（受信破棄数）をサービス稼働中に定期取得し、稼働区間内の増分を突合値として露出する。取得手段の確定は実機検証による（[M-8](#9-実測待ち一覧)）
- **粒度の限界を仕様として明記する**: システム全体統計は「本製品のソケットで落ちた」ことを直接は示さない（同居プロセスの UDP 破棄も含む）。意図した粒度（ソケット単位ないしポート単位）が実機検証で実現できない場合、システム全体統計への縮退を ADR-0002 の amendment として確定する（同 ADR の再評価トリガどおり）
- **受信バッファサイズを設定項目にする**: OS 側ロスの第一の緩和はソケット受信バッファの拡大である。既定値・指定可能な上限は実機検証で確定する（[M-2](#9-実測待ち一覧)）
- ベンチ（§5）の性能検証は、アプリ内カウンタだけでなく OS 統計との突合で行う（送った数 = 保存された数 + 全カウンタの合計、が成立することの確認）

### 4.3 メタデータ領域とカウンタの再起動をまたぐ永続化

- **メタデータ領域**は、カウンタ累積値・停止イベント・生存時刻を保存する**ホスト管轄の独立ローカルファイル**である。DB provider には依存しない——DB 障害中・provider 切替中にこそ記録が必要になるため、DB 内には置かない。したがって本番昇格（SQLite → SQL Server）の移行対象にも含まれない
- メタデータ領域は「ログが無い期間の説明責任」を担う証跡であり、**ログ本体と同等の保護対象として ACL は ADR-0004 決定 5 に従う**（スプールと同じ扱い）
- カウンタは累積値として**一定間隔でメタデータ領域に永続化し、正常停止時には必ず書き切る**（§1.3）。起動時に復元して継続する
- クラッシュ時は最終永続化以降の増分が失われ得る。損失窓を永続化間隔（仮値: 10 秒。[M-11](#9-実測待ち一覧)）以下に限定し、クラッシュ発生自体を §4.4 で可視化することで補う

### 4.4 受信断（再起動・クラッシュ）の可視化

再起動時間はロス時間である（ADR-0002 帰結）。計画再起動を含め、受信できなかった区間を記録として残す:

- **正常停止**: 停止イベント（時刻）を記録してから終了する。次回起動時に「停止〜起動」を受信断区間として保存する
- **クラッシュ**: 稼働中は一定間隔（仮値: 10 秒。カウンタ永続化と同一周期とするかは [M-11](#9-実測待ち一覧) で確定）で生存時刻をメタデータ領域に更新する。起動時に前回の正常停止イベントがなければ、最終生存時刻を近似の断点として受信断区間を保存する（近似であることを区間に印す）
- **受信断区間の保存**: 区間の確定は起動時に行い、保存は通常のパイプライン（スプールを含む耐障害経路）を通す——DB 障害中の起動でも記録が失われない
- **計上の限界を明記する**: クラッシュ（強制終了を含む）の瞬間に Q1・Q2 上およびスプール未書込だったログは、失われたうえ**件数をどのカウンタにも計上できない**（計上する主体ごと消えるため）。この喪失は件数ではなく、クラッシュ由来の受信断区間（近似断点の印付き）として**区間で観測可能にする**。「観測不能なログ喪失を作らない」（ADR-0006 基準 2 定義①）に対する本設計の回答は「件数計上が原理的に不可能な局面では、発生の事実と時間範囲を記録する」である
- **設定変更によるリスナ再構成の計画瞬断も、同じ区間記録の仕組みで記録・可視化する**（時刻・対象リスナ・契機となった設定変更。configuration.md §3）
- 受信断区間は検索・ダッシュボードで通常のログと同じ時間軸上に可視化する（「この時間帯はログが無いのではなく、受けられなかった」を見分けられるようにする。表示は ui.md で定める）

### 4.5 受信プロトコル別のロス特性

ロスの現れ方はプロトコルで異なる。カウンタの解釈を誤らせないため、特性を仕様として明記する:

- **UDP**: 到達しなかったこと自体をサーバ側は観測できない（ネットワーク上のロスは §4.1 のどのカウンタにも現れない）。サーバで観測できるのは OS バッファ以降のみ。「カウンタ全部 0 = ロス 0」ではなく「サーバ到達後のロス 0」である——この解釈は利用者向けドキュメントにも明記する（[§10](#10-利用者向けドキュメントへの申し送り)）
- **TCP**: 転送自体は信頼できるが、**接続断のときに送信側が未送信分をどう扱うか（保持・再送・破棄）は送信側実装に依存し、サーバからは制御も観測もできない**。サーバ側で観測・記録できる事象は「接続の確立・切断（相手・時刻）」「切断時の不完全メッセージ」であり、§4.1 のカウンタに計上する。サーバ側は Q1 満杯時に読み取りを止めて TCP のフロー制御に委ねる（§3.1）ことで、送信側の保持・再送能力を活かす設計とする。読み取り停止中も接続は保持されるため、**同時接続数に上限を設けて資源を有限化する**（設定項目。既定値は [M-14](#9-実測待ち一覧)。上限到達時は新規接続を拒否し、拒否は §4.1 のカウンタに計上する）。受信断（§4.4）中の TCP 送信元は接続拒否を検知できるため、送信側の再送機能が働き得る——これは UDP に対する利点として利用者向けドキュメントで案内する（[§10](#10-利用者向けドキュメントへの申し送り)）

### 4.6 ゲージと能動通知

カウンタ（累積値）は「起きたこと」の記録であり、「今どうなっているか」と「悪化しつつあること」は別の手段で見せる:

- **ゲージ**（現在値のメトリクス）: Q1・Q2 の使用率 / TCP 現在接続数（対上限比）・読み取り停止中の接続数 / スプール使用量（対上限比）/ スプール内最古レコードの滞留時間 / drain の未消化バックログ。これらは §3.2.2 の「振動していないこと」「速度不足が進行していること」を運用者が現在形で確認する手段である
- **能動通知**: UI を開いていない夜間にスプールが満ちていくのが最悪シナリオであり、pull 型の観測だけでは実効性がない。次の事象は **Windows イベントログへの警告書き出し（既定）**の対象とする: スプール退避の継続 / スプール上限接近・到達 / スプール書込失敗 / スプールなし縮退での起動（§1.2）/ 自己検証の失敗（§3.2.5）/ HTTPS 証明書の期限接近（configuration.md §6）。イベントログは Windows 管理者の既存監視（イベントログ転送等）のレールに乗る。通知の書き出し自体が失敗しても、ゲージとカウンタは独立した観測チャネルとして残る（通知失敗時の扱いの詳細は委任先で定める）。通知の詳細（イベント ID 採番はセキュリティ設計文書の監査系と統一する）と、通知を受けた運用者の行動手順は operations.md・セキュリティ設計文書に委ねる
- 監査記録（管理操作・拒否試行）の書き込み失敗の観測（ADR-0004 決定 7）は、本節の能動通知と同じ機構を使うが、要件の定義は**セキュリティ設計文書に委任する**

## 5. 性能とベンチマーク

### 5.1 ベンチハーネス（v0.1 の開発範囲）

- 構成: 負荷生成器（送信数を正確に把握できる自前実装）→ 本体 → 検証器（保存件数 + 全カウンタ + OS 統計の突合）
- 計測対象の最小セット: 受信スループット（UDP/TCP 別）、破棄ゼロで維持できる持続流量、**バースト負荷時の Q1 破棄の発生有無**（§3.1 の前提検証）、スプール発動 → 追いつきの所要、SQLite / SQL Server 各 provider の書き込み上限
- 容量・バッチサイズ等の設計値（[M-1](#9-実測待ち一覧)〜[M-3](#9-実測待ち一覧)）はこのハーネスの実測で確定し、本文書に記録する
- ベンチとは別に、スプールの結合部（drain → DB コミット → セグメント削除）を検証する機能テスト（発動 → 復旧 → 件数突合）を CI に置く（§3.2.5）

### 5.2 合否判定 — 回帰は基準比、公称は絶対値

判定方式は 2 層に分ける（ADR-0002 決定 4 の委任と ADR-0006 基準 1 の要求を、実行環境の揺らぎと両立させる）:

- **CI の回帰判定（毎 PR）は基準比とする**: リポジトリに記録した基準値との比を判定し、許容帯を超えて劣化したら不合格とする。CI 実行環境の揺らぎで絶対値は安定しないため、絶対値の合否は CI では行わない。許容帯は CI 環境の揺らぎを実測してから確定する（[M-5](#9-実測待ち一覧)）
- **基準値の更新はなし崩しを防ぐ手続きで行う**: 更新は PR として明示的に行い、**引き下げ方向の更新には理由の分類（実行環境の変更 / 機能追加による意図的なトレードオフ）を必須とする**。性能公称値（M-6）の確定後は、更新 PR に基準値と公称値の余裕率を明示し、余裕率が閾値を割る引き下げはリリース判定と同じ重さ（オーナー承認 + 公開の場での記録）で扱う——「環境変更への追従」名目の小さな引き下げの積み重ねでリリース時に初めて公称値割れが発覚する事態を防ぐ
- **公称値の判定（リリース時）は絶対値とする**: 定義済みの基準環境上で実測し、全体設計書に記録した性能公称値を満たすことを確認する。公称値と基準環境の定義は実測を経て確定する（[M-6](#9-実測待ち一覧)）。**確定後の公称値の引き下げは ADR-0006 の supersession に準ずる手続き**（PR + ペルソナレビュー + 変更理由の公開）を要する

### 5.3 SQLite の実用限界と昇格案内

ゼロ設定のまま大量ログを受けると SQLite は飽和し得る（ADR-0002 受け入れリスク）。「静かに落ちる」ことを許さないため:

- **飽和のシグナルはスプール退避カウンタ**とする。保存先が生きているのに退避が続く状態は「持続的な速度不足」そのものであり、専用の検知機構を持たずに §3.2.1 の機構から得られる
- **誤発火の防止**: 退避を「速度不足（発火に数える）」と「保存先の障害（数えない）」に分類する。判定は退避の形態（Q2 溢れかタイムアウトか）ではなく、**同一観測窓内に成功した書き込みが存在するか**で行う——成功が混在していれば保存先は生きており速度不足（飽和した SQLite の書き込みが Q2 溢れではなくタイムアウトとして現れても沈黙しない）、成功がなければ障害であり誤案内しない。SQLite の飽和が実際にどちらの形態で現れるかはベンチで確認する（[M-7](#9-実測待ち一覧)・[M-13](#9-実測待ち一覧) の決め方に含める）。**保持期間削除の実行中に発生した退避も速度不足に数えない**（メンテナンス由来の自己飽和を昇格の根拠にしない。database.md §3）
- **昇格案内の発火条件**: 速度不足由来のスプール退避が一定期間内に一定回数を超えたとき、Web UI で SQL Server への本番昇格を案内する。閾値は SQLite の実用限界の実測（[M-7](#9-実測待ち一覧)）を踏まえて確定する
- 案内は利用者が抑制でき、組み込み DB のまま使い続ける判断を尊重する（ADR-0002 決定 3）。**案内の文言は内部用語（スプール・カウンタ等）ではなく利用者の言葉で、次に取る手順とともに示す**（ADR-0003 決定 5 の平易化原則。具体的な文言・画面は ui.md で定める）

## 6. UI と受信の資源分離

単一プロセス同居の障害波及（UI の重い操作が受信と資源を奪い合う。ADR-0002 帰結）への防御:

- **絞り込みの強制はしない**: 時間範囲だけの検索・全件表示を拒否しない（Windows のイベントビューアと同じ操作感を基準にする）。防御は入力の制限ではなく出力の制限で行う
- **結果上限**: 1 回の検索が返す件数に上限を設ける（仮値: 10,000 件。[M-10](#9-実測待ち一覧)）。上限到達は「打ち切られた」ことを画面に明示する
- **一覧は軽量列のみ**: 一覧表示はメッセージ先頭の一定文字数までの射影とし、全文は詳細表示で個別取得する（転送量・メモリの両方を防御する。射影の文字数は仮値: 200 文字。[M-10](#9-実測待ち一覧)）
- **クエリタイムアウト**: 検索クエリに実行時間上限を設ける（仮値: 30 秒。[M-10](#9-実測待ち一覧)）。タイムアウトは「条件を絞る」案内とともに画面へ返す
- 検索の読み取りが受信の書き込みを妨げない分離（接続・ロックの設計）は provider ごとの性質に依存するため database.md で定める
- SignalR circuit 数の上限・到達時挙動はセキュリティ設計文書の管轄（ADR-0004 決定 6）

## 7. 永続化 provider 抽象（概要）

- 永続化層は初日から provider 抽象とし、v0.1 は SQLite（組み込み既定）と SQL Server（本番第一推奨）の 2 provider を必須とする（ADR-0002 決定 3）
- provider が満たすべき契約（インターフェース・適合テストスイート）、スキーマ、保持期間管理（自動削除）、昇格（接続切替・蓄積ログ移行）の設計は **database.md** で定める
- 本文書からの要求は次の 4 点: ①バッチ書き込みの提供（§2.1）、②書き込み失敗の区別可能な報告（一時障害か恒久障害か——リトライ・退避判断の入力。応答しないハングは永続化段側のタイムアウトが打ち切る）、③読み書き分離の性質の明示（§6）、④保持期間管理（自動削除）の提供（ログを失わない設計の裏面として「いつ・どう消すか」は初版から定義する。設計は database.md）
- カウンタ等のメタデータ領域（§4.3）は provider 管轄外（ホスト管轄のローカルファイル）であり、provider 契約には含まれない

## 8. セキュリティ挿入点との対応

opt-in 強化が「後付けの改造」ではなく「挿入点への装着」になる構造（ADR-0002 決定 6）と本文書の対応:

| 挿入点（ADR-0002 §6 / ADR-0004） | 本文書での位置 |
|---|---|
| 通信の保護（TLS 受信） | 受信段のリスナ構成（§2.1）。平文リスナと共存可能な構造とする |
| 利用者の認証・管理操作の認可 | UI モジュール境界（§1.1）。管理系エンドポイントの loopback 束縛は listener 構成の責務 |
| 保存データの保護 — スプール・組み込み DB・設定ファイル・メタデータ領域の ACL | §3.2.4・§4.3（ADR-0004 決定 5） |
| 保存データの保護 — DB 接続資格情報（DPAPI） | 永続化モジュールの接続管理。設定スキーマとの関係は configuration.md（ADR-0004 決定 5） |
| 実行権限の最小化（仮想サービスアカウント既定） | ホストモジュールのサービス統合（§1.1。ADR-0004 決定 4） |
| 管理操作の監査記録（管理操作全般 + 拒否された試行） | 管理操作の経路はホスト・UI モジュール。**拒否試行（非 localhost からの管理要求）の検知点は loopback リスナ構成側**にある。記録失敗の観測は §4.6 の能動通知機構を使う（要件はセキュリティ設計文書。ADR-0004 決定 7） |

loopback 束縛の CI 回帰テスト（v0.1 受け入れ条件。ADR-0004 決定 2）はセキュリティ設計文書で仕様化する。
