## 1. プロセス構成

### 1.1 単一サービスと内部境界

- **単一の Windows サービスプロセス**に受信・永続化・Web UI を同居させ、1 つのインストーラで配布する（ADR-0002 決定 1）
- 内部は次の 4 つのアセンブリ境界で分離する。境界は保守性のためのものであり、セキュリティ境界ではない

| モジュール | 責務 |
|---|---|
| ホスト | Windows サービス統合、起動・停止順序の制御、設定の読込、各モジュールの結線 |
| 受信 | ソケットの listen、syslog メッセージの解析、流量制御の挿入点 |
| 永続化 | provider 抽象（database.md）、書き込みバッチ、ディスクスプール |
| UI | Blazor Web UI（ADR-0003）、検索・ダッシュボード・セットアップウィザード |

- モジュール間の参照は「ホスト → 各モジュール」「受信・UI → 永続化の抽象インターフェース」のみとし、受信と UI は互いを参照しない

### 1.2 起動順序 — 受信を最初に開く

サービス再起動は受信断であり、syslog は再送されない（ADR-0002 帰結）。再起動によるロスをプロセス停止時間そのものまで縮めるため、起動順序を次のとおり固定する:

1. スプール領域を開く（前回退避分の存在確認を含む）
2. **受信ソケットを開き、受信を開始する**（この時点から取りこぼしが止まる）
3. DB provider を初期化する。完了までの間、受信済みログは通常のパイプライン経路でスプールに退避される（§3.2 の発動条件「保存先に書き込めない」に該当）
4. Web UI を開始する

DB や UI の初期化失敗・遅延が受信開始を遅らせない。これが「ゼロ設定ファーストラン」（SQLite の初期化は軽い）と「本番構成」（SQL Server への接続確立を待たない）の両方で成立する。

### 1.3 停止順序 — メモリ上のログをスプールへ逃がす

停止要求を受けたら:

1. 受信ソケットを閉じる（以降の到着分はプロセス停止と同じくロスであり、§4.4 の受信断として可視化される）
2. **メモリ上の未永続化ログを DB へ書き切るのを待たず、スプールへ追記して退避する**（ディスク追記は DB 書き込みより速く、停止時間を短縮する。退避分は次回起動時の drain で保存される）
3. カウンタを永続化し（§4.3）、正常停止イベントを記録して終了する

停止時間の目標値は実測で確定する（[M-9](#9-実測待ち一覧)）。Windows サービスの停止待機時間の既定（サービスコントロールマネージャ側の制約）に収まることを受け入れ条件とする。

## 2. 受信パイプライン

### 2.1 段の構成

受信から保存までを容量制限付きの非同期パイプラインとして構成する（ADR-0002 決定 2）:

```
送信元 ──UDP/TCP──> [OS ソケットバッファ]
                        │ 読み取り専念ループ
                        v
                    [受信段] ──(流量制御 挿入点)──> [解析段]
                        │ 有限キュー Q1                │ 有限キュー Q2
                        v                              v
                    （溢れ: 破棄+計上）            [永続化段] ──バッチ書込──> DB provider
                                                       │ 溢れ・障害時
                                                       v
                                                  [ディスクスプール] ──drain──> 永続化段へ合流
```

- **受信段**: ソケットからの読み取りに専念する。解析や書き込みを同じループで行わない（読み取りの停滞は OS ソケットバッファ溢れ = 計測困難なロスに直結するため）
- **解析段**: RFC 3164 / RFC 5424 の解析。**解析に失敗したメッセージは破棄しない**——受信した生データのまま、解析失敗の印を付けて保存する（「ログを失わない」原則。不正形式の頻発はそれ自体が観測対象であり、カウンタにも計上する）
- **永続化段**: provider へのバッチ書き込み。バッチサイズ・書込間隔は実測で確定する（[M-1](#9-実測待ち一覧)）

### 2.2 保存の意味論

- **at-least-once**（ADR-0002 決定 2）。スプールからの復旧時等に重複保存が起こり得ることを仕様として許容し、重複排除は約束しない
- **保存順序は保証しない**。ライブ経路とスプール drain 経路が並行するため、DB への挿入順は受信順と一致しないことがある。閲覧・検索は受信時刻列で並べ替えて提供する（時刻の契約——UTC 保存・表示層で変換——は ui.md / database.md で定める）

## 3. 信頼性機構

### 3.1 バックプレッシャ — 境界ごとの溢れ挙動

各境界の溢れ時挙動をブロックか破棄か明示する（ADR-0002 決定 2。受信段では、OS 側の計測不能なロスに転化する「ブロック」を選ばない）:

| 境界 | 容量 | 溢れ時の挙動 | 計上先カウンタ |
|---|---|---|---|
| OS ソケットバッファ | 受信バッファサイズ（設定項目。既定値は実機検証で確定 [M-2](#9-実測待ち一覧)） | OS が破棄（アプリからは直接見えない） | OS 統計との突合（§4.2） |
| 受信段 → 解析段（Q1） | 有限（実測で確定 [M-1](#9-実測待ち一覧)） | **破棄**（ブロックすると読み取りが停滞し OS 側ロスに転化するため） | 内部バッファ破棄 |
| 解析段 → 永続化段(Q2) | 有限（実測で確定 [M-1](#9-実測待ち一覧)） | **スプールへ退避**（破棄でもブロックでもない第 3 の経路。§3.2） | スプール退避（退避は損失ではないが観測する） |
| スプール | ディスク使用量上限（設定項目） | **新規破棄**（§3.2.3） | スプール破棄 |
| 流量制御 | —（挿入点。§3.3） | 破棄（実装後） | 流量制御破棄 |

破棄はすべて発生箇所別カウンタに計上する。「カウンタに計上されない喪失」は重大不具合である（ADR-0006 基準 2 の定義①）。

### 3.2 ディスクスプール

#### 3.2.1 発動条件と構造

- 発動条件は**永続化段のキュー（Q2）に収まらないこと**、これ一つに単純化する。保存先の停止（書き込みが返らない・失敗する）でも、持続的な速度不足（生きているが追いつかない）でも、同じ「Q2 が溢れる」という形で現れるため、障害検知のための別経路を持たない（ADR-0002 決定 2 の 2 条件を単一機構で充足する）
- 既定で有効（opt-out。ADR-0002 決定 2）
- 構造要件: 追記専用のセグメントファイル群とし、レコード単位の破損検出（チェックサム）を持つ。セグメントは DB へのコミット確認後に削除する（クラッシュ時は未確認セグメントを再 drain する——重複は at-least-once の範囲内）
- スプール領域はログ本体と同等の保護対象とし、ACL は ADR-0004 決定 5 に従う

#### 3.2.2 drain — 復旧後の追いつきと振動防止

復旧直後に drain がライブ流入と資源を奪い合うと、二次飽和 → 再スプールの振動が起こり得る。これを次の 2 点で防ぐ:

- **ライブ優先**: drain は Q2 の使用率が低水位を下回っている間だけ進める。ライブ流入が増えて Q2 が埋まれば drain は自動的に停止する（ヒステリシス。低水位・再開水位は実測で確定 [M-3](#9-実測待ち一覧)）
- **速度上限**: drain の投入速度に上限を設ける（既定値は実測で確定 [M-3](#9-実測待ち一覧)）

#### 3.2.3 上限到達時の挙動 — 新規破棄を採用

ディスク使用量が上限に達したら、**新規到着分を破棄する**（破棄はスプール破棄カウンタに計上）。

- 採用理由: ①スプール済みデータは「退避に成功したら保存される」という単純な約束を守れる（回転はこの約束を破る）、②損失区間が「上限到達時刻から空きができるまで」という連続した 1 区間になり、運用者への説明が単純、③満杯状態での回転（削除 + 追記の並行）という最も苦しい局面の複雑な I/O を持たない
- 却下した対案 — 古い順の回転: 復旧時点に近いログが残る利点はあるが、退避済みデータの事後破棄により「スプールに入った分は安全」という予測可能性を失い、損失区間も断片化する。上限到達が起きる状況はそもそも「上限の見積もりか保存先の復旧が破綻している」状況であり、どちらを残しても損失は損失として観測・対処させる方が一貫する
- いずれの挙動でも破棄は必ず計上する、という ADR-0002 の要求は新規破棄側で満たす

#### 3.2.4 削除保証

- 消化（DB コミット確認）済みセグメントと破棄されたセグメントは、**通常のファイル削除で速やかに削除する**。消化済みデータを削除待ちのまま滞留させない
- **ストレージ上の残留痕への対抗（上書き消去等）は行わない**。この層の保護は ACL（ADR-0004 決定 5）とディスク全体の暗号化（利用環境側の統制。operations.md で案内）が担い、アプリによる確実消去は約束しない——この限界は利用者向けドキュメントに明記する

#### 3.2.5 平常時の経路検証

障害時専用経路は最悪の瞬間に最少テストのコードが走る（ADR-0002 却下案 (c) の正当な懸念）。代償措置として:

- **定期自己検証**: 一定周期で合成レコードをスプールの書込 → 読出 → 照合の全経路に通し、結果をメトリクスとして露出する（周期は仮値: 1 日 1 回。[M-11](#9-実測待ち一覧)）。合成レコードは DB へは流さず、検証後に削除する
- **drain の DB 書き込みはライブ経路と同一コード**を使う（drain 専用の書込実装を持たない）。自己検証が覆わない「スプール → DB」の区間は、この共有によって平常時のライブトラフィックで常時検証されている状態を作る

### 3.3 流量制御（v0.1: 挿入点のみ）

- 挿入点は**受信段の直後・解析段の前**に置く（送信元の判別に解析は不要——送信元アドレスはソケット層で得られる。解析コストを浪費させないため解析より前）
- 送信元単位の制限とし、実装時の方式は token bucket を想定する（確定は実装設計時）
- v0.1 では判定・破棄は行わず、**挿入点のインターフェースと流量制御破棄カウンタの枠だけを設ける**（ADR-0002 決定 2 の到達水準どおり）
- 実装時の既定は opt-out（既定有効）を基本線とし、既定閾値は「通常運用で発火しない安全側」に実測で定める（[M-4](#9-実測待ち一覧)）。発火は必ず計測される

## 4. 観測性

### 4.1 発生箇所別ドロップカウンタ

計測は .NET 標準の計測 API（`System.Diagnostics.Metrics`）で実装し、出力先（ログ・Web UI・外部監視）は差し替え可能にする。あらゆるデータの外部送信は既定で無効（ADR-0002 決定 4）。

| カウンタ | 発生箇所 | 意味 |
|---|---|---|
| OS 受信破棄（突合値） | OS ソケットバッファ | アプリが読み取る前に OS が破棄した数（§4.2 の手段で観測） |
| 内部バッファ破棄 | Q1 溢れ | 受信したが解析段に渡せず破棄した数 |
| 流量制御破棄 | 流量制御 | 制限により破棄した数（v0.1 では常に 0） |
| スプール退避 | Q2 溢れ・保存先障害 | スプールへ退避した数（損失ではない。飽和の予兆シグナル） |
| スプール破棄 | スプール上限 | 上限到達により破棄した数 |
| 永続化失敗 | DB 書き込み | リトライとスプール退避でも救えず失われた数 |
| 解析失敗（保存済み） | 解析段 | 解析に失敗し生データのまま保存した数（損失ではない） |

原則: **損失は必ずどれかのカウンタに計上される**。カウンタ値は UI から 1 回のコピー操作で取得できるようにする（試用報告の様式が要求する。ADR-0006 基準 2）。

### 4.2 OS レベル取りこぼしの観測

アプリ内カウンタが 0 でも OS 側で落ちていることはあり得る、という前提で設計する（ADR-0002 決定 2）:

- **第一候補**: OS が提供するシステム全体の UDP 統計（受信破棄数）をサービス稼働中に定期取得し、稼働区間内の増分を突合値として露出する。取得手段の確定は実機検証による（[M-8](#9-実測待ち一覧)）
- **粒度の限界を仕様として明記する**: システム全体統計は「本製品のソケットで落ちた」ことを直接は示さない（同居プロセスの UDP 破棄も含む）。意図した粒度（ソケット単位ないしポート単位）が実機検証で実現できない場合、システム全体統計への縮退を ADR-0002 の amendment として確定する（同 ADR の再評価トリガどおり）
- **受信バッファサイズを設定項目にする**: OS 側ロスの第一の緩和はソケット受信バッファの拡大である。既定値・指定可能な上限は実機検証で確定する（[M-2](#9-実測待ち一覧)）
- ベンチ（§5）の性能検証は、アプリ内カウンタだけでなく OS 統計との突合で行う（送った数 = 保存された数 + 全カウンタの合計、が成立することの確認）

### 4.3 カウンタの再起動をまたぐ永続化

「破棄は必ず計上」を再起動越しに成立させる:

- カウンタは累積値として**一定間隔で組み込みのメタデータ領域に永続化し、正常停止時には必ず書き切る**。起動時に復元して継続する
- クラッシュ時は最終永続化以降の増分が失われ得る。損失窓を永続化間隔（仮値: 10 秒。[M-11](#9-実測待ち一覧)）以下に限定し、クラッシュ発生自体を §4.4 で可視化することで補う

### 4.4 受信断（再起動・クラッシュ）の可視化

再起動時間はロス時間である（ADR-0002 帰結）。計画再起動を含め、受信できなかった区間を記録として残す:

- **正常停止**: 停止イベント（時刻）を記録してから終了する。次回起動時に「停止〜起動」を受信断区間として保存する
- **クラッシュ**: 稼働中は一定間隔で生存時刻を更新する。起動時に前回の正常停止イベントがなければ、最終生存時刻を近似の断点として受信断区間を保存する（近似であることを区間に印す）
- 受信断区間は検索・ダッシュボードで通常のログと同じ時間軸上に可視化する（「この時間帯はログが無いのではなく、受けられなかった」を見分けられるようにする。表示は ui.md で定める）

### 4.5 受信プロトコル別のロス特性

ロスの現れ方はプロトコルで異なる。カウンタの解釈を誤らせないため、特性を仕様として明記する:

- **UDP**: 到達しなかったこと自体をサーバ側は観測できない（ネットワーク上のロスは §4.1 のどのカウンタにも現れない）。サーバで観測できるのは OS バッファ以降のみ。「カウンタ全部 0 = ロス 0」ではなく「サーバ到達後のロス 0」である——この解釈は利用者向けドキュメントにも明記する
- **TCP**: 転送自体は信頼できるが、**接続断のときに送信側が未送信分をどう扱うか（保持・再送・破棄）は送信側実装に依存し、サーバからは制御も観測もできない**。サーバ側で観測・記録できる事象は「接続の確立・切断（相手・時刻）」「切断時に解析途中だった不完全メッセージ」であり、これらを計測する。受信断（§4.4）中の TCP 送信元は接続拒否を検知できるため、送信側の再送機能が働き得る——これは UDP に対する利点として利用者向けドキュメントで案内する

## 5. 性能とベンチマーク

### 5.1 ベンチハーネス（v0.1 の開発範囲）

- 構成: 負荷生成器（送信数を正確に把握できる自前実装）→ 本体 → 検証器（保存件数 + 全カウンタ + OS 統計の突合）
- 計測対象の最小セット: 受信スループット（UDP/TCP 別）、破棄ゼロで維持できる持続流量、スプール発動 → 追いつきの所要、SQLite / SQL Server 各 provider の書き込み上限
- 容量・バッチサイズ等の設計値（[M-1](#9-実測待ち一覧)〜[M-3](#9-実測待ち一覧)）はこのハーネスの実測で確定し、本文書に記録する

### 5.2 合否判定 — 回帰は基準比、公称は絶対値

判定方式は 2 層に分ける（ADR-0002 決定 4 の委任と ADR-0006 基準 1 の要求を、実行環境の揺らぎと両立させる）:

- **CI の回帰判定（毎 PR）は基準比とする**: リポジトリに記録した基準値との比を判定し、許容帯を超えて劣化したら不合格とする。CI 実行環境の揺らぎで絶対値は安定しないため、絶対値の合否は CI では行わない。許容帯は CI 環境の揺らぎを実測してから確定する（[M-5](#9-実測待ち一覧)）。基準値の更新（改善の取り込み・環境変更への追従）は PR として明示的に行い、黙って動かさない
- **公称値の判定（リリース時）は絶対値とする**: 定義済みの基準環境上で実測し、全体設計書に記録した性能公称値を満たすことを確認する。公称値と基準環境の定義は実測を経て確定する（[M-6](#9-実測待ち一覧)）。**確定後の公称値の引き下げは ADR-0006 の supersession に準ずる手続き**（PR + ペルソナレビュー + 変更理由の公開）を要する

### 5.3 SQLite の実用限界と昇格案内

ゼロ設定のまま大量ログを受けると SQLite は飽和し得る（ADR-0002 受け入れリスク）。「静かに落ちる」ことを許さないため:

- **飽和のシグナルはスプール退避カウンタ**とする。保存先が生きているのに退避が続く状態は「持続的な速度不足」そのものであり、専用の検知機構を持たずに §3.2.1 の単一機構から得られる
- **昇格案内の発火条件**: 速度不足由来のスプール退避が一定期間内に一定回数を超えたとき、Web UI で SQL Server への本番昇格を案内する。閾値は SQLite の実用限界の実測（[M-7](#9-実測待ち一覧)）を踏まえて確定する
- 案内は利用者が抑制でき、組み込み DB のまま使い続ける判断を尊重する（ADR-0002 決定 3）

## 6. UI と受信の資源分離

単一プロセス同居の障害波及（UI の重い操作が受信と資源を奪い合う。ADR-0002 帰結）への防御:

- **絞り込みの強制はしない**: 時間範囲だけの検索・全件表示を拒否しない（Windows のイベントビューアと同じ操作感を基準にする）。防御は入力の制限ではなく出力の制限で行う
- **結果上限**: 1 回の検索が返す件数に上限を設ける（仮値: 10,000 件。[M-10](#9-実測待ち一覧)）。上限到達は「打ち切られた」ことを画面に明示する
- **一覧は軽量列のみ**: 一覧表示はメッセージ先頭の一定文字数までの射影とし、全文は詳細表示で個別取得する（転送量・メモリの両方を防御する。射影の文字数は仮値: 200 文字。[M-10](#9-実測待ち一覧)）
- **クエリタイムアウト**: 検索クエリに実行時間上限を設ける（仮値: 30 秒。[M-10](#9-実測待ち一覧)）。タイムアウトは「条件を絞る」案内とともに画面へ返す
- 検索の読み取りが受信の書き込みを妨げない分離（接続・ロックの設計）は provider ごとの性質に依存するため database.md で定める
- SignalR circuit 数の上限・到達時挙動はセキュリティ設計文書の管轄（ADR-0004 決定 6）

## 7. 永続化 provider 抽象（概要）

- 永続化層は初日から provider 抽象とし、v0.1 は SQLite（組み込み既定）と SQL Server（本番第一推奨）の 2 provider を必須とする（ADR-0002 決定 3）
- provider が満たすべき契約（インターフェース・適合テストスイート）、スキーマ、昇格（接続切替・蓄積ログ移行）の設計は **database.md** で定める
- 本文書からの要求は次の 3 点: ①バッチ書き込みの提供（§2.1）、②書き込み失敗の区別可能な報告（一時障害か恒久障害か——スプールの発動・リトライ判断の入力）、③読み書き分離の性質の明示（§6）

## 8. セキュリティ挿入点との対応

opt-in 強化が「挿入点への装着」になる構造（ADR-0002 決定 6）と本文書の対応:

| 挿入点（ADR-0004） | 本文書での位置 |
|---|---|
| 通信の保護（TLS 受信） | 受信段のリスナ構成（§2.1）。平文リスナと共存可能な構造とする |
| 利用者の認証・管理操作の認可 | UI モジュール境界（§1.1）。管理系エンドポイントの loopback 束縛は listener 構成の責務 |
| 保存データの保護 | スプール領域（§3.2.4）・組み込み DB・設定ファイルの ACL（ADR-0004 決定 5） |
| 設定変更の監査記録 | ホストモジュールの管理操作経路。詳細はセキュリティ設計文書 |

loopback 束縛の CI 回帰テスト（v0.1 受け入れ条件。ADR-0004 決定 2）はセキュリティ設計文書で仕様化する。
