## 1. provider 抽象

### 1.1 構成

- 永続化層は**初日から provider 抽象**とし、SQL 方言・機能差は provider 層に封じ込める（ADR-0002 決定 3）。上位（受信・UI・ホスト）は抽象インターフェースのみを参照し、特定 DB の SQL・接続 API を直接書かない
- v0.1 の必須 provider は **SQLite**（組み込み既定。§4）と **SQL Server**（本番第一推奨。§5）の 2 つ。PostgreSQL / MySQL は追加を意図する（約束ではない。ADR-0002 決定 3）
- 抽象の縫い目は 2 実装で検証するが、2 実装での検証が第 3 の provider への適合を保証するわけではない——十分性は適合テストスイート（§1.3）の設計で担保する（ADR-0002 の同趣旨を本文書が引き受ける）

### 1.2 provider 契約

provider が実装すべき操作と性質。インターフェースの C# シグネチャは実装設計で確定するが、**契約の意味論はここで固定する**:

| 契約 | 内容 | architecture.md §7 対応 | 主な利用者 |
|---|---|---|---|
| スキーマ管理 | スキーマの新規作成と版間移行（冪等——既に適用済みなら何もしない）。実行に必要な権限が不足する場合、**不足内容と、管理者がそのまま実行できる SQL を区別可能に報告する**（§5.2。**提示 SQL は秘密情報を含まない**） | —（本文書独自） | ホスト（起動時・昇格時） |
| バッチ挿入 | ログレコードのバッチ書き込み（architecture.md §2.1）。部分成功の扱い（全再試行で重複になっても at-least-once の範囲内）を含む | ① | 永続化段 |
| 失敗の分類報告 | 書き込み失敗を**一時障害（再試行に意味がある）・恒久障害（設定・スキーマ・権限の問題）・容量枯渇（保存領域の上限。削除で回復し得る）の 3 区分で報告する**。応答しないハングは呼び出し側のタイムアウトが打ち切る（architecture.md §3.2.1） | ② | 永続化段・ホスト |
| 対話的検索 | 条件（時間範囲・送信元・重大度・自由文等）+ **射影（一覧用の軽量列）+ 結果上限 + タイムアウト**を必須引数とする読み取り（architecture.md §6 の防御を契約レベルで強制する。**防御対象は UI の対話的検索であり、下記「契約拡張の予約」の管理経路には適用しない**） | —（本文書独自。architecture.md §6 の契約化） | UI |
| 保持期間の削除 | 基準時刻より古いレコードの削除。**受信の書き込みを長時間妨げない方式**（分割実行等）であること（§3） | ④ | ホスト（定期実行） |
| 統計 | 保存件数・DB サイズ（またはサイズ取得不能の明示。**SQL Server provider には適用しない**——§5.3 の必須要件）。Express の上限接近警告（§5.3）と昇格案内の入力 | —（本文書独自） | ホスト・UI |
| 読み書き分離の性質 | 「検索が書き込みをブロックし得るか」と付随する運用特性（§4 の WAL 肥大等）を provider ごとに明示する。**これは文書化義務であり、機械検証の対象ではない**（§1.3） | ③ | 設計・レビュー |

- **契約拡張の予約**: 次の 2 操作は現契約に含めないが、必要になる時点が既に見えているため、**§7 の互換性規則に従う optional 操作（capability 検出）として追加する**ことを予約する: (a) **一括読み出し**——移行（§6.2）・エクスポート用のストリーミング読み出し。上限・タイムアウトの必須化は適用せず、呼び出し経路を管理操作に限定する。(b) **集計**——ダッシュボード用の件数集計等（要否と形は ui.md 起筆時に確定）。対話的検索の防御を契約全体の禁止事項に一般化しない
- 契約に**含めない**もの: カウンタ等のメタデータ領域（ホスト管轄のローカルファイル。architecture.md §4.3）、スプール（永続化モジュール内だが provider の外側）

### 1.3 適合テストスイート — 外部貢献の入口

- **すべての provider が合格すべき単一の適合テストスイート**を v0.1 の開発範囲に含める（SQLite・SQL Server 自身もこのスイートで検証する——「2 実装があって初めて抽象が検証される」の実行形）
- 検証対象を 2 種に区別する: **機械検証する契約**（§1.2 の操作——スキーマ管理・バッチ挿入・失敗の分類・対話的検索・保持期間削除・統計——と境界ケース: 最大長メッセージ・不正 UTF-8・タイムスタンプ境界・空バッチ・保持期間削除と挿入の並行）と、**文書化義務**（読み書き分離の性質——テストではなくレビューで検証する）
- 対話的検索の一致規則（自由文の対象列・大文字小文字・部分一致の意味論）は、適合テストの整備と同時に §1.2 へ固定する（[DB-6](#8-確定待ち一覧)）。**特定 provider の挙動を暗黙の基準にしない**——規則を先に固め、全 provider を同じ規則で検証する
- **新 provider の追加提案は「適合テストスイートが通ること」を受け入れの技術条件とする**。提案は本文書に provider 節（§4・§5 と同格。読み書き分離の性質の明示を含む）を追加する PR として行う——性質明示の置き場は本文書である。提案手順（Issue → 設計対話 → PR）は CONTRIBUTING の管轄。実証完了前（SQLite + SQL Server が本番品質で動く前）に外部から provider 追加 PR が届いた場合、適合テストの整備を優先課題に繰り上げる（ADR-0002 再評価トリガの受け皿）
- 適合テストの CI 実行構成と、追加 DB の実行環境（コンテナ等）の提供責任の所在（貢献者側か受け入れ側か）は実装設計で確定する（[DB-3](#8-確定待ち一覧)）

## 2. 論理スキーマ

物理設計（DDL・索引・データ型の方言）は各 provider に委ねる。ここでは**全 provider が同じ意味で保存すべき論理形**を固定する。

### 2.1 ログレコード

| 項目 | 内容 | 備考 |
|---|---|---|
| Id | レコード識別子 | provider 内で一意。採番方式は provider 実装に委ねる |
| ReceivedAt | **サーバが受信した時刻（UTC）** | 検索・並び・保持期間の基準軸。刻印点は §2.2 |
| SourceAddress / SourcePort | 送信元 | ソケット層で取得 |
| Protocol | UDP / TCP（将来: TLS） | 受信リスナ由来 |
| DeviceTimestamp | 送信元が名乗った時刻（あれば） | **参考情報。基準軸にしない**（§2.2） |
| Facility / Severity | syslog PRI の分解値 | 解析失敗時は未設定 |
| Hostname / AppName / ProcId / MsgId | RFC 5424 ヘッダ（RFC 3164 は対応部分をマップ） | 解析できた範囲で設定 |
| StructuredData | RFC 5424 構造化データ | 原文のまま保存（要素分解はしない——加工しない方針。ADR-0001 スコープ外の「加工パイプライン」に踏み込まない） |
| Message | メッセージ本文 | 一覧は先頭 N 文字の射影で返す（architecture.md §6） |
| Raw | **受信したバイト列そのもの**（不正 UTF-8 を含み得るため、文字列変換を経ない） | **ParseStatus が解析失敗・不完全のときのみ保存する**（このとき Raw が一次保存物）。解析済みレコードには保存しない——却下案「常時保存」はストレージをほぼ倍化させる一方、StructuredData・Message を原文のまま保持する本スキーマでは追加で守れる情報がヘッダの整形差程度に留まるため採らない |
| ParseStatus | 解析済み / 解析失敗 / 不完全 | **排他の 3 値。不完全（TCP 切断による途中終端）は解析失敗に優先する**。解析失敗もログとして検索・閲覧できる |

### 2.2 時刻の契約

- **保存はすべて UTC**。表示層（UI・通知）でローカル時刻へ変換する。この契約は ui.md の画面設計より先に固定し、画面はこれに従う
- **基準軸は ReceivedAt（サーバ受信時刻）**とする。送信元の時計は信頼できない（進み・遅れ・未設定が現実に起こる）ため、DeviceTimestamp は保存するが、検索の並び・保持期間・受信断区間との突合には使わない。両者のずれが大きいことはそれ自体が運用上の発見であり、UI での見せ方は ui.md に委ねる
- **ReceivedAt は受信段で刻印し、以後変更しない**。スプール退避・drain・停止時退避・移行（§6.2）のどこを経ても再刻印しない——スプールのセグメント形式（architecture.md §3.2.1。凍結対象）は ReceivedAt を運ぶ。drain が数時間遅れてもログの時刻・保持期間の起算・受信断区間との突合は揺れない

### 2.3 システムイベント

受信断区間（architecture.md §4.4）等、製品自身が生成する記録はログレコードとは**別の論理テーブル**に保存する（利用者のログと製品の自己記録を混ぜない——検索結果に「本物のログではない行」が混入しない）:

| 項目 | 内容 |
|---|---|
| Kind | 受信断（正常停止 / クラッシュ近似）・保持期間削除の実行記録（§3）等 |
| StartAt / EndAt | 区間（UTC） |
| Approximate | 近似断点か（architecture.md §4.4 の印） |
| Details | 付帯情報（削除実行なら削除件数等） |

- システムイベントも保持期間管理の対象とするが、**保持期間はログ本体と同じ以上とする**（ログより先に「その期間の説明」が消える逆転を防ぐ。具体値は [DB-1](#8-確定待ち一覧)）
- ログの保持期間を越えた過去では、ログの不在は保持期間削除でも説明される——その期間の受信断区間だけを見せると「受信断以外はログがあったはず」という誤読を招くため、**表示上の前提（保持地平より古い期間の扱い）を ui.md へ申し送る**（homework.md に登録）

## 3. 保持期間管理

「ログを失わない」の裏面——溜まり続けたログをいつ・どう消すか——を初版から定義する（architecture.md §7 要求④）:

- **保持期間（日数）を設定項目とし、既定値を持つ**。既定値は「無期限」にしない——ゼロ設定ファーストランの環境こそ設定を触らないまま運用され、無期限の既定はディスク枯渇として現れるため。既定の日数は [DB-1](#8-確定待ち一覧) で確定する（ディスク消費の実測とオーナー判断。Express の 10 GB との整合を含む）
- **却下案 — 容量連動の保持（サイズ上限で古い順に削除）**: ディスク枯渇への直接対策ではあるが、「何日分のログが残っているか」が流量次第で変動し利用者に予測できない。監査・調査の要求は「○日分」という日数で来るため、日数を正とする。ただし日数既定は高流量環境での枯渇を防がない（保持期間内に容量へ先に当たる）ことを認め、**ディスク空き容量・DB 容量上限への接近は保持期間とは独立に監視・警告する**（§5.3、architecture.md §4.6 の経路）
- **設定を触らない利用者にも削除の存在が届くようにする**: 保持期間の現在値（「○日より古いログは自動的に削除されます」）をファーストラン完了時とダッシュボード上で常時明示する。表示設計は ui.md（homework.md に登録）
- 期限超過分の削除は**定期実行**とし、受信の書き込みを長時間妨げない方式（分割実行等）を provider 契約（§1.2）の条件とする。さらに**実行判断側も譲歩する**: スプール退避が進行中・Q2 が高水位・drain が進行中の間は削除を開始せず、実行中でも次のバッチ境界で中断できる（メンテナンスが自ら飽和を作らない）。**ただし容量枯渇（§1.2・§5.3）を契機とする前倒し削除は譲歩条件の例外とする**——このとき退避が進行しているのは削除でしか解けない状態であり、譲歩を適用すると復旧が永久に開始されない。**保持期間削除の実行中に発生したスプール退避は昇格案内の「速度不足」に数えない**（architecture.md §5.3 に反映済み）
- **保持期間を大幅に短縮した直後の一括削除は、複数回の定期実行に分けて漸進する**（数百万行規模の削除が一撃で走ることを防ぐ）
- 保持期間による削除は「損失」ではなく意図した仕様動作であり、architecture.md §4.1 のドロップカウンタには計上しない。**実行の事実（実行時刻・削除件数）はシステムイベント（§2.3）として保存する**——保持期間はシステムイベント側（ログ本体と同じ以上）に従うため、削除記録が削除対象のログより先に消えることはない。実行の失敗は architecture.md §4.6 の能動通知の対象とする
- 保持期間の変更は管理操作であり、監査記録の対象（ADR-0004 決定 7）

## 4. 組み込み provider: SQLite

- ゼロ設定ファーストランの既定 DB。**組み込みのまま使い続ける運用も正式サポート**（ADR-0002 決定 3）
- **WAL モードを用いる**。WAL では読み取りは書き込みをブロックせず、書き込みも読み取りをブロックしない（読み書きは並行できる）。一方 **writer は同時に 1 つ**である（いずれも SQLite 公式ドキュメント "Write-Ahead Logging" の記載。確認日 2026-07-04）
- 読み書き分離の性質（§1.2 の明示義務）:
  - 検索（読み取り）が受信の書き込みをブロックしない条件は WAL により成立する。**書き込みが単一 writer である制約は、書き込みを永続化段のバッチ経路に一本化している本設計と整合する**（保持期間削除も同じ書き込み経路に直列化する）
  - **付随する運用特性 — WAL ファイルの肥大**: 読み取りが常に重なり続けると checkpoint が完了できず、WAL ファイルが際限なく成長し得る（checkpoint starvation。SQLite 公式ドキュメント "Write-Ahead Logging" §6 の記載。確認日 2026-07-04）。本設計では対話的検索のタイムアウト（architecture.md §6・M-10）が個々の読者の上限時間を画す。checkpoint の実行方式（読者の切れ目での実行等）は実装設計で確定し、WAL ファイルサイズを観測対象（architecture.md §4.6 のゲージ）に含める
- 飽和特性: 書き込み上限は実測で確定し（architecture.md §9 M-7）、昇格案内（同 §5.3）の入力とする
- **容量枯渇の扱い**: ディスク満杯による書き込み失敗は §1.2 の「容量枯渇」として報告し、§5.3 と同じ復旧連鎖（保持期間削除の前倒し実行。§3 の譲歩条件の例外）を適用する。ただし**「削除で回復し得る」が SQLite で実際に成立するか**（行削除後のページ再利用の挙動、満杯状態での削除自体がディスク追記を要さないか）は自明ではないため、推測で確定せず実機検証で確認する（[DB-8](#8-確定待ち一覧)）
- ファイルの配置・ACL は ADR-0004 決定 5 に従う（既定の置き場所は configuration.md と合同で確定 [DB-2](#8-確定待ち一覧)）

## 5. 本番 provider: SQL Server

### 5.1 接続と認証

- **Windows 統合認証を第一推奨**とする。パスワード認証（SQL 認証）を使う場合、資格情報は DPAPI machine スコープで暗号化保存する（ADR-0004 決定 5。設定スキーマは configuration.md）

### 5.2 権限の分離 — 実行時最小・管理時都度

ADR-0004 決定 4 の具体化。実行時権限を最小化すると初回セットアップの手数が増えるという既知の緊張があり、次の 3 点で解消する:

- **実行時**: サービスの接続はログテーブルへの読み書き（+ 保持期間削除）に必要な最小権限のみ。スキーマ作成・変更権限を常時保持しない
- **セットアップ・昇格・版間移行時**: 必要な管理権限は管理者資格情報の都度入力で得て、使用後破棄する（保持しない。長時間処理での再入力方針は configuration.md の管轄——ADR-0004 委任の引き受け先を明記）
- **権限不足時の自走可能な失敗**: スキーマ管理（§1.2）が権限不足で失敗する場合、「何が足りないか」と**環境の実値が入った実行可能な SQL**（ログイン作成・権限付与・DB 作成）を提示する。DBA が別部署にいる環境では、この SQL がそのまま依頼文になる。スキーマ管理は冪等とし、途中失敗からの再実行を安全にする
- **提示 SQL は秘密情報を含まない**: ログイン作成は Windows 統合認証を既定とし、SQL 認証を選んだ場合もパスワード部は明示のプレースホルダとして**生成時点で実値を埋めない**（依頼文がメール・チケットを経由して流通する前提で設計する）。監査記録（ADR-0004 決定 7）には提示 SQL の原文を保存せず、要約のみ残す

### 5.3 SQL Server Express の適用限界

無償の Express を動作対象に含める（ADR-0002 決定 3）ため、その限界を仕様として記録する（Microsoft Learn「Editions and supported features of SQL Server 2022」の値。確認日 2026-07-04）:

- **データベース最大サイズ: 10 GB**（Express / Express with Advanced Services とも）
- バッファプール最大メモリ: インスタンスあたり 1,410 MB
- 計算能力: 1 ソケットまたは 4 コアの小さい方

含意と対応:

- 10 GB は中小環境のログ保存として実用になり得る規模だが、**保持期間と流量によっては到達する**。メモリ・コア制限は障害ではなく性能の頭打ちとして現れる（性能公称値の基準環境の定義——architecture.md M-6——とは別に扱う。利用者向け文書 DB-D2 では 10 GB を主とし、他 2 点は性能上の注記として平易に述べる）
- **SQL Server provider は DB サイズの取得とエディション判別（容量上限が適用されるか否か）を必須とする**（§1.2 統計の「取得不能の明示」の逃げ道はこの provider には適用しない——Express 接近警告の成立条件であるため）。**上限への接近を警告する**（architecture.md §4.6 の能動通知の経路）。何を「サイズ」として測るか（割当ファイルサイズか使用ページ量か）・削除後の空き解放の挙動・警告閾値は、実機検証を含めて [DB-4](#8-確定待ち一覧) で確定する（割当サイズを測ると削除しても警告が消えない、という運用者の混乱を設計段階で潰す）
- **到達後の復旧連鎖**: 容量上限由来の書き込み失敗は provider が「容量枯渇」として報告し（§1.2 の 3 区分。一時障害のリトライでも恒久障害の停止でもない第 3 の経路）、ホストは**保持期間削除の前倒し実行を試みて自走復旧を図る**（この前倒し削除は §3 の譲歩条件の例外である——同節参照）。回復までの受信分はスプールが受ける（architecture.md §3.2）。前倒し削除でも回復しない場合（保持期間内でも満杯）は能動通知で保持期間の短縮かエディション移行を促す。上限到達時の SQL Server の実挙動の確認は DB-4 の実機検証に含める

## 6. 本番昇格（SQLite → SQL Server）

ADR-0002 決定 3 の昇格支援 3 点 (i)〜(iii) のうち、(i) 案内は architecture.md §5.3、(ii)(iii) を本節で定める。

### 6.1 接続切替（v0.1。昇格 (ii)）

- **準備と本番を分離する**: 切替は「準備フェーズ」（接続・権限・スキーマの事前検証。§5.2 の SQL 提示 → DBA 依頼待ちのような長時間の停滞はここで吸収する）と「切替本番」（事前検証済みの接続への差し替え。短時間で完了する）に分ける。準備フェーズは何度でも中断・再開でき、本番はすべての事前検証が通ってから実行する
- **切替本番の手順**: ①旧 provider の書き込み停止 → ②システムイベントの複写（下記）→ ③新 provider へ差し替え（**この時点をもって切替の確定とする**）→ ④スプール分の drain（新 DB へ）。この間に到着したログは Q2 とスプール（architecture.md §3.2.1——書き込みが確定できない状態そのもの）が受ける。**切替以降の新規ログは新 DB へ**保存される
- **切替中の可視化と打ち切り**: 切替中であることと経過時間をゲージ・能動通知（architecture.md §4.6）に露出する。**打ち切り監視（[DB-7](#8-確定待ち一覧)）の区間は手順①〜③、すなわち差し替えの確定まで**とする——確定までに既定時間を超えた場合は**自動で旧 provider へ戻す**（戻った場合、スプール退避分の drain 先は旧 DB である——差し替え前なので通常経路のまま）。手順④の drain の長時間化は巻き戻しの対象ではなく、通常の drain 規則（architecture.md §3.2.2）で扱う
- **差し替え確定後の障害は昇格の失敗として扱わない**: 差し替え（手順③）以降に新 DB への書き込みが失敗し始めた場合は、巻き戻さず通常の障害経路（リトライ・スプール退避）で扱う——新 DB が正であり、旧へ戻すと新 DB 側のコミット済みログが管理外の複製になる（ADR-0004 決定 5 が嫌う状態）。事前検証の分離が、この局面に至るリスク自体を縮める
- **システムイベントは切替処理が新 DB へ複写する**: 受信断区間等（§2.3）はログ本体より件数が桁違いに少なく、蓄積ログ移行 (iii) を待たずに複写できる。「ログ本体と同じ以上の保持」（§2.3）を昇格が破らないようにするため、複写は手順②——**書き込み停止後・差し替え前**——に行う（停止後なので取り残しは生じない）。**複写の失敗は巻き戻しの対象**（まだ確定前であり安全に戻せる）。巻き戻し後の新 DB に複写済みイベントが残留し、再切替で重複し得ることは、システムイベントにも適用される at-least-once の範囲として許容する（切替自体の監査記録はホストの管理操作経路であり、この複写とは独立）
- **蓄積ログは移行機能 (iii) の提供まで Web UI から参照できなくなる**。ウィザードは**切替の確定前に**この帰結を利用者へ予告する（文言は ui.md）。参照が必要な場合の選択肢は「退避 + 手動移行手順（operations.md）」である
- **旧・組み込み DB ファイルは残置しない**（ADR-0004 決定 5）。切替完了時にウィザードが「明示の退避（利用者が指定する場所への移動）」か「削除」を選択させる:
  - **退避時は移動先へ ADR-0004 決定 5 相当の ACL を適用する**（適用できない場所なら警告する）。退避完了をもって保護責任は利用者へ移転する——この移転は利用者向けドキュメントに明記する（[§9](#9-利用者向けドキュメントへの申し送り) DB-D4）
  - 処分（退避 / 削除）が完了するまで昇格は「未完了」状態であり、**未完了の間も新 DB への保存は継続する**。未完了は UI の常時表示と能動通知で可視化し、ウィザードへ再入すると処分ステップから再開する（電話で中断されても迷子にならない）
  - 退避 / 削除の**選択内容・退避先・完了の事実は監査記録の対象**とする（切替本体と同様。ADR-0004 決定 7）
- 切替は管理操作であり監査記録の対象（ADR-0004 決定 7）

### 6.2 蓄積ログの移行（後続バージョン。昇格 (iii)）

- 蓄積済みログの移行機能は**それ単体で提案書 1 本分の規模**があり、時期と方式は個別に確定する（ADR-0002 決定 3 のとおり。着手判断の入力は [DB-5](#8-確定待ち一覧)）
- ただし**満たすべき要件は先に固定する**（提案書はこの要件を出発点にする）:
  1. 移行中も受信を止めない（新規ログの保存を移行処理が妨げない）
  2. **完全性の検証を移行機能自体が持つ**: 移行元・移行先の件数突合を実行し、結果を記録する。突合の軸は ReceivedAt の範囲とし、**合格条件は at-least-once と整合させる**——「移行先件数 ≥ 移行元件数、かつ差分のレコードを抽出して重複として説明できること」（件数の完全一致を合格条件にしない——重複と損失の相殺を隠すため）
  3. 中断・再開が安全（冪等または再開点の記録。at-least-once の意味論と整合）
  4. **ReceivedAt を再刻印しない**（§2.2 の契約。移行後も時刻・保持期間の起算は変わらない）
  5. **移行由来のレコードは投入経路を事後に識別できる**（監査で「このレコードはいつ・どの経路で入ったか」に答える。重複調査にも使う。識別の持たせ方は移行設計時に確定）
  6. 移行完了後の旧データの扱いは 6.1 と同じ原則（残置しない）
- 移行の読み出しは §1.2「契約拡張の予約」の一括読み出し操作として追加する（契約外の裏口 SQL で実装しない）
- 提供までの間は**手動移行手順を文書で提供する**（operations.md 起筆時に作成 [§9](#9-利用者向けドキュメントへの申し送り) DB-D1）。手順には要件 2 と同じ判定基準の件数突合を含め、**移行実施の事実（日時・移行元/先件数・実施者）を製品の監査記録に残すステップを置く**（手動移行が製品の証跡から断絶しないようにする。記録手段はセキュリティ設計文書の監査記録仕様と合同で確定する）

## 7. v1.0 で凍結する対象（ADR-0006 基準 4 の一覧）

v1.0 以降、後方互換を破る変更を MAJOR でのみ行う対象の一覧。**本節が ADR-0006 基準 4 の「凍結対象の一覧」の置き場である**:

| 凍結対象 | 定義場所 |
|---|---|
| DB 論理スキーマ（§2。ログレコード・システムイベントの項目と意味） | 本文書 |
| provider 契約（§1.2 の操作と意味論） | 本文書 |
| スプールのセグメント形式（未消化セグメントを残した版間更新が成立すること） | architecture.md §3.2.1 |
| メタデータ領域の形式（カウンタ・稼働記録の継続性） | architecture.md §4.3 |
| 設定スキーマ | [configuration.md](configuration.md) §8 |

- **provider 契約の「凍結」の意味を精密化する**（契約の消費者は実装者であり、利用者向け互換とは破壊の意味が異なる）: 凍結されるのは**既存操作の意味論とスキーマ項目の意味**である。**契約への操作追加は optional 操作（capability 検出——provider が実装有無を宣言し、ホストが検出する機構）として MINOR で行える**。既存操作の意味論変更・必須操作の追加は MAJOR とする。§1.2 の「契約拡張の予約」（一括読み出し・集計）はこの optional 機構で追加する
- v0.x の間はこれらの互換を破る変更を許容するが、conventions.md の deprecation 階段（1 バージョン警告 → 次バージョン削除）は v0.x でも守る
- v0.x → v1.0 の更新で蓄積データと設定が保全されることの実地実証は ADR-0006 基準 4 が要求する（本表がその検証範囲の定義になる）
