## 1. provider 抽象

### 1.1 構成

- 永続化層は**初日から provider 抽象**とし、SQL 方言・機能差は provider 層に封じ込める（ADR-0002 決定 3）。上位（受信・UI・ホスト）は抽象インターフェースのみを参照し、特定 DB の SQL・接続 API を直接書かない
- v0.1 の必須 provider は **SQLite**（組み込み既定。§4）と **SQL Server**（本番第一推奨。§5）の 2 つ。PostgreSQL / MySQL は追加を意図する（約束ではない。ADR-0002 決定 3）
- 抽象の縫い目は 2 実装で検証するが、2 実装での検証が第 3 の provider への適合を保証するわけではない——十分性は適合テストスイート（§1.3）の設計で担保する（ADR-0002 の同趣旨を本文書が引き受ける）

### 1.2 provider 契約

provider が実装すべき操作と性質。インターフェースの C# シグネチャは実装設計で確定するが、**契約の意味論はここで固定する**:

| 契約 | 内容 | architecture.md §7 対応 | 主な利用者 |
|---|---|---|---|
| スキーマ管理 | スキーマの新規作成と版間移行（冪等——既に適用済みなら何もしない）。実行に必要な権限が不足する場合、**不足内容と、管理者がそのまま実行できる SQL を区別可能に報告する**（§5.2。**提示 SQL は秘密情報を含まない**） | —（本文書独自） | ホスト（起動時・昇格時） |
| バッチ挿入 | ログレコードのバッチ書き込み（architecture.md §2.1）。部分成功の扱い（全再試行で重複になっても at-least-once の範囲内）を含む | ① | 永続化段 |
| 失敗の分類報告 | 書き込み失敗を**一時障害（再試行に意味がある）・恒久障害（設定・スキーマ・権限の問題）・容量枯渇（保存領域の上限。削除で回復し得る）の 3 区分で報告する**。応答しないハングは呼び出し側のタイムアウトが打ち切る（architecture.md §3.2.1） | ② | 永続化段・ホスト |
| 対話的検索 | 条件（時間範囲・送信元・**重大度の閾値（以下。Issue #148）**・**facility**・**解析状態（ParseStatus）**・自由文等）+ **射影（一覧用の軽量列）+ 結果上限 + タイムアウト**を必須引数とする読み取り（architecture.md §6 の防御を契約レベルで強制する。**防御対象は UI の対話的検索であり、下記「契約拡張の予約」の管理経路には適用しない**）。並びは `ReceivedAt` 降順、同一 `ReceivedAt` は `Id` 降順でタイブレークする（Issue #144。UDP バースト等の同一時刻多発でも結果順序を決定的にする） | —（本文書独自。architecture.md §6 の契約化） | UI |
| 保持期間の削除 | 基準時刻より古いレコードの削除。**受信の書き込みを長時間妨げない方式**（分割実行等）であること（§3） | ④ | ホスト（定期実行） |
| 統計 | 保存件数・DB サイズ（またはサイズ取得不能の明示。**SQL Server provider には適用しない**——§5.3 の必須要件）。Express の上限接近警告（§5.3）と昇格案内の入力 | —（本文書独自） | ホスト・UI |
| 読み書き分離の性質 | 「検索が書き込みをブロックし得るか」と付随する運用特性（§4 の WAL 肥大等）を provider ごとに明示する。**これは文書化義務であり、機械検証の対象ではない**（§1.3） | ③ | 設計・レビュー |

- **重大度・facility・解析状態の絞り込み（Issue #148。M8-3 追補）**: severity は完全一致ではなく**閾値方式**（`SeverityAtMost` 以下）で絞り込む——syslog は数値が小さいほど深刻なため、「N 以上の重大度」という運用語彙は「Severity ≤ N」に対応する（完全一致では「エラー以上」のつもりで選んでも、より深刻な緊急・警報・重大が結果から消える実害があった）。severity が未設定（PRI 自体を解析できなかった行）は閾値比較の対象外——それらを明示的に拾う手段は `ParseStatus` 条件（完全一致）。facility も完全一致。いずれも既存列のみを対象としスキーマ変更は不要。**自由文検索の対象列拡張（Hostname/AppName/StructuredData 等への拡張）は本 PR のスコープ外**——DB-6（下記）が確定した対象は Message のみであり、他列への拡張自体は別途の提案を要する
- **結果順序の決定性（Issue #144）**: `ORDER BY ReceivedAt DESC, Id DESC` に統一——`Id` は挿入順（採番順）と一致するため、同一 `ReceivedAt` 内は「新しく挿入された行が先」という決定的な順序になる。カーソル（キーセット）ページングは [DB-11](#8-確定待ち一覧) の索引設計と合わせて別途検討する（v0.1 時点では `Limit` による先頭 N 件のみ）
- **契約拡張の予約**: 次の 2 操作は現契約に含めないが、必要になる時点が既に見えているため、**§7 の互換性規則に従う optional 操作（capability 検出）として追加する**ことを予約する: (a) **一括読み出し**——移行（§6.2）・エクスポート用のストリーミング読み出し。上限・タイムアウトの必須化は適用せず、呼び出し経路を管理操作に限定する。(b) **集計**——ダッシュボード用の件数集計等（要否と形は ui.md 起筆時に確定）。対話的検索の防御を契約全体の禁止事項に一般化しない
- **M8-3（Issue #70）で追加した読み取り専用 3 操作**: 閲覧 3 画面の実装が要求する最小の読み取り口として、`ILogStore` へ次を追加した——①**詳細表示の個別取得** `FindByIdAsync`（全項目——Message 全文・StructuredData・Raw——を 1 件返す。architecture.md §6「全文は詳細表示で個別取得する」の契約化）、②**システムイベントの読み出し** `QuerySystemEventsAsync`（§2.3 の可視化面。区間の重なり判定 + 上限 + タイムアウト必須。Issue #150——保持期間削除の起動時キャッチアップ——で **Kind の完全一致フィルタ（省略可・既定は全種別）を最小の契約拡張として追加**した。理由は §3「起動時キャッチアップ」参照）、③**送信元別の集計** `QuerySourceActivityAsync`（予約 (b) の初の実体化。送信元ごとの最終受信時刻・件数を**最終受信時刻の古い順**で返す——ui.md UI-4 の無音化検出の要件「量の上位 N 抽出は不可」を集計の並び・打ち切り方向のレベルで満たす）。いずれもタイムアウトを必須引数とし、書き込みを行わない。v0.x のため通常のインターフェースメソッドとして追加した（optional 操作——capability 検出——へ移すかは §7 の v1.0 凍結判断で確定する）。適合テストスイート（§1.3）に 3 操作の検証を追加済み。(a) 一括読み出しは引き続き予約のまま
- **M8-5（Issue #159）で追加した読み取り専用 2 操作**: ダッシュボードに欠けていた「平常時からの逸脱検知」と「フラッディング検知」の 2 視点を満たすため、`ILogStore` へ予約 (b) 集計の追加実体化として次を追加した——④**重大度分布** `QuerySeverityDistributionAsync`（観測窓内の重大度別件数。PRI が解析できず Severity 未設定のレコードも 1 バケットとして返す——解析失敗の事実を隠さない）、⑤**受信量上位の送信元（Top talkers）** `QueryTopTalkersAsync`（観測窓内の件数降順。`QuerySourceActivityAsync` とは逆方向の集計であり、UI-4「量の上位 N 抽出は不可」の対象を無音化検出の一覧に限定するため、既存操作を書き換えずに別メソッドとして追加した）。**両操作は観測窓（`from`/`to`）を必須引数とする**——無条件の集計は大規模時にフルスキャンへ劣化し得るため、索引済みの `ReceivedAt` 範囲へ先に絞り込んでから集計する設計とした（Severity・SourceAddress の複合索引は Issue #145 のスキーマ v1→v2 移行で追加済みだが、窓の必須化はそれとは独立の防御として維持する）。ダッシュボード実装（M8-5）は受信量推移と同じ窓（直近 1 時間）を用いる。適合テストスイート（§1.3）に 2 操作の検証を追加済み
- **自由文検索の一致規則（[DB-6](#8-確定待ち一覧)。規則は確定。SQL Server は v1→v2 スキーマ移行で列 COLLATE を適用済み・SQLite の非 ASCII の実装と検証は未了（DB-9 待ち）。2026-07-09 オーナー決定「大文字小文字非区別で provider 間統一」）**: 対象列は `LogRecord.Message` のみ（[LogQuery.cs](../../src/Yagura.Storage/LogQuery.cs) の暫定規則を確定として採用。送信元等の他条件は完全一致のまま §1.2 表のとおり）。一致方式は部分一致（前後を問わない中間一致）。**大文字小文字は区別しない。そして折り畳むのは大文字小文字のみ**——アクセント記号の同一視（café ≒ cafe）・かな種の同一視（ひらがな ≒ カタカナ）・全角/半角の同一視は、いずれもこの規則の対象に**含めない**（同一視しない。したがって大文字小文字を持たない日本語文字——ひらがな・カタカナ・漢字——の一致判定はこの規則で変わらない）。検証の強度は二段階に分ける:
  - **ASCII 範囲（A–Z/a–z）の非区別は両 provider が確実に満たす最小要件**とする（適合テストで blocking——不合格は provider 受け入れを止める）
  - **非 ASCII の大文字小文字非区別は provider 間で揃えることを目標**とし、適合テストスイート（§1.3）の非 ASCII 大小ケースで検証する。合否の「正」は特定 provider の挙動ではなく **Unicode の simple case folding が一意に定まる文字の集合（保証集合）**に置く——保証集合の具体的な文字一覧は実装バッチで確定するが、**Issue #146 の再現例（CAFÉ/café）を含むラテン基本の大小ペアを必ず含める**。**保証集合は正例と負例の双方を blocking として含める**——正例（同一視されるべき大小ペア）に加え、負例（同一視されて**はならない**ペア: ひらがな「あ」/カタカナ「ア」・全角「Ａ」/半角「A」・café/cafe）を明記する（「規則を超えた過剰な同一視」——本改訂の初稿が SQL Server 照合順序の KS/WS 省略で作り込みかけた欠陥と同種——の回帰を機械検出するため）。保証集合外の既知の難例（トルコ語 İ/ı・ドイツ語 ß・ギリシャ語の語末シグマ等、言語依存・多対一の折り畳みを持つ文字）は非 blocking（情報表示）とし、provider 間の差異は各 provider 節に性質として文書化する（CI 回帰ベンチの `enforceRatio` と同じ「blocking / 情報表示を明示的に区別する」運用）
  - この二段階の置き方は「特定 provider の挙動を暗黙の基準にしない」（§1.3）の実行形である。**規則の確定は Issue #146 の不具合解消の宣言ではない**——不具合は、規則の確定（本項）と実現手段の適用（SQLite = §4、SQL Server = §5.4。Issue #145・#147 と同一のスキーマ移行バッチで実装・検証）の両方が揃って解消する
- 契約に**含めない**もの: カウンタ等のメタデータ領域（ホスト管轄のローカルファイル。architecture.md §4.3）、スプール（永続化モジュール内だが provider の外側）

### 1.3 適合テストスイート — 外部貢献の入口

- **すべての provider が合格すべき単一の適合テストスイート**を v0.1 の開発範囲に含める（SQLite・SQL Server 自身もこのスイートで検証する——「2 実装があって初めて抽象が検証される」の実行形）
- 検証対象を 2 種に区別する: **機械検証する契約**（§1.2 の操作——スキーマ管理・バッチ挿入・失敗の分類・対話的検索・保持期間削除・統計——と境界ケース: 最大長メッセージ・不正 UTF-8・タイムスタンプ境界・空バッチ・保持期間削除と挿入の並行）と、**文書化義務**（読み書き分離の性質——テストではなくレビューで検証する）
- 対話的検索の一致規則（自由文の対象列・大文字小文字・部分一致の意味論）は §1.2 で確定済み（[DB-6](#8-確定待ち一覧)。2026-07-09。ASCII = blocking・非 ASCII = 保証集合 blocking / 保証集合外は情報表示、の二段階）。適合テストスイートへの反映（ASCII・非 ASCII 双方の大文字小文字ケース——**同一視すべき正例と、かな種・全角/半角・アクセントを同一視してはならない負例の双方**（§1.2 の保証集合）——の追加。**性能側——タイムアウト予算内に収まるか——は適合テストではなく `tools/Yagura.Bench` の実測 [DB-9](#8-確定待ち一覧) が受け持つ**）は、SQL Server の列 COLLATE 付与・SQLite の比較関数導入（§4・§5.4）と同じスキーマ移行バッチ（Issue #145・#147 と合流）で行う。**特定 provider の挙動を暗黙の基準にしない**——規則を先に固め、全 provider を同じ規則で検証する。**反映状況**: ASCII の正例（error/ERROR 等）は両 provider の適合スイートで blocking として検証済み。非 ASCII の正例（CAFÉ/café）と負例（あ/ア・全角Ａ/半角A・café/cafe）は SQL Server 側の適合テストで blocking として検証済み——SQLite 側の非 ASCII ケースは実現手段の確定（DB-9）後に追加する
- **新 provider の追加提案は「適合テストスイートが通ること」を受け入れの技術条件とする**。提案は本文書に provider 節（§4・§5 と同格。読み書き分離の性質の明示を含む）を追加する PR として行う——性質明示の置き場は本文書である。提案手順（Issue → 設計対話 → PR）は CONTRIBUTING の管轄。実証完了前（SQLite + SQL Server が本番品質で動く前）に外部から provider 追加 PR が届いた場合、適合テストの整備を優先課題に繰り上げる（ADR-0002 再評価トリガの受け皿）
- 適合テストの CI 実行構成と、追加 DB の実行環境（コンテナ等）の提供責任の所在（貢献者側か受け入れ側か）は実装設計で確定する（[DB-3](#8-確定待ち一覧)）

## 2. 論理スキーマ

物理設計（DDL・索引・データ型の方言）は各 provider に委ねる。ここでは**全 provider が同じ意味で保存すべき論理形**を固定する。

### 2.1 ログレコード

| 項目 | 内容 | 備考 |
|---|---|---|
| Id | レコード識別子 | provider 内で一意。採番方式は provider 実装に委ねる |
| ReceivedAt | **サーバが受信した時刻（UTC）** | 検索・並び・保持期間の基準軸。刻印点は §2.2 |
| SourceAddress / SourcePort | 送信元 | ソケット層で取得 |
| Protocol | UDP / TCP（将来: TLS） | 受信リスナ由来 |
| DeviceTimestamp | 送信元が名乗った時刻（あれば） | **参考情報。基準軸にしない**（§2.2）。解析失敗時も、TIMESTAMP がその前段で解析済みなら保持する（後述「解析失敗時のフィールド保持」） |
| Facility / Severity | syslog PRI の分解値 | PRI 自体が解析できなかった場合は未設定。**PRI は解析済みで、より後段（本文等）の解析失敗で ParseStatus が解析失敗になった場合は保持する**（後述） |
| Hostname / AppName / ProcId / MsgId | RFC 5424 ヘッダ（RFC 3164 は対応部分をマップ） | 解析できた範囲で設定。**この「解析できた範囲」は ParseStatus が解析失敗のレコードにも及ぶ**（後述）。**長さの上限は設けない**（Issue #147。RFC 5424 の各上限を超える非準拠に長い値を送る機器が現実に存在し、黙った切り詰めは調査の手掛かりを消す。全 provider が全長を保存する——SQLite = TEXT・SQL Server = NVARCHAR(MAX)。適合テストの最大長超過ヘッダケースで検証する） |
| StructuredData | RFC 5424 構造化データ | 原文のまま保存（要素分解はしない——加工しない方針。ADR-0001 スコープ外の「加工パイプライン」に踏み込まない）。**一覧射影（`LogRecordSummary`）にも原文のまま含める**（切り詰めはしない——ui.md §4 の一覧で本文の前に接頭表示するための入力。2026-07-07 オーナー承認）。自身が非 UTF-8 で解析失敗した場合は未設定（後述） |
| Message | メッセージ本文 | 一覧は先頭 N 文字の射影で返す（architecture.md §6）。非 UTF-8 で解析失敗した場合は未設定（Raw に生バイト列が残る） |
| Raw | **受信したバイト列そのもの**（不正 UTF-8 を含み得るため、文字列変換を経ない） | **ParseStatus が解析失敗・不完全のときのみ保存する**（このとき Raw が一次保存物）。解析済みレコードには保存しない——却下案「常時保存」はストレージをほぼ倍化させる一方、StructuredData・Message を原文のまま保持する本スキーマでは追加で守れる情報がヘッダの整形差程度に留まるため採らない |
| ParseStatus | 解析済み / 解析失敗 / 不完全 | **排他の 3 値。不完全（TCP 切断による途中終端）は解析失敗に優先する**。解析失敗もログとして検索・閲覧できる |

**解析失敗時のフィールド保持**（Issue #139。2026-07-09）: PRI・HEADER・STRUCTURED-DATA・Message は
それぞれ独立に確定するタイミングを持つ（PRI → TIMESTAMP/HOSTNAME/AppName/ProcId/MsgId →
STRUCTURED-DATA → Message の順）。**ParseStatus が解析失敗になった時点より手前で既に確定した
項目は破棄せず保存する**。対象は原因を問わない——典型は Message/STRUCTURED-DATA の非 UTF-8
（CP932/Shift-JIS 等でログを送る国産機器を含む）だが、TIMESTAMP の値だけが RFC 3339 として
不正な場合（時計設定が壊れた機器等。確定済みの HOSTNAME 等は保持し DeviceTimestamp のみ未設定）や、
STRUCTURED-DATA 直後の構造違反（確定済みの HEADER・StructuredData は保持）も同じ規則に従う。
従来は解析失敗の原因がどこにあっても Raw のみを保持し確定済みヘッダごと捨てていたため、
「本文だけ非 UTF-8 の機器」のログがホスト名でさえ検索できなくなっていた（旧実装の欠陥。修正は
[src/Yagura.Ingestion/Parsing/SyslogParser.cs](../../src/Yagura.Ingestion/Parsing/SyslogParser.cs)）。
一方、フィールドのトークン化そのものが失敗した場合（PRI 不正、HEADER が ABNF 途中で途切れる等）は、
値が確定していないため該当フィールド以降を設定しない。**非 UTF-8 本文をフォールバックエンコーディング
（CP932 自動判定等）で読み替えて Message に格納することは本対応の範囲外**（Raw の生バイト列から
利用者が事後的に判読する運用を継続する。フォールバックエンコーディング対応は将来課題として追跡する）。

**RFC 3164 の寛容な TIMESTAMP 読み取り**（Issue #135。2026-07-09）: RFC 3164 は「べき」規定が多く実装依存の逸脱が現実に多い（§2.1 の best-effort 方針と同じ思想）。TIMESTAMP は厳密な `Mmm dd hh:mm:ss` に加え、次のベンダ拡張（主に Cisco IOS）を取りこぼさない: 秒の小数部（msec。例: `.345`）、TIMESTAMP 直後の TZ 付記（前述の優先順位①）、先頭シーケンス番号（`service sequence-numbers`。例: `45: Mar  1 ...`）、`Mmm dd yyyy hh:mm:ss` の 4 桁年変種。これらの形式に一致しない・後続の解析（HOSTNAME 等）が破綻する場合は、従来どおり TIMESTAMP 解析全体を諦めて CONTENT 側（Message）へ委ねる best-effort にフォールバックする（ログを失わない。値そのものは失われず本文に残る）。

### 2.2 時刻の契約

- **保存はすべて UTC**。表示層（UI・通知）でローカル時刻へ変換する。この契約は ui.md の画面設計より先に固定し、画面はこれに従う
- **基準軸は ReceivedAt（サーバ受信時刻）**とする。送信元の時計は信頼できない（進み・遅れ・未設定が現実に起こる）ため、DeviceTimestamp は保存するが、検索の並び・保持期間・受信断区間との突合には使わない。両者のずれが大きいことはそれ自体が運用上の発見であり、UI での見せ方は ui.md に委ねる
- **ReceivedAt は受信段で刻印し、以後変更しない**。スプール退避・drain・停止時退避・移行（§6.2）のどこを経ても再刻印しない——スプールのセグメント形式（architecture.md §3.2.1。凍結対象）は ReceivedAt を運ぶ。drain が数時間遅れてもログの時刻・保持期間の起算・受信断区間との突合は揺れない
- **RFC 3164 TIMESTAMP のタイムゾーン解釈**（Issue #134・#135。2026-07-09）: RFC 3164 の TIMESTAMP（`Mmm dd hh:mm:ss`）は年もタイムゾーンも持たず、値は送信機器のローカル時刻（RFC 3164 §4.1.2）。DeviceTimestamp への変換時、タイムゾーンは次の優先順位で解決する:
  1. **TIMESTAMP に送信元付記の TZ が取れた場合はそれを最優先で使う**（Cisco IOS の `service timestamps log datetime show-timezone` 等。数値オフセット `+HH:MM`/`+HHMM`、または曖昧でない略号 `UTC`/`GMT`/`JST` に限る——米国 4 系統の略号 `CST`/`EST`/`PST`/`MST` 等は Cisco の `clock timezone <name> <offset>` が任意の名称を任意のオフセットに割り当てられるため、略号だけからオフセットを断定できず意図的に解決対象へ含めない。未知の略号は TZ として消費せず、従来どおり HOSTNAME 側の解析に委ねる）
  2. **①が取れない場合、設定項目 `Ingestion:Rfc3164:DefaultTimeZone`（configuration.md）の既定タイムゾーンを適用する**。値は Windows タイムゾーン ID・IANA タイムゾーン ID のいずれも受理し、TIMESTAMP の年月日時刻に対する UTC オフセットを都度計算する（DST を持つゾーンでも季節に応じた正しいオフセットになる）
  3. **未設定時は UTC**（2026-07-09 以前の現状互換）
  年についても、`Mmm dd yyyy hh:mm:ss` の 4 桁年変種が取れた場合はそれを使い、取れない場合は ReceivedAt から最も近い年を補完する（既存の年またぎ補完。実装は [SyslogParser.TryReadRfc3164Timestamp](../../src/Yagura.Ingestion/Parsing/SyslogParser.cs)）。本節はいずれも DeviceTimestamp（参考情報。基準軸にしない）の精度改善であり、ReceivedAt ベースの検索・保持期間・受信断突合には影響しない。

### 2.3 システムイベント

受信断区間（architecture.md §4.4）等、製品自身が生成する記録はログレコードとは**別の論理テーブル**に保存する（利用者のログと製品の自己記録を混ぜない——検索結果に「本物のログではない行」が混入しない）:

| 項目 | 内容 |
|---|---|
| Kind | 受信断（正常停止 / クラッシュ近似）・保持期間削除の実行記録（§3）等 |
| StartAt / EndAt | 区間（UTC） |
| Approximate | 近似断点か（architecture.md §4.4 の印） |
| Details | 付帯情報（削除実行なら削除件数等） |

- システムイベントも保持期間管理の対象とするが、**保持期間はログ本体と同じ以上とする**（ログより先に「その期間の説明」が消える逆転を防ぐ。既定 30 日（2026-07-05 オーナー決定。実測根拠 = [tools/Yagura.Bench/results/2026-07-05-dev-machine/](../../tools/Yagura.Bench/results/2026-07-05-dev-machine/)）——システムイベントはログ本体と同じ既定日数を採用し、「ログ本体と同じ以上」の制約を満たす。システムイベント専用の別既定値は設けない）
- ログの保持期間を越えた過去では、ログの不在は保持期間削除でも説明される——その期間の受信断区間だけを見せると「受信断以外はログがあったはず」という誤読を招くため、**表示上の前提（保持地平より古い期間の扱い）を ui.md へ申し送る**（homework.md に登録）

## 3. 保持期間管理

「ログを失わない」の裏面——溜まり続けたログをいつ・どう消すか——を初版から定義する（architecture.md §7 要求④）:

- **保持期間（日数）を設定項目とし、既定値を持つ**。既定値は「無期限」にしない——ゼロ設定ファーストランの環境こそ設定を触らないまま運用され、無期限の既定はディスク枯渇として現れるため。**既定は 30 日**（2026-07-05 オーナー決定。[DB-1](#8-確定待ち一覧) 確定値。実測根拠 = [tools/Yagura.Bench/results/2026-07-05-dev-machine/](../../tools/Yagura.Bench/results/2026-07-05-dev-machine/)——レコード単価 ≈ メッセージ長 + 約 95 B、10 msg/s × 30 日 ≈ 7.8 GB は SQL Server Express の 10 GB 上限に収まる）。**設定ファイルで不正値（0 以下・数値以外）が指定された場合のフォールバック先は「削除しない」とし、既定 30 日へは自動フォールバックしない**（意図しない自動削除の開始を避ける安全側の判断。configuration.md §8 参照）。容量超過そのものへの対策ではなく、**ディスク空き容量・DB 容量上限への接近は保持期間とは独立に監視・警告する**（後述・§5.3）
- **却下案 — 容量連動の保持（サイズ上限で古い順に削除）**: ディスク枯渇への直接対策ではあるが、「何日分のログが残っているか」が流量次第で変動し利用者に予測できない。監査・調査の要求は「○日分」という日数で来るため、日数を正とする。ただし日数既定は高流量環境での枯渇を防がない（保持期間内に容量へ先に当たる）ことを認め、**ディスク空き容量・DB 容量上限への接近は保持期間とは独立に監視・警告する**（§5.3、architecture.md §4.6 の経路）
- **設定を触らない利用者にも削除の存在が届くようにする**: 保持期間の現在値（「○日より古いログは自動的に削除されます」）をファーストラン完了時とダッシュボード上で常時明示する。表示設計は ui.md（homework.md に登録）
- 期限超過分の削除は**定期実行**とし、受信の書き込みを長時間妨げない方式（分割実行等）を provider 契約（§1.2）の条件とする。さらに**実行判断側も譲歩する**: スプール退避が進行中・Q2 が高水位・drain が進行中の間は削除を開始せず、実行中でも次のバッチ境界で中断できる（メンテナンスが自ら飽和を作らない）。**ただし容量枯渇（§1.2・§5.3）を契機とする前倒し削除は譲歩条件の例外とする**——このとき退避が進行しているのは削除でしか解けない状態であり、譲歩を適用すると復旧が永久に開始されない。**保持期間削除の実行中に発生したスプール退避は昇格案内の「速度不足」に数えない**（architecture.md §5.3 に反映済み）
- **定期実行は起動時キャッチアップを持つ**: 定期実行の実行時刻（既定は深夜帯）に稼働していないホスト——日中のみ稼働する設置・再起動が実行時刻付近に偏る環境——では、定刻待ちだけの実装は削除を恒常的にスキップし、「保持期間を設定した＝消える」という利用者の前提を静かに裏切る（無期限の既定を排した本節の安全網に穴が開く）。これを防ぐため、スケジューラは**起動時に直近の削除実行記録（下記のシステムイベント）を検索し、前回実行から 1 日以上経過している（または記録が存在しない）場合は定刻を待たず即座に削除を実行する**。キャッチアップ判定の情報源は削除実行のシステムイベント（§2.3。既存の記録）であり、**専用のメタデータ領域は設けない**——実行の事実の記録と判定の情報源を一本化する。**判定の検索は Kind（`retention.delete`）でサーバ側フィルタする**——削除実行記録の StartAt は意図的な過去日付（cutoff）である一方、受信断イベントの StartAt は実時刻という非対称があり、種別を問わない直近 N 件の走査では受信断の蓄積が削除記録をウィンドウから恒常的に押し出すため（このために §1.2 の `QuerySystemEventsAsync` へ Kind フィルタを最小の契約拡張として追加した）。キャッチアップ直後に定刻がわずかな遅延で到来した場合の二重実行は、直前の実行からの経過時間による抑止（プロセス内メモリで足りる——抑止に失敗しても削除は冪等であり、防いでいるのは無駄な再実行のみ）で防ぐ。実装 = `Yagura.Host.Retention.RetentionScheduler`
- **保持期間を大幅に短縮した直後の一括削除は、複数回の定期実行に分けて漸進する**（数百万行規模の削除が一撃で走ることを防ぐ）
- 保持期間による削除は「損失」ではなく意図した仕様動作であり、architecture.md §4.1 のドロップカウンタには計上しない。**実行の事実（実行時刻・削除件数）はシステムイベント（§2.3）として保存する**——保持期間はシステムイベント側（ログ本体と同じ以上）に従うため、削除記録が削除対象のログより先に消えることはない。実行の失敗は architecture.md §4.6 の能動通知の対象とする
- 保持期間の変更は管理操作であり、監査記録の対象（ADR-0004 決定 7）

## 4. 組み込み provider: SQLite

- ゼロ設定ファーストランの既定 DB。**組み込みのまま使い続ける運用も正式サポート**（ADR-0002 決定 3）
- **WAL モードを用いる**。WAL では読み取りは書き込みをブロックせず、書き込みも読み取りをブロックしない（読み書きは並行できる）。一方 **writer は同時に 1 つ**である（いずれも SQLite 公式ドキュメント "Write-Ahead Logging" の記載。確認日 2026-07-04）
- 読み書き分離の性質（§1.2 の明示義務）:
  - 検索（読み取り）が受信の書き込みをブロックしない条件は WAL により成立する。**書き込みが単一 writer である制約は、書き込み呼び出しをホストの単一の書き込みゲートで直列化する本設計と整合する**——ライブ書き込み（永続化段）・スプール drain・保持期間削除（+ 実行記録のシステムイベント）の 3 経路は、ホストが構築する共有ゲート（実装 = `Yagura.Storage.LogStoreWriteGate`。結線 = `Yagura.Host.Program`）を通ることで provider 契約（§1.2「呼び出し側が直列化する」）を実配線で満たす。ゲート待ちの時間予算は DB 操作自体のタイムアウトとは独立させる（保持期間削除がゲートを保持する間のゲート待ちを「速度不足」と誤認してスプール退避が連発することを防ぐ——§3・architecture.md §5.3 の誤発火防止と同じ趣旨。設計判断の詳細は `LogStoreWriteGate` の doc コメント）。なお、ホスト起動処理の受信断記録（architecture.md §4.4。起動時 1 回の `WriteSystemEventAsync` 直接呼び出し）はゲートを通らない第 4 の書き込み経路だが、上記 3 経路のいずれもまだ開始されていない時点で実行されるため、非同時実行は起動順序により保証される（`ILogStore` の doc コメントに明記）
  - **付随する運用特性 — WAL ファイルの肥大**: 読み取りが常に重なり続けると checkpoint が完了できず、WAL ファイルが際限なく成長し得る（checkpoint starvation。SQLite 公式ドキュメント "Write-Ahead Logging" §6 の記載。確認日 2026-07-04）。本設計では対話的検索のタイムアウト（architecture.md §6・M-10）が個々の読者の上限時間を画す。checkpoint の実行方式（読者の切れ目での実行等）は実装設計で確定し、WAL ファイルサイズを観測対象（architecture.md §4.6 のゲージ）に含める
- 飽和特性: 書き込み上限は実測で確定し（architecture.md §9 M-7）、昇格案内（同 §5.3）の入力とする
- **容量枯渇の扱い**: ディスク満杯による書き込み失敗は §1.2 の「容量枯渇」として報告し、§5.3 と同じ復旧連鎖（保持期間削除の前倒し実行。§3 の譲歩条件の例外）を適用する。ただし**「削除で回復し得る」が SQLite で実際に成立するか**（行削除後のページ再利用の挙動、満杯状態での削除自体がディスク追記を要さないか）は自明ではないため、推測で確定せず実機検証で確認する（[DB-8](#8-確定待ち一覧)）
- **自由文検索の大文字小文字（[DB-6](#8-確定待ち一覧)。§1.2 の規則を SQLite でどう実現するか）**: SQLite の `LIKE` は既定で ASCII 範囲のみ大文字小文字を区別しない（SQLite 公式ドキュメント "The LIKE operator" — https://www.sqlite.org/lang_expr.html。確認日 2026-07-05）。組み込みの `NOCASE` 照合順序も同じ制限を持つ（SQLite 公式ドキュメント "Datatypes In SQLite" §7 Collating Sequences — https://www.sqlite.org/datatype3.html: 「the 26 upper case characters of ASCII are folded to their lower case equivalents... Note that only ASCII characters are case folded. SQLite does not attempt to do full UTF case folding due to the size of the tables required」。確認日 2026-07-09）——**SQLite に非 ASCII を含む大文字小文字非区別を標準機能だけで実現する手段はない**。
  - **ICU 拡張は採用しない**: SQLite 本体は ICU 拡張のソースを同梱するが、既定ビルドには含まれず `SQLITE_ENABLE_ICU` コンパイルオプションを指定した独自ビルドが要る（SQLite 公式ドキュメント "Compile-Time Options" — https://www.sqlite.org/compile.html: 「This option causes the International Components for Unicode ("ICU") extension to SQLite to be added to the build」——既定では追加されないことの裏返し。確認日 2026-07-09）。Yagura が利用するネイティブ SQLite バイナリ（SQLitePCLRaw 経由の配布物）はこのオプションを有効化した配布形態を持たず、独自コンパイルはプラットフォームごとの再ビルド・差し替えという継続的な運用コストを生む——本節冒頭の「組み込みのまま使い続ける運用も正式サポートする」という設計方針に反するため却下する
  - **第一候補の方式: 自由文検索専用にアプリケーション定義の比較関数を導入する**（**方式の最終確定は性能実測後**。下記の受け入れ条件参照）。Microsoft.Data.Sqlite は `SqliteConnection.CreateFunction` により .NET 側で実装したスカラー関数を SQL から呼び出し可能にする機構を公式に提供する（Microsoft Learn "User-defined functions - Microsoft.Data.Sqlite" — https://learn.microsoft.com/dotnet/standard/data/sqlite/user-defined-functions。確認日 2026-07-09）。**組み込み `LIKE` 演算子自体のグローバルな上書き**（同文書は `X LIKE Y` が内部で `like(Y, X)` 関数へ解決され、同名関数の再定義で演算子の挙動そのものを上書きできるとも説明する）は第一候補にしない——現時点で `LIKE` の利用箇所は自由文検索のみだが、演算子の意味を接続全体で差し替えると将来の利用箇所が暗黙に影響を受けるため、既定は呼び出し経路を限定した専用関数とする（両者の性能差は下記実測に含めてよい）
  - **方式確定の受け入れ条件（実装バッチ）**: 行ごとに managed コードへコールバックする比較関数は、ネイティブ `LIKE` の全表走査より遅くなる可能性が高い（Issue #145 が指摘済みの「中間一致は索引不可・常に全走査」の 1 行あたり単価を上げる方向の変更である）。**`tools/Yagura.Bench` で行数規模別にネイティブ `LIKE` と比較実測し、対話的検索のタイムアウト予算内に収まることを確認してから方式を確定する**（[DB-9](#8-確定待ち一覧)）。**許容できない場合のフォールバックをあらかじめ定める**: ASCII 限定のネイティブ `LIKE`（現行実装）を維持し、非 ASCII の大文字小文字非区別は Issue #145 の全文検索索引（FTS5）検討に委ねる——このとき §1.2 の非 ASCII 保証集合の検証は SQLite provider では「未達の性質」として文書化する（規則自体は変えない）
  - **Issue #145（FTS5 検討）との関係**: FTS5 はトークナイザベースの索引でありスカラー比較関数とは実現機構が別系統（大文字小文字の折り畳みはトークナイザ側の設定で扱う）。**凍結対象は §1.2 の一致規則であって実現機構ではない**——#145 で FTS5 を採用する場合、本比較関数は FTS5 側の実装に置き換わってよい（同じ規則を満たすことを適合テストで再検証する）。比較関数方式はその意味で「FTS5 の要否が確定するまでの規則充足手段」であり、#145 の選択肢を拘束しない
  - **.NET グローバリゼーションへの依存を前提として明記する**: 比較関数の非 ASCII 折り畳みは .NET のカルチャ非依存の大小変換（ICU ベース）に依存する。.NET には ICU を切り離す **Globalization Invariant Mode**（MSBuild プロパティ `InvariantGlobalization`。有効化すると「culture-specific data and behavior へのアクセスなし」で動作する——Microsoft Learn "Globalization config settings" — https://learn.microsoft.com/dotnet/core/runtime-config/globalization。確認日 2026-07-09）があり、これが有効な発行構成では非 ASCII の折り畳みが成立しない。**Yagura の発行設定は Invariant Mode を有効化しない（ICU を利用可能に保つ）ことを本方式の前提とし、実装バッチでビルド設定の固定とテストによる検知を含める**
  - **折り畳み範囲の非対称の申し送り**: SQL Server 側（§5.4）は `_SC` により補助文字（U+10000 以降）まで比較対象に含むが、SQLite 比較関数がサロゲートペアをどこまで折り畳むかは実装設計で確定する。差が残る場合は §1.2 の二段階検証（保証集合外の性質文書化）で扱う
- ファイルの配置・ACL は ADR-0004 決定 5 に従う（既定の置き場所は configuration.md と合同で確定 [DB-2](#8-確定待ち一覧)）

## 5. 本番 provider: SQL Server

### 5.1 接続と認証

- **Windows 統合認証を第一推奨**とする。パスワード認証（SQL 認証）を使う場合、資格情報は DPAPI machine スコープで暗号化保存する（ADR-0004 決定 5。設定スキーマは configuration.md）

### 5.2 権限の分離 — 実行時最小・管理時都度

ADR-0004 決定 4 の具体化。実行時権限を最小化すると初回セットアップの手数が増えるという既知の緊張があり、次の 3 点で解消する:

- **実行時**: サービスの接続はログテーブルへの読み書き（+ 保持期間削除）に必要な最小権限のみ。スキーマ作成・変更権限を常時保持しない
- **セットアップ・昇格・版間移行時**: 必要な管理権限は管理者資格情報の都度入力で得て、使用後破棄する（保持しない。長時間処理での再入力方針は configuration.md の管轄——ADR-0004 委任の引き受け先を明記）
- **権限不足時の自走可能な失敗**: スキーマ管理（§1.2）が権限不足で失敗する場合、「何が足りないか」と**環境の実値が入った実行可能な SQL**（ログイン作成・権限付与・DB 作成）を提示する。DBA が別部署にいる環境では、この SQL がそのまま依頼文になる。スキーマ管理は冪等とし、途中失敗からの再実行を安全にする
- **提示 SQL は秘密情報を含まない**: ログイン作成は Windows 統合認証を既定とし、SQL 認証を選んだ場合もパスワード部は明示のプレースホルダとして**生成時点で実値を埋めない**（依頼文がメール・チケットを経由して流通する前提で設計する）。監査記録（ADR-0004 決定 7）には提示 SQL の原文を保存せず、要約のみ残す

### 5.3 SQL Server Express の適用限界

無償の Express を動作対象に含める（ADR-0002 決定 3）ため、その限界を仕様として記録する（Microsoft Learn「Editions and supported features of SQL Server 2022」の値。確認日 2026-07-04）:

- **データベース最大サイズ: 10 GB**（Express / Express with Advanced Services とも）
- バッファプール最大メモリ: インスタンスあたり 1,410 MB
- 計算能力: 1 ソケットまたは 4 コアの小さい方

含意と対応:

- 10 GB は中小環境のログ保存として実用になり得る規模だが、**保持期間と流量によっては到達する**。メモリ・コア制限は障害ではなく性能の頭打ちとして現れる（性能公称値の基準環境の定義——architecture.md M-6——とは別に扱う。利用者向け文書 DB-D2 では 10 GB を主とし、他 2 点は性能上の注記として平易に述べる）
- **SQL Server provider は DB サイズの取得とエディション判別（容量上限が適用されるか否か）を必須とする**（§1.2 統計の「取得不能の明示」の逃げ道はこの provider には適用しない——Express 接近警告の成立条件であるため）。**上限への接近を警告する**（architecture.md §4.6 の能動通知の経路。**Issue #149（M4-6）で配線済み**——`Yagura.Host.Observability.ActiveNotification.ActiveNotificationMonitor` が周期監視し、使用率が閾値（仮値 0.8。architecture.md §9 M-16）以上でイベントログへ警告する。判定は `SqlServerLogStore.IsExpressEditionAsync` + `GetStatisticsAsync`（割当ファイルサイズ）を用いる）。何を「サイズ」として測るか（割当ファイルサイズか使用ページ量か——**現状は割当ファイルサイズのみ**）・削除後の空き解放の挙動・警告閾値の最終確定は、実機検証を含めて [DB-4](#8-確定待ち一覧) で行う（割当サイズを測ると削除しても警告が消えない、という運用者の混乱を設計段階で潰す）
- **到達後の復旧連鎖**: 容量上限由来の書き込み失敗は provider が「容量枯渇」として報告し（§1.2 の 3 区分。一時障害のリトライでも恒久障害の停止でもない第 3 の経路）、ホストは**保持期間削除の前倒し実行を試みて自走復旧を図る**（この前倒し削除は §3 の譲歩条件の例外である——同節参照）。回復までの受信分はスプールが受ける（architecture.md §3.2）。前倒し削除でも回復しない場合（保持期間内でも満杯）は能動通知で保持期間の短縮かエディション移行を促す。上限到達時の SQL Server の実挙動の確認は DB-4 の実機検証に含める

### 5.4 自由文検索の照合順序（COLLATE）

- **列単位で COLLATE を明示する**（[DB-6](#8-確定待ち一覧) 確定）: サーバの既定照合順序に委ねない。既定はインストール時の OS ロケールに依存し、大文字小文字を区別する照合順序（CS）が入ったサーバでは「自由文検索は大文字小文字を区別しない」という §1.2 の契約が配備先ごとに静かに反転する——Issue #146 で確認された不具合そのものであり、DDL に COLLATE 指定がなかったこと（本節改訂前の実装。SqlServerLogStore.cs の DDL・自由文検索コメントが自認していた暫定状態）が原因だった
- **採用する照合順序: `Latin1_General_100_CI_AS_KS_WS_SC`**（Windows 照合順序・version 100・大文字小文字非区別 `CI`・アクセント区別 `AS`・かな種区別 `KS`・全角/半角区別 `WS`・補助文字対応 `SC`）
  - **`CI` 以外の感度をすべて「区別する」側に明示するのは、§1.2 の規則「折り畳むのは大文字小文字のみ」を照合順序レベルで正確に実装するため**である。Windows 照合順序は `KS` を省略すると**かな種非区別**（ひらがな ≒ カタカナ）、`WS` を省略すると**全角/半角非区別**になる（Microsoft Learn "Windows collation name (Transact-SQL)" — https://learn.microsoft.com/sql/t-sql/statements/windows-collation-name-transact-sql: 「Omitting this option specifies kanatype-insensitive」「Omitting this option specifies width-insensitive」。確認日 2026-07-09）。つまり一般に推奨される `Latin1_General_100_CI_AS_SC`（KS/WS なし）をそのまま使うと、SQL Server だけが大文字小文字**以外**（かな種・全角/半角）まで同一視し、SQLite（かな種・全角/半角を区別する）との新たな provider 差を作り込む。`KS`・`WS` の明示でこの過剰な折り畳みを止める
  - `CI_AS_KS_WS` の組合せは Windows 照合順序の正規の構文（同文書の Syntax: `<CaseSensitivity>_<AccentSensitivity> [_<KanatypeSensitive>] [_<WidthSensitive>]`）であり、`_KS_WS_SC` を併記した照合順序の実例も公式ドキュメントに載る（Microsoft Learn "Collation and Unicode Support" — https://learn.microsoft.com/sql/relational-databases/collations/collation-and-unicode-support の例 `Japanese_Bushu_Kakusu_100_CS_AS_KS_WS_SC_UTF8`。確認日 2026-07-09）。**`Latin1_General_100_CI_AS_KS_WS_SC` が配備先インスタンスの `sys.fn_helpcollations()` に実在することの確認を、実装バッチのスキーマ管理（接続検証）に含める**
  - Windows 照合順序の大小比較は Windows システムロケールの規則に基づき、Unicode データにも同じアルゴリズムを適用する（同 "Collation and Unicode Support": 「Windows collations define rules for storing character data that are based on an associated Windows system locale... you can implement a comparison of non-Unicode data by using the same algorithm as that for Unicode data」。確認日 2026-07-09）——`CI` の折り畳みは ASCII に限られない
  - `SC`（補助文字対応）により `nvarchar` 列で全 Unicode 範囲（U+000000–U+10FFFF）を比較対象にできる（同文書: 「SQL Server 2012 (11.x) introduced a new family of supplementary character (_SC) collations that can be used with the nchar, nvarchar, and sql_variant data types to represent the full Unicode character range」。確認日 2026-07-09）。`_UTF8` 系照合順序は付与しない——UTF-8 エンコーディングは `char`/`varchar` 列向けの機能であり、本スキーマの文字列列はすべて `nvarchar`（UCS-2/UTF-16）のため効果がない（同文書のデータ型節）
  - **サーバ既定からの意図的な乖離**: Microsoft Learn は互換性照合順序（`SQL_Latin1_General_CP1_CI_AS` 等）から Windows 照合順序への切り替えを推奨している（「To take full advantage of SQL Server features, change the default installation settings to use Windows collations」。確認日 2026-07-09）。Yagura はこの推奨に従いつつ KS/WS を追加し、**サーバの既定照合順序が何であっても列単位指定で上書きする**——Issue #146 の症状（配備先次第で挙動が変わる）を構造的に解消する
- **適用列**: `dbo.LogRecords` の全 `NVARCHAR` 列（SourceAddress・Hostname・AppName・ProcId・MsgId・StructuredData・Message）と `dbo.SystemEvents` の Kind・Details。**自由文検索の対象は Message のみ（§1.2）だが、同一テーブル内で列ごとに照合順序が食い違うと、比較・結合を伴うクエリで「照合順序の競合」エラーを誘発しうるため、列を個別に選別せずテーブル単位で統一する**
- **スキーマ移行への申し送り**（Issue #145（絞り込み列の複合索引追加）・#147（列長の provider 間統一）と**同一のスキーマ移行バッチ**で実装する。3 件とも `ALTER TABLE` を伴う変更であり、メンテナンスの機会を 1 回にまとめる。**実装状況: スキーマ v1→v2 移行として実装済み**——新規 DDL は v2 形状で直接作成し、既存 v1 デプロイは `InitializeAsync`（スキーマ管理契約）が起動時に移行する。大規模 DB での所要時間・ロック挙動・受信継続性の見積もりは DB-10 の実機検証として残る）:
  - **DDL**: 新規作成の `CREATE TABLE` は列定義へ `COLLATE Latin1_General_100_CI_AS_KS_WS_SC` を追加する。既存デプロイへは `ALTER TABLE ... ALTER COLUMN ... COLLATE ...` で適用し、当該列に索引がある場合は再構築を伴う
  - **受信継続性**: 移行中に到着したログはスプール（architecture.md §3.2——書き込みが確定できない状態そのものの受け皿）が受け、**ログを損失させないことを移行の要件とする**。書き込み可用性が中断する幅（`NVARCHAR(MAX)` の Message 列を含む size-of-data 変更のロック・所要時間）は実機検証（[DB-10](#8-確定待ち一覧)）で見積もり、長時間の中断が不可避なら実行タイミングの利用者告知（[§9](#9-利用者向けドキュメントへの申し送り) DB-D5）を要件に含める
  - **「損失させない」をスプール容量計画に接続する**: 上記要件が成立する条件は、移行中の流入をスプール容量が吸収し切ることである（容量上限到達時の新規破棄——architecture.md §3.2.3——と正面から緊張する）。architecture.md M-12（スプール容量上限の既定値）が本番昇格切替について「切替中に受ける流入 × 想定切替時間」を容量算定の入力に含めるのと同じ扱いで、**DB-10 で実測した移行所要時間 × 想定流量を M-12 の容量算定の入力に含める**。吸収し切れない見積りになる場合の代替手段（移行前の一時的なスプール容量拡張、または §3 の分割削除に相当する移行の分割実行）を実装バッチで用意する——実行タイミングの選択（低トラフィック帯の指定）が効かない 24 時間運用の環境を想定条件に含める
  - **冪等性**: §1.2 スキーマ管理の冪等性要件をこの移行にも適用する——現在の列照合順序を（`sys.columns` 等で）確認し、適用済みなら何もしない。途中失敗からの再実行を安全にする
  - **失敗の扱い**: §5.2 の「自走可能な失敗」の枠組みを適用するが、**原因種別で提示内容を書き分ける**（実装バッチへの申し送り）——権限不足には「不足内容 + 実行可能な SQL の提示」がそのまま当てはまる。一方、**容量不足は「実行可能な SQL」では解決しない物理的対処**（空き領域の確保・増設・保持期間の短縮）であるため、「不足内容の明示 + 対処の案内」の形に書き分ける（§5.2 の原型は権限不足向けの設計であり、そのままの転用は成立しない）
  - **観測性**: 移行の実行中・進捗・所要時間をゲージ・能動通知（architecture.md §4.6）へ露出し、**適用したスキーマ版と適用日時を事後に問い合わせ可能な形で保持する**（検索の意味論が変わる変更が「いつ効き始めたか」に監査上答えるため。**適用版・適用日時の保持は両 provider の `SchemaMigrationHistory` 表として実装済み**——版ごとに 1 行を追記し、再実行では上書きしない）。移行の進捗のゲージ露出と、**スキーマ移行の実行を ADR-0004 決定 7 の監査記録対象に含めるか**は、DB-10（移行の実行方式とタイミングの確定）と合わせて確定する
  - **コード整合**: [LogQuery.cs](../../src/Yagura.Storage/LogQuery.cs)・[SqlServerLogStore.cs](../../src/Yagura.Storage/SqlServer/SqlServerLogStore.cs)・[SqliteLogStore.cs](../../src/Yagura.Storage/Sqlite/SqliteLogStore.cs) に残る「DB-6 確定までの暫定規則」コメントの更新（規則確定済みの旨への書き換え）を同バッチに含める（反映済み）
- **却下した代替案**:
  - **既定（COLLATE 未指定）のまま**: Issue #146 の不具合そのもの。配備先の OS ロケールに挙動が依存し、provider 契約（§1.2）の「特定 provider の挙動を暗黙の基準にしない」に反する。却下
  - **`SQL_Latin1_General_CP1_CI_AS`（互換性照合順序）を明示指定**: 配備依存は解消するが、レガシーな SQL Server 照合順序であり Unicode 比較の挙動保証が弱い（Microsoft Learn が Windows 照合順序への切り替えを推奨する理由——上記引用）。却下
  - **`Latin1_General_100_CI_AS_SC`（KS/WS なし。Microsoft の一般推奨そのまま）**: 本改訂の初稿が採っていた案。かな種・全角/半角まで同一視するため「折り畳むのは大文字小文字のみ」という §1.2 の規則を超過し、SQLite との差を大文字小文字以外の次元で新設する。ペルソナレビュー（PR #168）の指摘を受けて KS/WS 明示へ改めた。却下
  - **日本語照合順序（`Japanese_XJIS_140_CI_AS_KS_WS` 等）**: 照合順序指定子（`Latin1_General` / `Japanese_XJIS` 等）が定めるのは辞書順ソートの規則と `varchar` 用コードページである（Microsoft Learn "Windows collation name": 「The base collation rules cover: The sorting and comparison rules that are applied when dictionary sorting is specified... The code page used to store varchar data」。確認日 2026-07-09）。Yagura は文字列列で `ORDER BY` しない（並びは常に ReceivedAt——§2.2）ため辞書順の言語差は使われず、列はすべて `nvarchar` のためコードページ差も効かない。DB-6 の要件（大文字小文字のみの折り畳み）は `CI_AS_KS_WS` の感度指定で決まり、これは両系列で同一に指定できる——したがって日本語系列を選ぶ積極的理由が現時点でない一方、ロケール固有の系列を選ぶと「なぜこのロケールか」の説明責任を全配備先に対して負う。汎用系列（Latin1_General_100）を既定とし、**日本語ロケール固有の大小折り畳み差が適合テスト（非 ASCII 大小ケース）で実際に検出された場合に限り、実測を根拠に再評価する**。却下（条件付き——実測での再開扉を残す）
  - **大文字小文字を区別する方向へ統一（両 provider を CS 化）**: 現行の暫定規則・実務要求（Issue #146 の根拠——「error」で ERROR/Error を拾えることが自由文検索の根幹）に反する。却下
  - **ASCII 限定の折り畳みで統一する（SQLite の現状挙動に両者を揃える）**: この案の利点は SQLite 側を一切変更せずに済み、provider 間一致が「目標」でなく構造的に成立することにある。しかし Windows 照合順序の比較スタイルで大文字小文字の扱いは `CI` / `CS` の二値であり、折り畳みの適用範囲を ASCII に限定する修飾子は構文上存在しない（Microsoft Learn "Windows collation name (Transact-SQL)" — https://learn.microsoft.com/sql/t-sql/statements/windows-collation-name-transact-sql の Syntax: `<CaseSensitivity>` は `CI` | `CS` のみ。確認日 2026-07-09）——ASCII 限定の折り畳みを実現するには SQL Server 側にこそ独自機構（CS/BIN2 照合 + アプリ層での折り畳み等）が要る。**つまり「新規機構ゼロ」という見かけの利点は成立せず、独自機構の置き場所が SQLite から SQL Server へ移るだけ**である。さらに Issue #146 の再現例（CAFÉ/café）を「両者ともヒットしない」側で恒久化し、利用者の期待（大文字小文字を区別しない検索）を非 ASCII で裏切り続ける。機構コストが対称である以上、期待に沿う意味論を選ぶ。却下

## 6. 本番昇格（SQLite → SQL Server）

ADR-0002 決定 3 の昇格支援 3 点 (i)〜(iii) のうち、(i) 案内は architecture.md §5.3、(ii)(iii) を本節で定める。

### 6.1 接続切替（v0.1。昇格 (ii)）

- **実装参照（M8-4 / Issue #71 の骨格）**: 本番昇格ウィザード（ui.md §4 の `/admin/promotion`。契約 `IPromotionWizardService` / 実体 `Yagura.Host.Administration.PromotionWizardService`）が「準備フェーズ（接続検証——`ISqlServerConnectionValidator` 抽象により SQL Server 不在の開発機でも経路をテストできる）→ 切替確定前の予告 → 旧 DB ファイル処分の選択 → 切替実行（冪等トークンつき。監査対象 = security.md §4.3 の 2002/2003）」の経路を実装した。**M8-4 時点の切替実行の実効は「検証済み接続の設定保存 + サービス再起動での反映」であり、下記の切替本番手順①〜④（無瞬断切替）と旧 DB ファイルの実処分は後続 Issue の実装対象である**（本節が定める設計はこの骨格で変更しない）
- **準備フェーズの入力と検証失敗の扱い（骨格の UX 完成。2026-07-06 オーナー合意のモックアップの実装）**:
  - **接続は項目で入力し、接続文字列はサーバ側で組み立てる**: サーバ名・データベース名・認証方式（**Windows 統合認証を既定** = §5.1 の第一推奨。選択時は**接続に使うアカウント = サービスの実行アカウント名を画面に表示する**——SQL Server 側にどの名前で見えるかを推測させない）・SQL Server 認証時のユーザー名とパスワード・サーバ証明書の信頼（`TrustServerCertificate`。既定は信頼しない。チェックボックス自体に「暗号化は維持・証明書の検証のみ省略・なりすまし検知は行われない」の説明を常設する——自己署名証明書の閉域環境向けの選択であることを失敗前から読める形にする）。上級者向けに**接続文字列の直接入力を併存**する（項目が網羅しない接続オプションの逃げ道。既定は項目入力）
  - **パスワードは入力方式によらずマスク付きの専用欄からのみ入力する**: 直接入力欄のパスワード系キーは検証エラーとして拒否する。**拒否判定は文字列走査ではなく `SqlConnectionStringBuilder` で正規化パースした上で行う**（`Pwd` 別名・大文字小文字・空白ゆれを素通りさせない。単体テストで固定する）——画面・snapshot・設定画面の確認表示のいずれにもパスワードが平文で現れる経路を作らない（configuration.md §5。適用時の保存は従来どおり DPAPI 暗号化表現のみ）
  - **検証失敗は原因で分類し、原因別の次の一手を画面に示す**（分類の根拠 = Microsoft Learn 公式ドキュメント確認 2026-07-07）: ①**サーバ証明書が信頼されない**（SChannel `SEC_E_UNTRUSTED_ROOT` 0x80090325）→「サーバ証明書を信頼する」の有効化を案内、②**サーバへ到達できない**（タイムアウト・名前解決不能・ポート閉塞）→ サーバ名・ポート・ファイアウォールの確認を案内（修復 SQL なし）、③**ログイン失敗**（SQL Server エラー 18456）→ **18456 は誤パスワードでも DB 不在（state 38/46——クライアントへ返る情報は意図的に詳細を隠す）でも返るため、案内は「まずユーザー名・パスワードを再確認 → ログインが未作成の場合は次の SQL」の条件付き提示とする**、④**データベース不在**（同 4060）→ DB 作成を含む SQL を提示。③④は §5.2 の「実行可能な SQL」（ログイン作成・DB 作成・権限付与。環境の実値入り・SQL 認証のパスワードはプレースホルダ）をコピー可能に提示する。**分類できない失敗は生のエラーメッセージ + 汎用案内に留め、修復 SQL を断定提示しない**（誤分類の被害を画面内で閉じる安全側）。**提示のみでありサーバは実行しない**（§5.2 の統治——DBA 依頼文としての流通を前提とする）。監査記録（2002）には SQL 原文を残さず、失敗の分類値（分類不能を含む）と証明書信頼の選択値を付す
  - **v0.1 の提示 SQL は対象データベースの `db_owner` を付与する**（昇格・スキーマ作成を含む導入時権限として。実行時最小権限への分離——§5.2 の完全形——は後続 Issue の課題のまま変えない）。**提示 SQL 内のコメントで「導入時権限であり、実行時最小権限への分離で縮小予定」の暫定性を明記する**（依頼文として流通した先の DBA にも伝わる形にする）
  - **退避の選択時は退避先（絶対パス）の指定を必須とする**（下記「選択内容・退避先・完了の事実は監査記録の対象」の入力面。本改善での検証は形式——絶対パスであること——まで。**存在・書込可否の事前検証は旧 DB ファイルの実処分の実装（後続 Issue）の要件に含める**——準備フェーズで通った退避が本番で失敗する穴を後続で塞ぐ）
- **準備と本番を分離する**: 切替は「準備フェーズ」（接続・権限・スキーマの事前検証。§5.2 の SQL 提示 → DBA 依頼待ちのような長時間の停滞はここで吸収する）と「切替本番」（事前検証済みの接続への差し替え。短時間で完了する）に分ける。準備フェーズは何度でも中断・再開でき、本番はすべての事前検証が通ってから実行する
- **切替本番の手順**: ①旧 provider の書き込み停止 → ②システムイベントの複写（下記）→ ③新 provider へ差し替え（**この時点をもって切替の確定とする**）→ ④スプール分の drain（新 DB へ）。この間に到着したログは Q2 とスプール（architecture.md §3.2.1——書き込みが確定できない状態そのもの）が受ける。**切替以降の新規ログは新 DB へ**保存される
- **切替中の可視化と打ち切り**: 切替中であることと経過時間をゲージ・能動通知（architecture.md §4.6）に露出する。**打ち切り監視（[DB-7](#8-確定待ち一覧)）の区間は手順①〜③、すなわち差し替えの確定まで**とする——確定までに既定時間を超えた場合は**自動で旧 provider へ戻す**（戻った場合、スプール退避分の drain 先は旧 DB である——差し替え前なので通常経路のまま）。手順④の drain の長時間化は巻き戻しの対象ではなく、通常の drain 規則（architecture.md §3.2.2）で扱う
- **差し替え確定後の障害は昇格の失敗として扱わない**: 差し替え（手順③）以降に新 DB への書き込みが失敗し始めた場合は、巻き戻さず通常の障害経路（リトライ・スプール退避）で扱う——新 DB が正であり、旧へ戻すと新 DB 側のコミット済みログが管理外の複製になる（ADR-0004 決定 5 が嫌う状態）。事前検証の分離が、この局面に至るリスク自体を縮める
- **システムイベントは切替処理が新 DB へ複写する**: 受信断区間等（§2.3）はログ本体より件数が桁違いに少なく、蓄積ログ移行 (iii) を待たずに複写できる。「ログ本体と同じ以上の保持」（§2.3）を昇格が破らないようにするため、複写は手順②——**書き込み停止後・差し替え前**——に行う（停止後なので取り残しは生じない）。**複写の失敗は巻き戻しの対象**（まだ確定前であり安全に戻せる）。巻き戻し後の新 DB に複写済みイベントが残留し、再切替で重複し得ることは、システムイベントにも適用される at-least-once の範囲として許容する（切替自体の監査記録はホストの管理操作経路であり、この複写とは独立）
- **蓄積ログは移行機能 (iii) の提供まで Web UI から参照できなくなる**。ウィザードは**切替の確定前に**この帰結を利用者へ予告する（文言は ui.md）。参照が必要な場合の選択肢は「退避 + 手動移行手順（operations.md）」である
- **旧・組み込み DB ファイルは残置しない**（ADR-0004 決定 5）。切替完了時にウィザードが「明示の退避（利用者が指定する場所への移動）」か「削除」を選択させる:
  - **退避時は移動先へ ADR-0004 決定 5 相当の ACL を適用する**（適用できない場所なら警告する）。退避完了をもって保護責任は利用者へ移転する——この移転は利用者向けドキュメントに明記する（[§9](#9-利用者向けドキュメントへの申し送り) DB-D4）
  - 処分（退避 / 削除）が完了するまで昇格は「未完了」状態であり、**未完了の間も新 DB への保存は継続する**。未完了は UI の常時表示と能動通知で可視化し、ウィザードへ再入すると処分ステップから再開する（電話で中断されても迷子にならない）
  - 退避 / 削除の**選択内容・退避先・完了の事実は監査記録の対象**とする（切替本体と同様。ADR-0004 決定 7）
- 切替は管理操作であり監査記録の対象（ADR-0004 決定 7）

### 6.2 蓄積ログの移行（後続バージョン。昇格 (iii)）

- 蓄積済みログの移行機能は**それ単体で提案書 1 本分の規模**があり、時期と方式は個別に確定する（ADR-0002 決定 3 のとおり。着手判断の入力は [DB-5](#8-確定待ち一覧)）
- ただし**満たすべき要件は先に固定する**（提案書はこの要件を出発点にする）:
  1. 移行中も受信を止めない（新規ログの保存を移行処理が妨げない）
  2. **完全性の検証を移行機能自体が持つ**: 移行元・移行先の件数突合を実行し、結果を記録する。突合の軸は ReceivedAt の範囲とし、**合格条件は at-least-once と整合させる**——「移行先件数 ≥ 移行元件数、かつ差分のレコードを抽出して重複として説明できること」（件数の完全一致を合格条件にしない——重複と損失の相殺を隠すため）
  3. 中断・再開が安全（冪等または再開点の記録。at-least-once の意味論と整合）
  4. **ReceivedAt を再刻印しない**（§2.2 の契約。移行後も時刻・保持期間の起算は変わらない）
  5. **移行由来のレコードは投入経路を事後に識別できる**（監査で「このレコードはいつ・どの経路で入ったか」に答える。重複調査にも使う。識別の持たせ方は移行設計時に確定）
  6. 移行完了後の旧データの扱いは 6.1 と同じ原則（残置しない）
- 移行の読み出しは §1.2「契約拡張の予約」の一括読み出し操作として追加する（契約外の裏口 SQL で実装しない）
- 提供までの間は**手動移行手順を文書で提供する**（operations.md 起筆時に作成 [§9](#9-利用者向けドキュメントへの申し送り) DB-D1）。手順には要件 2 と同じ判定基準の件数突合を含め、**移行実施の事実（日時・移行元/先件数・実施者）を製品の監査記録に残すステップを置く**（手動移行が製品の証跡から断絶しないようにする。記録手段はセキュリティ設計文書の監査記録仕様と合同で確定する）

## 7. v1.0 で凍結する対象（ADR-0006 基準 4 の一覧）

v1.0 以降、後方互換を破る変更を MAJOR でのみ行う対象の一覧。**本節が ADR-0006 基準 4 の「凍結対象の一覧」の置き場である**:

| 凍結対象 | 定義場所 |
|---|---|
| DB 論理スキーマ（§2。ログレコード・システムイベントの項目と意味） | 本文書 |
| provider 契約（§1.2 の操作と意味論） | 本文書 |
| スプールのセグメント形式（未消化セグメントを残した版間更新が成立すること） | architecture.md §3.2.1 |
| メタデータ領域の形式（カウンタ・稼働記録の継続性） | architecture.md §4.3 |
| 設定スキーマ | [configuration.md](configuration.md) §8 |
| Windows イベントログのイベント ID の意味（additive-only。ID の追加は MINOR 可・意味の変更/転用は不可） | [security.md](security.md) §4.3 |

- **provider 契約の「凍結」の意味を精密化する**（契約の消費者は実装者であり、利用者向け互換とは破壊の意味が異なる）: 凍結されるのは**既存操作の意味論とスキーマ項目の意味**である。**契約への操作追加は optional 操作（capability 検出——provider が実装有無を宣言し、ホストが検出する機構）として MINOR で行える**。既存操作の意味論変更・必須操作の追加は MAJOR とする。§1.2 の「契約拡張の予約」（一括読み出し・集計）はこの optional 機構で追加する
- v0.x の間はこれらの互換を破る変更を許容するが、conventions.md の deprecation 階段（1 バージョン警告 → 次バージョン削除）は v0.x でも守る
- v0.x → v1.0 の更新で蓄積データと設定が保全されることの実地実証は ADR-0006 基準 4 が要求する（本表がその検証範囲の定義になる）
