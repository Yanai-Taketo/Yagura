## 1. デザインシステムの原則

- **すべての画面はデザイントークン（§2）と共通コンポーネント（§3）だけで組む**。ページ固有の CSS・JavaScript・インラインスタイル（`Style=` 属性）は原則禁止（ADR-0003 決定 2）
- **例外の手続き**: ページ固有であるべき理由を PR に明記し、[§11 ページ固有 CSS 例外一覧](#11-ページ固有-css-例外一覧) に登録した上で認める。例外は共通化の候補であり、共通コンポーネントへ吸収されたら一覧から消し込む
- **MudBlazor の低レベル API をページに散乱させない**（ADR-0003 決定 3 の緩和策）: 頻出パターンは共通コンポーネントへ集約し、ページは共通コンポーネントを組み合わせる。MudBlazor を直接使ってよいのは共通コンポーネントの実装内部と、単純な静的レイアウト部品（コンテナ・グリッド等）に限る
- **色・フォント・余白の直接指定を禁止する**: すべてトークン経由で参照し、テーマ切替（§2.4）で破綻しない構造を保つ（ADR-0003 決定 4）

## 2. デザイントークン

### 2.1 配色

方向性は「クリーンな業務系・ライト基調」（ADR-0003 決定 4）。値は §8 のコントラスト検証（UI-1。M8-2 / Issue #69 で実施済み——§8 の検証記録参照。ライトの state-warning / state-info は初期値が未達だったため調整済み）を通して確定した。**トークン値は本文書を正とし、実装のテーマ定義を変更する PR は本表の更新を同じ PR に含める**（configuration.md §8 と同型の同期規約）。

| トークン | ライト | ダーク | 用途 |
|---|---|---|---|
| primary | #1565C0 | #90CAF9 | 主要アクション・リンク・選択状態 |
| background | #F5F7FA | #121212 | 画面背景 |
| surface | #FFFFFF | #1E1E1E | カード・テーブル・ダイアログの面 |
| text-primary | #1A1A1A | #E0E0E0 | 本文 |

| divider | #D7DDE4 | #3A3A3A | 罫線・区切り |

**状態色**（意味を固定し、これ以外の意味で使わない）:

| トークン | ライト | ダーク | 意味 |
|---|---|---|---|
| state-ok | #2E7D32 | #81C784 | 正常・成功（受信中・保存成功） |
| state-warning | #A05F00 | #FFB74D | 警告・要注意（一時保管への退避中・上限接近・縮退運転）。ライトは UI-1 検証で #B26A00 から調整（§8） |
| state-error | #C62828 | #E57373 | 異常・失敗（受信断・書込失敗・破棄発生） |

- **色だけで意味を伝えない**: 状態は常にアイコンまたはテキストを伴う（色覚特性への配慮。§8）
- severity（syslog の重大度）の表示色は状態色と別系統のトークンとして実装設計で定義する（Error 以上 = state-error 系、Warning = state-warning 系に揃える）
- **実装形態**: トークンは MudBlazor のテーマ機構（テーマ定義のパレット）への割当を第一経路とし、独自の CSS 変数の併設は共通コンポーネントの実装内部に限る。具体の割当は §2.1.1 のとおり確定した（UI-8。M8-1 / Issue #68）

#### 2.1.1 MudBlazor テーマへの割当（UI-8 確定。2026-07-06）

実装側の唯一のテーマ定義は `src/Yagura.Web/Theming/YaguraTheme.cs`（MudTheme）。トークンは PaletteLight / PaletteDark の両方へ同名項目で割り当てる。割当が実際に CSS 変数（`--mud-palette-*`）として描画されることは実サーバ起動 + 描画 HTML の走査で確認済み（例: `--mud-palette-primary: rgba(21,101,192,1)` = #1565C0。確認日 2026-07-06）。

| トークン | MudBlazor パレット項目 |
|---|---|
| primary | Primary |
| background | Background |
| surface | Surface |
| text-primary | TextPrimary |
| text-secondary | TextSecondary |
| divider | Divider・LinesDefault・TableLines（罫線系 3 項目へ同値割当） |
| state-ok | Success |
| state-warning | Warning |
| state-error | Error |
| state-info | Info |

派生割当（独立トークンを増やさず、既存トークンの値を流用する項目）:

- AppbarBackground / DrawerBackground ← surface、AppbarText / DrawerText ← text-primary（共通骨格の面は「カードと同格の面」として扱う）
- PrimaryContrastText と各状態色の ContrastText（塗り面上の文字色）← ライトは #FFFFFF 固定、ダークは text-primary のライト値 #1A1A1A（ダークの状態色は明色のため文字は暗色）。コントラスト充足の検証は UI-1 の対象に含める
- 上表・派生割当にない MudBlazor パレット項目（Secondary・Tertiary 等）は MudBlazor 既定値のまま残るが、**トークンに存在しない色のため画面・共通コンポーネントから使用してはならない**（§1 の直接指定禁止と同じ扱い。§3.2 受け入れ条件①の「参照はテーマ由来の値とトークンに限られる」はこの限定を含む）

### 2.2 タイポグラフィ

- フォントは OS 提供のフォントスタックとし、Web フォントを同梱・参照しない（閉域完結・ADR-0003 決定 1 の npm/CDN 非依存と同じ向き）: `"Yu Gothic UI", "Segoe UI", Meiryo, sans-serif`。等幅（ログ本文・生データ表示）: `Consolas, "Courier New", monospace`
- **非 Windows の閲覧クライアント**（閲覧リスナは LAN 公開であり、閲覧側の OS は保証されない）では総称ファミリへのフォールバックとなることを許容する
- **フォントスタックの実在検証（UI-9 確定。2026-07-06）**: Windows 11（10.0.26200）の `System.Drawing.Text.InstalledFontCollection` で "Yu Gothic UI"・"Segoe UI"・"Meiryo"・"Consolas"・"Courier New" の 5 指名フォントすべての実在を確認した。非 Windows クライアントは指名フォントが解決されず末尾の総称ファミリ（sans-serif / monospace）へフォールバックする（前項の許容のとおり。総称ファミリの表示品質は閲覧側 OS の既定に委ね、個別検証はしない）。MudBlazor 9.6.0 の同梱 CSS はフォント名を一切ハードコードせず、すべてテーマ由来の CSS 変数（`--mud-typography-*-family`）を参照するため（実ファイル走査で Roboto 等の出現 0 件。§10）、テーマ定義のフォントスタック設定がそのまま全域に効く。MudBlazor 公式ドキュメントが案内する Google Fonts（Roboto）の `<link>` は追加していない（本節の「Web フォントを参照しない」の受け入れ検証は §10 ②が兼ねる）。描画 HTML に `--mud-typography-default-family: 'Yu Gothic UI', 'Segoe UI', Meiryo, sans-serif` が出力され、Roboto への参照が 0 件であることを実サーバで確認済み
- サイズ段階（px）: 12（注釈）/ 14（本文・既定）/ 16（強調・小見出し）/ 20（画面見出し）/ 28（数値の大表示——ダッシュボードのカウンタ等）。行間は 1.5 を基準とする
- **サイズ段階の MudBlazor Typo への割当（UI-9 の一部として M8-1 で確定）**: caption・overline = 12 / Default・body1・body2・subtitle2・button = 14 / h6・subtitle1 = 16 / h5 = 20 / h4 = 28。**h1〜h3 は h5 と同値（20）に固定**し、トークン外サイズの混入経路を塞ぐ（画面見出しは h5 を標準とする）。等幅スタック（ログ本文・生データ表示）は MudBlazor Typography に対応枠がないため、ログ表示系の共通コンポーネントの実装内部で適用する（§2.1 実装形態の「独自 CSS 変数の併設は共通コンポーネントの実装内部に限る」の範囲）

### 2.3 余白・形状

- 余白は **4px 基準のスケール**（4 / 8 / 12 / 16 / 24 / 32）のみ使用する
- 角丸は 4px（入力・ボタン）と 8px（カード・ダイアログ）の 2 値。影は「なし（フラット）/ カード / ダイアログ」の 3 段のみ使う
- **影 3 段の elevation 割当（UI-9 確定。2026-07-06）**: なし = elevation 0 / カード = elevation 1（MudPaper・MudCard の既定値であることを MudBlazor 9.6.0 の API ドキュメント（同梱 XML）で確認）/ ダイアログ = MudDialog の内蔵影（同梱 CSS の `.mud-dialog` が Material elevation 24 相当の `box-shadow` を直接保持することを実ファイルで確認。テーマの Shadows 配列では変更されない）。3 段の意味論は「フラット / 面 / 浮遊」であり、**浮遊系の部品（メニュー・ポップオーバー等。MudBlazor 既定 elevation 8）は「ダイアログ」段（浮遊）に属するものとして部品既定値を許容する**。ページ・共通コンポーネントが `Elevation` を明示指定してよい値は 0 と 1 のみ。テーマの Shadows 配列自体は MudBlazor 既定（Material 26 段）のまま使い、利用面をこの規約で 3 段に限定する
- **角丸 2 値の制約と方式（UI-9 で判明・確定。2026-07-06）**: MudBlazor のテーマ機構が持つ角丸は 1 値（`LayoutProperties.DefaultBorderRadius` → `--mud-default-borderradius`）のみで、本節の 2 値をテーマだけでは表現できない。割当: テーマ既定 = **4px**（入力・ボタン。描画 HTML への出力を実サーバで確認済み）。カード・ダイアログの 8px は、カード / ダイアログの共通コンポーネントの実装内部で適用する（§2.1 実装形態の許容範囲）——**M8-3 で適用済み**: カード = `YaguraCard`（`.yagura-card` 規則）、ダイアログ = 共通コンポーネント内部スタイル（`yagura-components.css` の `.mud-dialog` 規則。ダイアログはポータル描画のため部品側のラッパクラスでは面に届かず、ダイアログ段の形状規約として同スタイルシートで適用する）

### 2.4 テーマ

- 既定はライト。**ダークは同格のテーマ**であり、全トークンが両テーマの値を持つ（ライトの機械変換で作らない。ADR-0003 決定 4——ダッシュボードの暗所掲示運用を想定ユースケースに含む）
- 切替は UI 上のトグルで行い、**選択はブラウザ側（端末ごと）に保存する。サーバへは書き込まない**——閲覧リスナは認証なしで閲覧者の概念を持たず（ADR-0004 決定 2）、書き込みエンドポイントも持たない（§4）。端末ごとの保存は「掲示用の大型モニタはダーク・手元 PC はライト」の使い分けとも整合する
- OS 設定への追従は選択できる項目とし、既定では手動選択を尊重する（勝手に切り替わらない）
- **OS テーマ追従の JS 手書きなし実現可否（UI-9 確定: 可。2026-07-06）**: MudBlazor 9.6.0 の `MudThemeProvider` が `GetSystemDarkModeAsync` / `WatchSystemDarkModeAsync` を提供し、同梱 JS（MudBlazor.min.js）内の `matchMedia("(prefers-color-scheme: dark)")` で実現されていることを実ファイルと API ドキュメント（同梱 XML）で確認した。手書き JS の追加なしで追従を実装できる（ADR-0003 決定 1 と両立。方式の再設計は不要）。`MudThemeProvider` の自動追従（`ObserveSystemDarkModeChange`。既定 true）は**無効化して使う**——既定の手動選択尊重と衝突するため、追従は利用者が「OS の設定に合わせる」を選択した場合に限り `WatchSystemDarkModeAsync` の購読で行う
- **切替 UI と保存の実装（M8-1 で確定）**: 共通骨格（MainLayout）のアプリバー右端のメニューで「ライト / ダーク / OS の設定に合わせる」の 3 択。既定はライト。選択はブラウザの localStorage（キー `yagura.theme-mode`）へ保存する——標準 API の呼び出しは `IJSRuntime` 経由で行い、手書き JS ファイルは追加しない（ADR-0003 決定 1 の範囲内。§5.2 のような例外登録は不要）。保存済み選択の復元は circuit 確立後（prerender 中は JS 不可）のため、ダーク保存端末では初回表示の一瞬だけ既定のライトが見える——既定がライトである以上、v0.1 はこれを許容する（暗所掲示運用で問題が反復するなら再評価）

## 3. 共通コンポーネント規約

### 3.1 標準形（v0.1 の初期セット）

実装参照は M8-2（Issue #69）で実装した共通コンポーネント（`src/Yagura.Web/Components/Common/`）。表示文言は同ディレクトリの `UiText`（§7 の文言カタログ）に集約する。

| コンポーネント | 規約 | 実装参照 |
|---|---|---|
| ボタン | 主ボタン（primary 塗り）は 1 ビューに 1 つまで（**ビュー = 同時に視認される単位。ページ本体・ダイアログ・ウィザードの 1 ステップはそれぞれ別のビュー**）。破壊的操作（削除・昇格確定等）は state-error 系 + 確認ダイアログ必須 | `YaguraButton`（Role=Destructive は確認ダイアログ表示を内蔵し、確認内容なしでは実装エラーになる） |
| フォーム | ラベルは入力の上。検証エラーは項目直下に文言で表示（色だけにしない）。必須項目は「必須」表記（記号 * だけにしない） | `YaguraFormField`（共通枠）/ `YaguraTextField`（1 行テキスト入力）/ `YaguraSelectField`（選択。ネイティブ `select` をトークンで整形——「指定なし」を常に選べる）/ `YaguraDateTimeField`（日時。ネイティブ `datetime-local`。M8-3 で追加） |

| 通知（トースト） | 一時通知は自動で消えるが、**state-error の通知は手動で閉じるまで残る**（見逃し防止）。「閉じる」はその閲覧者の画面からのみ消えるセッションローカル操作であり、サーバ状態を変更しない。通知履歴は状態画面から辿れる | `IYaguraNotifier` / `YaguraSnackbarNotifier`（ページは ISnackbar を直接使わない。通知履歴は状態画面 = M8-3） |
| 空状態 | 「データがない」画面は、無地ではなく**次に取るべき行動**を必ず示す。特にログ未着の空状態は、**このサーバの受信先情報（IP アドレス・ポート番号）をコピーできる形で表示し**、届かないときの切り分け文書（configuration.md CF-D5）への導線を置く（インストーラが閉じた 30 分動線の続き——機器設定 → 最初の 1 件——を空状態画面が引き受ける） | `YaguraEmptyState`（NextAction 必須——無地の空状態を構造上作れない）+ `YaguraCopyField`（コピー可能な受信先表示） |
| 確認ダイアログ | 取り消せない操作は、何が起きるかの要約 + 明示的な確認ボタンで構成する。既定フォーカスは安全側（キャンセル）に置く | `YaguraConfirmDialog`（Esc で閉じられる = キャンセル） |
| 状態帯（ヘルスバー） | ダッシュボード・全画面共通ヘッダに置く現在状態の帯（§5.1）。state-ok / warning / error の 3 状態 + 一言サマリ | `YaguraStatusBand` |
| カード（M8-3） | カード面の共通部品。角丸 8px（§2.3）と影のカード段（elevation 1）を実装内部で適用する。ページは MudPaper を直接使わない | `YaguraCard` |
| 数値の大表示（M8-3) | ダッシュボード・状態画面のカウンタ/ゲージ現在値。数値は 28px 段（§2.2）。整形（単位・桁区切り）は利用側 | `YaguraStatCard` |
| 時間軸チャート（M8-3） | 受信量の推移（棒）+ 受信断区間（帯の重ね描き。§5.3）。外部チャートライブラリなしの手書き SVG。色はトークン由来の CSS 変数のみ・受信断は色 + 凡例文言のセット（§2.1・§8）。近似断点は薄い帯で区別 | `YaguraTimelineChart`（入力型 `YaguraTimelineBucket` / `YaguraTimelineInterval`） |

### 3.2 部品追加の提案手順 — 外部貢献の入口

（ADR-0003 決定 2。database.md §1.3 の provider 追加と同じ考え方。手順の詳細は CONTRIBUTING）

- 共通コンポーネントの追加・変更は、**本文書 §3.1 の表の更新を同じ PR に含める**提案として行う（Issue → 設計対話 → PR）
- 受け入れの技術条件: ①トークン（§2）のみで実装されている（**判定形: hex 値・任意の色/サイズの直書きがなく、参照はテーマ由来の値とトークンに限られる**）、②ライト・ダーク両テーマで §8 の基準を満たす、③画面文言が §7 の用語対応表・文言原則に従う、④ページ固有 CSS を持ち込まない

## 4. 画面レイアウトの骨格

- 全画面共通の骨格: **上部アプリバー**（製品名・状態帯・テーマ切替）+ **左ナビゲーション**（画面一覧。折りたたみ可）+ **コンテンツ領域**。この骨格自体も共通コンポーネントとする
- v0.1 の画面構成と責務:

| 画面 | 責務 | 主な設計委任元 |
|---|---|---|

## 5. 状態表示の原則

### 5.1 健康状態の一目確認（状態帯）

- ダッシュボード最上部（および全画面の状態帯）に「**稼働中 / 警告あり / 異常あり**」の 3 状態を常時表示する
- **判定は累積カウンタの現在値ではなく、直近の観測窓内の増分・現在のゲージ状態で行う**（過去の一過性破棄が永久に警告を灯し続けないため。窓幅・復帰条件・確認済み操作の要否は [UI-6](#12-確定待ち一覧)）。事象と状態の対応:

| 状態 | 判定入力（例） |
|---|---|
| 稼働中（state-ok） | 受信リスナが開いており、観測窓内に破棄・障害事象がない |
| 警告あり（state-warning） | 観測窓内のスプール退避の継続 / 上限・容量の接近 / 縮退運転中 / 証明書期限接近 |
| 異常あり（state-error） | 観測窓内の破棄（いずれかの破棄カウンタの増分）/ 保存先障害の継続 / 受信リスナが開けていない |

- 一過性事象で状態帯が復帰しても、**発生の事実は通知履歴・受信断履歴（状態画面）に残る**——状態帯は「今」を、履歴は「起きたこと」を担う
- 正常時は肯定形で表示するが、**観測できる範囲に限って言い切る**——対象と時間の両方を限定する: 「稼働中——現在、サーバに届いたログの取りこぼしは発生していません」（①ネットワーク上のロスはサーバから観測できない——architecture.md §4.5 / D-2 と矛盾する断言をしない、②判定は観測窓内であり累積の主張をしない——窓外に出た過去の破棄は履歴側で確認する。「状態帯は『今』、履歴は『起きたこと』」の分担のとおり）。「稼働中」はポートの待ち受けを意味し、ログが現に届いていることは意味しない——**全送信元合算の最終受信時刻を併記**し、送信元別の受信状況（無音化の検出は §12 UI-4）への導線を置く
- 実装参照（M8-2）: `YaguraStatusBand`（3 状態 + サマリ + 最終受信時刻の併記 + 送信元別への導線。判定自体はサーバ側観測性の管轄で、部品は表示のみを担う）
- 実装参照（M8-3）: 状態帯は共通骨格（`MainLayout`）が全画面へ常設し、更新間隔 10 秒（UI-2 の仮値）で自律更新する。判定の暫定実装は `Yagura.Host.Observability.SystemStatusReader`——観測窓は仮値 5 分（UI-6。読み出し履歴との差分で「窓内の増分」を判定。異常 = 破棄系カウンタの増分 / 警告 = スプール退避の増分・スプール使用量の上限接近（仮値 80%）・スプールなし縮退）。「受信リスナが開けていない」の判定入力は観測点が未整備のため未対応（リスナを開けない失敗は現状ホスト起動失敗として現れる——UI-6 の確定と合わせて観測点を追加する）。最終受信時刻は保存済みの最新 1 件の受信時刻（スプール滞留分は反映が遅れ得る——観測できる範囲の値）

### 5.2 最終更新時刻とステール警告

- **ダッシュボードは最終更新時刻を常時表示する**（ADR-0003 帰結）。表示が古くなった場合は、**画面全体にステール警告を重ねる**（時刻表示の色変化だけにしない——暗所の大型モニタで遠目に分かること）。閾値は仮値: 更新間隔の 3 倍（UI-2）。**起点はクライアントが最後にデータを受信した時刻**とする
- **ステール検知と警告の描画はクライアント側で自律動作させる**: サーバ停止・circuit 回復不能の局面でこそ警告が要るため、サーバ側レンダリングに依存できない。これは ADR-0003 決定 1 の「分離 JS モジュール」例外として**本文書で設計上確定した例外**であり、§11 の台帳（ページ固有の例外）とは区別する
- **ステール警告の文言は受信への影響を必ず含める**——ただし観測できる事実だけを言う: サーバの状態が確認できない場合は「表示が古くなっています（最終更新: ◯◯）。サーバの状態を確認できないため、ログの受信状況も不明です」。接続は維持されているのに表示の更新が止まっている場合も、受信の継続は保証できない（更新経路が異常な局面である）ため「画面とサーバの接続は維持されていますが、表示の更新が止まっています。ログの受信状況はこの画面から確認できません」とする。「受信は継続しています」と言い切ってよいのは §7.3 の再接続中（ブラウザとサーバ間のリンク断であり、サーバ健在を前提にできる）に限る
- 役割分担: **画面のステール警告は在室者の誤読防止**を担い、**無人時間帯の実効通知はイベントログ経由の既存監視**（architecture.md §4.6）が担う
- 実装参照（M8-2）: `YaguraStaleGuard`（最終更新時刻の常時表示 + 全画面オーバーレイ）+ 分離 JS モジュール `src/Yagura.Web/wwwroot/js/stale-guard.js`（本節が設計確定した例外の実装位置。監視タイマーと警告表示はブラウザ内で完結し、サーバと通信しない）。閾値の既定は更新間隔の 3 倍（UI-2 の仮値のまま。パラメータで上書き可）。circuit 生存中に更新だけが止まった局面の文言出し分けは更新経路を持つ画面側（M8-3 ダッシュボード）が引き受ける——**M8-3 で実装済み**: ダッシュボードの定期更新（10 秒。UI-2 仮値）が失敗した周期で「画面とサーバの接続は維持されていますが、表示の更新が止まっています。…」（本節の確定文言。`UiText.StaleWhileConnectedNotice`）を警告表示し、次の周期で自動再試行する

### 5.3 欠けているデータの明示

「無い」の理由を画面で見分けられるようにする:

- **受信断区間**（architecture.md §4.4）: 検索結果・ダッシュボードの時間軸上に区間として表示し、クラッシュ由来の近似断点はその旨を印す
- **保持地平**（database.md §2.3）: ログの保持期間より古い時間範囲を検索した場合、「この範囲は保持期間外（削除済み）」を結果に明示する。保持地平より古い受信断区間の表示は誤読（「区間以外はログがあったはず」）を招くため、保持地平の明示とセットでのみ見せる
- **検索の打ち切り**（architecture.md §6）: 結果上限・タイムアウトによる打ち切りは、件数・条件を絞る案内と共に表示する
- **保持期間の常時明示**（database.md §3）: ダッシュボードに「◯日より古いログは自動的に削除されます」の現在値を常時表示する。ファーストラン完了画面でも案内する
- 実装参照（M8-2）: `YaguraMissingDataNotice`（受信断区間・クラッシュ由来の近似注記・保持地平・検索の打ち切りの文言明示）/ `YaguraRetentionNotice`（保持期間の常時明示）
- 実装参照（M8-3）: 時間軸グラフ上の区間描画 = `YaguraTimelineChart`（§3.1。ダッシュボードの直近 1 時間・検索結果の時間軸の両方で受信断区間を帯として重ね描き。近似断点は薄い帯 + 一覧側の注記文言で印す）。保持地平の明示は検索画面が「検索範囲の下限なし・または下限が保持地平より古い」場合に必ず表示する（受信断区間の通知より上に置き、セット表示の規約を満たす）。保持期間が未適用（不正値フォールバック = 削除しない。database.md §3）の場合も「自動削除は行われない」ことをダッシュボードで常時明示する（`UiText.RetentionDisabledNotice`——削除の有無を画面から判別できない状態を作らない）

### 5.4 設定・システム由来の警告と案内

- configuration.md §1 のフォールバック（キー名・不正値・適用値の 3 点）と未知キー一覧は、設定画面と状態画面の両方に表示する（イベントログを見ない利用者にも届かせる）
- **反映方式の表示**（configuration.md §3 の引き受け）: 設定変更時に「即時反映 / 接続の瞬断あり / サービス再起動が必要」を操作前に示し、再読み込みの結果で**未反映のまま残った項目**を明示する
- **一時保管（スプール退避）中の表示**: state-warning の色 + アイコン + 補足文言（「ログの保存が追いついていません。一時保管領域に退避しています」）を**セットで常時表示**する——「一時保管」という言葉の無害さを色と文言で補う
- **昇格案内**（architecture.md §5.3 の文言の引き受け。文言例は表示面ごとに分ける）: 閲覧画面（ダッシュボード等）の通知は操作を含めない——「ログの保存が受信に追いついていません。保存先を SQL Server に切り替えることで改善が見込めます（切り替えと案内の抑制はサーバ上の設定画面から行えます）」。設定画面（管理リスナ）側では操作ボタンを含む——「…改善が見込めます。［切り替えの手順を見る］［今後表示しない］」。**抑制（今後表示しない）と解除は設定画面（管理リスナ）でのみ確定する**——閲覧画面の通知にはサーバ状態を変更する操作を置かない（§4 の不変条件を文言例のレベルでも守る）
- **切替（昇格）確定前の予告**（database.md §6.1 の文言の引き受け。文言例）: 「切り替えると、これまでに保存したログは移行機能の提供まで画面から参照できなくなります。あとで参照する可能性がある場合は『退避』を選んでください」
- Express 上限接近（database.md §5.3）は state-warning の通知として表示し、利用者の言葉 + 次に取る手順で書く（§7）
- 実装参照（M8-2）: `YaguraSystemNotice`（警告 = state-warning / 案内 = state-info。色 + アイコン + 補足文言のセット表示。**操作ボタンを持たない**——閲覧画面の通知にサーバ状態を変更する操作を置かない、の構造化）。一時保管・昇格案内（閲覧面）の確定文言は `UiText` に収載。操作を含む管理リスナ側の表示面は設定画面（M8-4）の管轄

### 5.5 OS 受信破棄ゲージの扱い（M8-3 で確定）

architecture.md §4.2 の M8 必須要求（「UI に表示する際は…常時可視の注記」と「表示自体の要否の判断」）の引き受けと判断記録:

- **判断: 値は表示しない。状態画面に常時可視の説明のみを掲示する**
- **理由**: M7-2 のクロスマシン実機検証（architecture.md §4.2「覆域の限界」）で、検証環境（Windows 10.0.26200）では本ゲージが**受信・破棄のいずれも一切計上しない**ことが実測確定した。常に 0 のままの数値を画面に置くと、注記を添えても「取りこぼしゼロ」という誤読（D-6 が明示的に禁じる解釈）の源になる——本設計は誤解を生まない側（非表示 + 説明）に倒す。ゲージが「動く環境では追加のシグナル」（同 §4.2 の降格後の位置づけ）であることは、計器（`yagura.os.udp.*`）自体を廃止しない理由であり、UI 表示の理由にはならない
- **常時掲示する説明の確定文言**（状態画面のカウンタ一覧直下。`UiText.OsUdpGaugeExplanation` / 同 `Supplement`）:
  - 本文: 「OS がこのアプリへ渡す前に破棄した受信データの数（OS の統計値）は、この画面に表示していません」
  - 補足: 「検証済みの Windows 環境では、この OS 統計は受信・破棄のどちらも計上しないことが実測で確認されています。0 という値を表示すると『取りこぼしなし』という誤解を生むため、値の表示自体を行いません。取りこぼしの確認には、上記のカウンタと、ダッシュボードの送信元別の受信状況（最終受信時刻）をあわせて確認してください」
- **再評価の条件**: 本ゲージが実際に計上する Windows 版・環境が確認された場合（M-15 の lab 検証・ADR-0002 の再評価トリガ）、「値の表示 + 常時注記（システム全体値・同居プロセス分の混入）」への切り替えを再検討する。その場合も D-6 の解釈ガイド（値 0 ≠ 取りこぼしゼロ）を注記に含めることが条件である

## 6. 時刻表示の契約

- 保存は UTC（database.md §2.2）。**表示はすべてサーバの OS タイムゾーンでローカル化し、タイムゾーンを画面に明示する**（例: `2026-07-04 21:05:11 (UTC+09:00)`）。オフセット表記は**そのレコードの時刻時点のオフセット**とする（夏時間のあるタイムゾーンでも一意に読める）
- **閲覧者ごとのタイムゾーン変換はしない**。主理由: ①閲覧リスナは認証なしで閲覧者の概念がなく、閲覧者ごとの設定を保持する場所がない（ADR-0004 決定 2 の帰結）、②ターゲットの中心は単一タイムゾーンの環境（ADR-0001）、③表示の一意性は実装も検証も単純にする。補助的に、同じ画面を見る管理者同士で時刻の読みが一致する（口頭・電話での突合が単純になる）。**リモート拠点からの閲覧で閲覧者別表示の要望が反復した場合（目安: 独立した利用者から 3 件以上）に再評価する**
- 基準軸はサーバ受信時刻（ReceivedAt）。送信元が名乗った時刻（DeviceTimestamp）は詳細表示で併記し、受信時刻との乖離が大きい場合はその旨を示す（装置の時計ずれの発見に使う。乖離の閾値は UI-2 と併せて確定する）
- **契約の実装位置（M8-2）**: 整形は `YaguraTimeFormatter`（`src/Yagura.Web/Components/Common/`）の 1 箇所に集約し、画面の時刻表示は `YaguraTimestamp` コンポーネント経由に統一する（`<time>` の datetime 属性に ISO 8601 の UTC 表記を併せて出す——機械可読の一意表現）。画面が独自の書式・変換を持つことを禁止する

## 7. 用語対応表と文言原則

### 7.1 文言原則（ADR-0003 決定 5）

- 画面の文言は平易な日本語とし、開発用語を利用者向けの画面に出さない。エラーメッセージ・通知（イベントログ警告の本文を含む）も対象
- **対応は 1 対 1 とする**: 画面の言葉 1 つは開発用語 1 つに一意対応させ、複数の開発用語を同じ平易語に写像しない（試用報告・監査報告が設計文書と突合できることを対応表の設計要件とする）
- 文言はリソースとして分離する（将来の英語化に備える。v0.1 の提供言語は日本語のみ）。実装参照（M8-2）: 共通コンポーネントの文言は `UiText`（文言カタログ。部品は文言リテラルを持たない）に分離済み——v0.1 は日本語のみのため静的カタログとし、英語化時に .resx 移行の単一起点とする
- **新しい開発用語を画面に導入する PR は、本表への追記を同じ PR に含める**。利用者向けヘルプ・マニュアル（operations.md 系）も本表を参照する。**syslog プロトコル用語（severity・facility 等）も本表の対象**であり、実装設計で画面に出す用語から順次追記する

## 8. アクセシビリティ基準（ADR-0003 決定 6 の個別化）

**ライト・ダーク両テーマで**次を受け入れ条件とする:

- **キーボード完遂**: 検索の実行・結果の閲覧・詳細表示・設定変更・ウィザード完了までの主要操作が、マウスなしで完遂できる。フォーカスは常に視認できる（フォーカスリングを消さない）。ダイアログは Esc で閉じられ、フォーカスが呼び出し元に戻る
- **コントラスト**: テキストと背景のコントラスト比は WCAG 2 の AA 基準値（通常テキスト 4.5:1 以上・大きなテキスト 3:1 以上）を満たす。状態色のテキスト利用時も同様。§2.1 の初期値はこの基準での検証を経て確定する（UI-1。未達の色はトークン側を調整する——トークン経由の強制（§1）が調整の影響を局所化する）
- **色以外の手がかり**: 状態・エラーは色 + アイコンまたは文言で伝える（§2.1）
- 検証はテーマごとに行い、結果（検証日・ツール・対象トークン）を本文書に記録する

**コントラスト検証記録（UI-1。検証日 2026-07-06。M8-2 / Issue #69）**:

- ツール: リポジトリ内テスト `DesignTokenContrastTests`（tests/Yagura.Web.Tests/Theming/。WCAG 2 の相対輝度式による機械計算。検証対象は実装のテーマ定義 `YaguraTheme.Instance` そのもので、**毎ビルド自動実行される**——以後のトークン変更はこのテストが恒常的に検証する）
- 対象: 両テーマの全実利用ペア——本文・補助テキスト・primary（リンク）・状態色 4 種のテキスト利用 × 背景/面、ContrastText × primary/状態色の塗り面、アプリバー/ドロワーの文字 × 面（計 21 ペア/テーマ）
- 結果: **ライトの state-warning 初期値 #B26A00（対 surface 4.24 / 対 background 3.95）と state-info 初期値 #0277BD（対 background 4.47）が 4.5:1 未達**だったため、UI-1 の手続きどおりトークン側を調整した——state-warning #A05F00（対 surface 5.08 / 対 background 4.74 / 白文字対塗り 5.08）・state-info #0270B2（対 surface 5.30 / 対 background 4.94 / 白文字対塗り 5.30）。他のライト全ペアとダーク全ペアは調整なしで基準を満たした
- 大きなテキスト（3:1）の緩和は使っていない——全ペアを通常テキスト基準（4.5:1）で判定した
