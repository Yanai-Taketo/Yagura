## 1. デザインシステムの原則

- **すべての画面はデザイントークン（§2）と共通コンポーネント（§3）だけで組む**。ページ固有の CSS・JavaScript・インラインスタイル（`Style=` 属性）は原則禁止（ADR-0003 決定 2）
- **例外の手続き**: ページ固有であるべき理由を PR に明記し、[§11 ページ固有 CSS 例外一覧](#11-ページ固有-css-例外一覧) に登録した上で認める。例外は共通化の候補であり、共通コンポーネントへ吸収されたら一覧から消し込む
- **MudBlazor の低レベル API をページに散乱させない**（ADR-0003 決定 3 の緩和策）: 頻出パターンは共通コンポーネントへ集約し、ページは共通コンポーネントを組み合わせる。MudBlazor を直接使ってよいのは共通コンポーネントの実装内部と、単純な静的レイアウト部品（コンテナ・グリッド等）に限る
- **色・フォント・余白の直接指定を禁止する**: すべてトークン経由で参照し、テーマ切替（§2.4）で破綻しない構造を保つ（ADR-0003 決定 4）

## 2. デザイントークン

### 2.1 配色

方向性は「クリーンな業務系・ライト基調」（ADR-0003 決定 4）。値は初期値であり、§8 のコントラスト検証（UI-1）を通して確定する。

| トークン | ライト | ダーク | 用途 |
|---|---|---|---|
| primary | #1565C0 | #90CAF9 | 主要アクション・リンク・選択状態 |
| background | #F5F7FA | #121212 | 画面背景 |
| surface | #FFFFFF | #1E1E1E | カード・テーブル・ダイアログの面 |
| text-primary | #1A1A1A | #E0E0E0 | 本文 |
| text-secondary | #5F6B7A | #A0A0A0 | 補助テキスト・ラベル |
| divider | #D7DDE4 | #3A3A3A | 罫線・区切り |

**状態色**（意味を固定し、これ以外の意味で使わない）:

| トークン | ライト | ダーク | 意味 |
|---|---|---|---|
| state-ok | #2E7D32 | #81C784 | 正常・成功（受信中・保存成功） |
| state-warning | #B26A00 | #FFB74D | 警告・要注意（スプール退避中・上限接近・縮退運転） |
| state-error | #C62828 | #E57373 | 異常・失敗（受信断・書込失敗・破棄発生） |
| state-info | #0277BD | #4FC3F7 | 情報（案内・昇格提案） |

- **色だけで意味を伝えない**: 状態は常にアイコンまたはテキストを伴う（色覚特性への配慮。§8）
- severity（syslog の重大度）の表示色は状態色と別系統のトークンとして実装設計で定義する（Error 以上= state-error 系、Warning = state-warning 系に揃える）

### 2.2 タイポグラフィ

- フォントは Windows 標準スタックとし、Web フォントを同梱・参照しない（閉域完結・ADR-0003 決定 1 の npm/CDN 非依存と同じ向き）: `"Yu Gothic UI", "Segoe UI", Meiryo, sans-serif`。等幅（ログ本文・生データ表示）: `Consolas, "Courier New", monospace`
- サイズ段階（px）: 12（注釈）/ 14（本文・既定）/ 16（強調・小見出し）/ 20（画面見出し）/ 28（数値の大表示——ダッシュボードのカウンタ等）。行間は 1.5 を基準とする

### 2.3 余白・形状

- 余白は **4px 基準のスケール**（4 / 8 / 12 / 16 / 24 / 32）のみ使用する
- 角丸は 4px（入力・ボタン）と 8px（カード・ダイアログ）の 2 値。影は MudBlazor の elevation を 0（フラット）・1（カード）・8（ダイアログ）の 3 段のみ使う

### 2.4 テーマ

- 既定はライト。**ダークは同格のテーマ**であり、全トークンが両テーマの値を持つ（ライトの機械変換で作らない。ADR-0003 決定 4——ダッシュボードの暗所掲示運用を想定ユースケースに含む）
- 切替は UI 上のトグルで行い、選択を保存する。OS 設定への追従は選択できる項目とし、既定では手動選択を尊重する（勝手に切り替わらない）

## 3. 共通コンポーネント規約

### 3.1 標準形（v0.1 の初期セット）

| コンポーネント | 規約 |
|---|---|
| ボタン | 主ボタン（primary 塗り）は 1 画面 1 つまで。破壊的操作（削除・昇格確定等）は state-error 系 + 確認ダイアログ必須 |
| フォーム | ラベルは入力の上。検証エラーは項目直下に文言で表示（色だけにしない）。必須項目は「必須」表記（記号 * だけにしない） |
| テーブル（ログ一覧） | 軽量射影列のみ表示（architecture.md §6）。行クリックで詳細へ。**打ち切り（結果上限到達）は表の末尾に件数と共に明示する** |
| 通知（トースト） | 一時通知は自動で消えるが、**state-error の通知は手動で閉じるまで残る**（見逃し防止）。通知履歴は状態画面から辿れる |
| 空状態 | 「データがない」画面は、無地ではなく**次に取るべき行動**（例: 機器の送信設定の案内）を必ず示す |
| 確認ダイアログ | 取り消せない操作は、何が起きるかの要約 + 明示的な確認ボタンで構成する。既定フォーカスは安全側（キャンセル）に置く |
| 状態帯（ヘルスバー） | ダッシュボード・全画面共通ヘッダに置く現在状態の帯（§5.1）。state-ok / warning / error の 3 状態 + 一言サマリ |

### 3.2 部品追加の提案手順 — 外部貢献の入口

（ADR-0003 決定 2。database.md §1.3 の provider 追加と同じ考え方）

- 共通コンポーネントの追加・変更は、**本文書 §3.1 の表の更新を同じ PR に含める**提案として行う（Issue → 設計対話 → PR。手順の詳細は CONTRIBUTING）
- 受け入れの技術条件: ①トークン（§2）のみで実装されている、②ライト・ダーク両テーマで §8 の基準を満たす、③画面文言が §7 の用語対応表・文言原則に従う、④ページ固有 CSS を持ち込まない
- 同型の UI が 2 画面以上に現れた時点で共通化を検討する（3 回目の複製を作らない）

## 4. 画面レイアウトの骨格

- 全画面共通の骨格: **上部アプリバー**（製品名・状態帯・テーマ切替）+ **左ナビゲーション**（画面一覧。折りたたみ可）+ **コンテンツ領域**。この骨格自体も共通コンポーネントとする
- v0.1 の画面構成と責務:

| 画面 | 責務 | 主な設計委任元 |
|---|---|---|
| ダッシュボード | 健康状態の一目確認（§5.1）・受信量の推移・送信元別の受信状況・スプール/カウンタの現在値 | architecture.md §4.6 |
| ログ検索 | 絞り込み強制なしの検索・軽量一覧・詳細表示・受信断区間の時間軸表示 | architecture.md §6・§4.4 |
| システム状態 | 全カウンタ・ゲージの一覧（コピー 1 回で取得可能な形式 = 試用報告様式対応）・通知履歴・受信断履歴 | architecture.md §4.1・ADR-0006 基準 2 |
| 設定（ウィザード群） | 初期セットアップ・本番昇格・設定変更（管理リスナ経由のみ） | configuration.md §3〜§7・database.md §6.1 |

- 閲覧系（ダッシュボード・検索・状態）は閲覧リスナ、書き込み系（設定）は管理リスナに属する（configuration.md §4.2）。閲覧画面には書き込み操作を置かない

## 5. 状態表示の原則

### 5.1 健康状態の一目確認

- ダッシュボード最上部（および全画面の状態帯）に「**受信中 / 警告あり / 異常あり**」の 3 状態を常時表示する。判定入力は architecture.md §4 のカウンタ・ゲージ（破棄カウンタの増加・スプール退避中・縮退運転・受信断等）
- 正常時は「受信中・直近の取りこぼしなし」と**肯定形で表示する**（「エラーなし」という否定形の空欄にしない——「全カウンタ 0 なら安心してよい」を一目で答える）。最終受信時刻を併記する

### 5.2 最終更新時刻とステール警告

- **ダッシュボードは最終更新時刻を常時表示する**（ADR-0003 帰結）。データが一定時間更新されない・再接続に失敗している場合は、**古いデータが「最新」の顔をしないよう画面全体にステール警告を重ねる**（時刻表示の色変化だけにしない——暗所の大型モニタで遠目に分かること）。閾値は仮値: 更新間隔の 3 倍（UI-2）
- Blazor の再接続中の表示・文言は §7 の文言原則に従う（「サーバが壊れた」と誤解させない。ADR-0003）

### 5.3 欠けているデータの明示

「無い」の理由を画面で見分けられるようにする:

- **受信断区間**（architecture.md §4.4）: 検索結果・ダッシュボードの時間軸上に区間として表示し、クラッシュ由来の近似断点はその旨を印す
- **保持地平**（database.md §2.3）: ログの保持期間より古い時間範囲を検索した場合、「この範囲は保持期間外（削除済み）」を結果に明示する。保持地平より古い受信断区間の表示は誤読（「区間以外はログがあったはず」）を招くため、保持地平の明示とセットでのみ見せる
- **検索の打ち切り**（architecture.md §6）: 結果上限・タイムアウトによる打ち切りは、件数・条件を絞る案内と共に表示する
- **保持期間の常時明示**（database.md §3）: ダッシュボードに「◯日より古いログは自動的に削除されます」の現在値を常時表示する。ファーストラン完了画面でも案内する

### 5.4 設定・システム由来の警告表示

- configuration.md §1 のフォールバック（キー名・不正値・適用値の 3 点）と未知キー一覧は、設定画面と状態画面の両方に表示する（イベントログを見ない利用者にも届かせる）
- 昇格案内（architecture.md §5.3）・Express 上限接近（database.md §5.3）は state-info / state-warning の通知として表示し、**利用者の言葉 + 次に取る手順**で書く（§7）。昇格案内は抑制でき、抑制状態は設定画面から解除できる

## 6. 時刻表示の契約

- 保存は UTC（database.md §2.2）。**表示はすべて閲覧環境ではなくサーバの OS タイムゾーンでローカル化し、タイムゾーンを画面に明示する**（例: `2026-07-04 21:05:11 (UTC+09:00)`）。閲覧者ごとのタイムゾーン変換はしない——複数管理者が同じ画面を見て同じ時刻を読める（電話で「21 時 5 分のログ」が通じる）ことを優先する
- 基準軸はサーバ受信時刻（ReceivedAt）。送信元が名乗った時刻（DeviceTimestamp）は詳細表示で併記し、受信時刻との乖離が大きい場合はその旨を示す（装置の時計ずれの発見に使う。database.md §2.2）

## 7. 用語対応表と文言原則

### 7.1 文言原則（ADR-0003 決定 5）

- 画面の文言は平易な日本語とし、開発用語を利用者向けの画面に出さない。エラーメッセージ・通知（イベントログ警告の本文を含む）も対象
- 文言はリソースとして分離する（将来の英語化に備える。v0.1 の提供言語は日本語のみ）
- **新しい開発用語を画面に導入する PR は、本表への追記を同じ PR に含める**。利用者向けヘルプ・マニュアル（operations.md 系）も本表を参照する

### 7.2 用語対応表（初版）

| 開発用語（設計文書） | 画面の言葉 |
|---|---|
| スプール / スプール退避 | 一時保管 / 一時保管中（保存が追いつくまでログをディスクに退避しています） |
| ドロップ / 破棄 | 取りこぼし |
| バックプレッシャ | （画面に出さない——挙動の説明は「受信量が処理を上回っています」） |
| provider | 保存先（データベース） |
| 本番昇格 | 保存先を SQL Server に切り替える |
| opt-in / opt-out | 追加で有効にする / 標準で有効（無効にできます） |
| 保持期間 | ログを保存しておく期間 |
| 受信断 | 受信できなかった時間帯 |
| 縮退運転 | 一部機能を停止して動作中 |
| 解析失敗（raw 保存） | 形式を解釈できなかったログ（原文のまま保存しています） |
| circuit / 再接続 | 画面とサーバの接続 / 再接続 |

### 7.3 再接続まわりの文言例（ADR-0003 / PR #5 佐藤の委任）

- 再接続中: 「サーバとの接続が一時的に切れました。自動的に再接続しています——ログの受信は継続しています」（**受信は止まっていないことを必ず添える**——閲覧の切断とログの喪失を混同させない）
- 再接続失敗: 「接続を回復できませんでした。ページを再読み込みしてください。表示が古い可能性があります（最終更新: ◯◯時◯◯分）」

## 8. アクセシビリティ基準（ADR-0003 決定 6 の個別化）

**ライト・ダーク両テーマで**次を受け入れ条件とする:

- **キーボード完遂**: 検索の実行・結果の閲覧・詳細表示・設定変更・ウィザード完了までの主要操作が、マウスなしで完遂できる。フォーカスは常に視認できる（フォーカスリングを消さない）。ダイアログは Esc で閉じられ、フォーカスが呼び出し元に戻る
- **コントラスト**: テキストと背景のコントラスト比は WCAG 2 の AA 基準値（通常テキスト 4.5:1 以上・大きなテキスト 3:1 以上）を満たす。状態色のテキスト利用時も同様。§2.1 の初期値はこの基準での検証を経て確定する（UI-1。未達の色はトークン側を調整する——トークン経由の強制（§1）が調整の影響を局所化する）
- **色以外の手がかり**: 状態・エラーは色 + アイコンまたは文言で伝える（§2.1）
- 検証はテーマごとに行い、結果（検証日・ツール・対象トークン）を本文書に記録する

## 9. UI メトリクス（circuit・再接続）

（PR #5 鈴木 / ADR-0004 決定 6 の委任）

- 次を architecture.md §4.6 のゲージ・カウンタ体系に加える: **circuit 現在数（対上限比）** / **再接続の発生回数** / **再接続失敗（circuit 喪失）回数**
- 上限の値・到達時挙動・認証失効の反映はセキュリティ設計文書の管轄。本文書は「上限に対する現在値が観測できること」を UI 側の要件として引き受ける（状態画面 §4 に表示）

## 10. MudBlazor の統治

- 採用と選定理由は ADR-0003 決定 3（バージョンは導入 PR 時に当日ライブ検証し、`Vendored:` 記録を残す——conventions.md）
- **同梱静的アセットの脆弱性監視の具体手段**（ADR-0003 決定 3 の委任）:
  1. **NuGet パッケージとしての監視**: CI の依存脆弱性スキャン（conventions.md 既定）と GitHub の依存アラートで MudBlazor 本体の既知脆弱性を検出し、検出時は PR をブロックする
  2. **公式セキュリティ勧告の追従**: MudBlazor の GitHub Security Advisories・リリースノートを、依存更新の定期 PR（頻度は実装期に確定）で確認する
  3. **同梱アセットの実体確認**: MudBlazor が同梱する CSS/JS に第三者由来のコード（別ライブラリのバンドル）が含まれるか否かを**導入 PR で実体確認し、結果を本節に記録する**（含まれる場合、その第三者ライブラリの監視手段を追加で定める）（UI-3）
- メジャー更新の追従・保守停滞時の再評価は ADR-0003 の再評価トリガに従う。ページが MudBlazor API を直接使う面を §1 の規約で狭く保つことが、更新・乗り換えの影響範囲を限定する（同 ADR の緩和策の実行形）
