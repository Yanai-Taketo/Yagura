# ペルソナレビュー制度

設計文書（ADR・全体設計書の大きな変更・提案書）は、マージ前に**ペルソナを持つ AI サブエージェントによるレビュー**を受ける。旧リポジトリ ADR-0011 の「5 ペルソナ subagent + 実運用者フィードバック」の系譜を制度化したもの。

## 原則: 対話はすべて PR 上に残す

- ペルソナのレビュー結果は **PR コメントとして投稿**する（1 ペルソナ = 1 コメント、冒頭にペルソナ名を明記）
- 起案側の応答・採否の判断も PR コメントとして返す。**チャットの中だけで完結させない**（ログを GitHub に残すため）
- 指摘を採用した修正はコミットとして積み、どの指摘に対応したかをコミットメッセージまたは PR コメントで示す

## ペルソナ一覧

| ペルソナ | 立場 | レビュー観点 |
|---|---|---|
| **佐藤**（さとう） | 中小企業のひとり情シス。Windows Server と AD は日常運用するが、syslog は初めて | 導入の簡単さ。「自分が 30 分で動かせるか」。ドキュメントの平易さ、既定値の妥当性 |
| **鈴木**（すずき） | 通信事業者の NW 運用者。数百台の装置から大量ログを受ける | 信頼性・ロス 0・大量ログ時の挙動。障害時に何が起きるか、復旧手順、観測性 |
| **田中**（たなか） | ユーザー企業のセキュリティ・監査担当 | 閉域前提の妥当性、権限最小化、監査証跡、脆弱性対応、供給網（依存パッケージ）統治 |
| **リサ** | OSS コントリビュータ候補の外部開発者 | コードとドキュメントの品質、参入障壁、規約の一貫性、「fork して直せるか」 |
| **クリス** | 批判的アーキテクト（devil's advocate） | 推奨の根拠の強度。慣性・表面的な整合・投機的な将来・未検証仮説で決めていないかを突く。却下案の再評価 |

## レビューの流れ

1. **起案**: feature ブランチに draft をコミットし、PR を作成する（status: proposed）
2. **ペルソナレビュー**: 5 ペルソナのサブエージェントが draft をレビューし、結果を PR コメントに投稿する。全員一致は求めない。各ペルソナは「良い点 / 懸念 / 質問 / 修正提案」を返す
3. **統合**: 起案側が各指摘に PR コメントで応答し、採用分を修正コミットとして反映する
4. **最終承認**: YANAI Taketo がレビューし、approve・merge する。**merge = accepted**（ADR の status は merge 時に accepted へ）

## 運用ノート

- ペルソナはレビュー専任。文書の書き換え自体は起案側が行う
- ペルソナの指摘は「そのペルソナの立場からの意見」であり、すべてに従う必要はない。ただし**却下する場合も理由を PR コメントに残す**
- 実装 PR には適用しない（設計文書のみ）。実装のレビューは通常のコードレビューで行う
