# 運用ガイド（利用者向け）

現在形の設計書（[design/](design/)）が「どう作るか」を定めるのに対し、本書は**導入・運用する人向け**の手順・注意をまとめる。まだ認証（管理 UI / 閲覧 UI）の運用に絞った初版であり、他の運用トピック（本番昇格の手順、逆引き参照ゾーンの整備、TLS 受信の証明書運用など）は各機能の実装に合わせて追記していく。

> **申し送りの受け皿**: 本書は複数の ADR が「利用者向けドキュメント（operations.md 系）へ申し送る」と定めた事項の受け皿である。認証に関する申し送りは [ADR-0010](adr/0010-admin-ui-authentication.md) 委任事項 11・[ADR-0013](adr/0013-admin-winauth-session.md) 委任事項 8。

## 1. 認証は既定でオフ（信頼ネットワーク前提 + opt-in 強化）

Yagura は既定で**認証なし**（[ADR-0004](adr/0004-security-model.md)・[ADR-0001](adr/0001-project-founding.md) の「信頼ネットワーク前提」）で動く。管理 UI（既定ポート 8515）は loopback 束縛のみ、閲覧 UI（既定ポート 8514）は LAN 公開。**機微なログを扱う環境で「LAN 内の誰でも読める」を締めたい場合にのみ**、認証を opt-in で有効化する。有効化しない限り本書の内容は関係しない。

## 2. 閲覧 UI 認証（8514）を有効化する

閲覧 UI 認証は **Windows 統合認証 + AD グループ**を主経路とする（[ADR-0010](adr/0010-admin-ui-authentication.md) 決定 7）。設定ファイル `yagura.json`（データルート直下。既定 `%ProgramData%\Yagura`）に次を追記して**サービスを再起動**する（設定の反映はサービス再起動——[configuration.md](design/configuration.md) §3）。

```json
{
  "Viewer": {
    "Authentication": {
      "Windows": {
        "Enabled": "true",
        "ViewerGroups": [ "CONTOSO\\LogViewers" ],
        "AdminGroups": [ "CONTOSO\\LogAdmins" ]
      }
    }
  }
}
```

- **グループの指定は名前（`DOMAIN\グループ名`）でも SID（`S-1-...`）でもよい**。名前は起動時に SID へ解決してキャッシュする。存在しない名前・タイプミス・DC 到達不能で解決できないグループは**起動を止めず警告してスキップ**する（そのグループの所属者は認可されないだけ——安全側）。イベントログに `sec9-group-unresolved` の警告が出たらグループ名を見直す。
  - **名前指定は起動時に DC へ問い合わせる**（`S-1-...` の SID 指定は問い合わせを発しない）。DC が一時的に到達不能なときにサービスを再起動すると、名前解決が最大 5 秒で打ち切られてそのグループはスキップされる（**syslog 受信の開始をブロックしないため**——受信は止めない）。この間に認可対象のグループが欠けると閲覧できない利用者が出るので、**DC 障害に強くしたい場合はグループを SID で指定する**とよい（SID なら DC 非依存で即解決できる）。
- **`AdminGroups` を指定すると平文で管理等価 Cookie が流れる点に注意**（機微環境向け）。閲覧リスナは既定で平文 HTTP のため、`AdminGroups` 所属者が発行される「管理」役割の Cookie は Secure 属性なしで LAN を流れ、host スコープゆえ管理リスナ（8515/8516）へも届く。機微環境では管理役割の付与は管理リスナ側（loopback またはリモート HTTPS）で行い、閲覧リスナには `ViewerGroups`（閲覧のみ）だけを指定する運用を検討する。起動時にイベントログへ `viewer-admingroups-plaintext` 注意が出る。
- **役割**: `ViewerGroups` の所属者は「閲覧」（ログの閲覧・検索・CSV エクスポート・自身の状態表示）。`AdminGroups` の所属者は「管理」で、**管理は閲覧を含む**（管理者は閲覧 UI も読める）。ネストグループ（グループの入れ子）は OS が展開したトークンで自動的に効くため、追加設定は要らない。
- **8514（閲覧リスナ）には管理機能の導線を出さない**。`AdminGroups` の利用者が 8514 にログインしても、設定変更などの管理画面は表示されない（管理操作は管理リスナ 8515 で行う）。
- **アプリ独自 ID/パスワードでも閲覧ログインできる**（`Admin:Authentication:App:Enabled = true` のとき、閲覧ログイン画面にも ID/パスワード欄が出る）。アプリ独自アカウントは常に「管理」役割のため、閲覧 UI へも到達できる。
- **自己ロックアウト注意**: `Viewer:...:Windows:Enabled = true` なのに `ViewerGroups`・`AdminGroups` をどちらも指定しない（または全て解決失敗）と、**Windows 経由で閲覧できる人がいなくなる**。起動時にイベントログへ `viewer-auth-no-groups` 警告が出る。閲覧が止まっても syslog 受信は止まらない——管理者は 8515（管理リスナ）や設定ファイルの手編集で復旧できる。

Kerberos のみを許可し NTLM を拒否したい場合は `Viewer:Authentication:Windows:KerberosOnly = "true"`（管理側の `Admin:...:KerberosOnly` とは独立——ポートごとに選べる）。

**有効化したら 1 人ログインできることを必ず確認する**。グループ名のタイプミスや所属設定漏れは、既定ではイベントログ（`sec9-group-unresolved` / `viewer-auth-no-groups`）にしか出ない。有効化・再起動後に、閲覧を許可したいユーザーで実際に 8514 へブラウザアクセスしてサインインできるか確かめる（誰もサインインできない状態に気づかず放置しないため）。

## 3. 管理 UI 認証と AD グループ（SEC-9）

管理 UI 認証（[ADR-0010](adr/0010-admin-ui-authentication.md) Phase 1）の「管理者」は既定で `BUILTIN\Administrators`（ローカル Administrators）のみ。**AD グループを管理者にマップしたい場合**は、既定判定に**加えて** `Admin:Authentication:Windows:AdminGroups` を指定する（ローカル Administrators 判定は残る）。

```json
{ "Admin": { "Authentication": { "Windows": { "Enabled": "true", "AdminGroups": [ "CONTOSO\\LogAdmins" ] } } } }
```

- ドメインコントローラー上では、ネットワークログオンのトークンに `BUILTIN\Administrators`（S-1-5-32-544）が載らないことがある。この場合、ローカル Administrators だけに頼ると DC 上の管理者がログインできない——**ドメイングループを `AdminGroups` に指定すれば解決する**（ドメイングループの SID はネットワークトークンに載る）。

## 4. セッションが切れたときの挙動

認証成立後は方式に依らない**単一のセッション Cookie**（[ADR-0013](adr/0013-admin-winauth-session.md)）で認証状態を持つ。**Cookie の寿命は認証方式で非対称**である:

| 方式 | 絶対寿命 | sliding（操作で延長） | 理由 |
|---|---|---|---|
| Windows 統合認証 | **1 時間（仮値）** | なし | Windows 権限（グループ所属）の剥奪は再ログインまで反映できないため、寿命を短くして陳腐化を有界化する |
| アプリ独自 ID/パスワード | 8 時間 | あり | 将来のログインは無効化で即座に絞れるため、既発行セッションは従来どおり |

- **切れたら再ログイン画面へ誘導される**。掲示用の大型モニタで閲覧ダッシュボードを開きっぱなしにする運用（NOC の壁掛けモニタ等）では、Windows 由来セッションが 1 時間で切れて再ログインを求める点に注意する。
  - **24 時間無人で映し続ける掲示・キオスクには、そもそも閲覧認証を課さないネットワーク区分での運用を第一に推奨する**。認証を有効化した閲覧リスナは、方式に依らずいずれセッションが切れる（Windows 1 時間、アプリ独自も無操作なら絶対寿命 8 時間で切れる。sliding は操作前提のため無人では延びない）。無人掲示に「切れない認証セッション」は用意しない設計であり、認証と 24 時間無人掲示は本質的に両立しにくい。掲示用の区分は閲覧認証オフ（既定）で公開し、機微な検索・エクスポートを行う端末側だけ認証オンの閲覧リスナ（別インスタンス/別区分）を使う、といった分離が現実的である。
- **Windows 権限を剥奪しても、反映は最大 Cookie 寿命（Windows は最大 1 時間）だけ遅れる**。即時に締めたい場合は下記の緊急全失効を使う。

## 5. 緊急全失効（退職者・漏洩疑い時）

発行済みの**全**認証セッション（管理・閲覧の両方）を即時無効化できる（[ADR-0013](adr/0013-admin-winauth-session.md) 決定 2。監査イベント ID 2013）。管理 UI から実行する（セッション世代番号をバンプする——DC に依存せずローカルで効く）。

- **all-or-nothing**: 特定の 1 人だけを狙う失効はできない。正規利用者を含む全員が次の操作で無効化され、再ログインが必要になる。退職者対応・資格情報漏洩疑いなど「全部落として入れ直す」場面のための機構である。
- **ログイン経路は殺さない**: 全失効後も、アプリ認証・Windows 再ログイン・設定ファイル手編集は生きている（ロックアウトしない）。

## 6. ブレークグラス（非常口）の推奨

- **リモート管理を Windows 統合認証で運用する場合も、アプリ独自 ID/パスワードの管理者アカウントを最低 1 つ持つことを強く推奨する**（[ADR-0010](adr/0010-admin-ui-authentication.md) 決定 3）。AD/DC 障害時に RDP 経路も同じ AD に依存していると、Windows 認証単独では AD 障害 1 点で管理面全体を失う。
- **loopback 認証 opt-in（`Admin:Authentication:RequireForLoopback = true`）を有効化した環境の最終復旧**は「物理到達（RDP/コンソール）→ 設定ファイル手編集で opt-in を解除 → サービス再起動」。ただし現状、**設定反映にはサービス再起動が必要で、その間 syslog 受信が止まる**（受信断）。この代償を承知の上で opt-in を有効化する。
- **手編集の実行可能者はローカル Administrators に限られる**（設定ファイルの書き込み ACL）。AD グループ経由で「管理」役割のみを持つ（ローカル Administrators 非所属の）利用者は、この手編集復旧を自分では行えない——別途ローカル Administrator への依頼が要る。
- **DC 障害中の相互作用**: DC 障害中は既存セッションは寿命まで生存するが、寿命が切れた後の Windows 再ログインはグループ解決ができず失敗する。実質のレジリエンス窓が Cookie 寿命に縮む——ブレークグラス（アプリ管理者アカウント／手編集）で救う。

## 7. NTLM 縮退・非ドメイン機

- Windows 統合認証は Kerberos を優先し、成立しない場合（ホスト名の SPN 未登録・IP アドレスでのアクセス等）は自動的に **NTLM へ縮退**する。非ドメイン参加機（ワークグループ）でもローカル Windows アカウントに対する NTLM 認証がそのまま使える——ドメイン必須ではない。
- NTLM 縮退時は、ブラウザに**資格情報の入力ダイアログ**が出る（Kerberos SSO のような透過ログインにはならない）。NTLM の既知の弱点（リレー攻撃・pass-the-hash・相互認証の欠如）を受容できない環境では、`KerberosOnly = "true"` で NTLM を拒否する（Kerberos SSO が成立する SPN 構成が前提）。

## 8. 監査（誰がアクセスしたか）

認証を経由した操作は、監査記録（`audit.jsonl` + Windows イベントログ、ソース `Yagura`）の「誰が」欄に方式（`windows`/`app`）と利用者名が残る（[security.md](design/security.md) §4）。認証まわりの主なイベント ID:

| ID | レベル | 意味 |
|---|---|---|
| 2008 | 情報 | 管理 UI へサインイン成功 |
| 2014 | 情報 | 閲覧 UI へサインイン成功（`role=viewer`/`role=admin`） |
| 2013 | 情報 | 認証セッションの緊急全失効 |
| 3003 | 警告 | Windows 統合認証のハンドシェイク失敗 |
| 3008 | 警告 | 認証成功後に管理者権限がなく拒否 |
| 3009 | 警告 | 閲覧で認証成功後に閲覧/管理グループ非所属で拒否 |

「ソース Yagura の警告以上を通知」で最低限の監視が組める（拒否・セキュリティ事象は警告レベル）。
