# コード署名ポリシー

Yagura のリリース成果物（MSI と自前アセンブリ）に付与する Authenticode コード署名の仕組み・体制・検証手順を開示します。設計判断の経緯は [ADR-0014](adr/0014-code-signing.md) を参照してください。

## 利用者向け要約（社内説明にそのまま使えます）

> 本ソフトウェア「Yagura」のインストーラーには、開発者本人によるコード署名が付与されています。署名者として表示される「**柳井建人**（YANAI Taketo）」は、本ソフトウェアを開発・公開している個人 OSS 開発者の実名です。署名に使う証明書は認証局 Sectigo が本人確認のうえ発行した OV（Organization Validation）コード署名証明書で、署名鍵は Google Cloud KMS の HSM（ハードウェアセキュリティモジュール）内にあり、開発者の手元にも CI 環境にも存在しません。配布物が正規のものかどうかは、ファイルのプロパティ →「デジタル署名」タブで署名者が「柳井建人」であることを確認することで検証できます（詳細な手順は本文書の「署名の検証手順」参照）。

## 署名の仕組み

- **証明書**: 個人（柳井建人 / YANAI Taketo）名義の Sectigo OV コード署名証明書。組織ではなく個人名義であり、発行元表示は実名になります
- **秘密鍵**: Google Cloud KMS の HSM 内（RSA 4096・HSM 保護レベル）で生成・保管され、**エクスポート不能**です。開発者の手元にも、GitHub にも、CI ランナーにも秘密鍵は存在しません
- **署名の実行**: GitHub Actions のリリースワークフローから、Workload Identity Federation（WIF）で取得した短命トークンにより KMS へ署名を依頼します。長期の資格情報（API キー・秘密鍵ファイル）はどこにも保存していません
- **承認ゲート**: 署名ジョブは GitHub Environment `release-signing`（required reviewers = オーナー）の承認がないと実行されません。この条件は GCP 側の WIF 信頼条件でも強制されており、承認を経ないジョブには GCP がトークン自体を発行しません
- **タイムスタンプ**: すべての署名に RFC 3161 タイムスタンプを付与します（主: Sectigo、副: DigiCert のフォールバック。取得できない場合、リリースは失敗します）。証明書の有効期限が切れても、タイムスタンプ時点で有効だった署名は有効であり続けます

## 署名者の期待値

検証時は「署名が有効か」だけでなく、**署名者がこの値と一致するか**を必ず確認してください（攻撃者が自分の正規証明書で署名した偽物も「有効な署名」にはなります）。

| 項目 | 期待値 |
|---|---|
| Subject（署名者） | `CN=柳井建人, O=柳井建人, S=Yamaguchi, C=JP` |
| 証明書 thumbprint（SHA-1） | `36257D1DC06F7CDECA61B9727A88320DA24D27B6` |
| 発行 CA | Sectigo Public Code Signing CA R36 |

証明書の更新（年次）で thumbprint は変わります。本表は証明書更新のたびに改訂し、リリースノートでも告知します（現行証明書の有効期間: 2026-07-23 〜 2027-07-16）。

## 署名の検証手順

実行しやすい順に記載します。

1. **GUI（追加ツール不要）**: MSI を右クリック → プロパティ →「デジタル署名」タブ。署名者名が「柳井建人」であり、「詳細」→「証明書の表示」の拇印が上記の期待値と一致することを確認します
2. **PowerShell（標準搭載）**:
   ```powershell
   Get-AuthenticodeSignature .\Yagura-<版>-x64.msi | Format-List Status, SignerCertificate
   ```
   `Status` が `Valid` で、`SignerCertificate` の Subject・Thumbprint が期待値と一致することを確認します
3. **signtool（Windows SDK が必要）**: `signtool verify /pa /v <ファイル>`。本番サーバには SDK を入れられないことが多いため、上の 2 つを推奨します

## 署名対象の範囲

- **署名するもの**: MSI 本体、MSI に含まれる自前アセンブリ（`Yagura.Host.exe`・`Yagura.*.dll`）
- **署名しないもの**: 同梱する .NET ランタイム（Microsoft 署名済み）と第三者 OSS の DLL（上流の配布形態のまま同梱）
- この混在のため、**WDAC / AppLocker の「発行元」ルールでインストール内容全体を一律に許可することはできません**（詳細は [SECURITY.md](../SECURITY.md)）

## 証明書の共用について（正確な開示）

この証明書は、同一開発者の別プロジェクト **[Open DUMP Viewer](https://github.com/Open-DUMP-Viewer/Open-DUMP-Viewer)（ODV）と 1 枚を共用**しています。

- **要約: どちらかのプロジェクトの CI が侵害されると、もう一方の「本物と区別できない」署名が作られえます**
- リポジトリごとに署名用サービスアカウントを分離し、Cloud Audit Logs で全署名操作を記録していますが、両者は同一の鍵に対する署名権限を持つため、これは**事後の検知・帰属**の仕組みであって**予防ではありません**。ODV の経路で作られた署名は、利用者が本ページの期待値（Subject・thumbprint）で照合しても Yagura の正規署名と区別できません
- 相互汚染リスクが許容できないと判明した場合は証明書を 2 枚に分離します（再評価トリガとして管理）

## 体制（実態どおりの開示）

| 役割 | 現在の担当 | 意味 |
|---|---|---|
| Authors | YANAI Taketo（リポジトリオーナー） | 追加レビューなしにソースを変更できる者 |
| Reviewers | YANAI Taketo | 外部からの PR は必ずレビューを経てマージされる |
| Approvers | YANAI Taketo | 署名（GitHub Environment）を承認する者 |

- 現在は単一メンテナ体制であり、**役割分離は形式的にしか成立していません**（同一人物が全役割を兼ねます）
- **ブランチ保護はオーナー自身には適用されていません**（`enforce_admins: false`）。外部 PR にはレビューが必須ですが、オーナー自身の変更にはレビューがありません。コントリビュータが増えた時点で役割を分け、`enforce_admins: true` へ格上げします
- コミット権・署名承認権を持つ者（現在はオーナーのみ）は GitHub / Google Cloud の **MFA を必須**としています。PR を送るだけの外部貢献者に氏名掲載や MFA 要件はありません

## プライバシー

- Yagura は**テレメトリを持たず**、利用者が設定した宛先（syslog の転送先・通知先など）以外へ情報を送信しません
- 例外的に外部への通信が発生するのは、**利用者が明示的に有効化・実行した操作**のみです: 逆引き（PTR）の DNS 問い合わせ（[ADR-0007](adr/0007-reverse-dns-display.md)。既定は無効）、フォワーダ配布キット生成時の Fluent Bit 取得（管理者の明示操作）
- 署名者の実名・所在地（県）は証明書の仕様上、署名済みバイナリを通じて公開されます。これは開発者本人が了承のうえ選択した公開です
