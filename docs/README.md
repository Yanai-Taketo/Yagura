# ドキュメント体系

仕様が差分文書の積み重ねになって全体像が見えなくなることを防ぐため、本リポジトリは **2 層構造** でドキュメントを維持する。

## 第 1 層: 全体設計書（常に現在形）— docs/design/

システムの「今の姿」を記述する少数の文書。アーキテクチャ・機能・設定・DB スキーマ・UI 設計をここに集約する。

- **機能を追加・変更する PR は、対応する全体設計書の更新を同じ PR に含めることを必須とする**（レビューで担保）
- 「歴史」は書かない。過去の経緯・却下案は ADR に書く
- 読者が「この文書だけ読めば現状がわかる」状態を維持する

## 第 2 層: ADR（決定の履歴）— docs/adr/

「なぜそうしたか」を時系列で積む。MADR 形式。一度書いた ADR は書き換えず、決定が変わったら supersession（新 ADR で置換）、決定は同じで補足だけなら amendment（改訂履歴セクションに追記）とする。

## 利用者ガイド — docs/guides/

利用者向けの手順書（how-to）。「Yagura と組み合わせて何かを実現する手順」を 1 テーマ 1 文書で置く
（例: [Windows イベントログを Yagura へ転送する](guides/forward-windows-eventlog.md)）。

- 全体設計書と同じく**常に現在形**を維持する（手順が変わったら文書を書き換える。差分を積まない）
- Yagura 自身の仕様は書かない（仕様の正は全体設計書。ガイドは外部ツールとの組み合わせ手順に限る）

## 提案書（実装着手前のみ有効）

大きな機能の実装着手前に提案書（proposal）を書く場合は docs/proposals/ に置くが、**マージ後の正は必ず全体設計書に反映し、提案書は archived 扱いにする**。提案書を「仕様の正」として参照し続けない。

## 開発規約 — docs/development/

コミット規約・ブランチ戦略・コーディング規約（[conventions.md](development/conventions.md)）と、設計文書のレビュー制度（[persona-review.md](development/persona-review.md)）。
