# 旧リポジトリ（yagura-dev）から蒸留した教訓

旧リポジトリ Yanai-Taketo/yagura-dev（v0.5.0 リリース済み、v0.6 途中で開発中断、PR 138 本 / Issue 16 本 / SPEC 26 本 / ADR 11 本）の全数調査（2026-07-03 実施）から抽出した教訓。**設計・実装の判断前に該当節を参照すること。** 出典の PR/Issue 番号は旧リポジトリのもの。

## A. アーキテクチャに効く技術的教訓

### A-1. カーネル UDP silent drop はアプリからは見えない（Issue #42）
実測: QEMU VM で OS は 5,812 msg/sec 受信、アプリの Parsed は約 325 msg/sec、アプリ内 Dropped カウンタは 0 のまま **約 5,500 msg/sec がカーネルのソケットバッファで silent drop** していた。
→ 新実装では **v0.1 から** OS レベルカウンタ（UDPv4 パフォーマンスカウンタ）の統合と SO_RCVBUF の設定化（既定 ~64KB は小さすぎる。Windows では約 1MB 超の指定は setsockopt が失敗する点に注意）を組み込む。性能要件の検証はアプリ内メトリクスでなく OS 側カウンタとの突合で行う。

### A-2. 内部キュー容量は実測なしに決めない（PR #146/#147/#153）
v0.1 で決めたチャネル容量 10,000 件が、v0.6 の初ベンチ（3k msg/sec）で 60% 損失を起こすまで**一度も実測検証されないまま放置**された。真因の特定にも「ドロップ内訳カウンタのベンチ結果への露出」が必要だった。
→ **スループットベンチハーネスを v0.1 相当の早期に整備**し、容量・バッチサイズ等の設計値は実測で決める。ドロップは発生箇所別（OS / 内部キュー / 流量制御）にカウンタを分けて最初から可視化する。

### A-3. DB 抽象化の後付けは失敗した（SPEC-023、PR #148 で草案止まり）
SQL Server 直書き（Dapper + 生 SQL）が Receiver / WebUI / CLI 全域に染み込み、provider 抽象の後付け改修は着手されないまま開発中断。26 SPEC 中、唯一の未実装。
→ 新実装では **永続化 provider 抽象を初日から**。SQL Server（既定）/ PostgreSQL / MySQL / SQLite（ゼロ設定用）。SQL 方言差は dialect 層に封じ込める。

### A-4. 信頼性機構は増築でなく標準装備に（SPEC-024/025、B-1）
ディスクスプール（DB 切断時退避）、送信元別流量制御（token bucket）、キュー容量拡張は、すべて v0.6 で後付けの増築になった。
→ 受信パイプラインの初期設計に、バックプレッシャ・スプール・流量制御の挿入点を最初から用意する。

### A-5. 検索 UX は「絞り込み強制なし + 上限 cap + 軽量 projection」が最終形（Issue #41/#104、SPEC-009→022）
「性能防御のため時間範囲のみの検索を拒否する」という初期設計は、主要 syslog ツールほぼすべてと逆の挙動で、operator の実利用と衝突し続け、**SPEC 3 本 + 実装 3 本かけて解体**された。技術的には clustered PK (ReceivedAt, Id) なら時間範囲クエリは元々速く、守るべきは自由文 LIKE と転送量だけ（一覧は LEFT(Message, 200) 程度の projection で 150MB→25MB に圧縮できる試算が #104 にある）。
→ 最初から Windows EventLog Viewer 型（フィルタなし全件可 + 結果上限で防御 + 一覧は軽量列のみ）で設計する。

### A-6. 時刻の扱いは UI 初回設計で契約を決める（Issue #78、PR #93）
サーバ UTC 値をそのまま datetime-local に渡し、日本では 9 時間ずれて見える状態が v0.4 まで放置された。後付け対応は JS 層のパッチになった。
→ 「DB は UTC 保存、表示層で変換」の契約を UI 設計書に明文化してから画面を作る。

### A-7. 最小権限のセットアップ UX 問題には確立済みの解法がある（Issue #35/#36/#37）
実行時権限を INSERT/SELECT に最小化すると、初回セットアップが「7 ステップ・3 コンテキスト跨ぎ」になる代償が生じる。旧リポジトリの解法 3 点セット: **管理操作は CLI サブコマンドに分離（init-db / retention）+ 失敗時に実環境値入りの実行可能 SQL を自動生成 + 冪等実行**。
→ 新実装ではこれを Web UI ウィザード + ゼロ設定ファーストラン（SQLite）でさらに前進させるが、SQL Server 昇格時の権限設計はこの 3 点セットを踏襲する。

### A-8. 設定キーの削除は既存環境を起動不能にする（#104 質問 2）
Options クラスから設定キーを消すと、既存 appsettings.local.json に残った値が検証違反で起動失敗する。
→ 「1 バージョン警告 → 次バージョン削除」の deprecation 階段を規約化する（[Obsolete] + 診断 ID）。設定スキーマは additive-only を原則とする。

## B. 検証・テストの教訓

### B-1. CI green ≠ 本番手順の正しさ（Issue #35）
CI の E2E は LocalDB（暗黙 sysadmin）で走っていたため、本番手順書の CREATE LOGIN 欠落（Error 18456 で起動不能）を検出できなかった。
→ **非特権サービスアカウント構成の E2E** を早期に持つ。

### B-2. 実環境でしか出ないバグ領域がある（PR #131/#133/#140）
gMSA の SPN 検証 false negative（AD lab で初検出）、MSI の Permanent/REINSTALLMODE 挙動差（実 CI で初検出）は、単体テストでは原理的に守れない。
→ AD / MSI アップグレード等の実環境依存機能は、リリース準備でなく**実装の受け入れ条件に lab 検証を組み込む**。

### B-3. 技術的同等性の主張は推測で通さない（旧リポジトリで 15 件の trap 累積）
「X は Y と同等」「A で B を代用できる」型の主張が 15 回、実機と食い違った。代表例: `IPNetwork.Parse(単一IP)` は公式ドキュメントの印象と異なり FormatException を投げる（PR #150 で事前予測により寸前阻止）/ `REINSTALLMODE=amus` は Major Upgrade の代用にならない（PR #140 hotfix）。
→ equivalence claim は公式ドキュメント引用または実機確認 1 回を通してから記述・実装する。

### B-4. 外部依存はライブ検証してから採用する（Issue #75）
学習時点の知識で「Chart.js 4.4.7 (latest stable)」と記載したが実際は 5 ヶ月 stale だった。「頭の中の最新」を「検証済みの最新」と誤申告する構造的リスクがある。
→ 採用時に `npm view` / `gh api` / NuGet 等で当日の最新版を確認し、`Vendored: <pkg> <ver> (verified <cmd>, YYYY-MM-DD)` 形式の記録を PR に残す。

### B-5. 時間窓テストは 1 つの基準時刻から両端を作る（PR #65）
`DateTime.UtcNow` を 2 回読むとマイクロ秒ドリフトで flaky になる。

### B-6. MSI は build ごとに非再現（PR #136）
同一コミットでも MSI の size/SHA256 は数十 KB 変動する。CHANGELOG には「約 N MB」+ Release artifact の .sha256 を正とし、byte-exact 値を書かない。**公開済みバージョンタグの再 push 禁止**（旧リポジトリで 2 回踏んだ）。

## C. プロセス・運用の教訓

### C-1. 個人運用のブランチ保護は enforce_admins:false + admin merge（Issue #2）
GitHub は自分の PR を自分で approve できないため、規約通りの保護設定では merge 不能になる。contributor 参加時に enforce_admins:true へ格上げするトリガ設計まで Issue #2 に残っている。

### C-2. squash merge race に注意（PR #140）
fix を push した 2 分後に merge すると fix が main に入らないことがある。**CI green を確認してから merge** を規約化。

### C-3. リリース workflow には contents: write が必要（PR #34）
GITHUB_TOKEN は既定 read-only。v0.1.0 リリースが 403 で失敗した。

### C-4. ドキュメント追従 PR が全体の 2 割を占めた
旧方式（SPEC status 遷移・実装 PR 紐付けの後追い docs PR）は PR 138 本中 2 割弱を消費した。新リポジトリの 2 層ドキュメント体系（設計書更新を実装 PR に同梱）はこの圧縮が目的。

### C-5. 監査は「問題がなかった証拠」も記録する（Issue #29）
指摘一覧だけでなく、grep コマンドレベルで「秘密情報 commit 0 件」等の陰性所見を残すと、外部への説明責任に使える。

### C-6. サイズ等の NFR 上限は依存追加時に事前承認する（PR #74/#103）
MSI 40MB 上限が Chart.js / MailKit 追加で 2 回事後緩和になった。依存追加を決める時点で NFR への影響を併記して承認を取る。

## D. 旧リポジトリで未完のまま残った宿題（新実装での扱い）

| 項目 | 旧状態 | 新実装での扱い |
|---|---|---|
| DB 抽象化（PostgreSQL/MySQL） | SPEC-023 草案止まり | 初期設計に内蔵（方針確定済み） |
| 自由文検索の高速化（FULLTEXT 等） | 2 回 deferral | 検索設計時に再評価 |
| 受信スレッド多重化（SO_REUSEPORT 相当） | 先送りのまま | ベンチ結果を見て判断 |
| 無限スクロール検索 UX | 先送りのまま | UI 全面刷新の中で再検討 |
| 大規模データ（1M/10M 行）実測 | 手動チェックリスト運用 | 計測環境の持ち方を初期に設計 |
| コード署名（Authenticode） | v0.7 目標のまま未達 | v1.0 公開基準に含める（旧 ADR-0011 継承） |
| Sender Agent 自作 | ADR-0007 で deferral（NXLog CE が主経路） | 継承（自作しない。再評価トリガも継承） |
